# PEP 518 – 指定 Python 项目的最低构建系统要求

**Metadata:**

- **Author**: Brett Cannon <brett&#32;&#97;t&#32;python.org>, Nathaniel J. Smith <njs&#32;&#97;t&#32;pobox.com>, Donald Stufft <donald&#32;&#97;t&#32;stufft.io>
- **BDFL-Delegate**: Alyssa Coghlan
- **Created**: 10-May-2016
- **Post-History**: 10-May-2016, 11-May-2016, 13-May-2016

## 摘要

本 PEP 指定了 Python 软件包如何指定其构建依赖，以便执行其选择的构建系统。作为此规范的一部分，引入了一个新的配置文件，供软件包用来指定其构建依赖（期望相同的配置文件将用于未来的配置详细信息）。

## 理由

当 Python 最初开发构建项目软件分发的工具时，distutils [1] 是选定的解决方案。随着时间的推移，setuptools [2] 获得了流行度，在 distutils 之上添加了一些功能。两者都使用了 setup.py 文件的概念，项目维护者执行该文件来构建其软件的分发（以及用户安装所述分发）。

使用可执行文件来指定 distutils 下的构建要求不是问题，因为 distutils 是 Python 标准库的一部分。将构建工具作为 Python 的一部分意味着 setup.py 没有外部依赖，项目维护者无需担心即可构建其项目的分发。无需指定任何依赖信息，因为唯一的依赖是 Python。

但是当项目选择使用 setuptools 时，使用像 setup.py 这样的可执行文件就成了问题。你不知道它的依赖关系就不能执行 setup.py 文件，但目前没有标准方法可以在不执行存储该信息的 setup.py 文件的情况下以自动化方式知道这些依赖关系是什么。这是一个先有鸡还是先有蛋的问题：文件在不知道其内容之前无法运行，而内容除非运行文件否则无法通过编程方式知道。

Setuptools 试图通过向其 setup() 函数添加 setup_requires 参数来解决这个问题 [3]。此解决方案存在许多问题，例如：

- 除了 setuptools 本身之外，没有工具可以在不执行 setup.py 的情况下访问这些信息，但如果没有安装这些项目就无法执行 setup.py。

- 虽然 setuptools 本身将安装在此列出的任何内容，但它们不会在安装过程中安装，而是在 setup() 函数执行期间安装，这意味着实际上使用此处添加的任何内容的唯一方法是通过越来越复杂的操作，将导入和使用这些模块延迟到 setup() 函数执行的后期。

- 这不能包括 setuptools 本身，也不能包括 setuptools 的替代品，这意味着像 numpy.distutils 这样的项目基本上无法利用它，项目无法利用更新的 setuptools 功能，直到他们的用户自然地将 setuptools 版本升级到更新版本。

- 列在 setup_requires 中的项目在每次执行 setup.py 时都会隐式安装，但执行 setup.py 的常见方式之一是通过另一个工具，如 pip，它已经在管理依赖关系。这意味着像 pip install spam 这样的命令可能最终导致 pip 和 setuptools 都下载和安装包，最终用户需要配置两个工具（对于 setuptools，不在控制调用范围内）以更改设置，例如从哪个仓库安装。这也意味着用户需要了解两个工具的发现规则，因为一个可能支持不同的包格式或以不同方式确定最新版本。

这最终导致 setup_requires 的使用很少，项目倾向于在 setup.py 文件之间简单地复制和粘贴片段，或者完全回避它，而选择在其他地方记录他们期望用户在尝试构建或安装项目之前手动安装的内容。

所有这些都导致 pip [4] 简单地假设在执行 setup.py 文件时需要 setuptools。但是，这里的问题是，如果另一个项目开始在社区中获得与 setuptools 一样的动力，它就无法扩展。它还阻止其他项目获得动力，因为当 pip 无法推断出需要 setuptools 以外的东西时，使用它与项目所需的摩擦。

本 PEP 试图通过指定一种以声明式方式在特定文件中列出项目构建系统的最小依赖关系的方法来纠正这种情况。这允许项目列出其构建依赖关系，例如从源代码检出到 wheel，而不会陷入 setup.py 的困境，即工具无法推断项目构建自身所需的内容。实现本 PEP 将允许项目提前指定它们依赖的构建系统，以便像 pip 这样的工具可以确保它们已安装，以便运行构建系统来构建项目。

为了为本文档提供更多背景和动机，请考虑生成项目的构建产物所需的（大致）步骤：

- 项目的源代码检出。

- 安装构建系统。

- 执行构建系统。

本 PEP 涵盖步骤#2。PEP 517 涵盖步骤#3，包括如何使构建系统动态指定构建系统执行其工作所需的其他依赖关系。但是，本 PEP 的目的是指定构建系统开始执行所需的最小要求集。

## 规范

### 文件格式

构建系统依赖关系将存储在名为 pyproject.toml 的文件中，该文件以 TOML 格式 [6] 编写。

选择此格式是因为它是人类可用的（不像 JSON [7]），它足够灵活（不像 configparser [9]），源自标准（也不像 configparser [9]），并且不过于复杂（不像 YAML [8]）。TOML 格式已经被 Rust 社区在其 Cargo 包管理器 [14] 中使用，在私人邮件中表示他们对选择 TOML 非常满意。关于为什么没有选择各种替代方案的更详细讨论可以在其他文件格式部分找到。但是，作者确实意识到，配置文件格式的选择最终是主观的，必须做出选择，作者在这种情况下更喜欢 TOML。

下面我们列出了工具应该识别/尊重的表。本 PEP 中未指定的表保留供其他 PEP 将来使用。

### build-system 表

[build-system] 表用于存储构建相关数据。最初，该表只有一个键有效并且是强制性的：requires。此键必须具有一个字符串列表的值，表示执行构建系统所需的 PEP 508 依赖关系（目前这意味着执行 setup.py 文件所需的依赖关系）。

对于绝大多数依赖 setuptools 的 Python 项目，pyproject.toml 文件将是：

```toml
[build-system]
# Minimum requirements for the build system to execute.
requires = ["setuptools"]  # PEP 508 specifications.
```

由于 setuptools 的使用在社区中非常广泛，构建工具预计将使用上面的示例配置文件作为其默认语义，当不存在 pyproject.toml 文件时。

工具不应要求 [build-system] 表存在。pyproject.toml 文件可用于存储构建相关数据以外的配置详细信息，因此合法地缺乏 [build-system] 表。如果文件存在但缺少 [build-system] 表，则应使用上面指定的默认值。如果指定了表但缺少必需字段，则工具应将其视为错误。

### tool 表

[tool] 表是任何与 Python 项目相关的工具（不仅是构建工具）可以让用户指定配置数据的地方，只要它们使用 [tool] 内的子表，例如 flit 工具将将其配置存储在 [tool.flit] 中。

我们需要某种机制来在 `tool.*` 命名空间中分配名称，以确保不同项目不会尝试使用相同的子表并发生冲突。我们的规则是，一个项目可以使用子表 `tool.$NAME`，当且仅当他们拥有 `Cheeseshop/PyPI` 中 `$NAME` 的条目。

### JSON Schema

为了仅为说明目的提供从 TOML 文件生成的数据的类型特定表示，以下 JSON Schema [15] 将匹配数据格式：

```json
{
  "$schema": "http://json-schema.org/schema#",

  "type": "object",
  "additionalProperties": false,

  "properties": {
  "build-system": {
  "type": "object",
  "additionalProperties": false,

  "properties": {
  "requires": {
  "type": "array",
  "items": {
  "type": "string"
  }
  }
  },
  "required": ["requires"]
  },

  "tool": {
  "type": "object"
  }
  }
}
```

## 被拒绝的想法

### 语义版本键

为了配置文件的结构的未来证明，最初提议了一个语义版本键。默认为 1，这个想法是，如果之前定义的键或表发生了任何不向后兼容的语义更改，则语义版本将递增到一个新数字。

但最终，决定这是一个过早的优化。预期配置文件中文本预定义的语义更改将相当保守。在发生向后不兼容更改的情况下，可以使用不同的名称来避免破坏旧工具。

### 更嵌套的命名空间

本 PEP 的早期草稿有一个顶级 `[package]` 表。这个想法是为语义版本控制方案强加一些作用域（参见为什么这个想法被拒绝的语义版本键）。随着对作用域需求的消除，拥有顶级表的观点变得多余。

### 其他表名

为 [build-system] 表提出的另一个名称是 [build]。替代名称更短，但不能传达存储在表中的信息的意图。在 distutils-sig 邮件列表上的投票之后，当前名称胜出。

### 其他文件格式

提出了几种其他文件格式供考虑，都因各种原因被拒绝。关键要求是格式可以由人类编辑，并且具有可以轻松由项目 vendored 的实现。这直接排除了某些格式，如 XML，这些格式对人类不友好，从未认真讨论过。

#### 考虑的文件格式概述

拒绝其他替代方案的关键原因总结如下，而完整的审查（包括支持 TOML 的正面论点）可以在 [16] 找到。

TOML 最终被选中，因为它提供了我们感兴趣的所有功能，同时避免了替代方案引入的缺点。

| 功能 | TOML | YAML | JSON | CFG/INI |
|------|------|------|------|---------|
| Well-defined | yes | yes | yes | |
| Real data types | yes | yes | yes | |
| Reliable Unicode | yes | yes | yes | |
| Reliable comments | yes | yes | | |
| Easy for humans to edit | yes | ?? | | ?? |
| Easy for tools to edit | yes | ?? | yes | ?? |
| In standard library | | yes | yes | |
| Easy for pip to vendor | yes | | n/a | n/a |

（表中的"??"表示大多数人倾向于回答"yes"的项目，但由于缺乏清晰的规范或底层文件格式规范出奇地复杂，在实践中会出现许多怪癖和边缘情况）

pytoml TOML 解析器是大约 300 行的纯 Python 代码，因此不在标准库中并不是很大的缺点。

Python 字面量也被讨论为潜在格式，但在文件格式审查中未考虑（因为它们不是常见的预存在文件格式）。

#### JSON

JSON 格式 [7] 最初被考虑但很快被拒绝。虽然作为人类可读的基于字符串的数据交换格式很棒，但语法不适合人类轻松编辑（例如，语法比必要的更冗长，同时不允许注释）。

拟议数据的示例 JSON 文件为：

```json
{
  "build": {
  "requires": [
  "setuptools",
  "wheel>=0.27"
  ]
  }
}
```

#### YAML

YAML 格式 [8] 旨在成为 JSON [7] 的超集，同时更易于手工处理。YAML 有三个主要问题。

一个是规范很大：如果打印在信纸大小的纸上为 86 页。这留下了某人可能使用 YAML 的一个功能，该功能与一个解析器工作但与另一个不工作。有人建议标准化一个子集，但这基本上意味着为此文件创建一个新标准，长期来看不切实际。

第二个是 YAML 本身默认不安全。规范允许任意执行代码，在处理配置数据时最好避免。当然可以避免这种行为——例如，PyYAML 提供了一个 safe_load 操作——但是如果任何工具不小心使用 load 而不是 safe_load，那么他们就打开了任意代码执行的大门。虽然本 PEP 专注于项目的构建，这 inherently 涉及代码执行，但其他配置数据，如项目名称和版本号，有一天可能会出现在同一个文件中，而不希望任意代码执行。

最后，YAML 最流行的 Python 实现是 PyYAML [10]，这是一个几千行代码和可选 C 扩展模块的大项目。这本身不一定是一个问题，但这对于像 pip 这样的项目来说变得更加成问题，他们可能需要将 PyYAML 作为依赖关系 vendored 以完全自包含（否则你最终会得到安装工具需要安装工具才能工作）。已经完成了 PyYAML 的概念验证重新设计，以查看可能 vendored 库的更简单版本的难易程度，这表明这是可能的。

示例 YAML 文件为：

```yaml
build:
  requires:
  - setuptools
  - wheel>=0.27
```

#### configparser

考虑了基于 configparser [9] 接受的 INI 风格配置文件。不幸的是，configparser 接受的规范不存在，导致版本之间的支持偏差。例如，Python 2.7 中的 ConfigParser 接受的内容与 Python 3 中的 configparser 接受的内容不同。虽然可以标准化 Python 3 接受的内容并简单地 vendored configparser 模块的 backport，但这意味着本 PEP 必须明确规定 configparser 的 backport 必须由所有希望消费本 PEP 指定的元数据的项目使用。这过于限制性，如果某人不知道需要特定版本的 configparser，可能会导致混淆。

示例 INI 文件为：

```ini
[build]
requires =
 setuptools
 wheel>=0.27
```

#### Python 字面量

有人提议使用 Python 字面量作为配置格式。文件将在顶级包含一个 dict，所有数据都在该 dict 内，部分由键定义。所有 Python 程序员都将习惯于该格式，隐式地没有第三方依赖关系来读取配置数据，如果由 ast.literal_eval() [13] 解析则是安全的。
Python 字面量可以与 JSON 相同，具有支持尾随逗号和注释的额外好处。此外，Python 更丰富的数据模型可能对某些未来的配置需求有用（例如非字符串 dict 键、浮点与整数值）。

另一方面，Python 字面量是 Python 特定格式，预计这些数据可能需要由不是用 Python 编写的打包工具等读取。

拟议数据的示例 Python 字面量文件为：

```python
# The build configuration
{"build": {"requires": ["setuptools",
 "wheel>=0.27", # note the trailing comma
 # "numpy>=1.10" # a commented out data line
 ]
# and here is an arbitrary comment.
 }
 }
```

### 坚持使用 setup.cfg

setuptools 使用的 setup.cfg 作为通用格式有两个问题。一个是它们是.ini 文件，如上面的 configparser 讨论中提到的有问题。另一个是该文件的模式从未严格定义，因此不知道哪种格式可以安全使用而不会混淆 setuptools 安装。

### 其他文件名

考虑并拒绝了几个其他文件名（虽然这非常是一个 bikeshedding 主题，所以决定主要取决于品味）。

- **pysettings.toml** - 最合理的替代方案。
- **pypa.toml** - 虽然引用 PyPA [11] 是有意义的，但它是一个有点小众的术语。最好让文件名在没有特定领域知识的情况下有意义。
- **pybuild.toml** - 从本 PEP 的限制性角度来看，这个文件名有意义，但是如果任何非构建元数据 ever 添加到文件中，那么名称就不再有意义。
- **pip.toml** - 太特定于工具。
- **meta.toml** - 太通用；项目可能想要自己的元数据文件。
- **setup.toml** - 虽然由于 setup.py 而保持传统，但它不一定匹配文件将来可能包含的内容（例如，知道项目名称 inherent 是其设置的一部分吗？）。
- **pymeta.toml** - 对编程和/或 Python 的新手来说不明显。
- **pypackage.toml & pypackaging.toml** - "package"是什么的名称混淆（项目与命名空间）。
- **pydevelop.toml** - 文件可能包含不特定于开发的详细信息。
- **pysource.toml** - 与源代码不直接相关。
- **pytools.toml** - 具有误导性，因为文件（目前）旨在进行项目管理。
- **dstufft.toml** - 太特定于个人。;)
