# PEP 3154 – Pickle 协议版本 4

**Metadata:**

- **Author**: Antoine Pitrou <solipsis&#32;&#97;t&#32;pitrou.net>
- **Created**: 11-Aug-2011
- **Python-Version**: 3.4

## Abstract

使用 pickle 模块序列化的数据必须能够跨 Python 版本移植。它还应支持最新的语言功能以及特定于实现的功能。因此，pickle 模块知道几种协议（目前编号从 0 到 3），每种协议出现在不同的 Python 版本中。使用较低的协议版本号允许与旧版 Python 交换数据，而使用较高的协议版本则允许访问新功能，有时还能更有效地利用资源（包括序列化和反序列化所需的 CPU 时间，以及数据传输所需的磁盘空间/网络带宽）。

## Rationale

最新的当前协议，巧合地命名为协议 3，随 Python 3.0 出现，支持语言中的新不兼容功能（主要是默认使用 unicode 字符串和新的 bytes 对象）。当时并没有借此机会以其他方式改进该协议。

本 PEP 旨在在一个新的 pickle 协议版本中推动一系列增量改进。使用 PEP 流程是为了尽可能多地收集改进，因为引入新的 pickle 协议应该是罕见的事情。

## Proposed changes

### Framing

传统上，当从流中反序列化对象时（通过调用 load() 而不是 loads()），可以在类文件对象上发出许多小的 read() 调用，这可能会产生巨大的性能影响。

相比之下，协议 4 具有二进制 framing。因此，pickle 的一般结构如下：

```python
+------+------+
| 0x80 | 0x04 |              protocol header (2 bytes)
+------+------+
|  OP  |                     FRAME opcode (1 byte)
+------+------+-----------+
| MM MM MM MM MM MM MM MM |  frame size (8 bytes, little-endian)
+------+------------------+
| .... |                     first frame contents (M bytes)
+------+
|  OP  |                     FRAME opcode (1 byte)
+------+------+-----------+
| NN NN NN NN NN NN NN NN |  frame size (8 bytes, little-endian)
+------+------------------+
| .... |                     second frame contents (N bytes)
+------+
  etc.
```

为了保持实现简单，禁止 pickle opcode 跨越 frame 边界。pickler 会注意不生成这样的 pickle，unpickler 也会拒绝它们。此外，没有"最后一个 frame"标记。最后一个 frame 只是以 STOP opcode 结尾的那个。

编写良好的 C 实现不需要为 framing 层进行额外的内存复制，从而保持一般的（反）序列化效率。

>**注意**
pickler 如何决定将 pickle 流分割成 frames 是一个实现细节。例如，"关闭"一个 frame 当它达到~64 KiB 时，对于性能和 pickle 大小开销来说都是一个合理的选择。

### Binary encoding for all opcodes

GLOBAL opcode 仍在协议 3 中使用，它使用 pickle 协议的所谓"文本"模式，这涉及在 pickle 流中查找换行符。它还使二进制 framing 的实现复杂化。

协议 4 禁止使用 GLOBAL opcode，并用 STACK_GLOBAL 替换它，这是一个新的 opcode，它从栈中获取操作数。

### Serializing more "lookupable" objects

默认情况下，pickle 只能序列化模块全局函数和类。支持其他类型的对象，如非绑定方法 [4]，是一个常见请求。实际上，第三方对其中一些（如绑定方法）的支持已在 multiprocessing 模块中实现 [5]。

PEP 3155 中的 `__qualname__` 属性使得通过名称查找更多对象成为可能。让 STACK_GLOBAL opcode 接受点分隔的名称将允许标准 pickle 实现支持所有这些类型的对象。

### 64-bit opcodes for large objects

当前协议版本将各种内置类型（str、bytes）的对象大小导出为 32 位整数。这禁止了大数据的序列化 [1]。需要新的 opcode 来支持非常大的 bytes 和 str 对象。

### Native opcodes for sets and frozensets

许多常见的内置类型（如 str、bytes、dict、list、tuple）都有专用的 opcode 来提高序列化和反序列化时的资源消耗；然而，sets 和 frozensets 没有。添加这样的 opcode 将是一个明显的改进。此外，专用的 set 支持可以帮助消除当前无法序列化自引用 sets 的问题 [2]。

### Calling \_\_new\_\_ with keyword arguments

目前，`__new__` 强制使用关键字参数的类不能被 pickle（或者说，不能被 unpickle）[3]。需要一个新的特殊方法（`__getnewargs_ex__`）和一个新的 opcode（NEWOBJ_EX）。如果存在，`__getnewargs_ex__` 方法必须返回一个二元组 (args, kwargs)，其中第一项是位置参数的元组，第二项是类的 `__new__` 方法的关键字参数的字典。

### Better string encoding

短 str 对象目前将其长度编码为 4 字节整数，这是浪费的。具有 1 字节长度的特定 opcode 将使许多 pickle 更小。

### Smaller memoization

PUT opcode 都需要一个显式索引来选择在 memo 字典的哪个条目中缓存栈顶元素。然而，在实践中，这些数字是按顺序分配的。一个新的 opcode，MEMOIZE，将栈顶对象存储在索引等于当前 memo 字典大小的位置。这使得 pickle 更短，因为 PUT opcode 为所有非原子数据类型发出。

## Summary of new opcodes

这些反映了所提议实现的状态（主要归功于 Alexandre Vassalotti 的工作）：

- `FRAME`：引入一个新 frame（后跟 8 字节 frame 大小和 frame 内容）。

- `SHORT_BINUNICODE`：推送一个 utf8 编码的 str 对象，带有一字节大小前缀（因此长度小于 256 字节）。

- `BINUNICODE8`：推送一个 utf8 编码的 str 对象，带有八字节大小前缀（对于长于 2**32 字节的字符串，因此无法使用 BINUNICODE 序列化）。

- `BINBYTES8`：推送一个 bytes 对象，带有八字节大小前缀（对于长于 2**32 字节的 bytes 对象，因此无法使用 BINBYTES 序列化）。

- `EMPTY_SET`：在栈上推送一个新的空 set 对象。

- `ADDITEMS`：将最顶层的栈项添加到 set 中（与 EMPTY_SET 一起使用）。

- `FROZENSET`：从最顶层的栈项创建一个 frozenset 对象，并将其推入栈中。

- `NEWOBJ_EX`：取最顶层的三个栈项 cls、args 和 kwargs，并推送调用 `cls.__new__(*args, **kwargs)` 的结果。

- `STACK_GLOBAL`：取最顶层的两个栈项 module_name 和 qualname，并推送在名为 module_name 的模块中查找点分隔的 qualname 的结果。

- `MEMOIZE`：将栈顶对象存储在 memo 字典中，索引等于当前 memo 字典的大小。

## Alternative ideas

### Prefetching

Serhiy Storchaka 建议用一个特殊的 PREFETCH opcode（带 2 或 4 字节参数）替换 framing，以显式声明已知的 pickle 块。大数据可以在这样的块之外进行 pickle。一个简单的 unpickler 应该能够跳过 PREFETCH opcode 并仍然正确解码 pickle，但良好的错误处理将需要检查 PREFETCH 长度是否落在 opcode 边界上。
