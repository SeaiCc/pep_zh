# PEP 3116 – 新 I/O

**元数据:**

- **作者**: Daniel Stutzbach <daniel at stutzbachenterprises.com>, Guido van Rossum <guido at python.org>, Mike Verdone <mike.verdone at gmail.com>
- **创建时间**: 26-Feb-2007
- **Python 版本**: 3.0
- **发布历史**: 26-Feb-2007

## 理由与目标

Python 允许使用各种流式（也称为文件式）对象，这些对象可以通过 read() 和 write() 调用来使用。任何提供 read() 和 write() 的对象都是流式的。然而，更多高级且极其有用的函数如 readline() 或 seek() 可能在每个流式对象上不一定可用。Python 需要一个基本的基于字节的 I/O 流规范，以便我们可以添加缓冲和文本处理功能。

一旦我们有了一个定义的原始基于字节的 I/O 接口，我们就可以在任何基于字节的 I/O 类之上添加缓冲和文本处理层。相同的缓冲和文本处理逻辑可用于文件、套接字、字节数组或 Python 程序员开发的自定义 I/O 类。开发一个标准流定义使我们能够将基于流的操作（如 read() 和 write()）与实现特定的操作（如 fileno() 和 isatty()）分离开来。它鼓励程序员编写使用流作为流的代码，而不要求所有流都支持特定于文件或特定于套接字的操作。

新 I/O 规范旨在与 Java I/O 库相似，但通常不太令人困惑。不想在新 I/O 世界中折腾的程序员可以期望 open() 工厂方法将生成一个与旧式文件对象向后兼容的对象。

## 规范

Python I/O 库将包含三个层：原始 I/O 层、缓冲 I/O 层和文本 I/O 层。每一层都由一个抽象基类定义，该基类可以有多个实现。原始 I/O 和缓冲 I/O 层处理字节单元，而文本 I/O 层处理字符单元。

## 原始 I/O

原始 I/O 的抽象基类是 RawIOBase。它有多个方法，这些方法是围绕适当的操作系统调用的包装器。如果其中一个函数在对象上没有意义，实现必须引发 IOError 异常。例如，如果文件是以只读方式打开的，.write() 方法将引发 IOError。再举一个例子，如果对象代表一个套接字，则 .seek()、.tell() 和 .truncate() 将引发 IOError。通常，对这些函数之一的调用映射到恰好一个操作系统调用。

`.read(n: int) -> bytes`
> 从对象中读取最多 n 个字节并返回它们。如果操作系统调用返回少于 n 个字节，则可能返回少于 n 个字节。如果返回 0 个字节，表示文件结尾。如果对象处于非阻塞模式且没有可用字节，调用返回 None。

`.readinto(b: bytes) -> int`

> 从对象中读取最多 len(b) 个字节并将它们存储在 b 中，返回读取的字节数。与 .read 类似，可能读取少于 len(b) 个字节，0 表示文件结尾。如果非阻塞对象没有可用字节，则返回 None。b 的长度永远不会改变。

`.write(b: bytes) -> int`

> 返回写入的字节数，可能小于 len(b)。

`.seek(pos: int, whence: int = 0) -> int`

`.tell() -> int`

`.truncate(n: int = None) -> int`

`.close() -> None`

此外，它还定义了一些其他方法：

`readable() -> bool`
> .如果对象是为读取而打开的，则返回 True，否则返回 False。如果为 False，调用 .read() 将引发 IOError。

`.writable() -> bool`

> 如果对象是为写入而打开的，则返回 True，否则返回 False。如果为 False，调用 .write() 和 .truncate() 将引发 IOError。

`.seekable() -> bool`

> 如果对象支持随机访问（如磁盘文件），则返回 True；如果对象仅支持顺序访问（如套接字、管道和 tty），则返回 False。如果为 False，调用 .seek()、.tell() 和 .truncate() 将引发 IOError。

`.__enter__() -> ContextManager`

> 上下文管理协议。返回 self。

`.__exit__(...) -> None`

> 上下文管理协议。与 .close() 相同。

当且仅当 RawIOBase 实现操作底层文件描述符时，它还必须提供 .fileno() 成员函数。这可以由实现特定定义，也可以使用混合类（需要就此作出决定）。

`.fileno() -> int`
> 返回底层文件描述符（一个整数）

最初，将提供三个实现 RawIOBase 接口的实现：FileIO、SocketIO（在 socket 模块中）和 ByteIO。每个实现必须确定对象是否支持随机访问，因为用户提供的信息可能不够（考虑 open("/dev/tty", "rw") 或 open("/tmp/named-pipe", "rw")）。例如，FileIO 可以通过调用 seek() 系统调用来确定这一点；如果它返回错误，则对象不支持随机访问。每个实现可以提供适合其类型的附加方法。ByteIO 对象类似于 Python 2 的 cStringIO 库，但是操作新的 bytes 类型而不是字符串。

## 缓冲 I/O

下一层是缓冲 I/O 层，它提供对文件式对象的更有效的访问。所有缓冲 I/O 实现的抽象基类是 BufferedIOBase，它提供与 RawIOBase 类似的方法：

`.read(n: int = -1) -> bytes`
> 从对象中返回下一个 n 个字节。如果遇到文件结尾或对象是非阻塞的，它可能返回少于 n 个字节。0 字节表示文件结尾。此方法可能多次调用 RawIOBase.read() 来收集字节，或者如果所有所需字节都已经缓冲，则可能不调用 RawIOBase.read()。

`.readinto(b: bytes) -> int`

`.write(b: bytes) -> int`

> 将 b 字节写入缓冲区。字节不保证立即写入原始 I/O 对象；它们可能会被缓冲。返回 len(b)。

`.seek(pos: int, whence: int = 0) -> int`

`.tell() -> int`

`.truncate(pos: int = None) -> int`

`.flush() -> None`

`.close() -> None`

`.readable() -> bool`

`.writable() -> bool`

`.seekable() -> bool`

`.__enter__() -> ContextManager`

`.__exit__(...) -> None`

此外，抽象基类提供一个成员变量：

`.raw`
> 对底层 RawIOBase 对象的引用。

BufferedIOBase 方法签名与 RawIOBase 的签名几乎完全相同（例外：write() 返回 None，read() 的参数是可选的），但可能有不同的语义。特别是，BufferedIOBase 实现可以读取比请求更多的数据，或者使用缓冲区延迟写入数据。在大多数情况下，这对用户是透明的（除非，例如，他们通过不同的描述符打开相同的文件）。此外，原始读取可能在没有任何特定原因的情况下返回短读取；缓冲读取仅在到达 EOF 时才返回短读取；原始写入可能返回短计数（即使未启用非阻塞 I/O！），而缓冲写入在无法写入或缓冲所有字节时将引发 IOError。

下面描述了 BufferedIOBase 抽象基类的四个实现。

### BufferedReader

BufferedReader 实现用于顺序访问只读对象。其 .flush() 方法是无操作。

### BufferedWriter

BufferedWriter 实现用于顺序访问只写对象。其 .flush() 方法强制所有缓存的数据被写入底层 RawIOBase 对象。

### BufferedRWPair

BufferedRWPair 实现用于顺序访问读写对象，如套接字和 tty。由于这些对象的读取和写入流是完全独立的，它可以通过简单地结合 BufferedReader 和 BufferedWriter 实例来实现。它提供了一个 .flush() 方法，具有与 BufferedWriter 的 .flush() 方法相同的语义。

### BufferedRandom

BufferedRandom 实现用于所有随机访问对象，无论是只读、只写还是读写。与操作顺序访问对象的先前类相比，BufferedRandom 类必须处理用户调用 .seek() 来重新定位流的情况。因此，BufferedRandom 的实例必须跟踪对象内的逻辑位置和真实位置。它提供了一个 .flush() 方法，强制所有缓存的写入数据被写入底层 RawIOBase 对象，所有缓存的读取数据被遗忘（以便将来的读取被迫回到磁盘）。

问：我们是否想在规范中规定在读写对象上切换读取和写入意味着 .flush()？还是这是用户不应依赖的实现便利？

对于只读的 BufferedRandom 对象，.writable() 返回 False，.write() 和 .truncate() 方法抛出 IOError。

对于只写的 BufferedRandom 对象，.readable() 返回 False，.read() 方法抛出 IOError。

## 文本 I/O

文本 I/O 层提供从流中读取和写入字符串的函数。一些新功能包括通用换行符和字符集编码和解码。文本 I/O 层由 TextIOBase 抽象基类定义。它提供几个与 BufferedIOBase 方法类似的方法，但是在每个字符的基础上操作，而不是在每个字节的基础上操作。这些方法是：

`.read(n: int = -1) -> str`

`.write(s: str) -> int`

`.tell() -> object`

返回描述当前文件位置的 cookie。cookie 的唯一支持用途是与 .seek() 一起使用，whence 设置为 0（即绝对寻址）。

`.seek(pos: object, whence: int = 0) -> int`

> 寻址到位置 pos。如果 pos 非零，它必须是 .tell() 返回的 cookie，且 whence 必须为零。

`.truncate(pos: object = None) -> int`

> 与 BufferedIOBase.truncate() 类似，除了 pos（如果不是 None）必须是之前由 .tell() 返回的 cookie。

与原始 I/O 不同，.seek() 的单位未指定 - 一些实现（例如 StringIO）使用字符，其他实现（例如 TextIOWrapper）使用字节。零的特殊情况是为了允许在没有先前的 .tell() 的情况下转到流的开头或结尾。实现可以在 .tell() 返回的 cookie 中包含流编码器状态。

TextIOBase 实现还提供几个直接传递给底层 BufferedIOBase 对象的方法：

`.flush() -> None`

`.close() -> None`

`.readable() -> bool`

`.writable() -> bool`

`.seekable() -> bool`

TextIOBase 类实现还额外提供以下方法：

`.readline() -> str`
> 读取直到换行符或 EOF，并返回该行，如果立即遇到 EOF 则返回 ""。

`.__iter__() -> Iterator`

> 返回一个从文件中返回行的迭代器（碰巧是 self）。

`.next() -> str`

> 与 readline() 相同，除了如果立即遇到 EOF 则引发 StopIteration。

Python 库将提供两个实现。主要实现 TextIOWrapper 包装一个缓冲 I/O 对象。每个 TextIOWrapper 对象有一个名为 .buffer 的属性，它提供对底层 BufferedIOBase 对象的引用。其初始化器具有以下签名：

`.__init__(self, buffer, encoding=None, errors=None, newline=None, line_buffering=False)`
> buffer 是对要用 TextIOWrapper 包装的 BufferedIOBase 对象的引用。
encoding 指的是用于在字节表示和字符表示之间转换的编码。如果它是 None，则系统区域设置将用作默认值。
errors 是一个可选字符串，指示错误处理。它可以在设置 encoding 时设置。默认为 'strict'。
newline 可以是 None、''、'\n'、'\r' 或 '\r\n'；所有其他值都是非法的。它控制行尾的处理。工作方式如下：

- 对于输入，如果 newline 是 None，则启用通用换行符模式。输入中的行可以以 '\n'、'\r' 或 '\r\n' 结尾，这些在返回给调用者之前被转换为 '\n'。如果它是 ''，则启用通用换行符模式，但行尾未经转换地返回给调用者。如果它具有任何其他合法值，输入行仅由给定字符串终止，行尾未经转换地返回给调用者。（换句话说，只有当 newline 是 None 时才发生到 '\n' 的转换。）

- 对于输出，如果 newline 是 None，任何写入的 '\n' 字符都被转换为系统默认行分隔符 os.linesep。如果 newline 是 ''，则不进行转换。如果 newline 是任何其他合法值，任何写入的 '\n' 字符都被转换为给定字符串。（注意，指导转换的规则对于输出与输入不同。）

> line_buffering，如果为 True，则导致如果写入的字符串包含至少一个 '\n' 或 '\r' 字符，write() 调用暗示 flush()。当 open() 检测到底层流是 TTY 设备时，或者当传递缓冲参数 1 时，这将被设置。
关于 newline 参数的进一步说明：

- 仍然需要 '\r' 支持用于某些产生使用 '\r' 行尾文件的 OSX 应用程序；Excel（导出到文本时）和 Adobe Illustrator EPS 文件是最常见的例子。

- 如果启用转换，无论调用哪个方法进行读取或写入，它都会发生。例如，f.read() 将始终产生与 ''.join(f.readlines()) 相同的结果。

- 如果在输入上请求不带转换的通用换行符（即 newline=''），如果系统读操作返回以 '\r' 结尾的缓冲区，则执行另一个系统读操作以确定它后面是否跟随 '\n'。在带转换的通用换行符模式中，第二个系统读操作可能会推迟到下一次读请求，如果接下来的系统读操作返回以 '\n' 开头的缓冲区，则该字符将被简单地丢弃。

另一个实现 StringIO 创建一个没有底层缓冲 I/O 对象的文件式 TextIO 实现。虽然可以通过将 BytesIO 对象包装在 TextIOWrapper 中来提供类似的功能，但 StringIO 对象允许更高的效率，因为它不需要实际执行编码和解码。字符串 I/O 对象可以简单地按原样存储编码的字符串。StringIO 对象的 `__init__` 签名接受一个可选字符串，指定初始值；初始位置始终为 0。它不支持编码或换行符转换；你总是精确地读取你写入的字符。

### Unicode 编码/解码问题

我们应该允许稍后更改编码和错误处理设置。Text I/O 操作在面对 Unicode 问题和歧义（例如变音符号、代理项、编码中的无效字节）时的行为应与 unicode encode()/decode() 方法相同。可能会引发 UnicodeError。

实现注意：我们应该能够重用 codecs 模块提供的大多数基础设施。如果它不提供我们需要的确切 API，我们应该重构它以避免重复造轮子。

## 非阻塞 I/O

非阻塞 I/O 仅在原始 I/O 级别完全支持。如果原始对象处于非阻塞模式，并且操作将会阻塞，则 .read() 和 .readinto() 返回 None，而 .write() 返回 0。为了使对象处于非阻塞模式，用户必须提取 fileno 并手动完成。

在缓冲 I/O 和文本 I/O 层，如果读取或写入由于非阻塞条件而失败，它们会引发 IOError，errno 设置为 EAGAIN。

最初，我们考虑传播原始 I/O 行为，但提出了许多角落情况和问题。为了解决这些问题，必须对缓冲 I/O 和文本 I/O 层进行重大更改。例如，.flush() 在缓冲非阻塞对象上应该做什么？用户如何指示对象"从缓冲区写入尽可能多的内容，但不要阻塞"？不一定会刷新所有可用数据的非阻塞 .flush() 是违反直觉的。由于非阻塞和阻塞对象在这些层具有如此不同的语义，大家同意放弃将它们组合成单一类型的努力。

## open() 内置函数

open() 内置函数由以下伪代码指定：

```python
def open(filename, mode="r", buffering=None, *,
         encoding=None, errors=None, newline=None):
    assert isinstance(filename, (str, int))
    assert isinstance(mode, str)
    assert buffering is None or isinstance(buffering, int)
    assert encoding is None or isinstance(encoding, str)
    assert newline in (None, "", "\n", "\r", "\r\n")
    modes = set(mode)
    if modes - set("arwb+t") or len(mode) > len(modes):
        raise ValueError("invalid mode: %r" % mode)
    reading = "r" in modes
    writing = "w" in modes
    binary = "b" in modes
    appending = "a" in modes
    updating = "+" in modes
    text = "t" in modes or not binary
    if text and binary:
        raise ValueError("can't have text and binary mode at once")
    if reading + writing + appending > 1:
        raise ValueError("can't have read/write/append mode at once")
    if not (reading or writing or appending):
        raise ValueError("must have exactly one of read/write/append mode")
    if binary and encoding is not None:
        raise ValueError("binary modes doesn't take an encoding arg")
    if binary and errors is not None:
        raise ValueError("binary modes doesn't take an errors arg")
    if binary and newline is not None:
        raise ValueError("binary modes doesn't take a newline arg")
    # XXX Need to spec the signature for FileIO()
    raw = FileIO(filename, mode)
    line_buffering = (buffering == 1 or buffering is None and raw.isatty())
    if line_buffering or buffering is None:
        buffering = 8*1024  # International standard buffer size
        # XXX Try setting it to fstat().st_blksize
    if buffering < 0:
        raise ValueError("invalid buffering size")
    if buffering == 0:
        if binary:
            return raw
        raise ValueError("can't have unbuffered text I/O")
    if updating:
        buffer = BufferedRandom(raw, buffering)
    elif writing or appending:
        buffer = BufferedWriter(raw, buffering)
    else:
        assert reading
        buffer = BufferedReader(raw, buffering)
    if binary:
        return buffer
    assert text
    return TextIOWrapper(buffer, encoding, errors, newline, line_buffering)
```
