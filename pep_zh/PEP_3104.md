# PEP 3104 – 访问外部作用域中的名称

**Metadata:**

- **Author**: Ka-Ping Yee <ping&#32;&#97;t&#32;zesty.ca>
- **Created**: 12-Oct-2006
- **Python-Version**: 3.0

## 摘要

在大多数支持嵌套作用域的语言中，代码可以引用或重新绑定（赋值给）最近的外部作用域中的任何名称。目前，Python 代码可以引用任何外部作用域中的名称，但它只能在两个作用域中重新绑定名称：局部作用域（通过简单赋值）或模块级全局作用域（使用 global 声明）。

这一限制在 Python-Dev 邮件列表和其他地方被多次提出，并引发了广泛的讨论和许多消除这一限制的提议。本 PEP 总结了各种被提议的替代方案，以及每种方案的优缺点。

## 基本原理

在 2.1 版本之前，Python 对作用域的处理类似于标准 C：在一个文件中只有两个作用域级别，全局和局部。在 C 中，这是函数定义不能嵌套这一事实的自然结果。但在 Python 中，尽管函数通常定义在顶层，但函数定义可以在任何地方执行。这使 Python 具有嵌套作用域的语法外观，却没有其语义，并产生了一些令程序员惊讶的不一致之处——例如，在顶层工作的递归函数在移到另一个函数内时将不再工作，因为递归函数自己的名称在其体的作用域中不再可见。这违反了一个函数在不同上下文中应表现一致这一直觉。这里有一个例子：

```python
def enclosing_function():
    def factorial(n):
        if n < 2:
            return 1
        return n * factorial(n - 1)  # 因 NameError 失败
    print factorial(5)
```

Python 2.1 通过使所有外部作用域中绑定的名称可见而更接近静态嵌套作用域（见 PEP 227）。这一变化使上述代码示例按预期工作。然而，由于对名称的任何赋值隐式声明该名称为局部变量，因此不可能重新绑定外部作用域中的名称（除非 global 声明强制该名称为全局变量）。因此，以下代码（旨在显示一个可以通过点击按钮递增和递减的数字）无法按熟悉词法作用域的人预期的那样工作：

```python
def make_scoreboard(frame, score=0):
    label = Label(frame)
    label.pack()
    for i in [-10, -1, 1, 10]:
        def increment(step=i):
            score = score + step  # 因 UnboundLocalError 失败
            label['text'] = score
        button = Button(frame, text='%+d' % i, command=increment)
        button.pack()
    return label
```

Python 语法没有提供方法来指示 increment 中提到的名称 score 指的是 make_scoreboard 中绑定的变量 score，而不是 increment 中的局部变量。

Python 的用户和开发者表示有兴趣消除这一限制，以便 Python 可以拥有 Algol 风格作用域模型的完整灵活性，该模型现在是许多编程语言的标准，包括 JavaScript、Perl、Ruby、Scheme、Smalltalk、带 GNU 扩展的 C 和 C# 2.0。

有人认为这样的功能不是必需的，因为可以通过将外部变量包装在可变对象中来模拟可重新绑定的外部变量：

```python
class Namespace:
    pass

def make_scoreboard(frame, score=0):
    ns = Namespace()
    ns.score = 0
    label = Label(frame)
    label.pack()
    for i in [-10, -1, 1, 10]:
        def increment(step=i):
            ns.score = ns.score + step
            label['text'] = ns.score
        button = Button(frame, text='%+d' % i, command=increment)
        button.pack()
    return label
```

然而，这种变通方法只是突出了现有作用域的不足之处：函数的目的是将代码封装在其自己的命名空间中，因此程序员不得不创建额外的命名空间来弥补现有局部作用域中缺失的功能，然后必须决定每个名称应该驻留在真实作用域还是模拟作用域中，这似乎很遗憾。

另一个常见的反对意见是，所需的功能可以写成类，尽管有些冗长。对这一反对意见的一个反驳是，存在不同的实现风格并不是让受支持的编程结构（嵌套作用域）功能不完整的原因。Python 有时被称为"多范式语言"，因为它从对多种编程范式的支持和优雅整合中获得了如此多的力量、实用灵活性和教学能力。

作用域语法的提议早在 1994 年就出现在 Python-Dev 上 [1]，远在 PEP 227 对嵌套作用域的支持被采用之前。当时，Guido 的回应是：

> 这非常接近引入 CSNS [经典静态嵌套作用域]。*如果*你要这样做，你提议的作用域语义似乎还可以。我仍然认为对 CSNS 的需求不足以证明这种结构的合理性……

在 PEP 227 之后，"外部名称重新绑定讨论"在 Python-Dev 上重新出现了足够多次，以至于它已经成为一个熟悉的事件，自 2003 年以来至少以来就以其现有形式出现 [2]。尽管这些讨论中提出的语言变化尚未被采用，但 Guido 已经承认值得考虑语言变化 [12]。

## 其他语言

为了提供一些背景，本节描述了一些其他语言如何处理嵌套作用域和重新绑定。

### JavaScript、Perl、Scheme、Smalltalk、GNU C、C# 2.0

这些语言使用变量声明来指示作用域。在 JavaScript 中，词法作用域的变量使用 var 关键字声明；未声明的变量名称被假定为全局变量。在 Perl 中，词法作用域的变量使用 my 关键字声明；未声明的变量名称被假定为全局变量。在 Scheme 中，所有变量必须声明（使用 define 或 let，或作为形式参数）。在 Smalltalk 中，任何块都可以通过在垂直线之间声明局部变量名称列表来开始。C 和 C# 要求所有变量都有类型声明。对于所有这些情况，变量属于包含声明的作用域。

### Ruby（截至 1.8 版本）

Ruby 是一个有启发性的例子，因为它似乎是唯一其他目前流行的语言，像 Python 一样，试图在不要求变量声明的情况下支持静态嵌套作用域，因此不得不想出一个不同寻常的解决方案。Ruby 中的函数可以包含其他函数定义，它们也可以包含用花括号括起来的代码块。块可以访问外部变量，但嵌套函数不能。在块内，对名称的赋值只有在它不会遮蔽已经在外部作用域中绑定的名称时才意味着局部变量的声明；否则，赋值被解释为外部名称的重新绑定。Ruby 的作用域语法和规则也被广泛辩论，Ruby 2.0 中似乎可能会发生变化 [28]。

## 提议概览

Python-Dev 上已经有许多不同的提议来重新绑定外部作用域中的名称。它们都属于两类：在名称被绑定的作用域中的新语法，或在名称被使用的作用域中的新语法。

### 在绑定（外部）作用域中的新语法

#### 作用域覆盖声明

这一类别中的提议都建议一种类似于 JavaScript 的 var 的新类型声明语句。为此目的已经提议了一些可能的关键字：

- scope x [4]
- var x [4] [9]
- my x [13]

在所有这些提议中，特定作用域 S 中的 var x 这样的声明将导致嵌套在 S 内的作用域中对 x 的所有引用都指的是在 S 中绑定的 x。

这一类提议的主要反对意见是函数定义的含义将变得与上下文相关。将函数定义移到某个其他块内可能会由于封闭块中的声明而导致函数中的任何局部名称引用变成非局部的。对于 Ruby 1.8 中的块，事实确实如此；在以下示例中，两个 setter 有不同的效果，尽管它们看起来相同：

```ruby
setter1 = proc { | x | y = x }  # y 在这里是局部的
y = 13
setter2 = proc { | x | y = x }  # y 在这里是非局部的
setter1.call(99)
puts y  # 打印 13
setter2.call(77)
puts y  # 打印 77
```

注意，尽管这个提议类似于 JavaScript 和 Perl 中的声明，但对语言的影响是不同的，因为在那些语言中未声明的变量默认是全局的，而在 Python 中未声明的变量默认是局部的。因此，在 JavaScript 或 Perl 中将函数移到某个其他块内只能减少先前全局名称引用的作用域，而在 Python 中带有这个提议，它可以扩展先前局部名称引用的作用域。

#### 必需的变量声明

一个更激进的提议 [21] 建议完全删除 Python 的作用域猜测约定，并要求所有名称必须在它们要被绑定的作用域中声明，就像 Scheme 一样。通过这个提议，var x = 3 将同时声明 x 属于局部作用域并将其绑定，而 x = 3 将重新绑定现有的可见 x。在没有包含 var x 声明的外部作用域的上下文中，语句 x = 3 将被静态确定为非法。

这个提议产生了一个简单一致的模型，但它将与所有现有的 Python 代码不兼容。

### 在引用（内部）作用域中的新语法

这一类别中有三种类型的提议。

#### 外部引用表达式

这种类型的提议建议一种在表达式中使用变量时引用外部作用域中变量的新方法。为此建议的一种语法是.x [7]，它将引用 x 而不会为它创建局部绑定。对这个提议的一个担忧是，在许多上下文中 x 和.x 可以互换使用，这会使读者感到困惑 [31]。一个密切相关的想法是使用多个点来指定要上升的作用域级别数 [8]，但大多数人认为这太容易出错 [17]。

#### 重新绑定运算符

这个提议建议一个新的类似赋值的运算符，它重新绑定名称而不会将该名称声明为局部变量 [2]。虽然语句 x = 3 既将 x 声明为局部变量又将其绑定到 3，但语句 x := 3 将更改 x 的现有绑定而不会将其声明为局部变量。

这是一个简单的解决方案，但根据 PEP 3099，它已被拒绝（可能是因为它太容易被忽略或与=混淆）。

#### 作用域覆盖声明

这一类别中的提议建议在内部作用域中声明一种新类型的语句，防止名称变成局部的。这个语句在性质上类似于 global 语句，但它不会使名称引用顶层模块级作用域中的绑定，而是使名称引用最近的外部作用域中的绑定。

这种方法因其与熟悉的 Python 构造的平行关系而具有吸引力，并且因为它保留了函数定义的上下文独立性。

从安全性和调试的角度来看，这种方法也有优势。结果 Python 不仅将匹配其他嵌套作用域语言的功能，而且将以一种可以说对防御性编程甚至更好的语法来实现。在大多数其他语言中，声明收缩了现有名称的作用域，因此不小心省略声明可能会产生比预期更远（即更危险）的效果。在带有这个提议的 Python 中，添加声明的额外工作与局部效果增加的風險相一致（即阻力最小的路径是更安全的路径）。

已经为这样的声明建议了许多拼写方式：

- scoped x [1]
- global x in f [3]（明确指定哪个作用域）
- free x [5]
- outer x [6]
- use x [9]
- global x [10]（更改 global 的含义）
- nonlocal x [11]
- global x outer [18]
- global in x [18]
- not global x [18]
- extern x [20]
- ref x [22]
- refer x [22]
- share x [22]
- sharing x [22]
- common x [22]
- using x [22]
- borrow x [22]
- reuse x [23]
- scope f x [25]（明确指定哪个作用域）

讨论最多的选择似乎是 outer、global 和 nonlocal。outer 已经在标准库中用作变量名和属性名。单词 global 有冲突的含义，因为"全局变量"通常被理解为具有顶层作用域的变量 [27]。在 C 中，关键字 extern 意味着名称引用不同编译单元中的变量。虽然 nonlocal 有点长，不如其他一些选项听起来愉快，但它确实具有完全正确的含义：它声明一个名称不是局部的。

## 提议的解决方案

本 PEP 提议的解决方案是在引用（内部）作用域中添加作用域覆盖声明。Guido 已经在 Python-Dev 上表达了对此类解决方案的偏好 [14]，并显示了对 nonlocal 作为关键字的赞同 [19]。

提议的声明：

```python
nonlocal x
```

防止 x 成为当前作用域中的局部名称。当前作用域中 x 的所有出现都将引用外部封闭作用域中绑定的 x。与 global 一样，允许多个名称：

```python
nonlocal x, y, z
```

如果外部作用域中没有预先存在的绑定，编译器将引发 SyntaxError。（将这种错误称为语法错误可能有点牵强，但到目前为止，SyntaxError 用于所有编译时错误，包括，例如，带有未知功能名称的__future__导入。）Guido 曾说，在没有外部绑定的情况下，这种声明应该被视为错误 [16]。

如果 nonlocal 声明与局部作用域中形式参数的名称冲突，编译器将引发 SyntaxError。

也允许简写形式，其中 nonlocal 附加到赋值或增强赋值之前：

```python
nonlocal x = 3
```

上述与 nonlocal x; x = 3 完全相同。（Guido 支持类似形式的 global 语句 [24]。）

在简写形式的左侧，只允许标识符，不允许像 x[0] 这样的目标表达式。否则，允许所有形式的赋值。nonlocal 语句的提议语法是：

```python
nonlocal_stmt ::=
    "nonlocal" identifier ("," identifier)*
    ["=" (target_list "=")+ expression_list]
    | "nonlocal" identifier augop expression_list
```

允许所有这些形式的赋值的基本原理是它简化了对 nonlocal 语句的理解。将简写形式分离为声明和赋值足以理解它的含义以及它是否有效。

> **注意**
>
> 简写语法没有在原 PEP 的实现中添加。后来的讨论 [29] [30] 得出结论，不应该实现这种语法。

## 向后兼容性

本 PEP 针对 Python 3000，正如 Guido 所建议的 [19]。然而，其他人注意到本 PEP 中考虑的一些选项可能足够小，可以在 Python 2.x 中实现 [26]，在这种情况下，本 PEP 可能可以移动成为 2.x 系列 PEP。
