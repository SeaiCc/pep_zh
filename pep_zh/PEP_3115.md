# PEP 3115 – Python 3000 中的元类

**Metadata:**

- **Author**: Talin <viridia&#32;&#97;t&#32;gmail.com>
- **Created**: 07-Mar-2007
- **Python-Version**: 3.0
- **Post-History**: 11-Mar-2007, 14-Mar-2007

## 摘要

本 PEP 提议更改声明元类的语法，并改变具有元类的类如何构建的语义。

## 理由

本 PEP 有两个理由，都相当微妙。

更改元类工作方式的主要原因是，有一些有趣的用例需要元类比当前可能的更早地参与到类构建过程中。目前，元类机制本质上是一个后处理步骤。随着类装饰器的出现，许多这些后处理工作可以由装饰器机制接管。

特别是，有一类重要的用例，保留类成员声明的顺序将非常有用。普通的 Python 对象将成员存储在字典中，在字典中排序并不重要，成员严格通过名称访问。然而，Python 经常被用来与外部系统接口，在这些系统中成员根据隐式排序组织。示例包括 C 结构体的声明；COM 对象；Python 类到 IDL 或数据库架构的自动转换（如 ORM 中使用）；等等。

在这种情况下，Python 程序员希望能够直接使用类成员的声明顺序来指定这种排序。目前，这种排序必须使用某种其他机制显式指定（参见 ctypes 模块示例。）

不幸的是，当前声明元类的方法不允许这样做，因为到元类发挥作用时，排序信息已经丢失。通过允许元类更早地参与到类构建过程中，新系统允许保留和检查排序或其他早期构建工件。

所提出的元类机制还支持许多其他有趣的用例，除了保留声明顺序外。一个用例是向类体的命名空间插入符号，这些符号仅在类构建期间有效。一个例子可能是"字段构造函数"，小函数用于创建类成员。另一个有趣的可能是支持前向引用，即在类体中更下面声明的 Python 符号的引用。

另一个较弱的理由纯粹是外观上的：当前指定元类的方法是赋值给特殊变量__metaclass__，这被一些人认为在美学上不太理想。其他人强烈不同意这种观点。本 PEP 将不会解决这个问题，只是注意到它，因为美学争论不能通过逻辑证明解决。

## 规范

在新模型中，指定元类的语法是通过基类列表中的关键字参数：

```python
class Foo(base1, base2, metaclass=mymeta):
 ...
```

这里也允许额外的关键字，并将传递给元类，如下面的示例：

```python
class Foo(base1, base2, metaclass=mymeta, private=True):
 ...
```

注意，本 PEP 不尝试定义这些其他关键字可能是什么——这由元类实现者决定。

更一般地说，传递给类定义的参数列表现在将支持函数调用的所有功能，意味着你现在可以在类基列表中使用*args 和**kwargs 样式的参数：

```python
class Foo(*bases, **kwds):
 ...
```

## 调用元类

在当前的元类系统中，元类对象可以是任何可调用类型。这没有改变，但是为了充分利用所有新功能，元类将需要一个额外的属性，该属性在类预构建期间使用。

此属性名为__prepare__，作为函数在评估类体之前被调用。__prepare__函数接受两个位置参数和任意数量的关键字参数。两个位置参数是：

| 参数 | 描述 |
| :--- | :--- |
| name | 正在创建的类的名称。 |
| bases | 基类列表。 |


解释器总是在调用之前测试__prepare__的存在；如果不存在，则使用常规字典，如下面的 Python 代码片段所示。

```python
def prepare_class(name, *bases, metaclass=None, **kwargs):
    if metaclass is None:
        metaclass = compute_default_metaclass(bases)
    prepare = getattr(metaclass, '__prepare__', None)
    if prepare is not None:
        return prepare(name, bases, **kwargs)
    else:
        return dict()
```

上面的示例说明了'class'的参数如何被解释。类名是第一个参数，后跟任意长度的基类列表。在基类之后，可能有一个或多个关键字参数，其中一个是 metaclass。注意，metaclass 参数不包含在 kwargs 中，因为它被正常的参数分配算法过滤掉了。（还要注意，metaclass 是按 PEP 3102 的关键字仅参数。）

即使__prepare__不是必需的，默认元类（'type'）也实现了它，为了方便子类通过 super() 调用它。

__prepare__返回一个字典样式的对象，用于在评估类体期间存储类成员定义。换句话说，类体被评估为函数块（就像现在一样），除了局部变量字典被__prepare__返回的字典替换。这个字典对象可以是常规字典或自定义映射类型。

这个字典样式的对象不需要支持完整的字典接口。支持有限集合字典操作的字典将限制在评估类体期间可以发生哪些类型的操作。最小实现可能只支持从字典添加和检索值——大多数类体在评估期间不会做更多。对于某些类，可能希望也支持删除。许多元类之后需要复制这个字典，因此迭代或其他读取字典内容的方法也可能有用。

__prepare__方法最常实现为类方法而不是实例方法，因为它在被调用时元类实例（即类本身）尚未创建。

一旦类体完成评估，元类将被调用（作为可调用对象），带有类字典，这与当前元类机制没有不同。

通常，元类将创建一个自定义字典——要么是 dict 的子类，要么是它的包装器——它将包含在评估类体之前或期间设置的附加属性。然后在第二阶段，元类可以使用这些附加属性进一步自定义类。

一个示例是一个使用成员声明排序信息创建 C 结构体的元类。元类将提供一个自定义字典，它简单地保留插入顺序的记录。这不需要是一个完整的'ordered dict'实现，而是一个（key,value）对的 Python 列表，每次插入时追加。

注意，在这种情况下，元类将需要处理重复键的可能性，但在大多数情况下那是微不足道的。元类可以使用第一个声明、最后一个、以某种方式组合它们，或者简单地抛出异常。元类决定如何处理这种情况。

## 示例

这是一个简单的元类示例，它创建一个列表中所有类成员的名称，按照它们被声明的顺序：

```python
# 自定义字典
class member_table(dict):
    def __init__(self):
        self.member_names = []

    def __setitem__(self, key, value):
        # 如果键尚未定义，添加到
        # 键列表。
        if key not in self:
            self.member_names.append(key)

        # 调用父类
        dict.__setitem__(self, key, value)

# 元类
class OrderedClass(type):

    # prepare 函数
    @classmethod
    def __prepare__(metacls, name, bases):  # 此情况下无关键字
        return member_table()

    # 元类调用
    def __new__(cls, name, bases, classdict):
        # 注意我们在传递给父类之前将
        # classdict 替换为常规
        # dict，这样我们在类
        # 创建后不再继续记录成员名称。
        result = type.__new__(cls, name, bases, dict(classdict))
        result.member_names = classdict.member_names
        return result

class MyClass(metaclass=OrderedClass):
    # method1 进入数组元素 0
    def method1(self):
        pass

    # method2 进入数组元素 1
    def method2(self):
        pass
```

## 替代提案

Josiah Carlson 提议使用名称'type'代替
'metaclass'，理论上真正指定的是类型的类型。虽然这在技术上是正确的，但从创建新类的程序员角度来看，这也是令人困惑的。从应用程序员的角度来看，他们感兴趣的'type'是他们正在编写的类；那个类型的类型是元类。

讨论中对'两阶段'创建过程有一些反对意见，元类被调用两次，一次创建类字典，一次'完成'类。一些人觉得这两个阶段应该完全分开，应该有不同的语法来指定自定义字典和指定元类。然而，在大多数情况下，这两个阶段将紧密相连，元类最有可能对类字典的内部细节有深入了解。要求程序员确保正确的字典类型和正确的元类类型一起使用，给程序员带来了额外和不需要的负担。

另一个好的建议是简单地对所有类使用有序字典，跳过整个'自定义字典'机制。这是基于大多数自定义字典的用例是为了保留排序信息的观察。然而，这个想法有几个缺点，首先因为它意味着有序字典实现必须添加到 Python 内置类型集合中，其次因为它将对所有类声明施加轻微的速度（和复杂性）惩罚。后来，几个人提出了除保留字段排序之外的自定义字典的用例想法，所以这个想法被放弃了。

## 向后兼容性

有可能保留现有的__metaclass__语法。或者，修改 Py3K 转换工具的语法规则从旧语法转换为新语法也不是太难。
