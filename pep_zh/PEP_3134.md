# PEP 3134 – 异常链式处理和嵌入式回溯

**Metadata:**

- **Author**: Ka-Ping Yee
- **Created**: 12-May-2005
- **Python-Version**: 3.0

## 编号说明

本 PEP 最初是 PEP 344。由于它现在针对 Python 3000，它已被移至 3xxx 空间。

## 摘要

本 PEP 为异常实例提出了三个标准属性：用于隐式链式异常的__context__属性，用于显式链式异常的__cause__属性，以及用于回溯的__traceback__属性。新的 raise ... from 语句设置__cause__属性。

## 动机

在处理一个异常（异常 A）期间，可能会发生另一个异常（异常 B）。在当今的 Python（版本 2.4）中，如果发生这种情况，异常 B 向外传播，异常 A 丢失。为了调试问题，了解两个异常很有用。__context__属性自动保留这些信息。

有时，异常处理程序故意重新抛出异常可能很有用，无论是提供额外信息还是将异常转换为另一种类型。__cause__属性提供了一种显式记录异常直接原因的方法。

在当今的 Python 实现中，异常由三部分组成：类型、值和回溯。sys 模块通过三个并行变量 exc_type、exc_value 和 exc_traceback 暴露当前异常，sys.exc_info() 函数返回这三部分的元组，raise 语句有一个接受这三部分的三参数形式。操作异常通常需要并行传递这三件事，这可能繁琐且容易出错。此外，except 语句只能提供对值的访问，而不能提供对回溯的访问。将__traceback__属性添加到异常值使所有异常信息都可以从一个地方访问。

## 历史

Raymond Hettinger [1] 在 2003 年 1 月在 Python-Dev 上提出了被掩盖异常的问题，并提出了一个 C 模块可以使用来用更多信息增强当前活动异常的 PyErr_FormatAppend() 函数。Brett Cannon [2] 在 2003 年 6 月再次提出链式异常，引发了漫长的讨论。

Greg Ewing [3] 确定了在由原始异常触发的展开期间在 finally 块中发生异常的情况，与在处理原始异常的 except 块中发生异常的情况不同。

Greg Ewing [4] 和 Guido van Rossum [5]，可能还有其他人，之前曾提到向 Exception 实例添加回溯属性。这在 PEP 3000 中有所说明。

本 PEP 的动机是最近 Python-Dev 重新发布的相同想法 [6] [7]。

## 理由

Python-Dev 讨论显示人们对两种完全不同目的的异常链式处理感兴趣。为了处理意外抛出次要异常，必须隐式保留异常。为了支持异常的有意转换，必须有一种显式链式异常的方法。本 PEP 解决了这两个问题。

在 Python-Dev [2] 上建议了几个用于链式异常的属性名，包括 cause、antecedent、reason、original、chain、chainedexc、exc_chain、excprev、previous 和 precursor。对于显式链式异常，本 PEP 建议__cause__，因为它有特定含义。对于隐式链式异常，本 PEP 提出名称__context__，因为预期含义比时间顺序更具体，但不如因果关系具体：异常在处理另一个异常的上下文中发生。

本 PEP 为这三个属性建议带有前导和尾随双下划线的名称，因为它们由 Python VM 设置。只有在非常特殊的情况下，它们才应由正常赋值设置。

本 PEP 以相同方式处理在 except 块和 finally 块期间发生的异常。读取回溯可以清楚地显示异常发生的位置，因此区分这两种情况的额外机制只会增加不必要的复杂性。

本 PEP 提出最外层的异常对象（通过 except 子句暴露以进行匹配的那个）应该是最近抛出的异常，以与当前行为兼容。

本 PEP 提出回溯最后显示最外层的异常，因为这与回溯的时间顺序（从最旧到最新的帧）一致，并且实际抛出的异常更容易在最后一行找到。

为了简化事情，用于设置异常的 C API 调用不会自动设置异常的__context__。Guido van Rossum 已表达了对进行此类更改的担忧 [8]。

至于其他语言，Java 和 Ruby 在 catch/rescue 或 finally/ensure 子句中出现另一个异常时都会丢弃原始异常。Perl 5 缺乏内置的结构化异常处理。对于 Perl 6，RFC 编号 88 [9] 提出了一个异常机制，该机制在名为@@的数组中隐式保留链式异常。在该 RFC 中，最近抛出的异常被暴露以进行匹配，如本 PEP 所示；此外，任意表达式（可能涉及@@）可以求值以进行异常匹配。

C# 中的异常包含一个只读 InnerException 属性，可能指向另一个异常。其文档 [10] 说"当异常 X 作为先前异常 Y 的直接结果被抛出时，X 的 InnerException 属性应包含对 Y 的引用。"此属性不是由 VM 自动设置的；相反，所有异常构造函数都接受一个可选的 innerException 参数来显式设置它。__cause__属性与 InnerException 履行相同的用途，但本 PEP 提出了一种新的 raise 形式，而不是扩展所有异常的构造函数。C# 还提供了 GetBaseException 方法，可以直接跳到 InnerException 链的末尾；本 PEP 没有提出类似方法。

所有这三个属性都在一个提案中一起提出的原因是__traceback__属性提供了对链式异常回溯的方便访问。

## 隐式异常链式处理

下面是一个示例来说明__context__属性：

```python
def compute(a, b):
    try:
        a/b
    except Exception, exc:
        log(exc)

def log(exc):
    file = open('logfile.txt')  # oops, forgot the 'w'
    print >>file, exc
    file.close()
```

调用 compute(0, 0) 会导致 ZeroDivisionError。compute() 函数捕获此异常并调用 log(exc)，但 log() 函数在尝试写入未打开以进行写入的文件时也会抛出异常。

在当今的 Python 中，compute() 的调用者会得到 IOError。ZeroDivisionError 丢失了。通过提议的更改，IOError 的实例有一个额外的__context__属性，保留 ZeroDivisionError。

下面这个更复杂的示例演示了 finally 和 except 子句混合的处理：

```python
def main(filename):
    file = open(filename)  # oops, forgot the 'w'
    try:
        try:
            compute()
        except Exception, exc:
            log(file, exc)
        finally:
            file.clos()  # oops, misspelled 'close'

def compute():
    1/0

def log(file, exc):
    try:
        print >>file, exc  # oops, file is not writable
    except:
        display(exc)

def display(exc):
    print ex  # oops, misspelled 'exc'
```

使用现有文件的名称调用 main() 将触发四个异常。最终结果将是由于 clos 拼写错误导致的 AttributeError，其__context__指向由于 ex 拼写错误导致的 NameError，其__context__指向由于文件只读导致的 IOError，其__context__指向 ZeroDivisionError，其__context__属性为 None。

提议的语义如下：

- 每个线程都有一个异常上下文，初始设置为 None。

- 每当抛出异常时，如果异常实例还没有__context__属性，解释器将其设置为线程的异常上下文。

- 在异常抛出后立即，线程的异常上下文设置为该异常。

- 每当解释器通过到达末尾或执行 return、yield、continue 或 break 语句退出 except 块时，线程的异常上下文设置为 None。

## 显式异常链式处理

异常对象上的__cause__属性始终初始化为 None。它由 raise 语句的新形式设置：

```python
raise EXCEPTION from CAUSE
```

等效于：

```python
exc = EXCEPTION
exc.__cause__ = CAUSE
raise exc
```

在下面的示例中，数据库为几种不同的存储类型提供实现，文件存储是其中之一。数据库设计者希望错误作为 DatabaseError 对象传播，这样客户端就不必了解存储特定的细节，但不想丢失底层错误信息。

```python
class DatabaseError(Exception):
    pass

class FileDatabase(Database):
    def __init__(self, filename):
        try:
            self.file = open(filename)
        except IOError, exc:
            raise DatabaseError('failed to open') from exc
```

如果对 open() 的调用抛出异常，问题将报告为 DatabaseError，__cause__属性揭示了 IOError 作为原始原因。

## 回溯属性

下面的示例说明了__traceback__属性。

```python
def do_logged(file, work):
    try:
        work()
    except Exception, exc:
        write_exception(file, exc)
        raise exc

from traceback import format_tb

def write_exception(file, exc):
    ...
    type = exc.__class__
    message = str(exc)
    lines = format_tb(exc.__traceback__)
    file.write(... type ... message ... lines ...)
    ...
```

在当今的 Python 中，do_logged() 函数必须从 sys.exc_traceback 或 sys.exc_info() [2] 提取回溯，并将值和回溯都传递给 write_exception()。通过提议的更改，write_exception() 只需一个参数，并使用__traceback__属性获取异常。

提议的语义如下：

- 每当捕获异常时，如果异常实例还没有__traceback__属性，解释器将其设置为新捕获的回溯。

## 增强报告

默认异常处理程序将修改为报告链式异常。通过遵循__cause__和__context__属性来遍历异常链，__cause__优先。遵循回溯的时间顺序，最近抛出的异常最后显示；即，显示从最内层异常的描述开始，然后回溯链到最外层异常。回溯按通常方式格式化，其中一行：

```
The above exception was the direct cause of the following exception:
```

或

```
During handling of the above exception, another exception occurred:
```

在回溯之间，取决于它们是否分别通过__cause__或__context__链接。以下是过程的草图：

```python
def print_chain(exc):
    if exc.__cause__:
        print_chain(exc.__cause__)
        print '\nThe above exception was the direct cause...'
    elif exc.__context__:
        print_chain(exc.__context__)
        print '\nDuring handling of the above exception, ...'
    print_exc(exc)
```

在 traceback 模块中，format_exception、print_exception、print_exc 和 print_last 函数将更新以接受一个可选的 chain 参数，默认为 True。当此参数为 True 时，这些函数将格式化或显示整个异常链，如上所述。当它为 False 时，这些函数将只格式化或显示最外层的异常。

cgitb 模块也应更新以显示整个异常链。

## C API

用于设置异常的 `PyErr_Set*`调用不会设置异常上的`__context__`属性。PyErr_NormalizeException 将始终将回溯属性设置为其 tb 参数，并将`__context__`和`__cause__`属性设置为 None。

新的 API 函数 PyErr_SetContext(context) 将帮助 C 程序员提供链式异常信息。此函数首先规范化当前异常使其成为实例，然后设置其`__context__`属性。类似的 API 函数 PyErr_SetCause(cause) 将设置`__cause__`属性。

## 兼容性

链式异常暴露最近异常的类型，因此它们仍然匹配与现在相同的 except 子句。

除非代码在异常实例上设置或使用名为__context__、__cause__或__traceback__的属性，否则提议的更改不应破坏任何代码。截至 2005-05-12，Python 标准库不包含此类属性的提及。

## 开放问题：额外信息

Walter Dörwald [11] 表达了在异常向上传播期间附加额外信息而不更改其类型的愿望。这可能是一个有用的功能，但本 PEP 未解决。它可以通过单独的 PEP 来解决，该 PEP 建立异常上其他信息属性的约定。

## 开放问题：抑制上下文

如所写，本 PEP 使得抑制__context__变得不可能，因为在 except 或 finally 子句中将 exc.__context__设置为 None 只会导致它在抛出 exc 时再次设置。

## 开放问题：限制异常类型

为了改进封装，库实现者可能希望用应用程序级异常包装所有实现级异常。人们可以尝试通过编写以下代码来包装异常：

```python
try:
    ... implementation may raise an exception ...
except:
    import sys
    raise ApplicationError from sys.exc_value
```

或这个：

```python
try:
    ... implementation may raise an exception ...
except Exception, exc:
    raise ApplicationError from exc
```

但两者都有些缺陷。能够在全捕获 except 子句中命名当前异常会很棒，但这在这里未解决。这样的功能将允许类似以下内容：

```python
try:
    ... implementation may raise an exception ...
except *, exc:
    raise ApplicationError from exc
```

## 开放问题：yield

当执行 yield 语句时，异常上下文丢失；在 yield 之后恢复帧不会恢复上下文。解决此问题超出了本 PEP 的范围；这不是一个新问题，如下面的示例所示：

```python
>>> def gen():
...     try:
...         1/0
...     except:
...         yield 3
...     raise
...
>>> g = gen()
>>> g.next()
3
>>> g.next()
TypeError: exceptions must be classes, instances, or strings
(deprecated), not NoneType
```

## 开放问题：垃圾回收

对此提议的最强烈反对是它在异常和堆栈帧之间创建循环 [12]。循环垃圾的收集（因此资源释放）可能会大大延迟。

```python
>>> try:
>>>     1/0
>>> except Exception, err:
>>>     pass
```

将引入从 err -> traceback -> stack frame -> err 的循环，保持同一作用域中的所有局部变量存活，直到下一次 GC 发生。

今天，这些局部变量将超出作用域。有许多代码假设"局部"资源——特别是打开的文件——将很快关闭。如果闭包必须等待下一次 GC，程序（今天运行良好）可能会用完文件句柄。

将__traceback__属性制作成弱引用将避免循环垃圾的问题。不幸的是，它将使稍后保存 Exception（如 unittest 所做）更加麻烦，并且不允许 sys 模块进行太多清理。

由 Adam Olsen 建议的可能替代解决方案是，当变量超出作用域时，将堆栈帧对 err 变量的引用转换为弱引用 [13]。

## 可能的未来兼容更改

这些更改与异常作为单个对象而不是三元组出现在解释器级别的外观一致。

- 如果接受 PEP 340 或 PEP 343，将__exit__的三个（类型、值、回溯）参数替换为单个异常参数。

- 弃用 sys.exc_type、sys.exc_value、sys.exc_traceback 和 sys.exc_info()，支持单个成员 sys.exception。

- 弃用 sys.last_type、sys.last_value 和 sys.last_traceback，支持单个成员 sys.last_exception。

- 弃用 raise 语句的三参数形式，支持一参数形式。

- 升级 cgitb.html() 以接受单个值作为其第一个参数，作为（类型、值、回溯）元组的替代方案。

## 可能的未来不兼容更改

这些更改可能值得为 Python 3000 考虑。

- 移除 sys.exc_type、sys.exc_value、sys.exc_traceback 和 sys.exc_info()。

- 移除 sys.last_type、sys.last_value 和 sys.last_traceback。

- 用一参数 API 替换三参数 sys.excepthook，并更改 cgitb 模块以匹配。

- 移除 raise 语句的三参数形式。

- 升级 traceback.print_exception 以接受异常参数，而不是类型、值和回溯参数。

## 实现

__traceback__和__cause__属性以及新的 raise 语法在修订版 57783 [14] 中实现。
