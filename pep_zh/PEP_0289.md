# PEP 289 – 生成器表达式

**Metadata:**

- **Author**: Raymond Hettinger <python&#32;&#97;t&#32;rcn.com>
- **Created**: 30-Jan-2002
- **Python-Version**: 2.4
- **Post-History**: 22-Oct-2003

## Abstract

本 PEP 引入了生成器表达式，作为列表推导式 PEP 202 和生成器 PEP 255 的高性能、内存高效的泛化形式。

## 理由

列表推导式的经验表明它们在 Python 中广泛有用。然而，许多使用场景并不需要在内存中创建完整的列表。相反，它们只需要一次迭代一个元素。

例如，以下求和代码将在内存中构建一个完整的平方列表，迭代这些值，然后在引用不再需要时删除该列表：

```python
sum([x*x for x in range(10)])
```

通过使用生成器表达式可以节省内存：

```python
sum(x*x for x in range(10))
```

容器对象的构造函数也获得类似的益处：

```python
s = set(word for line in page for word in line.split())
d = dict( (k, func(k)) for k in keylist)
```

生成器表达式对于将可迭代输入缩减为单个值的函数（如 sum()、min() 和 max()）特别有用：

```python
max(len(line) for line in file if line.strip())
```

生成器表达式也解决了一些使用 lambda 编码的函数式示例：

```python
reduce(lambda s, a: s + a.myattr, data, 0)
reduce(lambda s, a: s + a[3], data, 0)
```

这些简化为：

```python
sum(a.myattr for a in data)
sum(a[3] for a in data)
```

列表推导式大大减少了对 filter() 和 map() 的需求。同样，生成器表达式预计将最小化对 itertools.ifilter() 和 itertools.imap() 的需求。相比之下，其他 itertools 的实用性将通过生成器表达式得到增强：

```python
dotproduct = sum(x*y for x,y in itertools.izip(x_vector, y_vector))
```

具有与列表推导式相似的语法也使得在应用程序扩展时轻松将现有代码转换为生成器表达式成为可能。

早期计时显示生成器在性能上比列表推导式有显著优势。然而，后者在 Py2.4 中得到了高度优化，现在对于中小数据集性能大致相当。随着数据量增长，生成器表达式往往表现更好，因为它们不会耗尽缓存内存，并允许 Python 在迭代之间重用对象。

## BDFL Pronouncements

本 PEP 被接受用于 Py2.4。

## The Details

（对于来自火星的读者来说，这些都不够精确，但我希望示例能够充分传达意图，以便在 c.l.py 中进行讨论。Python 参考手册应包含 100% 精确的语义和语法规范。）

- 生成器表达式的语义等同于创建匿名生成器函数并调用它。例如：

```python
g = (x**2 for x in range(10))
print g.next()
```

等价于：

```python
def __gen(exp):
 for x in exp:
  yield x**2
g = __gen(iter(range(10)))
print g.next()
```

只有最外层的 for 表达式被立即求值，其他表达式延迟到生成器运行时：

```python
g = (tgtexp for var1 in exp1 if exp2 for var2 in exp3 if exp4)
```

等价于：

```python
def __gen(bound_exp):
 for var1 in bound_exp:
  if exp2:
   for var2 in exp3:
    if exp4:
     yield tgtexp
g = __gen(iter(exp1))
del __gen
```

语法要求生成器表达式必须始终直接位于一组括号内，并且两侧不能有逗号。参考 CVS 中的 Grammar/Grammar 文件，两条规则发生变化：

规则：
`atom: '(' [testlist] ')'`

变为：

`atom: '(' [testlist_gexp] ')'`

其中 testlist_gexp 与 listmaker 几乎相同，但只允许'for'...'in'之后有一个 test：

```python
testlist_gexp: test ( gen_for | (',' test)* [','] )
```

arglist 的规则需要类似的变化。

这意味着你可以写：

```python
sum(x**2 for x in range(10))
```

但你需要写：

```python
reduce(operator.add, (x**2 for x in range(10)))
```

还有：

```python
g = (x**2 for x in range(10))
```

也就是说，如果函数调用有一个位置参数，它可以是一个没有额外括号的生成器表达式，但在所有其他情况下你必须用括号括起来它。

精确细节已检查到 Grammar/Grammar 版本 1.49。

- 循环变量（如果是简单变量或简单变量元组）不暴露给周围函数。这有助于实现并使典型使用场景更可靠。在 Python 的某个未来版本中，列表推导式也将隐藏归纳变量，不向周围代码暴露（在 Py2.4 中，将针对访问归纳变量的代码发出警告）。例如：

```python
x = "hello"
y = list(x for x in "abc")
print x # 打印"hello"，不是"c"
```

列表推导式将保持不变。例如：

```python
[x for x in S] # 这是一个列表推导式。
[(x for x in S)] # 这是一个包含一个生成器表达式的列表。
```

不幸的是，目前有一个轻微的语法差异。表达式：

```python
[x for x in 1, 2, 3]
```

是合法的，意思是：

```python
[x for x in (1, 2, 3)]
```

但生成器表达式不允许前者版本：

```python
(x for x in 1, 2, 3)
```

是非法的。

前一种列表推导式语法在 Python 3.0 中将变为非法，并且应该在 Python 2.4 及更高版本中弃用。

列表推导式也会将其循环变量“泄漏”到周围作用域。这将在 Python 3.0 中改变，因此 Python 3.0 中列表推导式的语义定义将等价于 list(<生成器表达式>)。如果列表推导式的循环变量与直接周围作用域中使用的变量同名，Python 2.4 及更高版本应发出弃用警告。

## Early Binding versus Late Binding

经过大量讨论，决定第一个（最外层）for 表达式应立即求值，其余表达式应在生成器执行时求值。

当被要求总结绑定第一个表达式的理由时，Guido 提出 [1]：

>考虑 sum(x for x in foo())。现在假设 foo() 中有一个导致异常的 bug，sum() 中也有一个在它开始迭代参数之前引发异常的 bug。你期望看到哪个异常？如果 sum() 中的异常先于 foo() 中的异常被抛出，我会感到惊讶，因为对 foo() 的调用是 sum() 参数的一部分，我期望参数在函数被调用之前被处理。
>另一方面，在 sum(bar(x) for x in foo()) 中，sum() 和 foo() 没有 bug，但 bar() 引发异常，我们别无选择，只能延迟对 bar() 的调用，直到 sum() 开始迭代——这是生成器的合约的一部分。（它们在 next() 方法首次被调用之前什么都不做。）

提出了各种用例来绑定生成器定义时的所有自由变量。一些支持者认为，如果立即绑定，生成的表达式将更容易理解和调试。

然而，Python 对 lambda 表达式采用晚期绑定方法，并且没有自动早期绑定的先例。人们认为引入新范式将不必要地引入复杂性。

在探索了许多可能性之后，出现了一个共识，即绑定问题难以理解，用户应被强烈鼓励在立即消耗其参数的函数内部使用生成器表达式。对于更复杂的应用，完整的生成器定义在关于作用域、生命周期和绑定的明确性方面总是更优越 [2]。

## Reduction Functions

当与 sum()、min() 和 max() 等归约函数结合使用时，生成器表达式的实用性大大增强。Python 2.4 中的 heapq 模块包含两个新的归约函数：nlargest() 和 nsmallest()。两者都与生成器表达式配合良好，并且一次在内存中保持不超过 n 个项。
