# PEP 205 – 弱引用

**元数据：**

## 动机

弱引用有两个基本应用，Python 程序员已经注意到：对象缓存和减少循环引用带来的痛苦。

### 缓存（弱字典）

需要允许维护表示外部状态的物体，将单个实例映射到外部现实，其中允许多个实例映射到同一外部资源会导致在实例之间维护同步的不必要困难。在这些情况下，一种常见的习惯用法是支持实例缓存；工厂函数用于返回新实例或现有实例。

这种方法中的困难是必须容忍两件事之一：要么缓存无限制增长，要么需要在应用程序的其他地方显式管理缓存。后者可能非常乏味，导致比真正解决手头问题所需的更多代码，前者对于长时间运行的进程或甚至具有大量内存需求的相对短进程可能是不可接受的。

- 需要由单个实例表示的外部对象，无论有多少内部用户。这对于表示需要整体写回磁盘而不是每次使用时锁定和修改的文件可能很有用。

- 创建成本高但可能被多个内部消费者需要的对象。类似于第一种情况，但不一定绑定到外部资源，可能不是共享状态的问题。弱引用仅在这种情况下有用，如果存在某种"软"引用或如果单个对象的用户的寿命很可能重叠。

### 循环引用

- DOM 需要大量的循环引用（到父节点和文档节点），但可以使用弱字典从每个节点映射到其父节点来消除这些引用。这在像 xml.dom.pulldom 这样的上下文中可能特别有用，允许.unlink() 操作变为空操作。

## 解决方案空间方面

弱引用问题有两个不同的方面：

- 弱引用失效

- 向 Python 代码呈现弱引用

### 失效

过去弱引用失效的方法通常依赖于存储强引用并能够检查所有弱引用对象实例，当它们的被引用对象的引用计数变为 1 时使其失效（表示弱引用存储的引用是最后一个剩余引用）。这种方法的优点是 Python 中的内存管理机制无需更改，任何类型都可以被弱引用。

这种方法失效的缺点是它假设弱引用的管理被足够频繁地调用，使得弱引用的对象在合理的时间范围内被注意到；由于这意味着扫描某个数据结构以使引用失效，这是一个关于弱引用对象数量的 O(N) 操作，这对于任何单个弱引用对象来说都不是有效地摊销的。这也假设应用程序以某种频率调用处理弱引用对象的代码，这使得弱引用对于库代码不太有吸引力。

另一种失效方法是 de-allocation 代码意识到弱引用的可能性，并在每次对象被 deallocated 时特定地调用弱引用管理代码以进行失效。这需要更改弱可引用对象的 tp_dealloc 处理器；需要为支持弱引用的对象在处理器"顶部"添加额外的调用，并且需要一种有效的方式将对象映射到该对象的弱引用链。

### 呈现

弱引用向 Python 层呈现的两种方式是作为显式引用对象，需要某种操作才能检索到底层对象的可用引用，以及尽可能伪装成原始对象的代理对象。

引用对象在 Python 中添加额外的对象管理层时很容易处理；可以显式检查引用是否存活，而不必在引用体上调用操作并捕获使用无效弱引用时引发的特殊异常。

然而，许多用户喜欢代理方法，仅仅因为弱引用看起来非常像原始对象。

## 提议的解决方案

弱引用应该能够指向任何可能具有大量内存大小（直接或间接）的 Python 对象，或持有对外部资源（数据库连接、打开的文件等）的引用。

新模块 weakref 将包含用于创建弱引用的新函数。weakref.ref() 将创建"弱引用对象"，并可选择附加一个回调，当对象即将被最终化时将调用该回调。weakref.mapping() 将创建"弱字典"。第三个函数 weakref.proxy() 将创建一个代理对象，其行为有些像原始对象。

弱引用对象允许访问被引用的对象（如果它尚未被收集），并确定对象是否仍然存在于内存中。检索被引用对象是通过调用引用对象完成的。如果被引用对象不再存活，这将返回 None。

弱字典将任意键映射到值，但不拥有值的引用。当值被最终化时，它作为值的所有（键，值）对将从包含此类对的所有映射中移除。像字典一样，弱字典不可哈希。

代理对象是弱引用，尽可能表现得像它们代理的对象。无论底层类型如何，代理都不可哈希，因为它们作为弱引用的能力依赖于基本可变性，当用作字典键时会导致失败——即使在被引用对象死亡之前计算了正确的哈希值，结果代理也不能用作字典键，因为它在被引用对象过期后无法进行比较，而可比性对于字典键是必要的。在被引用对象死亡后对代理对象的操作在大多数情况下会导致引发 weakref.ReferenceError。"is"比较、type() 和 id() 将继续工作，但始终引用代理而不是被引用对象。

与弱引用注册的回调必须接受单个参数，该参数将是弱引用或代理对象本身。对象不能在回调中访问或复活。

## 实现策略

弱引用的实现将包括必须为每个弱可引用对象清除的引用容器列表。如果引用来自弱字典，首先清除字典条目。然后，任何关联的回调将以对象作为参数被调用。一旦所有回调都被调用，对象就被最终化并 deallocated。

许多内置类型将参与弱引用管理，任何扩展类型都可以选择这样做。类型结构将包含一个额外的字段，该字段提供到包含弱引用结构列表的实例结构的偏移量。如果字段的值<=0，对象不参与。在这种情况下，`weakref.ref()`、`<weakdict>.__setitem__()` 和`.setdefault()`，以及项赋值将引发 TypeError。如果字段的值>0，可以生成新的弱引用并添加到列表中。

采用这种方法是为了允许任意扩展类型参与，而不必为数字或其他小类型付出内存代价。

支持弱引用的标准类型包括实例、函数以及绑定和未绑定方法。随着 Python 2.2 中类类型（"新式类"）的添加，类型获得了对弱引用的支持。类类型的实例如果具有弱可引用的基类型、类未指定__slots__，或命名了一个__weakref__ 槽，则是弱可引用的。生成器也支持弱引用。

## Java 中的弱引用

http://java.sun.com/j2se/1.3/docs/api/java/lang/ref/package-summary.html

Java 提供三种形式的弱引用和一个有趣的辅助类。这三种形式称为"弱"、"软"和"幻影"引用。相关类在 java.lang.ref 包中定义。

对于每种引用类型，有一个选项在内存分配器使其失效时将引用添加到队列。此设施的主要目的似乎是它允许组合更大的结构以结合弱引用语义，而无需施加实质性的额外锁定要求。例如，使用此设施创建"弱"哈希表不会太困难，当引用不再在其他地方使用时，它会移除键和被引用对象。使用弱引用作为对象而没有某种失效通知队列导致哈希表所需的各种操作的实现更加乏味。如果存储对象的 deallocations 不频繁，这可能是性能瓶颈。

Java 的"弱"引用最像 Dianne Hackborn 的旧 vref 提议：引用对象引用单个 Python 对象，但不拥有该对象的引用。当该对象被 deallocated 时，引用对象被失效。引用对象的用户可以很容易地确定引用已被失效，或者当试图使用被引用的对象时引发 NullObjectDereferenceError。

"软"引用类似，但不会在所有其他对被引用对象的引用被释放后立即失效。"软"引用确实拥有引用，但如果内存需要在其他地方使用，它允许内存分配器释放被引用对象。不清楚这是否意味着软引用在 malloc() 实现调用 sbrk() 或其等效物之前被释放，或者软引用仅在 malloc() 返回 NULL 时清除。

"幻影"引用有点不同；不像弱和软引用，当引用添加到其队列时，被引用对象不会被清除。当对象的所有幻影引用都被出队时，对象被清除。这可以用来保持对象存活，直到执行一些额外的清理，这些清理需要在对象的.finalize() 方法被调用之前发生。

不像其他两种引用类型，"幻影"引用必须与失效队列关联。

### 提议：虚拟引用

试图部分解决关于引用计数与垃圾收集的反复讨论，我想提议一个 Python 扩展，应该有助于创建"结构良好"的循环图。特别是，它应该允许至少创建具有父反向指针的树和双向链表，而无需担心循环。

我想提议的基本机制是"虚拟引用"，或从现在开始"vref"。vref 本质上是一个对象的句柄，不增加对象的引用计数。这意味着持有对象的 vref 不会阻止对象被销毁。这将允许 Python 程序员，例如，创建上述树结构，当它不再被使用时自动销毁——通过将所有父反向引用转换为 vref，它们不再创建阻止树被销毁的引用循环。

为了实现这个机制，Python 核心必须确保没有真实指针永远引用不再存在的对象。我想提议的实现涉及对当前 Python 系统的两个基本添加：

- 新的"vref"类型，Python 程序员通过它创建和操作虚拟引用。内部，它基本上是一个 C 级 Python 对象，带有指向它所引用的 Python 对象的指针。然而，不像所有其他 Python 代码，它不更改此对象的引用计数。此外，它包含两个指针来实现双向链表，在下面使用。

- 添加新字段到基本 Python 对象 [PyObject_Head in object.h]，它是 NULL，或指向引用它的所有 vref 对象列表的头部。当 vref 对象附加到另一个对象时，它将自己添加到这个链表中。然后，如果带有任何 vref 的对象被 deallocated，它可以遍历这个列表并确保其上的所有 vref 指向某个安全值，例如 Nothing。

这个实现应该希望最小化对当前 Python 核心的影响——当没有 vref 存在时，它应该只为所有对象添加一个指针，并且每次对象被 deallocated 时检查 NULL 指针。

回到 Python 语言级别，我已经考虑了 vref 对象的两种可能语义——

### 指针语义

在这个模型中，vref 本质上像一个 Python 级指针；Python 程序必须显式取消引用 vref 来操作它引用的实际对象。

使用此模型的示例 vref 模块可以包括函数"new"；当用作'MyVref = vref.new(MyObject)'时，它返回一个新的 vref 对象，使得 MyVref.object == MyObject。MyVref.object 将在 MyObject 被 deallocated 时变为 Nothing。

对于具体示例，我们可以引入一些新的 C 风格语法：

- & – 一元运算符，在对象上创建 vref，与 vref.new() 相同。

- * – 一元运算符，取消引用 vref，与 VrefObject.object 相同。

然后我们可以定义：

```
1. type(&MyObject) == vref.VrefType
2. *(&MyObject) == MyObject
3. (*(&MyObject)).attr == MyObject.attr
4. &&MyObject == Nothing
5. *MyObject -> 异常
```

规则#4 很微妙，但我们已经对（没有真实引用的 vref）创建了 vref。因此，外部 vref 在内部 vref 不可避免地消失时被清除为 Nothing。

### 代理语义

在这个模型中，Python 程序员操纵 vref 对象就像她在操纵它引用的对象一样。这是通过实现 vref 使得所有对其的操作都被重定向到其引用的对象来完成的。使用此模型，取消引用运算符 (*) 不再有意义；相反，我们只有引用运算符 (&)，并定义：

```
1. type(&MyObject) == type(MyObject)
2. &MyObject == MyObject
3. (&MyObject).attr == MyObject.attr
4. &&MyObject == MyObject
```

同样，规则#4 很重要——这里，外部 vref 实际上是原始对象的引用，而不是内部 vref。这是因为应用于 vref 的所有操作实际上应用于其对象，所以创建 vref 的 vref 实际上导致创建后者的对象的 vref。

第一个，指针语义，优点是它实现起来非常容易；vref 类型极其简单，至少需要一个属性 object 和一个创建引用的函数。

然而，我真的喜欢代理语义。它不仅对 Python 程序员负担较小，而且允许你在任何使用实际对象的地方使用 vref 来做很好的事情。不幸的是，它可能是个极端痛苦，如果不是实际上不可能，在当前 Python 实现中实现。我确实有一些想法，如果它看起来有趣，如何做这个；一种可能性是引入新的类型检查函数来处理 vref。这将希望旧 C 模块不期望 vref 简单地返回类型错误，直到它们可以被修复。

最后，这个系统可以提供一些其他额外能力。一个对我来说特别有趣的涉及允许 Python 程序员添加"析构函数"到 vref——这个 Python 函数将在被引用的对象被 deallocated 之前立即调用，允许 Python 程序不可见地附加到另一个对象并等待它消失。这看起来很酷，虽然我还想不出任何实际用途……:)

– Dianne
