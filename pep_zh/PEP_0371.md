# PEP 371 – 将 multiprocessing 包添加到标准库

**元数据：**

- **作者**: Jesse Noller <jnoller&#32;&#97;t&#32;gmail.com>、Richard Oudkerk <r.m.oudkerk&#32;&#97;t&#32;googlemail.com>
- **创建日期**: 2008 年 5 月 6 日
- **Python 版本**: 2.6、3.0

## 摘要

本 PEP 提议将 pyProcessing [1] 包纳入 Python 标准库，并重命名为"multiprocessing"。

该处理包模仿标准库 threading 模块的功能，提供基于进程的多线程编程方法，使用户能够调度多个任务，从而有效地绕过全局解释器锁。

该包还提供服务器和客户端功能（processing.Manager），以提供对象和任务的远程共享和管理，因此应用程序不仅可以利用本地机器上的多个核心，还可以将对象和任务分布在联网机器的集群上。

虽然该包的分布式功能很有益，但本 PEP 的主要重点是包的核心类线程 API 和功能。

## 理由

当前的 CPython 解释器实现了全局解释器锁（GIL），除非在 Python 3000 或其他正在计划的版本 [2] 中进行工作，否则在可预见的未来，GIL 将在 CPython 解释器中保持不变。虽然 GIL 本身可以为解释器和扩展库提供简洁且易于维护的 C 代码，但对于那些利用多核机器的 Python 程序员来说，它经常成为一个问题。

GIL 本身在任何给定时间点都防止单个线程在解释器中运行，实际上消除了 Python 利用多处理器系统的能力。

pyprocessing 包提供了一种绕过 GIL 的方法，允许 CPython 中的应用程序利用多核架构，而无需用户完全改变其编程范式（即：放弃线程编程而采用另一种"并发"方法——Twisted、Actors 等）。

Processing 包为 CPython 提供了一个"已知 API"，以符合 PEP 8 的方式镜像 threading API，具有已知语义和易于扩展性。

在未来，如果 CPython 解释器启用"真正的"线程，该包可能不再那么相关，但是对于某些应用程序来说，分叉一个操作系统进程有时可能比使用轻量级线程更可取，尤其是在那些进程创建快速且优化的平台上。

例如，一个简单的线程应用程序：

```python
from threading import Thread as worker

def afunc(number):
    print number * 3

t = worker(target=afunc, args=(4,))
t.start()
t.join()
```

pyprocessing 包镜像 API 非常完美，只需简单更改导入为：

```python
from processing import process as worker
```

代码现在将通过 processing.process 类执行。显然，随着 API 重命名为符合 PEP 8，用户应用程序中需要进行额外的重命名，尽管只是微小的。

这种类型的兼容性意味着，通过（在大多数情况下）微小的代码更改，用户的应用程序将能够利用给定机器上的所有核心和处理器进行并行执行。在许多情况下，对于 I/O 绑定程序，pyprocessing 包甚至比正常线程方法更快。当然，这考虑到 pyprocessing 包是优化的 C 代码，而 threading 模块不是。

## "分布式"问题

在 Python-Dev 关于包含此包的讨论 [3] 中，关于本 PEP 的意图存在混淆，试图解决"分布式"问题——经常将此包的功能与其他解决方案进行比较，如基于 MPI 的通信 [4]、CORBA 或其他分布式对象方法 [5]。

"分布式"问题庞大而多样。在此领域工作的每个程序员要么对他们喜欢的模块/方法有非常强烈的意见，要么有一个高度定制的问题，现有解决方案都不适用。

接受此包并不排除或建议解决"分布式"问题的程序员不要为其问题域检查其他解决方案。包含此包的意图是为本地并发提供入门级功能，并提供基本支持将并发扩展到机器网络——尽管两者不是紧密耦合的，pyprocessing 包实际上可以与任何其他解决方案（包括 MPI 等）结合使用。

如果有必要——可以完全将包的本地并发能力与包的联网/共享方面解耦。然而，在没有严重担忧或原因的情况下，本 PEP 的作者不建议这种方法。

## 性能比较

正如我们都知道的——有"谎言、该死的谎言和基准测试"。这些速度比较虽然旨在展示 pyprocessing 包的性能，但绝不全面或适用于所有可能的用例或环境。特别是对于那些进程分叉时间缓慢的平台。

所有基准测试均使用以下内容运行：

- 4 核 Intel Xeon CPU @ 3.00GHz

- 16 GB RAM

- Python 2.5.2 在 Gentoo Linux（内核 2.6.18.6）上编译

- pyProcessing 0.52

所有这些代码可以从 http://jessenoller.com/code/bench-src.tgz 下载

这些基准测试的基本执行方法在 run_benchmarks.py [6] 脚本中，它只是一个包装器，用于通过单线程（线性）、多线程（通过 threading）和多进程（通过 pyprocessing）函数执行目标函数，迭代次数固定，执行循环和/或线程数递增。

run_benchmarks.py 脚本执行每个函数 100 次，通过 timeit 模块选择那 100 次迭代中最佳的运行。

首先，为了识别生成工作进程的开销，我们执行一个仅仅是 pass 语句（空）的函数：

```
cmd: python run_benchmarks.py empty_func.py
Importing empty_func
Starting tests ...
non_threaded (1 iters) 0.000001 seconds
threaded (1 threads) 0.000796 seconds
processes (1 procs) 0.000714 seconds

non_threaded (2 iters) 0.000002 seconds
threaded (2 threads) 0.001963 seconds
processes (2 procs) 0.001466 seconds

non_threaded (4 iters) 0.000002 seconds
threaded (4 threads) 0.003986 seconds
processes (4 procs) 0.002701 seconds

non_threaded (8 iters) 0.000003 seconds
threaded (8 threads) 0.007990 seconds
processes (8 procs) 0.005512 seconds
```

正如你所看到的，通过 pyprocessing 包的进程分叉比构建然后执行线程版本的代码更快。

## API 命名

虽然包的 API 旨在紧密模仿 Python 2.x 的 threading 和 Queue 模块，但这些模块不符合 PEP 8。已经决定不是"原封不动"地添加包从而延续不符合 PEP 8 的命名，我们将重命名所有 API、类等以完全符合 PEP 8。

此更改确实影响了那些使用 threading 模块的人的轻松替换，但在作者的视野中，这是一个可以接受的副作用，特别是考虑到 threading 模块自己的 API 将发生变化。

跟踪器中的问题 3042 提出，对于 Python 2.6，threading 模块将有两个 API——当前的和符合 PEP 8 的。当调用 -3 时将发出关于即将删除原始 java 风格 API 的警告。

在 Python 3000 中，threading API 将变得符合 PEP 8，这意味着 multiprocessing 模块和 threading 模块将再次具有匹配的 API。

## 时间/计划

有人对本 PEP 对于今年 2.6 和 3.0 版本的时间/迟到了表示担忧，然而作者和其他人都认为此包提供的功能超过了包含的风险。

然而，考虑到不希望破坏 Python-core 的稳定性，pyprocessing 代码的一些重构"进入"Python-core 可以推迟到下一个 2.x/3.x 版本。这意味着对 Python-core 的实际风险最小，并且主要局限于包本身。

## 未解决的问题

- 确认没有"默认"远程连接功能，如果需要，为那些提供远程功能的类默认启用远程安全机制。

- 一些 API（Queue 方法 qsize()、task_done() 和 join()）需要添加，或者需要明确识别和记录排除它们的原因。

## 已解决的问题

- roudkerk 在问题 1683 中提交的 PyGILState 错误补丁必须应用才能使包单元测试工作。

- 现有文档必须移动到 ReST 格式。

- 对 ctypes 的依赖：pyprocessing 包对 ctypes 的依赖阻止了包在 ctypes 不受支持的平台上运行。这不是此包的限制，而是 ctypes 的限制。

- 完成：将顶级包从"pyprocessing"重命名为"multiprocessing"。

- 完成：还要注意进程生成的默认行为使其无法与 IDLE 如原样兼容，这将被检查为一个 bug 修复或"setExecutable"增强。

- 完成：添加"multiprocessing.setExecutable()"方法来覆盖包的默认行为，使用当前可执行文件名生成进程而不是 Python 解释器。注意 Mark Hammond 建议为此提供一个工厂风格的接口 [7]。
