# PEP 3135 – 新的 Super

**Metadata:**

- **Author**: Calvin Spealman <ironfroggy&#32;&#97;t&#32;gmail.com>, Tim Delaney <timothy.c.delaney&#32;&#97;t&#32;gmail.com>, Lie Ryan <lie.1296&#32;&#97;t&#32;gmail.com>
- **Created**: 28-Apr-2007
- **Python-Version**: 3.0

## 编号说明

本 PEP 最初是 PEP 367。由于它现在是针对 Python 3000，因此已移至 3xxx 空间。

## 摘要

本 PEP 提议为 super 类型的使用提供语法糖，以自动构建 super 类型实例，绑定到定义方法的类以及方法当前正在操作的对象（或类方法的类对象）。

新的 super 用法建议的前提如下：

```python
super().foo(1, 2)
```

替换旧的：

```python
super(Foo, self).foo(1, 2)
```

## 理由

当前的 super 用法需要显式传递它必须操作的类和实例，这违反了 DRY（Don't Repeat Yourself）原则。这阻碍了任何类名的更改，并且经常被许多人认为是一个缺点。

## 规范

在规范部分，将使用一些特殊术语来区分相似和密切相关的概念。"super class"指的是实际名为"super"的内置类。"super instance"只是一个 super 类的实例，它与另一个类以及可能该类的实例相关联。

新的 super 语义仅在 Python 3.0 中可用。

替换旧的 super 用法，调用 MRO（方法解析顺序）中的下一个类时，无需显式传递类对象（尽管仍然支持这样做）。每个函数都将有一个名为__class__的单元格，其中包含定义该函数的类对象。

新的语法：

```python
super()
```

等价于：

```python
super(__class__, <firstarg>)
```

其中__class__是定义方法的类，<firstarg>是方法的第一个参数（通常是实例方法的 self，类方法的 cls）。对于在类体之外定义的函数，__class__未定义，将导致运行时 SystemError。

虽然 super 不是保留字，但解析器识别方法定义中 super 的使用，并且仅当发现此情况时才传入__class__单元格。因此，不带参数调用 super 的全局别名不一定有效。

## 已关闭的问题

### 确定要使用的类对象

类对象取自名为__class__的单元格。

### super 是否应该真正成为关键字？

不。super 不需要成为关键字。

### super 与__call__属性一起使用

人们认为这可能是一个问题，即以经典方式实例化 super 实例，因为调用它会查找__call__属性，从而尝试执行到 MRO 中下一个类的自动 super 查找。但是，这被发现是错误的，因为调用对象仅在对象的类型上直接查找__call__方法。以下示例显示了这一点。

```python
class A(object):
    def __call__(self):
        return '__call__'
    def __getattribute__(self, attr):
        if attr == '__call__':
            return lambda: '__getattribute__'
a = A()
assert a() == '__call__'
assert a.__call__() == '__getattribute__'
```

无论如何，这个问题完全消失了，因为对 super(<class>, <instance>) 的经典调用仍然以相同的含义受到支持。

## 替代提案

### 不更改

虽然保持现状总是很有吸引力的，但人们已经寻求更改 super 调用一段时间了，并且有很好的理由，前面提到的所有理由。

- 与类名解耦（可能不再绑定到正确的类！）

- 看起来更简单、更干净的 super 调用会更好

### super 类型上的动态属性

该提案为 super 类型添加了动态属性查找，它将自动确定正确的类和实例参数。每个 super 属性查找识别这些参数并在实例上执行 super 查找，就像当前的 super 实现通过显式调用类和实例上的 super 实例一样。

该提案依赖于 `sys._getframe()`，这对于原型实现以外的任何用途都不合适。

### `self.__super__.foo(*args)`

__super__属性在本 PEP 的几处提到，可以是完整解决方案的候选者，实际上显式使用它而不是直接使用任何 super 用法。但是，双下划线名称通常是内部细节，并试图保持在日常代码之外。

### `super(self, *args)` 或 `__super__(self, *args)`

此解决方案仅解决了类型指示的问题，不处理不同名称的 super 方法，并且明确说明了实例的名称。它不够灵活，无法在其他方法名称上执行，在需要这样做的情况下。一个失败的用例是，基类有一个工厂类方法，子类有两个工厂类方法，两者都需要正确地向基类中的一个进行 super 调用。

### `super.foo(self, *args)`

这种变体实际上消除了定位正确实例的问题，如果任何替代方案被推向前台，我希望是这一个。

### `super(*p, **kw)`

有人提议直接调用 `super(*p, **kw)` 将等价于调用 super 对象上与当前正在执行的方法名称相同的方法，即以下两种方法是等价的：

```python
def f(self, *p, **kw):
    super.f(*p, **kw)

def f(self, *p, **kw):
    super(*p, **kw)
```

对此有强烈的支持和反对意见，但实现和样式问题显而易见。Guido 建议，根据 KISS（Keep It Simple Stupid）原则，这应该从本 PEP 中排除。
