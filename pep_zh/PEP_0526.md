## 摘要

PEP 484 引入了类型提示，也称为类型注解。虽然其主要关注点是函数注解，但它也引入了类型注释的概念来注解变量：

```python
# 'primes' 是一个整数列表
primes = [] # type: List[int]

# 'captain' 是一个字符串（注意：初始值是个问题）
captain = ... # type: str

class Starship:
    # 'stats' 是一个类变量
    stats = {} # type: Dict[str, int]
```

本 PEP 旨在为 Python 添加用于注解变量类型（包括类变量和实例变量）的语法，而不是通过注释来表达它们：

```python
primes: List[int] = []

captain: str  # 注意：没有初始值！

class Starship:
    stats: ClassVar[Dict[str, int]] = {}
```

PEP 484 明确指出类型注释旨在帮助复杂情况下的类型推断，本 PEP 不改变这一意图。然而，由于在实践中类型注释也被用于类变量和实例变量，本 PEP 还讨论了类型注解在这些变量中的使用。

## 理由

虽然类型注释工作得足够好，但它们是通过注释表达的，这有一些缺点：

- 文本编辑器通常以不同于类型注解的方式高亮显示注释。
- 没有办法注解未定义变量的类型；需要将其初始化为 None（例如 a = None # type: int）。
- 在条件分支中注解的变量很难阅读：

```python
if some_value:
    my_var = function()  # type: Logger
else:
    my_var = another_function()  # 为什么这里没有类型？
```

由于类型注释实际上不是语言的一部分，如果 Python 脚本想要解析它们，需要自定义解析器而不是直接使用 ast。

类型注释在 typeshed 中被大量使用。将 typeshed 迁移为使用变量注解语法而不是类型注释将提高存根的可读性。

在正常使用注释和类型注释的情况下，很难区分它们：

```python
path = None  # type: Optional[str]  # 模块源的路径
```

在不尝试查找模块源代码并在运行时解析它的情况下，无法在运行时检索注解，这至少可以说是不优雅的。

这些问题中的大多数可以通过将语法作为语言的核心部分来解决。此外，拥有专用的注解语法用于类和实例变量（除了方法注解外）将为静态鸭子类型铺平道路，作为 PEP 484 定义的名义类型的补充。

### 非目标

虽然本提案伴随着对 typing.get_type_hints 的扩展用于运行时检索注解的标准库函数，但变量注解不是为运行时类型检查设计的。第三方包需要开发来实现此类功能。

还应该强调，Python 仍将保持动态类型语言，作者无意使类型提示成为强制性的，即使是约定俗成的。类型注解不应与变量声明在静态类型语言中混淆。注解语法的目标是为第三方工具提供一种简单的方式来指定结构化类型元数据。

本 PEP 不要求类型检查器更改其类型检查规则。它只是提供了一个更易读的语法来替代类型注释。

## 规范

类型注解可以添加到赋值语句或单个表达式，指示注解目标的所需类型给第三方类型检查器：

```python
my_var: int
my_var = 5  # 通过类型检查。
other_var: int = 'a'  # 被类型检查器标记为错误，但在运行时没问题。
```

此语法不引入 PEP 484 之外的任何新语义，因此以下三个语句是等价的：

```python
var = value  # type: annotation
var: annotation; var = value
var: annotation = value
```

下面我们在不同上下文中指定类型注解的语法及其运行时效果。

我们还建议类型检查器如何解释注解，但遵守这些建议不是强制性的。（这与 PEP 484 中的遵守态度一致。）

### 全局和局部变量注解

局部变量和全局变量的类型可以如下注解：

```python
some_number: int  # 没有初始值的变量
some_list: List[int] = []  # 有初始值的变量
```

能够省略初始值使得为在条件分支中赋值的变量更容易输入类型：

```python
sane_world: bool
if 2+2 == 4:
    sane_world = True
else:
    sane_world = False
```

注意，虽然语法允许元组打包，但不允许在使用元组**解包**时注解变量的类型：

```python
# 使用变量注解语法的元组打包
t: Tuple[int, ...] = (1, 2, 3)
# 或
t: Tuple[int, ...] = 1, 2, 3  # 这只在 Python 3.8+ 中有效

# 使用变量注解语法的元组解包
header: str
kind: int
body: Optional[List[str]]
header, kind, body = message
```

省略初始值会使变量未初始化：

```python
a: int
print(a)  # 引发 NameError
```

然而，注解局部变量将导致解释器始终将其视为局部变量：

```python
def f():
    a: int
    print(a)  # 引发 UnboundLocalError
    # 注释掉 a: int 会使其成为 NameError。
```

就像代码是：

```python
def f():
    if False: a = 0
    print(a)  # 引发 UnboundLocalError
```

重复的类型注解将被忽略。但是，静态类型检查器可能会为不同类型的相同变量注解发出警告：

```python
a: int
a: str  # 静态类型检查器可能会警告此问题。
```

### 类和实例变量注解

类型注解也可用于注解类体和方法中的类和实例变量。特别是，无值表示法 a: int 允许注解应该在__init__或__new__中初始化的实例变量。建议的语法如下：

```python
class BasicStarship:
    captain: str = 'Picard'  # 有默认值的实例变量
    damage: int  # 没有默认值的实例变量
    stats: ClassVar[Dict[str, int]] = {}  # 类变量
```

这里 ClassVar 是 typing 模块定义的特殊类，指示静态类型检查器此变量不应在实例上设置。

注意，ClassVar 参数不能包含任何类型变量，无论嵌套级别如何：如果 T 是类型变量，则 ClassVar[T] 和 ClassVar[List[Set[T]]] 都无效。

这可以通过更详细的例子来说明。在这个类中：

```python
class Starship:
    captain = 'Picard'
    stats = {}

    def __init__(self, damage, captain=None):
        self.damage = damage
        if captain:
            self.captain = captain  # 否则保留默认值

    def hit(self):
        Starship.stats['hits'] = Starship.stats.get('hits', 0) + 1
```

stats 旨在成为类变量（跟踪许多不同的per-game统计信息），而 captain 是具有默认值的实例变量，在类中设置。类型检查器可能看不到这种差异：两者都在类中初始化，但 captain 仅作为实例变量的方便默认值，而 stats 真正是类变量——它旨在被所有实例共享。

由于两个变量恰好在类级别初始化，通过将它们标记为用 ClassVar[...] 包装的类型来区分它们是有用的。这样类型检查器可以标记实例上同名属性的意外赋值。

例如，注解所讨论的类：

```python
class Starship:
    captain: str = 'Picard'
    damage: int
    stats: ClassVar[Dict[str, int]] = {}

    def __init__(self, damage: int, captain: str = None):
        self.damage = damage
        if captain:
            self.captain = captain  # 否则保留默认值

    def hit(self):
        Starship.stats['hits'] = Starship.stats.get('hits', 0) + 1

enterprise_d = Starship(3000)
enterprise_d.stats = {}  # 被类型检查器标记为错误
Starship.stats = {}  # 这是可以的
```

为了方便起见（也是约定），实例变量可以在__init__或其他方法中注解，而不是在类中：

```python
from typing import Generic, TypeVar
T = TypeVar('T')

class Box(Generic[T]):
    def __init__(self, content):
        self.content: T = content
```

### 注解表达式

注解的目标可以是任何有效的单个赋值目标，至少在语法上（类型检查器如何处理由它决定）：

```python
class Cls:
    pass

c = Cls()
c.x: int = 0  # 用 int 注解 c.x。
c.y: int  # 用 int 注解 c.y。

d = {}
d['a']: int = 0  # 用 int 注解 d['a']。
d['b']: int  # 用 int 注解 d['b']。
```

注意，即使是括号化的名称也被视为表达式，而不是简单名称：

```python
(x): int  # 用 int 注解 x，(x) 被编译器视为表达式。
(y): int = 0  # 这里情况相同。
```

### 不允许注解的地方

试图注解受同一函数作用域中的 global 或 nonlocal 约束的变量是非法的：

```python
def f():
    global x: int  # SyntaxError

def g():
    x: int  # 也是 SyntaxError
    global x
```

原因是 global 和 nonlocal 不拥有变量；因此，类型注解属于拥有变量的作用域。

只允许单个赋值目标和单个右侧值。此外，不能注解在 for 或 with 语句中使用的变量；可以提前注解它们，类似于元组解包：

```python
a: int
for a in my_iter:
    ...

f: MyFile
with myfunc() as f:
    ...
```

### 存根文件中的变量注解

由于变量注解比类型注释更易读，因此在存根文件中它们适用于所有版本的 Python，包括 Python 2.7。注意存根文件不由 Python 解释器执行，因此使用变量注解不会导致错误。类型检查器应该支持所有 Python 版本存根中的变量注解。例如：

```python
# 文件 lib.pyi

ADDRESS: unicode = ...

class Error:
    cause: Union[str, unicode]
```

### 变量注解的首选编码风格

模块级变量、类和实例变量、以及局部变量的注解应在对应冒号后有一个空格。冒号前不应有空格。如果赋值有右侧，则等号两边应恰好有一个空格。示例：

是：

```python
code: int

class Point:
    coords: Tuple[int, int]
    label: str = '<unknown>'
```

否：

```python
code:int  # 冒号后没有空格
code : int  # 冒号前有空格

class Test:
    result: int=0  # 等号周围没有空格
```

## 标准库和文档的更改

一个新的协变类型 ClassVar[T_co] 被添加到 typing 模块。它只接受一个参数，该参数应该是有效类型，用于注解不应在类实例上设置的类变量。此限制由静态检查器确保，但不在运行时。见 classvar 部分了解 ClassVar 用法的示例和解释，见被拒绝部分了解 ClassVar 背后的推理的更多信息。

typing 模块中的函数 get_type_hints 将被扩展，以便可以从模块和类以及函数在运行时检索类型注解。

注解作为字典返回，映射从变量或参数到它们的类型提示，前向引用已求值。对于类，它返回一个映射（可能是 collections.ChainMap）从方法解析顺序中的注解构建。

使用注解的推荐指南将添加到文档中，包含本 PEP 和 PEP 484 中描述的规范的教学性总结。此外，一个辅助脚本用于将类型注释转换为类型注解将独立于标准库发布。

## 类型注解的运行时效果

注解局部变量将导致解释器将其视为局部变量，即使它从未被赋值。**局部变量的注解不会被求值**：

```python
def f():
    x: NonexistentName  # 没有错误。
```

然而，如果它在模块或类级别，则类型将被求值：

```python
x: NonexistentName  # 错误！
class X:
    var: NonexistentName  # 错误！
```

此外，在模块或类级别，如果被注解的项目是简单名称，则它和注解将存储在该模块或类的__annotations__属性中（如果私有则扰乱）作为从名称到已求值注解的有序映射。这是一个例子：

```python
from typing import Dict
class Player:
    ...
players: Dict[str, Player]
__points: int

print(__annotations__)
# 打印：{'players': typing.Dict[str, __main__.Player],
# '_Player__points': <class 'int'>}
```

__annotations__是可写的，所以这是允许的：

```python
__annotations__['s'] = str
```

但尝试将__annotations__更新为除有序映射之外的内容可能会导致 TypeError：

```python
class C:
    __annotations__ = 42
    x: int = 5  # 引发 TypeError
```

（注意对__annotations__的赋值，这是罪魁祸首，被 Python 解释器接受而不加质疑——但随后的类型注解期望它是一个 MutableMapping 并将失败。）

在运行时获取注解的推荐方式是使用 typing.get_type_hints 函数；与所有双下划线属性一样，__annotations__的任何未记录使用都可能在不警告的情况下破坏：

```python
from typing import Dict, ClassVar, get_type_hints
class Starship:
    hitpoints: int = 50
    stats: ClassVar[Dict[str, int]] = {}
    shield: int = 100
    captain: str
    def __init__(self, captain: str) -> None:
        ...

assert get_type_hints(Starship) == {'hitpoints': int,
                                    'stats': ClassVar[Dict[str, int]],
                                    'shield': int,
                                    'captain': str}

assert get_type_hints(Starship.__init__) == {'captain': str,
                                             'return': None}
```

注意，如果静态找不到注解，则__annotations__字典根本不会被创建。此外，拥有本地可用的注解的价值并不能抵消每次函数调用时创建和填充注解字典的成本。因此，函数级别的注解不会被求值也不会被存储。

### 注解的其他用途

虽然带有本 PEP 的 Python 不会反对：

```python
alice: 'well done' = 'A+'
bob: 'what a shame' = 'F-'
```

因为它不会在乎类型注解之外的事情，只要"它求值不引发异常"，遇到它的类型检查器将标记它，除非用# type: ignore 或@no_type_check 禁用。

然而，由于 Python 不在乎"类型"是什么，如果上面的代码段在全局级别或在类中，__annotations__将包含{'alice': 'well done', 'bob': 'what a shame'}。

这些存储的注解可能用于其他目的，但通过本 PEP 我们明确推荐类型提示作为注解的首选用途。

## 被拒绝/推迟的提案

### 我们是否应该引入变量注解？

变量注解已经存在近两年了以类型注释的形式，由 PEP 484 批准。它们被第三方类型检查器（mypy、pytype、PyCharm 等）和使用类型检查器的项目广泛使用。然而，注释语法有许多缺点，列在理由中。本 PEP 不是关于类型注解的需求，而是关于什么应该是此类注解的语法。

- **引入新关键字**：

选择一个好的关键字很难，例如，它不能是 var，因为那是一个太常见的变量名，如果我们想用于类变量或全局变量，它也不能是 local。其次，无论我们选择什么，我们仍然需要__future__导入。

- **使用 def 作为关键字**：

提案将是：

```python
def primes: List[int] = []
def captain: str
```

这个问题在于 def 对几代 Python 程序员（和工具！）意味着"定义函数"，用它来定义变量不会增加清晰度。（当然这是主观的。）

- **使用基于函数的语法**：

有人提出使用 var = cast(annotation[, value]) 来注解变量类型。虽然这种语法缓解了一些类型注释的问题，比如 AST 中缺少注解，但它不能解决其他问题，如可读性，并且引入了可能的运行时开销。

- **允许元组解包的类型注解**：

这会导致歧义：不清楚这个语句是什么意思：

```python
x, y: T
```

x 和 y 都是类型 T，还是我们期望 T 是两个项目的元组类型，分布在 x 和 y 上，或者也许 x 有类型 Any 而 y 有类型 T？（后者是如果这出现在函数签名中的意思。）为了避免让（人类）读者猜测，我们禁止这个，至少现在是这样。

- **括号形式 (var: type) 用于注解**：

这在 python-ideas 上被提出作为对上述歧义的补救，但它被拒绝了，因为这种语法会很复杂，好处很小，可读性会很差。

- **允许链式赋值中的注解**：

这有与元组解包类似的歧义和可读性问题，例如在：

```python
x: int = y = 1
z = w: int = 1
```

不清楚 y 和 z 的类型应该是什么？另外，第二行很难解析。

- **允许 with 和 for 语句中的注解**：

这被拒绝了，因为在 for 中它会使得难以发现实际的可迭代对象，在 with 中它会混淆 CPython 的 LL(1) 解析器。

- **在函数定义时求值局部注解**：

这被 Guido 拒绝了，因为注解的位置强烈暗示它与周围代码在同一作用域中。

- **在函数作用域中也存储变量注解**：

拥有本地可用注解的价值不足以显著抵消每次函数调用时创建和填充字典的成本。

- **初始化未赋值的注解变量**：

有人在 python-ideas 上提出将 x: int 中的 x 初始化为 None 或额外的特殊常量，如 Javascript 的 undefined。然而，向语言添加另一个单例值需要在代码的每个地方进行检查。因此，Guido 只是直截了当地说"不"。

- **向 typing 模块添加 InstanceVar**：

这是多余的，因为实例变量比类变量常见得多。更常见的用法应该成为默认值。

- **只允许在方法中注解实例变量**：

问题是许多__init__方法除了初始化实例变量之外还做很多事情，对（人类）来说更难找到所有实例变量注解。有时__init__被分解成更多辅助方法，所以更难追踪它们。将实例变量注解放在一起在类中使它们更容易找到，并帮助代码的第一次读者。

- **使用语法 x: class t = v 用于类变量**：

这将需要更复杂的解析器，并且 class 关键字会混淆简单的语法高亮器。无论如何我们需要 ClassVar 存储类变量到__annotations__，所以选择了更简单的语法。

- **完全忘记 ClassVar**：

这被提出了，因为 mypy 似乎在没有区分类变量和实例变量的方法的情况下运行良好。但是类型检查器可以用额外信息做一些有用的事情，例如标记通过实例对类变量的意外赋值（这将创建一个遮蔽类变量的实例变量）。它还可以标记具有可变默认值的实例变量，这是一个众所周知的危险。

- **使用 ClassAttr 代替 ClassVar**：

ClassVar 更好的主要原因是：许多东西都是类属性，例如方法、描述符等。但只有特定的属性在概念上是类变量（或可能是常量）。

- **不求值注解，将它们视为字符串**：

这将与函数注解的行为不一致，函数注解总是被求值。虽然未来可能会重新考虑，但在 PEP 484 中决定这必须是一个单独的 PEP。

- **在类 docstring 中注解变量类型**：

许多项目已经使用各种 docstring 约定，通常没有太多一致性，并且通常不符合 PEP 484 注解语法。此外，这将需要一个特殊的复杂解析器。反过来，这将破坏 PEP 的目的——与第三方类型检查工具合作。

- **将__annotations__实现为描述符**：

这被提出以禁止将__annotations__设置为非字典或非 None 的内容。Guido 拒绝了这一想法，认为这是不必要的；相反，如果尝试更新__annotations__为映射之外的任何内容，将引发 TypeError。

- **将裸注解与 global 或 nonlocal 同等对待**：

被拒绝的提案将希望函数体中存在没有赋值的注解不应涉及任何求值。相反，PEP 意味着如果目标比单个名称更复杂，其"左侧部分"应该在它出现在函数体中的点被求值，只是为了强制它被定义。例如，在这个例子中：

```python
def foo(self):
    slef.name: str
```

名称 slef 应该被求值，只是为了如果它未定义（在这个例子中很可能如此 :-)），错误将在运行时被捕获。这更符合当有初始值时发生的情况，因此预计会导致更少的意外。（还要注意，如果目标是 self.name（这次正确拼写 :-)），优化编译器没有义务求值 self，只要它能证明它肯定会被定义。）
