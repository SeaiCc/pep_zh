# PEP 249 – Python 数据库 API 规范 v2.0

**元数据：**

- **作者**: Marc-André Lemburg <mal&#32;&#97;t&#32;lemburg.com>
- **创建日期**: 1999 年 4 月 12 日

## 简介

此 API 已被定义，以鼓励用于访问数据库的 Python 模块之间的相似性。通过这样做，我们希望实现一致性，从而导致更容易理解的模块、在数据库之间更具可移植性的代码，以及 Python 更广泛的数据库连接能力。

关于此规范的评论和问题可以发送给 Python 数据库接口 SIG。

有关 Python 数据库接口和可用包的更多信息，请参阅数据库主题指南。

本文档描述了 Python 数据库 API 规范 2.0 以及一组常见的可选扩展。前一版本 1.0 版本仍可作为参考，在 PEP 248 中。鼓励包编写者使用此版本的规范作为新接口的基础。

## 模块接口

### 构造函数

对数据库的访问通过连接对象提供。模块必须为这些提供以下构造函数：

**connect(*parameters…*)** 
用于创建数据库连接的构造函数。返回一个连接对象。它接受许多参数，这些参数依赖于数据库。[[1](#id1)]

### 全局变量

必须定义以下模块全局变量：

**apilevel** 
说明支持的 DB API 级别的字符串常量。
目前仅允许字符串"1.0"和"2.0"。如果未给出，则应假设 DB-API 1.0 级别接口。

**threadsafety** 
说明接口支持的线程安全级别的整数常量。可能的值为：

| threadsafety | 含义 |
|-------------|------|
| 0 | 线程不能共享模块。 |
| 1 | 线程可以共享模块，但不能共享连接。 |
| 2 | 线程可以共享模块和连接。 |
| 3 | 线程可以共享模块、连接和游标。 |

上述上下文中的共享意味着两个线程可以使用资源，而无需使用互斥信号量包装它来实现资源锁定。注意，你不能总是通过使用互斥管理访问来使外部资源线程安全：资源可能依赖于全局变量或其他你无法控制的外部源。

**paramstyle** 
说明接口期望的参数标记格式的字符串常量。可能的值为 [[2](#id2)]：

| paramstyle | 含义 |
|-----------|------|
| qmark | 问号样式，例如 ...WHERE name=? |
| numeric | 数字位置样式，例如 ...WHERE name=:1 |
| named | 命名样式，例如 ...WHERE name=:name |
| format | ANSI C printf 格式代码，例如 ...WHERE name=%s |
| pyformat | Python 扩展格式代码，例如 ...WHERE name=%(name)s |

### 异常

模块应通过这些异常或其子类提供所有错误信息：

**Warning** 
为重要警告引发的异常，如插入时的数据截断等。它必须是 Python Exception 类的子类 [[10](#id10)] [[11](#id11)]。

**Error** 
是所有其他错误异常的基类的异常。你可以使用它用一个单独的 except 语句捕获所有错误。警告不被视为错误，因此不应将此用作基类。它必须是 Python Exception 类的子类 [[10](#id10)]。

**InterfaceError** 
为与数据库接口相关而不是数据库本身的错误引发的异常。它必须是 Error 的子类。

**DatabaseError** 
为与数据库相关的错误引发的异常。它必须是 Error 的子类。

**DataError** 
为由于处理数据的问题引起的错误引发的异常，如除以零、数值超出范围等。它必须是 DatabaseError 的子类。

**OperationalError** 
为与数据库操作相关的错误引发的异常，而不一定在程序员的控制之下，例如发生意外断开连接、找不到数据源名称、无法处理事务、处理期间发生内存分配错误等。它必须是 DatabaseError 的子类。

**IntegrityError** 
当数据库的关系完整性受到影响时引发的异常，例如外键检查失败。它必须是 DatabaseError 的子类。

**InternalError** 
当数据库遇到内部错误时引发的异常，例如游标不再有效、事务不同步等。它必须是 DatabaseError 的子类。

**ProgrammingError** 
为编程错误引发的异常，例如表未找到或已存在、SQL 语句中的语法错误、指定了错误数量的参数等。它必须是 DatabaseError 的子类。

**NotSupportedError** 
在使用数据库不支持的方法或数据库 API 时引发的异常，例如在不支持事务或已关闭事务的连接上请求.rollback()。它必须是 DatabaseError 的子类。

这是异常继承布局 [[10](#id10)] [[11](#id11)]：

```
Exception
|__Warning
|__Error
 |__InterfaceError
 |__DatabaseError
 |__DataError
 |__OperationalError
 |__IntegrityError
 |__InternalError
 |__ProgrammingError
 |__NotSupportedError
```

注意：这些异常的值未定义。但它们应该给用户一个相当好的关于出了什么问题的想法。

## 连接对象

连接对象应该响应以下方法。

### 连接方法

**.close()** 
立即关闭连接（而不是在调用`.__del__()` 时）。从现在开始连接将无法使用；如果尝试对连接执行任何操作，将引发 Error（或子类）异常。同样的也适用于所有尝试使用连接的游标对象。注意，首先不提交更改就关闭连接将导致执行隐式回滚。

**.commit()** 
将任何挂起的事务提交到数据库。注意，如果数据库支持自动提交功能，则必须最初关闭它。可以提供接口方法来将其重新打开。

不支持事务的数据库模块应该用空功能实现此方法。

**.rollback()** 
此方法是可选的，因为并非所有数据库都提供事务支持。[[3](#id3)] 如果数据库确实提供事务，此方法导致数据库回滚到任何挂起事务的开始。首先不提交更改就关闭连接将导致执行隐式回滚。

**.cursor()** 
使用连接返回一个新的游标对象。如果数据库不直接提供游标概念，则模块必须使用其他手段来模拟游标，以满足本规范所需的程度。[[4](#id4)]

## 游标对象

这些对象代表数据库游标，用于管理获取操作的上下文。从同一连接创建的游标不是隔离的，即一个游标对数据库所做的任何更改立即对其他游标可见。从不同连接创建的游标可以或不可以隔离，具体取决于事务支持的实现方式（另请参见连接的.rollback() 和.commit() 方法）。

游标对象应该响应以下方法和属性。

### 游标属性

**.description** 此只读属性是 7 项序列的序列。这些序列中的每一个都包含描述一个结果列的信息：

- name
- type_code
- display_size
- internal_size
- precision
- scale
- null_ok

前两项（name 和 type_code）是强制性的，其他五项是可选的，如果无法提供有意义的值，则设置为 None。

对于不返回行的操作，或者如果游标尚未通过.execute*() 方法调用操作，此属性将为 None。

type_code 可以通过将其与下节中指定的类型对象进行比较来解释。

**.rowcount** 此只读属性指定最后一个.execute*() 生成的行数（对于像 SELECT 这样的 DQL 语句）或受影响的行数（对于像 UPDATE 或 INSERT 这样的 DML 语句）。[[9](#id9)] 如果未在游标上执行.rowcount*()，或者无法通过接口确定最后操作的 rowcount，则属性为 -1。[[7](#id7)]

注意：DB API 规范的将来版本可能会重新定义后一种情况，使对象返回 None 而不是 -1。

### 游标方法

**.callproc(*procname* [, *parameters*])** 
（此方法是可选的，因为并非所有数据库都提供存储过程。[[3](#id3)]）调用给定名称的存储数据库过程。参数序列必须为过程期望的每个参数包含一个条目。调用的结果作为输入序列的修改副本返回。输入参数保持不变，输出和输入/输出参数用可能的新值替换。

该过程还可以提供结果集作为输出。然后必须通过标准的.fetch*() 方法使其可用。

**.close()** 
立即关闭游标（而不是在调用__del__时）。从现在开始游标将无法使用；如果尝试对游标执行任何操作，将引发 Error（或子类）异常。

**.execute(*operation* [, *parameters*])** 
准备并执行数据库操作（查询或命令）。参数可以作为序列或映射提供，并将绑定到操作中的变量。变量以数据库特定的符号指定（有关详细信息，请参阅模块的 paramstyle 属性）。[[5](#id5)]

游标将保留对操作的引用。如果再次传递相同的操作对象，则游标可以优化其行为。这对于使用相同操作但绑定不同参数（多次）的算法最有效。

为了在重用操作时获得最大效率，最好使用.setinputsizes() 方法提前指定参数类型和大小。参数不匹配预定义信息是合法的；实现应该补偿，可能会降低效率。

参数也可以指定为元组列表，例如在单个操作中插入多行，但这种用法已被弃用：应使用.executemany()。

未定义返回值。

**.executemany(*operation*, *seq_of_parameters*)** 
准备数据库操作（查询或命令），然后针对序列*seq_of_parameters*中发现的所有参数序列或映射执行它。

模块可以使用多次调用.execute() 方法或使用数组操作让数据库在一次调用中处理整个序列来实现此方法。

对此产生一个或多个结果集的操作使用此方法构成未定义行为，实现被允许（但不需要）在检测到操作调用创建结果集时引发异常。

与.execute() 相同的注释也相应地适用于此方法。

未定义返回值。

**.fetchone()** 
获取查询结果集的下一行，返回单个序列，或者在没有更多数据时返回 None。[[6](#id6)] 如果之前对.execute*() 的调用没有生成任何结果集或未发出任何调用，则引发 Error（或子类）异常。

**.fetchmany([*size=cursor.arraysize*])** 
获取查询结果的下一组行，返回序列的序列（例如元组列表）。没有更多可用行时返回空序列。

每次调用要获取的行数由参数指定。如果未给出，则游标的 arraysize 确定要获取的行数。该方法应尝试获取 size 参数指示的尽可能多的行。如果由于指定的行数不可用而无法做到这一点，则可能返回较少的行。

如果之前对.execute*() 的调用没有生成任何结果集或未发出任何调用，则引发 Error（或子类）异常。

注意，size 参数涉及性能考虑。为了最佳性能，通常最好使用.arraysize 属性。如果使用 size 参数，则最好让它从一个.fetchmany() 调用到下一个调用保持相同的值。

**.fetchall()** 
获取查询结果的所有（剩余）行，将它们作为序列的序列返回（例如元组列表）。注意，游标的 arraysize 属性会影响此操作的性能。如果之前对.execute*() 的调用没有生成任何结果集或未发出任何调用，则引发 Error（或子类）异常。

**.nextset()** 
（此方法是可选的，因为并非所有数据库都支持多个结果集。[[3](#id3)]）此方法将使游标跳到下一个可用的结果集，丢弃当前集中的任何剩余行。

如果没有更多结果集，该方法返回 None。否则，它返回真值，随后对.fetch*() 方法的调用将返回下一个结果集中的行。

如果之前对.execute*() 的调用没有生成任何结果集或未发出任何调用，则引发 Error（或子类）异常。

**.arraysize** 
此可读/写属性指定与.fetchmany() 一起每次获取的行数。默认为 1，意味着每次获取单行。

实现必须遵守此值关于.fetchmany() 方法，但可以自由地一次与数据库交互单行。它也可以用于.executemany() 的实现中。

**.setinputsizes(*sizes*)** 
这可以在调用.execute*() 之前使用，以预定义操作参数的内存区域。sizes 指定为序列——每个输入参数一个项目。该项目应是一个与将使用的输入对应的类型对象，或者它应该是一个指定字符串参数最大长度的整数。如果项目为 None，则不会为该列保留预定义的内存区域（这可用于避免为大输入预定义区域）。

此方法在调用.execute*() 方法之前使用。

实现可以自由地让此方法什么都不做，用户也可以自由地不使用它。

**.setoutputsize(*size* [, *column*])** 
为获取大列（例如 LONG、BLOB 等）设置列缓冲区大小。列指定为结果序列中的索引。不指定列将为游标中的所有大列设置默认大小。

此方法在调用.execute*() 方法之前使用。

实现可以自由地让此方法什么都不做，用户也可以自由地不使用它。

## 类型对象和构造函数

许多数据库需要以特定格式输入以绑定到操作的输入参数。例如，如果输入 destined 用于 DATE 列，则必须以特定字符串格式绑定到数据库。对于"Row ID"列或大二进制项（例如 blobs 或 RAW 列）也存在类似问题。这给 Python 带来了问题，因为.execute*() 方法的参数是未类型的。当数据库模块看到 Python 字符串对象时，它不知道应该将其绑定为简单的 CHAR 列、原始 BINARY 项还是 DATE。

为了克服这个问题，模块必须提供下面定义的构造函数来创建可以保存特殊值的对象。当传递给游标方法时，模块然后可以检测输入参数的正确类型并相应地绑定它。

游标对象的 description 属性返回有关查询的每个结果列的信息。type_code 必须与下面定义的类型对象之一相等。类型对象可以等于多个类型代码（例如，DATETIME 可以等于日期、时间和时间戳列的类型代码；有关详细信息，请参阅下面的实现提示）。

模块导出以下构造函数和单例：

**Date(*year*, *month*, *day*)** 此函数构造一个保存日期值的对象。

**Time(*hour*, *minute*, *second*)** 此函数构造一个保存时间值的对象。

**Timestamp(*year*, *month*, *day*, *hour*, *minute*, *second*)** 此函数构造一个保存时间戳值的对象。

**DateFromTicks(*ticks*)** 此函数从给定的 ticks 值（自纪元以来的秒数；请参阅标准 Python time 模块的文档以获取详细信息）构造一个保存日期值的对象。

**TimeFromTicks(*ticks*)** 此函数从给定的 ticks 值（自纪元以来的秒数；请参阅标准 Python time 模块的文档以获取详细信息）构造一个保存时间值的对象。

**TimestampFromTicks(*ticks*)** 此函数从给定的 ticks 值（自纪元以来的秒数；请参阅标准 Python time 模块的文档以获取详细信息）构造一个保存时间戳值的对象。

**Binary(*string*)** 此函数构造一个能够保存二进制（长）字符串值的对象。

**STRING 类型** 此类型对象用于描述数据库中的基于字符串的列（例如 CHAR）。

**BINARY 类型** 此类型对象用于描述数据库中的（长）二进制列（例如 LONG、RAW、BLOBs）。

**NUMBER 类型** 此类型对象用于描述数据库中的数字列。

**DATETIME 类型** 此类型对象用于描述数据库中的日期/时间列。

**ROWID 类型** 此类型对象用于描述数据库中的"Row ID"列。

SQL NULL 值在输入和输出上由 Python None 单例表示。

注意：使用 Unix ticks 进行数据库接口可能会因为它们的有限日期范围而带来麻烦。

## 模块作者的实现提示

- 日期/时间对象可以实现为 Python datetime 模块对象（自 Python 2.3 起可用，自 2.4 起提供 C API）或使用 mxDateTime 包（自 1.5.2 起适用于所有 Python 版本）。它们都在 Python 和 C 级别提供所有必要的构造函数和方法。

- 这是基于 Unix ticks 的日期/时间构造函数的示例实现，将工作委托给通用构造函数：

```python
import time

def DateFromTicks(ticks):
    return Date(*time.localtime(ticks)[:3])

def TimeFromTicks(ticks):
    return Time(*time.localtime(ticks)[3:6])

def TimestampFromTicks(ticks):
    return Timestamp(*time.localtime(ticks)[:6])
```

- Binary 对象的首选对象类型是标准 Python 从版本 1.5.2 开始提供的缓冲区类型。请参阅 Python 文档以获取详细信息。有关 C 接口的信息，请查看 Python 源发行版中的 Include/bufferobject.h 和 Objects/bufferobject.c。

- 此 Python 类允许实现上述类型对象，即使描述类型代码字段为单个类型对象产生多个值：

```python
class DBAPITypeObject:
    def __init__(self,*values):
        self.values = values
    def __cmp__(self,other):
        if other in self.values:
            return 0
        if other < self.values:
            return 1
        else:
            return -1
```

生成的类型对象与传递给构造函数的所有值相等。

- 这是实现上述异常层次的 Python 代码片段 [[10](#id10)]：

```python
class Error(Exception):
    pass

class Warning(Exception):
    pass

class InterfaceError(Error):
    pass

class DatabaseError(Error):
    pass

class InternalError(DatabaseError):
    pass

class OperationalError(DatabaseError):
    pass

class ProgrammingError(DatabaseError):
    pass

class IntegrityError(DatabaseError):
    pass

class DataError(DatabaseError):
    pass

class NotSupportedError(DatabaseError):
    pass
```

在 C 中，你可以使用 PyErr_NewException(fullname, base, NULL) API 创建异常对象。

## 可选 DB API 扩展

在 DB API 2.0 的生命周期内，模块作者经常将其实现扩展到本 DB API 规范所需之外。为了增强兼容性并为规范的可能未来版本提供清晰的升级路径，本节定义了一组对核心 DB API 2.0 规范的常见扩展。

与所有 DB API 可选功能一样，数据库模块作者可以自由地不实现这些额外的属性和方法（使用它们将导致 AttributeError），或者在只能在运行时检查可用性时引发 NotSupportedError。

已提议通过 Python 警告框架发出 Python 警告，使这些扩展的使用对程序员可选可见。为了使此功能有用，必须标准化警告消息以能够屏蔽它们。这些标准消息在下文中称为警告消息。

**Cursor.rownumber** 此只读属性应提供结果集中游标的当前基于 0 的索引，或者如果无法确定索引则为 None。索引可以看作是序列（结果集）中游标的索引。下一个获取操作将获取该序列中.rownumber 索引的行。

警告消息："DB-API extension cursor.rownumber used"

**Connection.Error、Connection.ProgrammingError 等** DB API 标准定义的所有异常类都应作为属性暴露在连接对象上（除了在模块范围内可用）。这些属性简化了多连接环境中的错误处理。

警告消息："DB-API extension connection.<exception> used"

**Cursor.connection** 此只读属性返回创建游标的连接对象的引用。该属性简化了多连接环境中编写多态代码。

警告消息："DB-API extension cursor.connection used"

**Cursor.scroll(*value* [, *mode='relative'*])** 根据*mode*在结果集中滚动游标到新位置。如果 mode 是 relative（默认），value 被视为结果集中当前位置的偏移量，如果设置为 absolute，value 表示绝对目标位置。

如果滚动操作将离开结果集，则应引发 IndexError。在这种情况下，游标位置未定义（理想的是根本不移动游标）。

注意：此方法应使用本机可滚动游标（如果可用），或者回退到仅向前可滚动游标的模拟。该方法可以引发 NotSupportedError 来信号数据库不支持特定操作（例如向后滚动）。

警告消息："DB-API extension cursor.scroll() used"

**Cursor.messages** 这是一个 Python 列表对象，接口将元组（异常类，异常值）附加到其中，用于接口从此游标的底层数据库接收的所有消息。列表由所有标准游标方法调用（在执行调用之前）自动清空，除了.fetch*() 调用，以避免过度的内存使用，也可以通过执行 del cursor.messages[:] 来清空。

数据库生成的所有错误和警告消息都放入此列表中，因此检查列表允许用户验证方法调用的正确操作。

此属性的目的是消除对 Warning 异常的需要，这通常会导致问题（一些警告真的只有信息性质）。

警告消息："DB-API extension cursor.messages used"

**Connection.messages** 与 Cursor.messages 相同，除了列表中的消息是面向连接的。列表由所有标准连接方法调用（在执行调用之前）自动清空，以避免过度的内存使用，也可以通过执行 del connection.messages[:] 来清空。

警告消息："DB-API extension connection.messages used"

**Cursor.next()** 使用与.fetchone() 相同的语义返回当前执行的 SQL 语句的下一行。当结果集用尽时，对于 Python 2.2 及更高版本，引发 StopIteration 异常。早期版本没有 StopIteration 异常，因此方法应引发 IndexError。

警告消息："DB-API extension cursor.next() used"

**Cursor.__iter__()** 返回 self 以使游标与迭代协议兼容 [[8](#id8)]。

警告消息："DB-API extension cursor.__iter__() used"

**Cursor.lastrowid** 此只读属性提供最后修改行的 rowid（大多数数据库仅在執行单个 INSERT 操作时返回 rowid）。如果操作不设置 rowid 或数据库不支持 rowid，则此属性应设置为 None。在最后一次执行的语句修改多于一行时，.lastrowid 的语义未定义，例如使用.executemany() 时的 INSERT。

警告消息："DB-API extension cursor.lastrowid used"

**Connection.autocommit** 用于查询和设置连接的自动提交模式的属性。如果连接处于自动提交（非事务）模式，则返回 True。如果连接处于手动提交（事务）模式，则返回 False。

将属性设置为 True 或 False 会相应地调整连接模式。

从 True 更改为 False（禁用自动提交）将使数据库离开自动提交模式并开始新事务。从 False 更改为 True（启用自动提交）具有依赖于数据库的语义，关于如何处理挂起的事务。[[12](#id12)]

弃用通知：即使几个数据库模块都实现了此属性的读写性质，通过写入属性来设置自动提交模式已被弃用，因为这可能导致 I/O 和相关异常，使得在异步上下文中实现变得困难。[[13](#id13)]

警告消息："DB-API extension connection.autocommit used"

## 可选错误处理扩展

核心 DB API 规范仅引入了一组可以引发以向用户报告错误的异常。在某些情况下，异常可能对程序流程过于破坏性，甚至使执行变得不可能。

对于这些情况，并为了简化处理数据库时的错误处理，数据库模块作者可以选择实现用户可定义的错误处理器。本节描述了定义这些错误处理器的标准方法。

**Connection.errorhandler、Cursor.errorhandler** 可读/写属性，引用在满足错误条件时要调用的错误处理器。处理器必须是 Python 可调用对象，接受以下参数：

```python
errorhandler(connection, cursor, errorclass, errorvalue)
```

其中 connection 是对游标在其上操作的连接的引用，cursor 是对游标的引用（如果错误不适用于游标则为 None），errorclass 是要使用 errorvalue 作为构造参数实例化的错误类。

标准错误处理器应将错误信息添加到适当的.messages 属性（Connection.messages 或 Cursor.messages），并引发由给定 errorclass 和 errorvalue 参数定义的异常。

如果未设置.errorhandler（属性为 None），则应应用上述标准错误处理方案。

警告消息："DB-API extension .errorhandler used"

游标应在游标创建时从连接对象继承.errorhandler 设置。

## 可选的两阶段提交扩展

许多数据库支持两阶段提交（TPC），允许跨多个数据库连接和其他资源管理事务。

如果数据库后端提供两阶段提交支持，并且数据库模块作者希望公开此支持，则应实现以下 API。如果只能在运行时检查数据库后端对两阶段提交的支持，则应引发 NotSupportedError。

### TPC 事务 ID

由于许多数据库遵循 XA 规范，事务 ID 由三个组件组成：

- 格式 ID
- 全局事务 ID
- 分支限定符

对于特定的全局事务，前两个组件对所有资源应该相同。全局事务中的每个资源应分配不同的分支限定符。

各个组件必须满足以下标准：

- 格式 ID：非负 32 位整数。
- 全局事务 ID 和分支限定符：长度不超过 64 个字符的字节字符串。

事务 ID 使用.xid() 连接方法创建：

**.xid(*format_id*, *global_transaction_id*, *branch_qualifier*)** 返回一个事务 ID 对象，适合传递给此连接的.tpc_*() 方法。如果数据库连接不支持 TPC，则引发 NotSupportedError。

.xid() 返回的对象的类型未定义，但它必须提供序列行为，允许访问三个组件。符合规范的数据库模块可以选择使用元组而不是自定义对象来表示事务 ID。

### TPC 连接方法

**.tpc_begin(*xid*)** 使用给定的事务 ID *xid* 开始 TPC 事务。此方法应在事务之外调用（即自上次.commit() 或.rollback() 以来未执行任何操作）。

此外，在 TPC 事务内调用.commit() 或.rollback() 是错误。如果应用程序在活动的 TPC 事务期间调用.commit() 或.rollback()，则引发 ProgrammingError。

如果数据库连接不支持 TPC，则引发 NotSupportedError。

**.tpc_prepare()** 执行使用.tpc_begin() 启动的事务的第一阶段。如果在 TPC 事务之外调用此方法，则应引发 ProgrammingError。调用.tpc_prepare() 后，不能执行任何语句，直到调用.tpc_commit() 或.tpc_rollback()。

**.tpc_commit([ *xid* ])** 不带参数调用时，.tpc_commit() 提交之前使用.tpc_prepare() 准备的 TPC 事务。如果在.tpc_prepare() 之前调用.tpc_commit()，则执行单阶段提交。事务管理器可以选择这样做，如果只有单个资源参与全局事务。

使用事务 ID xid 调用时，数据库提交给定的事务。如果提供无效的事务 ID，则引发 ProgrammingError。此形式应在事务之外调用，旨在用于恢复。

返回时，TPC 事务结束。

**.tpc_rollback([ *xid* ])** 不带参数调用时，.tpc_rollback() 回滚 TPC 事务。它可以在.tpc_prepare() 之前或之后调用。使用事务 ID xid 调用时，它回滚给定的事务。如果提供无效的事务 ID，则引发 ProgrammingError。此形式应在事务之外调用，旨在用于恢复。

返回时，TPC 事务结束。

**.tpc_recover()** 返回待处理的事务 ID 列表，适合与.tpc_commit(xid) 或.tpc_rollback(xid) 一起使用。如果数据库不支持事务恢复，它可以返回空列表或引发 NotSupportedError。

## 常见问题解答

数据库 SIG 经常看到关于 DB API 规范的反复出现的问题。本节涵盖了人们有时对规范的一些问题。

**问题：**
如何从.fetch*() 返回的元组构建字典：

**答案：**
有几个现有的工具可以提供此任务的帮助程序。它们大多数使用使用游标属性.description 中定义的列名作为行字典中键的基础的方法。

注意，不扩展 DB API 规范以也支持.fetch*() 方法的字典返回值的原因是这种方法有几个缺点：

- 一些数据库不支持区分大小写的列名，或者自动将它们转换为所有小写或所有大写字符。
- 结果集中由查询生成的列（例如使用 SQL 函数）不映射到表列名，数据库通常以非常特定的方式为此类列生成名称。

因此，通过字典键访问列在数据库之间变化，使得编写可移植代码变得不可能。

## 从版本 1.0 到版本 2.0 的主要更改

Python 数据库 API 2.0 与 1.0 版本相比引入了一些重大更改。由于其中一些更改将导致现有基于 DB API 1.0 的脚本中断，因此调整了主要版本号以反映此更改。

这是从 1.0 到 2.0 的最重要更改：

- 不再需要单独的 dbi 模块，功能合并到模块接口本身。

- 为日期/时间值添加了新的构造函数和类型对象，RAW 类型对象重命名为 BINARY。生成的集合应覆盖现代 SQL 数据库中常见的所有基本数据类型。

- 添加了新常量（apilevel、threadsafety、paramstyle）和方法（.executemany()、.nextset()）以提供更好的数据库绑定。

- 调用存储过程的.callproc() 的语义现在已明确定义。

- .execute() 返回值的定义已更改。以前，返回值基于 SQL 语句类型（这很难正确实现）——现在未定义；使用更灵活的.rowcount 属性。模块可以自由地返回旧式返回值，但这些不再由规范强制要求，应视为数据库接口依赖。

- 基于类的异常已合并到规范中。模块实现者可以通过子类化本规范中定义的异常类来扩展定义的异常布局。

发布后添加到 DB API 2.0 规范：

- 指定了核心功能集之外的额外可选 DB API 扩展。

## 开放问题

虽然版本 2.0 规范澄清了在 1.0 版本中留下开放的许多问题，但仍有一些剩余问题应在未来版本中解决：

- 为可用新结果集的情况定义.nextset() 的有用返回值。

- 集成 decimal 模块 Decimal 对象，用作无损货币和十进制交换格式。

## 脚注

[[1](#id1)] 作为指南，连接构造函数参数应实现为关键字参数，以便更直观地使用，并遵循以下参数顺序：

| 参数 | 含义 |
|------|------|
| dsn | 作为字符串的数据源名称 |
| user | 作为字符串的用户名（可选） |
| password | 作为字符串的密码（可选） |
| host | 主机名（可选） |
| database | 数据库名称（可选） |

例如，connect 可能如下所示：

```python
connect(dsn='myhost:MYDB', user='guido', password='234$')
```

另请参阅 [[13](#id13)] 关于此列表的计划未来添加。

[[2](#id2)] 模块实现者应优先选择 numeric、named 或 pyformat，因为这些格式提供更大的清晰度和灵活性。

[3](#id3) （1, 2, 3）如果数据库不支持方法所需的功能，则接口应在使用方法时引发异常。首选方法是不实现该方法，因此在请求方法时让 Python 生成 AttributeError。这允许程序员使用标准的 hasattr() 函数检查数据库功能。

对于某些动态配置的接口，要求动态使方法可用可能不合适。这些接口然后在调用方法时引发 NotSupportedError，以指示无法执行回滚。

[[4](#id4)] 数据库接口可以选择支持命名游标，通过允许字符串参数传递给方法。此功能不是规范的一部分，因为它使.fetch*() 方法的语义复杂化。

[[5](#id22)] 模块将使用参数对象的__getitem__方法将位置（整数）或名称（字符串）映射到参数值。这允许序列和映射都用作输入。术语"bound"指的是将输入值绑定到数据库执行缓冲区的进程。在实用术语中，这意味着输入值直接用作操作中的值。客户端不应被要求"转义"值以便可以使用——值应等于实际数据库值。

[[6](#id26)] 注意，接口可以使用数组和其他优化实现行获取。不保证调用此方法只会将相关游标向前移动一行。

[[7](#id14)] rowcount 属性可以编码为动态更新其值。这对于仅在第一次调用.fetch*() 方法后返回可用 rowcount 值的数据库很有用。

[[8](#id38)] 实现注意：Python C 扩展将必须在游标对象上实现 tp_iter 槽，而不是`.__iter__()` 方法。

[[9](#id13)] 术语"受影响的行数"通常指的是上次在数据库游标上运行的语句删除、更新或插入的行数。大多数数据库将返回相应 WHERE 子句找到的总行数。一些数据库对 UPDATE 使用不同的解释，仅返回被 UPDATE 更改的行数，即使语句的 WHERE 子句可能找到更多匹配行。数据库模块作者应尝试实现更常见的返回 WHERE 子句找到的总行数的解释，或清楚地记录.rowcount 属性的不同解释。

[[10](#id10)] （1, 2, 3, 4）在 Python 2 和此 PEP 的早期版本中，StandardError 用作所有 DB-API 异常的基类。由于 StandardError 在 Python 3 中被移除，针对 Python 3 的数据库模块应使用 Exception 作为基类。PEP 已更新为在整篇文本中使用 Exception，以避免混淆。更改不应影响现有模块或这些模块的用途，因为所有 DB-API 错误异常类仍然根植于 Error 或 Warning 类。

[[11](#id11)] （1, 2）在 DB-API 的未来修订版中，Warning 的基类可能会更改为内置的 Warning 类。在 1999 年编写 DB-API 2.0 时，Python 中的警告框架尚不存在。

[[12](#id39)] 实现 autocommit 属性的许多数据库模块将自动提交任何挂起的事务，然后进入自动提交模式。通常建议在更改 autocommit 设置之前显式.commit() 或.rollback() 事务，因为这跨数据库模块可移植。

[[13](#id13)] （1, 2）在 DB-API 的未来修订版中，我们将引入新方法.setautocommit(value)，允许设置自动提交模式，并使.autocommit 成为只读属性。此外，我们正在考虑向连接构造函数添加新标准关键字参数 autocommit。鼓励模块作者添加这些更改以准备此更改。
