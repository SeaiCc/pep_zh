# PEP 8000 – Python 语言治理方案概述

**元数据：**

- **作者**: Barry Warsaw <barry&#32;&#97;t&#32;python.org>
- **创建时间**: 24-Aug-2018

## 摘要

本 PEP 概述了 Guido 退休后新的 Python 语言治理模型的选择过程。
一旦治理模型选定，它将被编入 PEP 13。

以下是与治理模型选择过程相关的 PEP 列表。
8000 低位的 PEP 描述了选择治理模型的一般过程。

- **PEP 8001 - Python 治理投票流程**

  本 PEP 描述了新治理模型的投票如何进行。它概述了投票方法、时间表、参与标准，以及有资格投票者的明确名单。

- **PEP 8002 - 开源治理调查**

  将对类似开源和自由软件项目的治理模型进行调查，这些模型的摘要将在此 PEP 中概述。这些调查将作为此类项目如何成功治理的有用参考，并可能为 Python 自己的治理模型提供灵感。Python 是独特的，因此预计它将拥有自己独特的治理方式，而不是直接采用任何已调查的模型。

801X 系列的 PEP 描述了 Python 治理的实际提案。预计这些 PEP 将涵盖广泛的治理范围，而细节上的差异（如理事会的规模）将在同一个 PEP 中涵盖，而不是在可能导致选票分散的单独 PEP 中。

- **PEP 8010 - 技术领袖治理模型**

  本 PEP 提议延续单一技术项目领袖模型。也在范围内的是咨询委员会是否协助或支持 BDFL。本 PEP 不指定下一任 BDFL，也不指定此类咨询委员会的成员。关于这一点，请参阅 PEP 13。

- **PEP 8011 - 由三位 Python 专家领导的 Python 治理模型**

  本 PEP 描述了一种由三位 Python 专家（TOP）领导的新 Python 治理模型。它描述了 Trio 的角色和职责。本 PEP 不指定 Trio 的成员。关于这一点，请参阅 PEP 13。

- **PEP 8012 - 社区治理模型**

  这是一个占位符 PEP，用于基于共识和投票的新 Python 治理模型，没有集中单一领袖或治理委员会的角色。它描述了如何、何时以及为什么对影响 Python 语言的决策进行投票。它还描述了投票资格的标准。

- **PEP 8013 - 外部治理模型**

  本 PEP 描述了一种基于外部委员会的新 Python 治理模型，该委员会负责确保良好的流程。该委员会由核心开发团队选举产生，可以拒绝不够详细、未考虑所有受影响用户或不适合即将发布的提案。本 PEP 不指定此类委员会的成员。关于这一点，请参阅 PEP 13。

- **PEP 8014 - 公共治理模型**

  本 PEP 描述了一种基于长老委员会的新 Python 治理模型，这些长老负责确保 PEP 在被接受之前得到 Python 社区足够多数人的支持。与其他一些治理 PEP 不同，它明确不指定谁拥有投票权以及多数票的含义。相反，这是由长老委员会逐案决定的。

- **PEP 8015 - Python 社区的组织**

  本 PEP 正式化了 Python 社区的现有组织，并提出了 3 个主要变化：正式化现有的"Python 团队"概念；给予 Python 团队更多的自主权；用一个新的 3 人"Python 委员会"替代 BDFL（Guido van Rossum），该委员会的角色有限，主要决定如何批准（或拒绝）PEP。

- **PEP 8016 - 指导委员会模型**

  本 PEP 提出了一种围绕指导委员会的 Python 治理模型。该委员会拥有广泛的权力，但他们寻求尽可能少地行使这些权力；相反，他们使用这种权力来建立标准流程，就像其他 801x 系列 PEP 中提出的那样。这遵循了一般哲学，即将大变化分解为一系列可以独立审查的小变化：不是试图在一个 PEP 中完成所有事情，而是专注于为进一步的治理决策提供一个最小但坚实的基础。

最终选择之前可能会添加其他治理模型。
