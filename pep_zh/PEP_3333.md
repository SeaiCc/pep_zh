这是 PEP 333 的更新版本，稍作修改以改善 Python 3 下的可用性，并纳入几个长期存在的既定 WSGI 协议修正案。（其代码示例也已移植到 Python 3。）

然而，在 Python 3 下，您的应用程序或服务器还必须遵循下面标题为"关于字符串类型"和"Unicode 问题"的章节中概述的规则。

## 摘要

本文档指定了 Web 服务器与 Python Web 应用程序或框架之间的建议标准接口，以促进 Web 应用程序在各种 Web 服务器之间的可移植性。

## 原始理由和目标（来自 PEP 333）

Python 目前拥有各种各样的 Web 应用程序框架，例如 Zope、Quixote、Webware、SkunkWeb、PSO 和 Twisted Web——仅举几例 [1]。这种广泛的选择对于新的 Python 用户来说可能是一个问题，因为一般来说，他们对 Web 框架的选择将限制他们可用的 Web 服务器选择，反之亦然。

相比之下，虽然 Java 拥有同样多的 Web 应用程序框架，但 Java 的"servlet"API 使得使用任何 Java Web 应用程序框架编写的应用程序都可以在支持 servlet API 的任何 Web 服务器中运行。

在 Python 的 Web 服务器中提供并广泛使用这样的 API——无论这些服务器是用 Python 编写的（例如 Medusa）、嵌入 Python（例如 mod_python），还是通过网关协议调用 Python（例如 CGI、FastCGI 等）——将把框架选择与 Web 服务器选择分开，让用户自由选择合适的配对，同时让框架和服务器开发者专注于他们偏好的专业领域。

因此，本 PEP 提出了 Web 服务器与 Web 应用程序或框架之间简单通用的接口：Python Web Server Gateway Interface (WSGI)。

但仅仅存在 WSGI 规范并不能解决 Python Web 应用程序的服务器和框架的现有状态。服务器和框架的作者和维护者必须实际实现 WSGI 才能产生任何效果。

然而，由于现有的服务器或框架都不支持 WSGI，因此对于实现 WSGI 支持的作者来说，几乎没有立竿见影的回报。因此，WSGI 必须易于实现，以便作者在接口上的初始投资可以相对较低。

因此，在接口两侧的服务器和框架方面实现的简单性对于 WSGI 接口的实用性至关重要，因此是任何设计决策的主要标准。

然而，需要注意的是，对框架作者来说实现的简单性与对 Web 应用程序作者来说的易用性不是同一回事。WSGI 向框架作者呈现了一个绝对"无附加功能"的接口，因为响应对象和 cookie 处理等花哨的功能只会妨碍现有框架对这些问题的处理。再次强调，WSGI 的目标是促进现有服务器与应用程序或框架之间的轻松互连，而不是创建一个新的 Web 框架。

还要注意，这一目标排除了 WSGI 要求任何在已部署版本的 Python 中尚未可用的内容。因此，本规范不提议或不需要新的标准库模块，WSGI 中的任何内容也不需要大于 2.2.2 的 Python 版本。（不过，未来的 Python 版本在标准库提供的 Web 服务器中包含对此接口的支持是个好主意。）

除了便于现有和未来框架和服务器实现之外，创建请求预处理器、响应后处理器和其他基于 WSGI 的"中间件"组件也应该很容易，这些组件对其包含的服务器来说看起来像一个应用程序，同时对其包含的应用程序充当服务器。

如果中间件可以既简单又健壮，并且 WSGI 在服务器和框架中广泛可用，那么就有可能出现一种全新的 Python Web 应用程序框架：由松散耦合的 WSGI 中间件组件组成。事实上，现有框架的作者甚至可能选择重构他们框架的现有服务，以这种方式提供，变得更加像与 WSGI 一起使用的库，而不太像单体框架。然后这将允许应用程序开发人员为特定功能选择"最佳"组件，而不是必须承诺单个框架的所有优缺点。

当然，就本文撰写之时而言，那一天无疑还很遥远。与此同时，WSGI 的一个足够的短期目标是使任何框架可以与任何服务器一起使用。

最后，应该提到的是，当前版本的 WSGI 并未规定任何用于"部署"应用程序以供 Web 服务器或服务器网关使用的特定机制。目前，这必然由服务器或网关定义实现。在足够多的服务器和框架实现了 WSGI 以提供具有不同部署需求的现场经验之后，创建另一个描述 WSGI 服务器和应用程序框架部署标准的 PEP 可能是有意义的。

## 规范概述

WSGI 接口有两方面："服务器"或"网关"一侧，以及"应用程序"或"框架"一侧。服务器端调用应用程序端提供的可调用对象。如何提供该对象的具体细节由服务器或网关决定。可以假设某些服务器或网关将要求应用程序的部署者编写一个短脚本来创建服务器或网关的实例，并向其提供应用程序对象。其他服务器和网关可能使用配置文件或其他机制来指定应从何处导入应用程序对象，或以其他方式获取应用程序对象。

除了"纯"服务器/网关和应用程序/框架外，还可以创建实现本规范两侧的"中间件"组件。此类组件对其包含的服务器充当应用程序，对包含的应用程序充当服务器，可用于提供扩展 API、内容转换、导航和其他有用功能。

在本规范中，我们将使用术语"可调用"来表示"函数、方法、类或具有__call__ 方法的实例"。由实现可调用对象的服务器、网关或应用程序选择适合其需求的适当实现技术。相反，调用可调用对象的服务器、网关或应用程序必须不依赖于提供给它的可调用对象类型。可调用对象只能被调用，而不能被内省。

### 关于字符串类型

一般来说，HTTP 处理的是字节，这意味着本规范主要是关于处理字节。

然而，这些字节的内容通常具有某种文本解释，而在 Python 中，字符串是处理文本最方便的方式。

但在许多 Python 版本和实现中，字符串是 Unicode，而不是字节。这需要在可用 API 与 HTTP 上下文中字节和文本之间的正确转换之间进行仔细平衡……尤其是为了支持在不同 str 类型的 Python 实现之间移植代码。

因此，WSGI 定义了两种"字符串"：

- "原生"字符串（始终使用名为 str 的类型实现），用于请求/响应头和元数据

- "字节串"（在 Python 3 中使用 bytes 类型实现，在其他地方使用 str），用于请求和响应的主体（例如 POST/PUT 输入数据和 HTML 页面输出）。

但不要混淆：即使 Python 的 str 类型实际上是 Unicode"under the hood"，原生字符串的内容仍然必须可以通过 Latin-1 编码转换为字节！（详见本文档后面的 Unicode 问题部分。）

简而言之：在本文档中看到"string"一词时，它指的是"原生"字符串，即 str 类型的对象，无论它内部实现为字节还是 unicode。看到对"bytestring"的引用时，应理解为"Python 3 下类型为 bytes 的对象，或 Python 2 下类型为 str 的对象"。

因此，即使 HTTP 在某种意义上"真的只是字节"，使用 Python 的默认 str 类型也有许多 API 便利。

### 应用程序/框架端

应用程序对象只是一个接受两个参数的可调用对象。术语"对象"不应被误解为需要实际的对象实例：函数、方法、类或具有__call__ 方法的实例都可用于作为应用程序对象。应用程序对象必须能够被调用多次，因为几乎所有服务器/网关（CGI 除外）都会发出此类重复请求。

（注意：虽然我们称它为"应用程序"对象，但这不应被理解为应用程序开发人员将使用 WSGI 作为 Web 编程 API！假定应用程序开发人员将继续使用现有的高级框架服务来开发他们的应用程序。WSGI 是框架和服务器开发人员的工具，不打算直接支持应用程序开发人员。）

以下是两个示例应用程序对象；一个是函数，另一个是类：

```python
HELLO_WORLD = b"Hello world!\n"

def simple_app(environ, start_response):
    """Simplest possible application object"""
    status = '200 OK'
    response_headers = [('Content-type', 'text/plain')]
    start_response(status, response_headers)
    return [HELLO_WORLD]

class AppClass:
    """Produce the same output, but using a class

    (Note: 'AppClass' is the "application" here, so calling it
    returns an instance of 'AppClass', which is then the iterable
    return value of the "application callable" as required by
    the spec.

    If we wanted to use *instances* of 'AppClass' as application
    objects instead, we would have to implement a '__call__'
    method, which would be invoked to execute the application,
    and we would need to create an instance for use by the
    server or gateway.
    """

    def __init__(self, environ, start_response):
        self.environ = environ
        self.start = start_response

    def __iter__(self):
        status = '200 OK'
        response_headers = [('Content-type', 'text/plain')]
        self.start(status, response_headers)
        yield HELLO_WORLD
```

### 服务器/网关端

服务器或网关为从 HTTP 客户端接收到的每个指向应用程序的请求调用一次应用程序可调用对象。为说明起见，这里有一个简单的 CGI 网关，实现为接受应用程序对象的函数。请注意，这个简单的示例具有有限的错误处理，因为默认情况下未捕获的异常将被转储到 sys.stderr 并由 Web 服务器记录。

```python
import os, sys

enc, esc = sys.getfilesystemencoding(), 'surrogateescape'

def unicode_to_wsgi(u):
    # Convert an environment variable to a WSGI "bytes-as-unicode" string
    return u.encode(enc, esc).decode('iso-8859-1')

def wsgi_to_bytes(s):
    return s.encode('iso-8859-1')

def run_with_cgi(application):
    environ = {k: unicode_to_wsgi(v) for k,v in os.environ.items()}
    environ['wsgi.input'] = sys.stdin.buffer
    environ['wsgi.errors'] = sys.stderr
    environ['wsgi.version'] = (1, 0)
    environ['wsgi.multithread'] = False
    environ['wsgi.multiprocess'] = True
    environ['wsgi.run_once'] = True

    if environ.get('HTTPS', 'off') in ('on', '1'):
        environ['wsgi.url_scheme'] = 'https'
    else:
        environ['wsgi.url_scheme'] = 'http'

    headers_set = []
    headers_sent = []

    def write(data):
        out = sys.stdout.buffer

        if not headers_set:
            raise AssertionError("write() before start_response()")

        elif not headers_sent:
            # Before the first output, send the stored headers
            status, response_headers = headers_sent[:] = headers_set
            out.write(wsgi_to_bytes('Status: %s\r\n' % status))
            for header in response_headers:
                out.write(wsgi_to_bytes('%s: %s\r\n' % header))
            out.write(wsgi_to_bytes('\r\n'))

        out.write(data)
        out.flush()

    def start_response(status, response_headers, exc_info=None):
        if exc_info:
            try:
                if headers_sent:
                    # Re-raise original exception if headers sent
                    raise exc_info[1].with_traceback(exc_info[2])
            finally:
                exc_info = None # avoid dangling circular ref
        elif headers_set:
            raise AssertionError("Headers already set!")

        headers_set[:] = [status, response_headers]

        # Note: error checking on the headers should happen here,
        # *after* the headers are set. That way, if an error
        # occurs, start_response can only be re-called with
        # exc_info set.

        return write

    result = application(environ, start_response)
    try:
        for data in result:
            if data: # don't send headers until body appears
                write(data)
            if not headers_sent:
                write(b'') # send headers now if body was empty
    finally:
        if hasattr(result, 'close'):
            result.close()
```

### 中间件：同时扮演两方的组件

请注意，单个对象可以相对于某些应用程序扮演服务器的角色，同时相对于某些服务器充当应用程序。此类"中间件"组件可以执行以下功能：

- 根据目标 URL 将请求路由到不同的应用程序对象，并相应重写 environ。

- 允许多个应用程序或框架在同一进程中并排运行

- 通过在网络上传输请求和响应来进行负载均衡和远程处理

- 执行内容后处理，如应用 XSL 样式表

中间件的存在对"服务器/网关"和"应用程序/框架"接口双方都是透明的，不需要特殊支持。希望将中间件合并到应用程序中的用户只需将中间件组件提供给服务器，就像它是一个应用程序一样，并配置中间件组件来调用应用程序，就像中间件组件是一个服务器一样。当然，中间件包装的"应用程序"实际上可能是包装另一个应用程序的另一个中间件组件，依此类推，形成所谓的"中间件栈"。

在大多数情况下，中间件必须符合 WSGI 服务器和应用程序两侧的限制和要求。然而，在某些情况下，中间件的要求比"纯"服务器或应用程序更严格，这些点将在规范中注明。

以下是一个（半开玩笑的）中间件组件示例，它将 text/plain 响应转换为 Pig Latin，使用 Joe Strout 的 piglatin.py。（注意："真实的"中间件组件可能会使用更健壮的方式来检查内容类型，并且还应该检查内容编码。此外，这个简单的示例忽略了单词可能跨越块边界的可能性。）

```python
from piglatin import piglatin

class LatinIter:

    """Transform iterated output to piglatin, if it's okay to do so

    Note that the "okayness" can change until the application yields
    its first non-empty bytestring, so 'transform_ok' has to be a mutable
    truth value.
    """

    def __init__(self, result, transform_ok):
        if hasattr(result, 'close'):
            self.close = result.close
        self._next = iter(result).__next__
        self.transform_ok = transform_ok

    def __iter__(self):
        return self

    def __next__(self):
        data = self._next()
        if self.transform_ok:
            return piglatin(data) # call must be byte-safe on Py3
        else:
            return data

class Latinator:

    # by default, don't transform output
    transform = False

    def __init__(self, application):
        self.application = application

    def __call__(self, environ, start_response):

        transform_ok = []

        def start_latin(status, response_headers, exc_info=None):

            # Reset ok flag, in case this is a repeat call
            del transform_ok[:]

            for name, value in response_headers:
                if name.lower() == 'content-type' and value == 'text/plain':
                    transform_ok.append(True)
                # Strip content-length if present, else it'll be wrong
                response_headers = [(name, value)
                for name, value in response_headers
                if name.lower() != 'content-length'
                ]
                break

            write = start_response(status, response_headers, exc_info)

            if transform_ok:
                def write_latin(data):
                    write(piglatin(data)) # call must be byte-safe on Py3
                return write_latin
            else:
                return write

        return LatinIter(self.application(environ, start_latin), transform_ok)

# Run foo_app under a Latinator's control, using the example CGI gateway
from foo_app import foo_app
run_with_cgi(Latinator(foo_app))
```

## 规范细节

应用程序对象必须接受两个位置参数。为了说明起见，我们将其命名为 environ 和 start_response，但它们不必具有这些名称。服务器或网关必须使用位置参数（而不是关键字参数）调用应用程序对象。（例如，通过调用 result = application(environ, start_response)，如上所示。）

environ 参数是一个字典对象，包含 CGI 风格的环境变量。此对象必须是一个内置 Python 字典（不是子类、UserDict 或其他字典模拟），并且应用程序可以按其意愿修改字典。字典还必须包含某些 WSGI 必需的变量（在后面一节中描述），并且还可以包含服务器特定的扩展变量，其命名遵循下面描述的约定。

start_response 参数是一个可调用对象，接受两个必需的位置参数和一个可选参数。为了说明起见，我们将这些参数命名为 status、response_headers 和 exc_info，但它们不必具有这些名称，应用程序必须使用位置参数调用 start_response 可调用对象（例如 start_response(status, response_headers)）。

status 参数是形式为"999 Message here"的状态字符串，response_headers 是描述 HTTP 响应头的 (header_name, header_value) 元组列表。可选的 exc_info 参数在下面关于 The start_response() Callable 和 Error Handling 的部分中描述。它仅在应用程序捕获错误并尝试向浏览器显示错误消息时使用。

start_response 可调用对象必须返回一个 write(body_data) 可调用对象，该对象接受一个位置参数：要写入 HTTP 响应主体的一部分的字节串。（注意：write() 可调用对象仅提供以支持某些现有框架的命令式输出 API；如果可能，新应用程序或框架不应使用它。详见下面的 Buffering and Streaming 部分。）

当被服务器调用时，应用程序对象必须返回一个可迭代对象，生成零个或多个字节串。这可以通过多种方式实现，例如返回字节串列表，或者应用程序是生成字节串的生成器函数，或者应用程序是其实例可迭代的类。无论如何实现，应用程序对象必须始终返回一个生成零个或多个字节串的可迭代对象。

服务器或网关必须以无缓冲方式向客户端传输生成的字节串，在请求另一个字节串之前完成每个字节串的传输。（换句话说，应用程序应自行执行缓冲。详见下面的 Buffering and Streaming 部分，了解更多关于如何处理应用程序输出的信息。）

服务器或网关应将生成的字节串视为二进制字节序列：特别是，它应确保换行符不被更改。应用程序负责确保要写入的字节串处于适合客户端的格式。（服务器或网关可以应用 HTTP 传输编码，或执行其他转换以实现 HTTP 功能，如字节范围传输。详见下面的 Other HTTP Features。）

如果对 len(iterable) 的调用成功，服务器必须能够依赖结果是准确的。也就是说，如果应用程序返回的可迭代对象提供了工作的__len__() 方法，它必须返回准确的结果。（详见 Handling the Content-Length Header 部分，了解这通常如何使用。）

如果应用程序返回的可迭代对象具有 close() 方法，服务器或网关必须在该请求完成后调用该方法，无论请求是正常完成，还是由于迭代期间的应用程序错误或浏览器提前断开连接而提前终止。（close() 方法要求是为了支持应用程序的资源释放。此协议旨在补充 PEP 342 的生成器支持，以及其他具有 close() 方法的常见可迭代对象。）

返回生成器或其他自定义迭代器的应用程序不应假设整个迭代器都将被消费，因为它可能被服务器提前关闭。

（注意：应用程序必须在可迭代对象生成其第一个主体字节串之前调用 start_response() 可调用对象，以便服务器可以在任何主体内容之前发送头。然而，此调用可以由可迭代对象的第一次迭代执行，因此服务器绝不能假设 start_response() 在它们开始遍历可迭代对象之前已被调用。）

最后，除非它是特定于该服务器或网关的类型的实例，例如由 wsgi.file_wrapper 返回的"file wrapper"（参见 Optional Platform-Specific File Handling），否则服务器和网关绝不能直接使用应用程序返回的可迭代对象的任何其他属性。在一般情况下，只接受此处指定的属性或通过 PEP 234 迭代 API 访问的属性。

### environ 变量

environ 字典必须包含这些 CGI 环境变量，如 Common Gateway Interface 规范 [2] 所定义。以下变量必须存在，除非其值将为空字符串，在这种情况下它们可以被省略，除非下面另有说明。

| 变量 | 说明 |
|------|------|
| REQUEST_METHOD | HTTP 请求方法，如"GET"或"POST"。这绝不能为空字符串，因此始终需要。 |
| SCRIPT_NAME | 请求 URL 的"path"的初始部分，对应于应用程序对象，以便应用程序知道其虚拟"位置"。如果应用程序对应于服务器的"root"，这**可以**为空字符串。 |
| PATH_INFO | 请求 URL 的"path"的其余部分，指定应用程序内请求目标的虚拟"位置"。如果请求 URL 指向应用程序根目录且没有尾部斜杠，这**可以**为空字符串。 |
| QUERY_STRING | 请求 URL 中"?"之后的部分（如果有）。可以为空或不存在。 |
| CONTENT_TYPE | HTTP 请求中任何 Content-Type 字段的内容。可以为空或不存在。 |
| CONTENT_LENGTH | HTTP 请求中任何 Content-Length 字段的内容。可以为空或不存在。 |
| SERVER_NAME, SERVER_PORT | 当 HTTP_HOST 未设置时，这些变量可以组合来确定默认值。详见下面的 [URL Reconstruction](#url-reconstruction) 部分。SERVER_NAME 和 SERVER_PORT 是必需的字符串，绝不能为空。 |
| SERVER_PROTOCOL | 客户端用于发送请求的协议版本。通常这是类似"HTTP/1.0"或"HTTP/1.1"的内容，应用程序可以使用它来确定如何处理任何 HTTP 请求头。（这个变量应该叫 REQUEST_PROTOCOL，因为它表示请求中使用的协议，不一定是服务器响应中使用的协议。然而，为了与 CGI 兼容，我们必须保留现有名称。） |
| HTTP_ 变量 | 对应于客户端提供的 HTTP 请求头的变量（即，名称以"HTTP_"开头的变量）。这些变量的存在或不存在应与请求中适当 HTTP 头的存在或不存在相对应。 |

服务器或网关应尝试提供尽可能多的其他适用的 CGI 变量。此外，如果使用 SSL，服务器或网关还应提供尽可能多的 Apache SSL 环境变量 [5]，如 HTTPS=on 和 SSL_PROTOCOL。然而，需要注意的是，使用上述列出的 CGI 变量之外的任何 CGI 变量的应用程序必然无法移植到不支持相关扩展的 Web 服务器。（例如，不发布文件的 Web 服务器将无法提供有意义的 DOCUMENT_ROOT 或 PATH_TRANSLATED。）

符合 WSGI 的服务器或网关应记录它提供了哪些变量，以及它们的定义（如适用）。应用程序应检查它们所需的任何变量的存在，并在该变量不存在的情况下有备用计划。

注意：缺少的变量（如未发生身份验证时的 REMOTE_USER）应从 environ 字典中省略。还要注意，CGI 定义的变量如果是存在的话必须是原生字符串。任何 CGI 变量的值为任何非 str 类型都违反本规范。

除了 CGI 定义的变量外，environ 字典还可以包含任意的操作系统"环境变量"，并且必须包含以下 WSGI 定义的变量：

| 变量 | 值 |
|------|-----|
| wsgi.version | 元组 (1, 0)，表示 WSGI 版本 1.0。 |
| wsgi.url_scheme | 表示应用程序被调用的 URL 的"scheme"部分的字符串。通常，这将具有"http"或"https"的值，视情况而定。 |
| wsgi.input | 输入流（类文件对象），从中可以读取 HTTP 请求主体字节。（服务器或网关可以根据应用程序的请求按需执行读取，或者它可以预读取客户端的请求主体并将其缓冲在内存或磁盘上，或者使用任何其他技术来提供此类输入流，根据其偏好。） |
| wsgi.errors | 输出流（类文件对象），可以写入错误输出，用于在标准化且可能集中的位置记录程序或其他错误。这应该是一个"文本模式"流；即，应用程序应使用"\n"作为行结束符，并假设它将被服务器/网关转换为正确的行结束符。（在 str 类型为 unicode 的平台上，错误流应接受并记录任意 unicode 而不引发错误；然而，允许替换无法在流的编码中呈现的字符。）对于许多服务器，wsgi.errors 将是服务器的主错误日志。或者，这可能是 sys.stderr，或某种日志文件。服务器的文档应包含如何配置此或在哪里找到记录的输出的解释。服务器或网关可以向不同的应用程序提供不同的错误流，如果需要的话。 |
| wsgi.multithread | 如果应用程序对象可能同时被同一进程中的另一个线程调用，此值应评估为 true，否则应评估为 false。 |
| wsgi.multiprocess | 如果等效的应用程序对象可能同时被另一个进程调用，此值应评估为 true，否则应评估为 false。 |
| wsgi.run_once | 如果服务器或网关期望（但不保证！）应用程序在其包含进程的生存期内仅被调用一次，此值应评估为 true。通常，这只基于 CGI（或类似物）的网关为 true。 |

最后，environ 字典还可以包含服务器定义的变量。这些变量应仅使用小写字母、数字、点和下划线来命名，并且应以定义服务器或网关独有的名称为前缀。例如，mod_python 可能定义名称如 mod_python.some_variable 的变量。

#### 输入和错误流

服务器提供的输入和错误流必须支持以下方法：

| 方法 | 流 | 注释 |
|------|-----|------|
| read(size) | input | 1 |
| readline() | input | 1, 2 |
| readlines(hint) | input | 1, 3 |
| __iter__() | input |  |
| flush() | errors | 4 |
| write(str) | errors |  |
| writelines(seq) | errors |  |

每个方法的语义如 Python Library Reference 中所记录，除了上表中列出的这些注释：

- 服务器不需要读取超过客户端指定的 Content-Length，如果应用程序尝试读取超过该点，应模拟文件结束条件。应用程序不应尝试读取超过 CONTENT_LENGTH 变量指定的数据。

- 服务器应允许在不带参数的情况下调用 read()，并返回客户端输入流的剩余部分。

- 服务器应从任何尝试从空或耗尽的输入流读取的操作返回空字节串。

- 服务器应支持 readline() 的可选"size"参数，但正如在 WSGI 1.0 中一样，它们可以省略对此的支持。（在 WSGI 1.0 中，不支持 size 参数，理由是它可能复杂实现，并且在实践中不常用……但随后 cgi 模块开始使用它，因此实际服务器不得不开始支持它！）

- 注意 readlines() 的 hint 参数对调用者和实现者都是可选的。应用程序可以自由不提供它，服务器或网关可以自由忽略它。

- 由于错误流可能无法倒带，服务器和网关可以自由地立即转发写入操作，而不进行缓冲。在这种情况下，flush() 方法可以是无操作。然而，可移植的应用程序不能假设输出是无缓冲的或 flush() 是无操作。如果他们需要确保输出确实已被写入，他们必须调用 flush()。（例如，以最小化写入同一错误日志的多个进程的数据混杂。）

上表中列出的方法必须被所有符合本规范的服务器支持。符合本规范的应用程序不得使用输入或错误对象的任何其他方法或属性。特别是，应用程序不得尝试关闭这些流，即使它们拥有 close() 方法。

### The start_response() 可调用对象

传递给应用程序对象的第二个参数是形式为 start_response(status, response_headers, exc_info=None) 的可调用对象。（与所有 WSGI 可调用对象一样，参数必须按位置提供，而不是按关键字。）start_response 可调用对象用于开始 HTTP 响应，它必须返回一个 write(body_data) 可调用对象（见下面的 Buffering and Streaming 部分）。

status 参数是 HTTP"status"字符串，如"200 OK"或"404 Not Found"。也就是说，它是一个由 Status-Code 和 Reason-Phrase 组成的字符串，按此顺序并用单个空格分隔，没有周围的空格或其他字符。（详见 RFC 2616，Section 6.1.1。）字符串不得包含控制字符，并且不得以回车、换行或两者的组合终止。

response_headers 参数是 (header_name, header_value) 元组列表。它必须是一个 Python 列表；即 type(response_headers) 是 ListType，并且服务器可以按其意愿更改其内容。每个 header_name 必须是有效的 HTTP 头字段名（如 RFC 2616，Section 4.2 所定义），没有尾部冒号或其他标点。

每个 header_value 不得包含任何控制字符，包括嵌入或结尾的回车或换行符。（这些要求是为了最小化服务器、网关和中间响应处理器必须执行的任何解析的复杂性，这些处理器需要检查或修改响应头。）

一般来说，服务器或网关负责确保向客户端发送正确的头：如果应用程序省略了 HTTP（或其他相关规范）所需的头，服务器或网关必须添加它。例如，HTTP Date: 和 Server: 头通常由服务器或网关提供。

（对服务器/网关作者的提醒：HTTP 头名称不区分大小写，因此在检查应用程序提供的头时务必考虑这一点！）

禁止应用程序和中间件使用 HTTP/1.1"hop-by-hop"功能或头、HTTP/1.0 中的任何等效功能，或任何会影响客户端与 Web 服务器连接持久性的头。这些功能是实际 Web 服务器的专属领域，服务器或网关应认为应用程序尝试发送它们是致命错误，如果将它们提供给 start_response() 则应引发错误。（有关"hop-by-hop"功能和头的更多详细信息，请参见下面的 Other HTTP Features 部分。）

服务器应在调用 start_response 时检查头中的错误，以便可以在应用程序仍在运行时引发错误。

然而，start_response 可调用对象必须不实际传输响应头。相反，它必须将它们存储起来，以便服务器或网关仅在应用程序返回值的第一次迭代生成非空字节串后，或在应用程序第一次调用 write() 可调用对象时传输它们。换句话说，响应头不得在可用实际主体数据之前发送，或在应用程序返回的可迭代对象耗尽之前发送。（此规则唯一可能的例外是响应头明确包含零的 Content-Length。）

这种延迟响应头传输是为了确保缓冲和异步应用程序可以在最后一刻用错误输出替换其原始预期的输出。例如，如果错误发生在应用程序缓冲区内生成主体时，应用程序可能需要将响应状态从"200 OK"更改为"500 Internal Error"。

exc_info 参数（如果提供）必须是 Python sys.exc_info() 元组。此参数应由应用程序仅在 start_response 由错误处理器调用时提供。如果提供了 exc_info，并且尚未输出任何 HTTP 头，start_response 应该用新提供的头替换当前存储的 HTTP 响应头，从而允许应用程序在发生错误时"改变主意"。

然而，如果提供了 exc_info，并且 HTTP 头已经发送，start_response 必须引发错误，并应该使用 exc_info 元组重新引发。即：

```python
raise exc_info[1].with_traceback(exc_info[2])
```

这将重新引发应用程序捕获的异常，原则上应该中止应用程序。（在 HTTP 头已经发送之后，应用程序尝试向浏览器进行错误输出是不安全的。）如果应用程序使用 exc_info 调用 start_response，则不得捕获 start_response 引发的任何异常。相反，它应该允许此类异常传播回服务器或网关。详见下面的 Error Handling。

应用程序可以多次调用 start_response，当且仅当提供 exc_info 参数时。更确切地说，如果 start_response 已经在当前应用程序调用中被调用过，则在没有 exc_info 参数的情况下调用 start_response 是致命错误。这包括第一次调用 start_response 引发错误的情况。（参见上面的示例 CGI 网关以了解正确逻辑的说明。）

注意：实现 start_response 的服务器、网关或中间件应确保在函数执行期间之后不保留对 exc_info 参数的引用，以避免通过涉及的 traceback 和帧创建循环引用。实现此目的的最简单方法是类似：

```python
def start_response(status, response_headers, exc_info=None):
    if exc_info:
        try:
            # do stuff w/exc_info here
        finally:
            exc_info = None # Avoid circular ref.
```

#### 处理 Content-Length 头

如果应用程序提供 Content-Length 头，服务器不应向客户端传输超过头允许的字节数，并应在发送足够数据后停止迭代响应，或者如果应用程序尝试写入超过该点则引发错误。（当然，如果应用程序没有提供足够的数据来满足其声明的 Content-Length，服务器应关闭连接并记录或以其他方式报告错误。）

如果应用程序不提供 Content-Length 头，服务器或网关可以选择几种处理方法之一。最简单的是在响应完成后关闭客户端连接。

然而，在某些情况下，服务器或网关可能能够生成 Content-Length 头，或者至少避免需要关闭客户端连接。如果应用程序不调用 write() 可调用对象，并返回一个 len() 为 1 的可迭代对象，则服务器可以通过获取可迭代对象生成的第一个字节串的长度来自动确定 Content-Length。

此外，如果服务器和客户端都支持 HTTP/1.1"chunked encoding"，则服务器可以使用 chunked encoding 为每个 write() 调用或可迭代对象生成的字节串发送一个块，从而为每个块生成 Content-Length 头。这允许服务器保持客户端连接活跃（如果它希望这样做）。注意服务器在这样做时必须完全遵守 RFC 2616，否则回退到其他处理缺少 Content-Length 的策略之一。

（注意：应用程序和中间件不得对其输出应用任何类型的 Transfer-Encoding，如 chunking 或 gzipping；作为"hop-by-hop"操作，这些编码是实际 Web 服务器/网关的专属领域。详见下面的 Other HTTP Features。）

### Buffering and Streaming

一般来说，应用程序将通过缓冲其（适度大小的）输出并一次性发送所有内容来实现最佳吞吐量。这是现有框架（如 Zope）中的常见方法：输出缓冲在 StringIO 或类似对象中，然后一次性传输，连同响应头。

WSGI 中对应的方法是应用程序简单地返回一个单元素可迭代对象（如列表），其中包含作为单个字节串的响应主体。这是绝大多数应用程序函数的推荐方法，这些函数渲染可以轻松放入内存的 HTML 页面。

然而，对于大文件或 HTTP streaming 的专门用途（如 multipart"server push"），应用程序可能需要以更小的块提供输出（例如，以避免将大文件加载到内存中）。有时也会出现响应的部分可能耗时很长的情况，但提前发送其前面的部分会很有用。

在这些情况下，应用程序通常返回一个以块方式产生输出的迭代器（通常是生成器迭代器）。这些块可以打破以与 multipart 边界一致（用于"server push"），或者就在耗时任务（如读取磁盘文件的另一个块）之前。

WSGI 服务器、网关和中间件不得延迟任何块的传输；它们必须要么将块完全传输到客户端，要么保证它们将继续传输，即使应用程序正在生成其下一个块。服务器/网关或中间件可以通过以下三种方式之一提供此保证：

- 在将控制权返回给应用程序之前，将整个块发送到操作系统（并请求刷新任何 O/S 缓冲区），或者

- 使用不同的线程确保块在应用程序生成下一个块时继续传输。

- （仅限中间件）将整个块发送到其父网关/服务器

通过提供此保证，WSGI 允许应用程序确保传输不会在其输出数据的任意点停滞。这对于例如 multipart"server push"streaming 的正常运行至关重要，其中 multipart 边界之间的数据应完整传输到客户端。

#### 中间件处理块边界

为了更好地支持异步应用程序和服务器，中间件组件不得阻塞迭代以等待来自应用程序可迭代对象的多个值。如果中间件需要来自应用程序的更多数据才能产生任何输出，它必须生成一个空字节串。

换一种方式说此要求，中间件组件必须每次其底层应用程序生成一个值时至少生成一个值。如果中间件无法生成任何其他值，它必须生成一个空字节串。

此要求确保异步应用程序和服务器可以合谋减少同时运行给定数量应用程序实例所需的线程数。

还要注意，此要求意味着中间件必须在其底层应用程序返回可迭代对象时尽快返回可迭代对象。中间件也不得使用 write() 可调用对象来传输由底层应用程序生成的数据。中间件只能使用其父服务器的 write() 可调用对象来传输底层应用程序使用中间件提供的 write() 可调用对象发送的数据。

#### The write() 可调用对象

一些现有的应用程序框架 API 以不同于 WSGI 的方式支持无缓冲输出。具体地说，它们提供某种"write"函数或方法来写入无缓冲的数据块，或者它们提供缓冲的"write"函数和"flush"机制来刷新缓冲区。

不幸的是，除非使用线程或其他特殊机制，否则无法用 WSGI 的"iterable"应用程序返回值来实现此类 API。

因此，为了允许这些框架继续使用命令式 API，WSGI 包含一个特殊的 write() 可调用对象，由 start_response 可调用对象返回。

新的 WSGI 应用程序和框架如果可能应避免使用 write() 可调用对象。write() 可调用对象严格是一个支持命令式流式 API 的 hack。一般来说，应用程序应通过其返回的可迭代对象产生输出，因为这使得 Web 服务器能够在同一 Python 线程中交错其他任务，从而可能为整个服务器提供更好的吞吐量。

write() 可调用对象由 start_response() 可调用对象返回，它接受单个参数：要写入 HTTP 响应主体的一部分的字节串，该字节串的处理方式完全就像它是由输出可迭代对象生成的一样。换句话说，在 write() 返回之前，它必须保证传入的字节串要么完全发送到客户端，要么缓冲起来以便传输，同时应用程序继续进行。

应用程序必须返回一个可迭代对象，即使它使用 write() 来产生其响应主体的全部或部分。返回的可迭代对象可以为空（即不生成任何非空字节串），但如果它确实生成非空字节串，该输出必须由服务器或网关正常处理（即，必须立即发送或排队）。应用程序不得在其返回的可迭代对象内调用 write()，因此可迭代对象生成的任何字节串都在传递给 write() 的所有字节串发送到客户端之后传输。

### Unicode Issues

HTTP 不直接支持 Unicode，本接口也不支持。所有编码/解码必须由应用程序处理；传递给服务器或从服务器接收的所有字符串必须是 str 或 bytes 类型，绝不能是 unicode。在需要字符串对象的地方使用 unicode 对象的结果是未定义的。

还要注意，作为状态或响应头传递给 start_response() 的字符串必须遵循 RFC 2616 关于编码的规定。也就是说，它们必须是 ISO-8859-1 字符，或使用 RFC 2047 MIME 编码。

在 str 或 StringType 类型实际上是 Unicode 基础的 Python 平台上（例如 Jython、IronPython、Python 3 等），本规范中提到的所有"字符串"必须仅包含可以在 ISO-8859-1 编码中表示的码点（\u0000 到\u00FF，含）。应用程序提供包含任何其他 Unicode 字符或码点的字符串是致命错误。同样，服务器和网关不得向应用程序提供包含任何其他 Unicode 字符的字符串。

再次强调，本规范中提到的所有"字符串"对象必须是 str 或 StringType 类型，绝不能是 unicode 或 UnicodeType 类型。而且，即使给定平台允许在 str/StringType 对象中每个字符超过 8 位，也只能使用较低的 8 位，用于本规范中作为"字符串"提到的任何值。

对于本规范中作为"字节串"提到的值（即，从 wsgi.input 读取的值、传递给 write() 的值或由应用程序生成的值），在 Python 3 下值必须是 bytes 类型，在早期版本的 Python 中是 str 类型。

### Error Handling

一般来说，应用程序应该尝试捕获自己的内部错误，并在浏览器中显示有用的消息。（"helpful"在此上下文中的含义由应用程序决定。）

然而，为了显示这样的消息，应用程序必须尚未向浏览器实际发送任何数据，否则它可能会破坏响应。WSGI 因此提供了一种机制，允许应用程序发送其错误消息，或自动中止：start_response 的 exc_info 参数。以下是其用法的示例：

```python
try:
    # regular application code here
    status = "200 Froody"
    response_headers = [("content-type", "text/plain")]
    start_response(status, response_headers)
    return ["normal body goes here"]
except:
    # XXX should trap runtime issues like MemoryError, KeyboardInterrupt
    # in a separate handler before this bare 'except:'...
    status = "500 Oops"
    response_headers = [("content-type", "text/plain")]
    start_response(status, response_headers, sys.exc_info())
    return ["error body goes here"]
```

当发生异常时，如果没有输出被写入，对 start_response 的调用将正常返回，应用程序将返回要发送到浏览器的错误主体。然而，如果任何输出已经发送到浏览器，start_response 将重新引发提供的异常。此异常不应被应用程序捕获，因此应用程序将中止。然后服务器或网关可以捕获此（致命）异常并中止响应。

服务器应捕获并记录任何中止应用程序或其返回值迭代异常的异常。如果应用程序错误发生时已部分响应已写入浏览器，服务器或网关可以尝试将错误消息添加到输出中，如果已发送的头表示服务器知道如何干净修改的 text/* 内容类型。

一些中间件可能希望提供额外的异常处理服务，或拦截和替换应用程序错误消息。在这种情况下，中间件可以选择不重新引发传递给 start_response 的 exc_info，而是引发中间件特定的异常，或简单地在存储提供的参数后返回而不引发异常。然后这将导致应用程序返回其错误主体可迭代对象（或调用 write()），允许中间件捕获和修改错误输出。这些技术将有效，只要应用程序作者：

- 在开始错误响应时始终提供 exc_info

- 在提供 exc_info 时从不捕获 start_response 引发的错误

### HTTP 1.1 Expect/Continue

实现 HTTP 1.1 的服务器和网关必须为 HTTP 1.1 的"expect/continue"机制提供透明支持。这可以通过以下几种方式之一完成：

- 对包含 Expect: 100-continue 请求的请求立即响应"100 Continue"响应，并正常进行。

- 正常进行请求，但向应用程序提供 wsgi.input 流，如果/当应用程序首次尝试从输入流读取时将发送"100 Continue"响应。读取请求必须保持阻塞，直到客户端响应。

- 等待客户端决定服务器不支持 expect/continue，并自行发送请求主体。（这是次优的，不推荐。）

注意这些行为限制不适用于 HTTP 1.0 请求，或不指向应用程序对象的请求。有关 HTTP 1.1 Expect/Continue 的更多信息，请参见 RFC 2616，sections 8.2.3 和 10.1.1。

### Other HTTP Features

一般来说，服务器和网关应该"装傻"，让应用程序完全控制其输出。它们应该只进行不改变应用程序响应有效语义的更改。应用程序开发人员始终可以添加中间件组件来提供额外功能，因此服务器/网关开发人员应该在实现中保守。在某种意义上，服务器应将自己视为 HTTP"gateway server"，应用程序是 HTTP"origin server"。（详见 RFC 2616，section 1.3，了解这些术语的定义。）

然而，因为 WSGI 服务器和应用程序不通过 HTTP 通信，RFC 2616 所称的"hop-by-hop"头不适用于 WSGI 内部通信。WSGI 应用程序不得生成任何"hop-by-hop"头，尝试使用需要它们生成此类头的 HTTP 功能，或依赖 environ 字典中任何传入"hop-by-hop"头的内容。WSGI 服务器必须自行处理任何支持的传入"hop-by-hop"头，例如通过解码任何传入的 Transfer-Encoding，包括 chunked encoding（如果适用）。

将这些原则应用于各种 HTTP 功能，很明显服务器可以通过 If-None-Match 和 If-Modified-Since 请求头以及 Last-Modified 和 ETag 响应头来处理缓存验证。然而，不要求这样做，如果应用程序希望支持该功能，应该执行自己的缓存验证，因为服务器/网关不要求执行此类验证。

同样，服务器可以重新编码或传输编码应用程序的响应，但应用程序应该自行使用合适的内容编码，并且不得应用传输编码。如果客户端请求，服务器可以传输应用程序响应的字节范围，而应用程序本机不支持字节范围。然而，如果希望，应用程序应该自行执行此功能。

注意这些对应用程序的限制不一定意味着每个应用程序都必须重新实现每个 HTTP 功能；许多 HTTP 功能可以通过中间件组件部分或完全实现，从而使服务器和应用程序作者免于一遍又一遍地实现相同功能。

### Thread Support

线程支持，或缺乏线程支持，也是服务器依赖的。可以并行运行多个请求的服务器还应提供以单线程方式运行应用程序的选项，以便不是线程安全的应用程序或框架仍然可以与该服务器一起使用。

## Implementation/Application Notes

### Server Extension APIs

一些服务器作者可能希望暴露更高级的 API，应用程序或框架作者可以将其用于专门用途。例如，基于 mod_python 的网关可能希望将 Apache API 的一部分作为 WSGI 扩展暴露。

在最简单的情况下，这只需要定义一个 environ 变量，如 mod_python.some_api。但是，在许多情况下，中间件的可能存在可能使这变得困难。例如，提供对 environ 变量中找到的相同 HTTP 头访问的 API，如果 environ 被中间件修改，可能会返回不同的数据。

一般来说，任何重复、取代或绕过 WSGI 功能某些部分的扩展 API 都有与中间件组件不兼容的风险。服务器/网关开发人员不应假设没有人会使用中间件，因为一些框架开发人员特别打算组织或重组他们的框架以几乎完全作为各种类型的中间件运行。

因此，为了提供最大兼容性，提供替换某些 WSGI 功能的扩展 API 的服务器和网关必须设计这些 API，以便它们使用它们替换的 API 部分来调用。例如，访问 HTTP 请求头的扩展 API 必须要求应用程序传入其当前 environ，以便服务器/网关可以验证通过 API 访问的 HTTP 头未被中间件更改。如果扩展 API 无法保证它总是与 environ 关于 HTTP 头的内容一致，它必须拒绝向应用程序提供服务，例如通过引发错误、返回 None 而不是头集合，或对 API 适当的内容。

同样，如果扩展 API 提供写入响应数据或头的替代方法，它应该要求传入 start_response 可调用对象，然后应用程序才能获得扩展服务。如果传入的对象不是服务器/网关最初提供给应用程序的对象，它无法保证正确操作，必须拒绝向应用程序提供扩展服务。

这些指南也适用于向 environ 添加信息（如解析的 cookie、表单变量、会话等）的中间件。具体地说，此类中间件应将这些功能作为在 environ 上操作的函数提供，而不是简单地将值放入 environ。这有助于确保信息是从 environ 计算的，在中间件完成任何 URL 重写或其他 environ 修改之后。

服务器/网关和中间件开发人员遵循这些"安全扩展"规则非常重要，以避免未来的中间件开发人员被迫从 environ 中删除任何和所有扩展 API，以确保他们的调解不会被使用这些扩展的应用程序绕过！

### Application Configuration

本规范未定义服务器如何选择或获取要调用的应用程序。这些和其他配置选项是高度特定于服务器的事项。预计服务器/网关作者将记录如何配置服务器以执行特定应用程序对象，以及使用什么选项（如线程选项）。

另一方面，框架作者应记录如何创建包装其框架功能的应用程序对象。用户已选择服务器和应用程序框架，必须将两者连接在一起。然而，由于框架和服务器现在有通用接口，这应该只是一个机械问题，而不是每个新服务器/框架对的重大工程工作。

最后，一些应用程序、框架和中间件可能希望使用 environ 字典接收简单的字符串配置选项。服务器和网关应通过允许应用程序的部署者指定要放在 environ 中的名称 - 值对来支持此功能。在最简单的情况下，此支持可以仅由将所有操作系统提供的环境变量从 os.environ 复制到 environ 字典组成，因为部署者原则上可以在服务器外部配置这些，或者在 CGI 情况下，他们可能能够通过服务器的配置文件设置它们。

应用程序应尝试将此类必需变量保持在最低限度，因为并非所有服务器都支持轻松配置它们。当然，即使在最坏的情况下，部署应用程序的人员也可以创建脚本来提供必要的配置值：

```python
from the_app import application

def new_app(environ, start_response):
    environ['the_app.configval1'] = 'something'
    return application(environ, start_response)
```

但是，大多数现有应用程序和框架可能只需要 environ 中的单个配置值，以指示其应用程序或框架特定配置文件的位置。（当然，应用程序应该缓存此类配置，以避免每次调用时重新读取它。）

### URL Reconstruction

如果应用程序希望重构请求的完整 URL，它可以按照 Ian Bicking 提供的以下算法进行：

```python
from urllib.parse import quote
url = environ['wsgi.url_scheme']+'://'

if environ.get('HTTP_HOST'):
    url += environ['HTTP_HOST']
else:
    url += environ['SERVER_NAME']

    if environ['wsgi.url_scheme'] == 'https':
        if environ['SERVER_PORT'] != '443':
            url += ':' + environ['SERVER_PORT']
    else:
        if environ['SERVER_PORT'] != '80':
            url += ':' + environ['SERVER_PORT']

url += quote(environ.get('SCRIPT_NAME', ''))
url += quote(environ.get('PATH_INFO', ''))
if environ.get('QUERY_STRING'):
    url += '?' + environ['QUERY_STRING']
```

注意，这样重构的 URL 可能不精确等于客户端请求的 URI。例如，服务器重写规则可能已经修改了客户端最初请求的 URL，以将其置于规范形式。

### Supporting Older (<2.2) Versions of Python

一些服务器、网关或应用程序可能希望支持较旧 (<2.2) 版本的 Python。如果 Jython 是目标平台，这一点尤其重要，因为就本文撰写之时而言，生产就绪版本的 Jython 2.2 尚未可用。

对于服务器和网关，这相对直接：针对 2.2 之前版本的 Python 的服务器和网关只需限制自己仅使用标准"for"循环来迭代应用程序返回的任何可迭代对象。这是确保与 2.2 之前的迭代器协议（下面进一步讨论）和"今天的"迭代器协议（见 PEP 234）源代码级兼容性的唯一方法。

（注意，此技术必然仅适用于用 Python 编写的服务器、网关或中间件。如何从其他语言正确使用迭代器协议的讨论超出了本 PEP 的范围。）

对于应用程序，支持 2.2 之前的 Python 版本稍微复杂一些：

- 您不能返回文件对象并期望它作为可迭代对象工作，因为在 Python 2.2 之前，文件不可迭代。（一般来说，您无论如何都不应该这样做，因为它大多数时候表现会很差！）使用 wsgi.file_wrapper 或应用程序特定的文件包装器类。（详见 Optional Platform-Specific File Handling 了解更多关于 wsgi.file_wrapper 的信息，以及一个您可以用来将文件包装为可迭代对象的示例类。）

- 如果您返回自定义可迭代对象，它必须实现 2.2 之前的迭代器协议。也就是说，提供接受整数键的__getitem__ 方法，并在耗尽时引发 IndexError。（注意内置序列类型也是可接受的，因为它们也实现此协议。）

最后，希望支持 2.2 之前版本的 Python 的中间件，并迭代应用程序返回值或自身返回可迭代对象（或两者），必须遵循上述适当建议。

（注意：不言而喻，要支持 2.2 之前的 Python 版本，任何服务器、网关、应用程序或中间件还必须仅使用目标版本中可用的语言功能，使用 1 和 0 而不是 True 和 False 等。）

### Optional Platform-Specific File Handling

一些操作环境提供特殊的高性能文件传输设施，如 Unix sendfile() 调用。服务器和网关可以通过 environ 中的可选 wsgi.file_wrapper 键暴露此功能。应用程序可以使用此"file wrapper"将文件或类文件对象转换为可迭代对象，然后返回它，例如：

```python
if 'wsgi.file_wrapper' in environ:
    return environ['wsgi.file_wrapper'](filelike, block_size)
else:
    return iter(lambda: filelike.read(block_size), '')
```

如果服务器或网关提供 wsgi.file_wrapper，它必须是一个可调用对象，接受一个必需的位置参数和一个可选的位置参数。第一个参数是要发送的类文件对象，第二个参数是可选的块大小"建议"（服务器/网关不必使用）。可调用对象必须返回一个可迭代对象，并且不得执行任何数据传输，直到并且除非服务器/网关实际上从应用程序接收可迭代对象作为返回值。（否则将阻止中间件能够解释或覆盖响应数据。）

要被考虑为"类文件"，应用程序提供的对象必须具有接受可选 size 参数的 read() 方法。它可能具有 close() 方法，如果是这样，wsgi.file_wrapper 返回的可迭代对象必须具有调用原始类文件对象 close() 方法的 close() 方法。如果"类文件"对象具有任何其他与 Python 内置文件对象名称匹配的方法或属性（例如 fileno()），wsgi.file_wrapper 可以假设这些方法或属性具有与内置文件对象相同的语义。

任何平台特定文件处理的实际实现必须在应用程序返回之后发生，服务器或网关检查是否返回了包装器对象。（再次，由于中间件、错误处理器等的存在，不保证创建的任何包装器实际上将被使用。）

除了 close() 的处理外，从应用程序返回文件包装器的语义应该与应用程序返回 iter(filelike.read, '') 相同。换句话说，传输应该从传输开始时"文件"内的当前位置开始，并继续直到到达末尾，或直到写入 Content-Length 字节。（如果应用程序不提供 Content-Length，服务器可以使用其对底层文件实现的知识从文件生成一个。）

当然，平台特定的文件传输 API 通常不接受任意的"类文件"对象。因此，wsgi.file_wrapper 必须内省提供的对象以查找诸如 fileno()（类 Unix 操作系统）或 java.nio.FileChannel（在 Jython 下）等内容，以确定类文件对象是否适合使用它支持的平台特定 API。

注意，即使对象不适合平台 API，wsgi.file_wrapper 仍然必须返回包装 read() 和 close() 的可迭代对象，以便使用文件包装器的应用程序可以在平台之间移植。这是一个简单的平台无关文件包装器类，适用于旧版（2.2 之前）和新版 Python：

```python
class FileWrapper:

    def __init__(self, filelike, blksize=8192):
        self.filelike = filelike
        self.blksize = blksize
        if hasattr(filelike, 'close'):
            self.close = filelike.close

    def __getitem__(self, key):
        data = self.filelike.read(self.blksize)
        if data:
            return data
        raise IndexError
```

这是来自使用它来访问平台特定 API 的服务器/网关的代码片段：

```python
environ['wsgi.file_wrapper'] = FileWrapper
result = application(environ, start_response)

try:
    if isinstance(result, FileWrapper):
        # check if result.filelike is usable w/platform-specific
        # API, and if so, use that API to transmit the result.
        # If not, fall through to normal iterable handling
        # loop below.

    for data in result:
        # etc.

finally:
    if hasattr(result, 'close'):
        result.close()
```

## Questions and Answers

**为什么 environ 必须是字典？使用子类有什么问题吗？**

要求字典的理由是为了最大化服务器之间的可移植性。替代方案是定义字典某些方法的子集作为标准和可移植接口。然而，在实践中，大多数服务器可能会发现字典足以满足其需求，因此框架作者将期望全套字典功能可用，因为它们通常存在。但是，如果某些服务器选择不使用字典，那么尽管该服务器"符合"规范，也会出现互操作性问题。因此，使字典成为强制性的简化了规范并保证了互操作性。

注意，这并不阻止服务器或框架开发人员作为 environ 字典内的自定义变量提供专门服务。这是提供任何此类增值服务的方法。

**- 为什么你可以调用 write() 并生成字节串/返回可迭代对象？我们不应该只选择一种方式吗？**

如果我们只支持迭代方法，那么假设"push"可用性的当前框架会受到影响。但是，如果我们只支持通过 write() 推送，那么服务器性能对于例如大文件的传输会受到影响（如果工作线程在发送所有输出之前不能开始新请求的工作）。因此，这种折衷允许应用程序框架支持两种方法，如适当，但对服务器实现者的负担仅比仅推送方法多一点。

**- close() 的作用是什么？**

当在应用程序对象执行期间进行写入时，应用程序可以使用 try/finally 块确保资源被释放。但是，如果应用程序返回可迭代对象，任何使用的资源直到可迭代对象被垃圾回收才释放。close() 惯用允许应用程序在请求结束时释放关键资源，并且它向前兼容 PEP 325 提出的生成器中的 try/finally 支持。

**- 为什么这个接口这么低级？我想要功能 X！（例如 cookie、会话、持久性、…）**

这不是 Yet Another Python Web Framework。它只是框架与 Web 服务器通信的方式，反之亦然。如果您想要这些功能，您需要选择一个提供您想要功能的 Web 框架。如果该框架让您创建 WSGI 应用程序，您应该能够在大多数支持 WSGI 的服务器中运行它。此外，一些 WSGI 服务器可能通过其 environ 字典中提供的对象提供额外服务；详见相关服务器文档。（当然，使用此类扩展的应用程序将无法移植到其他基于 WSGI 的服务器。）

**- 为什么使用 CGI 变量而不是古老的好 HTTP 头？为什么将它们与 WSGI 定义的变量混合？**

许多现有 Web 框架 heavily built upon CGI spec，现有 Web 服务器知道如何生成 CGI 变量。相比之下，表示传入 HTTP 信息的替代方式是分散的，缺乏市场份额。因此，使用 CGI"standard"似乎是利用现有实现的好方法。至于将它们与 WSGI 变量混合，分离它们将只需要传递两个字典参数，而不提供任何实际好处。

**- 关于状态字符串怎么样？我们不能只使用数字，传入 200 而不是"200 OK"吗？**

这样做会使服务器或网关复杂化，要求它们具有数字状态和相应消息的表。相比之下，应用程序或框架作者键入伴随特定响应代码的额外文本很容易，现有框架通常已经包含包含所需消息的表。因此，总的来说，让应用程序/框架负责而不是服务器或网关似乎更好。

**- 为什么 wsgi.run_once 不保证只运行一次应用程序？**

因为它仅仅是对应用程序的建议，应该"rig for infrequent running"。这是针对具有多种操作模式（用于缓存、会话等）的应用程序框架。在"multiple run"模式下，此类框架可能预加载缓存，并且在每次请求后不写入例如日志或会话数据到磁盘。在"single run"模式下，此类框架避免预加载并在每次请求后刷新所有必要写入。

然而，为了测试应用程序或框架以验证后者的正确操作，可能需要（或至少方便）多次调用它。因此，应用程序不应假设它肯定不会再次运行，仅仅因为它被调用时 wsgi.run_once 设置为 True。

**- 功能 X（字典、可调用对象等）在应用程序代码中使用很丑陋；我们为什么不使用对象？**

WSGI 的所有这些实现选择特别旨在将功能彼此解耦；将这些功能重新组合到封装对象中使编写服务器或网关稍微困难一些，并且编写仅替换或修改整体功能小部分的中间件难一个数量级。

本质上，中间件希望具有"Chain of Responsibility"模式，使其可以充当某些函数的"handler"，同时允许其他函数保持不变。如果接口要保持可扩展，这用普通 Python 对象很难做到。例如，必须使用__getattr__ 或__getattribute__ 覆盖，以确保扩展（如未来 WSGI 版本定义的属性）被传递。

这种类型的代码 notoriously difficult to get 100% correct，很少有人想自己编写它。他们将复制其他人的实现，但当他们复制的人纠正另一个边缘情况时，未能更新它们。

此外，这个必要的样板代码将是纯 excise，中间件开发人员支付以支持框架开发人员稍微漂亮的 API 的开发者税。但是，应用程序框架开发人员通常只会更新一个框架以支持 WSGI，并且在其框架的非常有限的部分中。这很可能是他们的第一个（也许是唯一一个）WSGI 实现，因此他们可能会准备好实现此规范。因此，使 API"prettier"与对象属性等的努力可能会浪费于此受众。

我们鼓励那些想要更漂亮（或其他改进）的 WSGI 接口用于直接 Web 应用程序编程（相对于 Web 框架开发）的人开发包装 WSGI 以便应用程序开发人员方便使用的 API 或框架。这样，WSGI 可以保持对服务器和中间件作者方便的低级，同时对应用程序开发人员不"ugly"。

## Proposed/Under Discussion

这些项目当前在 Web-SIG 和其他地方讨论中，或在 PEP 作者的"to-do"列表中：

- wsgi.input 应该是迭代器而不是文件吗？这将有助于异步应用程序和 chunked-encoding 输入流。

- 正在讨论暂停应用程序输出迭代直到输入可用或回调发生的可选扩展。

- 添加关于同步与异步应用程序和服务器的部分，相关的线程模型，以及这些领域的问题/设计目标。
