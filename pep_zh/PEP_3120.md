# PEP 3120 – 使用 UTF-8 作为默认源编码

**元数据:**

- **作者**: Martin von Löwis <martin&#32;&#97;t&#32;v.loewis.de>
- **创建日期**: 15-Apr-2007
- **Python 版本**: 3.0

## 规范

本 PEP 提议将默认源编码从 ASCII 更改为 UTF-8。对替代源编码的支持（PEP 263）继续存在；显式的编码声明优先于默认设置。

## 历史简述

在 Python 1 中，源编码未指定，除了源编码必须是系统基本执行字符集（即，在大多数系统上是 ASCII 超集）的超集。源编码仅与词法本身相关（表示关键字、标识符、标点符号、换行符等的字母的字节）。字符串文字的内容从源文件逐字复制。

在 Python 2.0 中，作为引入 Unicode 的副作用，源编码更改为 Latin-1。对于 Unicode 字符串文字，字符仍然从源文件逐字复制，但在字符基础上加宽。由于 Unicode 对代码点给出了固定解释，此算法有效地固定了源编码，至少对于在 Unicode 文字中包含非 ASCII 字符的文件如此。

PEP 263 确定了您只能在 Unicode 文字中使用也在 Latin-1 中的那些 Unicode 字符的问题，并引入了声明源编码的语法。如果没有给出源编码，默认应该是 ASCII。为了与 Python 2.0 和 2.1 兼容，文件在过渡期内被解释为 Latin-1。此过渡在 Python 2.5 结束时结束，如果遇到非 ASCII 字符且未声明源编码，则给出错误。

## 理由

有了 PEP 263，在 Python 文件中使用任意非 ASCII 字符是可能的，但很繁琐。必须显式添加编码声明。即使某些编辑器（如 IDLE 和 Emacs）支持 PEP 263 的声明，许多编辑器仍然不支持（也永远不会支持）；用户必须逐个文件显式调整编辑器假定的编码。

当默认编码更改为 UTF-8 时，向 Python 文件添加非 ASCII 文本变得更加容易和可移植：在某些系统上，编辑器在保存文本时会自动选择 UTF-8（例如，在使用 UTF-8 的区域设置的 Unix 系统上）。在其他系统上，编辑器在读取文件时会猜测编码，UTF-8 很容易猜测。还有其他编辑器支持将默认编码与文件扩展名关联，允许用户将 `.py` 与 UTF-8 关联。

对于 Python 2，使用非 UTF-8 编码的一个重要原因是字节字符串文字在运行时处于源编码中，允许它们原样输出到文件或呈现给用户。对于 Python 3，所有字符串都将是 Unicode 字符串，因此源的原始编码在运行时不会有任何影响。

## 实现

需要更改解析器以在没有指定源编码的情况下接受字节 > 127；与其给出错误，不如需要检查字节是否为格式良好的 UTF-8（不需要解码，因为解析器无论如何都将所有源代码转换为 UTF-8）。

需要更改 IDLE 以使用 UTF-8 作为默认编码。
