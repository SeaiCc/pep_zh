# PEP 3137 – 不可变字节和可变缓冲区

**元数据：**

- **作者**：Guido van Rossum <guido&#32;&#97;t&#32;python.org>
- **创建时间**：2007 年 9 月 26 日
- **Python 版本**：3.0
- **发布历史**：2007 年 9 月 26 日，2007 年 9 月 30 日

## 介绍

在发布带有可变字节类型的 Python 3.0a1 之后，压力骤增，需要添加一种表示不可变字节的方法。Gregory P. Smith 提出了一个补丁，允许通过请求使用 PEP 3118 的新缓冲区 API 锁定数据，使字节对象临时不可变。这对我来说似乎不是正确的方法。

Jeffrey Yasskin 在 Adam Hupp 的帮助下，随后准备了一个补丁，使字节类型不可变（通过粗略地移除所有可变 API），并修复了测试套件中的问题。这表明，除了从小块构建返回值的代码外，并没有那么多地方依赖于字节的可变性。

考虑到后果，注意到使用 array 模块作为可变字节类型的替代品远非理想，并回忆起 Talin 之前提出的建议，我提出了同时拥有可变和不可变字节类型的建议。（这以前曾被提出过，但在看到 Jeffrey 的补丁的证据之前，我不愿意接受这个建议。）

此外，一个可能的实现策略变得清晰：使用旧的 PyString 实现，简化以移除区域设置支持以及与 Unicode 的隐式转换，作为不可变字节类型，并保留新的 PyBytes 实现作为可变字节类型。

随后的讨论清楚地表明，这个想法是受欢迎的，但需要更精确地指定。因此有了这个 PEP。

## 优势

拥有不可变字节类型的一个优势是代码对象可以使用它。它还使得能够有效地使用字节作为键创建哈希表；这在解析基于表示文本的字节的协议（如 HTTP 或 SMTP）时可能很有用。

使用新设计移植操作 Python 2.x 中二进制数据（或编码文本）的代码将比使用带有可变字节的原始 3.0 设计更容易；只需将 str 替换为 bytes，并将'…'字面量更改为 b'…'字面量。

## 命名

我建议在 Python 级别使用以下类型名称：

- bytes 是一个不可变的字节数组（PyString）

- bytearray 是一个可变的字节数组（PyBytes）

- memoryview 是另一个对象上的字节视图（PyMemory）

旧类型 buffer 与新类型 memoryview（由 PEP 3118 引入）如此相似，以至于它是多余的。本 PEP 的其余部分不讨论 memoryview 的功能；这里只是提到它，以证明摆脱旧的 buffer 类型是合理的。（本 PEP 的早期版本建议将 buffer 作为 PyBytes 的新名称；最终，鉴于 buffer 一词的许多其他用途，这个名称被认为令人困惑。）

虽然最终更改 C API 名称是有意义的，但本 PEP 保留了旧的 C API 名称，这对所有的人来说都应该很熟悉。

### 摘要

这是一个简单的 ASCII 艺术表，总结了各种 Python 版本中的类型名称：

```python
+--------------+-------------+------------+--------------------------+
| C name       | 2.x    repr | 3.0a1 repr | 3.0a2               repr |
+--------------+-------------+------------+--------------------------+
| PyUnicode    | unicode u'' | str     '' | str                   '' |
| PyString     | str      '' | str8   s'' | bytes                b'' |
| PyBytes      | N/A         | bytes  b'' | bytearray bytearray(b'') |
| PyBuffer     | buffer      | buffer     | N/A                      |
| PyMemoryView | N/A         | memoryview | memoryview         <...> |
+--------------+-------------+------------+--------------------------+
```

## 字面量表示法

Python 3.0a1 中引入的 b'…'表示法返回一个不可变的字节对象，无论使用什么变体。要创建可变的字节数组，使用 bytearray(b'…') 或 bytearray([…])。后一种形式接受一个在 range(256) 范围内的整数列表。

## 功能

### PEP 3118 缓冲区 API

bytes 和 bytearray 都实现了 PEP 3118 缓冲区 API。bytes 类型仅实现只读请求；bytearray 类型还允许可写和数据锁定请求。元素数据类型始终是'B'（即无符号字节）。

### 构造函数

有四种形式的构造函数，适用于 bytes 和 bytearray：

- bytes(<bytes>)、bytes(<bytearray>)、bytearray(<bytes>)、bytearray(<bytearray>)：简单的复制构造函数，注意 bytes(<bytes>) 可能会返回其（不可变的）参数，但 bytearray(<bytearray>) 总是创建副本。

- bytes(<str>, <encoding>[, <errors>])、bytearray(<str>, <encoding>[, <errors>])：编码文本字符串。注意 str.encode() 方法返回一个不可变的字节对象。<encoding> 参数是必需的；<errors> 是可选的。<encoding> 和 <errors>（如果给出）必须是 str 实例。

- bytes(<memory view>)、bytearray(<memory view>)：从实现 PEP 3118 缓冲区 API 的任何内容构建 bytes 或 bytearray 对象。

- bytes(<iterable of ints>)、bytearray(<iterable of ints>)：从 range(256) 范围内的整数流构建 bytes 或 bytearray 对象。

- bytes(<int>)、bytearray(<int>)：构建给定长度的零初始化的 bytes 或 bytearray 对象。

### 比较

bytes 和 bytearray 类型可以相互比较和排序，例如 b'abc' == bytearray(b'abc') < b'abd'。

将任一类型与 str 对象进行相等性比较，无论任一操作数的内容如何，都返回 False。与 str 的顺序比较会引发 TypeError。这都符合不兼容类型之间比较和排序的标准规则。

（注意：在 Python 3.0a1 中，比较 bytes 实例与 str 实例会引发 TypeError，前提是这将更快地捕获偶尔的错误，特别是在从 Python 2.x 移植的代码中。然而，python-3000 列表上的一次长时间讨论指出了许多问题，这显然是一个糟糕的主意，无论本 PEP 其余部分的命运如何，都将在 3.0a2 中回滚。）

### 切片

切片 bytes 对象返回 bytes 对象。切片 bytearray 对象返回 bytearray 对象。

对 bytearray 对象的切片赋值接受实现 PEP 3118 缓冲区 API 的任何内容，或 range(256) 范围内的整数可迭代对象。

### 索引

索引 bytes 和 bytearray 返回小整数（如 3.0a1 中的 bytes 类型，以及列表或 array.array('B')）。

赋值给 bytearray 对象的项接受 range(256) 范围内的整数。（要从字节序列赋值，请使用切片赋值。）

### Str() 和 Repr()

str() 和 repr() 函数对这些对象返回相同的内容。bytes 对象的 repr() 返回 b'…'风格的字面量。bytearray 的 repr() 返回"bytearray(b'…')"形式的字符串。

### 运算符

以下运算符由 bytes 和 bytearray 类型实现，除非另有说明：

- b1 + b2：连接。对于混合 bytes/bytearray 操作数，返回类型是第一个参数的类型（在考虑 += 的工作原理之前，这似乎是任意的）。

- b1 += b2：如果 b1 是 bytearray 对象，则修改 b1。

- b * n, n * b：重复；n 必须是整数。

- b *= n：如果 b 是 bytearray 对象，则修改 b。

- b1 in b2, b1 not in b2：子串测试；b1 可以是实现 PEP 3118 缓冲区 API 的任何对象。

- i in b, i not in b：单字节成员测试；i 必须是整数（如果它是长度为 1 的字节数组，则被视为子串测试，结果相同）。

- len(b)：字节数。

- hash(b)：哈希值；仅由 bytes 类型实现。

注意，% 运算符未实现。它似乎不值得这么复杂。

### 方法

以下方法由 bytes 和 bytearray 实现，具有相似的语义。它们接受实现 PEP 3118 缓冲区 API 的任何内容作为字节参数，并返回调用其方法的对象的相同类型（"self"）：

```python
.capitalize(), .center(), .count(), .decode(), .endswith(),
.expandtabs(), .find(), .index(), .isalnum(), .isalpha(), .isdigit(),
.islower(), .isspace(), .istitle(), .isupper(), .join(), .ljust(),
.lower(), .lstrip(), .partition(), .replace(), .rfind(), .rindex(),
.rjust(), .rpartition(), .rsplit(), .rstrip(), .split(),
.splitlines(), .startswith(), .strip(), .swapcase(), .title(),
.translate(), .upper(), .zfill()
```

这正是 Python 2.x 中 str 类型存在的方法集，不包括 .encode()。签名和语义也相同。但是，每当使用字符类（如字母、空白、小写）时，都使用这些类的 ASCII 定义。（Python 2.x str 类型使用当前区域设置的定义，可通过 locale 模块设置。）.encode() 方法被省略，因为 Python 3000 中编码和解码的更严格定义：编码总是接受 Unicode 字符串并返回字节序列，解码总是接受字节序列并返回 Unicode 字符串。

此外，两种类型都实现了类方法 .fromhex()，它从包含十六进制值的字符串（字节之间可以有空格或没有空格）构建对象。

bytearray 类型实现了 MutableSequence ABC 的这些额外方法（参见 PEP 3119）：

> .extend()、.insert()、.append()、.reverse()、.pop()、.remove()。

### 字节和 Str 类型

与 Python 3.0a1 中的 bytes 类型一样，与 Python 2.x 中 str 和 unicode 之间的关系不同，尝试在不指定编码的情况下混合 bytes（或 bytearray）对象和 str 对象将引发 TypeError 异常。（但是，比较 bytes/bytearray 和 str 对象的相等性将简单地返回 False；参见上面的比较部分。）

bytes 或 bytearray 对象与 str 对象之间的转换必须始终是显式的，使用编码。有两个等效的 API：str(b, <encoding>[, <errors>]) 等同于 b.decode(<encoding>[, <errors>])，bytes(s, <encoding>[, <errors>]) 等同于 s.encode(<encoding>[, <errors>])。

有一个例外：我们可以通过编写 str(b) 将 bytes（或 bytearray）转换为 str，而无需指定编码。这产生的结果与 repr(b) 相同。这个例外是必要的，因为有一般的承诺可以打印任何对象，而打印只是转换为 str 的特殊情况。但是，没有承诺打印字节对象会将各个字节解释为字符（与 Python 2.x 不同）。

str 类型当前实现了 PEP 3118 缓冲区 API。虽然这可能偶尔方便，但也可能令人困惑，因为通过缓冲区 API 访问的字节表示平台依赖的编码：取决于平台的字节序和编译时配置选项，编码可能是 UTF-16-BE、UTF-16-LE、UTF-32-BE 或 UTF-32-LE。更糟糕的是，str 类型的不同实现可能会完全改变字节表示，例如改为 UTF-8，甚至使其根本无法作为连续的字节数组访问。因此，PEP 3118 缓冲区 API 将从 str 类型中移除。

### 基字符串类型

basestring 类型将从语言中移除。过去说 isinstance(x, basestring) 的代码应更改为使用 isinstance(x, str)。
