# PEP 428 – pathlib 模块——面向对象的文件系统路径

**元数据：**

- **作者**: Antoine Pitrou <solipsis&#32;&#97;t&#32;pitrou.net>
- **创建日期**: 2012 年 7 月 30 日
- **Python 版本**: 3.4

## 摘要

本 PEP 提议将一个第三方模块 pathlib 纳入标准库。纳入提议采用临时标签，如 PEP 411 所述。因此，API 更改可以在 PEP 过程中进行，或在标准库接受后进行（直到删除临时标签）。

该库的目标是提供一个简单的类层次结构来处理文件系统路径以及用户对其执行的常见操作。

## 相关工作

面向对象的文件系统路径 API 已经在 PEP 355 中提出并被拒绝。野外存在几种面向对象文件系统路径思想的第三方实现：

- Jason Orendorff、Jason R. Coombs 等人编写的历史 path.py 模块，它提供了一个 str 子类化 Path 类；

- Twisted 的略有专业化的 FilePath 类；

- 一个 AlternativePathClass 提议，子类化 tuple 而不是 str；

- Unipath，str 子类化方法的变体，有两个公共类，一个用于不执行 I/O 的操作的 AbstractPath 类和一个用于所有常见操作的 Path 类。

本提议试图从这些之前的尝试和 PEP 355 的拒绝中吸取教训。

## 实现

本提议的实现在 pathlib 的 Mercurial 存储库的 pep428 分支中跟踪。

## 为什么是面向对象 API

使用专用类表示文件系统路径的理由与其他无状态对象（如日期、时间或 IP 地址）相同。Python 一直缓慢地远离严格复制 C 语言的 API，而是为各种常见功能提供更好的、更有帮助的抽象。即使本 PEP 未被接受，很可能有一天标准库将采用另一种形式的文件系统处理抽象。

事实上，许多人更喜欢使用 datetime 模块提供的高级对象来处理日期和时间，而不是使用数字时间戳和 time 模块 API。此外，使用专用类可以默认启用可取的行为，例如 Windows 路径的大小写不敏感性。

## 提议

### 类层次结构

pathlib 模块实现了一个简单的类层次结构：

```
                +----------+
                |          |
       ---------| PurePath |--------
       |        |          |       |
       |        +----------+       |
       |             |             |
       |             |             |
       v             |             v
+---------------+    |    +-----------------+
|               |    |    |                 |
| PurePosixPath |    |    | PureWindowsPath |
|               |    |    |                 |
+---------------+    |    +-----------------+
       |             v             |
       |          +------+         |
       |          |      |         |
       |   -------| Path |------   |
       |   |      |      |     |   |
       |   |      +------+     |   |
       |   |                   |   |
       |   |                   |   |
       v   v                   v   v
  +-----------+           +-------------+
  |           |           |             |
  | PosixPath |           | WindowsPath |
  |           |           |             |
  +-----------+           +-------------+
```

此层次结构沿两个维度划分路径类：

- 路径类可以是纯的或具体的：纯类仅支持不需要执行任何实际 I/O 的操作，即大多数路径操作操作；具体类支持纯类的所有操作，加上执行 I/O 的操作。

- 路径类根据其表示的操作系统路径类型具有给定风味。pathlib 实现两种风味：Windows 路径表示 Windows 系统中的文件系统语义，POSIX 路径表示其他系统。

任何纯类都可以在任何系统上实例化：例如，你可以在 Windows 下操作 PurePosixPath 对象，在 Unix 下操作 PureWindowsPath 对象，等等。然而，具体类只能在匹配的系统上实例化：事实上，在 Unix 下开始使用 WindowsPath 对象执行 I/O，或者反之亦然，容易出错。

此外，还有两个也充当系统依赖工厂的基类：PurePath 将根据操作系统实例化 PurePosixPath 或 PureWindowsPath。类似地，Path 将实例化 PosixPath 或 WindowsPath。

预计，在大多数使用中，使用 Path 类就足够了，这就是为什么它拥有所有类中最短的名称。

### 不与内置类型混淆

在本提议中，路径类不从内置类型派生。这与某些其他 Path 类提议不同，后者从 str 派生。它们也不假装实现序列协议：如果你想要路径充当序列，你必须查找专用属性（parts 属性）。

不从 str 继承的关键理由是防止意外地执行路径字符串和非路径字符串之间的操作，例如 path + an_accident。由于与字符串的操作不一定会导致有效或预期的文件系统路径，因此通过不子类化 str 来避免与字符串的意外操作，"明确优于隐含"。一位 Python 核心开发人员的博客文章更详细地介绍了这一特定设计决策背后的原因。

### 不可变性

路径对象是不可变的，这使得它们可哈希，也防止了一类编程错误。

### 合理的行为

os.path 的功能几乎没有重用。许多 os.path 函数由于向后兼容性而绑定到令人困惑或完全错误的行为（例如，os.path.abspath() 简化".."路径组件而不首先解析符号链接的事实）。

### 比较

相同风味的路径可比且可排序，无论是纯的还是非纯的：

```python
>>> PurePosixPath('a') == PurePosixPath('b')
False
>>> PurePosixPath('a') < PurePosixPath('b')
True
>>> PurePosixPath('a') == PosixPath('a')
True
```

比较和排序 Windows 路径对象是大小写不敏感的：

```python
>>> PureWindowsPath('a') == PureWindowsPath('A')
True
```

不同风味的路径总是比较不相等，且不能排序：

```python
>>> PurePosixPath('a') == PureWindowsPath('a')
False
>>> PurePosixPath('a') < PureWindowsPath('a')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: unorderable types: PurePosixPath() < PureWindowsPath()
```

路径与内置类型（如 str）和任何其他类型的实例比较不相等，且不可排序。

### 有用的表示法

API 试图提供有用的表示法，同时避免魔法。一些示例：

```python
>>> p = Path('/home/antoine/pathlib/setup.py')
>>> p.name
'setup.py'
>>> p.suffix
'.py'
>>> p.root
'/'
>>> p.parts
('/', 'home', 'antoine', 'pathlib', 'setup.py')
>>> p.relative_to('/home/antoine')
PosixPath('pathlib/setup.py')
>>> p.exists()
True
```

## 纯路径 API

PurePath API 的哲学是提供一致的一系列有用的路径操作操作，而不像 os.path 那样暴露一堆函数。

### 定义

首先几个约定：

- 所有路径都可以有驱动器和根。对于 POSIX 路径，驱动器始终为空。

- 相对路径既没有驱动器也没有根。

- POSIX 路径如果有根则是绝对的。Windows 路径如果既有驱动器又有根则是绝对的。Windows UNC 路径（例如`\\host\share\myfile.txt`）总是有驱动器和根（分别是`\\host\share` 和`\`）。

- 具有驱动器或根的路径被称为锚定的。其锚是驱动器和根的拼接。在 POSIX 下，"锚定"与"绝对"相同。

### 构造

我们将构造和拼接一起呈现，因为它们暴露了类似的语义。

构造路径的最简单方法是传递其字符串表示：

```python
>>> PurePath('setup.py')
PurePosixPath('setup.py')
```

多余的路径分隔符和"."组件被消除：

```python
>>> PurePath('a///b/c/./d/')
PurePosixPath('a/b/c/d')
```

如果你传递多个参数，它们将自动拼接：

```python
>>> PurePath('docs', 'Makefile')
PurePosixPath('docs/Makefile')
```

拼接语义类似于 os.path.join，即锚定路径忽略之前拼接组件的信息：

```python
>>> PurePath('/etc', '/usr', 'bin')
PurePosixPath('/usr/bin')
```

然而，对于 Windows 路径，驱动器在必要时保留：

```python
>>> PureWindowsPath('c:/foo', '/Windows')
PureWindowsPath('c:/Windows')
>>> PureWindowsPath('c:/foo', 'd:')
PureWindowsPath('d:')
```

此外，路径分隔符被规范化为平台默认值：

```python
>>> PureWindowsPath('a/b') == PureWindowsPath('a\\b')
True
```

多余的路径分隔符和"."组件被消除，但".."组件不是：

```python
>>> PurePosixPath('a//b/./c/')
PurePosixPath('a/b/c')
>>> PurePosixPath('a/../b')
PurePosixPath('a/../b')
```

多个前导斜杠根据路径风味被不同地处理。它们始终保留在 Windows 路径上（因为 UNC 表示法）：

```python
>>> PureWindowsPath('//some/path')
PureWindowsPath('//some/path/')
```

在 POSIX 上，它们被折叠，除非正好有两个前导斜杠，这是 POSIX 规范中路径名解析的特殊情况（这对于 Cygwin 兼容性也是必要的）：

```python
>>> PurePosixPath('///some/path')
PurePosixPath('/some/path')
>>> PurePosixPath('//some/path')
PurePosixPath('//some/path')
```

不带任何参数调用构造函数创建一个指向逻辑"当前目录"的路径对象（而不查找其绝对路径，这是具体路径上 cwd() 类方法的工作）：

```python
>>> PurePosixPath()
PurePosixPath('.')
```

### 表示

要表示路径（例如传递给第三方库），只需调用 str()：

```python
>>> p = PurePath('/home/antoine/pathlib/setup.py')
>>> str(p)
'/home/antoine/pathlib/setup.py'
>>> p = PureWindowsPath('c:/windows')
>>> str(p)
'c:\\windows'
```

要强制使用正斜杠的字符串表示，使用 as_posix() 方法：

```python
>>> p.as_posix()
'c:/windows'
```

要获取字节表示（在 Unix 系统上可能有用），调用 bytes()，它内部使用 os.fsencode()：

```python
>>> bytes(p)
b'/home/antoine/pathlib/setup.py'
```

要将路径表示为 file: URI，调用 as_uri() 方法：

```python
>>> p = PurePosixPath('/etc/passwd')
>>> p.as_uri()
'file:///etc/passwd'
>>> p = PureWindowsPath('c:/Windows')
>>> p.as_uri()
'file:///c:/Windows'
```

路径的 repr() 始终使用正斜杠，即使在 Windows 下，也是为了可读性并提醒用户正斜杠是可以的：

```python
>>> p = PureWindowsPath('c:/Windows')
>>> p
PureWindowsPath('c:/Windows')
```

### 属性

每个路径上提供几个简单属性（每个可以为空）：

```python
>>> p = PureWindowsPath('c:/Downloads/pathlib.tar.gz')
>>> p.drive
'c:'
>>> p.root
'\\'
>>> p.anchor
'c:\\'
>>> p.name
'pathlib.tar.gz'
>>> p.stem
'pathlib.tar'
>>> p.suffix
'.gz'
>>> p.suffixes
['.tar', '.gz']
```

### 派生新路径

#### 拼接

路径可以使用 / 运算符与另一个拼接：

```python
>>> p = PurePosixPath('foo')
>>> p / 'bar'
PurePosixPath('foo/bar')
>>> p / PurePosixPath('bar')
PurePosixPath('foo/bar')
>>> 'bar' / p
PurePosixPath('bar/foo')
```

与构造函数一样，可以指定多个路径组件，折叠或分开：

```python
>>> p / 'bar/xyzzy'
PurePosixPath('foo/bar/xyzzy')
>>> p / 'bar' / 'xyzzy'
PurePosixPath('foo/bar/xyzzy')
```

还提供了 joinpath() 方法，具有相同的行为：

```python
>>> p.joinpath('Python')
PurePosixPath('foo/Python')
```

#### 更改路径的最终组件

with_name() 方法返回一个新路径，名称已更改：

```python
>>> p = PureWindowsPath('c:/Downloads/pathlib.tar.gz')
>>> p.with_name('setup.py')
PureWindowsPath('c:/Downloads/setup.py')
```

如果路径没有实际名称，它将以 ValueError 失败：

```python
>>> p = PureWindowsPath('c:/')
>>> p.with_name('setup.py')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "pathlib.py", line 875, in with_name
    raise ValueError("%r has an empty name" % (self,))
ValueError: PureWindowsPath('c:/') has an empty name
>>> p.name
''
```

with_suffix() 方法返回一个新路径，后缀已更改。然而，如果路径没有后缀，则添加新后缀：

```python
>>> p = PureWindowsPath('c:/Downloads/pathlib.tar.gz')
>>> p.with_suffix('.bz2')
PureWindowsPath('c:/Downloads/pathlib.tar.bz2')
>>> p = PureWindowsPath('README')
>>> p.with_suffix('.bz2')
PureWindowsPath('README.bz2')
```

#### 使路径相对

relative_to() 方法计算路径相对于另一个路径的相对差：

```python
>>> PurePosixPath('/usr/bin/python').relative_to('/usr')
PurePosixPath('bin/python')
```

如果方法无法返回有意义的值，则引发 ValueError：

```python
>>> PurePosixPath('/usr/bin/python').relative_to('/etc')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "pathlib.py", line 926, in relative_to
    .format(str(self), str(formatted)))
ValueError: '/usr/bin/python' does not start with '/etc'
```

### 序列式访问

parts 属性返回一个元组，提供对路径组件的只读序列访问：

```python
>>> p = PurePosixPath('/etc/init.d')
>>> p.parts
('/', 'etc', 'init.d')
```

Windows 路径将驱动器和根作为单个路径组件处理：

```python
>>> p = PureWindowsPath('c:/setup.py')
>>> p.parts
('c:\\', 'setup.py')
```

（将它们分开是错误的，因为 C: 不是 C:\\的父级）。

parent 属性返回路径的逻辑父级：

```python
>>> p = PureWindowsPath('c:/python33/bin/python.exe')
>>> p.parent
PureWindowsPath('c:/python33/bin')
```

parents 属性返回路径逻辑祖先的不可变序列：

```python
>>> p = PureWindowsPath('c:/python33/bin/python.exe')
>>> len(p.parents)
3
>>> p.parents[0]
PureWindowsPath('c:/python33/bin')
>>> p.parents[1]
PureWindowsPath('c:/python33')
>>> p.parents[2]
PureWindowsPath('c:/')
```

### 查询

is_relative() 如果路径是相对的则返回 True（见上文定义），否则返回 False。

is_reserved() 如果 Windows 路径是保留路径（如 CON 或 NUL）则返回 True。对于 POSIX 路径始终返回 False。

match() 将路径与 glob 模式匹配。它对单个部分操作并从右侧匹配：

```python
>>> p = PurePosixPath('/usr/bin')
>>> p.match('/usr/b*')
True
>>> p.match('usr/b*')
True
>>> p.match('b*')
True
>>> p.match('/u*')
False
```

此行为尊重以下期望：

- 像"*.py"这样的简单模式匹配任意长的路径，只要最后一部分匹配，例如"/usr/foo/bar.py"。

- 更长的模式也可用于更复杂的匹配，例如"/usr/foo/*.py"匹配"/usr/foo/bar.py"。

## 具体路径 API

除了纯 API 的操作外，具体路径还提供了实际访问文件系统以查询或更改信息的附加方法。

### 构造

类方法 cwd() 创建一个指向当前工作目录的路径对象，以绝对形式表示：

```python
>>> Path.cwd()
PosixPath('/home/antoine/pathlib')
```

### 文件元数据

stat() 返回文件的 stat() 结果；类似地，lstat() 返回文件的 lstat() 结果（仅当文件是符号链接时不同）：

```python
>>> p.stat()
posix.stat_result(st_mode=33277, st_ino=7483155, st_dev=2053, st_nlink=1, st_uid=500, st_gid=500, st_size=928, st_atime=1343597970, st_mtime=1328287308, st_ctime=1343597964)
```

更高级的方法帮助检查文件类型：

```python
>>> p.exists()
True
>>> p.is_file()
True
>>> p.is_dir()
False
>>> p.is_symlink()
False
>>> p.is_socket()
False
>>> p.is_fifo()
False
>>> p.is_block_device()
False
>>> p.is_char_device()
False
```

文件所有者和组名称（而不是数字 ID）通过相应方法查询：

```python
>>> p = Path('/etc/shadow')
>>> p.owner()
'root'
>>> p.group()
'shadow'
```

### 路径解析

resolve() 方法使路径绝对，解析途中任何符号链接（像 POSIX realpath() 调用）。它是唯一将删除".."路径组件的操作。在 Windows 上，此方法还将注意返回规范路径（具有正确的大小写）。

### 目录遍历

简单（非递归）目录访问通过调用 iterdir() 方法完成，该方法返回子路径的迭代器：

```python
>>> p = Path('docs')
>>> for child in p.iterdir():
...     child
PosixPath('docs/conf.py')
PosixPath('docs/_templates')
PosixPath('docs/make.bat')
PosixPath('docs/index.rst')
PosixPath('docs/_build')
PosixPath('docs/_static')
PosixPath('docs/Makefile')
```

这允许通过列表推导式进行简单过滤：

```python
>>> p = Path('.')
>>> [child for child in p.iterdir() if child.is_dir()]
[PosixPath('.hg'), PosixPath('docs'), PosixPath('dist'), PosixPath('__pycache__'), PosixPath('build')]
```

还提供简单和递归的 globbing：

```python
>>> for child in p.glob('**/*.py'):
...     child
PosixPath('test_pathlib.py')
PosixPath('setup.py')
PosixPath('pathlib.py')
PosixPath('docs/conf.py')
PosixPath('build/lib/pathlib.py')
```

### 文件打开

open() 方法提供了一个类似于内置 open() 方法的文件打开 API：

```python
>>> p = Path('setup.py')
>>> with p.open() as f:
...     f.readline()
'#!/usr/bin/env python3\n'
```

### 文件系统修改

几个常见的文件系统操作作为方法提供：touch()、mkdir()、rename()、replace()、unlink()、rmdir()、chmod()、lchmod()、symlink_to()。可以提供更多操作，例如 shutil 模块的一些功能。

建议 API 的详细文档可在 pathlib 文档中找到。

## 讨论

### 除法运算符

除法运算符在关于路径拼接运算符的投票中脱颖而出。pathlib 的初始版本使用方括号（即__getitem__）代替。

### joinpath()

joinpath() 方法最初称为 join()，但几个人反对它可能与具有不同语义的 str.join() 混淆。因此，它被重命名为 joinpath()。

### 大小写敏感性

Windows 用户认为文件系统路径是大小写不敏感的，并希望路径对象遵守这一特征，尽管在一些罕见情况下，一些外国文件系统挂载在 Windows 下可能是大小写敏感的。

用一位评论者的话说：

> "如果 glob("*.py") 在 Windows 上找不到 SETUP.PY，那将是一个可用性灾难"。——Paul Moore in
> https://mail.python.org/pipermail/python-dev/2013-April/125254.html
