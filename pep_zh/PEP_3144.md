# PEP 3144 – Python 标准库的 IP 地址操作库

**Metadata:**

- **Author**: Peter Moody <pmoody&#32;&#97;t&#32;google.com>

- **Python-Version**: 3.3

## 摘要

本 PEP 为 Python 提议了一个 IP 地址操作模块的设计和实现。

## PEP 接受

本 PEP 于 2012 年 5 月 15 日由 Alyssa Coghlan 接受。

## 动机

已经存在几个非常好的 Python IP 地址模块。事实是，它们都在遵循 Pythonic 原则和网络工程师及管理员依赖的简写之间苦苦挣扎。ipaddress 旨在取得正确的平衡。

## 理由

存在几个 Python IP 地址操作模块，这证明了对此模块试图提供的功能有着迫切的需求。

## 背景

PEP 3144 和 ipaddr 之前曾被提上纳入标准库的议程。这里指定的库版本与 PyPI 上的版本以及之前讨论的版本不向后兼容。为了避免混淆当前 ipaddr 的用户，我已将此版本的库重命名为 ipaddress。

ipaddr 和 ipaddress 之间的主要区别是：

- ipaddress *Network 类等同于 ipaddr *Network 类的对应类，strict 标志设置为 True。

- ipaddress *Interface 类等同于 ipaddr *Network 类的对应类，strict 标志设置为 False。

- ipaddress 中的工厂函数被重命名以与类区分开来。

- 一些属性也被重命名以区分它们的用途。（例如，network, network_address）

- ipaddr 中返回容器的许多方法和函数现在返回迭代器。这包括 subnets、address_exclude、summarize_address_range 和 collapse_address_list。

由于 ipaddress 和 ipaddr 之间存在不向后兼容的 API 更改，建议使用新的临时 API 状态添加该模块：

- http://docs.python.org/dev/glossary.html#term-provisional-package

## 规范

ipaddr 模块总共定义了 6 个新的公共类，其中 3 个用于操作 IPv4 对象，3 个用于操作 IPv6 对象。这些类如下：

- IPv4Address/IPv6Address - 这些定义了单个地址，例如对 www.google.com 的 A 记录查询返回的 IPv4 地址（74.125.224.84）或对 ipv6.google.com 的 AAAA 记录查询返回的 IPv6 地址（2001:4860:4001:801::1011）。

- IPv4Network/IPv6Network - 这些定义了网络或地址组，例如保留用于组播使用的 IPv4 网络（224.0.0.0/4）或保留用于组播的 IPv6 网络（ff00::/8，哇，很大）。

- IPv4Interface/IPv6Interface - 这些混合类指的是给定网络上的单个地址。例如，网络 192.0.2.0/24 上的 IPv4 地址 192.0.2.1 可以被称为 192.0.2.1/24。同样，网络 2001:DB8::/96 上的 IPv6 地址 2001:DB8::1 可以被称为 2001:DB8::1/96。非常常见的是以这种方式引用分配给计算机网络接口的地址，因此得名 Interface。

所有 IPv4 类共享某些特性和方法；表示它们所需的位数、它们是否属于某些特殊 IPv4 网络范围等。同样，所有 IPv6 类共享特性和方法。

ipaddr 广泛使用继承来尽可能避免代码重复。父类是私有的，但它们在这里概述如下：

- _IPAddrBase - 提供所有 ipaddr 对象共用的方法。

- _BaseAddress - 提供 IPv4Address 和 IPv6Address 共用的方法。

- _BaseInterface - 提供 IPv4Interface 和 IPv6Interface 共用的方法，以及 IPv4Network 和 IPv6Network（ipaddr 将 Network 类视为 Interface 的特殊情况）。

- _BaseV4 - 提供所有 IPv4 类共用的方法和变量（例如，_max_prefixlen）。

- _BaseV6 - 提供所有 IPv6 类共用的方法和变量。

不同 IP 版本的对象之间的比较会导致 TypeError [1]。此外，具有不同_Base 父类的对象的比较会导致 TypeError。_Base 父类限制的效果是 IPv4Interface 可以与 IPv4Network 进行比较，IPv6Interface 可以与 IPv6Network 进行比较。

诉诸权威是一种逻辑谬误，但 Vint Cerf 是一个不可忽视的权威。电子邮件全文如下：> 我看到了大量关于 IPv4 和 IPv6 比较的讨论，普遍共识是这些是不可比较的。如果我们采取非常简单的观点，我们可能会将这些视为纯整数，在这种情况下有一个排序，但不是有用的排序。

在 IPv4 世界中，“长度”很重要，因为我们在路由时首先取最长（最具体）的地址。长度由掩码决定，如你所知。

假设类似的论证在 IPv6 中有效，我们将不得不得出结论，将 IPv6 值纯粹作为整数与 IPv4 进行比较会导致一些非常奇怪的结果。

所有 IPv4 空间都将位于 IPv6 的 0::0/96 前缀的主机空间中。对于 IPv4 的任何有用解释，这都是不可行的。

我认为唯一合理的结论是 IPv4 值和 IPv6 值应该被视为不可比较的。
