# PEP 3156 – 异步 IO 支持重启："asyncio" 模块

## 摘要

这是 Python 3（从 Python 3.3 开始）异步 I/O 的提案。可以将其视为 PEP 3153 中缺失的具体提案。该提案包括可插拔的事件循环、类似于 Twisted 中的传输和协议抽象，以及基于 yield from（PEP 380）的高级调度器。建议的包名为 asyncio。

许多拟议功能需要 Python 3.3。参考实现（Tulip）除了 Python 3.3 之外不需要新的语言或标准库功能，不需要第三方模块或包，也不需要 C 代码，Windows 上的（可选）IOCP 支持除外。

### 模块命名空间

这里的规范位于一个新的顶层包 asyncio 中。不同的组件位于包的独立子模块中。该包将从其各自的子模块导入常用 API，并将它们作为包属性提供（类似于 email 包的工作方式）。对于这些常用 API，实际定义它们的子模块名称不是规范的一部分。不太常用的 API 可能必须显式地从其各自的子模块导入，在这种情况下，子模块名称是规范的一部分。

未指定子模块名称定义的类和函数假定位于顶层包的命名空间中。（但不要将这些与各种类的方法混淆，为了简洁起见，在某些上下文中，方法也未使用前缀。）

### 互操作性

事件循环是大多数互操作性发生的地方。像 Twisted、Tornado，甚至 gevent 的（Python 3.3 移植版）框架应该能够轻松适默认事件循环实现来满足他们的需求，使用轻量级适配器或代理，或者用自己的事件循环实现的适配版本替换默认事件循环实现。（某些框架，如 Twisted，有多个事件循环实现。这不应成为问题，因为它们都有相同的接口。）

在大多数情况下，两个不同的第三方框架应该能够互操作，要么通过共享默认事件循环实现（每个使用自己的适配器），要么通过共享任一框架的事件循环实现。在后一种情况下，将发生两个级别的适配（从框架 A 的事件循环到标准事件循环接口，然后到框架 B 的事件循环）。使用哪个事件循环实现应由主程序控制（尽管提供了事件循环选择的默认策略）。

为了使这种互操作性有效，第三方框架中首选的适配方向是保留默认事件循环并将其适配到框架的 API。理想情况下，所有第三方框架都将放弃自己的事件循环实现，转而采用标准实现。但并非所有框架都能对标准实现提供的功能感到满意。

为了支持两个方向的适配，指定了两个独立的 API：

- 管理当前事件循环的接口

- 符合标准的事件循环接口

事件循环实现可以提供额外的方法和保证，只要这些在文档中被指出是非标准的。事件循环实现也可以不实现某些方法，如果它们无法在给定的环境中实现；但是，这种对标准 API 的偏离应仅作为最后手段考虑，并且仅当平台或环境迫使该问题时。（一个例子是一个系统事件循环无法启动或停止的平台；见下面的"嵌入式事件循环"。）

事件循环 API 不依赖于 await/yield from。相反，它使用回调、额外接口（传输和协议）以及 Future 的组合。后者类似于 PEP 3148 中定义的，但有不同的实现，并且不绑定到线程。特别是，result() 方法在结果尚未就绪时抛出异常而不是阻塞；用户应使用回调（或 await/yield from）等待结果。

所有指定为返回协程的事件循环方法都允许返回 Future 或协程，由实现选择（标准实现总是返回协程）。所有记录为接受协程参数的事件循环方法必须接受 Future 和协程作为此类参数。（存在一个便利函数 ensure_future()，用于将协程或 Future 参数转换为 Future。）

对于不喜欢使用回调的用户（如我自己），提供了一个调度器，用于使用 PEP 380 yield from 或 PEP 492 await 表达式编写异步 I/O 代码作为协程。
调度器不是可插拔的；
可插拔性发生在事件循环级别，标准调度器实现应与任何符合标准的事件循环实现一起工作。（事实上，这是符合标准实现的重要测试。）

对于使用协程编写的代码与其他异步框架之间的互操作性，调度器定义了一个 Task 类，其行为像一个 Future。在事件循环级别进行互操作的框架可以通过向 Future 添加回调来等待 Future 完成。同样，调度器提供了一个操作来挂起协程，直到调用回调。

如果这样的框架不能使用原样 Future 和 Task 类，它可以重新实现 loop.create_future() 和 loop.create_task() 方法。这些应返回实现 Future/Task 接口（或其超集）的对象。

一个不太雄心勃勃的框架可以只是调用 loop.set_task_factory() 来替换 Task 类，而无需实现自己的事件循环。

事件循环 API 提供与线程的有限互操作性：有一个 API 可以将函数提交给执行器（参见 PEP 3148），它返回一个与事件循环兼容的 Future，还有一个方法可以以线程安全的方式从另一个线程将回调调度到事件循环。

### 传输和协议

对于那些不熟悉 Twisted 的人，有必要简要解释传输和协议之间的关系。在最高级别，传输关心的是字节如何传输，而协议决定传输哪些字节（以及在某种程度上何时传输）。

用另一种方式说同样的事情：传输是套接字（或类似 I/O 端点）的抽象，而协议是从传输的角度来看应用程序的抽象。

还有一种观点简单地认为传输和协议接口一起定义了使用网络 I/O 和进程间 I/O 的抽象接口。

传输和协议对象之间几乎总是存在 1:1 关系：协议调用传输方法来发送数据，而传输调用协议方法来传递接收到的数据。传输和协议方法都不会"阻塞"——它们启动事件然后返回。

最常见的传输类型是双向流传输。它表示一对缓冲流（每个方向一个），每个流传输字节序列。双向流传输最常见的例子可能是 TCP 连接。另一个常见的例子是 SSL/TLS 连接。但还有一些其他可以这样看待的东西，例如 SSH 会话或一对 UNIX 管道。通常没有太多不同的传输实现，其中大多数随事件循环实现一起提供。然而，并不要求所有传输都必须通过调用事件循环方法创建：第三方模块完全可以实现新传输，并提供一个仅接受事件循环作为参数或调用 get_event_loop() 的构造函数或工厂函数。

注意，传输不需要使用套接字，即使它们使用 TCP——套接字是平台特定的实现细节。

双向流传输有两个"端"：一端与网络（或另一个进程，或它包装的任意低级接口）通信，另一端与协议通信。前者使用实现传输所需的任何 API；但是传输和协议之间的接口由本 PEP 标准化。

协议可以表示某种"应用级"协议，如 HTTP 或 SMTP；它也可以实现多个协议共享的抽象，或整个应用程序。协议的主要接口是传输。虽然某些常用协议（和其他抽象）可能有标准实现，但通常应用程序实现自定义协议。从 PyPI 下载和安装有用的第三方协议实现库也有意义。

传输和协议的一般概念包括其他接口，其中传输包装其他通信抽象。示例包括用于发送和接收数据报（例如 UDP）的接口，或子进程管理器。关注点的分离与双向流传输和协议相同，但每种情况下传输和协议之间的特定接口不同。

各种标准类型的传输和协议定义的接口细节稍后给出。

## 事件循环接口规范

### 事件循环策略：获取和设置当前事件循环

事件循环管理由事件循环策略控制，它是一个全局（每进程）对象。存在一个默认策略，以及更改策略的 API。策略定义了上下文的概念；策略为每个上下文管理单独的事件循环。默认策略的上下文概念定义为当前线程。

某些平台或编程框架可能会将默认策略更改为更适合该平台或框架用户期望的内容。此类平台或框架必须记录其策略以及在初始化序列的哪个点设置策略，以避免多个活动框架都想覆盖默认策略时出现未定义行为。（另见下面的"嵌入式事件循环"。）

要获取当前上下文的事件循环，使用 get_event_loop()。这返回一个实现下面指定接口的事件循环对象，或者在没有为当前上下文设置事件循环且当前策略未指定创建一个时抛出异常。它绝不应返回 None。

要设置当前上下文的事件循环，使用 set_event_loop(event_loop)，其中 event_loop 是一个事件循环对象，即 AbstractEventLoop 的实例，或 None。将当前事件循环设置为 None 是可以的，在这种情况下，后续对 get_event_loop() 的调用将抛出异常。这对于测试不应依赖于默认事件循环存在的代码很有用。

预计 get_event_loop() 会根据上下文返回不同的事件循环对象（事实上，这是上下文的定义）。如果没有设置且策略允许创建，它可能会创建新事件循环对象。默认策略仅在主线线程中创建新事件循环（如 threading.py 所定义，它为线程使用特殊子类），并且仅在调用 get_event_loop() 之前从未调用过 set_event_loop() 时。（要重置此状态，重置策略。）在其他线程中，必须显式设置事件循环。其他策略可能有不同的行为。默认策略创建的事件循环是延迟的；即，如果对 get_event_loop() 的第一次调用根据需要创建事件循环实例，并且由当前策略指定。

为了单元测试和其他特殊情况的好处，有第三个策略函数：new_event_loop()，它根据策略的默认规则创建并返回新事件循环对象。要使其成为当前事件循环，您必须使用它调用 set_event_loop()。

要更改事件循环策略，调用 set_event_loop_policy(policy)，其中 policy 是事件循环策略对象或 None。如果非 None，策略对象必须是 AbstractEventLoopPolicy 的实例，定义方法 get_event_loop()、set_event_loop(loop) 和 new_event_loop()，所有行为都像上面描述的函数。

传递 None 作为策略值会恢复默认事件循环策略（覆盖平台或框架设置的备用默认值）。默认事件循环策略是 DefaultEventLoopPolicy 类的实例。可以通过调用 get_event_loop_policy() 检索当前事件循环策略对象。

待定：描述子进程处理的子进程监视器和 UNIX 怪癖。

#### 显式传递事件循环

可以编写使用事件循环而不依赖全局或每线程默认事件循环的代码。为此，所有需要访问当前事件循环的 API（不是事件类上的方法）都接受一个名为 loop 的可选关键字参数。如果此参数为 None 或未指定，此类 API 将调用 get_event_loop() 获取默认事件循环，但如果 loop 关键字参数设置为事件循环对象，它们将使用该事件循环，并将其传递给它们调用的任何其他此类 API。例如，Future(loop=my_loop) 将创建绑定到事件循环 my_loop 的 Future。当默认当前事件循环为 None 时，loop 关键字参数实际上是强制性的。

注意，显式传递的事件循环仍必须属于当前线程；loop 关键字参数不会神奇地更改事件循环使用方式的约束。

### 指定时间

像在 Python 中通常一样，所有超时、间隔和延迟都以秒为单位测量，可以是 int 或 float。但是，绝对时间不指定为 POSIX 时间戳。时钟的准确性、精度和纪元由实现决定。

默认实现使用 time.monotonic()。关于此选择的含义可以写很多书。最好阅读标准库 time 模块的文档。

### 嵌入式事件循环

在某些平台上，事件循环由系统提供。这样的循环可能在用户代码开始时已经在运行，并且可能没有办法在退出程序之前停止或关闭它。在这种情况下，启动、停止和关闭事件循环的方法可能无法实现，is_running() 可能总是返回 True。

### 事件循环类

没有名为 EventLoop 的实际类。有一个 AbstractEventLoop 类定义了所有方法而没有实现，主要用作文档。定义了以下具体类：

- SelectorEventLoop 是基于 selectors 模块（Python 3.4 中新）的完整 API 的具体实现。构造函数接受一个可选参数，selectors.Selector 对象。默认情况下，创建并使用 selectors.DefaultSelector 的实例。

- ProactorEventLoop 是 API 的具体实现，除了 I/O 事件处理和信号处理方法。它仅在 Windows 上定义（或支持类似"重叠 I/O"API 的其他平台上）。构造函数接受一个可选参数，Proactor 对象。默认情况下，创建并使用 IocpProactor 的实例。（IocpProactor 类不由本 PEP 指定；它只是 ProactorEventLoop 类的实现细节。）

### 事件循环方法概述

符合标准的事件循环的方法分为几类。第一组类别必须由所有符合标准的事件循环实现支持，例外情况是嵌入式事件循环可能不实现启动、停止和关闭的方法。（然而，部分符合标准的事件循环总比没有好。:-)

- 启动、停止和关闭：run_forever()、run_until_complete()、stop()、is_running()、close()、is_closed()。

- 基本和定时回调：call_soon()、call_later()、call_at()、time()。

- 线程交互：call_soon_threadsafe()、run_in_executor()、set_default_executor()。

- 互联网名称查找：getaddrinfo()、getnameinfo()。

- 互联网连接：create_connection()、create_server()、create_datagram_endpoint()。

- 包装套接字方法：sock_recv()、sock_sendall()、sock_connect()、sock_accept()。

- 任务和 Future 支持：create_future()、create_task()、set_task_factory()、get_task_factory()。

- 错误处理：get_exception_handler()、set_exception_handler()、default_exception_handler()、call_exception_handler()。

- 调试模式：get_debug()、set_debug()。

第二组类别可以由符合标准的事件循环实现支持。如果不支持，它们将抛出 NotImplementedError。（在默认实现中，UNIX 系统上的 SelectorEventLoop 支持所有这些；Windows 上的 SelectorEventLoop 支持 I/O 事件处理类别；Windows 上的 ProactorEventLoop 支持管道和子进程类别。）

- I/O 回调：add_reader()、remove_reader()、add_writer()、remove_writer()。

- 管道和子进程：connect_read_pipe()、connect_write_pipe()、subprocess_shell()、subprocess_exec()。

- 信号回调：add_signal_handler()、remove_signal_handler()。

### 事件循环方法

#### 启动、停止和关闭

（未关闭的）事件循环可以处于两种状态之一：运行或停止。这些方法处理事件循环的启动和停止：

- run_forever()。运行事件循环直到调用 stop()。当事件循环已经在运行时不能调用此方法。（这有一个长名称，部分是为了避免与此 PEP 的早期版本混淆，其中 run() 有不同的行为，部分是已经有太多具有名为 run() 方法的 API，部分是因为无论如何不应有太多地方调用此方法。）

- run_until_complete(future)。运行事件循环直到 Future 完成。如果 Future 已完成，返回其结果，或抛出其异常。当事件循环已经在运行时不能调用此方法。
此方法如果参数是协程，则创建新的 Task 对象。

- stop()。尽可能快地停止事件循环。随后使用 run_forever() 或 run_until_complete() 重启循环是可以的；如果这样做，不会丢失任何已计划的回调。注意：stop() 正常返回，当前回调允许继续。此后事件循环何时停止由实现决定，但意图是在轮询 I/O 之前停止，并且不运行任何未来计划的回调；实现的主要自由度是在停止之前处理多少"就绪队列"（已用 call_soon() 计划的回调）。

- is_running()。如果事件循环当前正在运行，返回 True，如果已停止，返回 False。

- close()。关闭事件循环，释放它可能持有的任何资源，如 epoll() 或 kqueue() 使用的文件描述符，以及默认执行器。不应在事件循环运行时调用此方法。调用后不应再使用事件循环。可以多次调用；后续调用是空操作。

- is_closed()。如果事件循环已关闭，返回 True，否则返回 False。（主要用于错误报告；请不要基于此方法实现功能。）

#### 基本回调

与同一事件循环关联的回调严格序列化：一个回调必须完成，然后才能调用下一个。这是一个重要的保证：当两个或多个回调使用或修改共享状态时，每个回调保证在运行时，共享状态不会被另一个回调更改。

- call_soon(callback, *args)。这计划尽快调用回调。返回一个 Handle（见下文），代表回调，其 cancel() 方法可用于取消回调。它保证按照计划顺序调用回调。

- call_later(delay, callback, *args)。安排在未来大约 delay 秒后调用 callback(*args)，一次，除非被取消。返回一个 Handle 代表回调，其 cancel() 方法可用于取消回调。在过去或以完全相同时间计划的回调将以未定义的顺序调用。

- call_at(when, callback, *args)。这类似于 call_later()，但时间表示为绝对时间。返回类似的 Handle。有一个简单的等价关系：loop.call_later(delay, callback, *args) 与 loop.call_at(loop.time() + delay, callback, *args) 相同。

- time()。根据事件循环的时钟返回当前时间。这可能是 time.time()、time.monotonic() 或其他系统特定的时钟，但它必须返回一个 float，表示自某个纪元以来约一秒的时间单位。（没有时钟是完美的——参见 PEP 418。）

注意：此 PEP 的先前版本定义了一个名为 call_repeatedly() 的方法，它承诺以固定间隔调用回调。这已被撤回，因为此类函数的设计过度指定。一方面，使用使用 call_later() 重新计划自己的回调的简单计时器循环可以轻松模拟；也很容易编写包含循环和 sleep() 调用（模块中的顶层函数，见下文）的协程。另一方面，由于精确时间keeping 的复杂性，这里有许多陷阱和隐患（参见 PEP 418），不同的用例在边缘情况下需要不同的行为。不可能为此目的提供一个在所有情况下都防弹的 API，因此认为最好让应用程序设计者自己决定实现什么类型的计时器循环。

#### 线程交互

- call_soon_threadsafe(callback, *args)。类似于 call_soon(callback, *args)，但当从另一个线程调用而事件循环被阻塞等待 I/O 时，它会解除事件循环的阻塞。返回一个 Handle。这是唯一可以从另一个线程安全调用的方法。（要以线程安全的方式计划稍后的回调，可以使用 loop.call_soon_threadsafe(loop.call_later, when, callback, *args)。）注意：从信号处理程序调用此方法不安全（因为它可能使用锁）。事实上，没有 API 是信号安全的；如果要处理信号，请使用下面描述的 add_signal_handler()。

- run_in_executor(executor, callback, *args)。安排在执行器中调用 callback(*args)（参见 PEP 3148）。返回一个 asyncio.Future 实例，其成功结果是该调用的返回值。这等效于 wrap_future(executor.submit(callback, *args))。如果 executor 为 None，则使用 set_default_executor() 设置的默认执行器。如果尚未设置默认执行器，则创建具有默认线程数的 ThreadPoolExecutor 并将其设置为默认执行器。（默认实现在此情况下使用 5 个线程。）

- set_default_executor(executor)。设置 run_in_executor() 使用的默认执行器。参数必须是 PEP 3148 Executor 实例或 None，以重置默认执行器。

另见关于 Future 的部分中描述的 wrap_future() 函数。

#### 互联网名称查找

这些方法在想要连接或绑定套接字到地址而不冒名称查找阻塞风险时很有用。它们通常由 create_connection()、create_server() 或 create_datagram_endpoint() 隐式调用。

- getaddrinfo(host, port, family=0, type=0, proto=0, flags=0)。类似于 socket.getaddrinfo() 函数，但返回一个 Future。Future 的成功结果将是与 socket.getaddrinfo() 返回的格式相同的列表，即 (address_family, socket_type, socket_protocol, canonical_name, address) 的列表，其中 address 对于 IPv4 地址是 2 元组 (ipv4_address, port)，对于 IPv6 地址是 4 元组 (ipv6_address, port, flow_info, scope_id)。如果 family 参数为零或未指定，返回的列表可能包含 IPv4 和 IPv6 地址的混合；否则返回的地址受 family 值约束（proto 和 flags 类似）。默认实现使用 run_in_executor() 调用 socket.getaddrinfo()，但其他实现可能选择实现自己的 DNS 查找。可选参数必须作为关键字参数指定。
注意：实现允许实现完整 socket.getaddrinfo() 接口的子集；例如，它们可能不支持符号端口名称，或者可能忽略或不完整实现 type、proto 和 flags 参数。但是，如果忽略 type 和 proto，传入的参数值应原样复制到返回元组的 socket_type 和 socket_protocol 元素中。（你不能忽略 family，因为 IPv4 和 IPv6 地址必须不同地查找。family 的唯一允许值是 socket.AF_UNSPEC（0）、socket.AF_INET 和 socket.AF_INET6，后两者仅当平台定义时才可用。）

- getnameinfo(sockaddr, flags=0)。类似于 socket.getnameinfo()，但返回一个 Future。Future 的成功结果将是元组 (host, port)。与 getaddrinfo() 相同的实现备注。

#### 互联网连接

这些是管理互联网连接的高级接口。推荐使用它们而不是相应的低级接口，因为它们抽象了基于选择器和基于 proactor 的事件循环之间的差异。

注意，流连接的客户端和服务端使用相同的传输和协议接口。但是，数据报端点使用不同的传输和协议接口。

- `create_connection(protocol_factory, host, port, <options>)`。创建到给定互联网主机和端口的流连接。这通常从连接客户端调用的任务。它创建实现特定的双向流 Transport 来表示连接，然后调用 protocol_factory() 实例化（或检索）用户的 Protocol 实现，最后将两者绑定在一起。（见下文 Transport 和 Protocol 的定义。）用户的 Protocol 实现通过不带参数调用 protocol_factory() 创建或检索 `(*)`。协程的成功结果是 (transport, protocol) 对；如果失败阻止成功连接的创建，将抛出适当的异常。注意，当协程完成时，协议的 connection_made() 方法尚未被调用；那将在连接握手完成时发生。
`(*)` 不要求 protocol_factory 是类。如果您的协议类需要向其构造函数传递特定参数，可以使用 lambda。您也可以传递一个返回先前构造的 Protocol 实例的平凡 lambda。
`<options>` 都使用可选关键字参数指定：

- - ssl：传递 True 以创建 SSL/TLS 传输（默认创建普通 TCP 传输）。或者传递 ssl.SSLContext 对象以覆盖要使用的默认 SSL 上下文对象。如果创建默认上下文，由实现配置合理的默认值。参考实现目前使用 PROTOCOL_SSLv23 并设置 OP_NO_SSLv2 选项，调用 set_default_verify_paths() 并将 verify_mode 设置为 CERT_REQUIRED。此外，每当上下文（默认或其他）指定 verify_mode 为 CERT_REQUIRED 或 CERT_OPTIONAL 时，如果给定主机名，成功握手后立即调用 ssl.match_hostname(peercert, hostname)，如果此调用抛出异常，则连接关闭。（要避免此行为，传递一个将 verify_mode 设置为 CERT_NONE 的 SSL 上下文。但这意味着您不安全，容易受到例如中间人攻击。）

  - family、proto、flags：要传递给 getaddrinfo() 的地址族、协议和标志。这些都默认为 0，意思是"未指定"。（套接字类型始终是 SOCK_STREAM。）如果未指定任何这些值，getaddrinfo() 方法将选择适当的值。注意：proto 与高级 Protocol 概念或 protocol_factory 参数无关。

  - sock：可选的套接字，用于代替使用 host、port、family、proto 和 flags 参数。如果给定，host 和 port 必须显式设置为 None。

  - local_addr：如果给定，用于将套接字绑定到本地的 (host, port) 元组。这很少需要，但在多宿主服务器上，您偶尔需要强制连接来自特定地址。这就是您这样做的方式。host 和 port 使用 getaddrinfo() 查找。

  - server_hostname：这仅在使用 SSL/TLS 时相关；当 ssl 未设置时不应使用。当 ssl 设置时，这设置或覆盖将验证的主机名。默认情况下使用 host 参数的值。如果 host 为空，则没有默认值，您必须传递 server_hostname 的值。要禁用主机名验证（这是严重的安全风险），您必须在这里传递空字符串，并传递一个将 verify_mode 设置为 ssl.CERT_NONE 的 ssl.SSLContext 对象作为 ssl 参数。

- `create_server(protocol_factory, host, port, <options>)`。进入接受连接的服务器循环。这是一个协程，一旦设置好服务循环完成。返回值是一个 Server 对象，可用于以受控方式停止服务循环（见下文）。如果指定地址允许多个套接字可以绑定，则允许多个套接字绑定。每次接受连接时，不带参数`(**)`调用 protocol_factory 创建 Protocol，创建双向流 Transport 来表示连接的网络端，并通过调用 protocol.connection_made(transport) 将两者绑定在一起。
`(**)` 见 create_connection() 的前一个脚注。但是，由于 protocol_factory() 每次为新传入连接调用一次，它应该在每次调用时返回新的 Protocol 对象。
`<options>` 都使用可选关键字参数指定：

  - ssl：传递 ssl.SSLContext 对象（或具有相同接口的对象）以覆盖要使用的默认 SSL 上下文对象。（与 create_connection() 不同，这里传递 True 没有意义——需要 SSLContext 对象来指定证书和密钥。）

  - backlog：传递给 listen() 调用的 backlog 值。默认值取决于实现；在默认实现中，默认值是 100。

  - reuse_address：是否在套接字上设置 SO_REUSEADDR 选项。默认值在 UNIX 上为 True，在 Windows 上为 False。

  - family、flags：要传递给 getaddrinfo() 的地址族和标志。family 默认为 AF_UNSPEC；flags 默认为 AI_PASSIVE。（套接字类型始终是 SOCK_STREAM；套接字协议始终设置为 0，让 getaddrinfo() 选择。）

  - sock：可选的套接字，用于代替使用 host、port、family 和 flags 参数。如果给定，host 和 port 必须显式设置为 None。

- `create_datagram_endpoint(protocol_factory, local_addr=None, remote_addr=None, <options>)`。创建用于发送和接收数据报（通常是 UDP 数据包）的端点。由于数据报流量的性质，没有单独的调用来设置客户端和服务端，因为通常单个端点同时充当客户端和服务端。这是一个协程，成功时返回 (transport, protocol) 对，失败时抛出异常。如果协程成功返回，传输将在接收到数据报或套接字关闭时调用协议上的回调；协议负责调用传输方法发送数据报。返回的传输是 DatagramTransport。返回的协议是 DatagramProtocol。这些稍后描述。
强制性位置参数：

  - protocol_factory：类或工厂函数，将不带参数调用一次，构建要返回的协议对象。数据报传输和协议之间的接口在下面描述。

   可选参数，可以位置或关键字参数指定：

  - local_addr：可选元组，指示套接字将绑定的地址。如果给定，这必须是 (host, port) 对。它将传递给 getaddrinfo() 解析，结果将传递给创建的套接字的 bind() 方法。如果 getaddrinfo() 返回多个地址，将依次尝试。如果省略，将不会进行 bind() 调用。

  - remote_addr：可选元组，指示套接字将"连接"到的地址。（由于不存在数据报连接这样的东西，这仅指定传出数据报的目的地址的默认值。）如果给定，这必须是 (host, port) 对。它将传递给 getaddrinfo() 解析，结果将与创建的套接字一起传递给 sock_connect()。如果 getaddrinfo() 返回多个地址，将依次尝试。如果省略，将不会进行 sock_connect() 调用。

  `<options>` 都使用可选关键字参数指定：

  - family、proto、flags：要传递给 getaddrinfo() 的地址族、协议和标志。这些都默认为 0，意思是"未指定"。（套接字类型始终是 SOCK_DGRAM。）如果未指定任何这些值，getaddrinfo() 方法将选择适当的值。

  注意，如果 local_addr 和 remote_addr 都存在，将尝试所有具有匹配地址族的本地和远程地址组合。

#### 包装套接字方法

以下用于套接字异步 I/O 的方法不是通用使用的。它们主要用于通过 ProactorEventLoop 类与 IOCP 一起工作的传输实现。但是，它们很容易为其他事件循环类型实现，所以没有理由不要求它们。socket 参数必须是非阻塞套接字。

- sock_recv(sock, n)。从套接字 sock 接收最多 n 字节。返回一个 Future，其成功结果将是 bytes 对象。

- sock_sendall(sock, data)。发送字节数据到套接字 sock。返回一个 Future，其成功结果将是 None。注意：名称使用 sendall 而不是 send，以反映此方法的语义和签名与标准库套接字方法 sendall() 而不是 send()。

- sock_connect(sock, address)。连接到给定地址。返回一个 Future，其成功结果将是 None。

- sock_accept(sock)。从套接字接受连接。套接字必须处于监听模式并绑定到地址。返回一个 Future，其成功结果将是元组 (conn, peer)，其中 conn 是连接的非阻塞套接字，peer 是对端地址。

#### I/O 回调

这些方法主要用于与选择器一起工作的传输实现。它们由 SelectorEventLoop 实现，但不由 ProactorEventLoop 实现。自定义事件循环实现可能实现也可能不实现它们。

下面的 fd 参数可以是整数文件描述符，或具有 fileno() 方法包装整数文件描述符的"file-like"对象。并非所有file-like 对象或文件描述符都可接受。套接字（和套接字文件描述符）总是被接受。在 Windows 上不支持其他类型。在 UNIX 上，管道和可能的 tty 设备也受到支持，但磁盘文件不支持。确切支持哪些特殊文件类型可能因平台和每个选择器实现而异。（实验性地，OS X 上至少有一种伪 tty 被 select 和 poll 支持但不被 kqueue 支持：它被 Emacs shell 窗口使用。）

- add_reader(fd, callback, *args)。安排当文件描述符 fd 被认为准备好读取时调用 callback(*args)。再次为同一文件描述符调用 add_reader() 意味着为同一文件描述符调用 remove_reader()。

- add_writer(fd, callback, *args)。类似于 add_reader()，但为写入而不是读取注册回调。

- remove_reader(fd)。取消文件描述符 fd 的当前读取回调（如果已设置）。如果文件描述符当前未设置回调，这是空操作并返回 False。否则，它移除回调安排并返回 True。

- remove_writer(fd)。这是 add_writer() 的关系，就像 remove_reader() 是 add_reader() 的关系。

#### 管道和子进程

这些方法由 UNIX 上的 SelectorEventLoop 和 Windows 上的 ProactorEventLoop 支持。

用于管道和子进程的传输和协议与用于常规流连接的传输和协议不同。这些稍后描述。

下面每个方法都有一个 protocol_factory 参数，类似于 create_connection()；这将不带参数调用一次，构建要返回的协议对象。

每个方法是一个协程，成功时返回 (transport, protocol) 对，失败时抛出异常。

- connect_read_pipe(protocol_factory, pipe)：从包装 UNIX 管道读取端的file-like 对象创建单向流连接，必须处于非阻塞模式。返回的传输是 ReadTransport。

- connect_write_pipe(protocol_factory, pipe)：从包装 UNIX 管道写入端的file-like 对象创建单向流连接，必须处于非阻塞模式。返回的传输是 WriteTransport；它没有任何读取相关方法。返回的协议是 BaseProtocol。

- `subprocess_shell(protocol_factory, cmd, <options>)`：从 cmd 创建子进程，cmd 是使用平台"shell"语法的字符串。这类似于标准库 subprocess.Popen() 类，使用 shell=True 调用。其余参数和返回值在下面描述。

- `subprocess_exec(protocol_factory, *args, <options>)`：从一个或多个字符串参数创建子进程，其中第一个字符串指定要执行的程序，其余字符串指定程序的参数。（因此，一起字符串参数形成程序的 sys.argv 值，假设它是 Python 脚本。）这类似于标准库 subprocess.Popen() 类，使用 shell=False 调用，字符串列表作为第一个参数传递；但是，Popen() 接受单个参数（字符串列表），subprocess_exec() 接受多个字符串参数。其余参数和返回值在下面描述。

除了指定要执行的程序的方式外，两个 subprocess_*() 方法行为相同。返回的传输是 SubprocessTransport，它具有不同于常见双向流传输的接口。返回的协议是 SubprocessProtocol，也有自定义接口。

`<options>` 都使用可选关键字参数指定：

- stdin：代表连接到子进程标准输入流的管道的文件-like 对象，使用 connect_write_pipe()，或常量 subprocess.PIPE（默认）。默认情况下将创建新管道并连接。

- stdout：代表连接到子进程标准输出流的管道的文件-like 对象，使用 connect_read_pipe()，或常量 subprocess.PIPE（默认）。默认情况下将创建新管道并连接。

- stderr：代表连接到子进程标准错误流的管道的文件-like 对象，使用 connect_read_pipe()，或常量 subprocess.PIPE（默认）或 subprocess.STDOUT 之一。默认情况下将创建新管道并连接。当指定 subprocess.STDOUT 时，子进程的标准错误流将连接到与标准输出流相同的管道。

- bufsize：创建管道时使用的缓冲区大小；这传递给 subprocess.Popen()。在默认实现中，这默认为零，在 Windows 上它必须为零；这些默认值偏离 subprocess.Popen()。

- executable、preexec_fn、close_fds、cwd、env、startupinfo、creationflags、restore_signals、start_new_session、pass_fds：这些可选参数传递给 subprocess.Popen() 而不解释。

#### 信号回调

这些方法仅在 UNIX 上支持。

- add_signal_handler(sig, callback, *args)。每当接收到信号 sig 时，安排调用 callback(*args)。为同一信号指定另一个回调会替换先前处理程序（每个信号只能有一个活动处理程序）。sig 必须是 signal 模块中定义的有效信号号。如果无法处理信号，这将抛出异常：ValueError 如果它不是有效信号或它是不可捕获的信号（例如 SIGKILL），RuntimeError 如果此特定事件循环实例无法处理信号（由于信号是每进程全局的，只有与主线程关联的事件循环可以处理信号）。

- remove_signal_handler(sig)。移除信号 sig 的处理程序（如果已设置）。抛出与 add_signal_handler() 相同的异常（除了它可能返回 False 而不是对不可捕获信号抛出 RuntimeError）。如果成功移除处理程序，返回 True；如果没有设置处理程序，返回 False。

注意：如果静态已知这些方法不受支持，它们可能抛出 NotImplementedError 而不是 RuntimeError。

### 回调的互斥

事件循环应强制回调互斥，即它绝不应在先前回调仍在运行时开始回调。这应适用于所有类型的回调，无论它们是使用 call_soon()、call_later()、call_at()、call_soon_threadsafe()、add_reader()、add_writer() 还是 add_signal_handler() 计划的。

### 异常

Python 中有两类异常：那些从 Exception 类派生的和那些从 BaseException 派生的。从 Exception 派生的异常通常会被捕获并适当处理；例如，它们将被 Future 传递，当它们出现在回调中时会被记录并忽略。

然而，仅从 BaseException 派生的异常通常不会被捕获，通常会导致程序带跟踪终止。在某些情况下它们被捕获并重新抛出。（此类示例包括 KeyboardInterrupt 和 SystemExit；通常不 wise 将这些与大多数其他异常相同对待。）

事件循环将后一类传递给其异常处理程序。这是一个接受上下文字典作为参数的回调：

```python
def exception_handler(context):
    ...
```

context 可以有许多不同的键，但其中几个非常广泛使用：

- 'message'：错误消息。

- 'exception'：异常实例；如果没有异常则为 None。

- 'source_traceback'：代表涉及错误的对象创建时堆栈的字符串列表。

- 'handle_traceback'：代表涉及错误的 handle 创建时刻堆栈的字符串列表。

循环有以下与异常处理相关的方法：

- get_exception_handler() 返回循环注册当前异常处理程序。

- set_exception_handler(handler) 设置异常处理程序。

- default_exception_handler(context) 此循环实现的默认异常处理程序。

- call_exception_handler(context) 将上下文传递给注册的异常处理程序。这允许第三方库统一处理未捕获的异常。
循环使用 default_exception_handler()，如果默认没有被显式 set_exception_handler() 调用覆盖。

### 调试模式

默认情况下，循环以发布模式运行。应用程序可以启用调试模式以获得更好的错误报告，但会牺牲一些性能。

在调试模式下，启用许多额外的检查，例如：

- 未处理的 Future/任务异常可用源跟踪。

- 循环检查慢回调以检测意外的 I/O 阻塞。
The loop.slow_callback_duration 属性控制两个 yield 点之间允许的最大执行时间，然后报告慢回调。默认值是 0.1 秒；可以通过赋值更改它。

有两个与调试模式相关的方法：

- get_debug() 如果启用调试模式返回 True，否则返回 False。

- set_debug(enabled) 如果参数为 True 则启用调试模式。

如果定义了 PYTHONASYNCIODEBUG 环境变量且不为空，则自动启用调试模式。

### Handles

注册一次性回调的各种方法（call_soon()、call_later()、call_at() 和 call_soon_threadsafe()）都返回一个代表注册的对象，可用于取消回调。此对象称为 Handle。Handles 是不透明的，只有一个公共方法：

- cancel()：取消回调。

注意 add_reader()、add_writer() 和 add_signal_handler() 不返回 Handles。

### Servers

create_server() 方法返回 Server 实例，它包装用于接受请求的套接字（或其他网络对象）。此类有两个公共方法：

- close()：关闭服务。这停止接受新请求，但不取消已经接受并且当前正在处理的请求。

- wait_closed()：协程，阻塞直到服务关闭并且所有接受的请求都已处理。

### Futures

这里的 asyncio.Future 类故意类似于 PEP 3148 指定的 concurrent.futures.Future 类，但有细微差别。每当本 PEP 谈论 Futures 或 futures 时，应理解为指的是 asyncio.Future，除非明确提到 concurrent.futures.Future。支持的公共 API 如下，指示与 PEP 3148 的差异：

- cancel()。如果 Future 已经完成（或已取消），什么都不做并返回 False。否则，这尝试取消 Future 并返回 True。如果取消尝试成功，最终 Future 的状态将变为已取消（因此 cancelled() 将返回 True），并且将计划回调。对于常规 Futures，取消将总是立即成功；但对于 Tasks（见下文），任务可能忽略或延迟取消尝试。

- cancelled()。如果 Future 成功取消，返回 True。

- done()。如果 Future 已完成，返回 True。注意已取消的 Future 也被视为完成（这里和到处）。

- result()。返回用 set_result() 设置的结果，或抛出用 set_exception() 设置的异常。如果已取消，抛出 CancelledError。与 PEP 3148 的差异：这没有 timeout 参数，也不等待；如果 future 尚未完成，它抛出异常。

- exception()。如果设置了用 set_exception() 设置的异常，返回它，或者如果结果用 set_result() 设置，返回 None。如果已取消，抛出 CancelledError。与 PEP 3148 的差异：这没有 timeout 参数，也不等待；如果 future 尚未完成，它抛出异常。

- add_done_callback(fn)。添加一个回调，当 Future 变为完成（或已取消）时运行。如果 Future 已经完成（或已取消），计划使用 call_soon() 调用回调。与 PEP 3148 的差异：回调从不立即调用，总是在调用者的上下文中——通常这是线程。你可以认为这是通过 call_soon() 调用回调。注意，为了匹配 PEP 3148，回调（与本 PEP 中定义的所有其他回调不同，并忽略下面"回调样式"部分中的约定）总是用单个参数（Future 对象）调用。（对用 call_soon() 计划的回调严格序列化的动机也适用于这里。）

- remove_done_callback(fn)。从回调列表中移除参数。此方法未由 PEP 3148 定义。参数必须等于（使用 ==）传递给 add_done_callback() 的参数。返回回调被移除的次数。

- set_result(result)。Future 不能已经完成（或已取消）。这使 Future 完成并计划回调。与 PEP 3148 的差异：这是公共 API。

- set_exception(exception)。Future 不能已经完成（或已取消）。这使 Future 完成并计划回调。与 PEP 3148 的差异：这是公共 API。

内部方法 set_running_or_notify_cancel() 不受支持；没有方法设置运行状态。同样，方法 running() 不受支持。

定义了以下异常：

- InvalidStateError。每当 Future 不处于被调用方法接受的状态时抛出（例如，对已经完成的 Future 调用 set_result()，或对尚未完成的 Future 调用 result()）。

- InvalidTimeoutError。当给出非零 timeout 参数时，由 result() 和 exception() 抛出。

- CancelledError。concurrent.futures.CancelledError 的别名。当对已取消的 Future 调用 result() 或 exception() 时抛出。

- TimeoutError。concurrent.futures.TimeoutError 的别名。可由 run_until_complete() 抛出。

Future 在创建时与事件循环关联。

asyncio.Future 对象不被 concurrent.futures 包中的 wait() 和 as_completed() 函数接受。但是，有类似的 API asyncio.wait() 和 asyncio.as_completed()，下面描述。

asyncio.Future 对象在协程中使用时可被 yield from 表达式接受。这是通过 Future 上的__iter__() 接口实现的。见下面"协程和调度器"部分。

当 Future 被垃圾回收时，如果它有相关异常但 result() 或 exception() 从未被调用，该

异常被记录。（当协程使用 yield from 等待 Future 时，该 Future 的 result() 方法在协程恢复时被调用。）

在未来（双关语），我们可能统一 asyncio.Future 和 concurrent.futures.Future，例如通过向后者添加__iter__() 方法，使其与 yield from 一起工作。为了防止意外地通过调用 result() 阻塞事件循环在尚未完成的 Future 上，阻塞操作可能会检测到当前线程中活动事件循环并转而抛出异常。然而，当前 PEP 力求没有 Python 3.3 之外的依赖项，因此对 concurrent.futures.Future 的更改目前不在考虑之列。

有一些与 Futures 相关的公共函数：

- asyncio.async(arg)。这接受协程对象或 Future（即，任何可以与 yield from 一起使用的东西）作为参数，并返回 Future。如果参数是 Future，它原样返回；如果它是协程对象，它用 Task 包装它（记住 Task 是 Future 的子类）。

- asyncio.wrap_future(future)。这接受 PEP 3148 Future（即，concurrent.futures.Future 的实例）并返回与事件循环兼容的 Future（即，asyncio.Future 实例）。

### 传输

传输和协议深受 Twisted 和 PEP 3153 的影响。用户很少实现或实例化传输——相反，事件循环提供实用方法来设置传输。

传输与协议协同工作。协议通常编写时不知道或不关心使用的确切传输类型，传输可以与多种协议一起使用。例如，HTTP 客户端协议实现可以与普通套接字传输或 SSL/TLS 传输一起使用。普通套接字传输可以与 HTTP 之外的许多不同协议一起使用（例如 SMTP、IMAP、POP、FTP、IRC、SPDY）。

最常见的传输类型是双向流传输。还有单向流传输（用于管道）和数据报传输（由 create_datagram_endpoint() 方法使用）。

#### 所有传输的方法

- get_extra_info(name, default=None)。这是一个万能方法，返回有关传输的实现特定信息。第一个参数是要检索的额外字段名称。可选的第二个参数是要返回的默认值。查阅实现文档以查找支持的额外字段名称。对于不支持的名称，始终返回默认值。

#### 双向流传输

双向流传输是套接字或类似事物（例如，一对 UNIX 管道或 SSL/TLS 连接）之上的抽象。

大多数连接具有不对称性质：客户端和服务器通常具有非常不同的角色和行为。因此，传输和协议之间的接口也是不对称的。从协议的角度来看，写入数据是通过调用传输对象上的 write() 方法完成的；这会缓冲数据并立即返回。但是，传输在读取数据方面采取更积极的角色：每当从套接字（或其他数据源）读取一些数据时，传输调用协议的 data_received() 方法。

尽管如此，双向流使用的传输和协议之间的接口对于客户端和服务器是相同的，因为客户端和服务器之间的连接本质上是一对流，每个方向一个。

双向流传输有以下公共方法：

- write(data)。写入一些字节。参数必须是 bytes 对象。返回 None。传输可以自由缓冲字节，但它最终必须导致字节传输到另一端实体，并且它必须维护流行为。即，t.write(b'abc'); t.write(b'def') 等效于 t.write(b'abcdef')，也等效于：

    ```python
    t.write(b'a')
    t.write(b'b')
    t.write(b'c')
    t.write(b'd')
    t.write(b'e')
    t.write(b'f')
    ```

- writelines(iterable)。等效于：
    ```python
    for data in iterable:
        self.write(data)
    ```

- write_eof()。关闭连接的写入端。不允许后续调用 write()。一旦所有缓冲数据传输，传输向另一端发出信号，表示将不再接收更多数据。某些协议不支持此操作；在这种情况下，调用 write_eof() 将抛出异常。（注意：这以前称为 half_close()，但除非您已经知道它的用途，否则该名称没有指示关闭哪一端。）

- can_write_eof()。如果协议支持 write_eof()，返回 True；如果支持，返回 False。（此方法通常返回仅取决于特定 Transport 类的固定值，而不取决于 Transport 对象的状态。它之所以需要，是因为某些协议需要在 write_eof() 不可用时更改其行为。例如，在 HTTP 中，要发送大小未知的时间，通常使用 write_eof() 指示数据结束；但是，SSL/TLS 不支持此，HTTP 协议实现在这种情况下必须使用"分块"传输编码。但如果数据大小事先已知，两种情况下最佳方法都是使用 Content-Length 标头。）

- get_write_buffer_size()。返回传输写入缓冲区的当前大小（以字节为单位）。这只知道传输显式管理的写入缓冲区；网络栈其他层或其他地方的缓冲不报告。

- set_write_buffer_limits(high=None, low=None)。设置流控制的高水位和低水位限制。这两个值控制何时调用协议的 pause_writing() 和 resume_writing() 方法。如果指定，低水位限制必须小于或等于高水位限制。两个值都不能为负。<br><br>

  默认值取决于实现。如果只给出高水位限制，低水位限制默认为小于或等于高水位限制的特定值。将 high 设置为零也会强制 low 为零，并导致每当缓冲区变为非空时调用 pause_writing()。将 low 设置为零会导致仅在缓冲区为空时调用 resume_writing()。对任一限制使用零通常次优，因为它减少了同时执行 I/O 和计算的机会。

- pause_reading()。暂停向协议传递数据，直到后续 resume_reading() 调用。在 pause_reading() 和 resume_reading() 之间，协议的 data_received() 方法不会被调用。

- resume_reading()。通过 data_received() 重新启动向协议传递数据。注意"暂停"是二进制状态——pause_reading() 应仅在传输未暂停时调用，而 resume_reading() 应仅在传输已暂停时调用。

- close()。与另一端实体的连接断开。write() 缓冲的任何数据最终将在连接实际关闭之前传输。协议的 data_received() 方法不会再被调用。一旦所有缓冲数据已刷新，协议的 connection_lost() 方法将以 None 作为参数被调用。注意此方法不等待所有这些发生。

- abort()。立即断开连接。传输缓冲的任何数据都被丢弃。很快，协议的 connection_lost() 方法将以 None 作为参数被调用。

#### 单向流传输

写入流传输支持为双向流传输描述的 write()、writelines()、write_eof()、can_write_eof()、close() 和 abort() 方法。

读取流传输支持为双向流传输描述的 pause_reading()、resume_reading() 和 close() 方法。

写入流传输仅在其关联协议上调用 connection_made() 和 connection_lost()。

读取流传输可以调用协议部分中指定的所有协议方法（即，前两个加上 data_received() 和 eof_received()）。

#### 数据报传输

数据报传输有以下方法：

- sendto(data, addr=None)。发送数据报（bytes 对象）。可选的第二个参数是目的地址。如果省略，remote_addr 必须在创建此传输的 create_datagram_endpoint() 调用中指定。如果存在，并且指定了 remote_addr，它们必须匹配。(data, addr) 对可以立即发送或缓冲。返回值是 None。

- abort()。立即关闭传输。缓冲的数据将被丢弃。

- close()。关闭传输。缓冲的数据将异步传输。

数据报传输调用关联协议对象上的以下方法：connection_made()、connection_lost()、error_received() 和 datagram_received()。（这些方法名称中的"连接"是轻微的误称，但概念仍然存在：connection_made() 意味着表示端点的传输已创建，connection_lost() 意味着传输已关闭。）

#### 子进程传输

子进程传输有以下方法：

- get_pid()。返回子进程的进程 ID。

- get_returncode()。如果进程已退出，返回进程返回代码；否则返回 None。

- get_pipe_transport(fd)。返回与参数对应的管道传输（单向流传输），参数应为 0、1 或 2，分别代表 stdin、stdout 或 stderr（子进程的）。如果没有这样的管道传输，返回 None。对于 stdin，这是写入传输；对于 stdout 和 stderr，这是读取传输。您必须使用此方法获取可用于向子进程 stdin 写入的传输。

- send_signal(signal)。向子进程发送信号。

- terminate()。终止子进程。

- kill()。杀死子进程。在 Windows 上，这是 terminate() 的别名。

- close()。这是 terminate() 的别名。

注意 send_signal()、terminate() 和 kill() 包装标准库 subprocess 模块中对应方法。

### 协议

协议总是与传输一起使用。虽然提供了一些常用协议（例如，不错但未必优秀的 HTTP 客户端和服务器实现），但大多数协议将由用户代码或第三方库实现。

与传输一样，我们区分流协议、数据报协议，以及可能其他自定义协议。最常见的协议类型是双向流协议。（没有单向协议。）

#### 流协议

（双向）流协议必须实现以下方法，这些方法将由传输调用。将这些视为由事件循环在正确上下文中始终调用的回调。（见上面的"上下文"部分。）

- connection_made(transport)。表示传输已准备好并连接到另一端实体。协议可能应将传输引用保存为实例变量（以便稍后调用其 write() 和其他方法），并可能在此点写入初始问候或请求。

- data_received(data)。传输已从连接读取一些字节。参数总是非空 bytes 对象。对通过此方式传递的数据的最小或最大大小没有保证。`p.data_received(b'abcdef')` 应被视为完全等效于：

    ```python
    p.data_received(b'abc')
    p.data_received(b'def')
    ```

- eof_received()。当另一端调用 write_eof()（或等效操作）时调用。如果返回假值（包括 None），传输将关闭自己。如果它返回真值，关闭传输由协议决定。但是，对于 SSL/TLS 连接，这被忽略，因为 TLS 标准要求不再发送数据，并且一旦收到"关闭警报"就关闭连接。
默认实现返回 None。

- pause_writing()。要求协议暂时停止向传输写入数据。听从请求是可选的，但如果您继续写入，传输的缓冲区可能会无限制增长。调用此的缓冲区大小可以通过传输的 set_write_buffer_limits() 方法控制。

- resume_writing()。告诉协议可以安全地再次向传输写入数据。注意这可能直接由传输的 write() 方法调用（而不是使用 call_soon() 间接调用），以便协议在 write() 返回后立即知道其暂停状态。

- connection_lost(exc)。传输已关闭或中止，检测到另一端已干净地关闭连接，或遇到意外错误。在前三种情况下，参数是 None；对于意外错误，参数是导致传输放弃的异常。

以下是基本调用顺序和多重性的表：

- connection_made() – 正好一次

- data_received() – 零次或多次

- eof_received() – 最多一次

- connection_lost() – 正好一次

对 pause_writing() 和 resume_writing() 的调用成对出现，仅在第 1 和第 4 之间。这些对不会嵌套。最终的 resume_writing() 调用可以省略；即暂停的连接可能丢失且从未恢复。

#### 数据报协议

数据报协议有 connection_made() 和 connection_lost() 方法，具有与流协议相同的签名。（正如关于数据报传输的部分所解释的，我们更喜欢稍微奇怪的术语，而不是定义不同的方法名称来指示套接字的打开和关闭。）

此外，它们有以下方法：

- datagram_received(data, addr)。表示已从远程地址 addr（IPv4 2 元组或 IPv6 4 元组）接收到数据报 data（bytes 对象）。

- error_received(exc)。表示发送或接收操作抛出了 OSError 异常。由于数据报错误可能是暂时的，如果协议想要关闭端点，则由其调用传输的 close() 方法。

以下是调用顺序和多重性的图表：

- connection_made() – 正好一次

- datagram_received()、error_received() – 零次或多次

- connection_lost() – 正好一次

#### 子进程协议

子进程协议有 connection_made()、connection_lost()、pause_writing() 和 resume_writing() 方法，具有与流协议相同的签名。此外，它们有以下方法：

- pipe_data_received(fd, data)。当子进程向其 stdout 或 stderr 写入数据时调用。fd 是文件描述符（1 代表 stdout，2 代表 stderr）。data 是 bytes 对象。

- pipe_connection_lost(fd, exc)。当子进程关闭其 stdin、stdout 或 stderr 时调用。fd 是文件描述符。exc 是异常或 None。

- process_exited()。当子进程退出时调用。要检索退出状态，使用传输的 get_returncode() 方法。

注意，根据子进程的行为，process_exited() 可能在 pipe_connection_lost() 之前或之后调用。例如，如果子进程创建共享其 stdin/stdout/stderr 的子 - 子进程然后自己退出，process_exited() 可能在所有管道仍然打开时调用。另一方面，当子进程关闭其 stdin/stdout/stderr 但不退出时，pipe_connection_lost() 可能为所有三个管道调用而不调用 process_exited()。如果（更常见的情况）子进程退出从而隐式关闭所有管道，调用顺序未定义。

### 回调样式

大多数接受回调的接口也接受位置参数。例如，为了安排调用 foo("abc", 42)，您调用 loop.call_soon(foo, "abc", 42)。要计划调用 foo()，使用 loop.call_soon(foo)。此约定大大减少了典型回调编程所需的小 lambdas 数量。

此约定特别不支持关键字参数。关键字参数用于传递有关回调的可选额外信息。这允许 API 的优雅演变，而无需担心关键字可能在某处对调用者有意义。如果您有一个必须用关键字参数调用的回调，可以使用 lambda。例如：

loop.call_soon(lambda: foo('abc', repeat=42))

## 协程和调度器

这是一个单独的顶级部分，因为其状态不同于事件循环接口。协程的使用是可选的，仅使用回调编写代码完全没问题。另一方面，只有一个调度器/协程 API 实现，如果您使用协程，那就是您使用的一个。

### 协程

协程是遵循某些约定的生成器。出于文档目的，所有协程都应使用&#64;asyncio.coroutine 装饰，但这无法严格强制。

协程使用 PEP 380 引入的 yield from 语法，而不是原始 yield 语法。

"协程"一词，像"生成器"一词一样，用于两个不同（但相关）的概念：

- 定义协程的函数（用 asyncio.coroutine 装饰的函数定义）。如果需要消歧，我们将其称为协程函数。

- 通过调用协程函数获得的对象。此对象表示最终将完成的计算或 I/O 操作（通常是组合）。如果需要消歧，我们将其称为协程对象。

协程可以做的事情：

- result = yield from future – 挂起协程直到 future 完成，然后返回 future 的结果，或抛出异常，这将被传播。（如果 future 被取消，它将抛出 CancelledError 异常。）注意任务是 futures，关于 futures 所说的一切也适用于任务。

- result = yield from coroutine – 等待另一个协程产生结果（或抛出异常，这将被传播）。协程表达式必须是对另一个协程的调用。

- return expression – 使用 yield from 等待此协程的协程产生结果。

- raise exception – 在使用 yield from 等待此协程的协程中抛出异常。

调用协程不会启动其代码运行——它只是一个生成器，调用返回的协程对象实际上是生成器对象，在对其迭代之前不会做任何事情。对于协程对象，有两种基本方法启动它运行：从另一个协程调用 yield from coroutine（假设另一个协程已经在运行！），或将其转换为 Task（见下文）。

协程（和任务）仅当事件循环运行时才能运行。

### 等待多个协程

要等待多个协程或 Futures，提供两个类似于 concurrent.futures 包中的 wait() 和 as_completed() API 的 API：

- asyncio.wait(fs, timeout=None, return_when=ALL_COMPLETED)。这是一个协程，等待 fs 给出的 Futures 或协程完成。协程参数将包装在 Tasks 中（见下文）。这返回一个 Future，其成功结果是两个 Futures 集合的元组，(done, pending)，其中 done 是已完成（或已取消）的原始 Futures（或包装的协程）集合，pending 是其余部分，即那些尚未完成（也未取消）的。注意，timeout 和 return_when 的默认值，done 将始终是空列表。可选参数 timeout 和 return_when 具有与 concurrent.futures.wait() 相同的含义和默认值：timeout，如果不是 None，指定整体操作的超时；return_when，指定何时停止。常量 FIRST_COMPLETED、FIRST_EXCEPTION、ALL_COMPLETED 使用与 PEP 3148 中相同的值和相同的含义定义：
  - ALL_COMPLETED（默认）：等待所有 Futures 完成（或直到发生超时）。

  - FIRST_COMPLETED：等待至少一个 Future 完成（或直到发生超时）。

  - FIRST_EXCEPTION：等待至少一个 Future 完成但未取消且设置了异常。（从条件中排除已取消的 Futures 令人惊讶，但 PEP 3148 就是这样做的。）

- asyncio.as_completed(fs, timeout=None)。返回迭代器，其值是 Futures 或协程；等待连续值等待来自集合 fs 的下一个 Future 或协程完成，并返回其结果（或抛出其异常）。可选参数 timeout 具有与 concurrent.futures.wait() 相同的含义和默认值：当超时时，迭代器返回的下一个 Future 在等待时将抛出 TimeoutError。
使用示例：

    ```python
    for f in as_completed(fs):
        result = yield from f # 可能抛出异常。
        # 使用 result。
    ```

  注意：如果您不等待迭代器产生的值，您的 for 循环可能无法取得进展（因为您不允许其他任务运行）。

- asyncio.wait_for(f, timeout)。这是等待单个协程或 Future 带有超时的便利方法。当超时时，它取消任务并抛出 TimeoutError。为了避免任务取消，用 shield() 包装它。

- asyncio.gather(f1, f2, ...)。返回一个 Future，等待所有参数（Futures 或协程）完成并返回它们对应结果的列表。如果一个或多个参数被取消或抛出异常，返回的 Future 被取消或设置其异常（匹配第一个参数发生的情况），剩余参数在后台继续运行。取消返回的 Future 不影响参数。注意协程参数使用 asyncio.async() 转换为 Futures。

- asyncio.shield(f)。等待 Future，屏蔽其免受取消。这返回一个 Future，其结果或异常与参数完全相同；但是，如果返回的 Future 被取消，参数 Future 不受影响。
此函数的一个用例是缓存协程查询结果的协程，该协程处理 HTTP 服务器中的请求。当请求被客户端取消时，我们可能（可以说）希望查询缓存协程继续运行，这样当客户端重新连接时，查询结果是（希望）缓存的。这可以写成如下：

```python
@asyncio.coroutine
def handle_request(self, request):
    ...
    cached_query = self.get_cache(...)
    if cached_query is None:
        cached_query = yield from asyncio.shield(self.fill_cache(...))
    ...
```

### 睡眠

协程 asyncio.sleep(delay) 在给定时间延迟后返回。

### 任务

Task 是管理独立运行协程的对象。Task 接口与 Future 接口相同，实际上 Task 是 Future 的子类。当协程返回或抛出异常时，任务变为完成；如果它返回结果，那成为任务的结果，如果它抛出异常，那成为任务的异常。

取消尚未完成的任务会向协程抛出 asyncio.CancelledError 异常。如果协程不捕获此（或重新抛出它），任务将被标记为已取消（即，cancelled() 将返回 True）；但是如果协程以某种方式捕获并忽略异常，它可能继续执行（并且 cancelled() 将返回 False）。

任务也用于协程与基于回调的框架（如 Twisted）之间的互操作。将协程转换为 Task 后，可以向任务添加回调。

要将协程转换为任务，调用协程函数并将结果协程对象传递给 loop.create_task() 方法。您也可以为此目的使用 asyncio.ensure_future()。

您可能问，为什么不自动将所有协程转换为 Tasks？&#64;asyncio.coroutine 装饰器可以做到这一点。但是，如果一个协程调用另一个（依此类推），这会大大降低速度，因为切换到"裸"协程的开销比切换到 Task 小得多。

Task 类从 Future 派生，添加新方法：

current_task(loop=None)。类方法，返回事件循环中当前运行的任务。如果 loop 为 None，方法返回默认循环的当前任务。每个协程在任务上下文中执行，要么是使用 ensure_future() 或 loop.create_task() 创建的 Task，要么是通过从另一个协程使用 yield from 或 await 调用。此方法在协程外调用时返回 None，例如在使用 loop.call_later() 计划的回调中。

- all_tasks(loop=None)。类方法，返回循环所有活动任务的集合。如果 loop 为 None，使用默认循环。

### 调度器

调度器没有公共接口。您通过 yield from future 和 yield from task 与之交互。事实上，没有单个对象代表调度器——其行为由 Task 和 Future 类使用事件循环的公共接口实现，因此它也将与第三方事件循环实现一起工作。

### 便利实用程序

提供了一些函数和类来简化基本基于流的客户端和服务器的编写，如 FTP 或 HTTP。这些是：

- asyncio.open_connection(host, port)：EventLoop.create_connection() 的包装器，不要求您提供 Protocol 工厂或类。这是一个协程，返回 (reader, writer) 对，其中 reader 是 StreamReader 的实例，writer 是 StreamWriter 的实例（两者都在下面描述）。

- asyncio.start_server(client_connected_cb, host, port)：EventLoop.create_server() 的包装器，接受简单回调函数而不是 Protocol 工厂或类。这是一个协程，返回 Server 对象，就像 create_server() 一样。每次接受客户端连接，调用 client_connected_cb(reader, writer)，其中 reader 是 StreamReader 的实例，writer 是 StreamWriter 的实例（两者都在下面描述）。如果 client_connected_cb() 返回的结果是协程，它自动包装在 Task 中。

- StreamReader：提供类似于只读二进制流的接口的类，除了各种读取方法是协程。它通常由 StreamReaderProtocol 实例驱动。注意应该只有一个读者。读者接口是：
  - readline()：协程，读取代表以'\n'结尾的文本行的字节字符串，或直到流结束，以先发生者为准。

  - read(n)：协程，读取最多 n 字节。如果 n 省略或为负，它读取直到流结束。

  - readexactly(n)：协程，读取正好 n 字节，或直到流结束，以先发生者为准。

  - exception()：返回使用 set_exception() 在流上设置的异常，或者如果没有设置异常则返回 None。

  驱动程序的接口是：

  - feed_data(data)：将数据（bytes 对象）附加到内部缓冲区。如果它提供足够的数据来满足读者的契约，这将解除阻塞的读取协程。

  - feed_eof()：信号缓冲区结束。这将解除阻塞的读取协程。此调用后不应再向读者提供更多数据。

  - set_exception(exc)：在流上设置异常。所有后续读取方法将抛出此异常。此调用后不应再向读者提供更多数据。

- StreamWriter：提供类似于只写二进制流的接口的类。它包装传输。接口是传输接口的扩展子集：以下方法的行为与相应的传输方法相同：write()、writelines()、write_eof()、can_write_eof()、get_extra_info()、close()。注意写入方法不是协程（这与传输相同，但与 StreamReader 类不同）。以下方法在传输接口之外：
  - drain()：这应在写入重要数据后用 yield from 调用，用于流控制目的。预期用途如下：

    ```python
    writer.write(data)
    yield from writer.drain()
    ```

  注意这技术上不是协程：它返回 Future 或空元组（两者都可以传递给 yield from）。使用此方法是可选的。但是，当未使用时，StreamWriter 底层传输的内部缓冲区可能充满曾经写入到 writer 的所有数据。如果应用程序对其写入的数据量没有严格限制，它应该偶尔调用 `yield from drain()` 以避免填满传输缓冲区。

- StreamReaderProtocol：用作双向流传输/协议接口与 StreamReader 和 StreamWriter 类之间的适配器的协议实现。它充当特定 StreamReader 实例的驱动程序，响应各种协议回调调用其方法 feed_data()、feed_eof() 和 set_exception()。它还控制 StreamWriter 实例的 drain() 方法的行为。

## 同步

模仿 threading 模块中的锁、事件、条件和信号量的锁、事件、条件和信号量已实现，可以通过导入 asyncio.locks 子模块访问。模仿 queue 模块中的队列已实现，可以通过导入 asyncio.queues 子模块访问。

通常这些与它们的线程对应物有密切对应，但是，阻塞方法（例如锁上的 acquire()，队列上的 put() 和 get()）是协程，并且不提供超时参数（不过，您可以使用 asyncio.wait_for() 为阻塞调用添加超时）。

模块中的文档字符串提供更完整的文档。

### 锁

以下类由 asyncio.locks 提供。对于所有这些除了 Event，可以在 with 语句与 yield from 组合使用以获取锁并确保无论 with 块如何离开都释放锁，如下所示：

```python
with (yield from my_lock):
    ...
```

Lock：基本互斥锁，方法有 `acquire()`（协程）、`locked()` 和 `release()`。
Event：事件变量，方法有 `wait()`（协程）、`set()`、`clear()` 和 `is_set()`。
Condition：条件变量，方法有 `acquire()`、`wait()`、`wait_for(predicate)`（三个都是协程）、`locked()`、`release()`、`notify()` 和 `notify_all()`。
Semaphore：信号量，方法有 `acquire()`（协程）、`locked()` 和 `release()`。构造函数参数是初始值（默认 1）。
BoundedSemaphore：有界信号量；这与 Semaphore 类似，但初始值也是最大值。

### 队列

以下类和异常由 asyncio.queues 提供。

- Queue：标准队列，方法有 get()、put()（都是协程）、get_nowait()、put_nowait()、empty()、full()、qsize() 和 maxsize()。

- PriorityQueue：Queue 的子类，按优先级顺序检索条目（最低优先）。

- LifoQueue：Queue 的子类，首先检索最近添加的条目。

- JoinableQueue：具有 task_done() 和 join() 方法（后者是协程）的 Queue 子类。

- Empty、Full：当对空或满的队列调用 get_nowait() 或 put_nowait() 时分别抛出的异常。

### 日志记录

asyncio 包执行的所有日志记录都使用单个 logging.Logger 对象，asyncio.logger。要自定义日志记录，您可以在此对象上使用标准 Logger API。（但请不要替换该对象。）

### UNIX 上的 SIGCHLD 处理

子进程协议上 process_exited() 方法的有效实现需要 SIGCHLD 信号处理程序。但是，信号处理程序只能设置在与主线程关联的事件循环上。为了支持从在其他线程中运行的事件循环生成子进程，存在一种机制允许多个事件循环之间共享 SIGCHLD 处理程序。有两个额外函数，asyncio.get_child_watcher() 和 asyncio.set_child_watcher()，以及事件循环策略上的对应方法。

有两个子进程监视器实现类，FastChildWatcher 和 SafeChildWatcher。两者都使用 SIGCHLD。SafeChildWatcher 类默认使用；当许多子进程同时存在时效率低下。FastChildWatcher 类效率高，但它可能干扰其他代码（C 代码或 Python 代码）不使用 asyncio 事件循环生成子进程。如果您确定不使用其他生成子进程的代码，要使用快速实现，在主线程中运行以下：

```python
watcher = asyncio.FastChildWatcher()
asyncio.set_child_watcher(watcher)
```

## 开放问题

（注意这些实际上已通过 PEP 的接受以现状解决。但是，PEP 的临时状态允许为 Python 3.5 修订这些决定。）

- 为什么 create_connection() 和 create_datagram_endpoint() 有 proto 参数但 create_server() 没有？为什么 getaddrinfo() 的 family、flag、proto 参数有时是零，有时是命名常量（其值也是零）？

- 我们需要另一个查询方法来判断循环是否正在停止过程中？

- Handle 更完整的公共 API？用例是什么？

- 调试 API？例如，记录很多东西的东西，或记录异常条件（如队列填满比排空快）甚至回调花费太多时间……

- 我们需要内省 API 吗？例如，给定文件描述符请求读取回调。或者下一个计划调用何时。或者与回调注册的文件描述符列表。现在这些都需要使用内部。

- 我们需要更多套接字 I/O 方法吗，例如 sock_sendto() 和 sock_recvfrom()，以及可能其他如 pipe_read()？我想用户可以自己编写（这不是火箭科学）。

- 我们可能需要控制各种超时的 API。例如，我们可能想要限制花在 DNS 解析、连接、ssl/tls 握手、空闲连接、关闭/关闭，甚至每会话上的时间。可能为某些方法添加 timeout 关键字参数就足够了，其他超时可能可以通过巧妙使用 call_later() 和 Task.cancel() 实现。但是，某些操作可能需要默认超时，我们可能想全局更改特定操作的默认值（即，每事件循环）。
