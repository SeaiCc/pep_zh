# PEP 263 – 定义 Python 源代码编码

**元数据：**

- **作者**: Marc-André Lemburg <mal at lemburg.com>, Martin von Löwis <martin at v.loewis.de>
- **创建时间**: 06-Jun-2001
- **Python 版本**: 2.3

## 摘要

本 PEP 提议引入一种语法来声明 Python 源文件的编码。编码信息随后将被 Python 解析器用来以指定的编码解释文件。最重要的是，这增强了源代码中 Unicode 字面量的解释，并使得使用例如 UTF-8 直接在支持 Unicode 的编辑器中编写 Unicode 字面量成为可能。

## 问题

在 Python 2.1 中，Unicode 字面量只能使用基于 Latin-1 的编码"unicode-escape"来编写。这使得编程环境对于那些生活在非 Latin-1 区域（如许多亚洲国家）的 Python 用户来说不太友好。程序员可以使用自己喜欢的编码编写 8 位字符串，但对于 Unicode 字面量却局限于"unicode-escape"编码。

## 提议的解决方案

我提议通过使用文件顶部的特殊注释来声明编码，使 Python 源代码编码在每个源文件的基础上既可见又可更改。

为了让 Python 意识到这个编码声明，需要对手持 Python 源代码数据的处理方式做出一些概念上的改变。

## 定义编码

如果没有给出其他编码提示，Python 将默认使用 ASCII 作为标准编码。

要定义源代码编码，必须将魔法注释放置在源文件的第一个或第二个行中，例如：

```python
# coding=<encoding name>
```

或者（使用流行编辑器识别的格式）：

```python
#!/usr/bin/python
# -*- coding: <encoding name> -*-
```

或者：

```python
#!/usr/bin/python
# vim: set fileencoding=<encoding name> :
```

更准确地说，第一行或第二行必须匹配以下正则表达式：

```regex
^[ \t\f]*#.*?coding[:=][ \t]*([-_.a-zA-Z0-9]+)
```

此表达式的第一组随后被解释为编码名称。如果 Python 未知该编码，则在编译期间会引发错误。包含编码声明的行上不得有任何 Python 语句。如果第一行匹配，则第二行被忽略。

为了帮助像 Windows 这样的平台（它会在 Unicode 文件的开头添加 Unicode BOM 标记），UTF-8 签名`\xef\xbb\xbf` 也将被解释为'utf-8'编码（即使没有给出魔法编码注释）。

如果源文件同时使用 UTF-8 BOM 标记签名和魔法编码注释，则注释的唯一允许编码是'utf-8'。任何其他编码都会导致错误。

## 示例

以下是一些示例，用于说明在 Python 源文件顶部定义源代码编码的不同样式：

带有解释器二进制文件并使用 Emacs 风格的文件编码注释：

```python
#!/usr/bin/python
# -*- coding: latin-1 -*-
import os, sys
...
```

```python
#!/usr/bin/python
# -*- coding: iso-8859-15 -*-
import os, sys
...
```

```python
#!/usr/bin/python
# -*- coding: ascii -*-
import os, sys
...
```

没有解释器行，使用纯文本：

```python
# This Python file uses the following encoding: utf-8
import os, sys
...
```

文本编辑器可能有不同的方式来定义文件的编码，例如：

```python
#!/usr/local/bin/python
# coding: latin-1
import os, sys
...
```

没有编码注释，Python 的解析器将假设 ASCII 文本：

```python
#!/usr/local/bin/python
import os, sys
...
```

不起作用的编码注释：

缺少"coding:"前缀：

```python
#!/usr/local/bin/python
# latin-1
import os, sys
...
```

编码注释不在第 1 或第 2 行：

```python
#!/usr/local/bin/python
#
# -*- coding: latin-1 -*-
import os, sys
...
```

不支持的编码：

```python
#!/usr/local/bin/python
# -*- coding: utf-42 -*-
import os, sys
...
```

## 概念

本 PEP 基于以下概念，这些概念必须实现才能启用这种魔法注释的使用：

完整的 Python 源文件应使用单一编码。不允许嵌入不同编码的数据，在 Python 源代码编译期间将导致解码错误。

任何允许按上述方式处理前两行的编码都允许作为源代码编码，这包括 ASCII 兼容编码以及某些多字节编码，如 Shift_JIS。但不包括对所有字符使用两个或更多字节的编码，如 UTF-16。这样做的目的是保持标记器中的编码检测算法简单。

- 转义序列的处理应继续像现在这样工作，但适用于所有可能的源代码编码，即标准字符串字面量（8 位和 Unicode）都受到转义序列扩展的影响，而原始字符串字面量仅扩展非常小的转义序列子集。

- Python 的标记器/编译器组合需要更新，以如下方式工作：
  - 读取文件
  - 假设固定的每文件编码将其解码为 Unicode
  - 将其转换为 UTF-8 字节字符串
  - 对 UTF-8 内容进行标记
  - 编译它，从给定的 Unicode 数据创建 Unicode 对象，并通过首先使用给定的文件编码将 UTF-8 数据重新编码为 8 位字符串数据，从 Unicode 字面量数据创建字符串对象

注意，Python 标识符仅限于编码的 ASCII 子集，因此在第 4 步之后不需要进一步的转换。

## 实现

为了与当前在字符串字面量中使用非 ASCII 而不声明编码的现有代码向后兼容，实现将分两个阶段引入：

- 允许在字符串字面量和注释中使用非 ASCII，通过在内部将缺少的编码声明视为"iso-8859-1"的声明。这将导致任意字节字符串在处理过程中正确地往返于第 2 步和第 5 步之间，并为包含非 ASCII 字节的 Unicode 字面量提供与 Python 2.2 的兼容性。如果发现输入中有非 ASCII 字节，将发出警告，每个编码不正确的输入文件一次。

- 删除警告，并将默认编码更改为"ascii"。

内置的 compile() API 将增强为接受 Unicode 作为输入。8 位字符串输入受上述标准编码检测过程的约束。

如果将带有编码声明的 Unicode 字符串传递给 compile()，将引发 SyntaxError。

SUZUKI Hisao 正在处理补丁；详见 [2]。仅实现第 1 阶段的补丁可在 [1] 获取。

## 阶段

上述步骤 1 和 2 的实现已在 2.3 中完成，除了将默认编码更改为"ascii"。

默认编码在 2.5 版本中设置为"ascii"。

## 范围

本 PEP 旨在提供一条从当前（或多或少）未定义的源代码编码情况到更健壮和可移植的定义的升级路径。
