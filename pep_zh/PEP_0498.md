**元数据：**

- **作者**: Eric V. Smith <eric&#32;&#97;t&#32;trueblade.com>

## 摘要

Python 支持多种格式化文本字符串的方式，包括 %-formatting [1]、str.format() [2] 和 string.Template [3]。每种方法都有其优势，但也存在缺点，使得它们在实际使用中不够方便。本 PEP 提议添加一种新的字符串格式化机制：字面量字符串插值（Literal String Interpolation）。在本 PEP 中，此类字符串被称为"f-strings"，取自用于表示此类字符串的开头字符，代表"formatted strings"（格式化字符串）。

本 PEP 不提议移除或弃用任何现有的字符串格式化机制。

F-strings 提供了一种使用极简语法在字符串字面量中嵌入表达式的方式。需要注意的是，f-string 实际上是在运行时求值的表达式，而不是常量值。在 Python 源代码中，f-string 是以'f'为前缀的字面量字符串，其中包含在花括号内的表达式。这些表达式会被替换为它们的值。一些示例如下：

```python
>>> import datetime
>>> name = 'Fred'
>>> age = 50
>>> anniversary = datetime.date(1991, 10, 12)
>>> f'My name is {name}, my age next year is {age+1}, my anniversary is {anniversary:%A, %B %d, %Y}.'
'My name is Fred, my age next year is 51, my anniversary is Saturday, October 12, 1991.'
>>> f'He said his name is {name!r}.'
"He said his name is 'Fred'."
```

PEP 215 曾提议过类似的功能。PEP 215 提议支持 Python 表达式的子集，但不支持类型特定的字符串格式化（`__format__()` 方法），而该方法是随 PEP 3101 引入的。

## 基本原理

本 PEP 的驱动力是希望拥有一种更简单的方式来格式化 Python 中的字符串。现有的格式化方式要么容易出错，要么不够灵活，要么使用繁琐。

%-formatting 在支持的类型方面有限制。它只支持 int、str 和 double 类型。所有其他类型要么不被支持，要么在格式化之前被转换为这些类型之一。此外，还有一个众所周知的陷阱，即当传递单个值时：

```python
>>> msg = 'disk failure'
>>> 'error: %s' % msg
'error: disk failure'
```

但如果 msg 是一个元组，同样的代码就会失败：

```python
>>> msg = ('disk failure', 32)
>>> 'error: %s' % msg
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: not all arguments converted during string formatting
```

为了防御这种情况，应该使用以下代码：

```python
>>> 'error: %s' % (msg,)
"error: ('disk failure', 32)"
```

str.format() 的添加是为了解决 %-formatting 的一些问题。特别是，它使用普通的函数调用语法（因此支持多个参数），并且可以通过被转换为字符串的对象的 `__format__()` 方法进行扩展。详见 PEP 3101 中的详细说明。本 PEP 重用了 str.format() 的许多语法和机制，以便与现有的 Python 字符串格式化机制保持连续性。

然而，str.format() 也存在问题。其中最主要的是它的冗长。例如，这里的文本值被重复了：

```python
>>> value = 4 * 20
>>> 'The value is {value}.'.format(value=value)
'The value is 80.'
```

即使在最简单的形式下也有一些样板代码，而且插入到占位符中的值有时离占位符很远：

```python
>>> 'The value is {}.'.format(value)
'The value is 80.'
```

使用 f-string，这变成了：

```python
>>> f'The value is {value}.'
'The value is 80.'
```

F-strings 提供了一种简洁、可读的方式来在字符串中包含 Python 表达式的值。

在这个意义上，string.Template 和 %-formatting 有与 str.format() 类似的缺点，但支持的格式化选项更少。特别是，它们不支持__format__协议，因此无法控制特定对象如何转换为字符串，也无法扩展到希望控制自身如何转换为字符串的额外类型（如 Decimal 和 datetime）。以下示例在使用 string.Template 时无法实现：

```python
>>> value = 1234
>>> f'input={value:#06x}'
'input=0x04d2'
```

而 %-formatting 和 string.Template 都无法控制如下格式化：

```python
>>> date = datetime.date(1991, 10, 12)
>>> f'{date} was on a {date:%A}'
'1991-10-12 was on a Saturday'
```

### 不使用 globals() 或 locals()

在 python-dev [4] 的讨论中，提出了许多使用 locals() 和 globals() 或其等效物的解决方案。所有这些方法都有各种问题。其中包括引用闭包中未使用的变量。考虑以下情况：

```python
>>> def outer(x):
...     def inner():
...         return 'x={x}'.format_map(locals())
...     return inner
...
>>> outer(42)()
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
 File "<stdin>", line 3, in inner
KeyError: 'x'
```

这会返回错误，因为编译器没有在闭包中添加对 x 的引用。你需要手动添加对 x 的引用才能使它工作：

```python
>>> def outer(x):
...     def inner():
...         x
...         return 'x={x}'.format_map(locals())
...     return inner
...
>>> outer(42)()
'x=42'
```

此外，使用 locals() 或 globals() 会引入信息泄漏。一个能够访问调用者的 locals() 或 globals() 的被调用例程可以访问远远超出字符串插值所需的信息。

Guido 指出 [5]，任何更好的字符串插值解决方案都不会在其实现中使用 locals() 或 globals()。（这并不禁止用户传入 locals() 或 globals()，只是不要求这样做，也不允许在底层使用这些函数。）

## 规范

在源代码中，f-strings 是以字母'f'或'F'为前缀的字符串字面量。本 PEP 使用'f'的地方，也可以使用'F'。'f'可以与'r'或'R'组合，顺序不限，以生成原始 f-string 字面量。'f'不能与'b'组合：本 PEP 不提议添加二进制 f-strings。'f'不能与'u'组合。

在对源文件进行分词时，f-strings 使用与普通字符串、原始字符串、二进制字符串和三引号字符串相同的规则。也就是说，字符串必须以与开始相同的字符结束：如果它以单引号开始，则必须以单引号结束，等等。这意味着任何当前扫描 Python 代码查找字符串的代码都应该可以轻易修改以识别 f-strings（当然，在 f-string 内部分析是另一回事）。

一旦分词完成，f-strings 就会被解析为字面量字符串和表达式。表达式出现在花括号'{'和'}'内。在扫描字符串查找表达式时，f-string 的字面量部分中的任何双花括号'{{'或'}}'都会被相应的单个花括号替换。双字面量开头花括号并不表示表达式的开始。字面量部分中的单个闭合花括号'}'是一个错误：字面量闭合花括号必须成对出现'}}'才能表示单个闭合花括号。

f-string 中花括号之外的部分是字面量字符串。这些字面量部分随后会被解码。对于非原始 f-strings，这包括将反斜杠转义序列（如`'\n'`、`'\"'`、`"\'"`、`'\xhh'`、`'\uxxxx'`、`'\Uxxxxxxxx'`）和命名 Unicode 字符`'\N{name}'`转换为其关联的 Unicode 字符 [6]。

反斜杠不能出现在表达式中的任何地方。使用'#'字符的注释不允许出现在表达式内。

在每个表达式之后，可以指定可选的类型转换。允许的转换是`'!s'`、`'!r'`或`'!a'`。这些的处理方式与 str.format() 中相同：`'!s'`对表达式调用 str()，`'!r'`对表达式调用 repr()，`'!a'`对表达式调用 ascii()。这些转换在调用 format() 之前应用。使用'!s'的唯一原因是你想要指定适用于 str 而不是表达式类型的格式说明符。

F-strings 使用与 str.format 相同的格式说明符微语言。与 str.format() 类似，可选的格式说明符可以包含在 f-string 中，与表达式（或类型转换，如果指定的话）用冒号分隔。如果未提供格式说明符，则使用空字符串。

因此，f-string 看起来像这样：

```
f '<text> { <expression> <optional !s, !r, or !a> <optional : format specifier> } <text> ... '
```

然后使用`__format__`协议格式化表达式，使用格式说明符作为参数。结果值用于构建 f-string 的值。

注意，`__format__()` 不是直接在每个值上调用的。实际代码使用等效于 `type(value).__format__(value, format_spec)` 或 `format(value, format_spec)` 的代码。详见内置 format() 函数的文档。

表达式不能包含':'或'!'（字符串或括号、方括号、花括号之外的）。例外情况是'!='运算符被允许作为特殊情况。

### 转义序列

反斜杠不能出现在 f-strings 的表达式部分中，因此你不能使用它们来转义 f-strings 中的引号：

```python
>>> f'{\'quoted string\'}'
 File "<stdin>", line 1
SyntaxError: f-string expression part cannot include a backslash
```

你可以在表达式中使用不同类型的引号：

```python
>>> f'{"quoted string"}'
'quoted string'
```

反斜杠转义可以出现在 f-string 的字符串部分中。

注意，要让字面量花括号出现在结果字符串值中的正确方法是加倍花括号：

```python
>>> f'{{ {4*10} }}'
'{ 40 }'
>>> f'{{{4*10}}}'
'{40}'
```

与 Python 中的所有原始字符串一样，原始 f-strings 不执行转义处理：

```python
>>> fr'x={4*10}\n'
'x=40\\n'
```

由于 Python 的字符串分词规则，f-string `f'abc {a['x']} def'`是无效的。分词器将其解析为 3 个 token：f'abc {a['、x 和'}] def'。就像普通字符串一样，这不能通过使用原始字符串来修复。有几种正确编写此 f-string 的方法：使用不同的引号字符：

```python
f"abc {a['x']} def"
```

或者使用三引号：

```python
f'''abc {a['x']} def'''
```

### 代码等价性

实现 f-strings 的确切代码未指定。但是，保证任何嵌入的值转换为字符串时将使用该值的__format__方法。这与 str.format() 用于将值转换为字符串的机制相同。

例如，以下代码：

```python
f'abc{expr1:spec1}{expr2!r:spec2}def{expr3}ghi'
```

可能会被求值为：

```python
'abc' + format(expr1, spec1) + format(repr(expr2), spec2) + 'def' + format(expr3) + 'ghi'
```

### 表达式求值

从字符串中提取的表达式在 f-string 出现的上下文中求值。这意味着表达式可以完全访问局部和全局变量。任何有效的 Python 表达式都可以使用，包括函数和方法调用。

由于 f-strings 在字符串出现在源代码中的位置求值，因此 f-strings 没有额外的表达能力。也没有额外的安全问题：你也可以将相同的表达式写在 f-string 之外：

```python
>>> def foo():
...     return 20
...
>>> f'result={foo()}'
'result=20'
```

等价于：

```python
>>> 'result=' + str(foo())
'result=20'
```

表达式使用等效于 `ast.parse('(' + expression + ')', '<fstring>', 'eval')` [7] 的方式进行解析。

注意，由于表达式在求值前被隐式括号包围，因此表达式可以包含换行符。例如：

```python
>>> x = 0
>>> f'''{x
... +1}'''
'1'

>>> d = {0: 'zero'}
>>> f'''{d[0
... ]}'''
'zero'
```

### 格式说明符

格式说明符也可以包含求值的表达式。这允许如下代码：

```python
>>> width = 10
>>> precision = 4
>>> value = decimal.Decimal('12.34567')
>>> f'result: {value:{width}.{precision}}'
'result: 12.35'
```

一旦格式说明符中的表达式被求值（如果需要），格式说明符就不再被 f-string 求值器解释。就像在 str.format() 中一样，它们只是被传递给被格式化对象的__format__() 方法。

### 连接字符串

相邻的 f-strings 和普通字符串会被连接。普通字符串在编译时连接，而 f-strings 在运行时连接。例如，表达式：

```python
>>> x = 10
>>> y = 'hi'
>>> 'a' 'b' f'{x}' '{c}' f'str<{y:^4}>' 'd' 'e'
```

产生值：

```python
'ab10{c}str< hi >de'
```

虽然这种运行时连接的确切方法未指定，但上面的代码可能会求值为：

```python
'ab' + format(x) + '{c}' + 'str<' + format(y, '^4') + '>de'
```

每个 f-string 在连接到相邻的 f-strings 之前完全求值。这意味着：

```python
>>> f'{x' f'}'
```

是一个语法错误，因为第一个 f-string 不包含闭合花括号。

### 错误处理

处理 f-strings 时可能会发生编译时或运行时错误。编译时错误限于在扫描 f-string 时可以检测到的错误。这些错误都会引发 SyntaxError。

不匹配的花括号：

```python
>>> f'x={x'
 File "<stdin>", line 1
SyntaxError: f-string: expecting '}'
```

无效的表达式：

```python
>>> f'x={!x}'
 File "<stdin>", line 1
SyntaxError: f-string: empty expression not allowed
```

运行时错误发生在求值 f-string 内部的表达式时。注意，f-string 可能被求值多次，有时正常工作，有时引发错误：

```python
>>> d = {0:10, 1:20}
>>> for i in range(3):
...     print(f'{i}:{d[i]}')
...
0:10
1:20
Traceback (most recent call last):
 File "<stdin>", line 2, in <module>
KeyError: 2
```

或：

```python
>>> for x in (32, 100, 'fifty'):
...     print(f'x = {x:+3}')
...
'x = +32'
'x = +100'
Traceback (most recent call last):
 File "<stdin>", line 2, in <module>
ValueError: Sign not allowed in string format specifier
```

### 表达式中的前导和尾随空白被忽略

为了易于阅读，表达式中的前导和尾随空白被忽略。这是在求值前将表达式括在括号中的副产品。

### 表达式的求值顺序

f-string 中的表达式按从左到右的顺序求值。只有当表达式有副作用时才能检测到这一点：

```python
>>> def fn(l, incr):
...     result = l[0]
...     l[0] += incr
...     return result
...
>>> lst = [0]
>>> f'{fn(lst,2)} {fn(lst,3)}'
'0 2'
>>> f'{fn(lst,2)} {fn(lst,3)}'
'5 7'
>>> lst
[10]
```

## 讨论

### python-ideas 讨论

python-ideas [8] 上的大多数讨论集中在三个问题上：

- 如何表示 f-strings，
- 如何在 f-strings 中指定表达式的位置，以及
- 是否允许完整的 Python 表达式。

#### 如何表示 f-strings

由于编译器必须参与求值内插字符串中包含的表达式，因此必须有一种方式来表示哪些字符串应该被求值。本 PEP 选择在字符串字面量之前使用前导'f'字符。这类似于'b'和'r'前缀在编译时改变字符串本身的含义。还建议了其他前缀，如'i'。没有选项似乎比其他选项更好，因此选择了'f'。

另一个选项是支持编译器已知的特殊函数，如 Format()。这对 Python 来说似乎太魔幻了：不仅有机会与现有标识符冲突，PEP 作者认为用字符串前缀字符来表示魔法更好。

#### 如何在 f-strings 中指定表达式的位置

本 PEP 支持与 str.format() 相同的语法来区分字符串内的替换文本：表达式包含在花括号内。还建议了其他选项，如 `string.Template` 的`$identifier` 或`${expression}`。

虽然`$identifier` 无疑对 shell 脚本编写者和某些其他语言的用户来说更熟悉，但在 Python 中 `str.format()` 被广泛使用。对 Python 标准库的快速搜索显示只有几个 `string.Template` 的使用，但数百个 `str.format()` 的使用。

另一个提议的替代方案是将替换文本放在`\{`和`}`之间或`\{`和`\}`之间。虽然这种语法如果所有字符串字面量都支持内插可能很理想，但本 PEP 只支持已经标记有前导'f'的字符串。因此，PEP 使用不带装饰的花括号来表示替换文本，以利用最终用户对 str.format() 的熟悉度。

#### 支持完整的 Python 表达式

python-ideas 讨论中的许多人希望只支持单个标识符或有限的 Python 表达式子集（如 str.format() 支持的子集）。本 PEP 在花括号内支持完整的 Python 表达式。没有完整的表达式，一些期望的用法会很繁琐。例如：

```python
>>> f'Column={col_idx+1}'
>>> f'number of items: {len(items)}'
```

会变成：

```python
>>> col_number = col_idx+1
>>> f'Column={col_number}'
>>> n_items = len(items)
>>> f'number of items: {n_items}'
```

虽然确实非常难看的表达式可以包含在 f-strings 中，但本 PEP 认为这种用法应该在 linter 或代码审查中解决：

```python
>>> f'mapping is { {a:b for (a, b) in ((1, 2), (3, 4))} }'
'mapping is {1: 2, 3: 4}'
```

### 其他语言中的类似支持

Wikipedia 对其他编程语言中的字符串内插有很好的讨论 [9]。这个功能在许多语言中都有实现，具有各种语法和限制。

### f-string 和 str.format 表达式之间的差异

str.format() 中允许的有限表达式与 f-strings 中允许的完整表达式之间有一个小差异。差异在于索引查找的执行方式。在 str.format() 中，看起来不像数字的索引值被转换为字符串：

```python
>>> d = {'a': 10, 'b': 20}
>>> 'a={d[a]}'.format(d=d)
'a=10'
```

注意，索引值在被字典查找时转换为字符串'a'。

然而，在 f-strings 中，你需要使用'a'的值的字面量：

```python
>>> f'a={d["a"]}'
'a=10'
```

这个差异是必需的，否则你将无法使用变量作为索引值：

```python
>>> a = 'b'
>>> f'a={d[a]}'
'a=20'
```

详见 [10] 的进一步讨论。正是这个观察结果导致了 f-strings 中支持完整的 Python 表达式。

此外，str.format() 理解的有限表达式不必是有效的 Python 表达式。例如：

```python
>>> '{i[";]}'.format(i={'";':4})
'4'
```

因此，str.format() 的"表达式解析器"不适合用于实现 f-strings。

### 三引号 f-strings

允许三引号 f-strings。这些字符串的解析方式与普通三引号字符串相同。解析和解码后，应用正常的 f-string 逻辑，并对每个值调用__format__()。

### 原始 f-strings

原始字符串和 f-strings 可以组合。例如，它们可以用来构建正则表达式：

```python
>>> header = 'Subject'
>>> fr'{header}:\s+'
'Subject:\\s+'
```

此外，原始 f-strings 可以与三引号字符串组合。

### 没有二进制 f-strings

与不支持 bytes.format() 的原因相同，你不能将'f'与'b'字符串字面量组合。主要问题是一个对象的__format__() 方法可能返回与 bytes 字符串不兼容的 Unicode 数据。

二进制 f-strings 首先需要 bytes.format() 的解决方案。这个想法过去曾被提出过，最近的一次是在 PEP 461 中。对此类功能的讨论通常建议：

- 添加__bformat__() 之类的方法，以便对象可以控制如何转换为字节，或

- 使 bytes.format() 不如 str.format() 通用或可扩展。

如果希望具有此类功能，这两种选择在未来仍然可用。

### !s、!r 和 !a 是冗余的

!s、!r 和 !a 转换不是严格必需的。因为 f-strings 内允许任意表达式，所以以下代码：

```python
>>> a = 'some string'
>>> f'{a!r}'
"'some string'"
```

等同于：

```python
>>> f'{repr(a)}'
"'some string'"
```

类似地，!s 可以被对 str() 的调用替换，!a 可以被对 ascii() 的调用替换。

然而，本 PEP 支持 !s、!r 和 !a，以最小化与 str.format() 的差异。!s、!r 和 !a 在 str.format() 中是必需的，因为它不允许执行任意表达式。

### 表达式内的 lambdas

由于 lambdas 使用`':'`字符，它们不能出现在表达式中的括号之外。冒号被解释为格式说明符的开始，这意味着 lambda 表达式的开始被看到且语法无效。由于在 f-string 表达式中使用普通 lambda 没有实际用途，这不被视为很大的限制。

如果你必须使用 lambdas，它们可以在括号内使用：

```python
>>> f'{(lambda x: x*2)(3)}'
'6'
```

### 不能与'u'组合

'u'前缀在 PEP 414 中添加到了 Python 3.3 中，作为与 Python 2.7 简化源代码兼容性的手段。因为 Python 2.7 永远不会支持 f-strings，所以能够将'f'前缀与'u'组合没有任何好处。

以下是 Python 源代码中目前使用 str.format() 的一些示例，以及它们使用 f-strings 时会是什么样子。本 PEP 不建议全面转换为 f-strings，这些只是 str.format() 的真实世界用法示例，以及它们如果用 f-strings 从头编写会是什么样子。

`Lib/asyncio/locks.py`:

```python
extra = '{},waiters:{}'.format(extra, len(self._waiters))
extra = f'{extra},waiters:{len(self._waiters)}'
```

`Lib/configparser.py`:

```python
message.append(" [line {0:2d}]".format(lineno))

```

`Tools/clinic/clinic.py`:

methoddef_name = "{}_METHODDEF".format(c_basename.upper())

print("Usage: {0} [{1}]".format(sys.argv[0], '|'.join('--'+opt for opt in valid_opts)), file=sys.stderr)
