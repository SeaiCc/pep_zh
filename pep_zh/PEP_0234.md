# PEP 234 – 迭代器

**Metadata:**

- **Author**: Ka-Ping Yee <ping&#32;&#97;t&#32;zesty.ca>, Guido van Rossum <guido&#32;&#97;t&#32;python.org>
- **Created**: 30-Jan-2001
- **Python-Version**: 2.1
- **Post-History**: 30-Apr-2001

## Abstract

本文档提出了一个迭代接口，对象可以提供该接口来控制 for 循环的行为。通过提供生成迭代器对象的方法来定制循环。迭代器提供一个获取下一个值的操作，每次调用时产生序列中的下一个项目，当没有更多项目可用时引发异常。

此外，还提出了字典键的特定迭代器和文件行的迭代器，并提出允许将 dict.has_key(key) 拼写为 key in dict。

注意：这是第二作者对此 PEP 的几乎完全重写，描述了实际检查到 Python 2.2 CVS 树主干中的实现。它仍然开放讨论。原始版本中一些更晦涩的提案目前已撤回；这些可能是未来单独 PEP 的主题。

## C API Specification

定义了一个新异常 StopIteration，可用于信号迭代结束。

在类型对象结构中添加了一个名为 tp_iter 的新槽，用于请求迭代器。这应该是一个函数，参数为一个 PyObject *，返回一个 PyObject *，或 NULL。要使用此槽，添加了一个新的 C API 函数 PyObject_GetIter()，具有与 tp_iter 槽函数相同的签名。

添加了另一个新槽 tp_iternext 到类型结构，用于获取迭代中的下一个值。要使用此槽，添加了新的 C API 函数 PyIter_Next()。槽和 API 函数的签名如下，尽管 NULL 返回条件不同：参数是 PyObject *，返回值也是 PyObject *。当返回值非 NULL 时，它是迭代中的下一个值。当它为 NULL 时，对于 tp_iternext 槽有三种可能性：

- 没有设置异常；这意味着迭代结束。
- 设置了 StopIteration 异常（或派生异常类）；这意味着迭代结束。
- 设置了其他异常；这意味着发生了应该正常传播的错误。

高级的 PyIter_Next() 函数在发生 StopIteration 异常（或派生异常）时清除它，因此其 NULL 返回条件更简单：

- 没有设置异常；这意味着迭代已结束。
- 设置了某个异常；这意味着发生了错误，应该正常传播。

用 C 实现的迭代器不应该实现与 tp_iternext 槽具有相似语义的 next() 方法！当类型字典初始化时（通过 PyType_Ready()），tp_iternext 槽的存在会导致一个包装该槽的 next() 方法被添加到类型的 tp_dict 中。（例外：如果类型不使用 PyObject_GenericGetAttr() 来访问实例属性，则类型 tp_dict 中的 next() 方法可能看不到。）（由于此 PEP 原始文本中的误解，在 Python 2.2 中，所有迭代器类型都实现了被包装器覆盖的 next() 方法；这已在 Python 2.3 中修复。）

为了确保二进制向后兼容性，在 tp_flags 字段的一组标志和默认标志宏中添加了新标志 Py_TPFLAGS_HAVE_ITER。在访问 tp_iter 或 tp_iternext 槽之前必须测试此标志。宏 PyIter_Check() 测试对象是否设置了适当的标志并且具有非 NULL tp_iternext 槽。对于 tp_iter 槽没有这样的宏（因为引用此槽的唯一位置应该是 PyObject_GetIter()，它可以检查 Py_TPFLAGS_HAVE_ITER 标志直接）。

（注意：tp_iter 槽可以存在于任何对象上；tp_iternext 槽应该只存在于充当迭代器的对象上。）

为了向后兼容，PyObject_GetIter() 函数在其参数是不实现 tp_iter 函数的序列时实现回退语义：在这种情况下构建一个轻量级序列迭代器对象，以自然顺序遍历序列的项目。

为 for 循环生成的 Python 字节码更改为使用新操作码 GET_ITER 和 FOR_ITER，这些操作码使用迭代器协议而不是序列协议来获取循环变量的下一个值。这使得使用 for 循环遍历支持 tp_iter 槽的非序列对象成为可能。解释器遍历序列值的其他位置也应该更改为使用迭代器。

迭代器应该将 tp_iter 槽实现为返回对自身的引用；这是使能够在 for 循环中使用迭代器（而不是序列）所必需的。

迭代器实现（用 C 或 Python）应该保证一旦迭代器信号其耗尽，后续调用 tp_iternext 或 next() 方法将继续这样做。没有规定迭代器是否在引发异常（StopIteration 除外）时进入耗尽状态。注意 Python 不能保证用户定义或第三方迭代器正确实现此要求。

## Python API Specification

StopIteration 异常作为标准异常之一可见。它派生自 Exception。

定义了一个新的内置函数 iter()，可以用两种方式调用：

- iter(obj) 调用 PyObject_GetIter(obj)。
- iter(callable, sentinel) 返回一种特殊类型的迭代器，它调用可调用对象来产生新值，并将返回值与哨兵值比较。如果返回值等于哨兵，则信号迭代结束并引发 StopIteration 而不是正常返回；如果返回值不等于哨兵，则作为迭代器的下一个值返回。如果可调用对象引发异常，则正常传播；特别是，函数可以引发 StopIteration 作为结束迭代的另一种方式。（此功能在 C API 中可用，作为 PyCallIter_New(callable, sentinel)。）

由 iter() 任一形式返回的迭代器对象有一个 next() 方法。此方法要么返回迭代中的下一个值，要么引发 StopIteration（或派生异常类）来信号迭代结束。任何其他异常都应该被视为表示错误并正常传播，而不意味着迭代结束。

类可以通过定义__iter__() 方法来定义如何遍历它们；这不应接受额外参数并返回有效的迭代器对象。想要成为迭代器的类应该实现两个方法：一个如上所述的 next() 方法，以及一个返回 self 的__iter__() 方法。

这两个方法对应两个不同的协议：

- 如果实现__iter__() 或__getitem__()，则对象可以用 for 遍历。
- 如果实现 next()，则对象可以充当迭代器。

容器状对象通常支持协议 1。迭代器目前需要支持两个协议。迭代的语义仅来自协议 2；协议 1 的存在是为了使迭代器表现得像序列；特别是使接收迭代器的代码可以在迭代器上使用 for 循环。

## Dictionary Iterators

- 字典实现一个 sq_contains 槽，实现与 has_key() 方法相同的测试。这意味着我们可以写

```python
if k in dict: ...
```

这等价于

```python
if dict.has_key(k): ...
```

- 字典实现一个 tp_iter 槽，返回一个高效的迭代器，遍历字典的键。在这样的迭代期间，字典不应该被修改，除了设置现有键的值是允许的（不允许删除或添加，也不允许 update() 方法）。这意味着我们可以写

```python
for k in dict: ...
```

这等价于，但比这快得多

```python
for k in dict.keys(): ...
```

只要不违反对字典修改的限制（无论是通过循环还是通过另一个线程）。

- 向字典添加方法，返回不同种类的迭代器：

```python
for key in dict.iterkeys(): ...

for value in dict.itervalues(): ...

for key, value in dict.iteritems(): ...
```

这意味着 for x in dict 是 for x in dict.iterkeys() 的简写。

其他映射，如果它们支持迭代器，也应该遍历键。然而，这不应被视为绝对规则；特定应用可能有不同的要求。

## File Iterators

以下提案有用，因为它为我们提供了一个很好的答案，投诉遍历文件行的常见惯用法既丑陋又慢。

- 文件实现一个 `tp_iter` 槽，等价于 `iter(f.readline, "")`。这意味着我们可以写

```python
for line in file:
 ...
```

作为简写

```python
for line in iter(file.readline, ""):
 ...
```

这等价于，但比这快

```python
while 1:
 line = file.readline()
 if not line:
  break
 ...
```

这也表明一些迭代器是破坏性的：它们消耗所有值，无法轻松创建第二个独立遍历相同值的迭代器。您可以再次打开文件，或 seek() 到开头，但这些解决方案不适用于所有文件类型，例如，当打开的文件对象真正代表管道或流套接字时不起作用。

因为文件迭代器使用内部缓冲区，将此与其他文件操作（例如 file.readline()）混合使用不起作用。另外，以下代码：

```python
for line in file:
 if line == "\n":
  break
for line in file:
 print line,
```

不像您期望的那样工作，因为第二个 for 循环创建的迭代器不考虑第一个 for 循环的缓冲区预读。正确编写此代码的方法是：

```python
it = iter(file)
for line in it:
 if line == "\n":
  break
for line in it:
 print line,
```

（这些限制的理由是 for line in file 应该成为遍历文件行的推荐标准方式，这应该尽可能快。迭代器版本比调用 readline() 快得多，因为迭代器中的内部缓冲区。）

## Rationale

如果包括提案的所有部分，这将以一致和灵活的方式解决许多问题。其主要优点如下四个——不，五个——不，六个——点：

- 它提供了一个可扩展的迭代器接口。
- 它允许列表迭代性能增强。
- 它允许字典迭代性能大幅提升。
- 它允许仅提供迭代接口而不假装提供对元素的随机访问。
- 它与所有现有用户定义类和扩展对象向后兼容，即使只实现 {`__getitem__`, `keys`, `values`, `items`} 子集的映射。
- 它使遍历非序列集合的代码更简洁和可读。

## Resolved Issues

以下主题已通过共识或 BDFL 宣告决定。

- 已提出但拒绝的 next() 两种替代拼写：`__next__()`，因为它对应于类型对象槽（tp_iternext）；和__call__()，因为这是唯一操作。

反对__next__() 的论点：虽然许多迭代器用于 for 循环，但预计用户代码也将直接调用 next()，所以必须写__next__() 很难看；另外，协议的一个可能扩展是允许 prev()、current() 和 reset() 操作；当然我们不希望使用`__prev__()`、`__current__()`、`__reset__()`。

反对__call__() 的论点（原始提案）：脱离上下文，x() 不是很可读，而 x.next() 很清晰；存在每个特殊目的对象都想对最常见操作使用__call__() 的危险，导致更多混乱而非清晰。

（回顾，选择__next__() 并有一个新内置 next(it) 调用`it.__next__()` 可能更好。但可惜，为时已晚；这已在 Python 2.2 中自 2001 年 12 月部署。）

- 一些人要求能够重新启动迭代器。这应该通过反复在序列上调用 iter() 来处理，而不是通过迭代器协议本身。（另见下面请求的扩展。）

- 有人质疑用异常来信号迭代结束是否太昂贵。已提出 StopIteration 异常的几种替代方案：特殊值 End 来信号结束、函数 end() 来测试迭代器是否完成，甚至重用 IndexError 异常。

特殊值的问题是，如果序列包含该特殊值，遍历该序列的循环将提前结束而没有任何警告。如果 C 空终止字符串的经验没有教会我们这可能引起的问题，想象一个 Python 内省工具遍历所有内置名称列表会有多大麻烦，假设特殊 End 值是一个内置名称！

- 调用 end() 函数需要每次迭代两次调用。两次调用比一次调用加异常测试昂贵得多。尤其是时间关键的 for 循环可以非常廉价地测试异常。

- 重用 IndexError 可能引起混淆，因为它可能是真正的错误，会被提前结束循环所掩盖。

- 一些人要求标准迭代器类型。大概所有迭代器都必须从此类型派生。但这不是 Python 的方式：字典是映射因为它们支持__getitem__() 和一些其他操作，而不是因为它们从抽象映射类型派生。

- 关于 if key in dict：毫无疑问，dict.has_key(x) 对 x in dict 的解释是最有用的解释，可能是唯一的有用解释。对此有阻力，因为 x in list 检查 x 是否存在于值中，而提案使 x in dict 检查 x 是否存在于键中。鉴于列表和字典之间的对称性非常弱，此论点没有多少分量。

- 名称 iter() 是缩写。提出的替代方案包括 iterate()、traverse()，但这些似乎太长。Python 有使用常见内置函数缩写的历史，例如 repr()、str()、len()。

决议：iter() 就是它。

- 使用相同名称表示两个不同操作（从对象获取迭代器和为带哨兵值的函数制作迭代器）有点丑陋。我没有看到第二个操作更好的名称，而且因为它们都返回迭代器，很容易记住。

决议：内置 iter() 接受一个可选参数，这是要查找的哨兵。

- 一旦特定迭代器对象引发 StopIteration，它是否也会在所有后续 next() 调用中引发 StopIteration？有人说需要此要求是有用的，其他人说将其留给各个迭代器是开放的。注意这可能需要一些迭代器实现的额外状态位（例如函数包装迭代器）。

决议：一旦引发 StopIteration，调用 it.next() 继续引发 StopIteration。

注意：事实上这在 Python 2.2 中未实现；有许多情况下迭代器的 next() 方法可以在一次调用中引发 StopIteration，但在下一次调用中不引发。这已在 Python 2.3 中得到补救。

- 已提出文件对象应该是自己的迭代器，带有返回下一行的 next() 方法。这有某些优势，并使此迭代器的破坏性更清楚。缺点是这会使实现前一个要点中提出的"sticky StopIteration"功能更加痛苦。<br>
  决议：暂时拒绝（尽管仍然有人为此争论）。<br><br>

- 一些人要求扩展迭代器协议，例如 prev() 获取上一个项目，current() 再次获取当前项目，finished() 测试迭代器是否完成，甚至其他如 `rewind()`、`__len__()`、`position()`。<br>
   虽然其中一些很有用，但许多不能在不添加任意缓冲的情况下轻松为所有迭代器类型实现，有时根本无法实现（或不合理）。例如，涉及反向方向的任何东西在遍历文件或函数时无法完成。也许可以起草单独的 PEP 来标准化这些操作名称（当可实现时）。<br>
   决议：拒绝。<br><br>

- 关于 for x in dict: ... 应该分配 x 字典的连续键、值或项目，已有长时间讨论。if x in y 和 for x in y 之间的对称性表明它应该遍历键。许多独立观察到此对称性，甚至用其中一个"解释"另一个。这是因为对于序列，if x in y 遍历 y 比较迭代值到 x。如果我们采用以上两个提案，这对字典也成立。<br>
使 for x in dict 遍历键的论点主要来自实用性角度：标准库扫描显示 for x in dict.items() 的使用次数与 for x in dict.keys() 差不多，items() 版本有小幅多数。大概许多使用 keys() 的循环无论如何使用相应值，通过写 dict[x]，所以（论点继续）通过使键和值都可用，我们可以支持最多案例。虽然这是真的，我（Guido）发现 for x in dict 和 if x in dict 之间的对应关系太有说服力而不能打破，而且必须写 dict[x] 明确获取值的开销不大。<br>
对于快速遍历项目，使用 for key, value in dict.iteritems()。我计时了
```python
for key in dict: dict[key]
```
和
```python
for key, value in dict.iteritems(): pass
```
之间的差异，发现后者仅快约 7%。

决议：通过 BDFL 宣告，for x in dict 遍历键，字典有 iteritems()、iterkeys() 和 itervalues() 返回不同口味的字典迭代器。
