# PEP 352 – 异常的必要超类

**元数据：**

- **作者**: Brett Cannon, Guido van Rossum
- **创建时间**: 27-Oct-2005
- **Python 版本**: 2.5

## 摘要

在 Python 2.4 及之前，任何（经典）类都可以作为异常被抛出。2.5 的计划是允许新式类，但这使问题更糟——这意味着任何类（或实例）都可以被抛出！这是一个问题，因为它阻止了对异常接口做出任何保证。

本 PEP 提议引入一个新的超类，所有抛出的对象必须继承它。强加这个限制将允许存在一个可以依赖的标准异常接口。它还导致所有异常都遵循一个已知的层次结构。

有人可能会反驳说，要求特定接口使用特定基类是不 Pythonic 的。然而，在异常的特殊情况下，有一个很好的原因（这在 python-dev 上已普遍同意）：要求层次结构有助于想要捕获异常的代码，通过编写 except BaseException: 而不是 except *: 来显式捕获所有异常成为可能。[1]

为异常引入一个新的超类也给了我们机会稍微重新排列异常层次结构以改进它。就目前而言，内置命名空间中的所有异常都继承自 Exception。这是一个问题，因为这包括两个异常（KeyboardInterrupt 和 SystemExit），这些异常通常需要从应用程序的异常处理中除外：关闭解释器而不显示回溯的默认行为通常比应用程序可能做的任何事情都更可取（可能例外的是模拟 Python 交互式命令循环的应用程序，带有>>>提示）。更改它使这两个异常从公共超类继承而不是 Exception 将使人们编写不过度且不会捕获应该向上传播的异常的 except 子句变得容易。

本 PEP 基于 PEP 348 之前的工作。

## 要求公共超类

本 PEP 提议引入一个新异常名为 BaseException，它是一个新式类，有一个单个属性 args。下面是该异常在 Python 3.0 中工作的代码（它在 Python 2.x 中如何工作在过渡计划部分中介绍）：

```python
class BaseException(object):

    """表示异常层次结构基础的超类。

    提供一个'args'属性，包含传递给构造函数的所有参数。
    建议的做法是，只传递单个字符串参数给构造函数。

    """

    def __init__(self, *args):
        self.args = args

    def __str__(self):
        if len(self.args) == 1:
            return str(self.args[0])
        else:
            return str(self.args)

    def __repr__(self):
        return "%s(*%s)" % (self.__class__.__name__, repr(self.args))
```

出于向后兼容的原因，对可以传递给 args 的内容没有限制。但实际上，应该只使用单个字符串参数。这保持了异常的字符串表示为关于异常的有用消息，人类可读；这就是为什么__str__方法对长度 1 的 args 值进行特殊处理。包括程序化信息（例如，错误代码号）应作为子类中的单独属性存储。

raise 语句将被更改，要求传递给它的任何对象必须继承自 BaseException。这将确保所有异常都落在以 BaseException 为锚的单一层次结构中 [1]。这也保证了从 BaseException 继承的基本接口。对 raise 的更改将从 Python 3.0 开始执行（见下面的过渡计划）。

随着 BaseException 成为异常层次结构的根，Exception 现在将继承它。

## 异常层次结构更改

随着异常层次结构现在因为有一个基本根而变得更加重要，需要对现有层次结构进行更改。就目前而言，如果一个人想要捕获所有表示错误且不意味着解释器应该被允许退出的异常，你必须在一个 except 子句中明确指定除两个异常之外的所有异常，或单独捕获这两个异常然后重新抛出它们，让所有其他异常落到一个裸 except 子句中：

```python
except (KeyboardInterrupt, SystemExit):
    raise
except:
    ...
```

这是不必要的显式。本 PEP 提议将 KeyboardInterrupt 和 SystemExit 移动为直接从 BaseException 继承。

```
- BaseException
  |- KeyboardInterrupt
  |- SystemExit
  |- Exception
  |- (所有其他当前内置异常)
```

这样做使得捕获 Exception 更加合理。它只会捕获表示错误的异常。表示解释器应该退出的异常将不会被捕获，因此被允许向上传播并允许解释器终止。

KeyboardInterrupt 已被移动，因为用户通常期望当他们按下中断键（通常是 Ctrl-C）时应用程序退出。如果人们有过于广泛的 except 子句，预期的行为不会发生。

SystemExit 已因类似原因被移动。由于当调用 sys.exit() 时抛出该异常，解释器通常应该被允许终止。不幸的是，过于广泛的 except 子句可以防止明确请求的退出发生。

为了确保人们大多数时候捕获 Exception，文档和教程的各个部分将需要更新，以强烈建议 Exception 应该是程序员想要使用的。裸 except 子句或直接捕获 BaseException 应该被劝阻，基于 KeyboardInterrupt 和 SystemExit 几乎总是应该被允许向上传播的事实。

## 过渡计划

由于正在提议对 Python 进行语义更改，因此需要过渡计划。目标是最终在 Python 3.0 中使用新的语义，同时为 2.x 代码提供平稳过渡。计划中提到的所有弃用都将导致在初始弃用后的版本中移除该语义。

这是在 2.x 系列中实现的 BaseException：

```python
class BaseException(object):

    """表示异常层次结构基础的超类。

    __getitem__方法用于向后兼容，将在某个时候被弃用。
    'message'属性也被弃用。

    """

    def __init__(self, *args):
        self.args = args

    def __str__(self):
        return str(self.args[0]
                   if len(self.args) <= 1
                   else self.args)

    def __repr__(self):
        func_args = repr(self.args) if self.args else "()"
        return self.__class__.__name__ + func_args

    def __getitem__(self, index):
        """索引到实例化期间传递的参数。

        用于向后兼容并将被弃用。

        """
        return self.args[index]

    def _get_message(self):
        """'message'属性的方法。"""
        warnings.warn("the 'message' attribute has been deprecated "
                     "since Python 2.6")
        return self.args[0] if len(args) == 1 else ''

    message = property(_get_message,
                      doc="access the 'message' attribute; "
                          "deprecated and provided only for "
                          "backwards-compatibility")
```

Python 2.9 中功能的弃用是可选的。这是因为目前不知道 Python 2.9（预计是 2.x 系列中的最后一个版本）是否会积极弃用不会在 3.0 中的功能。可以想象，2.9 中将不会使用弃用警告，因为 2.9 和 3.0 之间可能存在如此大的差异，以至于会使 2.9 在警告方面过于"嘈杂"。因此，拟议的 Python 2.9 弃用警告将在该版本开发开始时重新审视，以确定它们是否仍然需要。

  - 所有标准异常成为新式类 [完成]
  - 引入 BaseException [完成]
  - Exception、KeyboardInterrupt 和 SystemExit 继承自 BaseException [完成]
  - 弃用抛出字符串异常 [完成]

- **Python 2.6 [完成]**
  - 弃用捕获字符串异常 [完成]
  - 弃用 message 属性（见撤回的想法）[完成]

- **Python 3.0 [完成]**
  - 删除上面弃用的所有内容：
    - 字符串异常（抛出和捕获）[完成]

    - 删除__getitem__、message [完成]

## 撤回的想法

本 PEP 的以前版本在 Python 2.5 中实现，在 BaseException 上包含一个'message'属性。它的目的是开始向 BaseException 只接受单个参数的过渡。这是为了收紧接口并迫使人们使用子类中的属性来携带任意信息，而不是将它们全部塞入 args。

不幸的是，在 PyCon 2007 冲刺期间实现 Python 3.0 中 args 属性的移除时 [2]，发现过渡非常痛苦，特别是对于 C 扩展模块。决定最好在 Python 2.6 中弃用 message 属性（并在 Python 2.7 和 Python 3.0 中删除它），并考虑在 Python 3.0 中更长期的过渡策略，以删除 BaseException 中的多参数支持，优先接受单个参数。因此，message 的引入和 args 的原始弃用已被撤回。
