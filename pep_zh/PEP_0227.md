# PEP 227 – 静态嵌套作用域

**Metadata:**

- **Author**: Jeremy Hylton <jeremy&#32;&#97;t&#32;alum.mit.edu>
- **Created**: 01-Nov-2000
- **Python-Version**: 2.1

## 摘要

本 PEP 描述了为 Python 2.2 添加静态嵌套作用域（词法作用域），并作为 Python 2.1 的源代码级选项。此外，Python 2.1 将发出关于当启用此功能时含义可能改变的构造的警告。

旧的语言定义（2.0 及之前）定义了恰好三个用于解析名称的命名空间——局部、全局和内建命名空间。嵌套作用域的添加允许在封闭函数的命名空间中解析未绑定的局部名称。

此更改最明显的结果是 lambda（和其他嵌套函数）可以引用在周围命名空间中定义的变量。目前，lambda 通常必须使用默认参数来显式地在 lambda 的命名空间中创建绑定。

## 介绍

本提议改变了 Python 函数中解析自由变量的规则。新的名称解析语义将在 Python 2.2 中生效。通过在模块顶部添加"`from __future__ import nested_scopes`"，这些语义也可以在 Python 2.1 中使用。（参见 PEP 236。）

Python 2.0 定义指定了检查每个名称的恰好三个命名空间——局部命名空间、全局命名空间和内建命名空间。根据此定义，如果函数 A 定义在函数 B 内，B 中绑定的名称在 A 中不可见。本提议改变了规则，使得 B 中绑定的名称在 A 中可见（除非 A 包含隐藏 B 中绑定的名称绑定）。

本规范引入了在 Algol 类语言中常见的词法作用域规则。词法作用域和对一等函数的现有支持的结合让人想起 Scheme。

更改的作用域规则解决了两个问题——lambda 表达式（和嵌套函数）的有限实用性，以及熟悉支持嵌套词法作用域的其他语言的新用户经常产生的混淆，例如无法在模块级别之外定义递归函数。

lambda 表达式产生一个计算单个表达式的未命名函数。它通常用于回调函数。在下面的示例中（使用 Python 2.0 规则编写），lambda 体中使用的任何名称都必须明确作为默认参数传递给 lambda。

```python
from Tkinter import *
root = Tk()
Button(root, text="Click here",
 command=lambda root=root: root.test.configure(text="..."))
```

这种方法很笨拙，特别是当 lambda 体中使用几个名称时。长长的默认参数列表掩盖了代码的目的。提议的解决方案粗略地说自动实现了默认参数方法。"root=root"参数可以省略。

新的名称解析语义将导致一些程序的行为与 Python 2.0 下不同。在某些情况下，程序将编译失败。在其他情况下，以前使用全局命名空间解析的名称将使用封闭函数的局部命名空间解析。在 Python 2.1 中，将对所有行为不同的语句发出警告。

## 规范

Python 是一种具有块结构的静态作用域语言，遵循 Algol 的传统。代码块或区域，如模块、类定义或函数体，是程序的基本单位。

名称引用对象。名称通过名称绑定操作引入。程序文本中名称的每次出现都引用在包含该使用的最内层函数块中建立的那个名称的绑定。

名称绑定操作包括参数声明、赋值、类和函数定义、导入语句、for 语句和 except 子句。每个名称绑定都发生在由类或函数定义或模块级别（顶级代码块）定义的块内。

如果名称在代码块的任何地方绑定，则该块内名称的所有使用都视为对当前块的引用。（注意：当一个名称在绑定之前在块内使用时，这可能导致错误。）

如果 global 语句出现在块内，则语句中指定的名称的所有使用都引用顶级命名空间中那个名称的绑定。通过在内置命名空间（即__builtin__模块的命名空间）中搜索全局命名空间（即包含代码块的模块的命名空间）来在顶级命名空间中解析名称。首先搜索全局命名空间。如果在那里未找到名称，则搜索内置命名空间。global 语句必须位于名称的所有使用之前。

如果名称在代码块内使用，但未在那里绑定且未声明为 global，则该使用视为对最近封闭函数区域的引用。（注意：如果区域包含在类定义内，则类块中发生的名称绑定对封闭函数不可见。）

类定义是一个可执行语句，可以包含名称的使用和定义。这些引用遵循正常的名称解析规则。类定义的命名空间成为类的属性字典。

以下操作是名称绑定操作。如果它们出现在块内，则它们在当前块中引入新的局部名称，除非也有 global 声明。

```
函数定义：def name ...
参数声明：def f(...name...)、lambda ...name...
类定义：class name ...
赋值语句：name = ...
导入语句：import name、import module as name、from module import name
隐式赋值：名称由 for 语句和 except 子句绑定
```

有几种情况，当与包含自由变量的嵌套作用域一起使用时，Python 语句是非法的。

如果在封闭的作用域中引用变量，则删除名称是错误的。编译器将对'del name'引发 SyntaxError。

如果在函数中使用通配符形式的导入（import *），并且函数包含具有自由变量的嵌套块，则编译器将引发 SyntaxError。

如果在函数中使用 exec，并且函数包含具有自由变量的嵌套块，除非 exec 明确指定 exec 的局部命名空间，否则编译器将引发 SyntaxError。（换句话说，"exec obj"将是非法的，但"exec obj in ns"将是合法的。）

如果函数作用域中绑定的名称也是模块全局名称或标准内建名称，并且函数包含引用该名称的嵌套函数作用域，则编译器将发出警告。名称解析规则将在 Python 2.0 下导致与 Python 2.2 下不同的绑定。警告表示程序可能无法在所有版本的 Python 上正确运行。

## 讨论

指定的规则允许在函数中定义的名称在该函数中定义的任何嵌套函数中引用。名称解析规则对于静态作用域语言来说是典型的，有三个主要例外：

- 类作用域中的名称不可访问。

- global 语句短路正常规则。

- 变量未声明。

类作用域中的名称不可访问。名称在最近封闭的函数作用域中解析。如果类定义出现在嵌套作用域链中，则解析过程跳过类定义。此规则防止类属性和局部变量访问之间的奇怪交互。如果名称绑定操作发生在类定义中，它会在结果类对象上创建属性。要在方法中或在方法内嵌套的函数中访问此变量，必须使用属性引用，通过 self 或类名称。

另一种选择是允许类作用域中的名称绑定表现得完全像函数作用域中的名称绑定。此规则将允许通过属性引用或简单名称来引用类属性。此选项被排除，因为它将与所有其他形式的类和实例属性访问不一致，这些访问始终使用属性引用。使用简单名称的代码将是模糊的。

global 语句短路正常规则。根据提议，global 语句的效果与 Python 2.0 完全相同。值得注意的是，它允许在一个块中执行的名称绑定操作更改另一个块（模块）中的绑定。

变量未声明。如果名称绑定操作在函数中的任何地方发生，则该名称被视为局部于函数，所有引用都引用局部绑定。如果在名称绑定之前发生引用，则引发 NameError。唯一的声明类型是 global 语句，它允许使用可变全局变量编写程序。因此，不可能重新绑定在封闭作用域中定义的名称。赋值操作只能在当前作用域或全局作用域中绑定名称。缺少声明和无法在封闭作用域中重新绑定名称对于词法作用域语言来说是不寻常的；通常有创建名称绑定的机制（例如 Scheme 中的 lambda 和 let）和更改绑定的机制（Scheme 中的 set!）。

## 示例

包含几个示例来说明规则的工作原理。

```python
>>> def make_adder(base):
...     def adder(x):
...         return base + x
...     return adder
>>> add5 = make_adder(5)
>>> add5(6)
11
```

```python
>>> def make_fact():
...     def fact(n):
...         if n == 1:
...             return 1L
...         else:
...             return n * fact(n - 1)
...     return fact
>>> fact = make_fact()
>>> fact(7)
5040L
```

```python
>>> def make_wrapper(obj):
...     class Wrapper:
...         def __getattr__(self, attr):
...             if attr[0] != '_':
...                 return getattr(obj, attr)
...             else:
...                 raise AttributeError, attr
...     return Wrapper()
>>> class Test:
...     public = 2
...     _private = 3
>>> w = make_wrapper(Test())
>>> w.public
2
>>> w._private
Traceback (most recent call last):
 File "<stdin>", line 1, in ?
AttributeError: _private
```

Tim Peters 的一个示例展示了在没有声明的情况下嵌套作用域的潜在陷阱：

```python
i = 6
def f(x):
    def g():
        print i
        # ...
        # skip to the next page
        # ...
    for i in x: # ah, i *is* local to f, so this is what g sees
        pass
    g()
```

对 g() 的调用将引用由 for 循环在 f() 中绑定的变量 i。如果在执行循环之前调用 g()，将引发 NameError。

## 向后兼容性

嵌套作用域引起两种类型的兼容性问题。在一种情况下，以前以一种方式运行的代码因为嵌套作用域而表现不同。在其他情况下，某些构造与嵌套作用域交互不良，将在编译时触发 SyntaxError。

Skip Montanaro 的以下示例说明了第一种问题：

```python
x = 1
def f1():
    x = 2
    def inner():
        print x
    inner()
```

在 Python 2.0 规则下，inner() 内的 print 语句引用全局变量 x，如果调用 f1() 将打印 1。在新规则下，它引用 f1() 的命名空间，即最近具有绑定的封闭作用域。

问题仅在全局变量和局部变量共享相同名称并且嵌套函数使用该名称引用全局变量时发生。这是糟糕的编程实践，因为读者很容易混淆这两个不同的变量。在实现嵌套作用域期间，在 Python 标准库中发现了此问题的一个示例。

为了解决这个问题，这个问题不太可能发生，Python 2.1 编译器（当未启用嵌套作用域时）发出警告。

另一个兼容性问题是由在函数体中使用 import * 和'exec'引起的，当该函数包含嵌套作用域并且包含的作用域具有自由变量时。例如：

```python
y = 1
def f():
    exec "y = 'gotcha'" # or from module import *
    def g():
        return y
    ...
```

在编译时，编译器无法确定对局部命名空间操作的 exec 或 import * 是否会引入遮蔽全局 y 的名称绑定。因此，无法确定 g() 中对 y 的引用应该引用全局还是在 f() 中的局部名称。

在 python-list 的讨论中，人们支持两种可能的解释。一方面，一些人认为 g() 中的引用应该绑定到局部 y（如果存在）。此解释的一个问题是，人类代码读者无法通过局部检查确定 y 的绑定。这似乎可能引入微妙的错误。另一种解释是将 exec 和 import * 视为不影响静态作用域的动态特性。在此解释下，exec 和 import * 将引入局部名称，但这些名称永远不会对嵌套作用域可见。在上面的特定示例中，代码的行为将完全像 Python 的早期版本一样。

由于每种解释都有问题并且确切含义模糊，编译器引发异常。Python 2.1 编译器在未启用嵌套作用域时发出警告。

简要审查了三个 Python 项目（标准库、Zope 和 PyXPCOM 的 beta 版本），在大约 200,000 行代码中发现了四个向后兼容性问题。标准库中有一个案例#1 的示例（微妙的行为变化）和两个 import * 问题的示例。

（Python 2.1a2 中实现的 import * 和 exec 限制的解释要严格得多，基于参考手册中从未强制执行的措辞。发布后放宽了这些限制。）

## C API 的兼容性

实现导致几个 Python C API 函数更改，包括 PyCode_New()。因此，C 扩展可能需要更新才能与 Python 2.1 正确配合工作。

## locals() / vars()

这些函数返回包含当前作用域局部变量的字典。对字典的修改不影响变量的值。根据当前规则，使用 locals() 和 globals() 允许程序访问解析名称的所有命名空间。

不会为嵌套作用域提供类似的功能。根据本提议，将无法以字典样式访问所有可见作用域。

## 警告和错误

编译器将在 Python 2.1 中发出警告，以帮助识别可能在 Python 的未来版本中无法编译或正确运行的程序。在 Python 2.2 或 Python 2.1 中使用 nested_scopes future 语句时（在本节中统称为"future 语义"），编译器在某些情况下将引发 SyntaxError。

警告通常适用于包含具有自由变量的嵌套函数的函数。例如，如果函数 F 包含函数 G 并且 G 使用内建 len()，则 F 是包含具有自由变量（len）的嵌套函数（G）的函数。标签"free-in-nested"将用于描述这些函数。

### 在函数作用域中使用 import *

语言参考指定 import * 只能出现在模块作用域中。（第 6.11 节）C Python 的实现支持函数作用域中的 import *。

如果在 free-in-nested 函数的体中使用 import *，编译器将发出警告。根据 future 语义，编译器将引发 SyntaxError。

### 在函数作用域中使用 bare exec

exec 语句允许在关键字"in"后跟两个可选表达式，指定用于局部和全局的命名空间。省略这两个命名空间的 exec 语句是 bare exec。

如果在 free-in-nested 函数的体中使用 bare exec，编译器将发出警告。根据 future 语义，编译器将引发 SyntaxError。

### 局部遮蔽全局

如果 free-in-nested 函数具有局部变量的绑定，该变量 (1) 在嵌套函数中使用且 (2) 与全局变量相同，则编译器将发出警告。

### 在封闭作用域中重新绑定名称

存在一些技术问题使得支持在封闭作用域中重新绑定名称变得困难，但当前提议不允许的主要原因 Guido 反对它。他的动机：它很难支持，因为它需要一个新机制，允许程序员指定块中的赋值应该在封闭块中重新绑定名称； presumably 关键字或特殊语法（x := 3）将使这成为可能。考虑到这将鼓励使用局部变量来存储最好存储在类实例中的状态，不值得添加新语法来实现这一点（根据 Guido 的意见）。

提议的规则允许程序员实现重新绑定的效果，尽管很笨拙。封闭函数将有效重新绑定的名称绑定到容器对象。代替赋值，程序使用对容器的修改来实现所需的效果：

```python
def bank_account(initial_balance):
    balance = [initial_balance]
    def deposit(amount):
        balance[0] = balance[0] + amount
        return balance
    def withdraw(amount):
        balance[0] = balance[0] - amount
        return balance
    return deposit, withdraw
```

对嵌套作用域中重新绑定的支持将使此代码更清晰。定义 deposit() 和 withdraw() 方法并将 balance 作为实例变量的类将更加清晰。由于类似乎以更直接的方式实现相同的效果，因此它们更受青睐。

## 实现

C Python 的实现使用 flat closures [1]。执行的每个 def 或 lambda 表达式都将创建闭包，如果函数体或任何包含的函数具有自由变量。使用 flat closures，闭包的创建有些昂贵，但查找很便宜。

实现添加了几个新的 opcodes 和代码对象中的两种新类型的名称。变量可以是特定代码对象的 cell 变量或 free 变量。cell 变量由包含的作用域引用；因此，定义它的函数必须在每次调用时为它分配单独的存储。free 变量通过函数的闭包引用。

选择 free closures 基于三个因素。首先，假定嵌套函数使用不频繁，深度嵌套（多层嵌套）使用更少。其次，嵌套作用域中名称的查找应该很快。第三，嵌套作用域的使用，特别是返回访问封闭作用域的函数，不应该阻止未引用对象被垃圾回收器回收。
