# PEP 372 – 在 collections 中添加有序字典

**元数据：**

- **作者**: Armin Ronacher <armin.ronacher&#32;&#97;t&#32;active-4.com>, Raymond Hettinger <python&#32;&#97;t&#32;rcn.com>
- **创建时间**: 15-Jun-2008
- **Python 版本**: 2.7, 3.1

## 摘要

本 PEP 提议将有序字典作为 collections 模块的新数据结构，在本 PEP 中称为"OrderedDict"。所提议的 API 结合了从各种现实世界应用和其他编程语言中类似实现中获得的经验。

## 补丁

包括测试和文档的工作 Py3.1 补丁位于：

> [OrderedDict patch](https://github.com/python/cpython/issues/49647)

Check-in 在修订版：70101 和 70102

## 理由

在当前的 Python 版本中，广泛使用的内置 dict 类型不指定存储的键/值对的顺序。这使得很难将字典用作某些特定用例的数据存储。

一些动态编程语言如 PHP 和 Ruby 1.9 保证迭代的一定顺序。在这些语言和现有的 Python ordered-dict 实现中，项的顺序由键的插入时间定义。新键附加在末尾，但被覆盖的键不会移到末尾。

以下示例显示了简单赋值的行为：

```python
>>> d = OrderedDict()
>>> d['parrot'] = 'dead'
>>> d['penguin'] = 'exploded'
>>> d.items()
[('parrot', 'dead'), ('penguin', 'exploded')]
```

顺序的保留使 OrderedDict 对几种情况有用：

- XML/HTML 处理库目前会丢弃属性的顺序，使用列表代替字典（使过滤变得繁琐），或者实现自己的有序字典。这影响了 ElementTree、html5lib、Genshi 和许多其他库。

- 各种库和应用程序中有很多有序字典实现，它们大多微不兼容。此外，子类化 dict 是一项非平凡的任务，许多实现没有正确地重写所有方法，这可能导致意想不到的结果。此外，许多有序字典的实现效率低下，使许多操作比必要的更复杂。

- PEP 3115 允许元类更改用于类体的映射对象。有序字典可用于创建类似于 C 结构体的有序成员声明。例如，这对于未来的 ctypes 发布以及将数据库表定义为类的 ORM（如 Django 框架附带的 ORM）可能很有用。Django 目前使用一个丑陋的技巧来恢复数据库模型中成员的顺序。

- RawConfigParser 类接受 dict_type 参数，允许应用程序设置内部使用的字典类型。添加此参数的动机正是允许用户提供有序字典。[1]

- 从其他编程语言（如 PHP）移植的代码通常依赖于有序字典。在标准库中拥有一个保持顺序的字典实现可以简化过渡并提高不同库的兼容性。

## 有序字典 API

有序字典 API 将主要与 dict 和现有的有序字典兼容。注意：本 PEP 指的是 collections.Mapping 抽象基类中描述的 2.7 和 3.0 字典 API。

构造函数和 update() 都接受元组的可迭代对象，就像 dict 一样。与普通字典不同，插入顺序被保留。

```python
>>> d = OrderedDict([('a', 'b'), ('c', 'd')])
>>> d.update({'foo': 'bar'})
>>> d
collections.OrderedDict([('a', 'b'), ('c', 'd'), ('foo', 'bar')])
```

如果有序字典从普通字典更新，新键的顺序当然未定义。

所有迭代方法以及 keys()、values() 和 items() 按键首次插入的时间返回排序的值：

```python
>>> d['spam'] = 'eggs'
>>> d.keys()
['a', 'c', 'foo', 'spam']
>>> d.values()
['b', 'd', 'bar', 'eggs']
>>> d.items()
[('a', 'b'), ('c', 'd'), ('foo', 'bar'), ('spam', 'eggs')]
```

dict 上不可用的新方法：

**OrderedDict.__reversed__()** 支持按键反向迭代。

## 问题与答案

**如果现有键被重新赋值会发生什么？**

> 键不会移动，而是在原地分配新值。这与现有实现一致。

**如果键在传递给构造函数的列表中出现多次会发生什么？**

> 与普通字典一样——后者项覆盖前者。这具有副作用，即使用第一个键的位置，因为只有值实际上被覆盖：
> ```python
> >>> OrderedDict([('a', 1), ('b', 2), ('a', 3)])
> collections.OrderedDict([('a', 3), ('b', 2)])
> ```
> 这种行为与 Python 中的现有实现、PHP 数组和 Ruby 1.9 中的 hashmap 一致。

**有序字典是 dict 子类吗？为什么？**

> 是的。像 defaultdict 一样，有序字典继承自 dict。作为 dict 子类使某些方法更快（如 `__getitem__` 和 `__len__`）。更重要的是，作为 dict 子类使有序字典可用于像 json 这样的工具，这些工具通过测试 isinstance(d, dict) 坚持需要 dict 输入。

**从子类化 dict 是否会产生任何限制？**

> 是的。由于 dict 的 API 在 Py2.x 和 Py3.x 中不同，OrderedDict API 也必须不同。因此，Py2.7 版本需要重写 iterkeys、itervalues 和 iteritems。

**OrderedDict.popitem() 返回特定的键/值对吗？**

> 是的。它弹出最近插入的新键及其相应值。这对应于传统推/弹出对显示的通常 LIFO 行为。它在语义上等价于 k=list(od)[-1]; v=od[k]; del od[k]; return (k,v)。实际实现更有效，直接从排序键列表弹出。

**OrderedDict 支持索引、切片等吗？**

> 事实上，OrderedDict 不实现 Sequence 接口。相反，它是一个记住键插入顺序的 MutableMapping。唯一的类似序列的添加是对 reversed 的支持。
> 不允许索引的另一个优势是它为使用链表快速 C 实现留下了可能性。

**OrderedDict 支持字母顺序等替代排序顺序吗？**

> 不。那些想要不同排序顺序的人确实需要使用另一种技术。OrderedDict 完全是关于记录插入顺序。如果任何其他顺序感兴趣，那么另一种结构（如内存 dbm）可能是更好的选择。

**OrderedDict 与 json 模块、PyYAML 和 ConfigParser 配合得如何？**

> 对于 json，好消息是 json 的编码器尊重 OrderedDict 的迭代顺序：
> ```python
> >>> items = [('one', 1), ('two', 2), ('three',3), ('four',4), ('five',5)]
> >>> json.dumps(OrderedDict(items))
> '{"one": 1, "two": 2, "three": 3, "four": 4, "five": 5}'
> ```
> 
> 在 Py2.6 中，json 解码器的 object_hook 传递一个已经构建的字典，因此在对象钩子看到它之前顺序会丢失。这个问题正在通过添加新钩子来修复 Python 2.7/3.1，该钩子保留顺序（参见 https://github.com/python/cpython/issues/49631）。使用新钩子，顺序可以保留：
> ```python
> >>> jtext = '{"one": 1, "two": 2, "three": 3, "four": 4, "five": 5}'
> >>> json.loads(jtext, object_pairs_hook=OrderedDict)
> OrderedDict({'one': 1, 'two': 2, 'three': 3, 'four': 4, 'five': 5})
> ```
> 
> 对于 PyYAML，完整往返没有问题：
> ```python
> >>> ytext = yaml.dump(OrderedDict(items))
> >>> print ytext
> !!python/object/apply:collections.OrderedDict
> - - [one, 1]
>   - [two, 2]
>   - [three, 3]
>   - [four, 4]
>   - [five, 5]
> 
> >>> yaml.load(ytext)
> OrderedDict({'one': 1, 'two': 2, 'three': 3, 'four': 4, 'five': 5})
> ```
> 
> 对于 ConfigParser 模块，往返也没有问题。自定义字典在 Py2.6 中添加专门用于支持有序字典：
> ```python
> >>> config = ConfigParser(dict_type=OrderedDict)
> >>> config.read('myconfig.ini')
> >>> config.remove_option('Log', 'error')
> >>> config.write(open('myconfig.ini', 'w'))
> ```

**OrderedDict 如何处理相等性测试？**

比较两个有序字典意味着测试将对顺序敏感，使得 `list(od1.items())==list(od2.items())`。当有序字典与其他 Mappings 比较时，使用它们不敏感的顺序比较。这允许有序字典可以在使用普通字典的任何地方替换。

**__repr__ 格式如何在 repr/eval 往返期间保持顺序？**

> OrderedDict([('a', 1), ('b', 2)])

**可能的底层数据结构的权衡是什么？**

> - 保持排序键列表对于所有操作都很快，除了 `__delitem__()` 成为 O(n) 练习。这种数据结构导致非常简单的代码和很少浪费的空间。
> 
> - 保持单独的字典来记录插入序列号使代码稍微复杂一些。所有基本操作都是 O(1)，但常量因子对于 `__setitem__()` 和 `__delitem__()` 增加了，意味着每个用例都必须为此加速付费（因为所有构建都通过 `__setitem__`）。此外，第一次遍历产生一次性的 O(n log n) 排序成本。存储成本是排序键列表方法的两倍。
> 
> - 用 C 编写的版本可以使用链表。代码将比其他两种方法更复杂，但它将节省空间并保持与普通字典相同的大 O 性能。它是最快和最节省空间的。

## 参考实现

带有测试和文档的实现位于：

> [OrderedDict patch](https://github.com/python/cpython/issues/49647)

提议的版本有几个优点：

- 严格遵守 MutableMapping API，没有新方法，因此学习曲线几乎为零。它只是一个记住插入顺序的字典。

- 通常良好的性能。大 O 时间与普通字典相同，除了键删除是 O(n)。

其他在 various Python 项目或独立库中的有序字典实现，启发了这里提出的 API，包括：

- odict in Python
- odict in Babel
- OrderedDict in Django
- odict 模块
- ordereddict（odict 模块的 C 实现）
- StableDict
- Armin Rigo 的 OrderedDict

## 未来方向

随着标准库中有序字典的可用性，其他库可以利用这一点。例如，ElementTree 将来可以返回 odicts，保留源文件的属性顺序。
