## 摘要

本 PEP 提议引入一种新语法 `/`，用于在 Python 函数定义中指定位置参数（positional-only parameters）。

位置参数没有外部可用的名称。当调用接受位置参数的函数时，位置参数仅根据其顺序映射到这些参数。

在设计 API（应用程序编程接口）时，库作者试图确保 API 的正确和预期使用。如果没有能力指定哪些参数是位置参数，库作者在选择适当的参数名称时必须格外小心。即使对于必需参数，或者当参数对 API 调用者没有外部语义意义时，也必须采取这种谨慎。

在本 PEP 中，我们讨论：

- Python 位置参数的历史和当前语义

- 没有位置参数所遇到的问题

- 在没有语言原生支持的情况下如何处理这些问题

- 拥有位置参数的好处

在动机背景下，我们接着：

- 讨论为什么位置参数应该是语言内在特性

- 提出标记位置参数的语法

- 展示如何教授这个新特性

- 详细说明被拒绝的想法

## 动机

### Python 中位置参数语义的历史

Python 最初支持位置参数。早期版本的语言缺乏通过名称将参数绑定到函数的能力。大约在 Python 1.0 时，参数语义变为位置或关键字参数。从那时起，用户可以通过位置或函数定义中指定的关键字名称向函数提供参数。

在当前的 Python 版本中，许多 CPython "内置"和标准库函数只接受位置参数。通过使用关键字参数调用这些函数之一，可以轻松观察到结果语义：

```python
>>> help(pow)
...
pow(x, y, z=None, /)
...

>>> pow(x=5, y=3)
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: pow() takes no keyword arguments
```

pow() 通过 `/` 标记表达其参数是位置参数。然而，这只是一个文档约定；Python 开发人员不能在代码中使用此语法。

还有一些具有其他有趣语义的函数：

- range()，一个重载函数，在其必需参数的左侧接受一个可选参数。[4]

- dict()，其映射/迭代器参数是可选的，且语义上必须是位置参数。这个参数的任何外部可见名称都会掩盖进入 `**kwarg` 关键字变长参数 `dict` 的名称。[3]

可以通过接受 `(*args, **kwargs)` 并手动解析参数来在 Python 代码中模拟这些语义。然而，这导致函数定义与函数契约上接受的内容之间存在脱节。函数定义与参数处理逻辑不匹配。

此外，`/` 语法在 CPython 之外也用于指定类似的语义（即，[1] [2]）；因此，表明这些场景并非 CPython 和标准库独有。

### 没有位置参数的问题

没有位置参数，库作者和 API 用户都面临挑战。以下小节概述了各方遇到的问题。

#### 库作者面临的挑战

使用位置或关键字参数时，调用约定的混合并不总是可取的。作者可能希望通过不允许使用关键字参数调用 API 来限制 API 的使用，这会将参数名称暴露为公共 API 的一部分。这种方法对于已经具有语义意义的必需函数参数特别有用（例如，`namedtuple(typenames, field_names, …)`），或者当参数名称没有真正的外部意义时（例如，`min()` 的 `arg1`、`arg2`、…等）。如果 API 的调用者开始使用关键字参数，库作者就不能重命名参数，因为这将是破坏性的更改。

可以通过从 `*args` 中逐个提取参数来模拟位置参数。然而，如前所述，这种方法容易出错，并且与函数定义不同义。函数的使用不明确，迫使用户查看 `help()`、相关的自动生成的文档或源代码以理解函数契约上接受哪些参数。

#### API 用户面临的挑战

用户第一次遇到位置参数表示法时可能会感到惊讶。这是可以预料的，因为它最近才被记录在案 [13]，并且不能在 Python 代码中使用。由于这些原因，这种表示法目前是一个异常，只出现在用 C 开发的 CPython API 中。记录这种表示法并使其可以在 Python 代码中使用将消除这种脱节。

此外，当前关于位置参数的文档不一致：

- 有些函数通过嵌套方括号括起来来表示可选的位置参数组。[5]

- 有些函数通过提供具有不同数量参数的多个原型来表示可选的位置参数组。[6]

- 有些函数同时使用上述两种方法。[4] [7]

当前文档没有区分的另一个点是函数是否接受位置参数。`open()` 接受关键字参数；然而，`ord()` 不接受——仅通过阅读现有文档无法判断。

### 位置参数的好处

位置参数赋予库作者更多的控制权，以更好地表达 API 的预期用途，并允许 API 以安全、向后兼容的方式演进。此外，它使 Python 语言与现有文档以及各种"内置"和标准库函数的行为更加一致。

#### 增强库作者的能力

库作者将能够灵活地更改位置参数的名称而不会破坏调用者。这种灵活性减少了为必需参数或没有真正外部语义意义的参数选择适当的公共名称的认知负担。

位置参数在几种情况下很有用，例如：

- 当函数接受任何关键字参数，但也接受一个位置参数时

- 当参数没有外部语义意义时

- 当 API 的参数是必需的且对函数而言明确时

一个关键场景是函数接受任何关键字参数，但也接受一个位置参数。突出的例子是 `Formatter.format` 和 `dict.update`。例如，`dict.update` 接受一个字典（位置）、一系列键/值对（位置），或者多个关键字参数。在这种情况下，如果字典参数不是位置参数，用户就不能使用函数定义中用于该参数的名称，或者反过来，函数无法轻松区分接收到的参数是字典/可迭代对象还是用于更新键/值对的关键字参数。

另一个位置参数有用的场景是当参数名称没有真正的外部语义意义时。例如，假设我们要创建一个将一个类型转换为另一个类型的函数：

```python
def as_my_type(x):
 ...
```

参数名称没有提供任何内在价值，并迫使 API 作者永远维护其名称，因为调用者可能会将 `x` 作为关键字参数传递。

此外，当 API 的参数是必需的且相对于函数明确时，位置参数也很有用。例如：

```python
def add_to_queue(item: QueueItem):
 ...
```

函数名称清楚地表明了预期的参数。关键字参数提供的利益很小，也限制了 API 的未来演进。假设在以后的某个时候，我们希望这个函数能够接受多个项目，同时保持向后兼容性：

```python
def add_to_queue(items: Union[QueueItem, List[QueueItem]]):
 ...
```

或者通过使用参数列表来接受它们：

```python
def add_to_queue(*items: QueueItem):
 ...
```

作者将不得不始终保留原始参数名称，以避免可能破坏调用者。

通过能够指定位置参数，作者可以自由地更改参数名称，甚至将其更改为 `*args`，如前面的示例所示。标准库中有多个函数定义属于此类。例如，`collections.defaultdict` 的必需参数（在其文档中称为 `default_factory`）只能按位置传递。这种情况的一个特例是类方法的 `self` 参数：调用者从类调用方法时，通过关键字绑定到名称 `self` 是不理想的：

```python
io.FileIO.write(self=f, b=b"data")
```

事实上，标准库中用 C 实现的函数定义通常将 `self` 作为位置参数接受：

```python
>>> help(io.FileIO.write)
Help on method_descriptor:

write(self, b, /)
 Write buffer b to file, return number of bytes written.
```

#### 提高语言一致性

Python 语言将因为位置参数而更加一致。如果这个概念是 Python 的正常特性，而不是扩展模块独有的特性，它将减少用户遇到具有位置参数的函数时的困惑。一些主要的第三方包已经在函数定义中使用 `/` 表示法 [1] [2]。

弥合指定位置参数的"内置"函数和缺乏位置语法的纯 Python 实现之间的差距将提高一致性。`/` 语法已经暴露在现有文档中，例如当内置函数和接口由参数诊所生成时。

另一个需要考虑的重要方面是 PEP 399，它规定标准库中模块的纯 Python 版本必须具有与 C 实现的加速器模块相同的接口和语义。例如，如果 `collections.defaultdict` 有一个纯 Python 实现，它需要使用位置参数来匹配其 C 对应物的接口。

## 基本原理

我们提议将位置参数作为 Python 语言的新语法引入。

新语法将使库作者能够进一步控制其 API 的调用方式。它将允许指定哪些参数必须作为位置参数调用，同时防止它们作为关键字参数调用。

以前，（信息性）PEP 457 定义了语法，但范围要模糊得多。本 PEP 将原始提案向前推进一步，通过论证语法并为函数定义中的 `/` 语法提供实现。

### 性能

除了上述好处之外，位置参数的解析和处理也更快。这种性能优势可以在关于将关键字参数转换为位置的这个讨论中看到：[11]。由于这种加速，最近有一种趋势是将内置函数从关键字参数中移开：最近，进行了向后不兼容的更改，以禁止向 `bool`、`float`、`list`、`int`、`tuple` 传递关键字参数。

### 可维护性

在 Python 中提供一种指定位置参数的方式将使维护 C 模块的纯 Python 实现变得更加容易。此外，定义函数的库作者可以选择使用位置参数，如果他们确定传递关键字参数不会提供额外的清晰度。

### 逻辑顺序

位置参数还具有（次要）好处，即在使用它们的接口调用时强制某些逻辑顺序。例如，`range` 函数按位置接受所有参数，并禁止以下形式：

```python
range(stop=5, start=0, step=2)
range(stop=5, step=2, start=0)
range(step=2, start=0, stop=5)
range(step=2, stop=5, start=0)
```

代价是禁止使用关键字参数用于（唯一的）预期顺序：

```python
range(start=0, stop=5, step=2)
```

### 纯 Python 和 C 模块的兼容性

位置参数的另一个关键动机是 PEP 399：纯 Python/C 加速器模块兼容性要求。该 PEP 规定：

> 本 PEP 要求在这些情况下，C 代码必须通过纯 Python 代码使用的测试套件，尽可能像一个插入式替换

如果 C 代码使用现有能力通过参数诊所和相关机器实现位置参数，那么纯 Python 对应物无法匹配提供的接口和要求。这在 CPython 标准库中的一些函数和类与其他 Python 实现的接口之间造成了差异。例如：

```python
$ python3 # CPython 3.7.2
>>> import binascii; binascii.crc32(data=b'data')
TypeError: crc32() takes no keyword arguments

$ pypy3 # PyPy 6.0.0
>>> import binascii; binascii.crc32(data=b'data')
2918445923
```

其他 Python 实现可以手动重现 CPython API，但这违背了 PEP 399 的精神，即通过强制所有添加到 Python 标准库的模块必须具有相同接口和语义的纯 Python 实现来避免重复工作。

### 子类中的一致性

位置参数提供好处的另一个场景是当子类重写基类的方法并更改旨在位置参数的参数名称时：

```python
class Base:
 def meth(self, arg: int) -> str:
 ...

class Sub(Base):
 def meth(self, other_arg: int) -> str:
 ...

def func(x: Base):
 x.meth(arg=12)

func(Sub()) # 运行时错误
```

这种情况可以被视为 Liskov 违规——子类不能在期望基类实例的上下文中使用。重载方法时重命名参数可能发生在子类有理由为参数名称使用更适合子类特定领域的不同选择时（例如，当子类化 `Mapping` 来实现 DNS 查找缓存时，派生类可能不想使用通用参数名称'key'和'value'，而更愿意使用'host'和'address'）。使用具有位置参数的函数定义可以避免这个问题，因为用户将无法使用关键字参数调用接口。一般来说，设计子类化通常涉及预见尚未编写的代码，作者对其没有控制权。采取措施促进接口以向后兼容的方式演进将对库作者有用。

### 优化

支持位置参数的最后一个论点是，它们允许一些新的优化，就像参数诊所中已经存在的优化一样，因为参数期望按严格顺序传递。例如，CPython 的内部 `METH_FASTCALL` 调用约定最近已针对具有位置参数的函数进行专门化，以消除处理空关键字的成本。由于位置参数，在创建 Python 函数的评估帧时也可以应用类似的性能改进。

## 规范

### 语法和语义

从"万英尺视角"来看，省略 `*args` 和 `**kwargs` 以进行说明，函数定义的语法如下：

```python
def name(positional_or_keyword_parameters, *, keyword_only_parameters):
```

在此基础上，函数定义的新语法如下：

```python
def name(positional_only_parameters, /, positional_or_keyword_parameters, *, keyword_only_parameters):
```

以下规则适用：

- `/` 左侧的所有参数被视为位置参数。

- 如果函数定义中未指定 `/`，则该函数不接受任何位置参数。

- 位置参数可选值周围的逻辑与位置或关键字参数相同。

- 一旦位置参数指定了默认值，后续的位置参数和位置或关键字参数也需要具有默认值。

- 没有默认值的位置参数是必需的位置参数。

因此，以下是有效的函数定义：

```python
def name(p1, p2, /, p_or_kw, *, kw):
def name(p1, p2=None, /, p_or_kw=None, *, kw):
def name(p1, p2=None, /, *, kw):
def name(p1, p2=None, /):
def name(p1, p2, /, p_or_kw):
def name(p1, p2, /):
```

就像今天一样，以下是有效的函数定义：

```python
def name(p_or_kw, *, kw):
def name(*, kw):
```

而以下则是无效的：

```python
def name(p1, p2=None, /, p_or_kw, *, kw):
def name(p1=None, p2, /, p_or_kw=None, *, kw):
def name(p1=None, p2, /):
```

### 语义边缘情况

以下是规范的一个有趣推论。考虑这个函数定义：

```python
def foo(name, **kwds):
    return 'name' in kwds
```

没有可能的调用会使其返回 True。例如：

```python
>>> foo(1, **{'name': 2})
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: foo() got multiple values for argument 'name'
>>>
```

但使用 `/` 我们可以支持这个：

```python
def foo(name, /, **kwds):
    return 'name' in kwds
```

现在上面的调用将返回 True。

换句话说，位置参数的名称可以在 `**kwds` 中使用而不产生歧义。（另一个例子，这有利于 `dict()` 和 `dict.update()` 的签名。）

### ""/""作为分隔符的起源

使用 `/` 作为分隔符最初由 Guido van Rossum 于 2012 年提出 [8]：

> 替代方案：如何使用 '/'？它在某种意义上是 `'*'` 的反义词，`'*'` 表示"关键字参数"，而 '/' 不是一个新字符。

## 如何教授这个

引入专用语法来标记位置参数与现有的仅关键字参数密切相关。一起教授这些概念可能会简化如何教授用户可能遇到或设计的可能的函数定义。

本 PEP 建议在 Python 文档中添加一个新的小节，在"更多关于定义函数"部分中，其余的参数类型在那里讨论。以下段落作为这些添加的草案。它们将介绍位置参数和仅关键字参数的表示法。它不打算详尽无遗，也不应被视为要纳入文档的最终版本。

默认情况下，参数可以通过位置或显式通过关键字传递给 Python 函数。为了可读性和性能，限制参数传递的方式是有意义的，这样开发者只需要查看函数定义就可以确定项目是按位置传递、按位置或关键字传递，还是按关键字传递。

函数定义可能如下所示：

```python
def f(pos1, pos2, /, pos_or_kwd, *, kwd1, kwd2):
      -----------    ----------     ----------
        |             |                  |
        |        Positional or keyword   |
        |                                - Keyword only
         -- Positional only
```

其中 `/` 和 `*` 是可选的。如果使用，这些符号通过参数如何传递给函数来指示参数类型：位置参数、位置或关键字参数，以及仅关键字参数。关键字参数也称为命名参数。

### 位置或关键字参数

如果 `/` 和 `*` 不在函数定义中，参数可以通过位置或关键字传递给函数。

### 位置参数

更详细地看，可以将某些参数标记为位置参数。如果是位置参数，参数的顺序很重要，并且不能通过关键字传递参数。位置参数将放在 `/`（正斜杠）之前。`/` 用于逻辑上将位置参数与其他参数分开。如果函数定义中没有 `/`，则没有位置参数。

`/` 之后的参数可以是位置或关键字或仅关键字。

### 仅关键字参数

要将参数标记为仅关键字，表示参数必须通过关键字参数传递，请在第一个仅关键字参数之前在参数列表中放置 `*`。

### 函数示例

考虑以下示例函数定义，注意标记 `/` 和 `*`：

```python
>>> def standard_arg(arg):
...     print(arg)
...
>>> def pos_only_arg(arg, /):
...     print(arg)
...
>>> def kwd_only_arg(*, arg):
...     print(arg)
...
>>> def combined_example(pos_only, /, standard, *, kwd_only):
...     print(pos_only, standard, kwd_only)
```

第一个函数定义 `standard_arg`，最熟悉的形式，对调用约定不放置任何限制，参数可以通过位置或关键字传递：

```python
>>> standard_arg(2)
2

>>> standard_arg(arg=2)
2
```

第二个函数 `pos_only_arg` 限制为仅使用位置参数，因为函数定义中有 `/`：

```python
>>> pos_only_arg(1)
1

>>> pos_only_arg(arg=1)
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: pos_only_arg() got an unexpected keyword argument 'arg'
```

第三个函数 `kwd_only_args` 只允许关键字参数，如函数定义中的 `*` 所示：

```python
>>> kwd_only_arg(3)
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: kwd_only_arg() takes 0 positional arguments but 1 was given

>>> kwd_only_arg(arg=3)
3
```

最后一个在同一函数定义中使用所有三种调用约定：

```python
>>> combined_example(1, 2, 3)
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: combined_example() takes 2 positional arguments but 3 were given

>>> combined_example(1, 2, kwd_only=3)
1 2 3

>>> combined_example(1, standard=2, kwd_only=3)
1 2 3

>>> combined_example(pos_only=1, standard=2, kwd_only=3)
Traceback (most recent call last):
 File "<stdin>", line 1, in <module>
TypeError: combined_example() got an unexpected keyword argument 'pos_only'
```

### 总结

用例将确定在函数定义中使用哪些参数：

```python
def f(pos1, pos2, /, pos_or_kwd, *, kwd1, kwd2):
```

作为指导：

- 如果名称不重要或没有意义，并且只有几个参数将始终按相同顺序传递，请使用位置参数。

- 当名称有意义且函数定义通过明确使用名称更易理解时，请使用仅关键字参数。

## 参考实现

一个通过 CPython 测试套件的初始实现可供评估 [10]。

此实现的好处是处理位置参数的速度、与仅关键字参数实现的兼容性（PEP 3102），以及所有受此更改影响的工具和模块的更简单实现。

## 被拒绝的想法

### 不作为

总是一个选项——维持现状。虽然考虑过这个，但上述好处值得添加到语言中。

### 装饰器

有人在 python-ideas [9] 上建议为此功能提供一个用 Python 编写的装饰器。

这种方法的好处是不会污染函数定义额外的语法。然而，我们决定拒绝这个想法，因为：

- 它引入了参数行为声明的不对称性。

- 它使静态分析器和类型检查器难以安全地识别位置参数。他们需要查询 AST 获取装饰器列表，并通过名称或额外启发式方法识别正确的装饰器，而仅关键字参数直接在 AST 中暴露。为了让工具正确识别位置参数，他们需要执行模块以访问装饰器设置的任何元数据。

- 任何声明错误仅在运行时报告。

- 在长的函数定义中识别位置参数可能更加困难，因为它迫使用户计算它们以知道受装饰器影响的最后一个参数。

- `/` 语法已经为 C 函数引入。这种不一致将使实现任何处理此语法的工具和模块更具挑战性——包括但不限于参数诊所、`inspect` 模块和 `ast` 模块。

- 装饰器实现可能会产生运行时性能成本，特别是与直接添加到解释器相比。

### 每个参数标记

每个参数标记是另一个语言内在选项。该方法为每个参数添加一个令牌以指示它们是位置参数，并要求这些参数放在一起。示例：

```python
def (.arg1, .arg2, arg3):
```

注意 `.arg1` 和 `.arg2` 上的点（即 .）。虽然这种方法可能更容易阅读，但它已被拒绝，因为 `/` 作为显式标记与仅关键字参数的 `*` 一致，并且更少出错。

值得注意的是，一些库已经使用前导下划线 [12] 按惯例指示参数为位置参数。

### 使用 `"__"` 作为每个参数标记

一些库和应用程序（如 mypy 或 jinja）使用名称前缀双下划线（即 __）作为惯例来指示位置参数。我们拒绝了将 `__` 作为新语法引入的想法，因为：

- 这是一个向后不兼容的更改。

- 它与当前仅关键字参数声明的方式不对称。

- 查询 AST 获取位置参数需要检查正常参数并检查它们的名称，而仅关键字参数有一个关联属性（`FunctionDef.args.kwonlyargs`）。

- 需要检查每个参数以知道位置参数何时结束。

- 标记更冗长，强制标记每个位置参数。

- 它与双下划线前缀的其他用途冲突，如在类中调用名称 mangling。

### 用括号分组位置参数

元组参数解包是 Python 2 功能，允许在函数定义中将元组用作参数。它允许自动解包序列参数。一个例子是：

```python
def fxn(a, (b, c), d):
 pass
```

元组参数解包在 Python 3 中被移除（PEP 3113）。有人提议重用此语法来实现位置参数。我们拒绝了此语法来指示位置参数的几个原因：

- 语法与仅关键字参数声明的方式不对称。

- Python 2 使用此语法，这可能引起关于此语法行为的混淆。这对移植使用此功能的 Python 2 代码库的用户来说会感到惊讶。

- 此语法与元组字面量非常相似。这可能引起额外的混淆，因为它可能与元组声明混淆。

### 分隔符后提案

在 `/` 后标记位置参数是另一个考虑过的想法。然而，我们无法找到一种方法来修改标记后的参数。否则，将强制标记前的参数也为位置参数。例如：

```python
def (x, y, /, z):
```

如果我们定义 `/` 将 `z` 标记为位置参数，那么就不可能指定 `x` 和 `y` 为关键字参数。找到绕过此限制的方法将增加混淆，因为目前关键字参数不能后跟位置参数。因此，`/` 将使前面的和后面的参数都成为位置参数。
