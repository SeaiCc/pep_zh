## 摘要

本 PEP 描述了一个添加到标准库的功能，称为数据类（Data Classes）。尽管它们使用非常不同的机制，但数据类可以被认为是"带有默认值的可变命名元组"。因为数据类使用普通的类定义语法，您可以自由使用继承、元类、文档字符串、用户定义的方法、类工厂和其他 Python 类特性。

提供一个类装饰器，它检查类定义中是否有 PEP 526 "变量注释语法"中定义的带有类型注释的变量。在本文档中，这样的变量称为字段。使用这些字段，装饰器向类添加生成的方法定义，以支持实例初始化、repr、比较方法，以及可选的其他方法，如规范部分所述。这样的类称为数据类，但该类真的没有什么特别之处：装饰器向类添加生成的方法并返回给它的同一个类。

例如：

```python
@dataclass
class InventoryItem:
 '''用于跟踪库存中的项目。'''
 name: str
 unit_price: float
 quantity_on_hand: int = 0

 def total_cost(self) -> float:
 return self.unit_price * self.quantity_on_hand
```

`@dataclass` 装饰器将向 `InventoryItem` 类添加等效于以下方法的内容：

```python
def __init__(self, name: str, unit_price: float, quantity_on_hand: int = 0) -> None:
 self.name = name
 self.unit_price = unit_price
 self.quantity_on_hand = quantity_on_hand
def __repr__(self):
 return f'InventoryItem(name={self.name!r}, unit_price={self.unit_price!r}, quantity_on_hand={self.quantity_on_hand!r})'
def __eq__(self, other):
 if other.__class__ is self.__class__:
 return (self.name, self.unit_price, self.quantity_on_hand) == (other.name, other.unit_price, other.quantity_on_hand)
 return NotImplemented
def __ne__(self, other):
 if other.__class__ is self.__class__:
 return (self.name, self.unit_price, self.quantity_on_hand) != (other.name, other.unit_price, other.quantity_on_hand)
 return NotImplemented
def __lt__(self, other):
 if other.__class__ is self.__class__:
 return (self.name, self.unit_price, self.quantity_on_hand) < (other.name, other.unit_price, other.quantity_on_hand)
 return NotImplemented
def __le__(self, other):
 if other.__class__ is self.__class__:
 return (self.name, self.unit_price, self.quantity_on_hand) <= (other.name, other.unit_price, other.quantity_on_hand)
 return NotImplemented
def __gt__(self, other):
 if other.__class__ is self.__class__:
 return (self.name, self.unit_price, self.quantity_on_hand) > (other.name, other.unit_price, other.quantity_on_hand)
 return NotImplemented
def __ge__(self, other):
 if other.__class__ is self.__class__:
 return (self.name, self.unit_price, self.quantity_on_hand) >= (other.name, other.unit_price, other.quantity_on_hand)
 return NotImplemented
```

数据类为您节省了编写和维护这些方法的时间。

## 基本原理

有许多尝试来定义主要存储通过属性查找可访问的值的类。一些例子包括：

- 标准库中的 `collections.namedtuple`。

- 标准库中的 `typing.NamedTuple`。

- 流行的 attrs [1] 项目。

- George Sakkis 的 recordType 食谱 [2]，一种受 `collections.namedtuple` 启发的可变数据类型。

- 许多在线示例食谱 [3]、软件包 [4] 和问题 [5]。David Beazley 在 PyCon 2013 元类演讲中使用了数据类形式作为动机示例 [6]。

那么，为什么需要这个 PEP？

随着 PEP 526 的添加，Python 有一种简洁的方式来指定类成员的类型。本 PEP 利用该语法来提供一种简单、无侵入的方式来描述数据类。有两个例外，指定的属性类型注释被数据类完全忽略。

数据类不使用基类或元类。这些类的用户可以自由使用继承和元类，而不会受到数据类的任何干扰。装饰的类真正是"普通"的 Python 类。数据类装饰器不应干扰类的任何用法。

数据类的一个主要设计目标是支持静态类型检查器。使用 PEP 526 语法就是这样一个例子，但 `fields()` 函数和 `@dataclass` 装饰器的设计也是如此。由于它们的动态性质，上面提到的一些库很难与静态类型检查器一起使用。

数据类不是，也不打算成为上述所有库的替代机制。但在标准库中允许许多更简单的用例转而利用数据类。列出的许多库都有不同的功能集，当然将继续存在和发展。

哪里不适合使用数据类？

- 需要与元组或字典的 API 兼容性。

- 需要超出 PEP 484 和 526 提供的类型验证，或者需要值验证或转换。

## 规范

本 PEP 中描述的所有函数都存在于名为 `dataclasses` 的模块中。

提供一个函数 `dataclass`，通常用作类装饰器，用于后处理类并添加生成的方法，如下所述。

`dataclass` 装饰器检查类以查找字段。字段定义为在 `__annotations__` 中识别的任何变量。也就是说，具有类型注释的变量。有两个下面描述的例外，数据类机制不检查注释中指定的类型。

注意 `__annotations__` 保证是一个有序映射，按类声明顺序。所有生成的方法中字段的顺序是它们在类中出现的顺序。

`dataclass` 装饰器将向类添加各种"dunder"方法，如下所述。如果任何添加的方法已经存在于类上，将引发 `TypeError`。装饰器返回调用的同一个类：不创建新类。

`dataclass` 装饰器通常不带参数和括号使用。但是，它也支持以下逻辑签名：

```python
def dataclass(*, init=True, repr=True, eq=True, order=False, unsafe_hash=False, frozen=False)
```

如果 `dataclass` 仅作为不带参数的简单装饰器使用，它的作用就像具有此签名中记录的默认值一样。也就是说，这三种 `@dataclass` 用法是等效的：

```python
@dataclass
class C:
 ...

@dataclass()
class C:
 ...

@dataclass(init=True, repr=True, eq=True, order=False, unsafe_hash=False, frozen=False)
class C:
 ...
```

`dataclass` 的参数是：

- `init`：如果为真（默认值），将生成 `__init__` 方法。

- `repr`：如果为真（默认值），将生成 `__repr__` 方法。生成的 repr 字符串将具有类名以及每个字段的名称和 repr，按它们在类中定义的顺序。标记为从 repr 中排除的字段不包括在内。例如：`InventoryItem(name='widget', unit_price=3.0, quantity_on_hand=10)`。如果类已经定义了 `__repr__`，则忽略此参数。

- `eq`：如果为真（默认值），将生成 `__eq__` 方法。此方法将类比较为按顺序的字段元组。比较中的两个实例必须是完全相同的类型。如果类已经定义了 `__eq__`，则忽略此参数。

- `order`：如果为真（默认值为 False），将生成 `__lt__`、`__le__`、`__gt__` 和 `__ge__` 方法。这些将类比较为按顺序的字段元组。比较中的两个实例必须是完全相同的类型。如果 `order` 为真且 `eq` 为假，则引发 `ValueError`。如果类已经定义了 `__lt__`、`__le__`、`__gt__` 或 `__ge__` 中的任何一个，则引发 `ValueError`。

- `unsafe_hash`：如果为 False（默认值），`__hash__` 方法根据 `eq` 和 `frozen` 的设置生成。如果 `eq` 和 `frozen` 都为真，数据类将为您生成 `__hash__` 方法。如果 `eq` 为真且 `frozen` 为假，`__hash__` 将设置为 `None`，标记为不可哈希（它确实不可哈希）。如果 `eq` 为假，`__hash__` 将保持不变，意味着将使用超类的 `__hash__` 方法（如果超类是 `object`，这意味着它将回退到基于 id 的哈希）。

虽然不建议，但您可以使用 `unsafe_hash=True` 强制数据类创建 `__hash__` 方法。如果您的类在逻辑上是不可变的但可以被改变，这可能是这种情况。这是一个专门的用例，应该仔细考虑。

如果类已经有明确定义的 `__hash__`，则添加 `__hash__` 的行为会修改。当以下条件满足时，定义为明确定义的 `__hash__`：

- `__eq__` 在类中定义且 `__hash__` 定义为任何非 `None` 值。

- `__eq__` 在类中定义且任何非 `None` 的 `__hash__` 被定义。

- `__eq__` 未在类上定义，且任何 `__hash__` 被定义。

如果 `unsafe_hash` 为真且存在明确定义的 `__hash__`，则引发 `ValueError`。

如果 `unsafe_hash` 为假且存在明确定义的 `__hash__`，则不添加 `__hash__`。

有关更多信息，请参阅 Python 文档 [7]。

- `frozen`：如果为真（默认值为 False），分配给字段将生成异常。这模拟只读冻结实例。如果在类中定义了 `__getattr__` 或 `__setattr__`，则引发 `ValueError`。参见下面的讨论。

字段可以可选地指定默认值，使用普通 Python 语法：

```python
@dataclass
class C:
 a: int # 'a' 没有默认值
 b: int = 0 # 为 'b' 分配默认值
```

在此示例中，`a` 和 `b` 都将包含在添加的 `__init__` 方法中，该方法定义为：

```python
def __init__(self, a: int, b: int = 0):
```

如果无默认值的字段跟随有默认值的字段，则引发 `TypeError`。这在一个类中发生或由于类继承的结果都是如此。

对于常见和简单的用例，不需要其他功能。但是，有一些数据类功能需要额外的每字段信息。为了满足这种额外信息的需求，您可以将默认字段值替换为对提供的 `field()` 函数的调用。`field()` 的签名是：

```python
def field(*, default=MISSING, default_factory=MISSING, repr=True, hash=None, init=True, compare=True, metadata=None)
```

`MISSING` 值是一个用于检测是否提供了 `default` 和 `default_factory` 参数的哨兵对象。使用此哨兵是因为 `None` 是 `default` 的有效值。

`field()` 的参数是：

- `default`：如果提供，这将是此字段的默认值。这是需要的，因为字段调用本身替换了默认值的正常位置。

- `default_factory`：如果提供，它必须是一个零参数可调用对象，当需要此字段的默认值时将被调用。除了其他用途外，这可用于指定具有可变默认值的字段，如下所述。同时指定 `default` 和 `default_factory` 是错误的。

- `init`：如果为真（默认值），此字段作为参数包含在生成的 `__init__` 方法中。

- `repr`：如果为真（默认值），此字段包含在生成的 `__repr__` 方法返回的字符串中。

- `compare`：如果为 True（默认值），此字段包含在生成的相等性和比较方法中（`__eq__`、`__gt__` 等）。

- `hash`：这可以是布尔值或 `None`。如果为 `True`，此字段包含在生成的 `__hash__` 方法中。如果为 `None`（默认值），使用 `compare` 的值：这通常是预期行为。如果字段用于比较，则应在哈希中考虑它。不鼓励将此值设置为任何非 `None` 的值。设置 `hash=False` 但 `compare=True` 的一个可能原因是，如果字段计算哈希值很昂贵，该字段用于相等性测试，并且有其他字段贡献于类型的哈希值。即使字段从哈希中排除，它仍用于比较。

- `metadata`：这可以是映射或 `None`。`None` 被视为空字典。此值包装在 `types.MappingProxyType` 中以使其只读，并在 `Field` 对象上暴露。它完全不被数据类使用，而是作为第三方扩展机制提供。多个第三方可以各自有自己的键，用作元数据中的命名空间。

如果字段的默认值通过调用 `field()` 指定，则此字的类属性将替换为指定的默认值。如果没有提供默认值，则类属性将被删除。意图是在 `dataclass` 装饰器运行后，类属性将包含字段的默认值，就像指定了默认值本身一样。例如，之后：

```python
@dataclass
class C:
 x: int
 y: int = field(repr=False)
 z: int = field(repr=False, default=10)
 t: int = 20
```

类属性 `C.z` 将是 10，类属性 `C.t` 将是 20，类属性 `C.x` 和 `C.y` 将不会设置。

### 字段对象

字段对象描述每个定义的字段。这些对象在内部创建，并由 `fields()` 模块级方法返回（见下文）。用户绝不应直接实例化 `Field` 对象。其文档属性是：

- `name`：字段名称。

- `type`：字段类型。

- `default`、`default_factory`、`init`、`repr`、`hash`、`compare` 和 `metadata` 具有与它们在 `field()` 声明中完全相同的含义和值。

可能存在其他属性，但它们是私有的，不得检查或依赖它们。

### post-init 处理

生成的 `__init__` 代码将调用名为 `__post_init__` 的方法（如果它在类上定义）。它将作为 `self.__post_init__()` 调用。如果没有生成 `__init__` 方法，则不会自动调用 `__post_init__`。

除了其他用途外，这允许初始化依赖于一个或多个其他字段的字段值。例如：

```python
@dataclass
class C:
 a: float
 b: float
 c: float = field(init=False)

 def __post_init__(self):
    self.c = self.a + self.b
```

参见下面关于 init-only 变量的部分，了解如何将参数传递给 `__post_init__()`。另请参见关于 `replace()` 如何处理 `init=False` 字段的警告。

### 类变量

数据类实际检查字段类型的一个地方是确定字段是否是 PEP 526 中定义的类变量。它通过检查字段类型是否为 `typing.ClassVar` 来做到这一点。如果字段是 `ClassVar`，它被排除在字段考虑之外，并被数据类机制忽略。有关更多讨论，参见 [8]。这样的 `ClassVar` 伪字段不由模块级 `fields()` 函数返回。

### Init-only 变量

数据类检查类型注释的另一个地方是确定字段是否是 init-only 变量。它通过查看字段类型是否为 `dataclasses.InitVar` 类型来做到这一点。如果字段是 `InitVar`，它被视为称为 init-only 字段的伪字段。因为它不是真正的字段，所以它不由模块级 `fields()` 函数返回。Init-only 字段作为参数添加到生成的 `__init__` 方法中，并传递给可选的 `__post_init__` 方法。数据类不使用它们的其他用途。

例如，假设字段将从数据库初始化（如果创建类时未提供值）：

```python
@dataclass
class C:
    i: int
    j: int = None
    database: InitVar[DatabaseType] = None

    def __post_init__(self, database):
        if self.j is None and database is not None:
            self.j = database.lookup('j')

c = C(10, database=my_database)
```

在这种情况下，`fields()` 将返回 `i` 和 `j` 的 `Field` 对象，但不返回 `database` 的。

### 冻结实例

无法创建真正的不可变 Python 对象。但是，通过向 `@dataclass` 装饰器传递 `frozen=True`，您可以模拟不可变性。在这种情况下，数据类将向类添加 `__setattr__` 和 `__delattr__` 方法。调用这些方法时将引发 `FrozenInstanceError`。

使用 `frozen=True` 时有一个微小的性能惩罚：`__init__` 不能使用简单赋值来初始化字段，必须使用 `object.__setattr__`。

### 继承

当 `@dataclass` 装饰器创建数据类时，它按反向 MRO 遍历类的所有基类（即，从 `object` 开始），对于找到的每个数据类，将该基类的字段添加到字段的有序映射中。添加所有基类字段后，它将自己的字段添加到有序映射中。所有生成的方法都将使用此组合的、计算出的字段有序映射。因为字段按插入顺序排列，派生类覆盖基类。示例：

```python
@dataclass
class Base:
 x: Any = 15.0
 y: int = 0

@dataclass
class C(Base):
 z: int = 10
 x: int = 15
```

最终字段列表按顺序是 `x`、`y`、`z`。`x` 的最终类型是 `int`，如类 `C` 中所指定的。

生成的 `C` 的 `__init__` 方法看起来像：

```python
def __init__(self, x: int = 15, y: int = 0, z: int = 10):
```

### 默认工厂函数

如果字段指定了 `default_factory`，当需要字段的默认值时，它将以零参数调用。例如，要创建列表的新实例，使用：

```python
l: list = field(default_factory=list)
```

如果字段从 `__init__` 中排除（使用 `init=False`）并且字段也指定了 `default_factory`，则默认工厂函数将始终从生成的 `__init__` 函数调用。这是因为没有其他方法可以给字段初始值。

### 可变默认值

Python 在类属性中存储默认成员变量值。考虑这个不使用数据类的示例：

```python
class C:
    x = []
    def add(self, element):
        self.x += element

o1 = C()
o2 = C()
o1.add(1)
o2.add(2)
assert o1.x == [1, 2]
assert o1.x is o2.x
```

注意类 `C` 的两个实例共享相同的类变量 `x`，正如预期的那样。

使用数据类，如果此代码有效：

```python
@dataclass
class D:
    x: List = []
    def add(self, element):
        self.x += element
```

它将生成类似于以下的代码：

```python
class D:
    x = []
    def __init__(self, x=x):
        self.x = x
    def add(self, element):
        self.x += element

assert D().x is D().x
```

这与使用类 `C` 的原始示例有相同的问题。也就是说，创建类实例时不指定 `x` 值的类 `D` 的两个实例将共享 `x` 的相同副本。因为数据类只是使用普通 Python 类创建，所以它们也共享这个问题。数据类没有通用的方法来检测此条件。相反，如果检测到类型为 `list`、`dict` 或 `set` 的默认参数，数据类将引发 `TypeError`。这是一个部分解决方案，但它确实防止了许多常见错误。有关更多详细信息，请参阅被拒绝的想法部分中的自动支持可变默认值。

使用默认工厂函数是创建可变类型的新实例作为字段默认值的一种方法：

```python
@dataclass
class D:
 x: list = field(default_factory=list)

assert D().x is not D().x
```

### 模块级辅助函数

- `fields(class_or_instance)`：返回定义此数据类字段的 `Field` 对象元组。接受数据类或数据类的实例。如果没有传递数据类或其实例，则引发 `ValueError`。不返回是 `ClassVar` 或 `InitVar` 的伪字段。

- `asdict(instance, *, dict_factory=dict)`：将数据类实例转换为字典（通过使用工厂函数 `dict_factory`）。每个数据类转换为其字段的字典，作为 name:value 对。递归进入数据类、字典、列表和元组。例如：

```python
@dataclass
class Point:
 x: int
 y: int

@dataclass
class C:
 l: List[Point]

p = Point(10, 20)
assert asdict(p) == {'x': 10, 'y': 20}

c = C([Point(0, 0), Point(10, 4)])
assert asdict(c) == {'l': [{'x': 0, 'y': 0}, {'x': 10, 'y': 4}]}
```

如果 `instance` 不是数据类实例，则引发 `TypeError`。

- `astuple(*, tuple_factory=tuple)`：将数据类实例转换为元组（通过使用工厂函数 `tuple_factory`）。每个数据类转换为其字段值的元组。递归进入数据类、字典、列表和元组。继续上一个示例：

```python
assert astuple(p) == (10, 20)
assert astuple(c) == ([(0, 0), (10, 4)],)
```

如果 `instance` 不是数据类实例，则引发 `TypeError`。

- `make_dataclass(cls_name, fields, *, bases=(), namespace=None)`：创建一个新数据类，名称为 `cls_name`，字段如 `fields` 中所定义，基类如 `bases` 中所给出，并使用 `namespace` 中给出的命名空间初始化。`fields` 是一个可迭代对象，其元素是 `name`、`(name, type)` 或 `(name, type, Field)`。如果只提供了 `name`，则使用 `typing.Any` 作为类型。此函数不是严格必需的，因为任何创建具有 `__annotations__` 的新类的 Python 机制都可以然后应用 `dataclass` 函数将该类转换为数据类。此函数作为便利提供。例如：

```python
C = make_dataclass('C',
 [('x', int),
 'y',
 ('z', int, field(default=5))],
 namespace={'add_one': lambda self: self.x + 1})
```

等效于：

```python
@dataclass
class C:
 x: int
 y: 'typing.Any'
 z: int = 5

 def add_one(self):
    return self.x + 1
```

- `replace(instance, **changes)`：创建与 `instance` 相同类型的新对象，用 `changes` 中的值替换字段。如果 `instance` 不是数据类，则引发 `TypeError`。如果 `changes` 中的值未指定字段，则引发 `TypeError`。

新返回的对象通过调用数据类的 `__init__` 方法创建。这确保如果存在 `__post_init__`，也会被调用。

如果没有默认值的 init-only 变量（如果存在），必须在调用 `replace` 时指定，以便可以传递给 `__init__` 和 `__post_init__`。

如果 `changes` 包含任何定义为 `init=False` 的字段，则是错误的。这种情况下将引发 `ValueError`。

注意在调用 `replace()` 期间 `init=False` 字段的工作方式。它们不从源对象复制，而是在 `__post_init__()` 中初始化（如果根本初始化的话）。预期 `init=False` 字段将很少且谨慎地使用。如果使用它们，可能明智地拥有备用类构造器，或者也许自定义 `replace()`（或类似命名的）方法来处理实例复制。

- `is_dataclass(class_or_instance)`：如果其参数是数据类或其实例，则返回 `True`，否则返回 `False`。

如果您需要知道类是否是数据类的实例（而不是数据类本身），则添加进一步检查 `not isinstance(obj, type)`：

```python
def is_dataclass_instance(obj):
    return is_dataclass(obj) and not isinstance(obj, type)
```

## 讨论

### python-ideas 讨论

此讨论始于 python-ideas [9] 并移至 GitHub 仓库 [10] 以进一步讨论。作为此讨论的一部分，我们决定使用 PEP 526 语法来驱动字段发现。

### 支持自动设置 `__slots__`？

至少对于初始版本，不支持 `__slots__`。`__slots__` 需要在类创建时添加。数据类装饰器在类创建后调用，因此为了添加 `__slots__`，装饰器必须创建新类、设置 `__slots__` 并返回它。因为这种行为有点令人惊讶，数据类的初始版本将不支持自动设置 `__slots__`。有许多解决方法：

- 在类定义中手动添加 `__slots__`。

- 编写一个函数（可用作装饰器），使用 `fields()` 检查类并创建具有 `__slots__` 设置的新类。

有关更多讨论，参见 [11]。

### 为什么不直接使用 namedtuple？

- 任何 namedtuple 都可以意外地与具有相同数字字段的任何其他 namedtuple 比较。例如：`Point3D(2017, 6, 2) == Date(2017, 6, 2)`。使用数据类，这将返回 `False`。

- namedtuple 可以意外地与元组比较。例如，`Point2D(1, 10) == (1, 10)`。使用数据类，这将返回 `False`。

- 实例总是可迭代的，这使得添加字段变得困难。如果库定义了：

```python
Time = namedtuple('Time', ['hour', 'minute'])
def get_time():
    return Time(12, 0)
```

然后如果用户使用此代码作为：

```python
hour, minute = get_time()
```

那么在不破坏用户代码的情况下就不可能向 `Time` 添加第二个字段。

- 没有可变实例的选项。

- 不能指定默认值。

- 不能控制哪些字段用于 `__init__`、`__repr__` 等。

- 不能支持通过继承组合字段。

### 为什么不直接使用 typing.NamedTuple？

对于具有静态定义字段的类，它支持类似于数据类的语法，使用类型注释。这产生 namedtuple，因此它共享 namedtuple 的优点和一些缺点。与 `typing.NamedTuple` 不同，数据类支持通过继承组合字段。

### 为什么不直接使用 attrs？

- attrs 的移动速度快到如果将其移到标准库中则无法适应。

- attrs 支持此处未提议的附加功能：验证器、转换器、元数据等。数据类通过不实现这些功能来实现简单性的权衡。

有关更多讨论，参见 [12]。

### post-init 参数

在添加 `InitVar` 之前的本 PEP 的早期版本中，post-init 函数 `__post_init__` 从不接受任何参数。

参数化初始化的正常方式（不仅仅是数据类）是提供备用类方法构造器。例如：

```python
@dataclass
class C:
    x: int

    @classmethod
    def from_file(cls, filename):
        with open(filename) as fl:
            file_value = int(fl.read())
        return C(file_value)

c = C.from_file('file.txt')
```

因为 `__post_init__` 函数是在生成的 `__init__` 中调用的最后一件事，所以具有类方法构造器（也可以在构造对象后立即执行代码）在功能上等效于能够向 `__post_init__` 函数传递参数。

有了 `InitVars`，`__post_init__` 函数现在可以接受参数。它们首先传递给 `__init__`，然后传递给 `__post_init__`，用户代码可以根据需要使用它们。

备用类方法构造器和 `InitVar` 伪字段之间的唯一真正区别在于对象创建期间的必需非字段参数。使用 `InitVars`，使用 `__init__` 和模块级 `replace()` 函数，`InitVars` 必须始终指定。考虑需要上下文对象来创建实例但不存储为字段的情况。使用备用类方法构造器，上下文参数总是可选的，因为您仍然可以通过 `__init__` 创建对象（除非您抑制其创建）。哪种方法更合适将是应用程序特定的，但两种方法都受支持。

使用 `InitVar` 字段的另一个原因是类作者可以控制 `__init__` 参数的顺序。这对于具有默认值的常规字段和 `InitVar` 字段尤其重要，因为所有具有默认值的字段必须跟在所有没有默认值的字段之后。以前的设计将所有 init-only 字段放在常规字段之后。这意味着如果任何字段具有默认值，则所有 init-only 字段也必须有默认值。

### asdict 和 astuple 函数名称

模块级辅助函数 `asdict()` 和 `astuple()` 的名称可以说是不符合 PEP 8，应该是 `as_dict()` 和 `as_tuple()`。但是，经过讨论 [13] 后，决定保持与 `namedtuple._asdict()` 和 `attr.asdict()` 的一致性。

## 被拒绝的想法

### 在 replace() 中新对象创建后复制 init=False 字段

`init=False` 字段根据定义不传递给 `__init__`，而是用默认值初始化，或者通过在 `__init__` 中调用默认工厂函数，或者通过 `__post_init__` 中的代码初始化。

本 PEP 的早期版本指定 `init=False` 字段将在 `__init__` 返回后从源对象复制到新创建的对象，但这被认为与使用 `__init__` 和 `__post_init__` 初始化新对象不一致。例如，考虑这种情况：

```python
@dataclass
class Square:
    length: float
    area: float = field(init=False, default=0.0)

    def __post_init__(self):
        self.area = self.length * self.length

s1 = Square(1.0)
s2 = replace(s1, length=2.0)
```

如果 `init=False` 字段在运行 `__post_init__` 后从源复制到目标对象，那么 `s2` 最终将是 `Square(length=2.0, area=1.0)`，而不是正确的 `Square(length=2.0, area=4.0)`。

### 自动支持可变默认值

一个提议是自动复制默认值，因此如果字面列表 `[]` 是默认值，每个实例将获得一个新列表。这个决定有不良的副作用，因此最终决定是禁止 3 个已知的内置可变类型：`list`、`dict` 和 `set`。有关此和其他选项的完整讨论，参见 [14]。

## 示例

### 自定义 __init__ 方法

有时生成的 `__init__` 方法不足够。例如，假设您想要一个对象来存储 `*args` 和 `**kwargs`：

```python
@dataclass(init=False)
class ArgHolder:
    args: List[Any]
    kwargs: Mapping[Any, Any]

    def __init__(self, *args, **kwargs):
        self.args = args
        self.kwargs = kwargs

a = ArgHolder(1, 2, three=3)
```

### 一个复杂的示例

此代码存在于一个闭源项目中：

```python
class Application:
    def __init__(self, name, requirements, constraints=None, path='', executable_links=None, executables_dir=()):
        self.name = name
        self.requirements = requirements
        self.constraints = {} if constraints is None else constraints
        self.path = path
        self.executable_links = [] if executable_links is None else executable_links
        self.executables_dir = executables_dir
        self.additional_items = []

    def __repr__(self):
        return f'Application({self.name!r},{self.requirements!r},{self.constraints!r},{self.path!r},{self.executable_links!r},{self.executables_dir!r},{self.additional_items!r})'
```

这可以被替换为：

```python
@dataclass
class Application:
    name: str
    requirements: List[Requirement]
    constraints: Dict[str, str] = field(default_factory=dict)
    path: str = ''
    executable_links: List[str] = field(default_factory=list)
    executable_dir: Tuple[str] = ()
    additional_items: List[str] = field(init=False, default_factory=list)
```

数据类版本更具声明性，代码更少，支持类型，并包括其他生成的函数。
