# PEP 454 – 添加新的 tracemalloc 模块来追踪 Python 内存分配

**元数据：**

- **作者**：Victor Stinner <vstinner&#32;&#97;t&#32;python.org>
- **BDFL 代表**：Charles-François Natali <cf.natali&#32;&#97;t&#32;gmail.com>
- **创建时间**：2013 年 9 月 3 日
- **Python 版本**：3.4

## 摘要

本 PEP 提议添加一个新的 tracemalloc 模块来追踪 Python 分配的内存块。

## 理由

经典的通用工具如 Valgrind 可以获取内存块分配的 C 回溯。使用此类工具分析 Python 内存分配没有帮助，因为大多数内存块在相同的 C 函数中分配，例如在 PyMem_Malloc() 中。此外，Python 有一个名为"pymalloc"的小对象分配器，它为提高效率而保留空闲块。这些工具不能很好地处理这一点。

有专门用于 Python 语言的调试工具，如 Heapy、Pympler 和 Meliae，它们使用垃圾回收器模块（如 gc.get_objects()、gc.get_referrers() 和 gc.get_referents() 函数）列出所有存活的对象，计算它们的大小（例如：使用 sys.getsizeof()）并按类型分组对象。这些工具提供了应用程序内存使用的更好估计。当大多数内存泄漏是同一类型的实例且该类型仅在少数函数中实例化时，它们很有用。当对象类型非常常见，如 str 或 tuple，很难识别这些对象在哪里实例化时，会出现问题。

查找引用循环也是一个困难的问题。有不同的工具来绘制所有引用的图表。这些工具不能用于具有数千个对象的大型应用程序，因为图表太大而无法手动分析。

## 提议

使用 PEP 445 的自定义分配 API，可以很容易地在 Python 内存分配器上设置挂钩。挂钩可以检查 Python 内部以检索 Python 回溯。获取当前回溯的想法来自 faulthandler 模块。faulthandler 在崩溃时转储所有 Python 线程的回溯，这里的想法是在 Python 分配内存块时获取当前 Python 线程的回溯。

本 PEP 提议添加一个新的 tracemalloc 模块，一个调试工具来追踪 Python 分配的内存块。该模块提供以下信息：

- 对象分配位置的回溯

- 按文件名和行号的已分配内存块统计信息：总大小、已分配内存块的数量和平均大小

- 计算两个快照之间的差异以检测内存泄漏

tracemalloc 模块的 API 类似于 faulthandler 模块的 API：enable() / start()、disable() / stop() 和 is_enabled() / is_tracing() 函数，环境变量（PYTHONFAULTHANDLER 和 PYTHONTRACEMALLOC），以及-X 命令行选项（-X faulthandler 和-X tracemalloc）。参见 faulthandler 模块的文档。

追踪内存分配的想法并不新鲜。它首先在 2005 年的 PySizer 项目中实现。PySizer 的实现方式不同：回溯存储在帧对象中，一些 Python 类型通过对象类型名称与跟踪链接。CPython 上的 PySizer 补丁在性能和内存占用上增加开销，即使没有使用 PySizer。tracemalloc 将回溯附加到底层，到内存块，当模块没有追踪内存分配时没有开销。

tracemalloc 模块是为 CPython 编写的。Python 的其他实现可能无法提供它。

## API

要追踪 Python 分配的内存块，应尽早通过设置 PYTHONTRACEMALLOC 环境变量为 1，或使用-X tracemalloc 命令行选项来启动模块。可以在运行时调用 tracemalloc.start() 函数来启动追踪 Python 内存分配。

默认情况下，已分配内存块的跟踪仅存储最近的帧（1 帧）。要在启动时存储 25 帧：将 PYTHONTRACEMALLOC 环境变量设置为 25，或使用-X tracemalloc=25 命令行选项。set_traceback_limit() 函数可以在运行时用于设置限制。

### 函数

clear_traces() 函数：

> 清除 Python 分配的内存块的跟踪。
> 另见 stop()。

get_object_traceback(obj) 函数：

> 获取 Python 对象 *obj* 分配位置的回溯。
> 返回 Traceback 实例，或者如果 tracemalloc 模块没有追踪内存分配或没有追踪对象的分配，则返回 None。
> 另见 gc.get_referrers() 和 sys.getsizeof() 函数。

get_traceback_limit() 函数：

> 获取跟踪回溯中存储的最大帧数。
> tracemalloc 模块必须正在追踪内存分配以获取限制，否则引发异常。

限制由 start() 函数设置。

get_traced_memory() 函数：

> 获取 tracemalloc 模块追踪的内存块的当前大小和最大大小，作为元组：(size: int, max_size: int)。

get_tracemalloc_memory() 函数：

> 获取 tracemalloc 模块用于存储内存块跟踪的内存使用量（字节）。返回 int。

is_tracing() 函数：

> 如果 tracemalloc 模块正在追踪 Python 内存分配，则为 True，否则为 False。
> 另见 start() 和 stop() 函数。

start(nframe: int=1) 函数：

> 启动追踪 Python 内存分配：在 Python 内存分配器上安装挂钩。收集的跟踪回溯将限制为 *nframe* 帧。默认情况下，内存块的跟踪仅存储最近的帧：限制为 1。*nframe* 必须大于或等于 1。

存储超过 1 帧仅对按'traceback'分组计算统计信息或计算累计统计信息有用：参见 Snapshot.compare_to() 和 Snapshot.statistics() 方法。

存储更多帧会增加 tracemalloc 模块的内存和 CPU 开销。使用 get_tracemalloc_memory() 函数来测量 tracemalloc 模块使用了多少内存。

PYTHONTRACEMALLOC 环境变量（PYTHONTRACEMALLOC=NFRAME）和-X tracemalloc=NFRAME 命令行选项可用于在启动时启动追踪。

另见 stop()、is_tracing() 和 get_traceback_limit() 函数。

stop() 函数：

> 停止追踪 Python 内存分配：卸载 Python 内存分配器上的挂钩。还清除 Python 分配的内存块的跟踪。
> 调用 take_snapshot() 函数在清除之前获取跟踪的快照。

另见 start() 和 is_tracing() 函数。

take_snapshot() 函数：

> 获取 Python 分配的内存块的跟踪快照。返回新的 Snapshot 实例。

快照不包括 tracemalloc 模块开始追踪内存分配之前分配的内存块。

跟踪的回溯限制为 get_traceback_limit() 帧。使用 start() 函数的 nframe 参数来存储更多帧。

tracemalloc 模块必须正在追踪内存分配以获取快照，参见 start() 函数。

另见 get_object_traceback() 函数。

### Filter

Filter(inclusive: bool, filename_pattern: str, lineno: int=None, all_frames: bool=False) 类：

> 内存块跟踪的过滤器。
> 参见 fnmatch.fnmatch() 函数以获取 filename_pattern 的语法。'.pyc'和'.pyo'文件扩展名被替换为'.py'。

示例：

- `Filter(True, subprocess.__file__)` 仅包含 subprocess 模块的跟踪

- `Filter(False, tracemalloc.__file__)` 排除 tracemalloc 模块的跟踪

- `Filter(False, "<unknown>")` 排除空回溯

inclusive 属性：

> 如果 *inclusive* 为 True（包含），仅跟踪在文件名匹配 filename_pattern 且行号为 lineno 的文件中分配的内存块。
> 如果 inclusive 为 False（排除），忽略在文件名匹配 filename_pattern 且行号为 lineno 的文件中分配的内存块。

lineno 属性：

> 过滤器的行号（int）。如果 *lineno* 为 None，过滤器匹配任何行号。

filename_pattern 属性：

> 过滤器的文件名模式（str）。

all_frames 属性：

> 如果 *all_frames* 为 True，检查回溯的所有帧。如果 *all_frames* 为 False，仅检查最近的帧。
> 如果回溯限制小于 2，则忽略此属性。参见 get_traceback_limit() 函数和 Snapshot.traceback_limit 属性。

### Frame

Frame 类：

> 回溯的帧。
> Traceback 类是 Frame 实例的序列。

filename 属性：

> 文件名（str）。

lineno 属性：

> 行号（int）。

### Snapshot

Snapshot 类：

> Python 分配的内存块跟踪的快照。
> take_snapshot() 函数创建快照实例。

compare_to(old_snapshot: Snapshot, group_by: str, cumulative: bool=False) 方法：

> 计算与旧快照的差异。获取按 *group_by* 分组的统计信息，作为 StatisticDiff 实例的排序列表。
> 参见 statistics() 方法以获取 group_by 和 cumulative 参数。

结果从最大到最小排序：StatisticDiff.size_diff 的绝对值、StatisticDiff.size、StatisticDiff.count_diff 的绝对值、Statistic.count，然后按 StatisticDiff.traceback。

dump(filename) 方法：

> 将快照写入文件。
> 使用 load() 重新加载快照。

filter_traces(filters) 方法：

> 创建具有过滤跟踪序列的新 Snapshot 实例，*filters* 是 Filter 实例列表。如果 *filters* 是空列表，返回具有跟踪副本的新 Snapshot 实例。
> 所有包含过滤器同时应用，如果没有包含过滤器匹配跟踪，则忽略跟踪。如果至少一个独占过滤器匹配跟踪，则忽略跟踪。

load(filename) 类方法：

> 从文件加载快照。
> 另见 dump()。

statistics(group_by: str, cumulative: bool=False) 方法：

> 获取按 *group_by* 分组的统计信息，作为 Statistic 实例的排序列表：

group_by | description
-------|------------
'filename' | 文件名
'lineno' | 文件名和行号
'traceback' | 回溯

如果 cumulative 为 True，累计跟踪回溯的所有帧的内存块的大小和数量，而不仅仅是最近的帧。累计模式只能与 group_by 等于'filename'和'lineno'以及 traceback_limit 大于 1 一起使用。

结果从最大到最小排序：Statistic.size、Statistic.count，然后按 Statistic.traceback。

traceback_limit 属性：

> 跟踪回溯中存储的最大帧数：获取快照时 get_traceback_limit() 的结果。

traces 属性：

> Python 分配的所有内存块的跟踪：Trace 实例的序列。
> 序列具有未定义的顺序。使用 Snapshot.statistics() 方法获取统计信息的排序列表。

### Statistic

Statistic 类：

> 内存分配的统计信息。
> Snapshot.statistics() 返回 Statistic 实例列表。

另见 StatisticDiff 类。

count 属性：

> 内存块的数量（int）。

size 属性：

> 内存块的总大小（字节）（int）。

traceback 属性：

> 内存块分配位置的回溯，Traceback 实例。

### StatisticDiff

StatisticDiff 类：

> 旧和新 Snapshot 实例之间内存分配的统计差异。
> Snapshot.compare_to() 返回 StatisticDiff 实例列表。另见 Statistic 类。

count 属性：

> 新快照中的内存块数量（int）：如果内存块在新快照中已释放，则为 0。

count_diff 属性：

> 旧和新快照之间内存块数量的差异（int）：如果内存块在新快照中已分配，则为 0。

size 属性：

> 新快照中内存块的总大小（字节）（int）：如果内存块在新快照中已释放，则为 0。

size_diff 属性：

> 旧和新快照之间内存块总大小的差异（字节）（int）：如果内存块在新快照中已分配，则为 0。

traceback 属性：

> 内存块分配位置的回溯，Traceback 实例。

### Trace

Trace 类：

> 内存块的跟踪。
> Snapshot.traces 属性是 Trace 实例的序列。

size 属性：

> 内存块的大小（字节）（int）。

traceback 属性：

> 内存块分配位置的回溯，Traceback 实例。

### Traceback

Traceback 类：

> 从最近帧到最旧帧排序的 Frame 实例序列。
> 回溯包含至少 1 帧。如果 tracemalloc 模块无法获取帧，则使用第 0 行的文件名"<unknown>"。

当获取快照时，跟踪的回溯限制为 get_traceback_limit() 帧。参见 take_snapshot() 函数。

Trace.traceback 属性是 Traceback 实例。

## 被拒绝的替代方案

### 记录对内存分配器的调用

不同的方法是记录对 malloc()、realloc() 和 free() 函数的调用。调用可以记录到文件或通过网络发送到另一台计算机。日志条目的示例：函数名、内存块大小、内存块地址、发生分配的 Python 回溯、时间戳。

日志不能直接使用，获取内存的当前状态需要解析以前的日志。例如，不可能直接获取 Python 对象的回溯，如 get_object_traceback(obj) 使用跟踪所做的那样。

Python 使用非常短生命周期的对象，因此广泛使用内存分配器。它有一个针对短生命周期小对象（小于 512 字节）优化的分配器。例如，Python 测试套件每秒平均调用 malloc()、realloc() 或 free() 270,000 次。如果日志条目大小为 32 字节，日志每秒产生 8.2 MB 或每小时 29.0 GB。

替代方案被拒绝，因为它效率较低且功能较少。在不同进程或不同计算机上解析日志比在同一进程中维护已分配内存块的跟踪更慢。

## 先前工作

- Python Memory Validator (2005-2013)：
  Software Verification 开发的商业 Python 内存验证器。
  它使用 Python 反射 API。

- PySizer：Nick Smallbone 的 Google Summer of Code 2005 项目。

- Heapy (2006-2013)：
  Sverker Nilsson 编写的 Guppy-PE 项目的一部分。

- 草案 PEP：在 CPython 中支持跟踪低级内存使用
  （Brett Canon, 2006）

- Muppy：Robert Schuppenies 在 2008 年开发的项目。

- asizeof：
  Jean Brouwers 编写的纯 Python 模块，用于估计对象大小（2008）。

- Heapmonitor：
  它提供对单个对象进行大小测量的功能，并可以跟踪特定类的所有对象。
  它由 Ludwig Haehne 在 2008 年开发。

- Pympler (2008-2011)：
  基于 asizeof、muppy 和 HeapMonitor 的项目

- objgraph (2008-2012)

- Dozer：CherryPy 内存泄漏调试器的 WSGI 中间件版本，
  由 Marius Gedminas 编写（2008-2013）

- Meliae：
  John A Meinel 自 2009 年以来开发的 Python 内存使用分析器

- gdb-heap：Dave Malcolm 用
  Python 编写的 gdb 脚本（2010-2011），用于分析堆内存的使用

- memory_profiler：
  由 Fabian Pedregosa 编写（2011-2013）

- caulk：Ben Timby 在 2012 年编写

另见 Pympler 相关工作。
