# PEP 389 – argparse - 新的命令行解析模块

**元数据：**

- **作者**: Steven Bethard <steven.bethard at gmail.com>
- **创建时间**: 25-Sep-2009
- **Python 版本**: 2.7, 3.2
- **发布历史**: 27-Sep-2009, 24-Oct-2009

## 接受

本 PEP 于 2010 年 2 月 21 日在 python-dev 上由 Guido 批准 [17]。

## 摘要

本 PEP 提议将 argparse [1] 模块包含在 Python 2.7 和 3.2 的标准库中。

## 动机

argparse 模块是一个命令行解析库，它提供的功能比标准库中现有的命令行解析模块 getopt [2] 和 optparse [3] 更多。它包括对位置参数（不仅仅是选项）、子命令、必需选项、像"/f"和"+rgb"这样的选项语法、零个或多个以及一个或多个风格的参数的支持，以及其他两个模块缺乏的许多其他功能。

argparse 模块也已经是一个流行的第三方替换模块。它被用于 IPython（Scipy Python shell）[4] 等项目，包含在 Debian testing 和 unstable [5] 中，并且自 2007 年以来一直有各种请求将其包含在标准库中 [6] [7] [8]。这种受欢迎程度表明它可能是 Python 库的有价值补充。

## 为什么 getopt 和 optparse 不够？

反对添加 argparse 的一个论点是"标准库中已经有两个不同的选项解析模块"[9]。以下是 argparse 提供但在 getopt 或 optparse 中不存在的功能列表：

- 虽然确实有两个选项解析库，但没有完整的命令行解析库——getopt 和 optparse 都只支持选项，不支持位置参数。argparse 模块处理两者，因此能够生成更好的帮助消息，避免像 optparse 通常需要的 usage= 字符串这样的冗余。

- argparse 模块重视实用性而非纯粹性。因此，argparse 允许必需选项和自定义用于识别选项的字符，而 optparse 明确声明"'required option'这个短语是自相矛盾的"，并且选项语法 -pf、-file、+f、+rgb、/f 和/file "不被 optparse 支持，而且永远不会被支持"。

- argparse 模块允许选项使用 nargs='?'、nargs='*'或 nargs='+'接受可变数量的参数。optparse 模块为此功能的部分内容提供了一个未测试的食谱 [10]，但承认"当你想要一个选项接受可变数量的参数时，事情变得棘手。"

- argparse 模块支持子命令，其中主命令行解析器根据命令行参数分发到其他命令行解析器。这是命令行接口中的常见模式，例如 svn co 和 svn up。

## 为什么功能没有被添加到 optparse？

显然，上述所有功能都比 optparse 提供的有所改进。一个合理的问题是，为什么这些功能不作为补丁提供给 optparse，而是引入一个全新的模块。事实上，argparse 的原始开发打算这样做，但由于 optparse 的各种相当限制性的设计决策，这实际上是不可能的。一些问题包括：

- optparse 模块暴露了其解析算法的内部细节。特别是，parser.largs 和 parser.rargs 保证对回调可用 [11]。这使得改进解析算法变得极其困难，而 argparse 中正确处理位置参数和可变长度参数需要这样做。例如，argparse 中的 nargs='+'使用正则表达式匹配，因此没有像 parser.largs 这样的概念。

- optparse 扩展 API 极其复杂。例如，只是使用一个简单的自定义字符串到对象的转换函数，你必须子类化 Option、hack 类属性，然后向解析器指定自定义选项类型，如下所示：

```python
class MyOption(Option):
    TYPES = Option.TYPES + ("mytype",)
    TYPE_CHECKER = copy(Option.TYPE_CHECKER)
    TYPE_CHECKER["mytype"] = check_mytype
parser = optparse.OptionParser(option_class=MyOption)
parser.add_option("-m", type="mytype")
```

相比之下，argparse 简单允许转换函数直接用作 type= 参数，例如：

```python
parser = argparse.ArgumentParser()
parser.add_option("-m", type=check_mytype)
```

但鉴于 optparse 的华丽自定义 API，不清楚这样的功能应该如何与这些 API 交互，引入简单的 argparse API 很可能会破坏现有的自定义 Option 代码。

- optparse 和 argparse 都解析命令行参数并将它们分配为 parse_args 返回的对象属性。但是，optparse 模块保证自定义操作的 take_action 方法将始终被传递一个提供 ensure_value 方法的 values 对象 [12]，而 argparse 模块允许将属性分配给任何对象，例如：

```python
foo_object = ...
parser.parse_args(namespace=foo_object)
foo_object.some_attribute_parsed_from_command_line
```

修改 optparse 以允许传入任何对象将很困难，因为简单地传递 foo_object 而不是 Values 实例将破坏依赖 ensure_value 方法的现有自定义操作。

由于这些问题，使得 argparse 与 optparse API 保持兼容变得不合理地困难，argparse 被开发为一个独立模块。鉴于这些问题，将所有 argparse 功能合并到 optparse 中而不向后不兼容似乎不太可能。

## optparse 的弃用

因为 optparse 的所有功能在 argparse 中都可用，optparse 模块将被弃用。然而，由于 optparse 的广泛使用，弃用策略仅包含文档更改和默认情况下不可见的警告：

- **Python 2.7+ 和 3.2+** – optparse 文档中将添加以下说明：
  optparse 模块已弃用，将不再开发；开发将继续在 argparse 模块中进行。

- **Python 2.7+** – 如果在命令行提供 Python 3 兼容性标志 -3，则导入 optparse 将发出 DeprecationWarning。否则不会发出警告。

- **Python 3.2+** – 导入 optparse 将发出 PendingDeprecationWarning，默认不显示。

注意，没有为 optparse 提议删除日期。

## getopt 文档的更新

getopt 模块不会被弃用。但是，它的文档将在几个地方更新为指向 argparse。在模块顶部，将添加以下说明：

> getopt 模块是一个命令行选项解析器，其 API 设计为对 C getopt 函数的用户熟悉。
> 对 C getopt 函数不熟悉或希望编写更少代码并获得更好帮助和错误消息的用户应考虑使用 argparse 模块。

此外，在最后的 getopt 示例之后，将添加以下说明：

注意，通过使用 argparse 模块，可以用更少的代码生成等效的命令行接口：

```python
import argparse

if __name__ == '__main__':
    parser = argparse.ArgumentParser()
    parser.add_argument('-o', '--output')
    parser.add_argument('-v', dest='verbose', action='store_true')
    args = parser.parse_args()
    # ... do something with args.output ...
    # ... do something with args.verbose ..
```

## 延迟：字符串格式化

argparse 模块支持从 2.3 到 3.2 的 Python，因此依赖于传统的`%(foo)s` 风格字符串格式化。有人建议最好使用新风格的`{foo}` 字符串格式化 [13]。关于如何最好地为标准库中的模块这样做有一些讨论 [14]，几个人正在开发用于自动将%-formatting 转换为{}-formatting 的函数 [15] [16]。当其中一个被添加到标准库时，argparse 将使用它们来支持两种格式化风格。

## 拒绝：getopt 兼容方法

以前，当本 PEP 建议弃用 getopt 以及 optparse 时，有一些关于添加像这样的方法的讨论：

```python
ArgumentParser.add_getopt_arguments(options[, long_options])
```

然而，出于以下原因不会添加此方法：

- getopt 模块没有被弃用，所以需求较少。

- 这个方法实际上不会减轻任何已经在维护使用说明的 getopt 用户的过渡，因为上面的 API 没有提供向参数添加帮助消息的方法。

- getopt 的一些用户认为只需要单个函数调用非常重要。上面的 API 不满足此要求，因为还必须调用 `ArgumentParser()` 和 `parse_args()`。

## 范围之外：各种功能请求

在本 PEP 的讨论中提出了几个 argparse 的功能请求：

- 支持从环境变量获取参数默认值

- 支持从配置文件获取参数默认值

- 支持"foo --help subcommand"，除了当前支持的"foo subcommand --help"

这些都是 argparse 模块的合理功能请求，但超出了本 PEP 的范围，已重定向到 argparse 问题跟踪器。

## 讨论：sys.stderr 和 sys.exit

有一些担心 argparse 默认总是写入 sys.stderr 并且在提供无效参数时总是调用 sys.exit。这是围绕简单命令行接口的绝大多数 argparse 用例的期望行为。然而，在某些情况下，可能希望保持 argparse 不退出，或者让它将消息写入 sys.stderr 之外的东西。这些用例可以通过子类化 ArgumentParser 并覆盖 exit 或_print_message 方法来支持。后者是一个未记录的实现细节，但如果这成为一个常见需求，它可以正式公开。
