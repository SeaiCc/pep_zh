# PEP 324 – subprocess - 子进程模块

**元数据：**

- **作者**: Peter Astrand <astrand&#32;&#97;t&#32;lysator.liu.se>
- **创建日期**: 2003 年 11 月 19 日
- **Python 版本**: 2.4

## 摘要

本 PEP 描述了一个用于启动和与进程通信的新模块。

## 动机

启动新进程是任何编程语言中的常见任务，在像 Python 这样的高级语言中非常常见。需要良好的支持来完成此任务，因为：

- 不适当的启动进程函数可能意味着安全风险：如果程序通过 shell 启动，并且参数包含 shell 元字符，结果可能是灾难性的。[1]

- 它使 Python 成为过度复杂的 shell 脚本的更好的替代语言。

目前，Python 有许多不同的进程创建函数。这使得开发者难以选择。

subprocess 模块提供了以下优于先前函数的增强功能：

- 一个"统一"的模块提供所有先前函数的功能。

- 跨进程异常：在新进程开始执行之前在子进程中发生的异常会在父进程中重新引发。这意味着很容易处理 exec() 失败等。例如，使用 popen2，不可能检测执行是否失败。

- 在 fork 和 exec 之间执行自定义代码的钩子。这可用于，例如，更改 uid。

- 没有隐式调用 /bin/sh。这意味着不需要转义危险的 shell 元字符。

- 所有文件描述符重定向的组合都是可能的。例如，"python-dialog" [2] 需要生成一个进程并重定向 stderr，但不重定向 stdout。使用当前函数，不使用临时文件，这是不可能的。

- 使用 subprocess 模块，可以控制在执行新程序之前是否应该关闭所有打开的文件描述符。

- 支持连接多个子进程（shell"管道"）。

- 通用换行符支持。

- communicate() 方法，使发送 stdin 数据和读取 stdout 和 stderr 数据变得容易，而不必担心死锁。大多数人都意识到与子进程通信相关的流量控制问题，但并非所有人都有耐心或技能编写完全正确且无死锁的 select 循环。这意味着许多 Python 应用程序包含竞态条件。标准库中的 communicate() 方法解决了这个问题。

## 理由

以下要点总结了设计：

- subprocess 基于 popen2，这是经过验证的。

- popen2 中的工厂函数已被删除，因为我认为类构造函数同样易于使用。

- popen2 包含多个用于不同重定向组合的工厂函数和类。然而，subprocess 只包含一个类。由于 subprocess 模块支持 12 种不同的重定向组合，为每一种提供类或函数将是繁琐且不直观的。即使使用 popen2，这也是一个可读性问题。例如，许多人无法在不使用文档的情况下区分 popen2.popen2 和 popen2.popen4。

- 提供一个小的实用函数：subprocess.call()。它旨在作为 os.system() 的增强，同时仍然非常易于使用：它不使用 Standard C 函数 system()，后者有限制。

- 它不隐式调用 shell。

- 不需要引号；使用参数列表。

- 返回值更容易处理。

`call()` 实用函数接受一个'args'参数，就像 `Popen` 类构造函数一样。它等待命令完成，然后返回 returncode 属性。实现非常简单：

```python
def call(*args, **kwargs):
    return Popen(*args, **kwargs).wait()
```

call() 函数背后的动机很简单：启动一个进程并等待其完成是一个常见的任务。

虽然 Popen 支持广泛的选择，但许多用户有简单需求。许多人今天使用 os.system()，主要是因为它提供了一个简单的接口。考虑这个示例：

```python
os.system("stty sane -F " + device)
```

使用 subprocess.call()，这将看起来像：

```python
subprocess.call(["stty", "sane", "-F", device])
```

或者，如果通过 shell 执行：

```python
subprocess.call("stty sane -F " + device, shell=True)
```

"preexec"功能使得在 fork 和 exec 之间运行任意代码成为可能。人们可能会问为什么有专门用于设置环境和当前目录的参数，但没有用于设置 uid 的参数。答案是：更改环境和工作目录被认为是相当常见的。像 spawn() 这样的旧函数支持"env"参数。env 和 cwd 被认为是跨平台的：它们甚至在 Windows 上也有意义。

在 POSIX 平台上，不需要扩展模块：该模块使用 os.fork()、os.execvp() 等。在 Windows 平台上，该模块需要 Mark Hammond 的 Windows 扩展 [5]，或一个名为 _subprocess 的小扩展模块。

## 规范

此模块定义了一个名为 Popen 的类：

```python
class Popen(args, bufsize=0, executable=None,
            stdin=None, stdout=None, stderr=None,
            preexec_fn=None, close_fds=False, shell=False,
            cwd=None, env=None, universal_newlines=False,
            startupinfo=None, creationflags=0):
```

参数为：

- args 应该是一个字符串或程序参数序列。要执行的程序通常是 args 序列或字符串中的第一个项目，但可以使用 executable 参数显式设置。

在 UNIX 上，shell=False（默认）：在这种情况下，Popen 类使用 os.execvp() 来执行子程序。args 通常应该是一个序列。字符串将被视为只有一个项目（要执行的程序）的序列。

在 UNIX 上，shell=True：如果 args 是字符串，它指定通过 shell 执行的命令字符串。如果 args 是序列，第一个项目指定命令字符串，任何附加项目将被视为附加 shell 参数。

在 Windows 上：Popen 类使用 CreateProcess() 来执行子程序，它操作字符串。如果 args 是序列，它将使用 list2cmdline 方法转换为字符串。请注意，并非所有 MS Windows 应用程序都以相同的方式解释命令行：list2cmdline 是为使用与 MS C 运行时相同规则的应用程序设计的。

- bufsize，如果给定，具有与内置 open() 函数相应参数相同的含义：0 表示无缓冲，1 表示行缓冲，任何其他正值表示使用（大约）该大小的缓冲区。负 bufsize 表示使用系统默认值，这通常意味着完全缓冲。bufsize 的默认值为 0（无缓冲）。

- stdin、stdout 和 stderr 分别指定执行程序的标准输入、标准输出和标准错误文件句柄。有效值是 PIPE、现有文件描述符（正整数）、现有文件对象和 None。PIPE 表示应该创建到子进程的新管道。对于 None，不会发生重定向；子进程的文件句柄将从父进程继承。此外，stderr 可以是 STDOUT，这表示来自应用程序的 stderr 数据应该被捕获到与 stdout 相同的文件句柄中。

- 如果 preexec_fn 设置为可调用对象，此对象将在子进程执行之前在子进程中调用。

- 如果 close_fds 为 true，则在执行子进程之前将关闭除 0、1 和 2 之外的所有文件描述符。

- 如果 shell 为 true，则指定的命令将通过 shell 执行。

- 如果 cwd 不是 None，则在执行子进程之前当前目录将更改为 cwd。

- 如果 env 不是 None，它定义新进程的环境变量。

- 如果 universal_newlines 为 true，文件对象 stdout 和 stderr 作为文本文件打开，但行可以由`\n`（Unix 行尾约定）、`\r`（Macintosh 约定）或`\r\n`（Windows 约定）中的任何一个终止。Python 程序将所有这些外部表示都视为`\n`。注意：此功能仅在 Python 使用通用换行符支持构建时可用（默认）。此外，文件对象 stdout、stdin 和 stderr 的 newlines 属性不会被 communicate() 方法更新。

- startupinfo 和 creationflags，如果给定，将传递给底层的 CreateProcess() 函数。它们可以指定新进程的主窗口外观和优先级等事项。（仅限 Windows）

此模块还定义了两个快捷函数：

- `call(*args, **kwargs)`: 
使用参数运行命令。等待命令完成，然后返回 returncode 属性。参数与 Popen 构造函数相同。

示例：

```python
retcode = call(["ls", "-l"])
```

### 异常

在新程序开始执行之前在子进程中引发的异常将在父进程中重新引发。此外，异常对象将有一个名为'child_traceback'的额外属性，它是一个包含从子进程角度看的 traceback 信息的字符串。

最常见的异常是 OSError。例如，当尝试执行不存在的文件时就会发生这种情况。应用程序应该准备处理 OSErrors。

如果 Popen 被调用时参数无效，将引发 ValueError。

### 安全

与其他一些 popen 函数不同，此实现永远不会隐式调用 /bin/sh。这意味着所有字符，包括 shell 元字符，都可以安全地传递给子进程。

### Popen 对象

Popen 类的实例具有以下方法：

- poll() 检查子进程是否已终止。返回 returncode 属性。

- wait() 等待子进程终止。返回 returncode 属性。

- communicate(input=None) 与进程交互：向 stdin 发送数据。从 stdout 和 stderr 读取数据，直到到达文件末尾。等待进程终止。可选的 stdin 参数应该是发送到子进程的字符串，或者如果不应向子进程发送数据，则为 None。communicate() 返回一个元组 (stdout, stderr)。

注意：读取的数据在内存中缓冲，因此如果数据量大或无限，请不要使用此方法。

以下属性也可用：

- `stdin` 如果 stdin 参数是 PIPE，此属性是一个提供子进程输入的文件对象。否则，它是 None。

- `stdout` 如果 stdout 参数是 PIPE，此属性是一个提供子进程输出的文件对象。否则，它是 None。

- `stderr` 如果 stderr 参数是 PIPE，此属性是一个提供子进程错误输出的文件对象。否则，它是 None。

- pid 子进程的进程 ID。

- returncode 子进程返回码。None 值表示进程尚未终止。负值 -N 表示子进程被信号 N 终止（仅限 UNIX）。

## 使用 subprocess 模块替换旧函数

在本节中，"a ==> b"意味着 b 可以用作 a 的替代品。

注意：本节中的所有函数在找不到执行的程序时都会（或多或少）静默失败；此模块引发 OSError 异常。

在以下示例中，我们假设 subprocess 模块使用 from subprocess import * 导入。

### 替换 /bin/sh shell 反引号

```python
output=`mycmd myarg`
==> 
output = Popen(["mycmd", "myarg"], stdout=PIPE).communicate()[0]
```

### 替换 shell 管道行

```python
output=`dmesg | grep hda`
==> 
p1 = Popen(["dmesg"], stdout=PIPE)
p2 = Popen(["grep", "hda"], stdin=p1.stdout, stdout=PIPE)
output = p2.communicate()[0]
```

### 替换 os.system()

```python
sts = os.system("mycmd" + " myarg")
==> 
p = Popen("mycmd" + " myarg", shell=True)
sts = os.waitpid(p.pid, 0)
```

注意：

通过 shell 调用程序通常不是必需的。

- 查看 returncode 属性比退出状态更容易。

更现实的示例如下：

```python
try:
    retcode = call("mycmd" + " myarg", shell=True)
    if retcode < 0:
        print >>sys.stderr, "Child was terminated by signal", -retcode
    else:
        print >>sys.stderr, "Child returned", retcode
except OSError, e:
    print >>sys.stderr, "Execution failed:", e
```

### 替换 os.spawn*

P_NOWAIT 示例：

```python
pid = os.spawnlp(os.P_NOWAIT, "/bin/mycmd", "mycmd", "myarg")
==> 
pid = Popen(["/bin/mycmd", "myarg"]).pid
```

P_WAIT 示例：

```python
retcode = os.spawnlp(os.P_WAIT, "/bin/mycmd", "mycmd", "myarg")
==> 
retcode = call(["/bin/mycmd", "myarg"])
```

向量示例：

```python
os.spawnvp(os.P_NOWAIT, path, args)
==> 
Popen([path] + args[1:])
```

环境示例：

```python
os.spawnlpe(os.P_NOWAIT, "/bin/mycmd", "mycmd", "myarg", env)
==> 
Popen(["/bin/mycmd", "myarg"], env={"PATH": "/usr/bin"})
```

### 替换 os.popen*

```python
pipe = os.popen(cmd, mode='r', bufsize)
==> 
pipe = Popen(cmd, shell=True, bufsize=bufsize, stdout=PIPE).stdout

pipe = os.popen(cmd, mode='w', bufsize)
==> 
pipe = Popen(cmd, shell=True, bufsize=bufsize, stdin=PIPE).stdin

(child_stdin, child_stdout) = os.popen2(cmd, mode, bufsize)
==> 
p = Popen(cmd, shell=True, bufsize=bufsize,
          stdin=PIPE, stdout=PIPE, close_fds=True)
(child_stdin, child_stdout) = (p.stdin, p.stdout)

(child_stdin, child_stdout, child_stderr) = os.popen3(cmd, mode, bufsize)
==> 
p = Popen(cmd, shell=True, bufsize=bufsize,
          stdin=PIPE, stdout=PIPE, stderr=PIPE, close_fds=True)
(child_stdin, child_stdout, child_stderr) = (p.stdin, p.stdout, p.stderr)

(child_stdin, child_stdout_and_stderr) = os.popen4(cmd, mode, bufsize)
==> 
p = Popen(cmd, shell=True, bufsize=bufsize,
          stdin=PIPE, stdout=PIPE, stderr=STDOUT, close_fds=True)
(child_stdin, child_stdout_and_stderr) = (p.stdin, p.stdout)
```

### 替换 popen2.*

注意：如果 popen2 函数的 cmd 参数是字符串，命令通过 /bin/sh 执行。如果是列表，命令直接执行。

```python
(child_stdout, child_stdin) = popen2.popen2("somestring", bufsize, mode)
==> 
p = Popen(["somestring"], shell=True, bufsize=bufsize,
          stdin=PIPE, stdout=PIPE, close_fds=True)
(child_stdout, child_stdin) = (p.stdout, p.stdin)

(child_stdout, child_stdin) = popen2.popen2(["mycmd", "myarg"], bufsize, mode)
==> 
p = Popen(["mycmd", "myarg"], bufsize=bufsize,
          stdin=PIPE, stdout=PIPE, close_fds=True)
(child_stdout, child_stdin) = (p.stdout, p.stdin)
```

popen2.Popen3 和 popen3.Popen4 基本上工作如 subprocess.Popen，除了：

- subprocess.Popen 在执行失败时引发异常

- capturestderr 参数被 stderr 参数替换。

- 必须指定 stdin=PIPE 和 stdout=PIPE。

- popen2 默认关闭所有文件描述符，但你必须用 subprocess.Popen 指定 close_fds=True。

## 未解决的问题

一些功能已被请求但尚未实现。这包括：

- 支持管理整个 flock 的子进程

- 支持管理"守护"进程

- 内置杀死子进程的方法

虽然这些是有用的功能，但预计这些可以稍后添加，没有问题。

- expect-like 功能，包括 pty 支持。

pty 支持高度依赖于平台，这是一个问题。此外，已经有其他模块提供这种功能 [6]。

## 向后兼容性

由于这是一个新模块，预计不会有主要的向后兼容问题。模块名称"subprocess"可能与其它同名模块 [3] 冲突，但名称"subprocess"似乎是目前为止建议的最佳名称。此模块的第一个名称是"popen5"，但这个名称被认为太不直观。有一段时间，模块被称为"process"，但这个名称已经被 Trent Mick 的模块 [4] 使用。

此新模块试图替换的函数和模块（os.system、os.spawn*、os.popen*、popen2.*、commands.*）预计在将来的 Python 版本中长期可用，以保持向后兼容性。
