## 摘要

这是一个为 Python 3000 添加抽象基类（ABC）支持的提案。它建议：

- 一种重载 isinstance() 和 issubclass() 的方法。

- 一个新的模块 abc，作为"ABC 支持框架"。它定义了一个用于 ABC 的元类和一个可用于定义抽象方法的装饰器。

- 为容器和迭代器定义特定的 ABC，将添加到 collections 模块中。

该提案中的许多思考并非关于 ABC 的具体机制（与接口或通用函数（GFs）相比），而是关于阐明哲学问题，如"什么构成一个集合"、"什么构成一个映射"以及"什么构成一个序列"。

还有一个配套 PEP 3141，它定义了数值类型的 ABC。

## 理由

在面向对象编程领域中，与对象交互的使用模式可以分为两个基本类别，即"调用"和"检查"。

调用意味着通过调用对象的方法来与对象交互。通常这与多态性相结合，因此调用给定的方法可能会根据对象的类型运行不同的代码。

检查意味着外部代码（在对象方法之外）能够检查该对象的类型或属性，并根据该信息决定如何对待该对象。

这两种使用模式服务于相同的一般目的，即能够以统一的方式处理多样化和潜在新颖的对象，同时允许为每种不同类型的对象定制处理决策。

在经典 OOP 理论中，调用是首选的使用模式，而检查则受到积极劝阻，被认为是早期过程化编程风格的遗留物。然而，在实践中，这种观点过于教条和僵化，导致了一种与设计灵活性相悖的设计刚性，这与 Python 等语言的动态性质非常不符。

特别是，经常需要以对象类创建者未预料到的方式处理对象。将满足每个可能用户需求的内建方法构建到每个对象中并不总是最佳解决方案。此外，有许多强大的调度哲学与经典的 OOP 要求（行为严格封装在对象内）直接相对，例如基于规则或模式匹配的逻辑。

另一方面，经典 OOP 理论家对检查的批评之一是缺乏形式化和检查的临时性质。在像 Python 这样的语言中，对象的几乎任何方面都可以被外部代码反射和直接访问，有许多不同的方法来测试对象是否符合特定协议。例如，如果问"这个对象是一个可变序列容器吗？"，可以寻找"list"的基类，或者寻找名为"`__getitem__`"的方法。但请注意，虽然这些测试可能看起来很明显，但它们都不正确，因为一个会产生假阴性，另一个会产生假阳性。

普遍同意的补救方法是标准化测试，并将它们分组到正式安排中。这最容易通过将一组标准可测试属性与每个类相关联来完成，通过继承机制或其他手段。每个测试都带有一组承诺：它包含关于类一般行为的承诺，以及关于哪些其他类方法将可用的承诺。

本 PEP 提出了一种用于组织这些测试的特定策略，称为抽象基类（Abstract Base Classes），或 ABC。ABC 只是被添加到对象继承树中的 Python 类，用于向外部检查者信号该对象的某些特征。测试使用 isinstance() 完成，特定 ABC 的存在意味着测试已通过。

此外，ABC 定义了一组最小的方法，这些方法建立了类型的特征行为。基于其 ABC 类型区分类别的代码可以信任这些方法将始终存在。这些方法中的每一个都伴随着一个广义的抽象语义定义，在 ABC 的文档中描述。这些标准语义定义不被强制执行，但强烈建议遵循。

像 Python 中的所有其他事物一样，这些承诺本质上是友好协议，这意味着虽然语言确实强制执行 ABC 中做出的一些承诺，但具体类的实现者要确保其余承诺得到遵守。

## 规范

规范遵循摘要中列出的类别：

- 重载 isinstance() 和 issubclass() 的方法。

- 一个新的模块 abc，作为"ABC 支持框架"。它定义了一个用于 ABC 的元类和一个可用于定义抽象方法的装饰器。

- 为容器和迭代器定义特定的 ABC，将添加到 collections 模块中。

### 重载 isinstance() 和 issubclass()

在开发本 PEP 及其配套 PEP 3141 期间，我们反复面临在标准化更多细粒度 ABC 还是更少粗粒度 ABC 之间的选择。例如，在某个阶段，PEP 3141 引入了以下用于复数的基类堆栈：MonoidUnderPlus、AdditiveGroup、Ring、Field、Complex（每个都派生自前一个）。讨论还提到了几个其他代数分类，这些分类被排除在外：Algebraic、Transcendental、IntegralDomain 和 PrincipalIdealDomain。在当前 PEP 的早期版本中，我们考虑了像 Set、ComposableSet、MutableSet、HashableSet、MutableComposableSet、HashableComposableSet 这样单独类的用例。

这里的困境是，我们宁愿有更少的 ABC，那么需要不太精细 ABC 的用户该怎么办？例如，考虑一个数学家想要定义自己种类的 Transcendental 数字，但也希望 float 和 int 被视为 Transcendental 的困境。PEP 3141 最初提议为此修补 `float.__bases__`，但有一些很好的理由保持内置类型的不可变性（其中一个原因是，它们在同一地址空间中运行的所有 Python 解释器之间共享，就像 mod_python [16] 使用的那样）。

另一个例子是某人想要为任何具有 append() 方法的序列定义通用函数（PEP 3124）。Sequence ABC（见下文）不承诺 append() 方法，而 MutableSequence 不仅要求 append()，还要求各种其他变异方法。

为了解决这些和类似的困境，下一节将提出一个用于 ABC 的元类，这将允许我们将 ABC 作为"虚拟基类"（与 C++ 中的概念不同）添加到任何类，包括另一个 ABC。这允许标准库定义 ABC Sequence 和 MutableSequence，并将这些注册为内置类型（如 basestring、tuple 和 list）的虚拟基类，因此例如以下条件都为真：

```python
isinstance([], Sequence)
issubclass(list, Sequence)
issubclass(list, MutableSequence)
isinstance((), Sequence)
not issubclass(tuple, MutableSequence)
isinstance("", Sequence)
issubclass(bytearray, MutableSequence)
```

这里提出的主要机制是允许重载内置函数 isinstance() 和 issubclass()。重载的工作原理如下：调用 isinstance(x, C) 首先检查 C.`__instancecheck__` 是否存在，如果存在，则调用 `C.__instancecheck__(x)` 而不是其正常实现。同样，调用 issubclass(D, C) 首先检查 `C.__subclasscheck__` 是否存在，如果存在，则调用 `C.__subclasscheck__(D)` 而不是其正常实现。

注意，魔术名称不是 `__isinstance__` 和 `__issubclass__`；这是因为参数的反转可能导致混淆，尤其是对于 issubclass() 重载器。

此功能的原型实现在 [12] 中给出。

这是一个带有（天真简单）实现` __instancecheck__` 和 `__subclasscheck__` 的示例：

```python
class ABCMeta(type):

    def __instancecheck__(cls, inst):
        """Implement isinstance(inst, cls)."""
        return any(cls.__subclasscheck__(c)
                  for c in {type(inst), inst.__class__})

    def __subclasscheck__(cls, sub):
        """Implement issubclass(sub, cls)."""
        candidates = cls.__dict__.get("__subclass__", set()) | {cls}
        return any(c in candidates for c in sub.mro())

class Sequence(metaclass=ABCMeta):
    __subclass__ = {list, tuple}

assert issubclass(list, Sequence)
assert issubclass(tuple, Sequence)

class AppendableSequence(Sequence):
    __subclass__ = {list}

assert issubclass(list, AppendableSequence)
assert isinstance([], AppendableSequence)

assert not issubclass(tuple, AppendableSequence)
assert not isinstance((), AppendableSequence)
```

下一节提出了一个全功能实现。

### abc 模块：一个 ABC 支持框架

新的标准库模块 abc，用纯 Python 编写，作为 ABC 支持框架。它定义了一个元类 ABCMeta 和装饰器 @abstractmethod 和 @abstractproperty。示例实现由 [13] 给出。

ABCMeta 类覆盖了 `__instancecheck__` 和 `__subclasscheck__`，并定义了一个 register 方法。register 方法接受一个参数，必须是一个类；在调用 B.register(C) 之后，调用 issubclass(C, B) 将返回 True，因为 `B.__subclasscheck__(C)` 返回 True。此外，isinstance(x, B) 等同于 `issubclass(x.__class__, B)` 或 `issubclass(type(x), B)`。（`type(x)` 和 `x.__class__` 可能不是同一个对象，例如当 x 是一个代理对象时。）

这些方法旨在在元类为（派生自）ABCMeta 的类上调用；例如：

```python
from abc import ABCMeta

class MyABC(metaclass=ABCMeta):
    pass

MyABC.register(tuple)

assert issubclass(tuple, MyABC)
assert isinstance((), MyABC)
```

最后两个断言等同于以下两个：

```python
assert MyABC.__subclasscheck__(tuple)
assert MyABC.__instancecheck__(())
```

当然，你也可以直接子类化 MyABC：

```python
class MyClass(MyABC):
    pass

assert issubclass(MyClass, MyABC)
assert isinstance(MyClass(), MyABC)
```

当然，tuple 不是 MyClass：

```python
assert not issubclass(tuple, MyClass)
assert not isinstance((), MyClass)
```

你可以注册另一个类作为 MyClass 的子类：

```python
MyClass.register(list)

assert issubclass(list, MyClass)
assert issubclass(list, MyABC)
```

你也可以注册另一个 ABC：

```python
class AnotherClass(metaclass=ABCMeta):
    pass

AnotherClass.register(basestring)

MyClass.register(AnotherClass)

assert isinstance(str, MyABC)
```

最后一个断言需要追踪以下超类 - 子类关系：

```
MyABC -> MyClass（使用常规子类化）
MyClass -> AnotherClass（使用注册）
AnotherClass -> basestring（使用注册）
basestring -> str（使用常规子类化）
```

abc 模块还定义了一个新的装饰器 @abstractmethod，用于声明抽象方法。包含至少一个用此装饰器声明但尚未被覆盖的方法的类不能被实例化。此类方法可以从子类中的覆盖方法调用（使用 super 或直接调用）。例如：

```python
from abc import ABCMeta, abstractmethod

class A(metaclass=ABCMeta):
    @abstractmethod
    def foo(self): pass

A()  # raises TypeError

class B(A):
    pass

B()  # raises TypeError

class C(A):
    def foo(self): print(42)

C()  # works
```

注意：@abstractmethod 装饰器应仅用于类体内，且仅用于元类为（派生自）ABCMeta 的类。动态地向类添加抽象方法，或尝试在方法或类创建后修改其抽象状态，均不受支持。@abstractmethod 仅影响使用常规继承派生的子类；使用 register() 方法注册的"虚拟子类"不受影响。

实现：@abstractmethod 装饰器将函数属性 `__isabstractmethod__` 设置为值 True。`ABCMeta.__new__` 方法计算类型属性 `__abstractmethods__`，作为具有 `__isabstractmethod__` 属性且值为 true 的所有方法名称的集合。它通过组合基类的 `__abstractmethods__` 属性，添加新类 dict 中具有 true `__isabstractmethod__` 属性的所有方法名称，并移除新类 dict 中不具有 true `__isabstractmethod__` 属性的所有方法名称来完成此操作。如果 resulting `__abstractmethods__` 集合非空，则该类被视为抽象的，尝试实例化它将引发 TypeError。（如果这在 CPython 中实现，则可以使用内部标志 Py_TPFLAGS_ABSTRACT 来加速此检查 [6]。）

讨论：与 Java 的抽象方法或 C++ 的纯抽象方法不同，此处定义的抽象方法可以有实现。此实现可以通过 super 机制从覆盖它的类调用。这对于使用协作多重继承的框架 [7]、[8] 中的 super-call 的端点可能很有用。

定义了第二个装饰器 @abstractproperty，用于定义抽象数据属性。它的实现是内置 property 类的子类，添加了 `__isabstractmethod__` 属性：

```python
class abstractproperty(property):
    __isabstractmethod__ = True
```

它可以用两种方式使用：

```python
class C(metaclass=ABCMeta):

    # 只读属性：

    @abstractproperty
    def readonly(self):
        return self.__x

    # 可读写属性（不能使用装饰器语法）：

    def getx(self):
        return self.__x
    def setx(self, value):
        self.__x = value
    x = abstractproperty(getx, setx)
```

与抽象方法类似，继承抽象属性（使用装饰器语法或较长形式声明）的子类，除非用具体属性覆盖该抽象属性，否则不能被实例化。

### 用于容器和迭代器的 ABC

collections 模块将定义必要且足够的 ABC，用于处理集合、映射、序列以及一些辅助类型，如迭代器和字典视图。所有 ABC 都具有上述 ABCMeta 作为它们的元类。

ABC 提供了它们抽象方法的实现，这些实现在技术上有效但相当无用；例如，`__hash__` 返回 0，`__iter__` 返回空迭代器。通常，抽象方法表示指定类型的空容器的行为。

一些 ABC 还提供了具体的（即非抽象的）方法；例如，Iterator 类有一个 `__iter__` 方法返回自身，满足了迭代器的一个重要不变量（在 Python 2 中，每个迭代器类都必须重新实现）。这些 ABC 可以被视为"混合"类。

PEP 中定义的任何 ABC 都不覆盖 `__init__、__new__、__str__ 或 __repr__`。定义标准构造函数签名将不必要地限制自定义容器类型，例如 Patricia 树或 gdbm 文件。为集合定义特定字符串表示也留给各个实现。

注意：没有用于排序操作（`__lt__、__le__、__ge__、__gt__`）的 ABC。在基类（抽象或非抽象）中定义这些会遇到第二个操作数接受类型的问题。例如，如果类 Ordering 定义了 `__lt__`，人们会假设对于任何 Ordering 实例 x 和 y，x < y 都有定义（即使它只是定义了偏序）。但这不可能是真的：如果 list 和 str 都派生自 Ordering，这将意味着 [1, 2] < (1, 2) 应该有定义（并且可能返回 False），而实际上（在 Python 3000 中！）这样的"混合模式比较"操作被明确禁止并引发 TypeError。有关更多信息，请参阅 PEP 3100 和 [14]。（这是采用另一个相同类型参数的操作的更一般问题的特例）。

#### 单项马（One Trick Ponies）

这些抽象类代表单个方法，如 `__iter__` 或 `__len__`。

**Hashable** 
定义 `__hash__` 的类的基类。`__hash__` 方法应返回一个整数。抽象 `__hash__` 方法总是返回 0，这是一个有效（但效率低下）的实现。**不变量**：如果类 C1 和 C2 都派生自 Hashable，则条件 o1 == o2 必须隐含 hash(o1) == hash(o2)，对于 C1 的所有实例 o1 和 C2 的所有实例 o2。换句话说，两个对象不应比较相等，如果它们有不同的哈希值。

另一个约束是，哈希对象一旦被创建，应该永远不会改变它们的值（通过 == 比较）或它们的哈希值。如果类不能保证这一点，它不应派生自 Hashable；如果它不能为某些实例保证这一点，则那些实例的 `__hash__` 应引发 TypeError 异常。

注意：成为此类的实例并不意味着对象是不可变的；例如，包含列表作为成员的元组不是不可变的；其 `__hash__` 方法引发 TypeError。（这是因为它递归地尝试计算每个成员的哈希值；如果成员不可哈希，则引发 TypeError。）

**Iterable** 
定义 `__iter__` 的类的基类。`__iter__` 方法应始终返回 Iterator 的实例（见下文）。抽象 `__iter__` 方法返回空迭代器。

**Iterator** 
定义 `__next__` 的类的基类。它派生自 Iterable。抽象 `__next__` 方法引发 StopIteration。具体 `__iter__` 方法返回 self。注意 Iterable 和 Iterator 之间的区别：Iterable 可以被迭代，即支持 `__iter__` 方法；Iterator 是内置函数 iter() 返回的对象，即支持 `__next__` 方法。

**Sized** 
定义 `__len__` 的类的基类。`__len__` 方法应返回一个 Integer（见下文"Numbers"）>= 0。抽象 `__len__` 方法返回 0。**不变量**：如果类 C 派生自 Sized 以及 Iterable，则不变量 `sum(1 for x in c) == len(c)` 应对 C 的任何实例 c 成立。

**Container** 
定义 `__contains__` 的类的基类。`__contains__` 方法应返回一个 bool。抽象 `__contains__` 方法返回 False。**不变量**：如果类 C 派生自 Container 以及 Iterable，则 (x in c for x in c) 应该是一个生成器，仅产生 True 值，对于 C 的任何实例 c。

**开放问题**：
理论上，这些类可以不使用 ABCMeta 元类，而是覆盖 `__instancecheck__` 和 `__subclasscheck__` 来检查是否存在适用的特殊方法；例如：

```python
class Sized(metaclass=ABCMeta):
    @abstractmethod
    def __hash__(self):
        return 0
    @classmethod
    def __instancecheck__(cls, x):
        return hasattr(x, "__len__")
    @classmethod
    def __subclasscheck__(cls, C):
        return hasattr(C, "__bases__") and hasattr(C, "__len__")
```

这具有不需要显式注册的优点。然而，给定实例属性与类属性的令人困惑的语义，以及类是其元类的实例，很难完全正确地处理语义；对 `__bases__` 的检查只是所需语义的近似值。**草拟方案**：让我们这样做，但让我们以这样的方式安排它，使得注册 API 也起作用。

#### 集合（Sets）

这些抽象类代表只读集合和可变集合。最基本的集合操作是成员资格测试，写成 x in s，并由 `s.__contains__(x)` 实现。此操作已由上面定义的 Container 类定义。因此，我们将集合定义为具有数学集合论中某些不变量成立的 sized、iterable container。

内置类型 set 派生自 MutableSet。内置类型 frozenset 派生自 Set 和 Hashable。

**Set** 
这是一个 sized、iterable container，即 Sized、Iterable 和 Container 的子类。并非这三个类的每个子类都是集合！集合具有额外的不变量，即每个元素只出现一次（可以通过迭代确定），此外，集合定义了具体的运算符，实现不等式操作作为子集/超集测试。通常，数学中有穷集合的不变量成立。[[11]](#id19)具有不同实现的集合可以使用有穷集合的子集/超集操作的数学定义安全地、（通常）高效地、正确地比较。

排序操作有具体实现；子类可以覆盖它们以提高速度，但应保持语义。因为 Set 派生自 Sized，`__eq__` 可以采取捷径，如果比较两个长度不等的集合，则立即返回 False。同样，`__le__` 如果第一个集合的成员比第二个集合多，可以立即返回 False。注意集合包含仅实现偏序；例如，{1, 2} 和 {1, 3} 没有序（所有三个 <、== 和 > 对于这些参数都返回 False）。

集合不能相对于映射或序列排序，但它们可以与那些进行比较相等（然后它们总是比较不相等）。

该类还定义了具体的运算符来计算并集、交集、对称和非对称差，分别为 `__or__`、`__and__`、`__xor__` 和 `__sub__`。这些运算符应返回 Set 实例。默认实现调用可覆盖的类方法 _from_iterable()，带有可迭代参数。此工厂方法的默认实现返回一个 frozenset 实例；它可以被覆盖以返回另一个适当的 Set 子类。

最后，该类定义了一个具体方法 _hash，它从元素计算哈希值。Hashable 子类的 Set 可以通过调用 _hash 来实现 `__hash__`，或者它们可以更高效地重新实现相同的算法；但实现的算法应该是相同的。目前，算法仅由源代码 [15] 完全指定。

**注意**：Python 2 中 set 类型上找到的 issubset 和 issuperset 方法不受支持，因为这些大部分只是 `__le__` 和 `__ge__` 的别名。

**MutableSet** 这是 Set 的子类，实现了额外的操作来添加和删除元素。支持的方法具有来自 Python 2 中 set 类型的语义（除了 discard，它是模仿 Java 的）：

- **.add(x)** 抽象方法返回一个 bool，如果元素 x 不在集合中则添加它。如果 x 被添加，它应该返回 True，如果它已经在那里，则返回 False。抽象实现引发 NotImplementedError。
- **.discard(x)** 抽象方法返回一个 bool，如果存在则删除元素 x。如果元素存在，它应该返回 True，如果不存在则返回 False。抽象实现引发 NotImplementedError。
- **.pop()** 具体方法删除并返回任意项。如果集合为空，它引发 KeyError。默认实现删除集合迭代器返回的第一个项。
- **.toggle(x)** 具体方法返回一个 bool，如果 x 不在集合中则添加到集合，如果在那里则删除它。如果 x 被添加，它应该返回 True，如果被删除则返回 False。
- **.clear()** 具体方法清空集合。默认实现重复调用 self.pop() 直到捕获 KeyError。（**注意**：这可能比简单地创建新集合慢得多，即使实现用更快的方法覆盖它；但在某些情况下，对象标识很重要。）

这也支持原位变异操作 |=、&=、^=、-=。这些是具体方法，其右操作数可以是任意 Iterable，除了 &=，其右操作数必须是 Container。此 ABC 不提供在内置具体 set 类型上存在的执行（几乎）相同操作的方法。

#### 映射（Mappings）

这些抽象类代表只读映射和可变映射。Mapping 类代表最常见的只读映射 API。

内置类型 dict 派生自 MutableMapping。

**Mapping** 
Container、Iterable 和 Sized 的子类。映射的键自然地形成一个集合。（键，值）对（必须是元组）也被称为项（items）。项也形成一个集合。方法：

- **`.__getitem__(key)`** 抽象方法返回对应于 key 的值，或引发 KeyError。实现总是引发 KeyError。
- **`.get(key, default=None)`** 具体方法，如果 self[key] 不引发 KeyError，则返回 self[key]，否则返回默认值。
- **`.__contains__(key)`** 具体方法，如果 self[key] 不引发 KeyError，则返回 True，否则返回 False。
- **`.__len__()`** 抽象方法返回不同键的数量（即键集的长度）。
- **`.__iter__()`** 抽象方法返回键集中的每个键恰好一次。
- **`.keys()`** 具体方法返回键集作为 Set。默认具体实现返回键集上的"视图"（意味着如果底层映射被修改，视图的值相应地改变）；子类不要求返回视图，但它们应返回 Set。
- **`.items()`** 具体方法返回项作为 Set。默认具体实现返回项集上的"视图"；子类不要求返回视图，但它们应返回 Set。
- **`.values()`** 具体方法返回值作为 sized、iterable container（不是集合！）。默认具体实现返回映射值上的"视图"；子类不要求返回视图，但它们应返回 sized、iterable container。

以下不变量应对任何映射 m 成立：

```python
len(m.values()) == len(m.keys()) == len(m.items()) == len(m)
[value for value in m.values()] == [m[key] for key in m.keys()]
[item for item in m.items()] == [(key, m[key]) for key in m.keys()]
```

即，迭代项、键和值应以相同的顺序返回结果。

**MutableMapping** 
Mapping 的子类，还实现了一些标准变异方法。抽象方法包括 `__setitem__`、`__delitem__`。具体方法包括 pop、popitem、clear、update。**注意**：setdefault 不包括在内。**开放问题**：写出方法的规范。

#### 序列（Sequences）

这些抽象类代表只读序列和可变序列。

内置 list 和 bytes 类型派生自 MutableSequence。内置 tuple 和 str 类型派生自 Sequence 和 Hashable。

**Sequence** 
Iterable、Sized、Container 的子类。它定义了一个新的抽象方法 `__getitem__`，具有相当复杂的签名：当用整数调用时，它返回序列的元素或引发 IndexError；当用 slice 对象调用时，它返回另一个 Sequence。具体 `__iter__` 方法使用整数参数 0、1 等迭代元素，使用 `__getitem__`，直到引发 IndexError。长度应等于迭代器返回的值数。**开放问题**：其他候选方法，都可以有默认具体实现，仅依赖于 `__len__` 和带有整数参数的 `__getitem__`：`__reversed__`、`index`、`count`、`__add__`、`__mul__`。

**MutableSequence** 
Sequence 的子类，添加了一些标准变异方法。抽象变异方法：`__setitem__`（用于整数索引以及切片）、`__delitem__`（同样）、insert。具体变异方法：append、reverse、extend、pop、remove。具体变异运算符：+=、*=（这些在原位变异对象）。**注意**：这不定义 sort() —— 那只需要在真正的 list 实例上存在。

### 字符串

Python 3000 可能至少有两种内置字符串类型：字节字符串（bytes），派生自 MutableSequence，和（Unicode）字符字符串（str），派生自 Sequence 和 Hashable。

**开放问题**：定义这些的基本接口，以便替代实现和子类知道它们面临的挑战。这可能是新 PEP 或 PEPs 的主题（PEP 358 应该被征用于 bytes 类型）。

## ABC 与替代方案

在本节中，我将尝试比较和对比 ABC 与其他已提出的方法。

### ABC 与 Duck Typing

ABC 的引入意味着 Duck Typing 的终结吗？我不这么认为。Python 不会要求类在定义 `__getitem__` 方法时派生自 BasicMapping 或 Sequence，x[y] 语法也不会要求 x 是任一 ABC 的实例。只要具有 write 方法，你仍然可以将任何"file-like"对象分配给 sys.stdout。

当然，会有一些胡萝卜鼓励用户从适当的基类派生；这些从某些功能的默认实现到更好地区分映射和序列的能力。但没有棍棒。如果 hasattr(x, "`__len__`") 对你有用，太好了！ABC 旨在解决 Python 2 中根本没有良好解决方案的问题，例如区分映射和序列。

### ABC 与通用函数

ABC 与通用函数（GFs）兼容。例如，我自己的通用函数实现 [4] 使用参数类（类型）作为调度键，允许派生类覆盖基类。由于（从 Python 的角度来看）ABC 是非常普通的类，在 GF 的默认实现中使用 ABC 可能非常合适。例如，如果我有一个重载的 prettyprint 函数，完全有意义地定义集合的漂亮打印如下：

```python
@prettyprint.register(Set)
def pp_set(s):
    return "{" + ... + "}"  # 细节留作练习
```

可以为 Set 的特定子类添加实现。

我相信 ABC 也不会给 RuleDispatch（Phillip Eby 在 PEAK 中的 GF 实现 [5]）带来任何问题。

当然，GF 支持者可能会声称 GF（和具体或实现类）是你所需要的一切。但即使他们也不会否认继承的有用性；人们很容易将本 PEP 中提出的 ABC 视为可选的实现基类；没有要求所有用户定义的映射都派生自 BasicMapping。

### ABC 与接口

ABC 本质上与接口不兼容，但有相当大的重叠。现在，我将留给接口的支持者来解释为什么接口更好。我希望投入到例如定义各种"映射性"色调和命名法中的工作可以很容易地适应使用接口而不是 ABC 的提案。

"接口"在此上下文中指的是附加到类的一组提议的额外元数据元素，这些元素不是常规类层次结构的一部分，但允许某些类型的继承测试。

根据至少某些提议，这种元数据将设计为应用程序可以轻松修改，允许应用程序编写者覆盖对象的正常分类。

将可变元数据附加到类这一想法的缺点在于，类是共享状态，修改它们可能导致意图冲突。此外，可以使用通用函数更干净地完成覆盖对象分类的需求：在最简单的情况下，可以定义"类别成员资格"通用函数，在基本实现中简单地返回 False，然后为任何感兴趣的类提供返回 True 的覆盖。
