## 摘要

制作 Python 发布版本是一个令人兴奋又疯狂的过程。你听说过"赶猫"这个表达吗？想象一下，试图让那些咕噜作响的小家伙们背起鞍来，骑着它们进城，而它们的一些伙伴则牢牢地附着在你的裸背上，由新磨尖的爪子固定。至少它们很可爱，你这样提醒自己。

实际上，不，这有点夸张😉 多年来，Python 发布流程不断改进，现在在我们神奇社区的帮助下，真的并不难。本 PEP 尝试在一个地方收集制作 Python 发布所需的所有步骤。现在大多数步骤都已自动化或由自动化工具指导，因此不再需要手动按照此列表操作。

## 发布类型

你需要制作几种类型的发布。包括：

- alpha（alpha 版）

- begin beta，也称为 beta 1，也称为 new branch（新分支）

- beta 2+

- release candidate 1（发布候选版 1）

- release candidate 2+

- final（正式版）

- new branch（新分支）

- begin bugfix mode（开始 bug 修复模式）

- begin security-only mode（开始仅安全更新模式）

- end-of-life（生命周期结束）

某些发布类型实际上涉及不止一个发布分支。特别是，new branch 是发布周期中开始新特性发布周期的那个点。在当前 CPython Git 仓库的组织方式下，main 分支始终是接收新特性的目标。在下一个特性版本的发布周期的某个时间点，会进行 new branch 发布，创建一个新的独立分支用于稳定和后期维护当前进行中的特性版本（3.n.0），同时修改 main 分支以构建新版本（最终将作为 3.n+1.0 发布）。虽然 new branch 发布步骤可以在发布周期中的几个点之一发生，但当前的做法是将其安排在计划进行第一个 beta 发布的发布的功能代码截止日期。

在下面的描述中，特定于发布类型的步骤相应地标记，目前，new branch 和 final。

## 如何制作发布

以下是制作 Python 发布的步骤。某些步骤比其他步骤更模糊，因为没有什么可以自动化（例如编写 NEWS 条目）。当某个步骤通常由专家执行时，会给出该专家的角色。否则，假设该步骤由发布经理（RM）执行，即执行发布的指定人员。角色及其当前专家是：

- **RM** = Release Manager
  Savannah Ostrowski <savannah&#64;python&#46;org>（美国）
  Hugo van Kemenade <hugo&#64;python&#46;org>（芬兰）
  Thomas Wouters <thomas&#64;python&#46;org>（荷兰）
  Pablo Galindo Salgado <pablogsal&#64;python&#46;org>（英国）

- **WE** = Windows - Steve Dower <steve&#46;dower&#64;python&#46;org>

- **ME** = Mac - Ned Deily <nad&#64;python&#46;org>（美国）

> **注意**
>
> 强烈建议 RM 在发布前一天联系各位专家。因为世界是圆的，每个人生活在不同的时区，RM 必须确保在足够早的时间内创建发布标签，以便专家们可以制作二进制发布。

在所有专家更新他们的部分之前，你不应该公开发布（通过更新网站和发送公告）。在极少数 Windows 或 Mac 专家失联的情况下，你可以添加一条消息"(平台) 二进制文件将稍后提供"然后继续。

我们在下面的示例中使用以下约定。当给出版本号时，格式为 3.X.YaN，例如 Python 3.13.0 alpha 3 为 3.13.0a3，其中"a"== alpha，"b"== beta，"rc"== release candidate。

发布标签命名为 v3.X.YaN。次要版本维护分支的分支名为 3.X。

尽可能多地，发布由 run_release.py 脚本自动化和指导，该脚本在单独的仓库中可用：python/release-tools。这有助于自动化许多以下步骤，并指导你执行某些手动步骤。

- **登录 Discord 并加入 Python Core Devs 服务器**。向另一位 RM 请求邀请。你可能需要与世界各地的人们协调。这个沟通渠道是我们约定见面的地方。

- **检查是否有任何阻碍发布的严重 bug**。前往 https://github.com/python/cpython/issues 查找任何可能阻止此次发布的未解决 bug。你需要关注两个相关标签：
  - **release-blocker**：直接阻止发布。你不能在有未解决的 release blocker bug 的情况下进行任何发布。
  - **deferred-blocker**：不阻止本次发布，但会阻止未来的发布。你不能在有未解决的 deferred blocker bug 的情况下进行最终或候选版发布。

审查 release blockers，要么解决它们，要么将其降级为 deferred，要么停止发布并请求社区协助。如果你正在制作最终版或候选版发布，对任何未解决的 deferred blocker 做同样的处理。

- **检查稳定的 buildbots**。前往 https://buildbot.python.org/all/#/release_status

查看你正在制作的发布的 buildbots。忽略任何离线的（或通知社区以便它们可以被重新启动）。如果剩下的主要是绿色 buildbots，你就可以进行了。如果你有非离线的红色 buildbots，你可能想要暂停发布直到它们被修复。审查问题并运用你的判断，考虑到你正在制作 alpha、beta 还是最终版发布。

- **制作发布克隆**。在 GitHub 上的 CPython 仓库的分叉上，在其中创建一个发布分支（从现在起称为"发布克隆"）。你可以使用你用于 CPython 开发的同一个 GitHub 分叉。使用 Python 开发者指南中推荐的标准设置，你的分叉将被称为 origin，标准 CPython 仓库为 upstream。你将使用分叉上的分支进行发布工程工作，包括标记发布，并用于与其他专家共享以制作二进制文件。

对于最终版或 release candidate 2+ 发布，如果你要从自上次 rc 以来合并的所有更改中 cherry-pick 一部分更改用于下一个 rc 或 final，你应该从最近的 release candidate 标签（即 v3.8.0rc1）开始创建一个发布工程分支。然后根据需要 cherry-pick 标准发布分支中的更改到发布工程分支，然后像往常一样继续。如果你要采用自上次 rc 以来的所有更改，可以按正常方式继续。

- **确保发布克隆的当前分支是你想要从中发布的分支**（git status）。

- **运行 blurb release <version> 指定版本号**（例如 blurb release 3.4.7rc1）。这将所有最近的新闻 blurb 合并到一个用此版本号标记的单个文件中。

- **重新生成 Lib/pydoc-topics.py**。在 Doc 目录中，运行：

```bash
make pydoc-topics
cp build/pydoc-topics/topics.py ../Lib/pydoc_data/topics.py
```

提交你对 pydoc_topics.py 的更改（以及在文档中做的任何修复）。

考虑使用当前接受的标准版本运行 autoconf，以防 configure 或其他 Autoconf 生成的文件最后是用较新或较旧的版本提交的，可能包含虚假或有用的差异。目前，Autoconf 2.71 是我们的事实标准。如果有差异，提交它们。

- **确保 Doc/tools/extensions/pyspecific.py 中的 SOURCE_URI 指向 Git 仓库中的正确分支**（main 或 3.X）。对于 new branch 发布，将文件中的分支从 main 更改为你即将创建的新发布分支（3.X）。

- **通过发布脚本提升版本号**：

```bash
.../release-tools/release.py --bump 3.X.YaN
```

**提醒**：X、Y 和 N 应该是整数。
a 应该是 a、b 或 rc 之一（例如 3.4.3rc1）。
对于最终发布，省略 aN（3.4.3）。对于新版本的第一次发布，Y 应该是 0（3.6.0）。

这自动化了更新各种版本号，但你将需要手动修改几个文件。如果你的$EDITOR 环境变量设置正确，release.py 会弹出编辑器窗口显示你需要编辑的文件。

- **审查 blurb 生成的 Misc/NEWS 文件并根据需要编辑**。

- **确保所有更改已提交**。（release.py --bump 不会为你提交更改。）

- **对于主要的最终发布**，编辑 Doc/whatsnew/3.X.rst 的第一段以包含实际发布日期；例如"Python 2.5 于 2003 年 8 月 1 日发布。"对于 alpha 或 beta 发布不需要编辑此内容。

- **在此目录中执行 git status**。你不应该看到任何文件，即你不应该有任何未提交的更改。

- **为 3.X.YaN 标记发布**：

```bash
.../release-tools/release.py --tag 3.X.YaN
```

这执行带有-s 选项的 git tag 命令，以便仓库中的发布标签用你的 GPG 密钥签名。提示时，选择你用于签名发布 tarball 等的私钥。

- **对于 begin security-only mode 和 end-of-life 发布**，审查两个文件并在所有活动分支中相应地更新版本。

- **将你的提交推送到 GitHub 分叉中的远程发布分支**：

```bash
# 先做 dry run。
git push --dry-run --tags origin
# 确保你推送到你的 GitHub 分叉，
# *不是*主 python/cpython 仓库！
git push --tags origin
```

在 python/release-tools 中，进入 build-release workflow，选择"Run workflow"，并输入你刚刚创建的标签的详细信息。这将执行以下步骤：
  - 创建源 gzip 和 xz tarball。
  - 创建文档 tar 和 zip 文件。
  - 检查源 tarball 以确保完全干净的 virgin 构建通过回归测试。
  - 构建和测试 Android 二进制文件（如果是 Python 3.14 或更高版本）。

生成的工件将附加到 GitHub workflow 的摘要页面。源 tarball 可用后，下载并解压它以确保一切看起来合理，没有杂散的.pyc 文件等。

如果测试通过，那么你可以放心 tarball 没问题。如果某些测试失败，或者新解压的目录的任何内容看起来奇怪，你最好现在停下来找出问题所在。

- **通知专家们他们可以开始构建二进制文件**。

> **警告**
>
> **停止**：此时你必须获得其他专家的"绿灯"才能创建发布。在你等待期间有事情可以做，所以继续阅读直到你遇到下一个停止。

- **WE 使用 Azure Pipelines 生成和发布 Windows 文件**，位于.az ure-pipelines/windows-release/中的构建脚本，当前设置在 https://dev.azure.com/Python/cpython/_build?definitionId=21。

构建过程运行多个阶段，每个阶段的输出作为可下载的工件提供。阶段包括：
  - 编译所有变体的二进制文件（32 位、64 位、调试/发布），包括运行分析引导优化。
  - 使用 PSF 的证书对二进制文件进行代码签名。
  - 为 python.org、nuget.org、可嵌入发行版和 Windows Store 创建包。
  - 对安装程序进行基本验证。
  - 将包上传到 python.org 和 nuget.org，清除下载缓存并运行测试下载。

上传完成后，WE 从构建日志中复制生成的哈希值并通过电子邮件发送给 RM。Windows Store 包由 WE 手动上传到 https://partner.microsoft.com/dashboard/home。

- **ME 构建 Mac 安装程序包并上传到 downloads.nyc1.psf.io**，连同 GPG 签名文件。

- **scp 或 rsync build-release workflow 构建的所有文件**到你 downloads.nyc1.psf.io 上的主目录，连同任何签名、SBOM 等。

在你等待文件上传完成的同时，你可以继续完成其余任务。你也可以询问 Discord 和/或 discuss.python.org 上的人们在文件上传完成时下载它们，以便他们可以在自己的平台上测试。

- **现在你需要前往 downloads.nyc1.psf.io 并将所有文件就位在**。我们的政策是每个 Python 版本都有自己的目录，但每个目录包含该版本的所有发布。在 downloads.nyc1.psf.io 上，cd /srv/www.python.org/ftp/python/3.X.Y，如需要则创建它。确保它由 downloads 组拥有并且组可写。

- **将你上传到主目录的文件移动到发布目录**。Win/Mac 二进制文件通常由专家们自己放在那里。确保它们是世界可读的。它们还应该组可写，并由 downloads 组拥有。

- **使用 gpg --verify 确保它们完整上传**。

- **如果是最终版或 rc 发布**：将 doc zips 和 tarballs 移动到/srv/www.python.org/ftp/python/doc/3.X.Y[rcA]，如需要则创建目录，并调整.../doc 中的"current"符号链接指向该目录。不过注意，如果你正在为旧版本发布维护发布，不要更改 current 链接。

- **如果是最终版或 rc 发布（即使是维护发布）**，还要在 docs.nyc1.psf.io 上将 HTML 文档解压到/srv/docs.python.org/release/3.X.Y[rcA]。确保文件在 docs 组中并且组可写。

- **注意文档和下载都在缓存 CDN 后面**。如果你通过网站下载后更改归档，你需要像这样清除 CDN 中的旧数据：

```bash
curl -X PURGE https://www.python.org/ftp/python/3.12.0/Python-3.12.0.tar.xz
```

你总是应该清除目录列表的缓存，因为人们使用它浏览发布文件：

```bash
curl -X PURGE https://www.python.org/ftp/python/3.12.0/
```

对于特别偏执的人，对发布进行完全干净的测试。这包括从 www.python.org 下载 tarball。确保 md5 校验和匹配。然后解压 tarball，并执行干净的 make test：

```bash
make distclean
./configure
make test
```

以确保回归测试套件通过。如果没有，你某处搞砸了！

> **警告**
>
> **停止并确认**：
>
> - 你从 WE 那里获得绿灯了吗？
> - 你从 ME 那里获得绿灯了吗？

如果都是绿色，是时候将发布工程分支合并回主仓库了。

- **为了将更改推送到 GitHub，你必须暂时禁用对管理员的分支保护**。前往 Settings | Branches 页面：

https://github.com/python/cpython/settings/branches

"Edit"你正在发布的分支的设置。这将加载该分支的设置页面。
取消选中"Include administrators"框并按底部的"Save changes"按钮。

- **将你的发布克隆合并到主开发仓库**：

```bash
# 上游仓库分支的纯净副本
git clone git@github.com:python/cpython.git merge
cd merge

# Checkout 正确的分支：
# 1. 对于功能预发布直到和包括**new branch**发布，
# 即 alphas 和第一个 beta，checkout main 分支
git checkout main

# 2. 否则，对于所有其他发布，checkout 相应的发布分支。
git checkout 3.X

# 从你的克隆仓库获取新创建和签名的标签
git fetch --tags git@github.com:your-github-id/cpython.git v3.X.YaN
# 将临时发布工程分支合并回
git merge --no-squash v3.X.YaN
git commit -m 'Merge release engineering branch'
```

如果这是 new branch 发布，即第一个 beta，现在创建新发布分支：

```bash
git checkout -b 3.X
```

执行设置新发布分支所需的任何步骤，包括：
  - 在 README.rst 中，将所有从 main 的引用更改为新分支，特别是 GitHub 仓库 URL。

- **对于所有发布，使用发布脚本执行指导性的发布后步骤**：

```bash
.../release-tools/release.py --done 3.X.YaN
```

对于最终版或 release candidate 2+ 发布，你可能需要做一些合并后清理。检查顶层 README.rst 和 include/patchlevel.h 文件，确保它们现在反映持续开发的所需发布后值。patchlevel 应该是发布标签加一个+。另外，如果你从标准发布分支 cherry-pick 了更改到本次发布的发布工程分支，你现在需要手动从 Misc/NEWS.d/next 目录中移除每个 cherry-pick 的 blurb 条目，因为这些 blurb 条目现在已捕获在新发布的合并的 x.y.z.rst 文件中。否则，blurb 条目将出现在 changelog.html 文件中两次，一次在 Python next 下，另一次在 x.y.z 下。

审查并提交这些更改：

```bash
git commit -m 'Post release updates'
```

如果这是 new branch 发布（例如第一个 beta），更新 main 分支以开始开发下一个特性版本。完成后，main 分支将构建 Python X.Y+1。

首先，设置 main 为下一个发布，即 X.Y+1.a0：

```bash
git checkout main
.../release-tools/release.py --bump 3.9.0a0
```

- 编辑 README.rst 中的所有版本引用
- 编辑 Doc/tutorial/interpreter.rst（两个'[Pp]ython3x'引用，一个'Python 3.x'引用，还要使横幅中的日期一致）
- 编辑 Doc/tutorial/stdlib.rst 和 Doc/tutorial/stdlib2.rst，每个都有一个'[Pp]ython3x'引用
- 添加新的 whatsnew/3.x.rst 文件（带有顶部的注释和从上一个文件复制的顶级部分），并将其添加到 whatsnew/index.rst 的 toctree 中。但要注意，来自之前发布的初始 whatsnew/3.x.rst 提交可能不正确，因为初始的 midstream 对 blurb 的更改会传播到每次发布！帮助打破循环：如果有必要，进行以下更改：

```
-For full details, see the :source:`Misc/NEWS` file.
+For full details, see the :ref:`changelog <changelog>`.
```

- 更新 configure.ac 中的版本号并重新运行 autoconf
- 确保 Doc/tools/extensions/pyspecific.py 中的 SOURCE_URI 指向 main
- 更新 Windows 构建的版本号，其中有对 python38 的引用：

```bash
ls PC/pyconfig.h.in PCbuild/rt.bat | xargs sed -i 's/python3\(\.\?\)[0-9]\+/python3\19/g'
```

- 编辑.g ithub/ISSUE_TEMPLATE/中的 bug.yml 和 crash.yml 问题模板，将新分支添加到"versions"下拉框
- 将这些更改提交到 main 分支：

```bash
git status
git add ...
git commit -m 'Bump to 3.9.0a0'
```

在此目录中再执行一次 git status。你不应该看到任何文件，即你不应该有任何未提交的更改。

- **提交并推送到主仓库**：

```bash
# 先做 dry run。

# 对于**new branch**发布之前的功能预发布，
# 即功能 alpha 发布：
git push --dry-run --tags git@github.com:python/cpython.git main
# 如果看起来没问题，就跳下去。没有回头路了！
git push --tags git@github.com:python/cpython.git main

# 对于**new branch**发布，即第一个 beta：
git push --dry-run --tags git@github.com:python/cpython.git 3.X
git push --dry-run --tags git@github.com:python/cpython.git main
# 如果看起来没问题，就跳下去。没有回头路了！
git push --tags git@github.com:python/cpython.git 3.X
git push --tags git@github.com:python/cpython.git main

# 对于所有其他发布：
git push --dry-run --tags git@github.com:python/cpython.git 3.X
# 如果看起来没问题，就跳下去。没有回头路了！
git push --tags git@github.com:python/cpython.git 3.X
```

如果这是 new branch 发布，为新创建的分支（3.X）添加 Branch protection rule。查看上一个发布分支（3.X-1）的值并用作模板。

https://github.com/python/cpython/settings/branches

另外，将 3.x 和 needs backport to 3.X 标签添加到 GitHub 仓库。

https://github.com/python/cpython/labels

- **你现在可以重新启用对管理员的分支设置强制**。回到 Settings | Branch 页面：

https://github.com/python/cpython/settings/branches

"Edit"你正在发布的分支的设置。
重新选中"Include administrators"框并按底部的"Save changes"按钮。

现在是时候摆弄网站了。几乎这一切都没有自动化，抱歉。

要执行这些步骤，你必须有编辑网站的权限。如果没有，询问 pydotorg&#64;python&#46;org 上的某人获取适当的权限。

- **登录 https://www.python.org/admin**

- **为发布创建一个新的"release"**。当前"Releases"在"Downloads"下排序。

最简单的事情可能是从现有的 Python release"page"复制字段，边编辑边进行。

你可以使用 Markdown 或 reStructured Text 来描述你的发布。前者更简洁，后者有巧妙的集成用于引用 PEP 之类的事情。

将表单上的"Release page"字段留空。

- **"Save"发布**。

- **用可下载的文件填充发布**。你和我朋友 Georg Brandl 制作了一个可爱的工具叫做 add_to_pydotorg.py。你可以在 python/release-tools 仓库中找到它（在 release.py 旁边）。你在 downloads.nyc1.psf.io 上运行工具，如下：

```bash
AUTH_INFO=<username>:<python.org-api-key> python add_to_pydotorg.py <version>
```

这遍历<version>的正确下载目录，查找标记为<version>的文件，并用这些文件填充网站上正确"release"的"Release Files"。注意每次运行时都会清除相关版本的"Release Files"。你可以从任何目录运行它，如果文件发生变化，你可以运行任意多次。在 dl-files 的主目录中保留一份副本并保持更新。

如果发布中添加了新类型的文件，有人需要更新 add_to_pydotorg.py 以使其识别这些新文件。（在文件类型移除时更新 add_to_pydotorg.py 也是最好的。）

脚本还将签署到此时尚未用 Sigstore 签署的任何剩余文件。再次，如果发生这种情况，请在此过程中使用你的&#64;python.org 地址。更多信息：

https://www.python.org/downloads/metadata/sigstore/

- **如果 CDN 已经缓存了没有这些文件的 Downloads 页面版本**，你可以使缓存失效：

```bash
curl -X PURGE https://www.python.org/downloads/release/python-XXX/
```

- **在 discuss.python.org 上编写公告**。这是模糊的部分，因为没有什么可以自动化。你可以使用较早的公告作为模板，但要编辑内容！同时将公告发布到 Python Insider 博客。要添加新条目，前往 python/python-insider-blog。

- **更新发布 PEP**（例如 719）中的发布日期

- **更新 https://github.com/python/cpython/issues 上的标签**：
  - 将所有 deferred-blocker 问题翻回 release-blocker 用于下一个发布
  - 审查未解决的问题，因为这可能会发现潜伏的严重 bug，除了提醒人们修复他们忘记的简单问题

你可以从你的仓库克隆中删除远程发布克隆分支。

如果这是 new branch 发布，你需要确保开发基础设施的各个部分已为新分支更新。这些包括：

- 更新新分支的问题追踪器：将新版本添加到版本列表
- 更新 python-releases.toml 以反映新分支和版本
- 创建 PR 更新下载页面上的支持发布表（见 python/pythondotorg#1302）
- 确保为新分支定义了 buildbots（联系Łukasz或 Zach Ware）
- 确保各种 GitHub bot 根据需要更新以适配新分支。特别是，确保向后移植到新分支正常工作（联系 core-workflow 团队）
- 审查 main 和新发布分支的最近提交历史，识别并向后移植可能在发布工程阶段合并到 main 分支的合并，这些应该在发布分支中
- 验证 CI 对 main 和新发布分支的新 PR 是否正常工作，以及发布分支是否受到适当保护（没有直接推送等）
- 验证在线文档是否构建正确（网站上的完整构建可能需要长达 24 小时）

## 下一步？

**验证！**假装你是一个用户：从 www.python.org 下载文件，并从中制作 Python。这一步太容易被忽视，好几次我们有无用的发布文件。一次通用服务器问题导致所有文件神秘损坏；一次源 tarball 构建不正确；不止一次 SF 上的文件上传过程截断了文件；等等。

**庆祝**。喝酒。快乐。写一个像这样的 PEP。或者像 Guido 一样去度假。

你刚刚制作了 Python 发布！

## 迁移到生命周期结束

根据当前政策，发布分支通常在初次发布五年后达到生命周期结束状态。政策在 Python 开发者指南中有更详细的讨论。

达到生命周期结束时，需要执行一些任务，要么由你作为发布经理直接执行，要么确保其他人执行。其中一些任务包括：

- 可选地制作最终发布以发布任何剩余的未发布更改

- **通过创建当前 HEAD 的标签然后从 CPython 仓库删除分支来冻结发布分支的状态**。当前 HEAD 应该在分支的最终安全发布处或之后：

```bash
git fetch upstream
git tag --sign -m 'Final head of the former 3.3 branch' 3.3 upstream/3.3
git push upstream refs/tags/3.3
```

如果一切看起来不错，删除分支。这可能需要有仓库管理员权限的人协助：

```bash
git push upstream --delete 3.3 # 或从 GitHub Settings 页面执行
```

- 在 python-releases.toml 中，将分支状态设置为 end-of-life
- 更新或移除开发者指南中对分支的引用
- **从问题追踪器中退役发布**。任务包括：
  - 将此版本的问题更新到下一个支持的版本
  - 从版本列表中移除版本标签
  - 移除退役版本的 needs backport to 标签
  - 审查并处理标记为此分支的未解决问题

- 运行在线文档的最终构建以添加生命周期结束横幅

- **在通常的地方宣布分支退役**：
  - discuss.python.org
  - Python Insider 博客

享受你的退休，在出色完成工作的光芒中沐浴！

## Windows 说明

Windows 有 MSI 安装程序，各种版本的 Windows 有"特殊限制"，Windows 安装程序还打包预编译的"外来的"二进制文件（Tcl/Tk、expat 等）。

安装程序作为 Azure Pipeline 的一部分进行测试。过去，这些步骤是手动执行的。我们保留这些以供后世参考。

与上传安装程序同时，WE 从安装程序安装 Python 两次：一次到安装程序建议的默认目录，另一次到名称中嵌有空间的目录。对于每次安装，WE 从 DOS 盒子运行完整的回归套件，带和不带 -0。对于维护发布，WE 还测试升级安装是否成功。

WE 还尝试 Start -> Menu -> Python 组下创建的每个快捷方式。这样尝试 IDLE 时，你需要验证 Help -> Python Documentation 是否工作。这样尝试 pydoc 时（"Module Docs"开始菜单项），确保"Start Browser"按钮工作，并确保你可以搜索随机模块（比如"random"<眨眼>），然后确保"go to selected"按钮工作。

这里可能出错的程度令人惊讶——更令人惊讶的是，最后一刻的提交经常破坏这些事物之一。如果你是"Windows 极客"，记住你可能是常规在 Windows 上测试的唯一人，而 Windows 简直是个烂摊子。

对每个目标架构重复测试。尝试 Admin 和 plain User（不是 Power User）账户。
