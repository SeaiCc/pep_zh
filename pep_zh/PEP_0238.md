# PEP 238 – 改变除法运算符

**元数据:**

- **作者**: Moshe Zadka <moshez at zadka.site.co.il>, Guido van Rossum <guido at python.org>
- **创建日期**: 2001 年 3 月 11 日
- **Python 版本**: 2.2
- **发布历史**: 2001 年 3 月 16 日，2001 年 7 月 26 日，2001 年 7 月 27 日

## 摘要

当前的除法 (/) 运算符对于数值参数具有歧义的含义：如果参数是 int 或 long，它返回除法数学结果的 floor，但如果参数是 float 或 complex，它返回除法结果的合理近似值。这使得期望 float 或 complex 结果的表达式在整数不是预期的但可能作为输入时容易出错。

我们提议通过为不同操作引入不同的运算符来解决这个问题：x/y 返回除法数学结果的合理近似值（"真除法"），x//y 返回 floor（"向下取整除法"）。我们称当前 x/y 的混合含义为"经典除法"。

由于严重的向后兼容性问题，更不用说在 c.l.py 上的重大争论，我们提议以下过渡措施（从 Python 2.2 开始）：

- 经典除法将保持为 Python 2.x 系列的默认值；真除法将在 Python 3.0 中成为标准。

- // 运算符可用于明确要求向下取整除法。

- future division 语句，拼写为 `from __future__ import division`，将改变 / 运算符在整个模块中意味着真除法。

- 命令行选项将启用对应用于 int 或 long 参数的经典除法的运行时警告；另一个命令行选项将使真除法成为默认值。

- 标准库将在适当时使用 future division 语句和 // 运算符，以完全避免经典除法。

## 动机

经典除法运算符使得编写应该从任意数值输入给出正确结果的数值表达式变得困难。对于所有其他运算符，可以写下公式如 `x*y**2 + z`，并且计算结果将接近数学结果（当然在数值精度的限制内），适用于任何数值输入类型（int、long、float 或 complex）。但除法存在问题：如果两个参数的表达式碰巧都是整数类型，它实现向下取整除法而不是真除法。

这个问题是动态类型语言独有的：在像 C 这样的静态类型语言中，输入（通常是函数参数）将声明为 double 或 float，当调用传递整数参数时，它在调用时转换为 double 或 float。Python 没有参数类型声明，所以整数参数很容易进入表达式。

这个问题特别有害，因为在所有其他情况下 int 是 float 的完美替代品：math.sqrt(2) 返回与 math.sqrt(2.0) 相同的值，3.14*100 和 3.14*100.0 返回相同的值，等等。因此，数值例程的作者可能只使用浮点数来测试他的代码，并相信它工作正常，而用户可能意外地传入整数输入值并得到不正确的结果。

另一种看待这个问题的方式是，经典除法使得编写多态函数变得困难，这些函数应该对 float 或 int 参数都能很好地工作；所有其他运算符已经做了正确的事情。任何对 int 和 float 都有效的算法都不需要在一种情况下使用截断除法而在另一种情况下使用真除法。

正确的工作方式是微妙的：将参数转换为 float() 是错误的，如果它可能是复数；向参数添加 0.0 不会保留参数的符号，如果它是负零。没有这两种缺点的唯一解决方案是将参数（通常是第一个）乘以 1.0。这对 float 和 complex 保持值和符号不变，并将 int 和 long 转换为具有相应值的 float。

作者认为这是 Python 中的一个真实设计 bug，应该尽早修复。假设 Python 的使用将继续增长，将此 bug 留在语言中的成本最终将超过修复旧代码的成本——要修复的代码量有一个上限，但未来可能受此 bug 影响的代码量是无界的。

进行此更改的另一个原因是最终统一 Python 的数值模型。这是 PEP 228 的主题（目前尚不完整）。统一的数值模型消除了用户需要意识到不同数值类型的大部分需求。这对初学者很好，但也消除了高级程序员对不同数值行为的担忧。（当然，它不会消除对数值稳定性和准确性的担忧。）

在统一的数值模型中，不同的类型（int、long、float、complex，可能还有其他类型，如新的 rational 类型）主要用作存储优化，并在一定程度上指示正交属性，如不精确性或复杂性。在统一模型中，整数 1 应该与浮点数 1.0 无法区分（除了它的不精确性），并且两者应该在所有数值上下文中表现相同。显然，在统一的数值模型中，如果 `a==b` 且 `c==d`，则 a/c 应该等于 b/d（由于舍入对不精确数字采取一些自由），并且由于每个人都同意 1.0/2.0 等于 0.5，1/2 也应该等于 0.5。同样，由于 1//2 等于零，1.0//2.0 也应该等于零。

## 变体

从美学上讲，x//y 并不让每个人都满意，因此提出了几种变体。它们在这里被解决：

- x div y。这将引入一个新关键字。由于 div 是一个流行的标识符，这将破坏大量现有代码，除非新关键字仅在 future division 语句下被识别。由于预计大多数需要转换的代码都在分割整数，这将大大增加对 future division 语句的需求。即使有 future 语句，一般反对添加新关键字除非绝对必要的 sentiment 也反对这样做。

- div(x, y)。这使得旧代码的转换更加困难。用简单查询替换将 x/y 替换为 x//y 或 x div y；在大多数情况下，程序员可以容易地验证特定模块只与整数工作，因此所有 x/y 的出现都可以被替换。（仍然需要查询替换来剔除出现在注释或字符串字面量中的斜杠。）用 div(x, y) 替换 x/y 将需要更智能的工具，因为在决定 div(和) 部分的放置之前，必须分析/左侧和右侧的表达式范围。

- x \ y。反斜杠已经是一个令牌，意味着行续行，并且一般来说它对 Unix 眼睛暗示转义。此外（这归功于 Terry Reedy），这使得像 eval("x\y") 这样的事情更难正确。

## 替代方案

为了减少需要转换的旧代码量，提出了几种替代方案。这里是对每个提案（或提案类别）的简要讨论。如果您知道在 c.l.py 上讨论的这里没有提到的替代方案，请邮件给第二作者。

- 让 / 保持其经典语义；为真除法引入 //。这仍然在语言中留下了一个损坏的运算符，并邀请使用损坏的行为。它还关闭了通往 PEP 228 式统一数值模型的道路。

- 让 int 除法返回一个特殊的"混合"类型，在整数上下文中表现为整数，但在浮点上下文中像浮点一样。这个问题的困难在于，经过几次操作后，int 和 float 值可能相距甚远，不清楚在比较中应该使用哪个值，当然许多上下文（如转换为字符串）没有明确的整数或浮点偏好。

- 使用指令在模块中使用特定除法语义，而不是 future 语句。这保留了经典除法作为语言中的永久性缺点，要求未来几代 Python 程序员意识到问题和补救措施。

- 使用 `from __past__ import division` 在模块中使用经典除法语义。这也保留了经典除法作为永久性缺点，或者至少很长时间（最终 past division 语句可能会引发 ImportError）。

- 使用指令（或其他方式）指定特定代码片段的 Python 版本。这需要未来的 Python 解释器能够精确模拟几个以前的 Python 版本，而且在同一解释器内对多个版本这样做。这工作量太大了。一个更简单的解决方案是保持安装多个解释器。另一个反对理由是版本指令几乎总是过度指定：大多数为 Python X.Y 编写的代码也适用于 Python X.(Y-1) 和 X.(Y+1)，因此指定 X.Y 作为版本比需要的更严格。同时，没有办法知道代码将在哪个未来或过去版本中断。

## API 更改

在过渡阶段，我们必须在同一程序中支持三个除法运算符：经典除法（用于没有 future division 语句的模块中的 /），真除法（用于有 future division 语句的模块中的 /），以及向下取整除法（用于 //）。每个运算符有两种形式：常规和作为增强赋值运算符（/= 或 //=）。

与这些变体相关的名称是：

- 重载运算符方法：`__div__()`、`__floordiv__()`、`__truediv__()`；`__idiv__()`、`__ifloordiv__()`、`__itruediv__()`。

- 抽象 API C 函数：PyNumber_Divide()、PyNumber_FloorDivide()、PyNumber_TrueDivide()；PyNumber_InPlaceDivide()、PyNumber_InPlaceFloorDivide()、PyNumber_InPlaceTrueDivide()。

- 字节码操作码：BINARY_DIVIDE、BINARY_FLOOR_DIVIDE、BINARY_TRUE_DIVIDE；INPLACE_DIVIDE、INPLACE_FLOOR_DIVIDE、INPLACE_TRUE_DIVIDE。

- PyNumberMethod 槽：nb_divide、nb_floor_divide、nb_true_divide、nb_inplace_divide、nb_inplace_floor_divide、nb_inplace_true_divide。

添加的 PyNumberMethod 槽需要 tp_flags 中的额外标志；该标志将命名为 Py_TPFLAGS_HAVE_NEWDIVIDE，并将包含在 Py_TPFLAGS_DEFAULT 中。

真除法和向下取整除法 API 将查找相应的槽并调用它；当该槽为 NULL 时，它们将引发异常。没有回退到经典除法槽。

在 Python 3.0 中，经典除法语义将被移除；经典除法 API 将成为真除法的同义词。

## 命令行选项

-Q 命令行选项接受一个可以取四个值的字符串参数：old、warn、warnall 或 new。默认值是 old 在 Python 2.2 中，但在后来的 2.x 版本中将改为 warn。old 值意味着经典除法运算符如所述那样作用。warn 值意味着经典除法运算符在应用于 int 或 long 时发出警告（使用标准警告框架的 DeprecationWarning）。warnall 值也对应用于 float 或 complex 的经典除法发出警告；这是由下面提到的 fixdiv.py 转换脚本使用的。new 值全局改变默认值，使得 / 运算符始终被解释为真除法。new 选项仅 intended 用于某些教育环境，其中需要真除法，但要求学生将所有代码中的 future division 语句包含在内将是一个问题。

此选项在 Python 3.0 中将不受支持；Python 3.0 将始终将 / 解释为真除法。

（此选项最初提议为 -D，但结果是 Jython 的现有选项，因此是 Q——Quotient 的助记符。其他名称已被提议，如 -Qclassic、-Qclassic-warn、-Qtrue 或 -Qold_division 等；这些对我来说似乎更冗长而没有太多优势。毕竟术语经典除法在语言中完全未使用（仅在 PEP 中），术语真除法在语言中很少使用——仅在 `__truediv__` 中。）

## 向下取整除法的语义

向下取整除法将在所有 Python 数值类型中实现，并具有以下语义：

```python
a // b == floor(a/b)
```

除了结果类型将是 a 和 b 在操作之前强制的公共类型。

具体地说，如果 a 和 b 是相同类型，a//b 也将是该类型。如果输入是不同类型，它们首先使用与其他算术运算符相同的规则强制为公共类型。

特别是，如果 a 和 b 都是 int 或 long，结果具有与这些类型的经典除法相同的类型和值（包括混合输入类型的情况；int//long 和 long//int 都将返回 long）。

对于浮点输入，结果是 float。例如：

```python
3.5//2.0 == 1.0
```

对于复数，// 引发异常，因为复数的 floor() 是不允许的。

对于用户定义的类和扩展类型，所有语义都取决于类或类型的实现。

## 真除法的语义

对于 int 和 long 的真除法将参数转换为 float，然后应用浮点除法。也就是说，即使 2/1 也返回 float（2.0），而不是 int。对于 float 和 complex，它与经典除法相同。

2.2 版本的真除法实现就像 float 类型具有无界范围，因此除非数学结果的大小大到无法表示为 float，否则不会发生溢出。例如，在 x = 1L << 40000 之后，float(x) 引发 OverflowError（注意这也是 2.2 中的新内容：以前结果是平台依赖的，最常见的是 float 无穷大）。但 x/x 返回 1.0 没有异常，而 x/1 引发 OverflowError。

注意对于 int 和 long 参数，真除法可能会丢失信息；这是真除法的性质（只要 rational 不在语言中）。有意识使用 long 的算法应考虑使用 //，因为 long 的真除法保留不超过 53 位精度（在大多数平台上）。

如果和当 rational 类型被添加到 Python（见 PEP 239），int 和 long 的真除法应该返回 rational。这避免了 int 和 long 的真除法丢失信息的问题。但在那之前，为了保持一致性，float 是真除法的唯一选择。

## Future Division 语句

如果 `from __future__ import division` 存在于模块中，或者使用了 -Qnew，/ 和 /= 运算符被转换为真除法操作码；否则它们被转换为经典除法（直到 Python 3.0 到来，在那里它们始终被转换为真除法）。

future division 语句对 // 和 //=? 的识别或转换没有影响。

见 PEP 236 了解 future 语句的一般规则。

（已经提议使用更长的短语，如 true_division 或 modern_division。这些似乎没有添加太多信息。）

## 未解决问题

我们预计这些问题将随着时间的推移而解决，因为收到更多反馈或我们获得对初始实施的更多经验。

已经提议将 // 称为商运算符，/ 运算符称为比率运算符。我不确定这个——对于一些人来说商只是除法的同义词，比率暗示有理数，这是错误的。我宁愿术语稍微笨拙，如果那避免了歧义。另外，对于一些人来说商暗示向零截断，而不是像向下取整除法明确说的那样向负无穷。

- 已经争论说改变默认值的命令行选项是邪恶的。它在错误的手中肯定可能是危险的：例如，不可能组合一个需要 -Qnew 的第三方库包和另一个需要 -Qold 的包。但我相信 VPython 的人们需要一种方法来默认启用真除法，其他教育者可能需要同样的。这些通常对其环境中的可用库包有足够的控制。

- 类必须支持所有三个 `__div__()`、`__floordiv__()` 和 `__truediv__()` 看起来很痛苦；在 3.0 中怎么办？也许我们只需要 `__div__()` 和 `__floordiv__()`，或者至少真除法应该先尝试 `__truediv__()`，然后是 `__div__()`。

## 已解决问题

- **问题**：对于非常大的 long 整数，真除法定义为返回 float 导致问题，因为 Python long 的范围比 Python float 大得多。如果和当支持有理数时，这个问题将消失。

**解决方案**：对于 long 真除法，Python 使用具有本机双精度但无界范围的内部 float 类型，因此除非商太大无法表示为本机 double，否则不会发生 OverflowError。

- **问题**：在过渡期间，也许 long-to-float 转换可以在 long 超出范围时引发 OverflowError。

**解决方案**：这已经实现，但是，如上所述，long 真除法的输入大小无关紧要；只有商的大小才重要。

- **问题**：Tim Peters 将确保每当返回范围内的 float 时，保证不错的精度。

**解决方案**：只要 long 真除法的商可以表示为 float，它就不会遭受超过 3 个舍入误差：一个用于将输入转换为具有本机双精度但无界范围的内部 float 类型，一个用于除法。但是，注意如果商的大小太小无法表示为本机 double，则返回 0.0 没有异常（"静默下溢"）。

## FAQ

### Python 3.0 何时发布？

> 我们没有计划那么远，所以我们不能肯定地说。我们希望允许至少两年的过渡期。如果 Python 3.0 更早发布，我们将保持 2.x 版本活着以保持向后兼容性，直到至少从 Python 2.2 发布两年后。实际上，在 Python 3.0 发布后，您将能够继续使用 Python 2.x 版本几年，因此您可以从容地进行过渡。站点预计将同时安装 Python 2.x 和 Python 3.x。

### 为什么真除法不叫浮点除法？

> 因为我想保持开放 *可能* 引入有理数并让 1/2 返回有理数而不是 float 的大门。见 [PEP 239](../pep-0239/)。

### 为什么需要 `__truediv__` 和 `__itruediv__`?

> 我们不希望使用户定义的类成为二等公民。肯定不是在类型/类统一进行的情况下。

### 如何编写在经典规则和新规则下都工作的代码而不使用 // 或 future division 语句？

> 使用 x*1.0/y 进行真除法，divmod(x, y) ([PEP 228](../pep-0228/)) 进行 int 除法。尤其是后者最好隐藏在函数内。如果您确定不期望复数，也可以为真除法编写 float(x)/y。如果您知道整数从不为负，可以使用 int(x/y)——虽然 int() 的文档说 int() 可以根据 C 实现舍入或截断，但我们知道没有 C 实现不截断，我们将改变 int() 的规范来承诺截断。注意经典除法（和向下取整除法）向负无穷舍入，而 int() 向零舍入，对负数给出不同的答案。

### 如何为 input()、compile()、execfile()、eval() 和 exec 指定除法语义？

> 它们从调用模块继承选择。[PEP 236](../pep-0236/) 现在将此列为已解决的问题，参考 [PEP 264](../pep-0264/)。

### codeop 模块编译的代码怎么办？

> 这已适当处理；见 [PEP 264](../pep-0264/)。

### 会有转换工具或辅助吗？

> 当然。虽然这些超出了 PEP 的范围，我应该指出两个将随 Python 2.2a3 发布的简单工具：Tools/scripts/finddiv.py 找到除法运算符（比 grep / 稍微聪明一点），Tools/scripts/fixdiv.py 可以根据运行时分析生成补丁。

### 为什么我的问题没有在这里回答？

> 因为我们没有意识到它。如果它已经在 c.l.py 上讨论过，您认为答案具有普遍兴趣，请通知第二作者。（我们没有时间或倾向回答每封私人邮件中发送的问题，因此要求它首先在 c.l.py 上讨论。）
