## 摘要

本提案建议在表达式中使用 `NAME := expr` 这种语法来给变量赋值。

作为此变更的一部分，还更新了字典推导式的求值顺序，确保键表达式在值表达式之前执行（允许将键绑定到名称，然后在计算对应值时重用）。

在讨论本 PEP 期间，该运算符非正式地被称为"海象运算符"。该结构的正式名称是"赋值表达式"（如 PEP 标题所示），但也可能被称为"命名表达式"（例如，CPython 参考实现在内部使用该名称）。

## 基本原理

为表达式的结果命名是编程的重要组成部分，允许使用描述性名称代替较长的表达式，并允许重用。目前，此功能仅以语句形式提供，在列表推导式和其他表达式上下文中不可用。

此外，为大表达式的子部分命名可以帮助交互式调试器，提供有用的显示钩子和部分结果。如果没有办法内联捕获子表达式，这将需要重构原始代码；而有了赋值表达式，只需插入几个 `name :=` 标记即可。消除重构的需要降低了代码在调试过程中被无意更改的可能性（海森堡缺陷的常见原因），也更容易向另一位程序员口述。

### 真实代码的重要性

在开发本 PEP 期间，许多人（支持者和批评者都有）往往一方面关注简单的示例，另一方面关注过于复杂的示例。

简单示例的危险是双重的：它们往往太抽象，无法让人产生"哦，这很有吸引力"的感觉，而且容易被"我无论如何都不会那样写"所反驳。

过于复杂的示例的危险在于，它们为提案的批评者提供了方便的稻草人（"那很晦涩"）。

然而，极其简单和极其复杂的示例都有一定的用途：它们有助于阐明预期的语义。
因此，下面会有每种示例。

然而，为了具有说服力，示例应该植根于真实代码，即作为有用应用程序的一部分编写的代码，无论大小，编写时都没有考虑本 PEP。Tim Peters 提供了极大的帮助，他查阅了自己的个人代码库，挑选了他认为如果（谨慎地）使用赋值表达式重写会更清晰的代码示例。他的结论：当前的提案确实可以在相当多的代码片段中带来适度但明显的改进。

真实代码的另一个用途是间接观察程序员对紧凑性有多重视。Guido van Rossum 搜索了一个 Dropbox 代码库，发现了一些证据表明程序员更重视编写更少的行数而不是更短的行。

一个例子是：Guido 发现了一些程序员重复子表达式的例子，从而减慢了程序速度，只为了节省一行代码，例如，他们不写：

```python
match = re.match(data)
group = match.group(1) if match else None
```

而是写：

```python
group = re.match(data).group(1) if re.match(data) else None
```

另一个例子说明，程序员有时会做更多的工作来节省额外的缩进层级：

```python
match1 = pattern1.match(data)
match2 = pattern2.match(data)
if match1:
    result = match1.group(1)
elif match2:
    result = match2.group(2)
else:
    result = None
```

这段代码即使 pattern1 有匹配也会尝试匹配 pattern2（在这种情况下，pattern2 的匹配从未被使用）。更高效的改写应该是：

```python
match1 = pattern1.match(data)
if match1:
    result = match1.group(1)
else:
    match2 = pattern2.match(data)
    if match2:
        result = match2.group(2)
    else:
        result = None
```

## 语法和语义

在可以使用任意 Python 表达式的大多数上下文中，命名表达式可以出现。它的形式是 `NAME := expr`，其中 `expr` 是任何有效的 Python 表达式（未加括号的元组除外），`NAME` 是标识符。

这种命名表达式的值与所包含的表达式相同，附加的副作用是目标被分配了该值：

```python
# 处理匹配的正则表达式
if (match := pattern.search(data)) is not None:
    # 对 match 做一些事情

# 一个无法简单使用 2 参数 iter() 重写的循环
while chunk := file.read(8192):
    process(chunk)

# 重用计算昂贵的值
[y := f(x), y**2, y**3]

# 在推导式过滤条件及其输出之间共享子表达式
filtered_data = [y for x in data if (y := f(x)) is not None]
```

### 特殊情况

有几个地方不允许赋值表达式，以避免歧义或用户混淆：

- 在表达式语句的顶层禁止未加括号的赋值表达式。示例：
  ```python
  y := f(x)  # 无效
  (y := f(x))  # 有效，但不推荐
  ```

  这条规则是为了简化用户在赋值语句和赋值表达式之间的选择——没有语法位置两者都有效。

- 在赋值语句右侧的顶层禁止未加括号的赋值表达式。示例：
  ```python
  y0 = y1 := f(x)  # 无效
  y0 = (y1 := f(x))  # 有效，但不鼓励
  ```

  同样，这条规则是为了避免两种视觉上相似的方式来表达同一件事。

- 在调用中关键字参数的值上禁止未加括号的赋值表达式。示例：
  ```python
  foo(x = y := f(x))  # 无效
  foo(x=(y := f(x)))  # 有效，但可能令人困惑
  ```

  这条规则是为了禁止过于令人困惑的代码，也因为解析关键字参数已经足够复杂了。

- 在函数默认值的顶层禁止未加括号的赋值表达式。示例：
  ```python
  def foo(answer = p := 42):  # 无效
      ...
  def foo(answer=(p := 42)):  # 有效，但不是很好的风格
      ...
  ```

  这条规则是为了不鼓励在某个位置产生副作用，该位置的精确语义对许多用户来说已经令人困惑（参见反对可变默认值的常见风格建议），也是为了呼应调用中的类似禁令（上一个要点）。

- 在参数、返回值和赋值的注解中禁止未加括号的赋值表达式。示例：
  ```python
  def foo(answer: p := 42 = 5):  # 无效
      ...
  def foo(answer: (p := 42) = 5):  # 有效，但可能毫无用处
      ...
  ```

  这里的原因与前两种情况类似；这种由 `:` 和 `=` 组成的未分组符号和操作符组合很难正确阅读。

- 在 lambda 函数中禁止未加括号的赋值表达式。示例：
  ```python
  (lambda: x := 1)  # 无效
  lambda: (x := 1)  # 有效，但不太可能有用
  (x := lambda: 1)  # 有效
  lambda line: (m := re.match(pattern, line)) and m.group(1)  # 有效
  ```

  这允许 lambda 始终比 `:=` 绑定得更松散；在 lambda 函数顶层进行名称绑定不太可能有用，因为没有办法利用它。在名称将被使用多次的情况下，表达式无论如何都需要加括号，因此此禁令很少会影响代码。

- f-string 内部的赋值表达式需要括号。示例：
  ```python
  >>> f'{(x:=10)}'  # 有效，使用赋值表达式
  '10'
  >>> x = 10
  >>> f'{x:=10}'  # 有效，将 '=10' 传递给格式化器
  ' 10'
  ```

  这表明 f-string 中看起来像赋值运算符的东西并不总是赋值运算符。f-string 解析器使用 `:` 来表示格式化选项。为了保持向后兼容性，f-string 内部的赋值运算符用法必须加括号。如上所述，不推荐使用赋值运算符的这种用法。

### 目标的作用域

赋值表达式不引入新的作用域。在大多数情况下，目标将被绑定的作用域是不言自明的：它是当前作用域。如果此作用域包含目标的 `nonlocal` 或 `global` 声明，赋值表达式会遵守该声明。lambda（作为一个显式的、尽管匿名的函数定义）对此目的算作一个作用域。

有一个特殊情况：发生在列表、集合或字典推导式或生成器表达式（下面统称为"推导式"）中的赋值表达式在包含作用域中绑定目标，遵守该作用域中目标的 `nonlocal` 或 `global` 声明（如果存在）。就本规则而言，嵌套推导式的包含作用域是包含最外层推导式的作用域。lambda 算作一个包含作用域。

此特殊情况的双重动机是。首先，它允许我们方便地捕获 `any()` 表达式的"见证"或 `all()` 的反例，例如：

```python
if any((comment := line).startswith('#') for line in lines):
    print("第一个注释:", comment)
else:
    print("没有注释")

if all((nonblank := line).strip() == '' for line in lines):
    print("所有行都是空白的")
else:
    print("第一行非空白行:", nonblank)
```

其次，它提供了一种从推导式更新可变状态的紧凑方法，例如：

```python
# 在列表推导式中计算部分和
total = 0
partial_sums = [total := total + v for v in values]
print("总计:", total)
```

然而，赋值表达式目标名称不能与包含赋值表达式的任何推导式中出现的 `for` 目标名称相同。后者的名称对于它们出现的推导式是局部的，因此对于相同名称的包含使用引用包含最外层推导式的作用域来说是矛盾的。

例如，`[i := i+1 for i in range(5)]` 是无效的：`for i` 部分建立 `i` 对于推导式是局部的，但 `i :=` 部分坚持 `i` 对于推导式不是局部的。
同样的原因使这些示例也无效：

```python
[[(j := j) for i in range(5)] for j in range(5)]  # 无效
[i := 0 for i, j in stuff]  # 无效
[i+1 for i in (i := stuff)]  # 无效
```

虽然在技术上可以为这些情况分配一致的语义，但在缺乏真实用例的情况下，很难确定这些语义是否实际上有意义。因此，参考实现 [1] 将确保此类情况引发 `SyntaxError`，而不是以实现定义的行为执行。

即使赋值表达式从未被执行，此限制也适用：

```python
[False and (i := 0) for i, j in stuff]  # 无效
[i for i, j in stuff if True or (j := 1)]  # 无效
```

对于推导式主体（第一个 `for` 关键字之前的部分）和过滤表达式（`if` 之后和任何嵌套 `for` 之前的部分），此限制仅适用于也用作推导式迭代变量的目标名称。出现在这些位置的 lambda 表达式引入一个新的显式函数作用域，因此可以使用赋值表达式而无需额外的限制。

由于参考实现中的设计约束（符号表分析器无法轻易检测名称在最左侧推导式可迭代表达式和推导式其余部分之间是否被重用），命名表达式完全不允许作为推导式可迭代表达式的一部分（每个 `in` 之后，以及任何后续 `if` 或 `for` 关键字之前的部分）：

```python
[i+1 for i in (j := stuff)]  # 无效
[i+1 for i in range(2) for j in (k := stuff)]  # 无效
[i+1 for i in [j for j in (k := stuff)]]  # 无效
[i+1 for i in (lambda: (j := stuff))()]  # 无效
```

当赋值表达式出现在其包含作用域是类作用域的推导式中时，还有进一步的例外。如果上述规则导致目标被分配给该类的作用域，则赋值表达式明确无效。此情况也引发 `SyntaxError`：

```python
class Example:
    [(j := i) for i in range(5)]  # 无效
```

（后者的例外是因为推导式创建的隐式函数作用域——目前没有任何运行时机制可以让函数引用包含类作用域中的变量，我们也不想添加这样的机制。如果这个问题将来得到解决，此特殊情况可能会从赋值表达式的规范中删除。注意，从推导式中使用在类作用域中定义的变量已经存在这个问题。）

参见附录 B，了解推导式中目标规则如何转换为等效代码的一些示例。

### `:=` 的相对优先级

`:=` 运算符在所有合法的语法位置上比逗号分组更紧密，但比其他所有运算符分组更松散，包括 `or`、`and`、`not` 和条件表达式（`A if C else B`）。如上文"特殊情况"一节所述，它从不允许与 `=` 处于同一层级。如果需要不同的分组，应使用括号。

`:=` 运算符可直接用于位置函数调用参数；然而，它不能直接用于关键字参数。

一些示例以澄清技术上有效或无效的内容：

```python
# 无效
x := 0

# 有效的替代方案
(x := 0)

# 无效
x = y := 0

# 有效的替代方案
x = (y := 0)

# 有效
len(lines := f.readlines())

# 有效
foo(x := 3, cat='vector')

# 无效
foo(cat=category := 'vector')

# 有效的替代方案
foo(cat=(category := 'vector'))
```

上述大多数"有效"示例都不推荐，因为快速浏览代码的 Python 源代码人类读者可能会错过区别。但简单的情况是可以接受的：

```python
# 有效
if any(len(longline := line) >= 100 for line in lines):
    print("极长的行:", longline)
```

本 PEP 建议在 `:=` 周围始终放置空格，类似于 PEP 8 对用于赋值的 `=` 的建议，而后者禁止在用于关键字参数的 `=` 周围放置空格。）

### 求值顺序的变更

为了具有精确定义的语义，本提案要求求值顺序定义良好。这 technically 不是新要求，因为函数调用可能已经有副作用。Python 已经有一条规则，即子表达式通常从左到右求值。然而，赋值表达式使这些副作用更加可见，我们提议对当前求值顺序进行单一变更：

- 在字典推导式 `{X: Y for ...}` 中，`Y` 当前在 `X` 之前求值。我们提议更改此行为，使 `X` 在 `Y` 之前求值。（在像 `{X: Y}` 这样的字典表示中，这已经是这种情况，在 `dict((X, Y) for ...)` 中也是如此，后者应该明显等价于字典推导式。）

### 赋值表达式与赋值语句之间的差异

最重要的是，由于 `:=` 是一个表达式，它可以在语句非法的上下文中使用，包括 lambda 函数和推导式。

相反，赋值表达式不支持赋值语句中的高级功能：

- 不直接支持多个目标：
  ```python
  x = y = z = 0  # 等价：(z := (y := (x := 0)))
  ```

- 不支持单个 NAME 以外的单个赋值目标：
  ```python
  # 没有等价形式
  a[i] = x
  self.rest = []
  ```

- 逗号的优先级不同：
  ```python
  x = 1, 2  # 将 x 设置为 (1, 2)
  (x := 1, 2)  # 将 x 设置为 1
  ```

- 不支持可迭代打包和解包（常规或扩展形式）：
  ```python
  # 等价形式需要额外的括号
  loc = x, y  # 使用 (loc := (x, y))
  info = name, phone, *rest  # 使用 (info := (name, phone, *rest))

  # 没有等价形式
  px, py, pz = position
  name, phone, email, *other_info = contact
  ```

- 不支持内联类型注解：
  ```python
  # 最接近的等价形式是单独的 "p: Optional[int]" 声明
  p: Optional[int] = None
  ```

- 不支持增强赋值：
  ```python
  total += tax  # 等价：(total := total + tax)
  ```

## 实现期间的规范变更

基于实现经验和 PEP 首次接受后及 Python 3.8 发布前的额外审查，进行了以下更改：

- 为了与其他类似的异常保持一致，并避免锁定一个不一定会提高最终用户清晰度的异常名称，最初提议的 `TargetScopeError` `SyntaxError` 子类被放弃，改为直接引发 `SyntaxError`。[3]

- 由于 CPython 符号表分析过程的限制，参考实现对推导式可迭代表达式内部所有命名表达式的用法都引发 `SyntaxError`，而不仅仅是在命名表达式目标与推导式中的一个迭代变量冲突时引发。如果提供足够有说服力的示例，这可能会被重新审视，但实现更有选择性的限制所需的额外复杂性似乎对于纯假设的用例来说不值得。

## 示例

### 来自 Python 标准库的示例

#### site.py

`env_base` 仅在这些行中使用，将其赋值放在 `if` 上将其移动为块的"头部"。

**当前:**
```python
env_base = os.environ.get("PYTHONUSERBASE", None)
if env_base:
    return env_base
```

**改进后:**
```python
if env_base := os.environ.get("PYTHONUSERBASE", None):
    return env_base
```

#### _pydecimal.py

避免嵌套 `if` 并减少一个缩进层级。

**当前:**
```python
if self._is_special:
    ans = self._check_nans(context=context)
    if ans:
        return ans
```

**改进后:**
```python
if self._is_special and (ans := self._check_nans(context=context)):
    return ans
```

#### copy.py

代码看起来更规则，并避免多个嵌套 `if`。
（见此示例的起源，参考附录 A。）

**当前:**
```python
reductor = dispatch_table.get(cls)
if reductor:
    rv = reductor(x)
else:
    reductor = getattr(x, "__reduce_ex__", None)
    if reductor:
        rv = reductor(4)
    else:
        reductor = getattr(x, "__reduce__", None)
        if reductor:
            rv = reductor()
        else:
            raise Error(
                "un(deep)copyable object of type %s" % cls)
```

**改进后:**
```python
if reductor := dispatch_table.get(cls):
    rv = reductor(x)
elif reductor := getattr(x, "__reduce_ex__", None):
    rv = reductor(4)
elif reductor := getattr(x, "__reduce__", None):
    rv = reductor()
else:
    raise Error("un(deep)copyable object of type %s" % cls)
```

#### datetime.py

`tz` 仅用于 `s += tz`，将其赋值移到 `if` 内部有助于显示其作用域。

**当前:**
```python
s = _format_time(self._hour, self._minute,
                 self._second, self._microsecond,
                 timespec)
tz = self._tzstr()
if tz:
    s += tz
return s
```

**改进后:**
```python
s = _format_time(self._hour, self._minute,
                 self._second, self._microsecond,
                 timespec)
if tz := self._tzstr():
    s += tz
return s
```

#### sysconfig.py

在 `while` 条件中调用 `fp.readline()`，在 `if` 行上调用 `.match()` 使代码更紧凑，而不会使其更难理解。

**当前:**
```python
while True:
    line = fp.readline()
    if not line:
        break
    m = define_rx.match(line)
    if m:
        n, v = m.group(1, 2)
        try:
            v = int(v)
        except ValueError:
            pass
        vars[n] = v
    else:
        m = undef_rx.match(line)
        if m:
            vars[m.group(1)] = 0
```

**改进后:**
```python
while line := fp.readline():
    if m := define_rx.match(line):
        n, v = m.group(1, 2)
        try:
            v = int(v)
        except ValueError:
            pass
        vars[n] = v
    elif m := undef_rx.match(line):
        vars[m.group(1)] = 0
```

### 简化列表推导式

列表推导式可以通过捕获条件来有效地映射和过滤：

```python
results = [(x, y, x/y) for x in input_data if (y := f(x)) > 0]
```

同样，子表达式可以在主表达式中重用，通过首次使用时给它一个名称：

```python
stuff = [[y := f(x), x/y] for x in range(5)]
```

注意，在这两种情况下，变量 `y` 都绑定在包含作用域中（即与 `results` 或 `stuff` 同一层级）。

### 捕获条件值

赋值表达式可以有效地用于 `if` 或 `while` 语句的头部：

```python
# 循环和半次
while (command := input("> ")) != "quit":
    print("你输入了:", command)

# 捕获正则表达式匹配对象
# 参见 Lib/pydoc.py，它使用了此效果的多行拼写
if match := re.search(pat, text):
    print("找到:", match.group(0))
# 与使用赋值语句的等价形式不同，
# 相同的语法可以很好地链接到 'elif' 语句。
elif match := re.search(otherpat, text):
    print("备用找到:", match.group(0))
elif match := re.search(third, text):
    print("回退找到:", match.group(0))

# 读取 socket 数据直到返回空字符串
while data := sock.recv(8192):
    print("接收到的数据:", data)
```

特别是对于 `while` 循环，这可以消除拥有无限循环、赋值和条件的需要。它还在仅使用函数调用作为条件的循环和也使用该条件但还使用实际值的循环之间创建了平滑的并行。

### 分支

来自低级 UNIX 世界的示例：

```python
if pid := os.fork():
    # 父进程代码
else:
    # 子进程代码
```

## 被拒绝的替代提案

与此提案大致相似的提案在 python-ideas 上频繁出现。
下面是几种替代语法，其中一些特定于推导式，已被拒绝，而采用上述方案。

### 更改推导式的作用域规则

本 PEP 的先前版本提议了对推导式作用域规则的细微更改，使其在类作用域中更可用，并统一"最外层可迭代对象"和推导式其余部分的作用域。然而，提案的这部分将导致向后不兼容，因此已被撤回，以便 PEP 可以专注于赋值表达式。

### 替代拼写

大致与当前提案相同的语义，但拼写不同。

**EXPR as NAME:**
```python
stuff = [[f(x) as y, x/y] for x in range(5)]
```

由于 `EXPR as NAME` 在 `import`、`except` 和 `with` 语句中已有含义（具有不同的语义），这将造成不必要的混淆或需要特殊处理（例如，禁止在这些语句的头部内进行赋值）。

（注意 `with EXPR as VAR` 不仅仅将 `EXPR` 的值分配给 `VAR`——它调用 `EXPR.__enter__()` 并将结果分配给 `VAR`。）

选择 `:=` 而不是此拼写的其他原因包括：

- 在 `if f(x) as y` 中，赋值目标不会突出显示——它读起来就像 `if f x blah blah`，而且在视觉上太类似于 `if f(x) and y`。

- 在所有其他允许 `as` 子句的情况下，即使是中等水平的读者也会由开始行的关键字引导来预期该子句（无论是否可选），并且语法将该关键字与 `as` 子句紧密联系在一起：
  ```python
  import foo as bar
  except Exc as var
  with ctxmgr() as var
  ```

- 相反，赋值表达式不属于开始行的 `if` 或 `while`，我们故意允许赋值表达式在其他上下文中使用。

- `NAME = EXPR` 与 `if NAME := EXPR` 之间的平行节奏加强了赋值表达式的视觉识别。

**EXPR -> NAME:**
```python
stuff = [[f(x) -> y, x/y] for x in range(5)]
```

此语法受到 R 和 Haskell 等语言以及一些可编程计算器的启发。（注意，左向箭头 `y <- f(x)` 在 Python 中是不可能的，因为它会被解释为小于和一元负号。）此语法相对于 `as` 有一个轻微的优势，即它不与 `with`、`except` 和 `import` 冲突，但在其他方面是等价的。但它与 Python 对 `->` 的其他用途（函数返回类型注解）完全无关，与 `:=`（可追溯到 Algol-58）相比，它的传统要弱得多。

**用前导点装饰语句局部名称:**
```python
stuff = [[(f(x) as .y), x/.y] for x in range(5)]  # 使用 "as"
stuff = [[(.y := f(x)), x/.y] for x in range(5)]  # 使用 ":="
```

这有一个优势，即泄漏的用法可以轻易被检测到，消除了一些形式的语法歧义。然而，这将是 Python 中唯一将变量作用域编码到其名称中的地方，使得重构更加困难。

**向任何语句添加 where:**
```python
value = x**2 + 2*x where:
    x = spam(1, 4, 7, q)
```

执行顺序是反转的（缩进主体先执行，然后是"头部"）。这需要一个新的关键字，除非重新利用现有的关键字（最可能是 `with:`）。参见 PEP 3150 关于此主题的先前讨论（提议的关键字是 `given:`）。

**TARGET from EXPR:**
```python
stuff = [[y from f(x), x/y] for x in range(5)]
```

此语法比 `as` 有更少的冲突（仅与 `raise Exc from Exc` 表示法冲突），但在其他方面是可比的。它没有与 `with expr as target:` 平行（这可能有用，但也可能令人困惑），但没有平行关系，但具有启发性。

### 特殊处理条件语句

最受欢迎的用例之一是 `if` 和 `while` 语句。而不是更通用的解决方案，此提案增强了这两个语句的语法，以增加捕获比较值的手段：

```python
if re.search(pat, text) as match:
    print("找到:", match.group(0))
```

这只有在所需的条件基于捕获值的真假性时才能完美工作。因此，它对特定的用例（正则表达式匹配、返回 '' 表示完成的 socket 读取）有效，但在更复杂的情况下完全无用（例如，条件是 `f(x) < 0` 而你想要捕获 `f(x)` 的值）。它对列表推导式也没有任何好处。

**优点:** 没有语法歧义。**缺点:** 只回答了可能用例的一小部分，即使在 `if`/`while` 语句中也是如此。

### 特殊处理推导式

另一个常见的用例是推导式（列表/集合/字典和生成器表达式）。如上所述，已经提出了针对推导式的解决方案。

**where, let, 或 given:**
```python
stuff = [(y, x/y) where y = f(x) for x in range(5)]
stuff = [(y, x/y) let y = f(x) for x in range(5)]
stuff = [(y, x/y) given y = f(x) for x in range(5)]
```

这将为子表达式带来一个位于 `for` 循环和表达式之间的位置。它引入了一个额外的语言关键字，这会造成冲突。在三个中，`where` 读起来最清晰，但也有最大的冲突潜力（例如，SQLAlchemy 和 numpy 有 `where` 方法，标准库中的 tkinter.dnd.Icon 也有）。

**with NAME = EXPR:**
```python
stuff = [(y, x/y) with y = f(x) for x in range(5)]
```

如上所述，但重用 `with` 关键字。读起来不算太差，不需要额外的语言关键字。但仅限于推导式，并且不能容易地转换为"长手"for 循环语法。具有 C 的问题，即表达式中的等号现在可以创建名称绑定，而不是执行比较。会引发为什么 `with NAME = EXPR:` 不能单独用作语句的问题。

**with EXPR as NAME:**
```python
stuff = [(y, x/y) with f(x) as y for x in range(5)]
```

如选项 2 所述，但使用 `as` 而不是等号。在语法上与其他用于名称绑定的 `as` 用法对齐，但一个简单的转换为 for 循环长手形式会产生截然不同的语义；推导式内部的 `with` 的含义将完全不同于作为独立语句的含义，同时保留相同的语法。

无论选择哪种拼写，这都会在推导式和循环的等效展开长手形式之间引入明显的差异。不再可能在不重新处理任何名称绑定的情况下将循环展开为语句形式。唯一可以重新用于此任务的关键字是 `with`，从而在推导式中给它偷偷地不同的语义，而不是在语句中；或者，需要一个新的关键字，带有其中所有的成本。

### 降低运算符优先级

`:=` 运算符有两个逻辑优先级。要么它应该尽可能松散地绑定，就像语句赋值一样；或者它应该比较运算符更紧密地绑定。将其优先级放在比较和算术运算符之间（确切地说：略低于按位或）允许大多数在 `while` 和 `if` 条件内的用法在不使用括号的情况下拼写，因为你最可能想要捕获某物的值，然后对其执行比较：

```python
pos = -1
while pos := buffer.find(search_term, pos + 1) >= 0:
    ...
```

一旦 `find()` 返回 -1，循环就终止。如果 `:=` 像 `=` 一样松散地绑定，这将捕获比较的结果（通常是 `True` 或 `False`），这用处较小。

虽然这种行为在许多情况下会很方便，但它也比"`:=` 运算符的行为就像赋值语句"更难解释，因此，`:=` 的优先级已尽可能接近 `=` 的优先级（例外是它比逗号绑定得更紧）。

### 允许右侧有逗号

一些批评者声称赋值表达式应该允许右侧有未加括号的元组，以使这两个等价：

```python
(point := (x, y))
(point := x, y)
```

（根据本提案的当前版本，后者等价于 `((point := x), y)`。）

然而，采取这种立场在逻辑上将导致这样的结论，即当在函数调用中使用时，赋值表达式也比逗号绑定得更松，因此我们将有以下令人困惑的等价性：

```python
foo(x := 1, y)
foo(x := (1, y))
```

较不令人困惑的选择是使 `:=` 比逗号绑定得更紧密。

### 始终需要括号

有人建议始终在赋值表达式周围需要括号。这将解决许多歧义，而且括号经常需要提取所需的子表达式。但在以下情况下，额外的括号感觉多余：

```python
# if 中的顶层
if match := pattern.match(line):
    return match.group(1)

# 短调用
len(lines := f.readlines())
```

## 经常提出的异议

### 为什么不把现有的赋值变成表达式？

C 及其衍生语言将 `=` 运算符定义为表达式，而不是像 Python 那样的语句。这允许在更多上下文中进行赋值，包括比较更常见的上下文。`if (x == y)` 和 `if (x = y)` 之间的语法相似性掩盖了它们截然不同的语义。因此，本提案使用 `:=` 来澄清区别。

### 有了赋值表达式，为什么还要麻烦赋值语句？

这两种形式有不同的灵活性。`:=` 运算符可用于更大的表达式内部；`=` 语句可以增强为 `+=` 及其朋友，可以链接，并且可以分配给属性和下标。

### 为什么不使用子局部作用域并防止命名空间污染？

本提案的先前版本涉及子局部作用域（限制为单个语句），防止名称泄漏和命名空间污染。虽然在许多情况下这是一个确定的优势，但这在许多其他情况下增加了复杂性，而且成本不值得收益。为了语言简单性的利益，这里创建的名称绑定与任何其他名称绑定完全等价，包括在类或模块作用域使用将创建外部可见的名称。这与 for 循环或其他构造没有不同，可以以相同的方式解决：一旦不再需要就 `del` 名称，或者在其前加上下划线。

（作者希望感谢 Guido van Rossum 和 Christoph Groth 建议将提案朝这个方向移动。[2]）

## 风格指南建议

由于表达式赋值有时可以等价地用作语句赋值，因此应该优先使用哪个的问题会出现。为了 PEP 8 等风格指南的利益，提出两条建议。

- 如果赋值语句或赋值表达式都可以使用，优先使用语句；它们是意图的明确声明。

- 如果使用赋值表达式会导致执行顺序的歧义，则重构它以使用语句。

## 附录 A: Tim Peters 的发现

以下是 Tim Peters 就此话题写的一篇简短文章。

我不喜欢"繁忙"的代码行，也不喜欢将概念上不相关的逻辑放在同一行上。所以，例如，我不喜欢：

```python
i = j = count = nerrors = 0
```

我更喜欢：

```python
i = j = 0
count = 0
nerrors = 0
```

所以我怀疑我会找到很少想要使用赋值表达式的场合。我甚至不考虑那些已经延伸到屏幕一半的行。在其他情况下，"不相关"规则：

```python
mylast = mylast[1]
yield mylast[0]
```

是比简短版本巨大的改进：

```python
yield (mylast := mylast[1])[0]
```

原始两个语句在做完全不同的概念性事情，将它们 slam 在一起在概念上是疯狂的。

在其他情况下，组合相关逻辑使其更难理解，例如重写：

```python
while True:
    old = total
    total += term
    if old == total:
        return total
    term *= mx2 / (i*(i+1))
    i += 2
```

为更简短的：

```python
while total != (total := total + term):
    term *= mx2 / (i*(i+1))
    i += 2
return total
```

那里的 while 测试太微妙了，关键依赖于非短路或方法链上下文中的严格从左到右求值。我的大脑不是那样连接的。

但像这样的情况很少见。名称绑定非常频繁，"稀疏优于密集"并不意味着"几乎空优于稀疏"。例如，我有很多函数返回 `None` 或 `0` 来传达"在这种情况下我没有有用的东西要返回，但由于这经常发生，我不会用异常烦你"。这基本上与正则表达式搜索函数在没有匹配时返回 `None` 相同。所以有很多这样的代码：

```python
result = solution(xs, n)
if result:
    # 使用 result
```

我发现这更清晰，当然少一些打字和模式匹配阅读，如：

```python
if result := solution(xs, n):
    # 使用 result
```

用少量水平空白换取另一行周围代码也是不错的。我一开始没有太重视这个，但它如此频繁以至于累积起来，我很快就因为不能实际运行更简短的代码而感到恼火。这让我惊讶！

还有其他赋值表达式真正发光的情况。而不是从我的代码中再选一个，Kirill Balunov 从标准库的 copy() 函数中给了一个很好的例子：

```python
reductor = dispatch_table.get(cls)
if reductor:
    rv = reductor(x)
else:
    reductor = getattr(x, "__reduce_ex__", None)
    if reductor:
        rv = reductor(4)
    else:
        reductor = getattr(x, "__reduce__", None)
        if reductor:
            rv = reductor()
        else:
            raise Error("un(shallow)copyable object of type %s" % cls)
```

不断增加的缩进在语义上是误导性的：逻辑在概念上是平坦的，"第一个成功的测试获胜"：

```python
if reductor := dispatch_table.get(cls):
    rv = reductor(x)
elif reductor := getattr(x, "__reduce_ex__", None):
    rv = reductor(4)
elif reductor := getattr(x, "__reduce__", None):
    rv = reductor()
else:
    raise Error("un(shallow)copyable object of type %s" % cls)
```

使用简单的赋值表达式允许代码的视觉结构强调逻辑的概念平坦性；不断增加的缩进掩盖了它。

来自我的代码的一个更小的例子让我很高兴，既允许将本质上相关的逻辑放在一行，又允许删除一个烦人的"人工"缩进层级：

```python
diff = x - x_base
if diff:
    g = gcd(diff, n)
    if g > 1:
        return g
```

变成了：

```python
if (diff := x - x_base) and (g := gcd(diff, n)) > 1:
    return g
```

那个 if 大约是我想要行有多长，但仍然容易遵循。

所以，总的来说，在大多数绑定名称的行中，我不会使用赋值表达式，但因为那个构造如此频繁，这留下了许多我会使用的地方。在后者的大多数中，我发现了一个小胜利，由于它发生的频率而累积起来，在其余中，我发现了一个中度到主要的胜利。我肯定会比三元 if 更频繁地使用它，但比增强赋值显著少。

### 一个数字示例

我有另一个例子，当时让我相当印象深刻。

当所有变量都是正整数，且 `a` 至少和 `x` 的 `n` 次根一样大时，此算法返回 `x` 的 `n` 次根的 floor（并且每迭代大致将准确位数加倍）：

```python
while a > (d := x // a**(n-1)):
    a = ((n-1)*a + d) // n
return a
```

为什么它工作并不明显，但在"循环和半次"形式中也不更明显。没有建立在正确的见解（"算术平均 - 几何平均不等式"）上，并知道一些关于嵌套 floor 函数如何行为的非平凡事情，很难证明正确性。也就是说，挑战在于数学，而不在编码。

如果你知道所有这些，那么赋值表达式形式很容易读作"当当前猜测太大时，获取更小的猜测"，其中"太大？"测试和新猜测共享一个昂贵的子表达式。

在我看来，原始形式更难理解：

```python
while True:
    d = x // a**(n-1)
    if a <= d:
        break
    a = ((n-1)*a + d) // n
return a
```

## 附录 B: 推导式的粗略代码转换

本附录试图阐明（但不规范）当目标出现在推导式或生成器表达式中的规则。
对于许多说明性示例，我们显示包含推导式的原始代码，以及转换后的代码，其中推导式已被等效的生成器函数和一些脚手架替换。

由于 `[x for ...]` 等价于 `list(x for ...)`，这些示例都使用列表推导式而不失一般性。
并且由于这些示例旨在阐明规则的边缘情况，它们并不试图看起来像真实代码。

注意：推导式已经通过合成像本附录中那样的嵌套生成器函数来实现。新部分是添加适当的声明以建立赋值表达式目标的预期作用域（与如果在包含最外层推导式的块中执行赋值相同的范围）。对于类型推断目的，这些说明性扩展并不意味着赋值表达式目标总是 `Optional`（但它们确实指示目标绑定作用域）。

让我们从提醒没有赋值表达式的生成器表达式生成什么代码开始。

- **原始代码**（EXPR 通常引用 VAR）：
  ```python
  def f():
      a = [EXPR for VAR in ITERABLE]
  ```

  **转换**（让我们不要担心名称冲突）：
  ```python
  def f():
      def genexpr(iterator):
          for VAR in iterator:
              yield EXPR
      a = list(genexpr(iter(ITERABLE)))
  ```

让我们添加一个简单的赋值表达式。

- **原始代码:**
  ```python
  def f():
      a = [TARGET := EXPR for VAR in ITERABLE]
  ```

  **转换:**
  ```python
  def f():
      if False:
          TARGET = None  # 死代码以确保 TARGET 是局部变量
      def genexpr(iterator):
          nonlocal TARGET
          for VAR in iterator:
              TARGET = EXPR
              yield TARGET
      a = list(genexpr(iter(ITERABLE)))
  ```

让我们在 `f()` 中添加一个 `global TARGET` 声明。

- **原始代码:**
  ```python
  def f():
      global TARGET
      a = [TARGET := EXPR for VAR in ITERABLE]
  ```

  **转换:**
  ```python
  def f():
      global TARGET
      def genexpr(iterator):
          global TARGET
          for VAR in iterator:
              TARGET = EXPR
              yield TARGET
      a = list(genexpr(iter(ITERABLE)))
  ```

或者让我们添加一个 `f()` 中的 `nonlocal TARGET` 声明。

- **原始代码:**
  ```python
  def g():
      TARGET = ...
      def f():
          nonlocal TARGET
          a = [TARGET := EXPR for VAR in ITERABLE]
  ```

  **转换:**
  ```python
  def g():
      TARGET = ...
      def f():
          nonlocal TARGET
          def genexpr(iterator):
              nonlocal TARGET
              for VAR in iterator:
                  TARGET = EXPR
                  yield TARGET
          a = list(genexpr(iter(ITERABLE)))
  ```

最后，让我们嵌套两个推导式。

- **原始代码:**
  ```python
  def f():
      a = [[TARGET := i for i in range(3)] for j in range(2)]
      # 即，a = [[0, 1, 2], [0, 1, 2]]
      print(TARGET)  # 打印 2
  ```

  **转换:**
  ```python
  def f():
      if False:
          TARGET = None
      def outer_genexpr(outer_iterator):
          nonlocal TARGET
          def inner_generator(inner_iterator):
              nonlocal TARGET
              for i in inner_iterator:
                  TARGET = i
                  yield i
          for j in outer_iterator:
              yield list(inner_generator(range(3)))
      a = list(outer_genexpr(range(2)))
      print(TARGET)
  ```

## 附录 C: 不更改作用域语义

因为它一直是一个混淆点，请注意 Python 的作用域语义没有任何改变。函数局部作用域继续在编译时解析，并且在运行时具有无限的持续时间（"完全闭包"）。示例：

```python
a = 42
def f():
    # `a` 对 `f` 是局部的，但保持未绑定
    # 直到调用者执行此生成器表达式：
    yield ((a := i) for i in range(3))
    yield lambda: a + 100
    print("done")
    try:
        print(f"`a` 绑定到 {a}")
        assert False
    except UnboundLocalError:
        print("`a` 尚未绑定")
```

然后：

```python
>>> results = list(f())  # [genexp, lambda]
done
`a` 尚未绑定
# f 的执行帧在 CPython 中不再存在，
# 但只要仍然可以引用，f 的局部变量就存在。
>>> list(map(type, results))
[<class 'generator'>, <class 'function'>]
>>> list(results[0])
[0, 1, 2]
>>> results[1]()
102
>>> a
42
```
