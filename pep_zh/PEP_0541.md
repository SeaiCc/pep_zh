# PEP 541 – 软件包索引名称保留

**元数据：**

- **作者**：Łukasz Langa <lukasz&#32;&#97;t&#32;python.org>
- **BDFL 代表**：Mark Mangoba <mmangoba&#32;&#97;t&#32;python.org>
- **创建时间**：2017 年 1 月 12 日

## 摘要

本 PEP 提议扩展软件包索引 [2] 的使用条款 [1]，明确软件包所有者对软件包索引上软件包名称所有权的期望，特别是关于冲突解决。

现有的软件包存储库，如 CPAN [3]、NPM [4] 和 GitHub [5]，将作为此领域的现有技术进行调查。

## 理由

鉴于索引上的软件包名称共享单个扁平命名空间，唯一名称是一种有限资源。软件包索引的日益增长导致了当前名称使用与相同名称的不同建议使用之间冲突情况的持续增加。

本文档旨在为解决此类冲突的最典型情况提供一般指导方针。

## 批准流程

由于此政策的适用对 Python 软件基金会具有潜在的法律影响，因此使用的批准流程比大多数 PEP 使用的流程更正式。

BDFL 代表不会直接接受 PEP，而是建议将其接受提交给 PSF 的打包工作组。在与 PSF 总法律顾问协商后，政策的采用将取决于工作组内的正式投票。

此正式批准流程将用于政策的初步采用和任何未来修订的采用。

## 规范

本文档背后的主要思想是软件包索引服务于社区。每个用户都被邀请在使用条款下向软件包索引上传内容，理解这是由用户独自承担风险。

虽然软件包索引不是备份服务，但软件包索引的维护者尽其最大努力以发布的形式永久保留该内容可访问。但是，在某些边缘情况下，更大社区的需求可能会超过个人对软件包名称所有权的期望。

本文档涵盖的使用场景是：

- 被遗弃的项目：
  由不同的用户集继续维护；或

- 从索引中移除以用于不同的项目。

- 活跃项目：
  解决名称争议。

- 无效项目：
  涉及知识产权侵权指控的项目。

使用条款的提议扩展，如实现部分所述，将作为单独文档发布在软件包索引上，链接到主页页脚中现有使用条款旁边。

## 实现

### 可联系性

软件包索引的用户全权负责就用户拥有的项目事项可被软件包索引维护者联系。在每个需要联系用户的情况下，维护者将尝试至少三次使用以下联系方式：

- 软件包索引上用户档案中的电子邮件地址；

- 上传到索引的给定项目的作者字段中列出的电子邮件地址；以及

- 在索引上或列出的主页上的给定项目文档中找到的任何电子邮件地址。

维护者在六周后停止尝试联系用户。

### 被遗弃的项目

当满足以下所有条件时，项目被视为被遗弃：

- 所有者无法联系（见上文可联系性）；

- 过去十二个月内没有发布；以及

- 所有者在项目主页上没有活动（或未列出主页）。

所有其他项目被视为活跃。

### 继续维护被遗弃的项目

如果候选人似乎愿意继续维护被遗弃的项目，当满足以下所有条件时，名称的所有权被转移：

- 项目已被上述规则确定为被遗弃；

- 候选人能够证明自己尝试联系现有所有者的失败；

- 候选人能够证明在候选人自己的项目分叉中进行的改进；

- 候选人能够证明为什么在不同名称下的分叉不是可接受的变通方法；以及

- 软件包索引的维护者没有任何额外的保留意见。

在任何情况下，都不会违背可联系所有者的意愿重新分配名称。

### 被遗弃项目的移除

项目绝不会仅基于被遗弃而从软件包索引中移除。上传到软件包索引的工件具有固有的历史价值。

当满足以下所有条件时，可以将被遗弃的项目转移给新所有者以重用名称：

- 项目已被上述规则确定为被遗弃；

- 候选人能够证明自己尝试联系现有所有者的失败；

- 候选人能够证明建议重用名称的项目已经存在并满足知名度要求；

- 候选人能够证明为什么在不同名称下的分叉不是可接受的变通方法；

- 软件包索引上现有软件的下载统计数据显示项目未被使用；以及

- 软件包索引的维护者没有任何额外的保留意见。

### 活跃项目的名称冲突解决

软件包索引的维护者不是活跃项目争议的仲裁者。这里有许多可能的场景，下面列出一个非排他性列表，描述一些现实世界的示例。以下没有任何一个符合软件包名称所有权转移：

- 用户 A 和用户 B 共享项目 X。一段时间后，他们分道扬镳，每个人都想在名称 X 下继续项目。

- 用户 A 在软件包索引之外拥有项目 X。用户 B 在索引上创建名称为 X 的软件包。一段时间后，用户 A 想在索引上发布项目 X，但发现名称已被占用。即使用户 A 的项目 X 获得知名度且用户 B 的项目 X 不著名，这也是正确的。

- 用户 A 将项目 X 发布到软件包索引。一段时间后，用户 B 向项目提出错误修复，但用户 A 没有发布新版本。即使用户 A 同意发布新版本但后来没有发布，即使用户 B 的更改被合并到项目 X 的源代码仓库中，这也是正确的。

再次，上面的列表不是排他性的。软件包索引的维护者建议用户相互联系，并通过尊重的沟通解决问题（参见 PSF 行为准则 [6]）。

### 无效项目

在软件包索引上发布的项目满足以下任何条件被视为无效，将从索引中移除：

- 项目不符合使用条款；

- 项目是恶意软件（设计用于直接或间接利用或危害系统或用户，促进命令和控制攻击，或执行数据渗出）；

- 项目是垃圾邮件（设计用于广告或 solicitation 商品或服务）；

- 项目包含非法内容；

- 项目侵犯版权、商标、专利或许可证；

- 项目是名称抢注（软件包没有功能或为空）；

- 项目名称、描述或内容违反行为准则；

- 项目使用混淆来隐藏或掩盖功能；

- 项目滥用软件包索引用于非预期目的。

软件包索引维护者出于安全原因预先声明某些软件包名称不可用。

### 知识产权政策

Python 软件基金会和软件包索引维护者的政策是对第三方知识产权侵权索赔做出适当响应。Python 软件基金会和软件包索引维护者的政策不是预先筛选上传的软件包是否有任何类型的知识产权侵权。

可能侵权的软件包应报告给 legal&#64;python&#46;org，Python 软件基金会的法律顾问将确定适当的响应。Python 软件基金会可以全权决定移除或转移软件包给新所有者以解决侵权索赔。

在软件包索引上发布的项目满足以下任何条件可能被视为侵权，并受到从索引中移除或转移给新所有者的处理：

- 项目包含来自第三方的未授权版权材料，并受到 DMCA 下的适当声明；

- 项目使用第三方商标的方式不符合名义或合理使用指南；

- 项目明确涉及专利系统或流程，并是投诉的对象；或

- 项目是活跃诉讼的对象。

在知识产权侵权投诉的情况下，投诉副本将发送给软件包所有者。在某些情况下，软件包索引维护者可能在所有者响应之前采取行动。

### Python 软件基金会的作用

Python 软件基金会 [7] 是提供软件包索引作为社区服务的非营利法律实体。

软件包索引维护者可以将本文档涵盖的问题升级给打包工作组解决，如果事项不够清楚。一些决定需要理事会的额外判断，特别是在行为准则违规或法律索赔的情况下。理事会提出的建议发送给打包工作组 [8] 审查。

打包工作组对本文档涵盖的任何争议有最终决定权，并可以在仔细考虑后决定重新分配或从软件包索引中移除项目，即使这里列出的所有要求都未满足。

## 如何请求名称转移

如果您想接管 PyPI 上的现有项目名称，请遵循以下步骤：

- 尝试直接联系当前所有者：如果他们可以找到相关仓库，给他们发邮件并打开问题。这里描述的流程是如果无法联系所有者时的最后手段。

- 检查上述标准，看看何时允许转移。特别是，为不同项目重用名称的标准比继续维护同一项目的标准更严格——尽管在任何一种情况下都不容易获得名称转移。

- 搜索 PyPI 支持问题，看看是否有人已经在请求相同的名称。

- 如果满足转移名称所有权的所有标准，打开新问题请求它，详细说明为什么您认为每个相关标准都得到满足。

## 现有技术

NPM 包含一个从主页链接的单独部分，称为软件包名称争议。它被描述为"活文档"，截至 2017 年 1 月，其内容可以概括如下：

- 软件包名称抢注被禁止；

- 想要重用项目名称的用户必须联系现有作者，抄送 support&#64;npmjs&#46;com；

- 所有联系必须符合 NPM 行为准则；

- 如果几周后没有解决，npm inc. 保留对此事项的最终决定权。

CPAN 允许任何用户上传具有相同名称的模块。PAUSE，一个相关索引，仅列出由主要维护者或列出的共同维护者上传的模块。CPAN 文档不解决其他争议。

GitHub 的服务条款包含一个详尽的列表，描述不符合一般使用条件的行为。虽然没有任何地方编码，但 GitHub 确实同意用户通过归档被遗弃的账户并让另一个用户或组织重命名他们的账户来重新获得被遗弃的账户名称。这是逐案处理的。

## 被拒绝的提议

最初的方法是希望最好，并在没有书面政策的情况下解决问题。这是不可持续的。缺乏书面的一般可用的软件包名称冲突解决指南正在造成不必要的紧张。从用户的角度来看，软件包索引维护者在没有书面指南的情况下做出的决定可能显得随意。从软件包索引维护者的角度来看，解决名称冲突是一项有压力的任务，因为由于缺乏定义的政策而导致无意伤害的风险。
