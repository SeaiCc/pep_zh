# PEP 282 – 日志系统

**元数据:**

- **作者**: Vinay Sajip <vinay_sajip@red-dove.com>, Trent Mick <trentm at activestate.com>
- **创建日期**: 2002 年 2 月 4 日
- **Python 版本**: 2.3

## 摘要

本 PEP 描述了为 Python 标准库提议的日志记录包。

基本上，该系统涉及用户创建一个或多个记录器对象，在这些对象上调用方法来记录调试注释、一般信息、警告、错误等。可以使用不同的日志记录"级别"来区分重要消息和较不重要的消息。

维护一个命名单例记录器对象的注册表，以便：

- 存在不同的逻辑日志流（或"频道"）（例如，一个用于'zope.zodb'相关内容，另一个用于'mywebsite'特定内容）

- 不必四处传递记录器对象引用。

该系统可以在运行时配置。此配置机制允许人们调整日志记录的级别和类型，而无需更改应用程序本身。

## 动机

如果标准库中确立了一个统一的日志记录机制，那么 1) 日志记录更有可能做得"好"，并且 2) 多个库将能够集成到可以合理一致地记录的大型应用程序中。

## 影响来源

本提案是在研究了以下日志记录包后制定的：

- JDK 1.4 中的 java.util.logging（也称为 JSR047）[1]

- log4j [2]

- Protomatter 项目的 Syslog 包 [3]

- MAL 的 mx.Log 包 [4]

## 简单示例

此示例展示了如何使用日志记录包生成简单的 stderr 日志输出。

```python
--------- mymodule.py -------------------------------
import logging
log = logging.getLogger("MyModule")

def doIt():
    log.debug("Doin' stuff...")
    #do stuff...
    raise TypeError, "Bogus type error for testing"
-----------------------------------------------------
```

```python
--------- myapp.py ----------------------------------
import mymodule, logging

logging.basicConfig()

log = logging.getLogger("MyApp")

log.info("Starting my app")
try:
    mymodule.doIt()
except Exception, e:
    log.exception("There was a problem.")
log.info("Ending my app")
-----------------------------------------------------
```

```shell
$ python myapp.py

INFO:MyApp: Starting my app
DEBUG:MyModule: Doin' stuff...
ERROR:MyApp: There was a problem.
Traceback (most recent call last):
  File "myapp.py", line 9, in ?
    mymodule.doIt()
  File "mymodule.py", line 7, in doIt
    raise TypeError, "Bogus type error for testing"
TypeError: Bogus type error for testing

INFO:MyApp: Ending my app
```

上面的示例显示了默认输出格式。输出格式的所有方面都应该是可配置的，因此您可以拥有如下格式的输出：

```shell
2002-04-19 07:56:58,174 MyModule DEBUG - Doin' stuff...

或者只是 

Doin' stuff...
```

## 控制流

应用程序在 Logger 对象上进行日志记录调用。Loggers 组织在一个分层命名空间中，子 Loggers 从命名空间中的父级继承某些日志记录属性。

Logger 名称符合"点分隔名称"命名空间，点（句号）表示子命名空间。因此，Logger 对象的命名空间对应于单个树数据结构。

- "" 是命名空间的根

- "Zope" 将是根的子节点

- "Zope.ZODB" 将是 "Zope" 的子节点

这些 Logger 对象创建 LogRecord 对象，并将其传递给 Handler 对象进行输出。Loggers 和 Handlers 都可以使用日志记录级别和（可选的）Filters 来决定它们是否对某个 LogRecord 感兴趣。当需要向外输出 LogRecord 时，Handler 可以（可选地）使用 Formatter 在将其发送到 I/O 流之前本地化和格式化消息。

每个 Logger 跟踪一组输出 Handlers。默认情况下，所有 Loggers 还将输出发送到其祖先 Logger 的所有 Handlers。但是，也可以配置 Loggers 忽略树中更高层的 Handlers。

API 的结构使得当日志记录被禁用时，Logger API 上的调用可以很廉价。如果给定日志级别禁用了日志记录，则 Logger 可以进行廉价的比较测试并返回。如果给定的日志级别启用了日志记录，Logger 在将 LogRecord 传递给 Handlers 之前仍会小心地最小化成本。特别是，本地化和格式化（相对昂贵）将被推迟到 Handler 请求它们时。

整个 Logger 层次结构也可以关联一个级别，它优先于各个 Loggers 的级别。这是通过模块级函数完成的：

```python
def disable(lvl):
    """
    对于严重性小于 'lvl' 的请求，不生成任何 LogRecord。
    """
    ...
```

## 级别

日志记录级别按重要性递增顺序排列如下：

- DEBUG

- INFO

- WARN

- ERROR

- CRITICAL

术语 CRITICAL 优先于 log4j 使用的 FATAL。这些级别在概念上是相同的——严重或非常严重的错误。但是，FATAL 意味着死亡，在 Python 中意味着引发未捕获的异常、回溯和退出。由于日志记录模块不强制从 FATAL 级别日志条目产生这样的结果，因此使用 CRITICAL 而不是 FATAL 是有意义的。

这些只是整数常量，以便简单比较重要性。经验表明，太多的级别会让人困惑，因为它们导致对应该将哪个级别应用于任何特定日志请求的主观解释。

虽然强烈建议上述级别，但日志记录系统不应具有规定性。用户可以定义自己的级别，以及任何级别的文本表示。但是，用户定义的级别必须遵守它们都是正整数并且按严重性递增顺序增加的约束。

用户定义的日志记录级别通过两个模块级函数支持：

```python
def getLevelName(lvl):
    """返回级别 'lvl' 的文本。"""
    ...

def addLevelName(lvl, lvlName):
    """
    添加级别 'lvl' 及其关联文本 'levelName'，或者
    将现有级别 'lvl' 的文本表示设置为 'lvlName'。"""
    ...
```

## Loggers

每个 Logger 对象跟踪其感兴趣的日志级别（或阈值），并丢弃低于该级别的日志请求。

Manager 类实例维护命名 Logger 对象的分层命名空间。代际用点分隔的名称表示：Logger "foo" 是 Loggers "foo.bar" 和 "foo.baz" 的父级。

Manager 类实例是一个单例，不直接向用户暴露，用户使用各种模块级函数与其交互。

通用日志记录方法是：

```python
class Logger:
    def log(self, lvl, msg, *args, **kwargs):
        """在日志级别 'lvl' 记录 'str(msg) % args'。"""
        ...
```

但是，为每个日志级别定义了方便函数：

```python
class Logger:
    def debug(self, msg, *args, **kwargs): ...
    def info(self, msg, *args, **kwargs): ...
    def warn(self, msg, *args, **kwargs): ...
    def error(self, msg, *args, **kwargs): ...
    def critical(self, msg, *args, **kwargs): ...
```

目前只识别一个关键字参数——"exc_info"。如果为 true，调用方希望在日志输出中提供异常信息。仅当需要在任何日志级别提供异常信息时才需要此机制。在更常见的情况下，仅当错误发生时才需要将异常信息添加到日志中，即在 ERROR 级别，则提供另一个便捷方法：

```python
class Logger:
    def exception(self, msg, *args): ...
```

这只应在异常处理程序的上下文中调用，是指定希望在日志中包含异常信息的优选方式。其他便捷方法旨在仅在罕见情况下与 exc_info 一起调用，例如，您希望在 INFO 消息的上下文中提供异常信息。

上面的 "msg" 参数通常是格式字符串；但是，它可以是任何对象 x，其中 str(x) 返回格式字符串。这促进了例如使用获取特定于区域设置的对象的国际化和/或本地化应用程序，可能使用标准 gettext 模块。示例概要：

```python
class Message:
    """表示一条消息"""
    def __init__(self, id):
        """用消息 ID 初始化"""

    def __str__(self):
        """返回适当的本地化消息文本"""

    ...

logger.info(Message("abc"), ...)
```

收集和格式化日志消息的数据可能很昂贵，如果记录器本来就要丢弃该消息，那就是浪费。要查看请求是否会被记录器接受，可以使用 isEnabledFor() 方法：

```python
class Logger:
    def isEnabledFor(self, lvl):
        """
        如果级别 'lvl' 的请求不会被丢弃，则返回 true。
        """
        ...
```

因此，代替这个昂贵且可能浪费的 DOM 到 XML 转换：

```python
    ...
    hamletStr = hamletDom.toxml()
    log.info(hamletStr)
    ...
```

可以这样做：

```python
if log.isEnabledFor(logging.INFO):
    hamletStr = hamletDom.toxml()
    log.info(hamletStr)
```

创建新记录器时，它们被初始化为表示"无级别"的级别。可以使用 setLevel() 方法显式设置级别：

```python
class Logger:
    def setLevel(self, lvl): ...
```

如果未设置记录器的级别，系统将咨询其所有祖先，在层次结构中向上遍历，直到找到显式设置的级别。这被视为记录器的"有效级别"，可以通过 getEffectiveLevel() 方法查询：

```python
def getEffectiveLevel(self): ...
```

Logger 从不直接实例化。相反，使用模块级函数：

```python
def getLogger(name=None): ...
```

如果未指定名称，则返回根记录器。否则，如果存在该名称的记录器，则返回它。如果没有，则初始化并返回新记录器。这里，"name" 与 "channel name" 同义。

用户可以指定 Logger 的自定义子类，以便系统在新实例化记录器时使用：

```python
def setLoggerClass(klass): ...
```

传递的类应该是 Logger 的子类，其 `__init__` 方法应该调用 `Logger.__init__`。

## Handlers

Handlers 负责对所给 LogRecord 执行有用的操作。将实现以下核心 Handlers：

- StreamHandler：用于写入文件类对象的处理程序。

- FileHandler：用于写入单个文件或一组轮换文件的处理程序。

- SocketHandler：用于写入远程 TCP 端口的处理程序。

- DatagramHandler：用于写入 UDP 套接字的处理程序，用于低成本日志记录。Jeff Bauer 已经有了这样的系统 [5]。

- MemoryHandler：在内存中缓冲日志记录的处理程序，直到缓冲区已满或发生特定条件 [1]。

- SMTPHandler：用于通过 SMTP 发送到电子邮件地址的处理程序。

- SysLogHandler：用于通过 UDP 写入 Unix syslog 的处理程序。

- NTEventLogHandler：用于写入 Windows NT、2000 和 XP 上事件日志的处理程序。

- HTTPHandler：用于使用 GET 或 POST 语义写入 Web 服务器的处理程序。

也可以使用 setLevel() 方法为 Handlers 设置级别：

```python
def setLevel(self, lvl): ...
```

FileHandler 可以设置为创建一组轮换的日志文件。在这种情况下，传递给构造函数的文件名被视为"基础"文件名。通过向基础文件名附加 .1、.2 等来创建轮换的附加文件名，最多到请求轮换时指定的最大值。

setRollover 方法用于指定日志文件的最大大小和轮换中备份文件的最大数量。

```python
def setRollover(maxBytes, backupCount): ...
```

如果 maxBytes 指定为零，则永远不会发生轮换，日志文件无限增长。如果指定了非零大小，当即将超过该大小时，发生轮换。rollover 方法确保基础文件名始终是最新的，.1 是次最新的，.2 是下一个最新的，依此类推。

在 [6] 附带的测试/示例脚本中实现了许多额外的处理程序——例如，XMLHandler 和 SOAPHandler。

## LogRecords

LogRecord 作为关于日志记录事件的信息的容器。它只不过是一个字典，尽管它定义了一个 getMessage 方法，该方法将消息与可选的运行参数合并。

## Formatters

Formatter 负责将 LogRecord 转换为字符串表示。Handler 在写入记录之前可以调用其 Formatter。将实现以下核心 Formatters：

- Formatter：提供 printf 样式的格式化，使用 % 运算符。

- BufferingFormatter：为多条消息提供格式化，支持页眉和页脚格式化。

通过在处理程序上调用 setFormatter() 将 Formatters 与 Handlers 关联：

```python
def setFormatter(self, form): ...
```

Formatters 使用 % 运算符格式化日志消息。格式字符串应包含 %(name)x，并且使用 LogRecord 的属性字典来获取特定于消息的数据。提供以下属性：

| 属性 | 说明 |
|--------|------|
| `%(name)s` | Logger 的名称（日志记录频道） |
| `%(levelno)s` | 消息的数值日志级别（DEBUG、INFO、WARN、ERROR、CRITICAL） |
| `%(levelname)s` | 消息的文本日志级别（"DEBUG"、"INFO"、"WARN"、"ERROR"、"CRITICAL"） |
| `%(pathname)s` | 发出日志记录调用的源文件的完整路径名（如果可用） |
| `%(filename)s` | 路径名的文件名部分 |
| `%(module)s` | 发出日志记录调用的模块 |
| `%(lineno)d` | 发出日志记录调用的源行号（如果可用） |
| `%(created)f` | 创建 LogRecord 的时间（time.time() 返回值） |
| `%(asctime)s` | 创建 LogRecord 的文本时间 |
| `%(msecs)d` | 创建时间的毫秒部分 |
| `%(relativeCreated)d` | 创建 LogRecord 时的时间（以毫秒为单位），相对于加载日志记录模块的时间（通常在应用程序启动时） |
| `%(thread)d` | 线程 ID（如果可用） |
| `%(message)s` | record.getMessage() 的结果，在发出记录时计算 |

如果格式化程序看到格式字符串包含 "%(asctime)s"，则创建时间被格式化为 LogRecord 的 asctime 属性。为了在格式化日期方面提供灵活性，Formatters 用整个消息的格式字符串和日期/时间的单独格式字符串初始化。日期/时间格式字符串应为 time.strftime 格式。消息格式的默认值为 "%(message)s"。日期/时间格式的默认值为 ISO8601。

格式化程序使用类属性 "converter" 来指示如何将时间从秒转换为元组。默认情况下，"converter" 的值为 "time.localtime"。如果需要，可以在单独的格式化程序实例上设置不同的转换器（例如 "time.gmtime"），或者更改类属性以影响所有格式化程序实例。

## Filters

当基于级别的过滤不足时，Logger 或 Handler 可以调用 Filter 来决定是否应输出 LogRecord。Loggers 和 Handlers 可以安装多个过滤器，其中任何一个都可以否决输出 LogRecord。

```python
class Filter:
    def filter(self, record):
        """
        返回一个值，指示 true 如果记录要被处理。如果过滤器认为合适，可能会修改记录。
        """
```

默认行为允许 Filter 用 Logger 名称初始化。这将只允许使用命名记录器或其任何子级生成的事件通过。例如，用 "A.B" 初始化的过滤器将允许由记录器 "A.B"、"A.B.C"、"A.B.C.D"、"A.B.D" 等记录的事件，但不允许 "A.BB"、"B.A.B" 等。如果初始化为空字符串，则所有事件都通过过滤器。这种过滤器行为在希望关注应用程序的特定区域时很有用；只需更改附加到根记录器的过滤器即可更改焦点。

[6] 中提供了许多 Filter 示例。

## 配置

此类日志记录系统的主要好处是，可以在不更改应用程序源代码的情况下控制从应用程序获得多少和什么日志输出。因此，虽然可以通过日志记录 API 执行配置，但也必须能够在根本不更改应用程序的情况下更改日志记录配置。对于像 Zope 这样长时间运行的程序，应该能够在程序运行时更改日志记录配置。

配置包括以下内容：

- Logger 或 Handler 应感兴趣的日志级别。

- 应将哪些 handlers 附加到哪些 loggers。

- 应将哪些过滤器附加到哪些 handlers 和 loggers。

- 指定特定于某些 handlers 和 filters 的属性。

一般来说，每个应用程序都有用户如何配置日志输出自己的要求。但是，每个应用程序将通过标准机制向日志记录系统指定所需配置。

最简单的配置是单个 handler，写入 stderr，附加到根记录器。一旦导入日志记录模块，通过调用 basicConfig() 函数设置此配置。

```python
def basicConfig(): ...
```

对于更复杂的配置，此 PEP 不做具体建议，原因如下：

- 具体建议可能被视为具有规定性。

- 没有 Python 社区广泛实践经验的益处，无法知道任何给定的配置方法是否良好。这种实践真正到来只能在日志记录模块被使用之后，这意味着在 Python 2.3 发布之后。

- 不同类型的应用程序可能需要不同的配置方法，因此没有"一刀切"的方案。

参考实现 [6] 有一个工作的配置文件格式，为了证明概念和提出一种可能的替代方案而实现。可能会创建单独的扩展模块，不属于核心 Python 发行版的一部分，用于日志记录配置和日志查看、辅助处理程序和其他不为大多数社区感兴趣的特性。

## 线程安全

日志记录系统应支持线程安全操作，而无需用户采取任何特殊措施。

## 模块级函数

为了支持在短脚本和小应用程序中使用日志记录机制，提供了模块级函数 debug()、info()、warn()、error()、critical() 和 exception()。这些以与 Logger 相应命名的方法相同的方式工作——实际上它们委托给根记录器上的相应方法。这些函数提供的另一个便利是，如果未进行任何配置，则自动调用 basicConfig()。

在应用程序退出时，可以通过调用以下函数刷新所有处理程序：

```python
def shutdown(): ...
```

这将刷新并关闭所有处理程序。
