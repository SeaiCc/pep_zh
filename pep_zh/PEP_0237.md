# PEP 237 – 统一长整型和整型

**元数据：**

- **作者**: Moshe Zadka, Guido van Rossum
- **创建时间**: 11-Mar-2001
- **Python 版本**: 2.2
- **发布历史**: 16-Mar-2001, 14-Aug-2001, 23-Aug-2001

## 摘要

Python 目前区分两种整数（int）：常规或短整数，受 C long 大小限制（通常为 32 或 64 位），以及长整数，仅受可用内存限制。当短整数的运算结果不适合 C long 时，会引发错误。还有一些其他区别。本 PEP 提议消除大部分语义差异，从 Python 用户角度统一这两种类型。

## 理由

许多程序事后发现需要处理更大的数字，而稍后更改算法会很麻烦。当所有算术运算都使用长整数时（无论是否需要），可能会降低正常情况下的性能。

将机器字长暴露给语言会阻碍可移植性。例如，因此 Python 源文件和 .pyc 在 32 位和 64 位机器之间不可移植。

此外，还有一般愿望，即当不必要的细节对大多数应用程序无关紧要时，将其隐藏起来。例如，内存分配在 C 中是显式的，但在 Python 中是自动的，这为我们提供了字符串、列表等无限大小的便利。将这种便利扩展到数字是合理的。

这将使新 Python 程序员（无论他们是否对编程本身很新）少学一件事就可以开始使用语言。

## 实现

最初，提出了两种替代实现方案（每位作者各提出一种）：

- PyInt 类型用于 C long 的槽将变为：
  ```c
  union {
      long i;
      struct {
          unsigned long length;
          digit digits[1];
      } bignum;
  };
  ```
  只有 long 的低 n-1 位有意义；最高位始终设置。这区分了联合。所有 PyInt 函数在决定使用哪种类型的操作之前都将检查此位。

- 现有的短整型和长整型保持不变，但是当结果无法表示为短整数时，运算返回长整数而不是引发 OverflowError。可能会引入一个新类型 integer，它是一个抽象基类型，int 和 long 实现类型都继承自它。这很有用，这样程序可以用单个测试检查整数性：
  ```python
  if isinstance(i, integer): ...
  ```

经过一些考虑后，选择了第二种实现方案，因为它更容易实现，在 C API 层面是向后兼容的，此外还可以部分实现作为过渡措施。

## 不兼容性

以下操作对短整数和长整数具有（通常微妙的）不同语义，其中一个或另一个必须以某种方式更改。这旨在成为详尽列表。如果您知道任何其他操作的结果取决于传递的是短整数还是具有相同值的长整数，请写信给第二位作者。

目前，除了 `<<` 之外的所有短整数算术运算符，如果结果无法表示为短整数，都会引发 `OverflowError`。这将改为返回长整数。以下运算符当前可能引发 `OverflowError`：`x+y`、`x-y`、`x*y`、`x**y`、`divmod(x, y)`、`x/y`、`x%y` 和 `-x`。（最后四个仅在涉及值 `-sys.maxint-1` 时才会溢出。）

- 目前，`x<<n` 对于短整数可能会丢失位。这将改为返回包含所有移位位的长整数，如果返回短整数会丢失位（其中改变符号被视为丢失位的特殊情况）。

- 目前，短整数的 hex 和 oct 字面量可以指定负值；例如，在 32 位机器上 `0xffffffff == -1`。这将改为等于 `0xffffffffL（2**32-1）`。

- 目前，`%u`、`%x`、`%X` 和 `%o` 字符串格式化运算符以及 hex() 和 oct() 内置函数对负数的行为不同：负短整数被格式化为无符号 C long，而负长整数被格式化为带负号。这将改为在所有情况下使用长整数语义（但没有当前区分 hex() 和 oct() 长整数输出的尾随 L）。注意，这意味着 %u 成为 %d 的别名。它最终将被删除。

- 目前，长整数的 repr() 返回以 L 结尾的字符串，而短整数的 repr() 不返回。L 将被删除；但在 Python 3.0 之前不会。

- 目前，具有长操作数的运算永远不会返回短整数。这可能会改变，因为它允许一些优化。（目前在此领域尚未进行任何更改，也没有计划。）

- 表达式 `type(x).__name__` 取决于 x 是短整数还是长整数。由于选择了实现替代方案 2，这种差异将保留。（在 Python 3.0 中，我们可能能够部署一个技巧来隐藏这种差异，因为向用户代码揭示这种差异令人烦恼，而且随着两种类型之间的差异变得不太可见，更是如此。）

- marshal 模块以及 pickle 和 cPickle 模块对长整数和短整数的处理方式不同。这种差异将保留（至少在 Python 3.0 之前）。

- 小值（通常在 -1 到 99 之间，包含两端）的短整数是内部化的——每当结果具有这样的值时，返回具有相同值的现有短整数。对于具有相同值的长整数不这样做。这种差异将保留。（由于没有这种内部化的保证，因此这是语义差异是可以争论的——但可能存在使用 is 比较短整数的代码，并且由于这种内部化而工作。如果用长整数使用这种代码，可能会失败。）

## 字面量

整数字面量末尾的尾随 L 将不再有任何意义，最终将变得非法。编译器将仅基于值选择适当的类型。（直到 Python 3.0，它将强制字面量为长整数；但没有尾随 L 的字面量也可能是长整数，如果它们无法表示为短整数。）

## 内置函数

函数 int() 将根据参数值返回短整数或长整数。在 Python 3.0 中，函数 long() 将调用函数 int()；在此之前，它将继续强制结果为长整数，但其他方式与 int() 相同。内置名称 long 将保留在语言中代表长实现类型（除非在 Python 3.0 中完全消除），但仍然建议使用 int() 函数，因为它将在需要时自动返回长整数。

## C API

C API 保持不变；C 代码仍然需要知道短整数和长整数之间的区别。（Python 3.0 C API 可能完全不兼容。）

PyArg_Parse*() API 已经接受长整数，只要它们在 C int 或 long 可表示的范围内，因此接受 C int 或 long 参数的函数不必担心处理 Python 长整数。

## 过渡

过渡有三个主要阶段：

- 当前引发 OverflowError 的短整数运算改为返回长整数值。这是此阶段的唯一更改。字面量仍将区分短整数和长整数。上面列出的其他语义差异（包括 << 的行为）将保持不变。因为此阶段仅更改当前引发 OverflowError 的情况，所以假设这不会破坏现有代码。（依赖此异常的代码必须过于复杂，以至于不必担心它。）对于那些担心极端向后兼容性的人，命令行选项（或对 warnings 模块的调用）将允许在此点发出警告或错误，但这默认是关闭的。

- 解决剩余的语义差异。在所有情况下，长整数语义将占上风。由于这将引入向后不兼容性，这将破坏一些旧代码，此阶段可能需要 future 语句和/或警告，以及漫长的过渡阶段。尾随 L 将继续用于 longs 作为输入并由 repr() 使用。
  - 关于将在阶段 2B 中更改其数值结果的运算发出警告，特别是 hex() 和 oct()、%u、%x、%X 和 %o、在（包含）范围 [sys.maxint+1, sys.maxint*2+1] 中的 hex 和 oct 字面量，以及丢失位的左移。
  - 实现这些运算的新语义。给出与之前不同结果的运算将不会发出警告。
  - 从 repr() 中删除尾随 L，并在输入中使其非法。（如果可能，long 类型完全消失。）从 hex() 和 oct() 中也删除尾随 L。

阶段 1 将在 Python 2.2 中实现。

阶段 2 将逐步实现，2A 在 Python 2.3 中，2B 在 Python 2.4 中。

阶段 3 将在 Python 3.0 中实现（至少在 Python 2.4 发布两年之后）。

## OverflowWarning

以下是指导当前引发 OverflowError 的情况中生成的警告的规则。这适用于过渡阶段 1。历史备注：尽管阶段 1 在 Python 2.2 中完成，阶段 2A 在 Python 2.3 中完成，但没有人注意到 OverflowWarning 在 Python 2.3 中仍然生成。它最终在 Python 2.4 中被禁用。Python 内置 OverflowWarning 和相应的 C API PyExc_OverflowWarning 在 Python 2.4 中不再生成或使用，但将保留到 Python 2.5（不太可能）的用户代码情况。

- 引入一个新的警告类别 OverflowWarning。这是一个内置名称。

- 如果 int 结果溢出，则发出 OverflowWarning 警告，带有指示操作的参数，例如 "integer addition"。这可能会导致警告消息显示在 sys.stderr 上，也可能导致异常被引发，全部由 -W 命令行和 warnings 模块控制。

- OverflowWarning 警告默认被忽略。

- OverflowWarning 警告可以像所有警告一样通过 -W 命令行选项或通过 warnings.filterwarnings() 调用来控制。例如：
  ```bash
  python -Wdefault::OverflowWarning
  ```
  导致 OverflowWarning 在特定源行第一次出现时显示，而：
  ```bash
  python -Werror::OverflowWarning
  ```
  导致 OverflowWarning 在每次发生时转换为异常。以下代码从程序内部启用警告：
  ```python
  import warnings
  warnings.filterwarnings("default", "", OverflowWarning)
  ```
  请参阅 python man page 了解 -W 选项和 warnings 模块文档了解 filterwarnings()。

- 如果 OverflowWarning 警告被转换为错误，则替换 OverflowError。这是为了向后兼容性。

- 除非警告被转换为异常，否则运算结果（例如，x+y）在将参数转换为长整数后重新计算。

## 示例

如果您将长整数传递给接受整数的 C 函数或内置运算，它将作为短整数处理，只要值适合（由于 PyArg_ParseTuple() 的实现方式）。如果长值不适合，它仍然会引发 OverflowError。例如：

```python
def fact(n):
    if n <= 1:
        return 1
    return n*fact(n-1)

A = "ABCDEFGHIJKLMNOPQ"
n = input("Gimme an int: ")
print A[fact(n)%17]
```

对于 n >= 13，这当前会引发 OverflowError（除非用户输入尾随 L 作为输入的一部分），即使计算出的索引始终在 range(17) 内。使用新方法，此代码将做正确的事情：索引将作为长整数计算，但其值将在范围内。

## 已解决的问题

这些问题先前是开放的，现已解决。

- hex() 和 oct() 应用于长整数将继续产生尾随 L，直到 Python 3000。上面的原始文本对此不清楚，但由于它没有发生在 Python 2.4 中，所以最好保持原样。BDFL 宣布在此：https://mail.python.org/pipermail/python-dev/2006-June/065918.html

- 如何处理 sys.maxint？保留它，因为它在与短整数和长整数的区别仍然相关时仍然相关（例如，检查值的类型时）。

- 我们应该完全删除 %u 吗？删除它。

- 我们应该警告 << 不截断整数吗？是的。

- 溢出警告应该基于可移植的最大大小吗？不。

## 实现

Python 2.x 系列的实现工作已完成；阶段 1 随 Python 2.2 发布，阶段 2A 随 Python 2.3 发布，阶段 2B 将随 Python 2.4 发布（已经在 CVS 中）。
