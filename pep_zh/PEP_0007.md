# PEP 7 – C 代码风格指南

**元数据:**

- **作者**: Guido van Rossum <guido&#32;&#97;t&#32;python.org>, Barry Warsaw <barry&#32;&#97;t&#32;python.org>
- **创建日期**: 05-Jul-2001

## 介绍

本文档给出了构成 Python C 实现的 C 代码的编码约定。请参阅配套的说明性 PEP，其中描述了 Python 代码的风格指南。

注意，规则就是用来打破的。打破特定规则的两个好理由：

- 当应用该规则会使代码可读性降低时，即使对于习惯阅读遵循规则代码的人来说也是如此。

- 为了与也打破该规则的周围代码保持一致（可能是出于历史原因）——尽管这也是清理他人烂摊子的机会（以真正的 XP 风格）。

## C 标准

遵循以下标准。
对于相关标准中没有的功能，使用 CPython 特定的包装器（例如：_Py_atomic_store_int32、Py_ALWAYS_INLINE、Py_ARITHMETIC_RIGHT_SHIFT；公共头文件中的_Py_ALIGNED_DEF）。
添加此类包装器时，尽量使其易于调整以支持不受支持的编译器。

- Python 3.11 及更新版本使用 C11，不使用可选功能。
公共 C API 应与 C99 和 C++ 兼容。（提醒任何阅读本文的用户：此 PEP 是风格指南；这些规则就是用来打破的。）

- Python 3.6 到 3.10 使用 C89，带有几个精选的 C99 功能：
  - <stdint.h>和<inttypes.h>中的标准整数类型。我们需要固定宽度的整数类型。
  - static inline 函数
  - designated initializers（特别适用于类型声明）
  - 交错的声明
  - 布尔值
  - C++ 风格的行注释

- Python 3.6 之前的版本使用 ANSI/ISO 标准 C（1989 年版标准）。这意味着，除其他事项外，所有声明都在块的顶部。

## 通用 C 代码约定

- 不要使用编译器特定的扩展，例如 GCC 或 MSVC 的扩展。例如，不要编写没有尾部反斜杠的多行字符串。

- 所有函数声明和定义必须使用完整的原型。也就是说，指定所有参数的类型，并使用 (void) 声明没有参数的函数。

- 主要编译器（gcc、VC++ 等）没有编译器警告。

- 新代码中应优先使用 static inline 函数而不是宏。

## 代码布局

- 使用 4 空格缩进，不使用制表符。

- 任何行不应超过 79 个字符。如果此规则和上一条规则一起不能给您足够的编码空间，则您的代码太复杂了——考虑使用子程序。

- 任何行不应以空白字符结尾。如果您认为需要重要的尾部空白，再想一想——某人的编辑器可能会将其作为常规操作删除。

- 函数定义风格：函数名在第 1 列，最外层花括号在第 1 列，局部变量声明后空一行。

```c
static int
extra_ivars(PyTypeObject *type, PyTypeObject *base)
{
    int t_size = PyType_BASICSIZE(type);
    int b_size = PyType_BASICSIZE(base);

    assert(t_size >= b_size); /* type smaller than base! */
    ...
    return 1;
}
```

代码结构：if、for 等关键字与后面的左括号之间有一个空格；括号内没有空格；所有地方都需要花括号，即使 C 允许省略它们，但不要将花括号添加到您未修改的代码中。所有新 C 代码都需要花括号。花括号应按所示格式排列：

```c
if (mro != NULL) {
    ...
}
else {
    ...
}
```

return 语句不应有冗余的括号：

```c
return(albatross); /* incorrect */
```

而是：

```c
return albatross; /* correct */
```

函数和宏调用风格：foo(a, b, c)——左括号前没有空格，括号内没有空格，逗号前没有空格，每个逗号后有一个空格。

赋值、布尔和比较运算符周围始终有空格。在使用大量运算符的表达式中，在最外层（最低优先级）运算符周围添加空格。

换行长行：如果可以，在最外层参数列表的逗号后换行。始终适当缩进续行，例如：

```c
PyErr_Format(PyExc_TypeError,
             "cannot create '%.100s' instances",
             type->tp_name);
```

当您在二元运算符处换行长表达式时，花括号应按所示格式排列：

```c
if (type->tp_dictoffset != 0
    && base->tp_dictoffset == 0
    && type->tp_dictoffset == b_size
    && (size_t)t_size == b_size + sizeof(PyObject *))
{
    return 0; /* "Forgive" adding a __dict__ only */
}
```

将运算符放在行尾是可以的，特别是为了与周围代码保持一致。
（有关更详细的讨论，请参阅 PEP 8。）

- 在多行宏中垂直对齐行续行字符。

- 设计为语句使用的宏应使用 do { ... } while (0) 宏惯用法，不带末尾分号。
示例：

```c
#define ADD_INT_MACRO(MOD, INT) \
    do { \
        if (PyModule_AddIntConstant((MOD), (#INT), (INT)) < 0) { \
            goto error; \
        } \
    } while (0)

// 用作带分号的语句：
ADD_INT_MACRO(m, SOME_CONSTANT);
```

使用后#undef 文件局部宏。

函数、结构定义和函数内主要部分周围放空行。

注释放在它们描述的代码之前。

所有函数和全局变量应声明为 static，除非它们是已发布接口的一部分。

对于外部函数和变量，我们在"Include"目录中的适当头文件中始终有一个声明，它使用 PyAPI_FUNC() 宏和 PyAPI_DATA() 宏，如下所示：

```c
PyAPI_FUNC(PyObject *) PyObject_Repr(PyObject *);

PyAPI_DATA(PyTypeObject) PySuper_Type;
```

## 命名约定

公共函数使用 Py 前缀；静态函数永不用。Py_前缀保留用于全局服务例程，如 Py_FatalError；特定例程组（例如特定对象类型 API）使用更长的前缀，例如字符串函数使用 PyString_。

公共函数和变量使用带下划线的 MixedCase，如下所示：PyObject_GetAttr、Py_BuildValue、PyExc_TypeError。

偶尔，"内部"函数必须对加载器可见；我们为此使用_Py 前缀，例如：_PyObject_Dump。

宏应该有 MixedCase 前缀，然后使用大写字母，例如：PyString_AS_STRING、Py_PRINT_RAW。

宏参数应使用 ALL_CAPS 风格，以便它们很容易与 C 变量和结构成员区分开来。

## 文档字符串

使用 PyDoc_STR() 或 PyDoc_STRVAR() 宏用于 docstrings，以支持不使用 docstrings 构建 Python（./configure --without-doc-strings）。

每个函数 docstring 的第一行应该是"签名行"，给出参数和返回值的简要概述。例如：

```c
PyDoc_STRVAR(myfunction__doc__,
            "myfunction(name, value) -> bool\n\n\"
            "Determine whether name and value make a valid pair.");
```

始终在签名行和描述文本之间包含一个空行。

如果函数的返回值始终为 None（因为没有有意义的返回值），则不要包含返回类型的指示。

- 编写多行 docstrings 时，请确保始终使用反斜杠续行，如上例所示，或使用字符串字面量连接：

```c
PyDoc_STRVAR(myfunction__doc__,
            "myfunction(name, value) -> bool\n\n"
            "Determine whether name and value make a valid pair.");
```

虽然一些 C 编译器接受不带这两种方式的字符串字面量：

```c
/* BAD -- don't do this! */
PyDoc_STRVAR(myfunction__doc__,
            "myfunction(name, value) -> bool\n\n
            Determine whether name and value make a valid pair.");
```

但并非所有编译器都接受；已知 MSVC 编译器会对此发出抱怨。
