**元数据:**

- **作者**: Barry Warsaw, Jeremy Hylton, David Goodger, Alyssa Coghlan
- **创建**: 2000 年 6 月 13 日
- **发布历史**: 2001 年 3 月 21 日，2002 年 7 月 29 日，2003 年 5 月 3 日，2012 年 5 月 5 日，2013 年 4 月 7 日

## 什么是 PEP？

PEP 是 Python Enhancement Proposal（Python 增强提案）的缩写。PEP 是一份设计文档，向 Python 社区提供信息，或描述 Python 及其流程或环境的新功能。PEP 应提供该功能的技术规范及其设计理由。

我们的意图是让 PEP 成为提出主要新功能、收集社区对某个问题的意见以及记录 Python 设计决策的主要机制。PEP 作者负责在社区内建立共识并记录不同意见。

由于 PEP 作为版本化存储库中的文本文件进行维护，因此其修订历史就是功能提案的历史记录。该历史记录可通过正常的 git 命令检索旧版本，也可在 GitHub 上浏览。

## PEP 受众

PEP 的主要受众通常是 CPython 参考解释器的核心开发人员及其选举产生的指导委员会，以及其他 Python 语言规范实现的语言规范的开发人员。

然而，Python 社区的其他部分也可能选择使用此流程（特别是对于信息性 PEP）来记录预期的 API 约定并管理需要跨多个项目协作的复杂设计协调问题。

## PEP 类型

有三种类型的 PEP：

- **标准Track** PEP 描述 Python 的新功能或实现。它还可以描述将在标准库之外支持的可互操作标准，以便在后续 PEP 在未来版本中添加标准库支持之前支持当前 Python 版本。

- **信息性** PEP 描述 Python 设计问题，或为 Python 社区提供一般指南或信息，但不提出新功能。信息性 PEP 不一定代表 Python 社区共识或建议，因此用户和实现者可以自由选择忽略信息性 PEP 或遵循其建议。

- **流程** PEP 描述与 Python 相关的流程，或提议对流程（或流程中的事件）进行更改。流程 PEP 类似于标准轨道 PEP，但适用于 Python 语言本身以外的领域。它们可以提出实现方案，但并非针对 Python 代码库；它们通常需要社区共识；与信息性 PEP 不同，它们不仅仅是建议，用户通常不能自由选择忽略它们。示例包括程序、指南、决策流程的更改以及 Python 开发中使用的工具或环境的更改。任何元 PEP 也被视为流程 PEP。

## PEP 工作流程

### Python 指导委员会

本 PEP 中有多处提及"指导委员会"或"委员会"。这指的是 PEP 13 中描述的被选举产生的指导委员会的现任成员，他们作为 PEP 是否被接受或拒绝的最终决定者。

### Python 核心开发人员

本 PEP 中有多处提及"核心开发人员"。这指的是 PEP 13 中描述的当前活跃的 Python 核心团队成员。

### Python 的 BDFL

本 PEP 的早期版本使用"BDFL-Delegate"（BDFL 委托人）作为 PEP 决策者的头衔。这是对 Python 先前治理模式的历史性引用，其中所有设计权最终都源自 Python 编程语言的原始创建者 Guido van Rossum。相比之下，指导委员会的设计权来自于当前活跃的核心开发人员的选举。现在，PEP-Delegate 用来代替 BDFL-Delegate。

### PEP 编辑

PEP 编辑是负责管理 PEP 工作流程的行政和编辑方面（例如分配 PEP 编号和更改其状态）的个人。详见 PEP 编辑职责与工作流程。

PEP 编辑职位由现任编辑邀请担任，他们可以通过在 GitHub 上提及 @python/pep-editors 进行联系。所有 PEP 工作流程都可以通过 GitHub PEP 存储库的问题和拉取请求进行。

### 从 Python 的一个想法开始

PEP 流程始于一个关于 Python 的新想法。强烈建议单个 PEP 包含一个关键提案或新想法；PEP 越聚焦，往往越成功。大多数增强和错误修复不需要 PEP，可以直接提交到 Python 问题跟踪器。

PEP 编辑保留拒绝 PEP 提案的权利，如果它们显得过于分散或过于宽泛。如有疑问，请将您的 PEP 拆分为几个聚焦良好的 PEP。

每个 PEP 必须有一个倡导者——某人使用下面描述的样式和格式编写 PEP，在适当的论坛中引导讨论，并尝试围绕该想法建立社区共识。PEP 倡导者（也称为作者）应首先尝试确定该想法是否符合 PEP 要求。发布到 Python Discourse 的想法类别通常是处理此事的最佳方式，除非有更专业的场所更合适，例如 Python Discourse 上的类型类别（用于静态类型想法）或打包类别（用于打包想法）。

在编写 PEP 之前公开审核一个想法旨在节省潜在作者的时间。许多改变 Python 的想法因各种原因被拒绝。首先询问 Python 社区一个想法是否原创，有助于防止在注定因先前讨论而被拒绝的事情上花费过多时间（搜索互联网并不总能解决问题）。这还有助于确保该想法适用于整个社区，而不仅仅是作者。仅仅因为一个想法对作者听起来不错，并不意味着它在 Python 使用的大多数领域对大多数人都有用。

一旦倡导者询问了 Python 社区某个想法是否有被接受的可能，就应该将草稿 PEP 提交到上述适当场所。这给了作者一个机会来充实草稿 PEP，使其格式正确、质量高，并解决关于该提案的初步关注。

### 提交 PEP

在上述初步讨论之后，工作流程取决于 PEP 的合著者中是否有核心开发人员。如果一个或多个 PEP 的合著者是核心开发人员，他们负责遵循下面概述的流程。否则（即没有合著者是核心开发人员），则 PEP 作者需要为 PEP 找到一位赞助人。

理想情况下，会确定一位核心开发人员赞助人，但非核心赞助人也可以在指导委员会批准下被选中。GitHub"PEP 编辑"团队和类型委员会（PEP 729）的成员已被预先批准为赞助人。赞助人的工作是向 PEP 作者提供指导，帮助他们通过 PEP 流程的后勤工作（有点像导师）。成为赞助人并不会使该人失去日后成为合著者或 PEP-Delegate 的资格（但不能同时担任两者）。PEP 的赞助人记录在标题的"Sponsor:"字段中。

一旦赞助人或与 PEP 合著的核心开发人员认为 PEP 准备好提交，提案应作为草稿 PEP 通过 GitHub 拉取请求提交。草稿必须按照下面描述的 PEP 风格编写，否则将立即审查失败（尽管小错误可以由编辑纠正）。

标准 PEP 工作流程是：

- 您，PEP 作者，分叉 PEP 存储库，并创建一个名为 pep-NNNN.rst 的文件，其中包含您的新 PEP。NNNN 应该是下一个可用的 PEP 编号，未被已发布或在 PR 中的 PEP 使用。

- 在"PEP:"标题字段中，输入与您的文件名匹配的 PEP 编号作为您的草稿 PEP 编号。

- 在"Type:"标题字段中，输入"Standards Track"、"Informational"或"Process"，对于"Status:"字段输入"Draft"。完整详情，请参阅 PEP 标题前置部分。

- 更新 .github/CODEOWNERS，使任何对 PEP 存储库具有写入权限的合著者或赞助人都列在您的新文件中。这确保了任何更改该文件的未来拉取请求都会被分配给他们。

- 将此推送到您的 GitHub 分叉并提交拉取请求。

- PEP 编辑审查您的 PR 的结构、格式和其他错误。对于 reST 格式的 PEP，PEP 12 提供了一个模板。它还提供了在 PEP 中使用的 reST 标记的完整介绍。审批标准是：它健全且完整。这些想法必须在技术上讲得通。编辑不考虑它们是否可能被接受。

- 标题准确描述内容。

- PEP 的语言（拼写、语法、句子结构等）和代码风格（示例应符合 PEP 7 和 PEP 8）应该正确且符合规范。PEP 文本将在提交拉取请求时自动检查正确的 reStructuredText 格式。具有无效 reST 标记的 PEP 将不会被批准。

编辑通常对此初步审查相当宽容，预期问题会在审查过程中得到纠正。注意：PEP 的批准并不能保证没有令人尴尬的错误！正确性是作者和审查者的责任，而不是编辑的责任。

如果 PEP 尚未准备好批准，编辑会将其退回给作者进行修订，并附上具体说明。

- 一旦获得批准，他们将为您分配一个 PEP 编号。

一旦审查流程完成，并且 PEP 编辑批准它（注意这与接受您的 PEP 不同！），他们将会话合并您的拉取请求到主分支。

PEP 编辑不会无理拒绝发布 PEP。拒绝 PEP 状态的理由包括重复劳动、技术上不合理、未提供适当的动机或未解决向后兼容性问题，或不符合 Python 哲学。指导委员会可以在审批阶段被咨询，并且是草稿 PEP 可行性的最终裁决者。

对 PEP 存储库具有写入权限的开发人员可以通过创建并提交新 PEP 直接申请 PEP 编号。这样做时，开发人员必须处理通常由 PEP 编辑处理的任务（参见 PEP 编辑职责与工作流程）。这包括确保初始版本符合提交 PEP 的预期标准。或者，即使是开发人员也应该通过拉取请求提交 PEP。这样做时，通常预期您自己处理流程；如果需要 PEP 编辑的帮助，在 GitHub 上提及 @python/pep-editors。

随着更新的需要，PEP 作者可以签入新版本，如果他们（或协作开发人员）对 PEP 存储库具有写入权限。早期分配 PEP 编号对于引用方便可能很有用，特别是当同时考虑多个草稿 PEP 时。

标准轨道 PEP 包括两部分：设计文档和参考实现。通常建议至少在原型实现与 PEP 共同开发，因为原则上听起来不错的想法在经受实现测试时可能会变得不切实际。

### 讨论 PEP

一旦分配了 PEP 编号并且草稿 PEP 已提交到 PEP 存储库，就应该为 PEP 创建一个讨论线程，以提供一个集中讨论和审查其内容的地方，并且 PEP 应该更新，使 Discussions-To 标题链接到该线程。

PEP 作者（或赞助人，如果适用）可以选择任何合理的场所进行讨论，只要满足以下标准：

- 论坛适合 PEP 的主题。

- 线程在网络上公开可用，以便所有感兴趣各方都能参与。

- 讨论受 Python 社区行为准则约束。

- 在 PEP 的 Discussions-To 标题下提供指向当前讨论线程的直接链接。

Python Discourse 的 PEPs 类别是大多数新 PEP 的首选选择，而历史上 Python-Dev 邮件列表常被使用。某些专业主题有特定场所，例如 Python Discourse 上的 Typing 类别和 Packaging 类别分别用于类型和打包 PEP。如果 PEP 作者不确定最佳场所，PEP 赞助人和 PEP 编辑可以相应提供建议。

如果 PEP 经历重大重写或其他重大实质性更改其提出的规范，通常应该在选定的场所创建新线程以征求更多反馈。如果发生这种情况，必须更新 Discussions-To 链接并添加新的 Post-History 条目指向这个新线程。

如果未选作讨论场所，则应在 PEPs 类别发布简短公告帖子，至少包含指向渲染后的 PEP 和 Discussions-To 线程的链接，当草稿 PEP 提交到存储库时，以及如果发生重大更改触发新线程时。

PEP 作者负责在提交 PEP 审查之前收集社区对 PEP 的反馈。然而，为了避免冗长和无休止的讨论，应考虑诸如在早期设计阶段征求私人或更有针对性的反馈、与在 PEP 主题方面有专业知识的其他社区成员协作，以及为 PEP 主题选择适当专业化的讨论（如果适用）等策略。PEP 作者应在此处自行判断。

一旦 PEP 被分配编号并提交到 PEP 存储库，实质性问题通常应该在规范公共线程上讨论，而不是私人渠道、GitHub 拉取请求审查或不相关场所。这确保每个人都能关注和贡献，避免讨论碎片化，并确保它作为 PEP 审查过程的一部分得到充分考虑。此指定线程上的评论、支持、关注和其他反馈是指导委员会或 PEP-Delegate 审查 PEP 时将考虑的关键部分。

### PEP 审查与决议

一旦作者完成了 PEP，他们可以从 PEP 编辑处请求审查风格和一致性。然而，内容审查和 PEP 接受最终是指导委员会的责任，这通过打开指导委员会问题正式发起，一旦作者（和赞助人，如果有的话）确定 PEP 准备好最终审查和决议。

为了在选定情况下加快流程（例如，当更改明显有益且准备接受，但 PEP 尚未正式提交审查时），指导委员会也可以发起 PEP 审查，首先通知 PEP 作者并给他们修改的机会。

PEP 批准的最高权威是指导委员会。然而，每当提出新 PEP 时，任何认为自己有足够的经验对该 PEP 做出最终决定的核心开发人员都可以提议担任其 PEP-Delegate，通过通知指导委员会他们的意图。如果指导委员会批准他们的提议，PEP-Delegate 然后将有权批准或拒绝该 PEP。对于与 Python 类型系统相关的 PEP，类型委员会（PEP 729）向指导委员会提供建议。要请求此类建议，在类型委员会问题跟踪器上打开问题。

术语"PEP-Delegate"用于指导委员会治理模型下 PEP 的指定决策者，其记录在 PEP 标题的"PEP-Delegate"字段中。术语"BDFL-Delegate"是 PEP-Delegate 的已弃用别名，是 Python 由 BDFL 领导的时代的遗留物。任何对"BDFL-Delegate"的遗留引用应视为等同于"PEP-Delegate"。

个人提议自任 PEP-Delegate 必须通知相关作者和（如果存在）PEP 的赞助人，并向指导委员会提交请求（可以通过新问题完成）。承担此责任的人可以随时从指导委员会寻求额外指导，并期望考虑其他核心开发人员的建议和观点。

指导委员会通常会默认批准此类自我提名，但也可以选择拒绝它们。指导委员会拒绝自我提名为 PEP-Delegate 的可能原因包括但不限于感知到潜在的利益冲突（例如，与 PEP 提交者在同一家组织工作），或简单地认为另一个潜在 PEP-Delegate 更合适。如果核心开发人员（或其他社区成员）对任何给定 PEP 的 PEP-Delegate 的适宜性有顾虑，他们可以请求指导委员会审查委派。

如果没有志愿者站出来，指导委员会将接触具有相关专业知识的核心开发人员（以及潜在的 Python 社区其他成员），试图找到一个愿意担任该 PEP 的 PEP-Delegate 的候选人。如果找不到合适的候选人，则 PEP 将被标记为"Deferred"（延期），直到有一个可用。

先前任命的 PEP-Delegate 可能选择辞职，或被委员会要求辞职，在这种情况下，将以与新 PEP 相同的方式任命新的 PEP-Delegate（包括如果没有合适的替代者则延期 PEP）。如果 PEP-Delegate 被要求辞职，这将推翻任何先前的 PEP 接受或拒绝，它将恢复到"Draft"（草稿）状态。

当此类常设委派到位时，指导委员会将保持足够的公开记录，以允许后续委员会、核心开发人员和更广泛的 Python 社区理解当前存在的委派、它们为何被设置，以及它们可能不再需要的情况。

要接受 PEP，它必须满足某些最低标准。它必须是所提出增强的清晰完整描述。该增强必须代表净改进。所提出的实现（如果适用）必须稳固，并且不得使解释器过于复杂。最后，所提出的增强必须"pythonic"才能被指导委员会接受。（然而，"pythonic"是一个不精确的术语；它可以被定义为指导委员会可接受的任何内容。这种逻辑是故意循环的。）有关标准库模块接受标准，请参阅 PEP 2。

除指导委员会另有批准外，PEP 决议的公告将发布到 Python Discourse 的 PEPs 类别。

一旦 PEP 被接受，必须完成参考实现。当参考实现完成并合并到主源代码存储库时，状态将更改为"Final"（最终）。

为了在承诺语言功能或标准库 API 的长期稳定性之前收集额外的设计和接口反馈，PEP 也可以标记为"Provisional"（临时）。这代表"临时接受"，表示提案已被接受纳入参考实现，但在完整设计可以被认为"最终"之前需要额外的用户反馈。与普通接受的 PEP 不同，临时接受的 PEP 即使相关更改已包含在 Python 版本后仍可能被"Rejected"（拒绝）或"Withdrawn"（撤回）。

尽可能，认为缩小提案范围以避免依赖"Provisional"状态更可取（例如，通过将某些功能推迟到后续 PEP），因为此状态可能导致更广泛 Python 生态系统中的版本兼容性问题。PEP 411 提供了关于 Provisional 状态潜在用例的更多详细信息。

PEP 也可以被分配"Deferred"（延期）状态。当 PEP 没有进展时，PEP 作者或编辑可以为此 PEP 分配此状态。一旦 PEP 被延期，PEP 编辑可以将其重新分配为草稿状态。

PEP 也可以被"Rejected"（拒绝）。也许经过所有讨论后发现它不是一个好主意。记录这一事实仍然很重要。"Withdrawn"（撤回）状态类似——它意味着 PEP 作者自己决定 PEP 实际上是个坏主意，或者接受了竞争提案是更好的替代方案。

当 PEP 被 Accepted、Rejected 或 Withdrawn 时，PEP 应相应更新。除了更新 Status 字段外，至少要添加 Resolution 标题，并直接链接到做出 PEP 决策的相关帖子。

PEP 也可以被不同的 PEP superseded（取代），使原来的过时。这适用于信息性 PEP，其中 API 的 v2 版本可以替换 v1 版本。

### PEP 维护

通常，PEP 在达到 Accepted、Final、Rejected 或 Superseded 状态后不再进行实质性修改。一旦达成决议，PEP 被视为历史文档而非活动规范。预期行为的正式文档应在其他地方维护，例如核心功能的语言参考、标准库模块的库参考或打包的 PyPA 规范。

如果基于实现经验和用户反馈对 Standards track PEP 进行更改，而它们处于 Provisional 或（经 SC 批准）Accepted 状态，它们应该在 PEP 中注明，以便 PEP 准确描述其被标记为 Final 时的实现。

Active（Informational 和 Process）PEP 可能会随着时间更新以反映开发实践和其他细节的更改。这些情况下遵循的确切流程将取决于问题 PEP 的性质和目的。

偶尔，Deferred 甚至 Withdrawn PEP 可能会通过重大更新复活，但通常最好直接提出新的 PEP。

## 成功的 PEP 应该包含什么？

每个 PEP 应包含以下部分/章节：

- Preamble（前置部分）– RFC 2822 样式的标题，包含关于 PEP 的元数据，包括 PEP 编号、简短描述性标题（最多 44 个字符）、每个作者的姓名，以及可选的联系信息等。

- Abstract（摘要）– 关于所解决技术问题的简短描述（约 200 字）。

- Motivation（动机）– 动机对于想要更改 Python 语言、库或生态系统的 PEP 至关重要。它应清楚解释为什么现有语言规范不足以解决 PEP 解决的问题。这可以包括从 Python 生态系统中的重要项目收集对 PEP 的支持文件。没有足够动机的 PEP 提交可能会被拒绝。

- Specification（规范）– 技术规范应描述任何新语言功能的语法和语义。规范应足够详细，以允许至少当前主要 Python 平台（CPython、Jython、IronPython、PyPy）的竞争、可互操作实现。

- Rationale（理由）– 理由通过描述为何做出特定设计决定来充实规范。它应描述被考虑的其他设计和相关工作，例如其他语言如何支持该功能。理由应提供社区内共识的证据，并讨论讨论期间提出的重要反对意见或关注。

- Backwards Compatibility（向后兼容性）– 所有引入向后不兼容性的 PEP 必须包括描述这些不兼容性及其严重程度的部分。PEP 必须解释作者打算如何处理这些不兼容性。没有足够向后兼容性论述的 PEP 提交可能被直接拒绝。

- Security Implications（安全影响）– 如果与 PEP 相关存在安全顾虑，这些顾虑应明确写出，以确保 PEP 审查者意识到它们。

- How to Teach This（如何教授此内容）– 对于添加新功能或更改语言行为的 PEP，包含如何教用户（新的和有经验的）如何将 PEP 应用于他们工作的部分很有帮助。本节可能包括关键点和推荐的文档更改，以帮助用户采用新功能或将代码迁移以使用语言更改。

- Reference Implementation（参考实现）– 参考实现必须在任何 PEP 被赋予"Final"状态前完成，但不需要在 PEP 被接受前完成。虽然在编写代码之前在规范和理由上达成共识的方法有其价值，但"大致共识和运行代码"的原则在解决许多 API 细节讨论时仍然有用。最终实现必须包括测试代码和适用于 Python 语言参考或标准库参考的文档。

- Rejected Ideas（被拒绝的想法）– 在 PEP 讨论过程中，会提出各种未被接受的想法。这些被拒绝的想法应该连同被拒绝的理由一起记录。这既有助于记录最终版本 PEP 背后的思考过程，也防止人们在后续讨论中再次提出相同的被拒绝想法。在某种程度上，这个部分可以被视为 Rationale 部分的分支部分，专门关注为何某些想法最终未被追求。

- Open Issues（未决问题）– 当 PEP 处于草稿状态时，可能会出现需要进一步讨论的想法。这些想法应该被记录下来，以便人们知道它们正在被思考但没有具体解决方案。这有助于确保 PEP 准备考虑所需的所有问题都已完成，并减少人们重复之前的讨论。

- Acknowledgements（致谢）– 用于感谢和承认帮助开发、讨论或起草 PEP 的人，或任何其他目的。该节可用于认可不是合著者的工作贡献者。

- Footnotes（脚注）– PEP 中引用的脚注集合，以及列出非内联超链接目标的地方。


- Change History (变更历史) – 基于讨论和反馈，对 PEP 经历的主要变更进行总结。您可以将其视为 PEP 的“变更日志”或“发布说明”。

通常，每当您更新“发布历史”标题以显示主要变更时，

请按最新在前（即时间倒序）的顺序添加新的项目符号，格式为 DD-MMM-YYYY，

并使用子项目符号总结变更。您可以考虑将其链接到

与“发布历史”链接相同的链接。这不是强制性的，

由 PEP 作者自行决定，但这样的部分有助于关注者

了解您的 PEP 的演变过程。

此处提供一个

[示例](../pep-0694/#change-history)。

- 版权/许可 – 每个新的 PEP 都必须采用双重许可：

公共领域和 CC0-1.0-Universal（参见此 PEP 示例）。

## PEP 格式和模板

PEP 文件是使用 reStructuredText 格式的 UTF-8 编码文本文件。

reStructuredText 允许使用丰富的标记，同时保持良好的可读性，并生成美观且功能完善的 HTML 代码。PEP 12

包含相关说明和 PEP 模板。

PEP 文本文件会自动

转换为 HTML

（通过基于 Sphinx 的构建系统）

以便于在线阅读。

## PEP 头部导言

每个 PEP 文件都必须以符合 RFC 2822 标准的头部导言开头。头部
必须按以下顺序出现。标有“*”的头部是

可选的，将在下文描述。所有其他头部都是必需的。

```
  PEP: <pep number>
  Title: <pep title>
  Author: <list of authors' names and optionally, email addrs>
* Sponsor: <name of sponsor>
* PEP-Delegate: <PEP delegate's name>
  Discussions-To: <URL of current canonical discussion thread>
  Status: <Draft | Active | Accepted | Provisional | Deferred | Rejected |
           Withdrawn | Final | Superseded>
  Type: <Standards Track | Informational | Process>
* Topic: <Governance | Packaging | Release | Typing>
* Requires: <pep numbers>
  Created: <date created on, in dd-mmm-yyyy format>
* Python-Version: <version number>
  Post-History: <dates, in dd-mmm-yyyy format,
                 inline-linked to PEP discussion threads>
* Replaces: <pep number>
* Superseded-By: <pep number>
* Resolution: <date in dd-mmm-yyyy format, linked to the acceptance/rejection post>
```

作者头列出了 PEP 所有作者/所有者的姓名，以及可选的电子邮件地址。

作者头的格式

值必须为：

```Random J. User <random@example.com>```

如果包含电子邮件地址，则为：

```Random J. User```

如果未提供地址。大多数 PEP 作者都使用真实姓名，但

如果您更喜欢使用其他姓名，并在与 PEP 相关的讨论中始终使用该姓名，

请随意在此处使用。

如果有多位作者，每位作者应单独占一行，

遵循 RFC 2822 的续行约定。请注意，PEP 中的个人

电子邮件地址将被隐藏，以防止垃圾邮件收集器

收集。

“赞助人”字段记录哪个开发者（核心开发者或其他经指导委员会批准的开发者）赞助了该 PEP。

如果 PEP 的作者之一是
核心开发者，则无需赞助人，因此应省略此字段。



“PEP 代表”字段用于记录由
指导委员会指定的最终决定是否批准或

拒绝 PEP 的个人。

注意：仅标准跟踪 PEP 需要“决议”标头。

它包含一个 URL，该 URL 应指向电子邮件或其他网络资源，其中发布了关于 PEP 的声明（即批准或拒绝）。

Discussions-To 标头提供 PEP 当前权威讨论主题的 URL。

对于电子邮件列表，这应该是指向列表存档中该主题的直接链接，而不是 mailto: 或指向列表本身的超链接。

Type 标头指定 PEP 的类型：标准跟踪、信息或流程。

可选的 Topic 标头列出 PEP 所属的特定主题（如有）。

有关现有主题，请参阅主题索引。

Created 标头记录 PEP 被分配编号的日期，

而 Post-History 用于记录 PEP 的 Discussions-To 主题的日期和相应的 URL，

前者作为链接文本，后者作为链接目标。

两组日期都应采用 dd-mmm-yyyy 格式，例如 14-Aug-2001。

标准跟踪 PEP 通常会有一个 Python-Version 标头，

该标头指示该功能将随 Python 版本一起发布。

没有 Python-Version 标头的标准跟踪 PEP 表示
互操作性标准，这些标准最初将通过

外部库和工具提供支持，然后可能会由后续的 PEP 补充，

以添加对标准库的支持。信息性和流程 PEP 不需要 Python-Version 标头。

PEP 可能有一个 Requires 标头，指示该 PEP 所依赖的 PEP 编号。

PEP 也可能有一个 Superseded-By 标头，指示某个 PEP 已被
后续文档取代；该值是
P 的编号。
取代当前文档的EP。新的PEP必须包含一个

Replaces标头，其中包含被其取代的PEP编号。



### 辅助文件

PEP可以包含辅助文件，例如图表。此类文件应命名为
pep-XXXX-Y.ext，其中“XXXX”是PEP编号，“Y”是

序列号（从1开始），“ext”替换为实际的文件扩展名（例如“png”）。

或者，所有支持文件可以放在名为
pep-XXXX 的子目录中，其中“XXXX”是PEP编号。使用子目录时，
对文件名没有限制。

## 修改现有PEP

PEP草案在提交指导委员会或

PEP代表审查和决议之前，可自由讨论和提出修改建议，

作者可自行决定。实质性内容变更通常应首先在 PEP 的讨论帖中提出，该讨论帖列于 PEP 的

Discussions-To 标题下；而文字编辑和更正可以提交为
GitHub issue 或 GitHub pull request。

拥有 PEP 仓库写入权限的 PEP 作者可以通过

使用 git push 或 GitHub PR 提交更改来更新 PEP。

有关修改其他 PEP 的指导，请参阅 PEP 维护部分。

更多详情请参阅贡献指南，如有疑问，
请先咨询 PEP 作者和/或 PEP 编辑。

## PEP 所有权转移

有时需要将 PEP 的所有权转移给

新的负责人。通常，最好保留原作者作为
新 PEP 的共同作者，但这最终取决于

原作者的意愿。转移所有权的一个合理理由是：

原作者不再有时间或兴趣更新它，

不再参与 PEP 流程，或者已经销声匿迹（例如，无法联系或不回复邮件）。一个不恰当的转移所有权的理由是：

作者不同意 PEP 的方向。PEP 流程的目标之一是尝试围绕 PEP 达成共识，

但如果无法达成共识，作者可以随时提交一个与之竞争的 PEP。

如果您有兴趣获得某个 PEP 的所有权，您也可以通过以下方式实现：

提交 pull request。fork PEP 仓库，修改您的所有权，

然后提交 pull request。您应该在 pull request 的评论中同时提及原作者和

#python/pep-editors （如果原作者的 GitHub 用户名未知，请使用电子邮件。）如果原作者未能及时回复，PEP 编辑将做出单方面决定（当然，这样的决定并非不可撤销）。

## PEP 编辑职责及工作流程

PEP 编辑必须被添加到 GitHub 上的 `python/pep-editors` 群组，并且必须关注 PEP 代码库。

请注意，拥有 PEP 代码库写入权限的开发者可以处理通常由 PEP 编辑负责的任务。

此外，开发者也可以通过在 GitHub 上提及 `python/pep-editors` 群组来请求 PEP 编辑的帮助。

对于编辑器收到的每个新 PEP，执行以下操作：

- 确保该 PEP 由核心开发人员共同撰写，或由核心开发人员作为发起人，或由指导委员会专门批准的发起人。

- 阅读 PEP 以检查其是否准备就绪：内容合理且完整。即使某些想法看起来不太可能被采纳，也必须在技术上可行。

- 标题应准确描述内容。

- 文件扩展名正确（例如 .rst）。

- 确保所有被列为 PEP 发起人或共同作者且拥有 PEP 代码库写入权限的人员都已添加到 .github/CODEOWNERS 列表中。

- 浏览 PEP，检查是否存在明显的语言（拼写、语法、句子结构等）和代码风格缺陷（示例应符合 PEP 7 和 PEP 8）。编辑可以自行修正问题，但并非必须这样做（reStructuredText 语法由仓库的持续集成系统 (CI) 检查）。

- 如果某个项目被描述为受益于或支持 PEP，请确保包含来自该项目的直接说明，以明确其支持。这样做是为了避免 PEP 无意中将某个项目描述为支持该 PEP，而实际上这种支持只是基于推测。

如果 PEP 尚未准备就绪，编辑会将其退回给作者进行修改，并提供具体说明。如果 reST 格式存在问题，请作者使用 PEP 12 作为模板重新提交。

PEP 准备好提交到代码库后，PEP 编辑将执行以下操作：

- 检查作者是否已选择有效的 PEP 编号，如果没有，则为其分配一个编号（通常情况下，会分配下一个可用的编号，但

有时会分配一个特殊/玩笑编号，例如 666 或 3141）。请记住，编号小于 100 的编号是元 PEP。

- 检查作者是否已正确标记 PEP 的类型（“标准跟踪”、“信息性”或“流程”），

并将其状态标记为“草案”。

- 确保所有 CI 构建和 lint 检查均无错误通过，

并且没有明显的错误。

渲染预览输出中存在的问题。

- 合并新的（或更新的）PEP。

- 通知作者后续步骤（例如，发起讨论帖并

更新 PEP，发布公告等）。

对现有 PEP 的更新应以 GitHub pull request 的形式提交。

许多 PEP 由拥有 Python 代码库写入权限的开发者编写和维护。

PEP 编辑人员会监控 PEP 仓库的变更，
并纠正他们发现的任何结构、语法、拼写或

标记错误。

PEP 编辑人员不对 PEP 进行评判。他们仅负责

管理和编辑工作（通常工作量较小）。
