# PEP 3106 – 重新设计 dict.keys()、.values() 和 .items()

**元数据：**

- **作者**：Guido van Rossum
- **创建时间**：2006 年 12 月 19 日
- **Python 版本**：3.0

## 摘要

本 PEP 提议修改内置 dict 类型的 .keys()、.values() 和 .items() 方法，使其返回一个类似集合或无序的容器对象，该对象的内容源自底层字典，而不是返回键等的副本列表；并移除 .iterkeys()、.itervalues() 和 .iteritems() 方法。

该方法的灵感来自 Java Collections Framework [1]。

## 介绍

长期以来一直计划修改内置 dict 类型的 .keys()、.values() 和 .items() 方法，使其返回比列表更轻量级的对象，并摆脱 .iterkeys()、.itervalues() 和 .iteritems()。其想法是，当前（在 2.x 中）读取如下代码：

```python
for k, v in d.iteritems(): ...
```

应重写为：

```python
for k, v in d.items(): ...
```

（.itervalues() 和 .iterkeys() 也类似，只是后者是多余的，因为我们可以将该循环写为 for k in d。）

当前读取如下代码：

```python
a = d.keys() # 假设我们真的想要一个列表
```

（等等）应重写为

```python
a = list(d.keys())
```

有（至少）两种方法可以实现这一点。最初的计划是简单地让 .keys()、.values() 和 .items() 返回一个迭代器，即在 Python 2.x 中 iterkeys()、itervalues() 和 iteritems() 返回的内容。然而，Java Collections Framework [1] 表明有更好的解决方案：这些方法返回具有集合行为（对于 .keys() 和 .items()）或多集合（== 多重集）行为（对于 .values()）的对象，这些对象不包含键、值或项的副本，而是引用底层字典并在需要时从字典中获取值。

这种方法的优势在于，人们仍然可以编写如下代码：

```python
a = d.items()
for k, v in a: ...
# 稍后再次：
for k, v in a: ...
```

实际上，Python 3.0 中的 iter(d.keys())（等等）将执行 Python 2.x 中 d.iterkeys()（等等）的操作；但在大多数情况下，我们不必编写 iter() 调用，因为 for 循环隐含了它。

.keys() 和 .items() 方法返回的对象表现得像集合。.values() 方法返回的对象表现得像一个更简单的无序集合——它不能是集合，因为可能存在重复值。

由于集合行为，可以通过以下方式检查两个字典是否具有相同的键：

```python
if a.keys() == b.keys(): ...
```

.items() 也类似。

这些操作仅在线程不安全的方式使用它们可能会导致异常但不会导致内部表示损坏的程度上是线程安全的。

与 Python 2.x 一样，在使用迭代器迭代字典的同时修改它会产生未定义的影响，并且在大多数情况下会引发 RuntimeError 异常。（这与 Java Collections Framework 提供的保证类似。）

.keys() 和 .items() 返回的对象与内置 set 和 frozenset 类型的实例完全兼容；例如：

set(d.keys()) == d.keys()

保证为 True（除非 d 正被另一个线程同时修改）。

## 规范

我使用伪代码来指定语义：

```python
class dict:

    # 为简洁起见省略所有其他 dict 方法。
    # .iterkeys()、.itervalues() 和 .iteritems() 方法
    # 将被移除。

    def keys(self):
        return d_keys(self)

    def items(self):
        return d_items(self)

    def values(self):
        return d_values(self)

class d_keys:

    def __init__(self, d):
        self.__d = d

    def __len__(self):
        return len(self.__d)

    def __contains__(self, key):
        return key in self.__d

    def __iter__(self):
        for key in self.__d:
            yield key

    # 应实现以下操作以与集合兼容；
    # 可以通过利用上述基本操作来完成：
    #
    # <, <=, ==, !=, >=, >（返回 bool）
    # &, |, ^, -（返回一个新的、真实的集合对象）
    #
    # 以及它们的方法对应物（.union() 等）。
    #
    # 为了指定语义，我们可以指定 x == y 为：
    #
    # set(x) == set(y) 如果 x 和 y 都是 d_keys 实例
    # set(x) == y 如果 x 是 d_keys 实例
    # x == set(y) 如果 y 是 d_keys 实例
    #
    # 以及其他所有操作。

class d_items:

    def __init__(self, d):
        self.__d = d

    def __len__(self):
        return len(self.__d)

    def __contains__(self, (key, value)):
        return key in self.__d and self.__d[key] == value

    def __iter__(self):
        for key in self.__d:
            yield key, self.__d[key]

    # 以及上面提到的 d_keys 的集合操作。
    # 然而，那里建议的规范在值不可哈希时将不起作用。
    # 幸运的是，这些操作仍然可以高效实现。例如，这是
    # 交集的规范：

    def __and__(self, other):
        if isinstance(other, (set, frozenset, d_keys)):
            result = set()
            for item in other:
                if item in self:
                    result.add(item)
            return result
        if not isinstance(other, d_items):
            return NotImplemented
        d = {}
        if len(other) < len(self):
            self, other = other, self
        for item in self:
            if item in other:
                key, value = item
                d[key] = value
        return d.items()

    # 这里是相等性：

    def __eq__(self, other):
        if isinstance(other, (set, frozenset, d_keys)):
            if len(self) != len(other):
                return False
            for item in other:
                if item not in self:
                    return False
            return True
        if not isinstance(other, d_items):
            return NotImplemented
        # XXX 我们也可以只比较底层字典……
        if len(self) != len(other):
            return False
        for item in self:
            if item not in other:
                return False
        return True

    def __ne__(self, other):
        # XXX 也许 object.__ne__() 应该这样定义。
        result = self.__eq__(other)
        if result is not NotImplemented:
            result = not result
        return result

class d_values:

    def __init__(self, d):
        self.__d = d

    def __len__(self):
        return len(self.__d)

    def __contains__(self, value):
        # 这很慢，如果 __contains__ 未定义，
        # "x in y" 会将其用作后备；但我宁愿明确
        # 支持它。
        for v in self:
             if v == value:
                 return True
        return False

    def __iter__(self):
        for key in self.__d:
            yield self.__d[key]

    def __eq__(self, other):
        if not isinstance(other, d_values):
            return NotImplemented
        if len(self) != len(other):
            return False
        # XXX 有时这可以优化，但这是语义：
        # 我们不能依赖值是哈希或可比较的。
        olist = list(other)
        for x in self:
            try:
                olist.remove(x)
            except ValueError:
                return False
        assert olist == []
        return True

    def __ne__(self, other):
        result = self.__eq__(other)
        if result is not NotImplemented:
            result = not result
        return result
```

注释：

视图对象不是直接可变的，但不实现 `__hash__()`；如果底层字典被修改，其值可能会改变。

对底层字典的唯一要求是它实现 `__getitem__()`、`__contains__()`、`__iter__()` 和 `__len__()`。

我们不实现 .copy() – .copy() 方法的存在表明副本与原始类型相同，但如果不复制底层字典，这是不可行的。如果您想要特定类型的副本，如列表或集合，只需将上述之一传递给 list() 或 set() 构造函数。

规范意味着 .keys()、.values() 和 .items() 返回项的顺序是相同的（就像在 Python 2.x 中一样），因为顺序都是从字典迭代器派生的（这显然是任意的，但只要字典未被修改，它就是稳定的）。这可以通过以下不变式来表达：

```python
list(d.items()) == list(zip(d.keys(), d.values()))
```

## 未决问题

我们需要更多的动机吗？我认为能够对键和项执行集合操作而无需复制它们应该不言而喻。

我略去了各种集合操作的实现。这些操作仍可能带来一些小惊喜。

多次调用 d.keys()（等等）返回相同的对象是可以的，因为该对象的唯一状态是它所引用的字典。这值得在字典对象中拥有额外的槽位吗？那应该是一个弱引用，还是 d_keys（等等）对象一旦创建就永远存在？临时方案：可能不值得在每个字典中拥有额外的槽位。

d_keys、d_values 和 d_items 是否应该有一个公共实例变量或方法，通过它可以检索底层字典？临时方案：是（但应该叫什么？）。

我正在征集比 d_keys、d_values 和 d_items 更好的名称。这些类可以是公开的，以便其他映射的 .keys()、.values() 和 .items() 方法可以重用它们的实现。或者它们应该这样吗？

d_keys、d_values 和 d_items 类是否应该可重用？临时方案：是。

它们是否应该可子类化？临时方案：是（但请见下文）。

一个特别棘手的问题是，是否以其他操作（例如 .discard()）指定的操作必须真正以这些其他操作来实现；这似乎无关紧要，但如果这些类被子类化，它就会变得相关。历史上，Python 在明确指定高度优化的内置类型语义方面记录很差；我的临时方案是继续这一趋势。子类化可能仍然有助于添加新方法。

我将把决定（特别是关于命名）交给提交工作实现的人。
