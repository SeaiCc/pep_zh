**元数据：**

## 添加条件表达式

在 2005 年 9 月 29 日，Guido 决定以"X if C else Y"的形式添加条件表达式。[1]

推动这一决定的用例是普遍存在的、容易出错的尝试使用"and"和"or"来实现相同效果。[2]

之前社区努力添加条件表达式因无法就最佳语法达成共识而受阻。该问题通过简单地委托给 BDFL 的最佳判断得到了解决。

通过审查语法在整个标准库中的应用情况来验证这一决定（此次审查近似于对真实世界用例的采样，涵盖多种应用程序，由不同背景的程序员编写）。[3]

以下更改将应用于语法。（or_test 符号是新的，其他是修改的。）

```
test: or_test ['if' or_test 'else' test] | lambdef
or_test: and_test ('or' and_test)*
...
testlist_safe: or_test [(',' or_test)+ [',']]
...
gen_for: 'for' exprlist 'in' or_test [gen_iter]
```

新语法几乎引入了一个微小的语法向后不兼容。在以前的 Python 版本中，以下是合法的：

```python
[f for f in lambda x: x, lambda x: x**2 if f(1) == 1]
```

（即一个列表推导式，其中'in'后面的序列是一系列未加括号的 lambda——或者甚至只是一个 lambda。）

在 Python 3.0 中，lambda 序列必须加括号，例如：

```python
[f for f in (lambda x: x, lambda x: x**2) if f(1) == 1]
```

这是因为 lambda 的绑定比 if-else 表达式更松散，但在这种上下文中，lambda 后面已经可以跟一个绑定更松散的'if'关键字（详情考虑上面显示的语法更改）。

然而，在 Python 2.5 中，使用了一个略有不同的语法，它更向后兼容，但限制了用于此位置的 lambda 的语法，禁止 lambda 的体包含未加括号的条件表达式。示例：

```python
[f for f in (1, lambda x: x if x >= 0 else -1)]  # 可以
[f for f in 1, (lambda x: x if x >= 0 else -1)]  # 可以
[f for f in 1, lambda x: (x if x >= 0 else -1)]  # 可以
[f for f in 1, lambda x: x if x >= 0 else -1]    # 无效
```

## PEP 早期草案的介绍（保留供历史目的）

对 if-then-else（"三元"）表达式的需求不断在 comp.lang.python 上出现。本 PEP 包含了一个相当 Pythonic 语法的具体提议。这是社区的唯一机会：如果本 PEP 以明显多数获得批准，它将在 Python 2.4 中实现。如果没有，PEP 将增加被拒绝原因的摘要，主题最好不要再提。虽然 BDFL 是本 PEP 的合著者，但他既不赞成也不反对这个提议；由社区决定。如果社区无法决定，BDFL 将拒绝该 PEP。

在史无前例的社区响应后（支持和反对的论据都非常有力），本 PEP 在 Raymond Hettinger 的帮助下进行了修订。没有经过完整的修订历史，主要变化是不同的建议语法、提议替代方案的概述、当前讨论的状态以及对短路行为的讨论。

经过讨论后，进行了一次投票。虽然总体上对某种形式的 if-then-else 表达式感兴趣，但没有一种格式能够获得多数支持。因此，由于缺乏压倒性多数的改变，PEP 被拒绝。此外，Python 设计原则之一是在对采取哪条路径有疑问时倾向于维持现状。

## 提议

建议的语法如下：

```python
(if <condition>: <expression1> else: <expression2>)
```

注意外层的括号不是可选的。

结果表达式的评估方式如下：

- 首先，评估`<condition>`。

- 如果`<condition>`为真，评估`<expression1>`，它是整个表达式的结果。

- 如果`<condition>`为假，评估`<expression2>`，它是整个表达式的结果。

此语法的自然扩展是允许一个或多个'elif'部分：

```python
(if <cond1>: <expr1> elif <cond2>: <expr2> ... else: <exprN>)
```

如果提议被接受，这将得到实现。

提议的缺点是：

- 必需的括号

- 可能与语句语法混淆

- 冒号的额外语义负载

注意<expression1>和<expression2>中最多只有一个被评估。这被称为"短路表达式"；它类似于'and'/'or'的第二个操作数仅在第一个操作数为真/假时才被评估的方式。

模拟 if-then-else 表达式的一种常见方式是：

```python
<condition> and <expression1> or <expression2>
```

然而，这工作方式不同：当`<expression1>`为假时它返回`<expression2>`！参见 FAQ 4.16 了解有效的方法——然而，它们相当丑陋且需要更多努力来理解。

## 替代方案

Holger Krekel 提出了一种新的、最小侵入性的变体：

```python
<condition> and <expression1> else <expression2>
```

其背后的概念是，使用 and/or 已经存在一个几乎完整的三元运算符，这个提议是使其完整的最低侵入性更改。许多新闻组响应者认为这是最令人心动的替代方案。然而，一些响应者能够发布心理上难以解析的示例。后来有人指出，这个结构的工作方式是让"else"改变"and"的现有含义。

结果，Christian Tismer 提出的相同想法的变体获得了越来越多的支持：

```python
<condition> then <expression1> else <expression2>
```

优点是简单的视觉解析、不需要括号、不改变现有关键字的语义、不像提议那样容易与语句语法混淆，并且不会进一步过载冒号。缺点是引入新关键字的实现成本。然而，与其他新关键字不同，"then"这个词似乎不太可能在现有程序中被用作名称。

---

许多源自 C 的语言使用此语法：

```python
<condition> ? <expression1> : <expression2>
```

Eric Raymond 甚至实现了这一点。BDFL 出于几个原因拒绝了这一点：冒号在 Python 中已经有很多用途（尽管实际上它不会模棱两可，因为问号需要一个匹配的冒号）；对于不习惯源自 C 语言的人来说，它很难理解。

---

本 PEP 的原始版本建议了以下语法：

```python
<expression1> if <condition> else <expression2>
```

发现这种乱序排列对许多讨论参与者来说太不舒服；特别是当<expression1>很长时，快速浏览时很容易错过条件。

---

有些人建议添加一个新的内置函数，而不是扩展语言的语法。例如：

```python
cond(<condition>, <expression1>, <expression2>)
```

这不会像语法扩展那样工作，因为 expression1 和 expression2 必须在调用函数之前被评估。没有办法短路表达式评估。如果'cond'（或其他名称）被制成关键字，它可以工作，但这具有添加新关键字的所有缺点，再加上令人困惑的语法：它看起来像函数调用，所以随意读者可能期望<expression1>和<expression2>都被评估。

## 当前讨论状态摘要

群体分为三个阵营之一：

- 采用使用标点符号构建的三元运算符：

```python
<condition> ? <expression1> : <expression2>
```

- 采用使用新或现有关键字构建的三元运算符。领先的示例是：

```python
<condition> then <expression1> else <expression2>
(if <condition>: <expression1> else: <expression2>)
```

- 什么都不做。

前两个立场相对相似。

有些人认为任何形式的标点符号都会使语言更加神秘。另一些人发现标点符号风格适用于表达式而不是语句，并有助于避免 COBOL 风格：3 plus 4 times 5。

适应现有关键字试图通过显式含义和更整洁的外观来改进标点符号。缺点是损失了一些标点符号运算符提供的表达经济性。另一个缺点是它在两个含义和两个用法的关键字之间创造了一定程度的混淆。

那些困难通过引入新关键字的选项来克服，这需要更多的努力来实现。

最后一个立场是无所作为。支持论点包括保持语言简单简洁；维护向后兼容性；以及任何用例已经可以用"if"和"else"来表达。Lambda 表达式是一个例外，因为它们需要将条件分解到单独的函数定义中。

反对无所作为的论点是其他选择允许更大的表达经济性，当前做法显示出对"and"、"or"或其更复杂、视觉上不太吸引人的解决方案的错误使用的倾向。

## 短路行为

三元运算符和 cond() 函数之间的主要区别是后者提供表达式形式但不提供短路评估。

短路评估在三种情况下是可取的：

- 当表达式具有副作用时

- 当一个或两个表达式资源密集时

- 当条件作为表达式有效性的保护时。

```python
# 所有三个原因都适用的示例
data = isinstance(source, file) ? source.readlines()
  : source.split()
```

1. readlines() 移动文件指针
2. 对于长源，两个替代方案都需要时间
3. split() 仅对字符串有效，readlines() 仅对文件对象有效。

cond() 函数的支持者指出，对短路评估的需求很少见。扫描现有的代码目录，他们发现 if/else 不常发生；其中只有少数包含 cond() 或三元运算符可以帮助的表达式；大多数不需要短路评估。因此，cond() 足以满足大多数需求，并且可以节省更改语言语法的努力。

更多的支持证据来自对 C 代码库的扫描，显示其三元运算符使用非常罕见（作为代码行数的百分比）。

对这一分析的反驳是，三元运算符的可用性在每个情况下都有助于程序员，因为它节省了搜索副作用的需要。此外，它将排除因引入副作用的远程修改而产生的错误。后一种情况随着属性的出现而变得更加现实，甚至属性访问也可以具有副作用。

BDFL 的立场是短路行为对于将 if-then-else 结构添加到语言是必不可少的。
