## 前言

## 摘要

本文档规定了 Web 服务器与 Python Web 应用程序或框架之间的一项建议标准接口，以促进 Web 应用程序在各种 Web 服务器之间的可移植性。

## 设计原理与目标

Python 目前拥有多种 Web 应用程序框架，如 Zope、Quixote、Webware、SkunkWeb、PSO 和 Twisted Web——仅列举少数几个 [1]。这种广泛的选择对新 Python 用户来说可能是一个问题，因为一般来说，他们对 Web 框架的选择将限制他们对可用 Web 服务器的选择，反之亦然。

相比之下，虽然 Java 同样拥有众多 Web 应用程序框架，但 Java 的"servlet"API 使得使用任何 Java Web 应用程序框架编写的应用程序都可以在支持 servlet API 的任何 Web 服务器中运行。

在 Python Web 服务器中拥有并广泛使用这样的 API——无论这些服务器是用 Python 编写的（例如 Medusa），嵌入 Python（例如 mod_python），还是通过网关协议调用 Python（例如 CGI、FastCGI 等）——都将把框架的选择与 Web 服务器的选择分离开来，使用户能够自由选择合适的组合，同时使框架和服务器开发者能够专注于他们偏好的专业领域。

因此，本 PEP 提出了一种 Web 服务器与 Web 应用程序或框架之间的简单通用接口：Python Web Server Gateway Interface（WSGI）。

但仅仅存在 WSGI 规范并不能解决 Python Web 应用程序服务器和框架的现有状态。服务器和框架的作者和维护者必须实际实现 WSGI 才能产生任何效果。

然而，由于现有服务器或框架都不支持 WSGI，因此对于实现 WSGI 支持的作者来说，几乎没有立竿见影的回报。因此，WSGI 必须易于实现，这样作者对接口初始投入可以相对较低。

因此，接口两端（服务器端和框架端）实现的简单性对于 WSGI 接口的实用性绝对关键，因此是任何设计决策的主要标准。

然而需要注意的是，框架作者实现的简单性与 Web 应用程序作者使用的易用性并非同一回事。WSGI 向框架作者提供了一个绝对"无修饰"的接口，因为响应对象和 cookie 处理等附加功能只会妨碍现有框架处理这些问题。WSGI 的目标是促进现有服务器与应用程序或框架之间的轻松连接，而不是创建一个新的 Web 框架。

还需要注意的是，这一目标排除了 WSGI 要求任何在已部署版本的 Python 中尚未可用的功能。因此，本规范未提出或不需要新的标准库模块，WSGI 中没有任何内容需要 Python 版本高于 2.2.2。（不过，未来的 Python 版本在标准库提供的 Web 服务器中支持此接口是一个好主意。）

除了为现有和未来框架和服务器提供易于实现外，还应该能够轻松创建请求预处理器、响应后处理器和其他基于 WSGI 的"中间件"组件，这些组件对其包含的服务器来说看起来像一个应用程序，同时对其包含的应用程序来说充当服务器。

如果中间件可以既简单又健壮，并且 WSGI 在服务器和框架中广泛可用，那么就有可能出现一种全新的 Python Web 应用程序框架：由松耦合的 WSGI 中间件组件组成的框架。事实上，现有框架作者甚至可能选择重构其框架的现有服务，以便以这种方式提供，变得更像是与 WSGI 配合使用的库，而不是单体框架。这将允许应用程序开发者为特定功能选择"最优秀"的组件，而无需接受单一框架的所有优缺点。

当然，就目前而言，那一天无疑还很遥远。同时，WSGI 的短期目标足以实现任何框架与任何服务器的配合使用。

最后，应该提到的是，当前版本的 WSGI 未规定"部署"应用程序以供 Web 服务器或服务器网关使用的任何特定机制。目前，这必须由服务器或网关定义。在足够多的服务器和框架实现了 WSGI 以提供对不同部署要求的实际经验后，创建另一个 PEP 描述 WSGI 服务器和应用程序框架的部署标准可能是有意义的。

## 规范概述

WSGI 接口有两端："服务器"或"网关"端，以及"应用程序"或"框架"端。服务器端调用由应用程序端提供的可调用对象。如何提供该对象的具体细节由服务器或网关决定。可以假设某些服务器或网关将要求应用程序的部署者编写一个简短的脚本来创建服务器或网关的实例，并向其提供应用程序对象。其他服务器和网关可能使用配置文件或其他机制来指定应从何处导入应用程序对象或以其他方式获取。

除了"纯"服务器/网关和应用程序/框架外，还可以创建实现本规范两端的"中间件"组件。这些组件对其包含的服务器充当应用程序，对其包含的应用程序充当服务器，可用于提供扩展 API、内容转换、导航和其他有用功能。

在本规范的整个过程中，我们将使用术语"可调用对象"来表示"函数、方法、类或具有__call__方法的实例"。由实现可调用对象的服务器、网关或应用程序选择适合其需求的实现技术。相反，调用可调用对象的服务器、网关或应用程序不得依赖于提供给它的是什么类型的可调用对象。可调用对象只应被调用，而不应被内省。

### 应用程序/框架端

应用程序对象只是一个接受两个参数的可调用对象。术语"对象"不应被误解为需要实际的对象实例：函数、方法、类或具有__call__方法的实例都适用于用作应用程序对象。应用程序对象必须能够被多次调用，因为几乎所有服务器/网关（CGI 除外）都会进行此类重复请求。

（注意：虽然我们称之为"应用程序"对象，但这不应被理解为应用程序开发者将使用 WSGI 作为 Web 编程 API！假设应用程序开发者将继续使用现有的高级框架服务来开发他们的应用程序。WSGI 是框架和服务器开发者的工具，并不打算直接支持应用程序开发者。）

下面是两个示例应用程序对象；一个是函数，另一个是类：

```python
def simple_app(environ, start_response):
    """最简单的应用程序对象"""
    status = '200 OK'
    response_headers = [('Content-type', 'text/plain')]
    start_response(status, response_headers)
    return ['Hello world!\n']

class AppClass:
    """使用类产生相同的输出

    （注意：'AppClass'是这里的"应用程序"，所以调用它
    返回'AppClass'的实例，这然后就是
    "应用程序可调用对象"的迭代返回值，如规范所要求。）

    如果我们想用'AppClass'的*实例*作为应用程序
    对象，我们必须实现一个'__call__'
    方法，它将被调用来执行应用程序，
    我们需要创建一个实例供服务器或网关使用。
    """

    def __init__(self, environ, start_response):
        self.environ = environ
        self.start = start_response

    def __iter__(self):
        status = '200 OK'
        response_headers = [('Content-type', 'text/plain')]
        self.start(status, response_headers)
        yield "Hello world!\n"
```

### 服务器/网关端

服务器或网关对从 HTTP 客户端接收到的每个请求调用一次应用程序可调用对象，该请求指向应用程序。为说明起见，这里是一个简单的 CGI 网关，实现为一个接受应用程序对象的函数。注意这个简单示例的错误处理有限，因为默认情况下未捕获的异常将被转储到 sys.stderr 并由 Web 服务器记录。

```python
import os, sys

def run_with_cgi(application):

    environ = dict(os.environ.items())
    environ['wsgi.input'] = sys.stdin
    environ['wsgi.errors'] = sys.stderr
    environ['wsgi.version'] = (1, 0)
    environ['wsgi.multithread'] = False
    environ['wsgi.multiprocess'] = True
    environ['wsgi.run_once'] = True

    if environ.get('HTTPS', 'off') in ('on', '1'):
        environ['wsgi.url_scheme'] = 'https'
    else:
        environ['wsgi.url_scheme'] = 'http'

    headers_set = []
    headers_sent = []

    def write(data):
        if not headers_set:
            raise AssertionError("write() before start_response()")

        elif not headers_sent:
            # 在第一次输出之前，发送存储的头部
            status, response_headers = headers_sent[:] = headers_set
            sys.stdout.write('Status: %s\r\n' % status)
            for header in response_headers:
                sys.stdout.write('%s: %s\r\n' % header)
            sys.stdout.write('\r\n')

        sys.stdout.write(data)
        sys.stdout.flush()

    def start_response(status, response_headers, exc_info=None):
        if exc_info:
            try:
                if headers_sent:
                    # 如果头部已发送，则重新抛出原始异常
                    raise exc_info[0], exc_info[1], exc_info[2]
            finally:
                exc_info = None # 避免悬空循环引用
        elif headers_set:
            raise AssertionError("Headers already set!")

        headers_set[:] = [status, response_headers]
        return write

    result = application(environ, start_response)
    try:
        for data in result:
            if data: # 在出现主体之前不发送头部
                write(data)
            if not headers_sent:
                write('') # 如果主体为空，现在发送头部
    finally:
        if hasattr(result, 'close'):
            result.close()
```

### 中间件：扮演两端的组件

请注意，单个对象可以对某些应用程序扮演服务器的角色，同时对某些服务器充当应用程序。这样的"中间件"组件可以执行以下功能：

- 根据目标 URL 将请求路由到不同的应用程序对象，并在相应地重写 environ 后执行。

- 允许多个应用程序或框架在同一进程中并排运行

- 通过网络转发请求和响应，实现负载均衡和远程处理

- 执行内容后处理，如应用 XSL 样式表

中间件的存在对"服务器/网关"和"应用程序/框架"接口两端都是透明的，不需要特殊支持。希望将中间件合并到应用程序中的用户只需将中间件组件提供给服务器，就像它是一个应用程序一样，并配置中间件组件来调用应用程序，就像中间件组件是一个服务器一样。当然，中间件包装的"应用程序"实际上可能是另一个包装另一个应用程序的中间件组件，依此类推，形成所谓的"中间件堆栈"。

在大多数情况下，中间件必须符合 WSGI 服务器端和应用程序端的限制和要求。然而在某些情况下，中间件的要求比"纯"服务器或应用程序更为严格，这些点将在规范中说明。

这里是一个（开玩笑的）中间件组件示例，它将 text/plain 响应转换为猪拉丁语，使用 Joe Strout 的 piglatin.py。（注意："真正的"中间件组件可能会使用更稳健的方式来检查内容类型，并且还应该检查内容编码。此外，这个简单示例忽略了单词可能被分割在块边界处的可能性。）

```python
from piglatin import piglatin

class LatinIter:

    """如果合适，将迭代输出转换为猪拉丁语

    注意"是否合适"在应用程序生成第一个非空字符串之前可能会改变，
    所以'transform_ok'必须是一个可变真值。
    """

    def __init__(self, result, transform_ok):
        if hasattr(result, 'close'):
            self.close = result.close
        self._next = iter(result).next
        self.transform_ok = transform_ok

    def __iter__(self):
        return self

    def next(self):
        if self.transform_ok:
            return piglatin(self._next())
        else:
            return self._next()

class Latinator:

    # 默认情况下，不转换输出
    transform = False

    def __init__(self, application):
        self.application = application

    def __call__(self, environ, start_response):

        transform_ok = []

        def start_latin(status, response_headers, exc_info=None):

            # 重置 ok 标志，以防这是重复调用
            del transform_ok[:]

            for name, value in response_headers:
                if name.lower() == 'content-type' and value == 'text/plain':
                    transform_ok.append(True)
                    # 如果存在则删除 content-length，否则它将不正确
                    response_headers = [(name, value)
                        for name, value in response_headers
                        if name.lower() != 'content-length'
                    ]
                break

            write = start_response(status, response_headers, exc_info)

            if transform_ok:
                def write_latin(data):
                    write(piglatin(data))
                return write_latin
            else:
                return write

        return LatinIter(self.application(environ, start_latin), transform_ok)

# 在 Latinator 的控制下运行 foo_app，使用示例 CGI 网关
from foo_app import foo_app
run_with_cgi(Latinator(foo_app))
```


## 规范详情

应用程序对象必须接受两个位置参数。为了说明起见，我们将其命名为 environ 和 start_response，但它们不需要具有这些名称。服务器或网关必须使用位置参数（而不是关键字参数）调用应用程序对象。（例如，通过调用 result = application(environ, start_response)，如上所示。）

environ 参数是一个字典对象，包含 CGI 风格的环境变量。该对象必须是一个内置 Python 字典（不是子类、UserDict 或其他字典仿真），并且应用程序可以按需要修改字典。字典还必须包含某些 WSGI 必需的变量（在后面的部分中描述），并且还可以包含服务器特定的扩展变量，按照下面描述的约定命名。

start_response 参数是一个可调用对象，接受两个必需的位置参数和一个可选参数。为了说明起见，我们将其命名为 status、response_headers 和 exc_info，但它们不需要具有这些名称，应用程序必须使用位置参数调用 start_response 可调用对象（例如 start_response(status, response_headers)）。

status 参数是一个状态字符串，格式为"999 Message here"，response_headers 是一个(header_name, header_value)元组列表，描述 HTTP 响应头部。可选的 exc_info 参数在下面关于 The start_response() Callable 和 Error Handling 的部分中描述。它仅当应用程序已捕获错误并尝试向浏览器显示错误消息时使用。

start_response 可调用对象必须返回一个 write(body_data) 可调用对象，它接受一个位置参数：要写入 HTTP 响应主体一部分的字符串。（注意：write() 可调用对象仅提供以支持某些现有框架的命令式输出 API；如果可能，新应用程序或框架不应使用它。详见 Buffering and Streaming 部分。）

当被服务器调用时，应用程序对象必须返回一个可迭代对象，生成零个或多个字符串。这可以通过多种方式实现，例如返回一个字符串列表，或者应用程序是一个生成字符串的生成器函数，或者应用程序是一个其实例是可迭代的类。无论如何实现，应用程序对象必须始终返回一个可迭代对象，生成零个或多个字符串。

服务器或网关必须以无缓冲方式向客户端传输生成的字符串，在完成每个字符串的传输后再请求下一个字符串。（换句话说，应用程序应该自己进行缓冲。详见下面的 Buffering and Streaming 部分，了解如何处理应用程序输出。）

服务器或网关应将生成的字符串视为二进制字节序列：特别是，它应确保换行符不被更改。应用程序负责确保要写入的字符串格式适合客户端。（服务器或网关可以应用 HTTP 传输编码，或执行其他转换以实现 HTTP 功能，如字节范围传输。详见下面的 Other HTTP Features。）

如果对 len(iterable) 的调用成功，服务器必须能够依赖结果是准确的。也就是说，如果应用程序返回的可迭代对象提供了一个有效工作的__len__() 方法，它必须返回准确的结果。（详见 Handling the Content-Length Header 部分，了解这通常如何使用。）

如果应用程序返回的可迭代对象具有 close() 方法，服务器或网关必须在当前请求完成后调用该方法，无论请求是正常完成还是由于错误而提前终止（这是为了支持应用程序的资源释放）。此协议旨在补充 PEP 325 的生成器支持，以及其他带有 close() 方法的常见可迭代对象。

（注意：应用程序必须在可迭代对象生成第一个主体字符串之前调用 start_response() 可调用对象，以便服务器可以在任何主体内容之前发送头部。然而，此调用可以由可迭代对象的第一次迭代执行，因此服务器不得假设在它们开始遍历可迭代对象之前已经调用了 start_response()。）

最后，除非可迭代对象是特定于该服务器或网关的类型（例如由 wsgi.file_wrapper 返回的"文件包装器"，见 Optional Platform-Specific File Handling），否则服务器和网关不得直接使用应用程序返回的可迭代对象的任何其他属性。在一般情况下，只允许使用此处指定的属性或通过 PEP 234 迭代 API 访问的属性。

### environ 变量

environ 字典必须包含以下 CGI 环境变量，如 Common Gateway Interface 规范 [2] 所定义。以下变量必须存在，除非它们的值将是空字符串，在这种情况下它们可能被省略，除非另有说明。

**REQUEST_METHOD** 
HTTP 请求方法，如"GET"或"POST"。这绝不能是空字符串，因此总是必需的。

**SCRIPT_NAME** 
请求 URL 的"路径"的初始部分，对应于应用程序对象，以便应用程序知道其虚拟"位置"。如果应用程序对应于服务器的"根"，这**可能**是空字符串。

**PATH_INFO** 
请求 URL 的"路径"的其余部分，指定请求目标在应用程序内的虚拟"位置"。如果请求 URL 指向应用程序根且没有尾部斜杠，这**可能**是空字符串。

**QUERY_STRING** 
请求 URL 中"?"之后的部分（如果有）。可能为空或缺失。

**CONTENT_TYPE** 
HTTP 请求中任何 Content-Type 字段的内容。可能为空或缺失。

**CONTENT_LENGTH** 
HTTP 请求中任何 Content-Length 字段的内容。可能为空或缺失。

**SERVER_NAME, SERVER_PORT** 
与 SCRIPT_NAME 和 PATH_INFO 结合使用时，这些变量可以用于完成 URL。注意，如果存在，HTTP_HOST 应优先于 SERVER_NAME 用于重建请求 URL。详见下面的 [URL Reconstruction](#url-reconstruction) 部分。SERVER_NAME 和 SERVER_PORT 绝不能是空字符串，因此总是必需的。

**SERVER_PROTOCOL** 
客户端用于发送请求的协议版本。通常这将是类似于"HTTP/1.0"或"HTTP/1.1"的内容，应用程序可用于确定如何处理任何 HTTP 请求头部。（此变量应该称为 REQUEST_PROTOCOL，因为它表示请求中使用的协议，不一定是服务器响应中使用的协议。然而，为了与 CGI 兼容，我们必须保留现有名称。）

**HTTP_ 变量** 
对应于客户端提供的 HTTP 请求头部的变量（即名称以"HTTP_"开头的变量）。这些变量的存在或缺失应与请求中适当 HTTP 头部的存在或缺失相对应。

服务器或网关应尝试提供尽可能多的其他适用 CGI 变量。此外，如果使用 SSL，服务器或网关还应提供尽可能多的适用 Apache SSL 环境变量 [3]，如 HTTPS=on 和 SSL_PROTOCOL。然而需要注意的是，使用上述变量之外的任何 CGI 变量的应用程序必然无法移植到不支持相关扩展的 Web 服务器。（例如，不发布文件的 Web 服务器将无法提供有意义的 DOCUMENT_ROOT 或 PATH_TRANSLATED。）

WSGI 兼容的服务器或网关应记录它提供的变量及其适当的定义。应用程序应检查它们需要的任何变量的存在，并在变量缺失时拥有备用计划。

注意：缺失的变量（例如未发生身份验证时的 REMOTE_USER）应从 environ 字典中省略。还要注意，CGI 定义的变量必须是字符串（如果存在的话）。CGI 变量的值为任何非 str 类型都违反本规范。

除了 CGI 定义的变量外，environ 字典还可以包含任意操作系统"环境变量"，并必须包含以下 WSGI 定义的变量：

| 变量 | 值 |
|------|-----|
| wsgi.version | 元组 (1, 0)，表示 WSGI 版本 1.0。 |
| wsgi.url_scheme | 表示应用程序被调用的 URL 的"方案"部分的字符串。通常，这将具有"http"或"https"的值，视情况而定。 |
| wsgi.input | 输入流（文件-like 对象），从中可以读取 HTTP 请求主体。（服务器或网关可以按需执行应用程序请求的读取，或者它可以预读客户端的请求主体并缓冲在内存或磁盘上，或使用任何其他技术来提供此类输入流，根据其偏好。） |
| wsgi.errors | 输出流（文件-like 对象），可以向其写入错误输出，以在标准化且可能集中的位置记录程序或其他错误。这应该是"文本模式"流；即，应用程序应使用"\n"作为换行符，并假设它将被服务器/网关转换为正确的换行符。对于许多服务器，wsgi.errors 将是服务器的主要错误日志。或者，这可能是 sys.stderr，或某种日志文件。服务器的文档应包含如何配置此或在哪里查找记录的输出的说明。如果需要，服务器或网关可以向不同应用程序提供不同的错误流。 |
| wsgi.multithread | 如果应用程序对象可能被同一进程中的另一个线程同时调用，此值应求值为真，否则应求值为假。 |
| wsgi.multiprocess | 如果等效的应用程序对象可能被另一个进程同时调用，此值应求值为真，否则应求值为假。 |
| wsgi.run_once | 如果服务器或网关期望（但不保证！）应用程序在其包含进程的生命周期内只被调用一次，此值应求值为真。通常，这仅对于基于 CGI（或类似物）的网关才为真。 |

最后，environ 字典还可以包含服务器定义的变量。这些变量应仅使用小写字母、数字、点和下划线命名，并且应该以前缀命名，该前缀对定义服务器或网关是唯一的。例如，mod_python 可能定义名称如 mod_python.some_variable 的变量。

#### 输入和错误流

服务器提供的输入和错误流必须支持以下方法：

| 方法 | 流 | 备注 |
|------|-----|------|
| `read(size)` | input | 1 |
| `readline()` | input | 1, 2 |
| `readlines(hint)` | input | 1, 3 |
| `__iter__()` | input | |
| `flush()` | errors | 4 |
| `write(str)` | errors | |
| `writelines(seq)` | errors | |

每个方法的语义如 Python Library Reference 中所记录，除了上面表中列出的这些注释：

1. 服务器不需要读取超过客户端指定的 Content-Length，如果应用程序尝试读取超过该点，允许模拟文件结束条件。应用程序不应尝试读取超过 CONTENT_LENGTH 变量指定的数据。

2. readline() 的可选"size"参数不受支持，因为它对服务器作者来说实现可能很复杂，并且在实践中不常用。

3. 注意 readlines() 的 hint 参数对调用者和实现者都是可选的。应用程序可以自由不提供它，服务器或网关可以自由忽略它。

4. 由于错误流可能无法倒带，服务器和网关可以自由地立即转发写入操作，而无需缓冲。在这种情况下，flush() 方法可能是无操作。然而，便携式应用程序不能假设输出是无缓冲的或 flush() 是无操作。如果它们需要确保输出确实已被写入，它们必须调用 flush()。（例如，最小化写入同一错误日志的多个进程的数据混合。）

上面表中列出的方法必须由所有符合本规范的服务器支持。符合本规范的应用程序不得使用输入或错误对象的任何其他方法或属性。特别是，应用程序不得尝试关闭这些流，即使它们拥有 close() 方法。

### The start_response() 可调用对象

传递给应用程序对象的第二个参数是一个形式为 start_response(status, response_headers, exc_info=None) 的可调用对象。（与所有 WSGI 可调用对象一样，参数必须按位置提供，而不是按关键字。）start_response 可调用对象用于开始 HTTP 响应，它必须返回一个 write(body_data) 可调用对象（见下面的 Buffering and Streaming 部分）。

status 参数是一个 HTTP"状态"字符串，如"200 OK"或"404 Not Found"。也就是说，它是一个由 Status-Code 和 Reason-Phrase 组成的字符串，按此顺序并用单个空格分隔，没有周围的空白或其他字符。（详见 RFC 2616，Section 6.1.1。）字符串不得包含控制字符，并且不得以回车、换行或其组合终止。

response_headers 参数是一个(header_name, header_value)元组列表。它必须是一个 Python 列表；即 type(response_headers) 是 ListType，服务器**可以**按需要更改其内容。每个 header_name 必须是有效的 HTTP 头部字段名（如 RFC 2616，Section 4.2 所定义），没有尾部冒号或其他标点符号。

每个 header_value **不得**包含任何控制字符，包括回车或换行，无论是嵌入的还是结尾的。（这些要求是为了最小化必须由服务器、网关和中间响应处理器执行的任何解析的复杂性，这些处理器需要检查或修改响应头部。）

一般来说，服务器或网关负责确保向客户端发送正确的头部：如果应用程序省略了 HTTP（或其他相关规范）要求的头部，服务器或网关**必须**添加它。例如，`HTTP Date:` 和 `Server:` 头部通常由服务器或网关提供。

（给服务器/网关作者的提醒：HTTP 头部名称不区分大小写，所以在检查应用程序提供的头部时务必考虑这一点！）

禁止应用程序和中间件使用 HTTP/1.1"hop-by-hop"功能或头部、HTTP/1.0 中的任何等效功能，或任何会影响客户端与 Web 服务器连接持久性的头部。这些功能是实际 Web 服务器的专属领域，服务器或网关**应**将应用程序尝试发送它们视为致命错误，如果它们被提供给 start_response()，应引发错误。（有关"hop-by-hop"功能和头部的更多具体信息，请参阅下面的 Other HTTP Features 部分。）

start_response 可调用对象**不得**实际传输响应头部。相反，它必须存储它们供服务器或网关传输，**只有**在应用程序返回值的第一次迭代生成非空字符串时，或应用程序第一次调用 write() 可调用对象时。（换句话说，直到有实际主体数据可用，或应用程序返回的可迭代对象耗尽，才不应发送响应头部。（此规则的唯一可能例外是响应头部显式包含 Content-Length 为零。）

这种延迟响应头部传输是为了确保缓冲和异步应用程序可以在最后一刻将其原本打算的输出替换为错误输出。例如，如果错误发生在应用程序缓冲区内生成主体时，应用程序可能需要将响应状态从"200 OK"更改为"500 Internal Error"。

如果提供，exc_info 参数必须是 Python sys.exc_info() 元组。此参数应由应用程序仅在 start_response 被错误处理程序调用时提供。如果提供了 exc_info，并且尚未输出任何 HTTP 头部，start_response 应该用新提供的头部替换当前存储的 HTTP 响应头部，从而允许应用程序在发生错误时"改变主意"关于输出。

然而，如果提供了 exc_info，并且 HTTP 头部已经发送，start_response 必须引发错误，并且**应该**引发 exc_info 元组。即：

```python
raise exc_info[0], exc_info[1], exc_info[2]
```

这将重新抛出应用程序捕获的异常，原则上应该中止应用程序。（应用程序在 HTTP 头部已经发送后尝试向浏览器输出错误是不安全的。）如果应用程序调用 start_response 时带有 exc_info，应用程序**不得**捕获 start_response 引发的任何异常。相反，它应该允许此类异常传播回服务器或网关。详见下面的 Error Handling。

应用程序**可以**多次调用 start_response，当且仅当提供了 exc_info 参数时。更准确地说，如果在当前应用程序调用内已经调用过 start_response，则在不提供 exc_info 参数的情况下调用 start_response 是致命错误。（见上面的示例 CGI 网关以了解正确逻辑的说明。）

注意：实现 start_response 的服务器、网关或中间件**应**确保在函数执行期间之后不保留对 exc_info 参数的引用，以避免通过涉及的跟踪和帧创建循环引用。实现此目的最简单的方法是类似：

```python
def start_response(status, response_headers, exc_info=None):
    if exc_info:
        try:
            # 在这里对 exc_info 做事情
        finally:
            exc_info = None # 避免循环引用
```

示例 CGI 网关提供了此技术的另一个说明。

#### 处理 Content-Length 头部

如果应用程序不提供 Content-Length 头部，服务器或网关可以选择几种处理方法之一。最简单的是在响应完成时关闭客户端连接。

然而在某些情况下，服务器或网关可能能够生成 Content-Length 头部，或至少避免需要关闭客户端连接。如果应用程序不调用 write() 可调用对象，并返回一个 len() 为 1 的可迭代对象，则服务器可以通过获取可迭代对象生成的第一个字符串的长度来自动确定 Content-Length。

此外，如果服务器和客户端都支持 HTTP/1.1"chunked encoding"，则服务器**可以**对每个 write() 调用或可迭代对象生成的字符串使用 chunked encoding 发送一个块，从而为每个块生成 Content-Length 头部。这允许服务器保持客户端连接存活（如果希望的话）。注意服务器在执行此操作时**必须**完全遵守 RFC 2616，否则回退到其他处理缺少 Content-Length 的策略之一。

（注意：应用程序和中间件不得对其输出应用任何类型的 Transfer-Encoding，如分块或压缩；作为"hop-by-hop"操作，这些编码是实际 Web 服务器/网关的领域。详见下面的 Other HTTP Features。）

### Buffering and Streaming

一般来说，应用程序将通过缓冲其（适度大小的）输出并一次性发送所有内容来实现最佳吞吐量。这是现有框架（如 Zope）中的常见方法：输出缓冲在 StringIO 或类似对象中，然后一次性传输，与响应头部一起。

WSGI 中对应的方法是应用程序简单地返回一个单元素可迭代对象（如列表），其中包含响应主体作为单个字符串。这是绝大多数应用程序函数的推荐方法，这些函数渲染的 HTML 页面的文本可以轻松放入内存。

然而，对于大文件或 HTTP 流式的特殊用途（如多部分"server push"），应用程序可能需要以小得多的块提供输出（例如，避免将大文件加载到内存中）。有时，响应的一部分可能很耗时才能生成，但提前发送其前面的响应部分很有用。

在这些情况下，应用程序通常返回一个迭代器（通常是生成器迭代器），以块方式生成输出。这些块可以被打断以与多部分边界重合（用于"server push"），或在耗时任务之前（如读取磁盘文件中的另一个块）。

WSGI 服务器、网关和中间件不得延迟任何块的传输；它们必须完全向客户端传输块，或保证它们将继续传输，即使应用程序正在生成下一个块。服务器/网关或中间件可以通过以下三种方式之一提供此保证：

- 将整个块发送到操作系统（并请求刷新任何操作系统缓冲区），然后再将控制权返回给应用程序，或者

- 使用不同的线程确保块继续传输，同时应用程序生成下一个块。

- （仅中间件）将整个块发送到其父网关/服务器

通过提供此保证，WSGI 允许应用程序确保传输不会在输出数据的任意点停滞。这对于多部分"server push"流等的正确功能至关重要，其中多部分边界之间的数据应完整传输到客户端。

#### 中间件处理块边界

为了更好地支持异步应用程序和服务器，中间件组件不得阻塞迭代以等待来自应用程序可迭代对象的多个值。如果中间件需要在产生任何输出之前从应用程序累积更多数据，它必须生成空字符串。

换句话说，中间件组件每次其底层应用程序生成一个值时必须至少生成一个值。如果中间件无法生成任何其他值，它必须生成空字符串。

此要求确保异步应用程序和服务器可以共谋减少同时运行给定数量的应用程序实例所需的线程数。

注意此要求还意味着中间件必须在其底层应用程序返回可迭代对象时尽快返回可迭代对象。还禁止中间件使用 write() 可调用对象来传输由底层应用程序生成的数据。中间件只能使用其父服务器的 write() 可调用对象来传输底层应用程序使用中间件提供的 write() 可调用对象发送的数据。

#### The write() 可调用对象

某些现有应用程序框架 API 以不同于 WSGI 的方式支持无缓冲输出。具体来说，它们提供某种"write"函数或方法来写入无缓冲的数据块，或者它们提供缓冲的"write"函数和"flush"机制来刷新缓冲区。

不幸的是，除非使用线程或其他特殊机制，否则无法用 WSGI 的"可迭代"应用程序返回值来实现此类 API。

因此，为了允许这些框架继续使用命令式 API，WSGI 包含一个特殊的 write() 可调用对象，由 start_response 可调用对象返回。

新的 WSGI 应用程序和框架如果可能应避免使用 write() 可调用对象。write() 可调用对象严格是一个用于支持命令式流 API 的 hack。一般来说，应用程序应通过其返回的可迭代对象生成输出，因为这使得 Web 服务器能够在同一 Python 线程中交错其他任务，可能为整个服务器提供更好的吞吐量。

write() 可调用对象由 start_response() 可调用对象返回，它接受单个参数：要写入 HTTP 响应主体一部分的字符串，处理完全如同它由输出可迭代对象生成一样。换句话说，在 write() 返回之前，它必须保证传入的字符串已完全发送到客户端，或已缓冲供传输，同时应用程序继续前进。

应用程序必须返回一个可迭代对象，即使它使用 write() 来生成其响应主体的全部或部分。返回的可迭代对象可能为空（即不生成任何非空字符串），但如果它确实生成非空字符串，该输出必须由服务器或网关正常处理（即，它必须立即发送或入队）。应用程序不得从返回的可迭代对象内部调用 write()，因此可迭代对象生成的任何字符串在传递给 write() 的所有字符串发送到客户端之后传输。

### Unicode 问题

HTTP 不直接支持 Unicode，此接口也不支持。所有编码/解码必须由应用程序处理；传递给或来自服务器的所有字符串必须是标准 Python 字节字符串，不是 Unicode 对象。在需要字符串对象的地方使用 Unicode 对象的结果未定义。

注意传递给 start_response() 作为状态或响应头部的字符串**必须**遵循 RFC 2616 关于编码的规定。也就是说，它们必须是 ISO-8859-1 字符，或使用 RFC 2047 MIME 编码。

在 str 或 StringType 类型实际上是 Unicode 基础的 Python 平台上（如 Jython、IronPython、Python 3000 等），本规范中提到的所有"字符串"必须只包含可在 ISO-8859-1 编码中表示的代码点（\u0000 到 \u00FF，含）。应用程序提供包含任何其他 Unicode 字符或代码点的字符串是致命错误。同样，服务器和网关不得向应用程序提供包含任何其他 Unicode 字符的字符串。

再次强调，本规范中提到的所有字符串**必须**是 str 或 StringType 类型，**不得**是 unicode 或 UnicodeType 类型。而且，即使给定平台允许在 str/StringType 对象中每个字符多于 8 位，对于本规范中称为"字符串"的任何值，只能使用较低的 8 位。

### 错误处理

一般来说，应用程序**应**尝试捕获自己的内部错误，并在浏览器中显示有帮助的消息。（"有帮助"在此上下文中的含义由应用程序决定。）

然而，要显示这样的消息，应用程序必须尚未实际向浏览器发送任何数据，否则它冒着破坏响应的风险。WSGI 因此提供了一种机制，允许应用程序发送其错误消息，或自动中止：start_response 的 exc_info 参数。以下是其使用示例：

```python
try:
    # 这里放常规应用程序代码
    status = "200 Froody"
    response_headers = [("content-type", "text/plain")]
    start_response(status, response_headers)
    return ["normal body goes here"]
except:
    # XXX 应该在此裸'except:'之前单独处理 MemoryError、KeyboardInterrupt 等运行时问题...
    status = "500 Oops"
    response_headers = [("content-type", "text/plain")]
    start_response(status, response_headers, sys.exc_info())
    return ["error body goes here"]
```

如果发生异常时没有输出已写入，对 start_response 的调用将正常返回，应用程序将返回错误主体以发送到浏览器。然而，如果任何输出已经发送到浏览器，start_response 将重新抛出提供的异常。此异常不应被应用程序捕获，因此应用程序将中止。服务器或网关可以捕获此（致命）异常并中止响应。

服务器应捕获并记录任何中止应用程序或其返回值迭代过程的异常。如果应用程序发生错误时已部分响应已写入浏览器，服务器或网关可以尝试向输出添加错误消息（如果已发送的头部表示服务器知道如何干净修改的 text/* 内容类型）。

某些中间件可能希望提供额外的异常处理服务，或拦截和替换应用程序错误消息。在这种情况下，中间件可以选择不重新抛出提供给 start_response 的 exc_info，而是引发中间件特定的异常，或者简单地返回而不引发异常，同时存储提供的参数。这然后将导致应用程序返回其错误主体可迭代对象（或调用 write()），允许中间件捕获和修改错误输出。只要应用程序作者遵循以下规则，这些技术就能正常工作：

- 在开始错误响应时始终提供 exc_info

- 在提供 exc_info 时从不捕获 start_response 引发的错误

### HTTP 1.1 Expect/Continue

实现 HTTP 1.1 的服务器和网关必须为 HTTP 1.1 的"expect/continue"机制提供透明支持。这可以通过以下几种方式之一完成：

- 对包含 Expect: 100-continue 请求的请求立即响应"100 Continue"，然后正常继续。

- 正常继续处理请求，但向应用程序提供 wsgi.input 流，该流将在应用程序首次尝试从输入流读取时发送"100 Continue"响应。读取请求必须保持阻塞状态，直到客户端响应。

- 等待客户端决定服务器不支持 expect/continue，并自行发送请求主体。（这是次优的，不推荐。）

注意这些行为限制不适用于 HTTP 1.0 请求，或不指向应用程序对象的请求。有关 HTTP 1.1 Expect/Continue 的更多信息，请参见 RFC 2616，第 8.2.3 和 10.1.1 节。

### Other HTTP Features

一般来说，服务器和网关应该"装傻"，允许应用程序完全控制其输出。它们应仅进行不会改变应用程序响应有效语义的更改。应用程序开发者总是可以添加中间件组件来提供额外功能，因此服务器/网关开发者应保守实现。在某种意义上，服务器应认为自己类似于 HTTP"网关服务器"，应用程序是 HTTP"源服务器"。（详见 RFC 2616，第 1.3 节，了解这些术语的定义。）

然而，由于 WSGI 服务器和应用程序不通过 HTTP 通信，RFC 2616 所称的"hop-by-hop"头部不适用于 WSGI 内部通信。WSGI 应用程序不得生成任何"hop-by-hop"头部，尝试使用需要它们生成此类头部的 HTTP 功能，或依赖 environ 字典中任何传入"hop-by-hop"头部的内容。WSGI 服务器必须自行处理任何支持的传入"hop-by-hop"头部，如解码任何传入的 Transfer-Encoding，包括 chunked encoding（如果适用）。

将这些原则应用于各种 HTTP 功能，应该清楚的是，服务器可以通过 If-None-Match 和 If-Modified-Since 请求头部以及 Last-Modified 和 ETag 响应头部处理缓存验证。然而，这不是必需的，如果应用程序希望支持此功能，它应该执行自己的缓存验证，因为服务器/网关不要求执行此类验证。

类似地，服务器可以重新编码或传输编码应用程序的响应，但应用程序应该在其自身上使用适当的内容编码，并且不得应用传输编码。如果客户端请求，服务器可以传输应用程序响应的字节范围，而应用程序本机不支持字节范围。同样，如果希望，应用程序应该在其自身上执行此功能。

注意对应用程序的这些限制不一定意味着每个应用程序必须重新实现每个 HTTP 功能；许多 HTTP 功能可以通过中间件组件部分或完全实现，从而解放服务器和应用程序作者，无需一遍又一遍地实现相同的功能。

### 线程支持

线程支持（或缺乏 thereof）也是服务器依赖的。可以并行运行多个请求的服务器还应提供以单线程方式运行应用程序的选项，以便不是线程安全的应用程序或框架仍可与该服务器配合使用。

## 实现/应用程序说明

### 服务器扩展 API

某些服务器作者可能希望公开更高级的 API，应用程序或框架作者可以将其用于特殊目的。例如，基于 mod_python 的网关可能希望将 Apache API 的一部分公开为 WSGI 扩展。

在最简单的情况下，这只需要定义一个 environ 变量，如 mod_python.some_api。但是，在许多情况下，中间件的可能存在可能使这变得困难。例如，提供访问 environ 变量中找到的相同 HTTP 头部的 API，如果在 environ 被中间件修改后返回不同的数据。

一般来说，任何重复、取代或绕过 WSGI 功能某些部分的扩展 API 都有与中间件组件不兼容的风险。服务器/网关开发者不应假设没有人会使用中间件，因为某些框架开发者专门打算组织或重组他们的框架，使其几乎完全作为各种类型的中间件运行。

因此，为了提供最大兼容性，提供替换某些 WSGI 功能的扩展 API 的服务器和网关必须设计这些 API，以便它们使用它们替换的 API 部分来调用。例如，访问 HTTP 请求头部的扩展 API 必须要求应用程序传入其当前 environ，以便服务器/网关可以验证通过 API 可访问的 HTTP 头部未被中间件更改。如果扩展 API 不能保证它总是与 environ 关于 HTTP 头部的内容一致，它必须拒绝为应用程序服务，例如通过引发错误、返回 None 而不是头部集合，或对 API 适当的任何其他操作。

类似地，如果扩展 API 提供了写入响应数据或头部的替代方法，它应该在应用程序能够获得扩展服务之前要求传入 start_response 可调用对象。如果传入的对象不是服务器/网关最初提供给应用程序的那个，它不能保证正确操作，必须拒绝向应用程序提供扩展服务。

这些指导原则也适用于向 environ 添加信息（如解析的 cookie、表单变量、会话等）的中间件。具体来说，此类中间件应该将这些功能作为对 environ 操作的函数提供，而不是简单地将值塞入 environ。这有助于确保信息是从 environ 计算的，在任何中间件执行任何 URL 重写或其他 environ 修改之后。

这些"安全扩展"规则必须由服务器/网关和中间件开发者遵循，这是非常重要的，以避免未来中间件开发者被迫从 environ 中删除所有扩展 API，以确保它们的中介不被使用这些扩展的应用程序绕过！

### 应用程序配置

本规范未定义服务器如何选择或获取要调用的应用程序。这些和其他配置选项是高度服务器特定的事项。预期服务器/网关作者将记录如何配置服务器以执行特定应用程序对象，以及使用什么选项（如线程选项）。

框架作者应记录如何创建包装其框架功能的应用程序对象。选择了服务器和应用程序框架的用户必须将两者连接在一起。然而，由于框架和服务器现在有了通用接口，这应该只是一个机械问题，而不是每个新服务器/框架对的重大工程努力。

最后，某些应用程序、框架和中间件可能希望使用 environ 字典来接收简单的字符串配置选项。服务器和网关应该通过允许应用程序的部署者指定要放置在 environ 中的名称 - 值对来支持此功能。在最简单的情况下，此支持可以仅包含将所有操作系统提供的环境变量从 os.environ 复制到 environ 字典，因为部署者原则上可以在服务器外部配置这些，或者在 CGI 情况下，它们可能能够通过服务器的配置文件设置。

应用程序应尝试将此类必需变量保持在最低限度，因为不是所有服务器都支持轻松配置它们。当然，即使在最坏的情况下，部署应用程序的人员也可以创建脚本来提供必要的配置值：

```python
from the_app import application

def new_app(environ, start_response):
    environ['the_app.configval1'] = 'something'
    return application(environ, start_response)
```

但是，大多数现有应用程序和框架可能只需要 environ 中的一个配置值，以指示其应用程序或框架特定配置文件的位置。（当然，应用程序应该缓存此类配置，以避免每次调用时都要重新读取。）

### URL 重建

如果应用程序希望重建请求的完整 URL，它可以按照 Ian Bicking 提供的以下算法执行：

```python
from urllib import quote
url = environ['wsgi.url_scheme']+'://'

if environ.get('HTTP_HOST'):
    url += environ['HTTP_HOST']
else:
    url += environ['SERVER_NAME']

    if environ['wsgi.url_scheme'] == 'https':
        if environ['SERVER_PORT'] != '443':
            url += ':' + environ['SERVER_PORT']
    else:
        if environ['SERVER_PORT'] != '80':
            url += ':' + environ['SERVER_PORT']

url += quote(environ.get('SCRIPT_NAME', ''))
url += quote(environ.get('PATH_INFO', ''))
if environ.get('QUERY_STRING'):
    url += '?' + environ['QUERY_STRING']
```

注意这样重建的 URL 可能不完全是客户端请求的相同 URI。例如，服务器重写规则可能已经修改了客户端最初请求的 URL 以将其置于规范形式。

### 支持旧版本（<2.2）的 Python

某些服务器、网关或应用程序可能希望支持旧版本（<2.2）的 Python。如果目标是 Jython，这一点尤为重要，因为就本文撰写时而言，生产就绪版本的 Jython 2.2 尚不可用。

对于服务器和网关，这相对简单：针对 Python 2.2 之前版本的服务器和网关必须简单地将自己限制为仅使用标准"for"循环来遍历应用程序返回的任何可迭代对象。这是确保与 2.2 之前的迭代器协议（下面进一步讨论）和"今天的"迭代器协议（见 PEP 234）在源代码级别兼容的唯一方法。

（注意此技术仅适用于用 Python 编写的服务器、网关或中间件。如何从其他语言正确使用方法迭代器协议的讨论超出本 PEP 的范围。）

对于应用程序，支持 2.2 之前的 Python 版本稍微复杂一些：

- 你不得返回文件对象并期望它作为可迭代对象工作，因为在 Python 2.2 之前，文件不是可迭代的。（一般来说，你无论如何都不应该这样做，因为大多数时候它表现会很差！）使用 wsgi.file_wrapper 或特定于应用程序的文件包装器类。（详见 Optional Platform-Specific File Handling 了解更多关于 wsgi.file_wrapper 的信息，以及你可以用来包装文件作为可迭代对象的示例类。）

- 如果你返回自定义可迭代对象，它必须实现 2.2 之前的迭代器协议。也就是说，提供一个接受整数键并在耗尽时引发 IndexError 的__getitem__方法。（注意内置序列类型也是可以接受的，因为它们也实现此协议。）

最后，希望支持 2.2 之前 Python 版本并遍历应用程序返回值或自身返回可迭代对象（或两者）的中间件，必须遵循上述适当建议。

（注意：不言而喻，要支持 2.2 之前的 Python 版本，任何服务器、网关、应用程序或中间件还必须仅使用目标版本中可用的语言功能，使用 1 和 0 而不是 True 和 False 等。）

### 可选的平台特定文件处理

某些操作系统环境提供特殊的高性能文件传输功能，如 Unix sendfile() 调用。服务器和网关可以通过 environ 中的可选 wsgi.file_wrapper 键公开此功能。应用程序可以使用此"文件包装器"将文件或文件-like 对象转换为可迭代对象，然后返回它，例如：

```python
if 'wsgi.file_wrapper' in environ:
    return environ['wsgi.file_wrapper'](filelike, block_size)
else:
    return iter(lambda: filelike.read(block_size), '')
```

如果服务器或网关提供 wsgi.file_wrapper，它必须是一个可调用对象，接受一个必需的位置参数和一个可选的位置参数。第一个参数是要发送的文件-like 对象，第二个参数是可选的块大小"建议"（服务器/网关可以不必使用）。可调用对象必须返回一个可迭代对象，并且不得执行任何数据传输，直到服务器/网关实际从应用程序接收可迭代对象作为返回值。（否则将阻止中间件能够解释或覆盖响应数据。）

要被视为"文件-like"，应用程序提供的对象必须具有接受可选 size 参数的 read() 方法。它可能具有 close() 方法，如果有，wsgi.file_wrapper 返回的可迭代对象必须具有调用原始文件-like 对象 close() 方法的 close() 方法。如果"文件-like"对象具有任何其他名称与 Python 内置文件对象匹配的方法或属性（如 fileno()），wsgi.file_wrapper 可以假设这些方法或属性具有与内置文件对象相同的语义。

任何平台特定文件处理的实际实现必须在应用程序返回之后发生，服务器或网关检查是否返回了包装器对象。（再次，由于中间件、错误处理器等的存在，不保证创建的任何包装器实际上将被使用。）

除了 close() 的处理外，从应用程序返回文件包装器的语义应该与应用程序返回 iter(filelike.read, '') 相同。换句话说，传输应该从传输开始时"文件"内的当前位置开始，并继续直到到达末尾。

当然，平台特定的文件传输 API 通常不接受任意的"文件-like"对象。因此，wsgi.file_wrapper 必须内省提供的对象以查找 fileno()（类 Unix 操作系统）或 java.nio.FileChannel（在 Jython 下）等内容，以确定文件-like 对象是否适合使用其支持的平台特定 API。

注意即使对象不适合平台 API，wsgi.file_wrapper 仍然必须返回一个包装 read() 和 close() 的可迭代对象，以便使用文件包装器的应用程序可以在平台间移植。这里是一个简单的与平台无关的文件包装器类，适用于旧版（2.2 之前）和新型 Python：

```python
class FileWrapper:

    def __init__(self, filelike, blksize=8192):
        self.filelike = filelike
        self.blksize = blksize
        if hasattr(filelike, 'close'):
            self.close = filelike.close

    def __getitem__(self, key):
        data = self.filelike.read(self.blksize)
        if data:
            return data
        raise IndexError
```

这是来自使用它来访问平台特定 API 的服务器/网关的代码片段：

```python
environ['wsgi.file_wrapper'] = FileWrapper
result = application(environ, start_response)

try:
    if isinstance(result, FileWrapper):
        # 检查 result.filelike 是否可用于平台特定
        # API，如果是，使用该 API 传输 result。
        # 如果不是，则落入下面的正常可迭代对象
        # 处理循环。

    for data in result:
        # 等等。

finally:
    if hasattr(result, 'close'):
        result.close()
```

## 问题与解答

**为什么 environ 必须是字典？使用子类有什么问题？**

要求字典的原理是最大化服务器之间的可移植性。替代方案是定义字典方法的某个子集作为标准和可移植接口。然而，在实践中，大多数服务器可能会发现字典足以满足其需求，因此框架作者将期望完整的字典功能可用，因为它们通常都存在。但是，如果某些服务器选择不使用字典，那么即使该服务器"符合"规范，也会出现互操作性问题。因此，使字典成为强制性简化了规范并保证了互操作性。

注意这并不阻止服务器或框架开发者在 environ 字典内部提供特殊服务作为自定义变量。这是提供此类增值服务的方法。

- **为什么你可以调用 write() 并生成字符串/返回可迭代对象？我们不应该只选择一种方式吗？**

如果我们只支持迭代方法，那么当前假设可用"push"的框架会受苦。但是，如果我们只支持通过 write() 推送，那么服务器性能在传输大文件时会受到影响（如果工作线程在所有输出发送之前不能开始新请求的工作）。因此，这种妥协允许应用程序框架支持两种方法，视情况而定，但给服务器实现者的负担仅比仅推送方法稍多一点。

- **close() 是做什么的？**

当在应用程序对象执行期间执行写入时，应用程序可以使用 try/finally 块确保资源被释放。但是，如果应用程序返回可迭代对象，任何使用的资源直到可迭代对象被垃圾回收之前都不会被释放。close() 习语允许应用程序在请求结束时释放关键资源，并且它与 PEP 325 提议的生成器中 try/finally 支持向前兼容。

- **为什么这个接口如此低级？我想要功能 X！（如 cookie、会话、持久性...）**

这不是又一个 Python Web 框架。它只是框架与 Web 服务器通信的方式，反之亦然。如果你想要这些功能，你需要选择一个提供你想要功能的 Web 框架。如果该框架让你创建 WSGI 应用程序，你应该能够在大多数支持 WSGI 的服务器中运行它。此外，某些 WSGI 服务器可能通过其 environ 字典中提供的对象提供额外服务；详见相关服务器文档。（当然，使用此类扩展的应用程序将无法移植到其他基于 WSGI 的服务器。）

- **为什么使用 CGI 变量而不是老式的 HTTP 头部？为什么将它们与 WSGI 定义的变量混合？**

许多现有 Web 框架建立在 CGI 规范之上，现有 Web 服务器知道如何生成 CGI 变量。相比之下，表示传入 HTTP 信息的替代方法是碎片化的，缺乏市场份额。因此，使用 CGI"标准"似乎是利用现有实现的好方法。至于将它们与 WSGI 变量混合，分离它们将只需要传递两个字典参数，而不会提供任何真正的好处。

- **状态字符串呢？我们不能只使用数字，传入 200 而不是"200 OK"吗？**

这样做会使服务器或网关复杂化，要求它们拥有数字状态和相应消息的表。相比之下，应用程序或框架作者输入与特定响应代码一起的额外文本很容易，现有框架通常已经拥有包含所需消息的表。因此，总的来说，让应用程序/框架负责似乎更好，而不是服务器或网关。

- **为什么 wsgi.run_once 不保证只运行一次应用程序？**

因为这只是对应用程序的建议，它应该"为不频繁运行做准备"。这是为具有多种操作模式的应用程序框架设计的，用于缓存、会话等。在"多次运行"模式下，此类框架可以预加载缓存，并且可能不会在每次请求后将日志或会话数据写入磁盘。在"单次运行"模式下，此类框架避免预加载并在每次请求后刷新所有必要的写入。

然而，为了测试应用程序或框架以验证后一种模式中的正确操作，可能需要（或至少方便）多次调用它。因此，应用程序不应假设它肯定不会再次运行，仅因为它被调用时 wsgi.run_once 设置为 True。

- **功能 X（字典、可调用对象等）在应用程序代码中用于很难看；我们为什么不使用对象？**

WSGI 的所有这些实现选择专门旨在将功能相互解耦；将这些功能重新组合到封装的对象中使得编写服务器或网关稍微困难一些，编写仅替换或修改整体功能小部分的中间件难一个数量级。

本质上，中间件想要有一个"责任链"模式，通过它可以充当某些功能的"处理程序"，同时允许其他功能保持不变。如果接口要保持可扩展性，使用普通 Python 对象很难做到这一点。例如，必须使用__getattr__或__getattribute__覆盖，以确保扩展（如未来 WSGI 版本定义的属性）被传递。

这类代码众所周知很难 100% 正确，很少有人想自己编写它。因此，他们将复制其他人的实现，但当他们复制的人纠正另一个角落情况时，他们未能更新它们。

此外，这个必要的样板代码将是纯粹的浪费，是中间件开发者为支持应用程序框架开发者稍微更好看的 API 而支付的开发者税。但是，应用程序框架开发者通常只会更新一个框架以支持 WSGI，并且在其整个框架的非常有限的部分。这很可能是他们的第一个（也许是唯一一个）WSGI 实现，因此他们可能会实现此规范。因此，使 API"更好看"的努力与对象属性等内容可能会浪费在这个受众身上。

我们鼓励那些想要更好看（或其他改进）的 WSGI 接口用于直接 Web 应用程序编程（相对于 Web 框架开发）的人开发包装 WSGI 以便应用程序开发者方便使用的 API 或框架。这样，WSGI 可以保持方便的低级供服务器和中间件作者使用，同时对应用程序开发者来说不"难看"。

- Mark Nottingham，他广泛审查了规范，查找与 HTTP RFC 合规性的问题，特别是关于我甚至不知道存在的 HTTP/1.1 功能，直到他指出。
