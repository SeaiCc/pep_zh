# PEP 405 – Python 虚拟环境

**元数据:**

- **作者**: Carl Meyer <carl at oddbird.net>
- **BDFL 代表**: Alyssa Coghlan
- **创建日期**: 2011 年 6 月 13 日

- **发布历史**: 2011 年 10 月 24 日，2011 年 10 月 28 日，2012 年 3 月 6 日，2012 年 5 月 24 日

## 摘要

本 PEP 提议为 Python 添加一种轻量级"虚拟环境"机制，具有自己的站点目录，可选择性地与系统站点目录隔离。每个虚拟环境都有自己的 Python 二进制文件（允许创建具有各种 Python 版本的环境），并且可以拥有自己独立的一组已安装的 Python 包在其站点目录中，但与基础安装的 Python 共享标准库。

## 动机

Python 虚拟环境的实用性已经通过现有第三方虚拟环境工具的普及得到了很好的确立，主要是 Ian Bicking 的 virtualenv。虚拟环境已经广泛用于依赖管理、隔离、无需系统管理员访问即可安装和使用 Python 包、以及在多个 Python 版本上自动化测试 Python 软件等用途。

现有的虚拟环境工具缺乏 Python 本身行为的支持。像 rvirtualenv 这样的工具不会将 Python 二进制文件复制到虚拟环境中，无法提供可靠的与系统站点目录的隔离。Virtualenv 确实复制 Python 二进制文件，被迫复制 Python 的 site 模块的大部分，并手动符号链接/复制一组不断变化的标准库模块到虚拟环境中，以便在每次启动时执行微妙的引导舞蹈。（Virtualenv 必须复制二进制文件以提供隔离，因为 Python 在搜索 sys.prefix 之前会解引用符号链接的可执行文件。）

PYTHONHOME 环境变量，Python 唯一的现有内置虚拟环境解决方案，需要将整个标准库复制/符号链接到每个环境中。复制整个标准库不是轻量级解决方案，并且对符号链接的跨平台支持仍然不一致（即使在支持符号链接的 Windows 平台上，创建它们通常也需要管理员权限）。

与 Python 集成并借鉴多年现有第三方工具经验的虚拟环境机制可以降低维护成本、提高可靠性，并且更容易为所有 Python 用户提供。

## 规范

当 Python 二进制文件执行时，它尝试确定其前缀（存储在 sys.prefix 中），然后用于查找标准库和其他关键文件，并由 site 模块用于确定站点包目录的位置。目前，前缀是通过首先在文件系统树中向上查找标记文件（os.py），该文件标志着标准库的存在，如果未找到，则回退到二进制文件中硬编码的构建时前缀来找到。

本 PEP 提议向此搜索添加一个新的第一步。如果在 Python 可执行文件旁边或在其上方一个目录中找到 pyvenv.cfg 文件（如果可执行文件是符号链接，则不解引用），则扫描该文件以查找形式为 key = value 的行。如果找到 home 键，则表示 Python 二进制文件属于虚拟环境，home 键的值是包含用于创建此虚拟环境的 Python 可执行文件的目录。

在这种情况下，使用前缀查找继续正常进行，使用 home 键的值作为有效的 Python 二进制文件位置，从而找到基础安装的前缀。sys.base_prefix 设置为此值，而 sys.prefix 设置为包含 pyvenv.cfg 的目录。

（如果未找到 pyvenv.cfg 或不包含 home 键，则前缀查找正常继续，sys.prefix 将等于 sys.base_prefix。）

另外，添加 sys.base_exec_prefix，并相对于 sys.exec_prefix 类似处理。（sys.exec_prefix 相当于 sys.prefix，但是针对平台特定文件；默认情况下它具有与 sys.prefix 相同的值。）

site 和 sysconfig 标准库模块被修改，使得标准库和头文件相对于 sys.base_prefix / sys.base_exec_prefix 查找，而站点包目录（在 sysconfig 术语中的"purelib"和"platlib"）仍然相对于 sys.prefix / sys.exec_prefix 查找。

因此，最简单的 Python 虚拟环境形式仅由 Python 二进制文件的副本或符号链接、一个 pyvenv.cfg 文件和一个 site-packages 目录组成。

### 与系统 site-packages 的隔离

默认情况下，虚拟环境与系统级 site-packages 目录完全隔离。

如果 pyvenv.cfg 文件还包含一个值为 true（不区分大小写）的 include-system-site-packages 键，则 site 模块还将系统站点目录添加到虚拟环境站点目录之后的 sys.path 中。因此，系统安装的包仍然可以导入，但在虚拟环境中安装的具有相同名称的包将优先。

PEP 370 用户级 site-packages 被视为系统 site-packages 的一部分：它们不可用于隔离的 venv，但可用于 include-system-site-packages = true 的 venv。

### 创建虚拟环境

本 PEP 还提议向标准库添加一个新的 venv 模块，该模块实现了虚拟环境的创建。此模块可以使用-m 标志执行：

```python
python3 -m venv /path/to/new/virtual/environment
```

还提供了 pyvenv 安装脚本以使这更方便：

pyvenv /path/to/new/virtual/environment

运行此命令创建目标目录（创建任何尚不存在的父目录），并在其中放置一个 pyvenv.cfg 文件，其中包含一个指向运行此命令的 Python 安装的 home 键。它还创建一个 bin/（或 Windows 上的 Scripts）子目录，其中包含 python3 可执行文件的副本（或符号链接），以及来自打包标准库模块的 pysetup3 脚本（以促进从 PyPI 轻松安装包到新的 venv）。并且它创建一个（最初为空的）lib/pythonX.Y/site-packages（或 Windows 上的 Lib\site-packages）子目录。

如果目标目录已经存在，将引发错误，除非提供了 --clear 选项，在这种情况下将删除目标目录，虚拟环境创建将照常进行。

创建的 pyvenv.cfg 文件还包括 include-system-site-packages 键，如果 pyvenv 与 --system-site-packages 选项一起运行，则设置为 true，默认为 false。

可以向 pyvenv 提供多个路径，在这种情况下，将根据给定选项在每个提供的路径处创建相同的 venv。

venv 模块还将 POSIX 和 Windows 系统的"shell 激活脚本"放置在 venv 的 bin 或 Scripts 目录中。这些脚本只是将虚拟环境的 bin（或 Scripts）目录添加到用户 shell PATH 的开头。这对于使用虚拟环境不是严格必要的（因为可以使用显式路径到 venv 的 python 二进制文件或脚本），但它很方便。

为了允许 pysetup 和其他 Python 包管理器以与安装到普通 Python 安装相同的方式将包安装到虚拟环境中，并在 sysconfig 中避免对虚拟环境的特殊处理，除了在适当的地方使用 sys.base_prefix 代替 sys.prefix，内部虚拟环境布局模仿每个平台上 Python 安装本身的布局。因此，典型的 POSIX 系统上的虚拟环境布局为：

```
pyvenv.cfg
bin/python3
bin/python
bin/pysetup3
include/
lib/python3.3/site-packages/
```

而在 Windows 系统上：

```
pyvenv.cfg
Scripts/python.exe
Scripts/python3.dll
Scripts/pysetup3.exe
Scripts/pysetup3-script.py
 ... 其他 DLLs 和 pyds...
Include/
Lib/site-packages/
```

安装到虚拟环境中的第三方包将把其 Python 模块放在 site-packages 目录中，将其可执行文件放在 bin/或 Scripts 中。

**注意**

在正常的 Windows 系统级安装上，Python 二进制文件本身不会放在"Scripts/"子目录中，就像它在默认 venv 布局中那样。这在虚拟环境中很有用，因为用户只需要将单个目录添加到 shell PATH 即可有效地"激活"虚拟环境。

**注意**

在 Windows 上，还需要从编译的 stdlib 模块复制或符号链接 DLL 和 pyd 文件到环境中，因为如果 venv 是从非系统范围的 Python 安装创建的，当从 venv 运行 Python 时，Windows 将无法找到 Python 安装的这些文件副本。

### Sysconfig 安装方案和 user-site

这种方法明确选择不为 venv 引入新的 sysconfig 安装方案。相反，通过修改 sys.prefix，我们确保基于 sys.prefix 的现有安装方案将直接在 venv 中工作。安装到其他安装方案（例如 user-site 方案）的路径不相对于 sys.prefix，将完全不受 venv 影响。

创建一个基于虚拟特定 sysconfig 方案的 Python 虚拟环境的替代实现可能是可行的，但它将不太稳健，因为它需要更多的代码来知道它是否在虚拟环境中运行。

### 副本与符号链接

本 PEP 中的技术在一般情况下对复制的或符号链接的 Python 二进制文件（以及 Windows 上所需的 DLL）同样有效。符号链接在可能的情况下是可取的，因为在升级底层 Python 安装的情况下，venv 中复制的 Python 可执行文件可能与安装的标准库不同步，需要手动升级。

符号链接存在一些跨平台困难：

- 不是所有 Windows 版本都支持符号链接，即使在支持的那些版本上，创建它们通常也需要管理员权限。

- 在 OS X framework 版本的 Python 上，sys.executable 只是一个执行真实 Python 二进制文件的存根。符号链接此存根不起作用；必须复制它。（幸运的是存根也很小，并且不会因为 Python 的 bugfix 升级而更改，因此复制它不是问题）。

因此，本 PEP 提议在所有平台上符号链接二进制文件，除了 Windows 和 OS X framework 版本。可用 --symlink 选项强制在支持符号链接的 Windows 版本上使用符号链接，如果具有适当的权限。（此选项对 OS X framework 版本没有影响，因为符号链接在那里永远无法工作，也没有优势）。

在 Windows 上，如果不使用 --symlink，这意味着如果底层 Python 安装被升级，venv 中的 Python 二进制文件和 DLL 应该更新，否则可能会与升级的标准库出现不匹配问题。pyvenv 脚本接受一个 --upgrade 选项，以便在现有 venv 上轻松执行此升级。

### Include 文件

当前的 virtualenv 以这种方式处理 include 文件：

在 POSIX 系统上，已安装的 Python 的 include 文件位于 `${base_prefix}/include/pythonX.X`，virtualenv 创建 `${venv}/include/`并符号链接 `${base_prefix}/include/pythonX.X` 到 `${venv}/include/pythonX.X`。在 Windows 上，Python 的 include 文件位于`{{sys.prefix}}/Include`，符号链接不可靠可用，virtualenv 复制`{{sys.prefix}}/Include` 到 `${venv}/Include`。这确保在 virtualenv 内构建和安装的扩展模块总是在相对于 sys.prefix 的期望位置找到它们需要的 Python 头文件。

当扩展模块安装自己的头文件时，此解决方案不是理想的，因为这些头文件的默认安装位置可能是对可能不可写的系统目录的符号链接。一个安装程序 pip 明确通过安装头文件到非标准位置 ${venv}/include/site/pythonX.X/ 来解决这个问题，因为在 Python 中目前还没有针对特定于站点的 include 目录的标准抽象。

本 PEP 提出了一种略有不同的方法，尽管效果基本相同，并且具有相同的优势和劣势集。不是将 include 文件符号链接或复制到 venv，我们只是修改 sysconfig 方案，使得头文件始终相对于 base_prefix 而不是 prefix 查找。（我们还在 venv 内创建一个 include/目录，以便安装程序可以将安装到 env 内的 include 文件放在某处）。

在 distutils/packaging 和扩展 pyvenv 中更好地处理 include 文件是一个可能值得自己未来 PEP 的领域。目前，我们提议 virtualenv 的行为迄今为止在实践中证明至少"足够好"。

### API

上述高级方法使用简单的 API，为第三方虚拟环境创建者提供根据需求自定义环境创建的机制。

venv 模块包含一个 EnvBuilder 类，在实例化时接受以下关键字参数：

- system_site_packages - 一个布尔值，指示系统 Python site-packages 应对环境可用。默认为 False。

- clear - 一个布尔值，如果为 true，将删除任何现有目标目录而不是引发异常。默认为 False。

- symlinks - 一个布尔值，指示是否尝试符号链接 Python 二进制文件（以及任何必要的 DLL 或其他二进制文件，例如 pythonw.exe），而不是复制。默认为 False。

实例化的 env-builder 有一个 create 方法，它将目标目录的路径（绝对或相对于当前目录）作为必需参数，该目录将包含虚拟环境。create 方法要么在指定目录中创建环境，要么引发适当的异常。

venv 模块还提供了一个模块级 create 函数作为便利：

```python
def create(env_dir,
    system_site_packages=False, clear=False, use_symlinks=False):
    builder = EnvBuilder(
        system_site_packages=system_site_packages,
        clear=clear,
        use_symlinks=use_symlinks)
    builder.create(env_dir)
```

第三方虚拟环境工具的创建者可以自由使用提供的 EnvBuilder 类作为基类。

EnvBuilder 类的 create 方法说明了可用的自定义钩子：

```python
def create(self, env_dir):
    """
    在目录中创建虚拟化的 Python 环境。

    :param env_dir: 要创建环境的目标目录。

    """
    env_dir = os.path.abspath(env_dir)
    context = self.create_directories(env_dir)
    self.create_configuration(context)
    self.setup_python(context)
    self.post_setup(context)
```

create_directories、create_configuration、setup_python 和 post_setup 的每个方法都可以被覆盖。这些方法的函数是：

- create_directories - 创建环境目录和所有必要的目录，并返回一个上下文对象。这只是属性（如路径）的容器，供其他方法使用。

- create_configuration - 在环境中创建 pyvenv.cfg 配置文件。

- setup_python - 在环境中创建 Python 可执行文件的副本（以及在 Windows 下，DLL）。

- post_setup - 一个（默认无操作）钩子方法，可以在第三方子类中被覆盖，以在虚拟环境中预安装包或安装脚本。

此外，EnvBuilder 提供了一个实用方法，可以从子类中的 post_setup 调用，以帮助将自定义脚本安装到虚拟环境中。install_scripts 方法接受上下文对象（见上文）和目录路径作为参数。该目录应包含子目录"common"、"posix"、"nt"，每个包含用于环境中的 bin 目录的脚本。在对占位符进行一些文本替换后，复制"common"和对应于 os.name 的目录的内容：

- `__VENV_DIR__` 被替换为环境目录的绝对路径。

- `__VENV_NAME__` 被替换为环境名称（环境目录的最终路径段）。

- `__VENV_BIN_NAME__` 被替换为 bin 目录的名称（bin 或 Scripts）。

- `__VENV_PYTHON__`被替换为环境的可执行文件的绝对路径。

参考实现中的 DistributeEnvBuilder 子类说明了如何在实践中使用自定义钩子预安装 Distribute 到虚拟环境中。不打算将 DistributeEnvBuilder 实际添加到 Python 核心，但它使参考实现更立即可用于测试和探索目的。

## 向后兼容性

### 分割 sys.prefix 的含义

任何此类虚拟环境工具（试图隔离 site-packages，同时仍然使用基础 Python 的标准库而无需将其符号链接到虚拟环境）都提出了分割当前都包含在 sys.prefix 中的两个不同含义（以及其他含义）：问题"标准库在哪里？"和"应该安装第三方模块的 site-packages 位置在哪里？"的答案。

这种分割可以通过为前者前缀或后者前缀引入新的 sys 属性来处理。任一选项都可能引入一些与编写为假设 sys.prefix 的其他含义的软件向后不兼容。（此类软件最好使用 site 和 sysconfig 模块中的 API 来回答这些问题，而不是直接使用 sys.prefix，在这种情况下没有向后兼容问题，但实际上 sys.prefix 有时被使用。）

sys.prefix 的文档将其描述为"一个字符串，给出平台无关的 Python 文件安装的特定于站点的目录前缀"，并特别提到标准库和头文件在 sys.prefix 下找到。它没有提到 site-packages。

保持此文档定义意味着让 sys.prefix 指向基础系统安装（这是找到标准库和头文件的地方），并在 sys 中引入一个新值（类似于 sys.site_prefix）指向 site-packages 的前缀。这将保持 sys.prefix 的文档语义，但如果第三方代码使用 sys.prefix 而不是 sys.site_prefix 或适当的 site API 来查找 site-packages 目录，则可能会破坏隔离。

最显著的案例可能是 setuptools 及其 fork distribute，它们主要使用 distutils 和 sysconfig API，但直接使用 sys.prefix 来构建站点目录列表，用于预飞行检查 pth 文件可以有用放置的地方。

否则，Google Code Search 显示出使用 sys.prefix 构建 site-packages 路径的包和使用它来例如从代码执行跟踪中消除标准库的包之间大致均匀的使用混合。

虽然它需要修改 sys.prefix 的文档定义，但本 PEP 更喜欢让 sys.prefix 指向虚拟环境（找到 site-packages 的地方），并引入 sys.base_prefix 指向标准库和 Python 头文件。选择此选项的理由：

- 最好倾向于虚拟环境的更大隔离。

- Virtualenv 已经修改 sys.prefix 指向虚拟环境，在实践中这没有问题。

- 不需要对 setuptools/distribute 进行修改。

### 对其他 Python 实现的影响

本 PEP 的大多数更改发生在标准库中，它被其他 Python 实现共享，不应造成任何问题。

其他 Python 实现需要复制解释器引导的新 sys.prefix 查找行为，包括定位和解析 pyvenv.cfg 文件（如果存在）。
