## 摘要

本 PEP 提议在标准库中添加一个新的目录迭代函数 `os.scandir()`。这个新函数增加了有用的功能，并通过在大多数情况下避免调用 `os.stat()` 使 `os.walk()` 的速度提高了 2-20 倍（取决于平台和文件系统）。

## 理由

Python 内置的 `os.walk()` 比它需要的要慢得多，因为 – 除了在每个目录上调用 `os.listdir()` 之外 – 它还需要执行 `stat()` 系统调用或 `GetFileAttributes()` 来确定条目是否为目录。

但底层的系统调用 – Windows 上的 `FindFirstFile` / `FindNextFile` 和 POSIX 系统上的 `readdir` – 已经告诉您返回的文件是否为目录，因此不需要进一步的系统调用。此外，Windows 系统调用返回目录条目的 `stat_result` 对象的所有信息，如文件大小和最后修改时间。

简而言之，您可以将像 `os.walk()` 这样的树函数所需的系统调用数量从大约 2N 减少到 N，其中 N 是树中文件和目录的总数。（而且由于目录树通常宽于深，所以效果通常比这好得多。）

在实践中，删除所有这些额外的系统调用使 `os.walk()` **在 Windows 上快约 8-9 倍**，在 **POSIX 系统上快约 2-3 倍**。所以我们不是在谈论微优化。参见更多 [基准测试](https://github.com/benhoyt/scandir#benchmarks)。

有些相关的是，许多人（见 Python [问题 11406](http://bugs.python.org/issue11406)）也希望有一个版本的 `os.listdir()`，在迭代时产生文件名，而不是将它们作为一个大列表返回。这提高了遍历非常大的目录的内存效率。

因此，除了提供一个可以直接调用的 `scandir()` 迭代器函数外，Python 现有的 `os.walk()` 函数也可以大幅提速。

## 提议的具体内容

### os.scandir()

具体而言，本 PEP 提议向标准库中的 `os` 模块添加一个函数 `scandir`，它接受一个可选的字符串参数：

```
scandir(path='.') -> 生成 DirEntry 对象的生成器
```

与 `listdir` 一样，`scandir` 调用操作系统的目录迭代系统调用来获取给定 `path` 中的文件名，但它与 `listdir` 有两个不同之处：

- 它不返回纯文件名字符串，而是返回轻量级的 `DirEntry` 对象，这些对象持有文件名字符串，并提供简单的方法来访问操作系统可能返回的附加数据。
- 它返回一个生成器而不是列表，因此 `scandir` 作为真正的迭代器，而不是立即返回完整的列表。

`scandir()` 为 `path` 中的每个文件和子目录产生一个 `DirEntry` 对象。与 `listdir` 一样，`'.'` 和 `'..'` 伪目录被跳过，条目按系统相关顺序产生。每个 `DirEntry` 对象具有以下属性和方法：

- `name`：条目的文件名，相对于 `scandir` 的 `path` 参数（对应于 `os.listdir` 的返回值）
- `path`：条目的完整路径名（不一定是绝对路径）– 相当于 `os.path.join(scandir_path, entry.name)`
- `inode()`：返回条目的 inode 号。结果缓存在 `DirEntry` 对象上，使用 `os.stat(entry.path, follow_symlinks=False).st_ino` 获取最新信息。在 Unix 上，不需要系统调用。
- `is_dir(*, follow_symlinks=True)`：类似于 `pathlib.Path.is_dir()`，但返回值缓存在 `DirEntry` 对象上；大多数情况下不需要系统调用；如果 `follow_symlinks` 为 False 则不跟随符号链接
- `is_file(*, follow_symlinks=True)`：类似于 `pathlib.Path.is_file()`，但返回值缓存在 `DirEntry` 对象上；大多数情况下不需要系统调用；如果 `follow_symlinks` 为 False 则不跟随符号链接
- `is_symlink()`：类似于 `pathlib.Path.is_symlink()`，但返回值缓存在 `DirEntry` 对象上；大多数情况下不需要系统调用
- `stat(*, follow_symlinks=True)`：类似于 `os.stat()`，但返回值缓存在 `DirEntry` 对象上；在 Windows 上不需要系统调用（符号链接除外）；如果 `follow_symlinks` 为 False 则不跟随符号链接（类似于 `os.lstat()`）

所有*方法*在某些情况下可能执行系统调用，因此可能引发 `OSError` – 详见“异常处理说明”部分。

`DirEntry` 属性和方法名称尽可能与新的 `pathlib` 模块中的名称相同，以保持一致性。功能上的唯一区别是 `DirEntry` 方法在首次调用后将值缓存在条目对象上。

与 `os` 模块中的其他函数一样，`scandir()` 接受 bytes 或 str 对象作为 `path` 参数，并返回与 `path` 类型相同的 `DirEntry.name` 和 `DirEntry.path` 属性。但是，*强烈建议*使用 str 类型，因为这确保了跨平台对 Unicode 文件名的支持。（在 Windows 上，bytes 文件名自 Python 3.3 起已被弃用）。

### os.walk()

作为本提议的一部分，`os.walk()` 也将被修改为使用 `scandir()` 而不是 `listdir()` 和 `os.path.isdir()`。这将非常显著地提高 `os.walk()` 的速度（如上所述，提高 2-20 倍，取决于系统）。

## 示例

首先，一个非常简单的 `scandir()` 示例，展示 `DirEntry.name` 属性和 `DirEntry.is_dir()` 方法的使用：

```python
def subdirs(path):
    """在给定路径下产生不以'.'开头的目录名。"""
    for entry in os.scandir(path):
        if not entry.name.startswith('.') and entry.is_dir():
            yield entry.name
```

在 Windows 和 POSIX 系统上，尤其是中等大小或大型目录上，使用 `scandir` 的 `subdirs()` 函数将比使用 `os.listdir()` 和 `os.path.isdir()` 快得多。

或者，对于获取目录树中文件的总大小，展示 `DirEntry.stat()` 方法和 `DirEntry.path` 属性的使用：

```python
def get_tree_size(path):
    """返回给定路径和子目录中文件的总大小。"""
    total = 0
    for entry in os.scandir(path):
        if entry.is_dir(follow_symlinks=False):
            total += get_tree_size(entry.path)
        else:
            total += entry.stat(follow_symlinks=False).st_size
    return total
```

这也展示了 `is_dir()` 的 `follow_symlinks` 参数的使用 – 在这样的递归函数中，我们可能不想跟随链接。（要在这样的递归函数中正确跟随链接，我们需要对跟随符号链接导致递归循环的情况进行特殊处理。）

注意，`get_tree_size()` 在 Windows 上将获得巨大的速度提升，因为不需要额外的 stat 调用，但在 POSIX 系统上，目录迭代函数不返回大小信息，所以这个函数在那里不会获得任何提升。

### 关于缓存的说明

`DirEntry` 对象相对简单 – `name` 和 `path` 属性显然总是被缓存，`is_X` 和 `stat` 方法缓存它们的值（在 Windows 上通过 `FindNextFile` 立即缓存，在 POSIX 系统上通过 `stat` 系统调用在首次使用时缓存），并且从不重新从系统获取。

因此，`DirEntry` 对象旨在在迭代后使用并丢弃，而不是存储在长期存在的数据结构中并反复调用方法。

如果开发者需要“刷新”行为（例如，用于监视文件大小变化），他们可以直接使用 `pathlib.Path` 对象，或调用常规的 `os.stat()` 或 `os.path.getsize()` 函数，这些函数每次调用都从操作系统获取新鲜数据。

### 关于异常处理的说明

`DirEntry.is_X()` 和 `DirEntry.stat()` 明确是方法而不是属性或特性，这是为了清楚表明它们可能不是廉价操作（尽管通常是的），并且它们可能执行系统调用。因此，这些方法可能引发 `OSError`。

例如，`DirEntry.stat()` 将在基于 POSIX 的系统上始终执行系统调用，如果 `readdir()` 不支持 `d_type` 或返回值为 `DT_UNKNOWN` 的 `d_type`（这在某些条件下或某些文件系统上可能发生），则 `DirEntry.is_X()` 方法将在这样的系统上执行 `stat()` 系统调用。

通常这并不重要 – 例如，标准库中定义的 `os.walk()` 只捕获 `listdir()` 调用周围的错误。

此外，由于 `DirEntry.is_X` 方法的异常引发行为与 `pathlib` 相匹配 – `pathlib` 仅在权限或其他致命错误的情况下引发 `OSError`，但如果路径不存在或是损坏的符号链接则返回 False – 因此通常不需要在 `is_X()` 调用周围捕获错误。

但是，当用户需要细粒度的错误处理时，可能希望在所有方法调用周围捕获 `OSError` 并适当处理。

例如，下面是上面所示的 `get_tree_size()` 示例的版本，但添加了细粒度的错误处理：

```python
def get_tree_size(path):
    """返回路径和子目录中文件的总大小。如果
    is_dir() 或 stat() 失败，将错误消息打印到 stderr
    并假定为零大小（例如，文件已被删除）。
    """
    total = 0
    for entry in os.scandir(path):
        try:
            is_dir = entry.is_dir(follow_symlinks=False)
        except OSError as error:
            print('Error calling is_dir():', error, file=sys.stderr)
            continue
        if is_dir:
            total += get_tree_size(entry.path)
        else:
            try:
                total += entry.stat(follow_symlinks=False).st_size
            except OSError as error:
                print('Error calling stat():', error, file=sys.stderr)
    return total
```

## 被否决的想法

### 命名

这个函数名的唯一其他真正的竞争者是 `iterdir()`。然而，Python 中的 `iterX()` 函数（主要存在于 Python 2 中）倾向于是非迭代器对应物的简单迭代器版本。例如，`dict.iterkeys()` 只是 `dict.keys()` 的迭代器版本，但返回的对象是相同的。而在 `scandir()` 的情况下，返回值是完全不同的对象（`DirEntry` 对象 vs 文件名字符串），因此这应该通过名称上的差异来反映 – 因此是 `scandir()`。

参见 [python-dev 上的相关讨论](https://mail.python.org/pipermail/python-dev/2014-June/135228.html)。

### 通配符支持

Windows 上的 `FindFirstFile`/`FindNextFile` 支持传递“通配符”如 `*.jpg`，所以一开始大家（包括本 PEP 的作者）觉得将 `windows_wildcard` 关键字参数包含到 `scandir` 函数中是个好主意，这样用户可以传入。

然而，经过进一步的思考和讨论，决定这是一个坏主意，*除非它可以实现跨平台*（`pattern` 关键字参数或类似的东西）。这看起来很容易 – 在 Windows 上使用 OS 通配符支持，在 POSIX 系统上使用 `fnmatch` 或 `re` 之类的东西。

不幸的是，确切的 Windows 通配符匹配规则在微软的任何地方都没有真正记录，而且它们相当古怪（见这个 [博客文章](http://blogs.msdn.com/b/oldnewthing/archive/2007/12/17/6785519.aspx)），这意味着使用 `fnmatch` 或正则表达式模拟非常成问题。

所以共识是 Windows 通配符支持是个坏主意。如果以后有跨平台的方式实现它，可以稍后添加，但初始版本不行。

阅读更多关于 [2012 年 11 月的 python-ideas 线程](https://mail.python.org/pipermail/python-ideas/2012-November/017770.html) 和 [2014 年 6 月关于 PEP 471 的 python-dev 线程](https://mail.python.org/pipermail/python-dev/2014-June/135217.html)。

### 方法默认不跟随符号链接

在 python-dev 上有大量关于 `DirEntry` 方法是否应该跟随符号链接的争论（当 `is_X()` 方法没有 `follow_symlinks` 参数时）（见 [这个线程](https://mail.python.org/pipermail/python-dev/2014-July/135485.html) 中的消息）。

最初它们不跟随（参见本 PEP 的先前版本和 scandir.py 模块），但 Victor Stinner 在 python-dev 上提出了一个相当有说服力的论点，认为默认跟随符号链接是个更好的主意，因为：

- 跟随链接通常是你想要的（在标准库 92% 的情况下，使用 `os.listdir()` 和 `os.path.isdir()` 的函数都会跟随符号链接）
- 这是类似函数 `os.path.isdir()` 和 `pathlib.Path.is_dir()` 确立的先例，因此否则会很令人困惑
- 使用不跟随链接的方法，如果你想跟随链接，你就得写类似 `if (entry.is_symlink() and os.path.isdir(entry.path)) or entry.is_dir()` 的东西，这很笨拙

作为一个证明不跟随符号链接版本容易出错的例子，本 PEP 的作者在他的 `scandir.walk()` 初始实现中有一个 bug，就是由这个完全测试错误引起的（见 [这里的 Issue #4](https://github.com/benhoyt/scandir/issues/4)）。

最终，并没有完全达成一致认为方法应该跟随符号链接，但在最参与的参与者中存在基本共识，本 PEP 的作者认为上述论点足够有力，足以证明默认跟随符号链接是合理的。

此外，如果希望其他行为， straightforward 地调用相关方法并传入 `follow_symlinks=False` 即可。

### DirEntry 属性是特性

在某些方面，`DirEntry` 的 `is_X()` 和 `stat()` 作为特性而不是方法会更好，以表明它们非常便宜或免费。然而，事实并非完全如此，因为 `stat()` 在基于 POSIX 的系统上需要 OS 调用，但在 Windows 上不需要。甚至 `is_dir()` 等也可能在 POSIX 系统上执行 OS 调用，如果 `dirent.d_type` 值是 `DT_UNKNOWN`（在某些文件系统上）。

此外，人们会期望属性访问 `entry.is_dir` 只引发 `AttributeError`，而不是在底层执行系统调用时引发 `OSError`。调用代码必须在看起来像简单属性访问的东西周围有 `try`/`except`，因此将它们做成*方法*要好得多。

参见 [2013 年 5 月的 python-dev 线程](https://mail.python.org/pipermail/python-dev/2013-May/126184.html)，其中本 PEP 作者提出这个论点并得到了核心开发者的同意。

### DirEntry 字段是“静态”属性只有对象

在 [2014 年 7 月的 python-dev 消息](https://mail.python.org/pipermail/python-dev/2014-July/135303.html) 中，Paul Moore 提出了一个解决方案，即“OS 功能的薄包装”，其中 `DirEntry` 对象只有静态属性：`name`、`path` 和 `is_X`，`st_X` 属性仅在 Windows 上存在。想法是使用这个更简单、更低级的函数作为高级函数的构建块。

起初，大家普遍同意这样简化是件好事。然而，这种方法有两个问题。首先，假设 `is_dir` 和类似属性在 POSIX 上总是存在，事实并非如此（如果 `d_type` 不存在或是 `DT_UNKNOWN`）。其次，它在实践中是一个更难用的 API，因为甚至 `is_dir` 属性在 POSIX 上也不总是存在，需要用 `hasattr()` 测试，如果不存在则调用 `os.stat()`。

参见 [2014 年 7 月的 python-dev 回复](https://mail.python.org/pipermail/python-dev/2014-July/135312.html)，本 PEP 作者详细说明了为什么这个选项不是理想解决方案，以及随后 Paul Moore 的回复表示同意。

### DirEntry 字段是静态的，带有 ensure_lstat 选项

另一个看似更简单和有吸引力的选项由 Alyssa Coghlan 在 [2014 年 6 月的 python-dev 消息](https://mail.python.org/pipermail/python-dev/2014-June/135261.html) 中提出：使 `DirEntry.is_X` 和 `DirEntry.lstat_result` 成为特性，并在迭代时填充 `DirEntry.lstat_result`，但仅当在 `scandir()` 调用中指定了新参数 `ensure_lstat=True` 时。

这确实比上面有优势，因为如果你需要 stat 结果，你可以轻松地从 `scandir()` 获取。然而，它有一个严重的缺点，即细粒度错误处理很混乱，因为 `stat()` 将在迭代期间被调用（因此可能引发 `OSError`），导致相当丑陋的手工迭代循环：

```python
it = os.scandir(path)
while True:
    try:
        entry = next(it)
    except OSError as error:
        handle_error(path, error)
    except StopIteration:
        break
```

或者这意味着 `scandir()` 必须接受一个 `onerror` 参数 – 一个在迭代期间 `stat()` 出错时调用的函数。这对本 PEP 的作者来说，似乎不如在 `DirEntry.stat()` 调用周围使用 `try`/`except` 那么直接和 Pythonic。

另一个缺点是 `os.scandir()` 是为了让代码更快而编写的。在 POSIX 上总是调用 `os.lstat()` 不会带来任何速度提升。在大多数情况下，你不需要完整的 `stat_result` 对象 – `is_X()` 方法就足够了，而且这些信息已经知道了。

参见 [Ben Hoyt 2014 年 7 月的回复](https://mail.python.org/pipermail/python-dev/2014-July/135312.html)，总结了这一点，并详细说明了为什么他认为原始的 [PEP 471](../pep-0471/ "PEP 471 – os.scandir() 函数 – 一个更好更快速的目录迭代器") 提议终究是“正确的那个”。

### 返回值是 (name, stat_result) 二元组

最初，本 PEP 的作者将这个概念提议为一个名为 `iterdir_stat()` 的函数，它产生 (name, stat_result) 二元组。这确实有一个优势，即没有引入新类型。然而，在基于 POSIX 的系统上，`stat_result` 只部分填充（大多数字段设置为 `None` 和其他怪异之处），所以它们根本不是 `stat_result` 对象，这将需要彻底记录为不同于 `os.stat()`。

此外，Python 对具有属性和方法的适当对象有很好的支持，这比二元组提供了一个更合理和更简单的 API。它也使 `DirEntry` 对象更具可扩展性和面向未来，因为操作系统添加功能而我们想要在 `DirEntry` 中包含这些功能时。

参见之前的一些讨论：

- [2013 年 5 月的 python-dev 线程](https://mail.python.org/pipermail/python-dev/2013-May/126148.html)，其中 Alyssa Coghlan 提出了 `DirEntry` 风格对象的原始论点。
- [2014 年 6 月的 python-dev 线程](https://mail.python.org/pipermail/python-dev/2014-June/135244.html)，其中 Alyssa Coghlan 提出了（另一个）反对二元组方法的好论点。

### 返回值是重载的 stat_result 对象

另一个讨论过的替代方案是使返回值成为带有 `name` 和 `path` 属性的重载 `stat_result` 对象。然而，除了这是一种奇怪（且牵强！）的重载之外，它还有上面提到的相同问题 – `stat_result` 的大部分信息在 POSIX 系统上不会被 `readdir()` 获取，只有（部分的）`st_mode` 值。

### 返回值是 pathlib.Path 对象

随着 Antoine Pitrou 的新标准库 `pathlib` 模块，`scandir()` 返回 `pathlib.Path` 实例最初看起来是个很棒的主意。然而，`pathlib.Path` 的 `is_X()` 和 `stat()` 函数明确不缓存，而 `scandir` 必须按设计缓存它们，因为它（经常）从原始目录迭代系统调用返回值。

如果 `scandir` 返回的 `pathlib.Path` 实例缓存了 stat 值，但普通的 `pathlib.Path` 对象明确不缓存，那将有点令人困惑。

Guido van Rossum 明确拒绝了在 scandir 上下文中 `pathlib.Path` 缓存 stat [在这里](https://mail.python.org/pipermail/python-dev/2013-November/130583.html)，使得 `pathlib.Path` 对象成为 scandir 返回值的糟糕选择。

## 可能的改进

可以对 scandir 进行许多可能的改进，但这里是本 PEP 作者心中的一些简短列表：

- scandir 可能通过在每个 `Py_BEGIN_ALLOW_THREADS` 块中调用 `readdir` / `FindNextFile` 约 50 次来进一步加速，这样它可以在 C 扩展模块中停留更长时间，因此可能稍快一些。这种方法还没有测试过，但是由 Antoine Pitrou 在 Issue 11406 中建议的。[来源 9](http://bugs.python.org/msg130125)
- scandir 可以使用一个自由列表来避免每次迭代的内存分配成本 – 一个短的自由列表，10 个或甚至 1 个可能会有帮助。由 Victor Stinner 在 [6 月 27 日的 python-dev 线程](https://mail.python.org/pipermail/python-dev/2014-June/135232.html) 中建议。
