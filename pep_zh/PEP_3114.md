# PEP 3114 – 将 `iterator.next()` 重命名为 `iterator.__next__()`

**Metadata:**

- **Author**: Ka-Ping Yee <ping&#32;&#97;t&#32;zesty.ca>
- **Created**: 04-Mar-2007
- **Python-Version**: 3.0

## 摘要

Python 2.x 中的迭代器协议由两个方法组成：在可迭代对象上调用的__iter__() 用于生成迭代器，以及在迭代器对象上调用的 next() 用于生成序列中的下一个项。使用 for 循环遍历可迭代对象隐式调用这两个方法。本 PEP 提议将 next 方法重命名为__next__，与 Python 中所有其他协议一致，在这些协议中，方法作为语言级协议的一部分被隐式调用，并且引入一个名为 next 的内置函数来调用__next__方法，与显式调用其他协议的方式一致。

## 双下划线名称

在 Python 中，名称前后的双下划线用于区分属于语言本身的名称。解释器隐式使用或创建的属性和方法采用此命名约定；一些示例是：

- `__file__` - 由解释器自动创建的属性

- `__dict__` - 对解释器具有特殊含义的属性

- `__init__` - 由解释器隐式调用的方法

注意，此约定适用于由程序员显式定义的方法（如__init__），以及只能通过显式命名访问的属性（如__file__），因此它包括由解释器使用或创建的名称。

（并非所有被称为"协议"的东西都是由具有双下划线名称的方法组成的。例如，`__contains__`方法有双下划线，因为语言结构 x in y 隐式调用`__contains__`。但即使 read 方法是文件协议的一部分，它也没有双下划线，因为没有语言结构隐式调用 x.read()。）

双下划线的使用为 Python 语言定义的一部分名称创建了一个单独的命名空间，因此程序员可以自由创建以字母开头的变量、属性和方法，而无需担心与具有语言定义目的的名称静默冲突。（与保留关键字冲突仍然是一个问题，但至少这将立即产生语法错误。）

迭代器上的 next 方法的命名是此约定的一个例外。从不包含对 next 方法的显式调用的代码仍然可能受到此类方法存在的静默影响。因此，本 PEP 提议迭代器应该具有__next__方法而不是 next 方法（语义没有变化）。

## 双下划线方法和内置函数

Python 语言定义了几个通过定义具有双下划线名称的方法来实施或自定义的协议。在每种情况下，协议都是由作为解释器中的 C 函数实现的内部方法提供的。对于在 Python 中定义的对象，此 C 函数通过隐式调用具有双下划线名称的 Python 方法来支持自定义（它通常除了调用 Python 方法之外还会做一些额外的工作。）

有时协议由语法结构调用：

```
- x[y] –> 内部 tp_getitem –> x.__getitem__(y)

- x + y –> 内部 nb_add –> x.__add__(y)

- -x –> 内部 nb_negative –> x.__neg__()
```

有时没有语法结构，但仍然能够显式调用协议是有用的。对于这种情况，Python 提供了一个同名的内置函数，但没有双下划线。

```
- len(x) –> 内部 sq_length –> x.__len__()

- hash(x) –> 内部 tp_hash –> x.__hash__()

- iter(x) –> 内部 tp_iter –> x.__iter__()
```

按照这个模式，处理 next 的自然方法是添加一个 next 内置函数，其行为完全相同。

```
- next(x) –> 内部 tp_iternext –> x.__next__()
```

此外，建议 next 内置函数接受一个哨兵值作为可选的第二个参数，遵循 getattr 和 iter 内置函数的风格。当使用两个参数调用时，next 捕获 StopIteration 异常并返回哨兵值，而不是传播异常。这在 iter 和 next 之间创建了一个很好的对偶性：

> iter(function, sentinel) <–> next(iterator, sentinel)

## 之前的提议

这个提议不是一个新想法。这里提出的想法得到了 BDFL 在 python-dev 上的支持 [1]，甚至在原始迭代器 PEP PEP 234 中提到：

（回顾一下，采用__next__() 并有一个新的内置函数 next(it)，它调用 `it.__next__()` 可能更好。但唉，太迟了；这已经在 2001 年 12 月以来部署在 Python 2.2 中。）

## 反对意见

对添加更多内置函数有一些反对意见。特别是，Martin von Loewis 写道 [2]：

我不喜欢引入更多的内置函数，除非它们具有真正的通用性（即可能在许多程序中需要）。对于这一个，我认为__next__的正常用法是与 for 循环一起，所以我认为人们不会经常需要显式的 next() 调用。

也不是大多数协议都通过内置函数显式调用。相反，大多数协议可以通过 operator 模块中的方法显式调用。因此，按照传统，它应该是 operator.next。

...

作为替代，我提议 object 增长一个.next() 方法，默认调用__next__。

## 过渡计划

将向 2to3 转换工具 [3] 添加两个额外的转换：

- 名为 next 的方法定义将重命名为__next__。

- 对 next 方法的显式调用将被替换为对内置 next 函数的调用。例如，x.next() 将变为 next(x)。

Collin Winter 研究了根据模块级绑定到 next 的存在自动决定是否执行第二个转换的可能性 [4]，发现它"丑陋且慢"。相反，转换工具将在检测到此类绑定时发出警告。Collin 已为以下条件提出警告 [5]：

- 对 next 的模块级赋值。

- 名为 next 的函数的模块级定义。

- next 名称的模块级导入。

- 对__builtin__.next 的赋值。

## 批准

本 PEP 于 2007 年 3 月 6 日由 Guido 接受 [6]。
