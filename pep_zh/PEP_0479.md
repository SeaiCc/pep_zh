# PEP 479 – 修改生成器中 StopIteration 的处理

**元数据：**

- **作者**: Chris Angelico <rosuav&#32;&#97;t&#32;gmail.com>，Guido van Rossum <guido&#32;&#97;t&#32;python.org>
- **创建日期**: 2014 年 11 月 15 日
- **Python 版本**: 3.5
- **发布历史**: 2014 年 11 月 15 日，2014 年 11 月 19 日，2014 年 12 月 5 日

## 摘要

本 PEP 提议对生成器进行一项更改：当 StopIteration 在生成器内部被引发时，它将被替换为 RuntimeError。（更准确地说，当异常即将从生成器的堆栈帧冒泡出来时发生这种情况。）由于此更改向后不兼容，该功能最初使用__future__语句引入。

## 接受

本 PEP 于 11 月 22 日被 BDFL 接受。由于从初稿到接受的异常短的时间，接受后提出的主要反对意见被认真考虑并反映在下面的"替代提案"部分中。然而，讨论都没有改变 BDFL 的想法，PEP 的接受现在是最终的。（对于澄清编辑的建议仍然受欢迎——与 IETF RFC 不同，PEP 的文本在接受后不是不可更改的，尽管核心设计/计划/规范在接受后不应更改。）

## 理由

生成器和 StopIteration 的交互目前有些令人惊讶，可能隐藏难以察觉的 bug。意外的异常不应导致微妙改变的行为，而应该导致嘈杂且易于调试的跟踪回溯。目前，在生成器函数内意外引发的 StopIteration 将被驱动生成器的循环结构解释为迭代的结束。

本提案的主要目标是简化调试情况，其中未加保护的 next() 调用（可能在几个堆栈帧深处）引发 StopIteration 并导致生成器控制的迭代静默终止。（相比之下，当引发其他异常时，会打印跟踪回溯，指出问题的原因。）

这在结合 PEP 380 的 yield from 构造时尤为有害，因为它破坏了子生成器可以从生成器中因子化的抽象。该 PEP 指出了这一限制，但指出"这些的用例很少到不存在"。不幸的是，虽然故意使用很少，但很容易偶然遇到这些情况：

```python
import contextlib

@contextlib.contextmanager
def transaction():
    print('begin')
    try:
        yield from do_it()
    except:
        print('rollback')
        raise
    else:
        print('commit')

def do_it():
    print('Refactored initial setup')
    yield # with 语句的主体在这里执行
    print('Refactored finalization of successful transaction')

def gene():
    for i in range(2):
        with transaction():
            yield i
    # return
    raise StopIteration # 这是错误的
    print('Should not be reached')

for i in gene():
    print('main: i =', i)
```

这里将 do_it 因子化为子生成器引入了一个微妙的 bug：如果包装的块引发 StopIteration，在当前行为下，此异常将被上下文管理器吞没；更糟的是，最终化被静默跳过！类似的问题行为发生在 asyncio 协程引发 StopIteration 时，导致它静默终止，或者当 next 用于从不期望为空的迭代器中获取第一个结果时，例如：

```python
# 使用上面相同的上下文管理器
import pathlib

with transaction():
    print('commit file {}'.format(
        # 我从不记得 README 扩展名是什么
        next(pathlib.Path('/some/dir').glob('README*'))))
```

在这两种情况下，yield from 的重构抽象都在客户端代码中存在 bug 的情况下失效。

此外，该提案减少了列表推导式和生成器表达式之间的差异，防止诸如启动此讨论的意外情况 [2]。此后，以下语句如果产生任何结果将产生相同的结果：

```python
a = list(F(x) for x in xs if P(x))
a = [F(x) for x in xs if P(x)]
```

在当前状态下，可以编写一个函数 F(x) 或谓词 P(x)，导致第一种形式产生一个（截断的）结果，而第二种形式引发异常（即 StopIteration）。通过提议的更改，两种形式都将在此点引发异常（尽管第一种情况是 RuntimeError，第二种是 StopIteration）。

最后，该提案还澄清了如何终止生成器的混淆：正确的方式是 return，不是 raise StopIteration。

作为一个额外的优点，上述更改使生成器函数更多与常规函数保持一致。如果你想将一段作为生成器呈现的代码转换为其他东西，你通常可以通过将每个 yield 替换为 print() 或 list.append() 调用来相当简单地做到这一点；但是，如果代码中有任何裸露的 next() 调用，你必须意识到它们。如果代码最初编写时不依赖 StopIteration 终止函数，转换将更容易。

## 背景信息

当生成器帧作为__next__()（或 send() 或 throw()）调用的结果而（重新）启动时，可能发生三种结果之一：

- 达到 yield 点，返回 yield 的值。
- 帧被返回；引发 StopIteration。
- 引发异常，它冒泡出来。

在后两种情况下，帧被放弃（生成器对象的 gi_frame 属性被设置为 None）。

## 提案

如果 StopIteration 即将从生成器帧冒泡出来，它被替换为 RuntimeError，这导致 next() 调用（它调用了生成器）失败，传递该异常出去。从那时起，它就像任何旧异常一样。[3]

这影响上面列出的第三种结果，不改变任何其他效果。此外，它仅影响此结果当引发的异常是 StopIteration（或其子类）时。

注意，提议的替换发生在异常即将从帧冒泡出来的点，即在任何 except 或 finally 块可能影响它之后退出。由从帧返回引发的 StopIteration 不受影响（要点是 StopIteration 意味着生成器"正常"终止，即它没有引发异常）。

一个微妙的问题是如果调用者捕获了 RuntimeError，再次调用生成器对象的__next__() 方法会发生什么。答案是从此点开始它将引发 StopIteration——行为与生成器引发任何其他异常时相同。

提案的另一个逻辑后果：如果有人使用 g.throw(StopIteration) 将 StopIteration 异常抛入生成器，如果生成器不捕获它（它可以使用 yield 周围的 try/except 来做到这一点），它将被转换为 RuntimeError。

在过渡阶段，新功能必须使用以下每模块启用：

```python
from __future__ import generator_stop
```

在此指令影响下构造的任何生成器函数将在其代码对象上设置 REPLACE_STOPITERATION 标志，设置该标志的生成器将根据此提案行事。一旦该功能成为标准，标志可能会被丢弃；代码不应检查生成器的标志。

已创建一个概念验证补丁以促进测试。[4]

## 对现有代码的影响

此更改将影响依赖 StopIteration 冒泡的现有代码。pure Python 参考实现中的 groupby [5] 目前有注释"退出 StopIteration"，其中期望异常将传播然后被处理。这将是不寻常的，但不是未知的，这样的结构将失败。其他例子比比皆是，例如 [6]，[7]。

（Alyssa Coghlan 评论："如果你想要因子化一个辅助函数来终止生成器，你必须做"return yield from helper()"而不是仅仅"helper()"。"）

还有生成器表达式的例子依赖表达式、目标或谓词（而不是 for 循环本身隐含的__next__() 调用）引发的 StopIteration。

### 编写向后和向前兼容的代码

除了提升 StopIteration 来退出生成器表达式的 hack，编写在旧版 Python 版本和新语义下同样有效的代码很容易。

这是通过将生成器体中期望 StopIteration 的地方（例如裸露的 next() 调用或在某些情况下期望引发 StopIteration 的辅助函数）封装在 try/except 构造中，当 StopIteration 被引发时返回。try/except 构造应直接出现在生成器函数中；在本身不是生成器的辅助函数中这样做不起作用。如果 raise StopIteration 直接出现在生成器中，只需将其替换为 return。

### 破坏示例

明确引发 StopIteration 的生成器通常可以更改为简单地 return 代替。这将与所有现有 Python 版本兼容，并且不受__future__影响。这里是标准库的一些说明。

Lib/ipaddress.py：

```python
if other == self:
    raise StopIteration
```

变为：

```python
if other == self:
    return
```

在某些情况下，这可以与 yield from 结合简化代码，例如 Lib/difflib.py：

```python
if context is None:
    while True:
        yield next(line_pair_iterator)
```

变为：

```python
if context is None:
    yield from line_pair_iterator
    return
```

（return 对于严格等价的转换是必要的，尽管在这个特定文件中，没有进一步的代码，可以省略 return。）对于与预 3.3 版本的 Python 兼容，这可以写成显式的 for 循环：

```python
if context is None:
    for line in line_pair_iterator:
        yield line
    return
```

更复杂的迭代模式需要显式的 try/except 构造。例如，一个假设的解析器像这样：

```python
def parser(f):
    while True:
        data = next(f)
        while True:
            line = next(f)
            if line == "- end -": break
            data += line
        yield data
```

需要重写为：

```python
def parser(f):
    while True:
        try:
            data = next(f)
            while True:
                line = next(f)
                if line == "- end -": break
                data += line
            yield data
        except StopIteration:
            return
```

或者可能：

```python
def parser(f):
    for data in f:
        while True:
            line = next(f)
            if line == "- end -": break
            data += line
        yield data
```

后一种形式通过声称用 for 循环迭代文件来掩盖迭代，但在循环体内也从同一个迭代器获取更多数据。它确实清楚地区分"正常"终止（初始行之后的 StopIteration）和"异常"终止（在内层循环中找不到结束标记，现在将引发 RuntimeError）。

StopIteration 的这种效果已被用于截断生成器表达式，创建一种 takewhile 形式：

```python
def stop():
    raise StopIteration
print(list(x for x in range(10) if x < 5 or stop()))
# 打印 [0, 1, 2, 3, 4]
```

在当前提案下，这种非局部流程控制不受支持，必须用语句形式重写：

```python
def gen():
    for x in range(10):
        if x >= 5: return
        yield x
print(list(gen()))
# 打印 [0, 1, 2, 3, 4]
```

虽然这是功能的微小损失，但这是经常以可读性为代价的功能，就像 lambda 有比 def 更多的限制，生成器表达式也有比生成器函数更多的限制。在许多情况下，转换为完整的生成器函数将非常简单，并可能提高结构清晰度。

## 生成器、迭代器和 StopIteration 的解释

提案不改变生成器和迭代器之间的关系：生成器对象仍然是迭代器，不是所有迭代器都是生成器。生成器有迭代器没有的额外方法，如 send 和 throw。所有这些都不变。生成器用户没有任何变化——只有生成器函数的作者可能必须学习一些新东西。（这包括依赖条件中引发的 StopIteration 提前终止迭代的生成器表达式的作者。）

迭代器是具有__next__方法的对象。像许多其他特殊方法一样，它可以返回一个值，或者引发特定异常——在这种情况下，StopIteration——来信号它没有值要返回。在这方面，它类似于__getattr__（可以引发 AttributeError），__getitem__（可以引发 KeyError），等等。迭代器的辅助函数可以编写为遵循相同的协议；例如：

```python
def helper(x, y):
    if x > y: return 1 / (x - y)
    raise StopIteration

def __next__(self):
    if self.a: return helper(self.b, self.c)
    return helper(self.d, self.e)
```

两种信号形式都被传递：返回的值被返回，异常冒泡上去。辅助函数被编写为匹配调用函数的协议。

生成器函数是包含 yield 表达式的函数。每次它（重新）启动时，它可以 yield 一个值，或者返回（包括"掉出末尾"）。生成器的辅助函数也可以编写，但它也必须遵循生成器协议：

```python
def helper(x, y):
    if x > y: yield 1 / (x - y)

def gen(self):
    if self.a: return (yield from helper(self.b, self.c))
    return (yield from helper(self.d, self.e))
```

在两种情况下，任何意外异常都将冒泡上去。由于生成器和迭代器的性质，生成器内部意外的 StopIteration 将被转换为 RuntimeError，但除此之外，所有异常都将正常传播。

## 过渡计划

- **Python 3.5**: 在__future__导入下启用新语义；如果 StopIteration 从不在__future__导入下的生成器冒泡出来，则静默弃用警告。
- **Python 3.6**: 非静默弃用警告。
- **Python 3.7**: 在所有地方启用新语义。

## 替代提案

### 引发 RuntimeError 之外的东西

而不是通用的 RuntimeError，引发一个新的异常类型 UnexpectedStopIteration 可能更有意义。这有缺点，即隐式鼓励捕获它；正确的操作是捕获原始的 StopIteration，而不是链式异常。

### 提供返回时要引发的特定异常

Alyssa（Nick）Coghlan 建议了一种向生成器提供特定 StopIteration 实例的方法；如果引发任何其他 StopIteration 实例，它是一个错误，但如果引发那个特定的实例，生成器已正确完成。此子提案已被撤回，以支持更好的选项，但保留供参考。

### 使 return 触发的 StopIteration 显而易见

对于某些情况，更简单且完全向后兼容的解决方案可能就足够了：当生成器返回时，不引发 StopIteration，而是引发 StopIteration 的特定子类（GeneratorReturn），然后可以检测到它。如果不是那个子类，它是一个逃逸异常而不是 return 语句。

这个替代提案的灵感来自 Alyssa 的观察 [8]，如果 asyncio 协程 [9] 意外引发 StopIteration，它当前静默终止，这可能呈现给开发者一个难以调试的谜团。主要提案将这样的意外变成明显可区分的 RuntimeError 异常，但如果被拒绝，这个替代提案将使 asyncio 能够区分 return 语句和意外引发的 StopIteration 异常。

在上述三种结果中，两种改变：

- 如果达到 yield 点，值显然仍然会被返回。
- 如果帧被返回，引发 GeneratorReturn（而不是 StopIteration）。
- 如果会引发 GeneratorReturn 的实例，相反会引发 StopIteration 的实例。任何其他异常正常冒泡。

在第三种情况下，StopIteration 将具有原始 GeneratorReturn 的值，并在其__cause__中引用原始异常。如果未被捕获，这清楚地显示异常链。

这个替代不影响生成器表达式和列表推导式之间的差异，但允许生成器感知代码（如 contextlib 和 asyncio 模块）可靠地区分上面列出的第二和第三种结果。

然而，一旦存在依赖这个 GeneratorReturn 和 StopIteration 之间区别的代码，调用另一个生成器并依赖后者的 StopIteration 来冒泡出来的生成器仍然可能是错误的，取决于对两个异常类型之间区别的使用。

### 在 next() 内部转换异常

Mark Shannon 建议 [10] 问题可以在 next() 中解决，而不是在生成器函数的边界处。通过让 next() 捕获 StopIteration 并引发 ValueError，所有意外的 StopIteration 冒泡都将被阻止；然而，向后不兼容的担忧比当前提案严重得多，因为每个 next() 调用现在需要重写以防范 ValueError 而不是 StopIteration——更不用说没有办法编写一个代码块可靠地在多个 Python 版本上工作。（使用专用异常类型，可能子类化 ValueError，会有所帮助；然而，所有代码仍然需要重写。）

注意调用 next(it, default) 捕获 StopIteration 并替换给定的默认值；此功能通常有用以避免 try/except 块。

### 子提案：装饰器显式请求当前行为

Alyssa Coghlan 建议 [11] 当前行为所需的情况可以通过装饰器支持：

```python
from itertools import allow_implicit_stop

@allow_implicit_stop
def my_generator():
    ...
    yield next(it)
    ...
```

这在语义上等价于：

```python
def my_generator():
    try:
        ...
        yield next(it)
        ...
    except StopIteration:
        return
```

但更快，因为它可以通过简单地允许 StopIteration 直接冒泡来实现。

单源 Python 2/3 代码在 3.7+ 世界中也将受益，因为像 six 和 python-future 这样的库可以定义自己的"allow_implicit_stop"版本，引用 3.5+ 中的新内置函数，并在其他版本中实现为恒等函数。

然而，由于所需的实现复杂性，持续的兼容性问题，装饰器效果的微妙性，以及它会鼓励"快速修复"解决方案，只需将装饰器拍在所有生成器上而不是正确修复问题代码，此子提案已被拒绝。[12]

## 批评

非正式和传说统计表明，这很少，如果有的话，是一个问题。[13] 确实存在依赖当前行为的代码（例如 [3]，[6]，[7]），并且有这种将是不必要的代码变动来实现很少或没有收益的担忧。

Steven D'Aprano 在 comp.lang.python 上开始了非正式调查 [14]；撰写本文时只收到两个回复：一个支持更改列表推导式以匹配生成器表达式（！），另一个支持本 PEP 的主要提案。

现有模型已被比作完全可接受的每个异常有特殊含义的情况固有的问题。例如，__getitem__方法内意外的 KeyError 将被解释为失败，而不是允许冒泡。然而，有一个区别。特殊方法使用 return 来指示正常性，使用 raise 来信号异常；生成器 yield 来指示数据，使用 return 来信号异常状态。这使得明确引发 StopIteration 完全多余，可能令人惊讶。如果其他特殊方法有专用关键字来区分它们的返回路径，它们也可以将意外异常转换为 RuntimeError；它们不能的事实不应排除生成器这样做。

### 为什么不修复所有__next__() 方法？

当实现常规__next__() 方法时，唯一的方法来指示迭代结束是引发 StopIteration。所以在这里捕获 StopIteration 并将其转换为 RuntimeError 将适得其反。这是对生成器函数特殊地位的提醒：在生成器函数中，引发 StopIteration 是多余的，因为迭代可以通过简单的 return 终止。
