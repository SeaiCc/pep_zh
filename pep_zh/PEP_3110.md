# PEP 3110 – Python 3000 中的异常捕获

**Metadata:**

- **Author**: Collin Winter <collinwinter&#32;&#97;t&#32;google.com>
- **Created**: 16-Jan-2006
- **Python-Version**: 3.0

## 摘要

本 PEP 介绍了旨在帮助消除 Python 语法中的歧义、简化异常类、简化异常的垃圾回收以及减少 Python 3.0 语言规模的更改。

## 理由

- Python 2.x 中的 except 子句存在语法歧义，解析器无法区分 `except <expression>, <expression>`:

应该被解释为
```
except <type>, <type>:
```
或
```
except <type>, <name>:
```
Python 2 选择后一种语义，代价是需要将前者括起来，如
```
except (<type>, <type>):
```
如 PEP 352 中所指定的，将异常作为元组处理的能力将被移除，意味着此代码将不再工作：
```python
except os.error, (errno, errstr):
```
由于自动解包不再可能，最好移除使用元组作为 except 目标的能力。

- 如 PEP 344 中所指定的，Python 3 中的异常实例将具有__traceback__属性。该 PEP 的"开放问题"部分包括一段关于由此属性引起的垃圾回收困难的段落，即"exception -> traceback -> stack frame -> exception"引用循环，其中所有局部变量都保留在作用域中，直到下一次 GC 运行。本 PEP 旨在通过在 Python 3 的 except 子句中添加清理语义来解决此问题，在 except 套件结束时删除目标名称。

- 本着"应该有一种——而且最好只有一种——显而易见的方法来做某件事"的精神，最好合并重复的功能。为此，sys 模块的 exc_value、exc_type 和 exc_traceback 属性 [1] 将被移除，取而代之的是 sys.exc_info()，它提供相同的信息。这些属性已经在 PEP 3100 中列为目标移除。

## 文法更改

在 Python 3 中，except 语句的文法将从 [4]
```
except_clause: 'except' [test [',' test]]
```
更改为
```
except_clause: 'except' [test ['as' NAME]]
```
使用 as 代替逗号标记意味着
```
except (AttributeError, os.error):
```
可以清楚地理解为异常类的元组。这种新语法首先由 Greg Ewing [2] 提出，并得到 BDFL 的认可（[2]、[3]）。

此外，将 as 后面的标记从 test 限制为 NAME 意味着只有有效的标识符可以用作 except 目标。

注意，上面的文法总是要求异常类使用括号括起来的元组。这样，歧义的
```
except A, B:
```
在 Python 2.x 和 3.x 中意味着不同的事情——导致难以捕获的 bug——在 3.x 代码中不能合法地出现。

## 语义更改

为了解决与 PEP 344 相关的垃圾回收问题，Python 3 中的 except 语句将生成额外的字节码来删除目标，从而消除引用循环。源到源的转换，如 Phillip J. Eby 所建议的 [5]，是
```python
try:
 try_body
except E as N:
 except_body
...
```

被转换为（以 Python 2.5 术语）
```py
try:
 try_body
except E, N:
 try:
  except_body
 finally:
  N = None
  del N
...
```
实现已经检查到 py3k（前"p3yk"）分支 [6]。

## 兼容性问题

几乎所有 except 子句都需要更改。具有标识符目标的 except 子句将从
```
except E, N:
```
转换为
```
except E as N:
```
具有非元组、非标识符目标（例如，a.b.c[d]）的 except 子句需要从
```
except E, T:
```
转换为
```
except E as t:
 T = t
```
这两种情况都可以使用 Guido van Rossum 的 2to3 工具 [7] 和 except fixer [8] 处理。

具有元组目标的 except 子句需要逐个手动转换。这些更改通常需要伴随异常类本身的更改。虽然这些更改通常无法自动化，但 2to3 工具能够指出 except 子句目标是元组的情况，简化转换。

需要在 except 套件结束后保留异常实例的情况可以轻松转换如下：

```py
try:
 ...
except E as N:
 ...
...
```

变为

```py
try:
 ...
except E as N:
 n = N
 ...
...
```

这样，当 N 在块结束时被删除时，n 将保留并可以正常使用。

最后，所有对 sys 模块的 exc_type、exc_value 和 exc_traceback 属性的使用都需要移除。它们可以分别用 sys.exc_info()[0]、sys.exc_info()[1] 和 sys.exc_info()[2] 替换，这种转换可以由 2to3 的 sysexcattrs fixer 执行。

### 2.6 - 3.0 兼容性

为了促进 Python 2.6 和 3.0 之间的前向兼容性，except ... as ...: 语法将移植到 2.x 系列。文法将从：
```
except_clause: 'except' [test [',' test]]
```
更改为：
```
except_clause: 'except' [test [('as' | ',') test]]
```
except 语句的套件结束清理语义将不包含在 2.x 系列版本中。

## 开放问题

### 替换或删除"sys.exc_info()"

删除 sys.exc_info() 或用 sys.exception 属性或 sys.get_exception() 函数替换它的想法已经在 python-3000 上多次提出（[9]、[10]），并在 PEP 344 的"开放问题"部分提到。

虽然一个简单的 2to3 fixer 来替换对 sys.exc_info() 的调用和一些属性访问将是微不足道的，但对于静态分析来查找和修复期望 sys.exc_info() 的值作为参数的函数要困难得多。同样，这也没有解决需要重写所有以 sys.exc_info() 定义的 API 的文档的需求。

## 实现

本 PEP 在修订版 53342 [11] 和 53349 [12] 中实现。对 2.6 中新 except 语法的支持在修订版 55446 [13] 中实现。
