# PEP 309 – 偏函数应用

**元数据：**

- **作者**: Peter Harris <scav at blueyonder.co.uk>
- **创建时间**: 08-Feb-2003
- **Python 版本**: 2.5
- **发布历史**: 10-Feb-2003, 27-Feb-2003, 22-Feb-2004, 28-Apr-2006

## 注意

在本 PEP 被接受之后，python-dev 和 comp.lang.python 上的进一步讨论表明需要几个操作函数对象的工具，但它们与函数式编程无关。与其为这些工具创建一个新模块，大家同意 [1] 将"functional"模块重命名为"functools"，以反映其新扩展的范围。

本 PEP 中对"functional"模块的引用出于历史原因被保留。

## 摘要

本提议是为一个函数或可调用类，允许从可调用对象和部分参数列表（包括位置参数和关键字参数）构建新的可调用对象。

我提议一个名为"functional"的标准库模块，用于存放有用的高阶函数，包括 partial() 的实现。

实现已提交到 SourceForge [2]。

## 接受

补丁#941881 于 2005 年被接受并应用于 Py2.5。它基本上如这里概述的那样，一个 partial() 类型构造函数绑定最左边的位置参数和任何关键字。partial 对象有三个只读属性 func、args 和 keywords。对 partial 对象的调用可以指定关键字来覆盖对象本身的关键字。

还有一个单独的持续讨论，关于是否通过添加__get__ 方法来修改 partial 实现以更接近地模拟等效函数的行为。

## 动机

在函数式编程中，函数柯里化是一种用单参数函数实现多参数函数的方式。一个有 N 个参数的函数实际上是一个有 1 个参数的函数，返回另一个接受 (N-1) 个参数的函数。像 Haskell 和 ML 这样的语言中的函数应用工作方式是一个函数调用：

```
f x y z
```

实际上意味着：

```
(((f x) y) z)
```

这只是一个模糊的理论问题，除非在实际编程中它被证明非常有用。用对另一个函数的参数偏应用来表达函数既优雅又强大，在函数式语言中被广泛使用。

在某些函数式语言中（例如 Miranda），你可以使用像 (+1) 这样的表达式来表示等同于 Python 的 (lambda x: x + 1)。

一般来说，像这样的语言是强类型的，所以编译器总是知道期望的参数数量，并在面对函子和少于期望的参数数量时可以做出正确的处理。

Python 不通过柯里化实现多参数函数，所以如果你想要一个具有部分应用参数的函数，你可能会使用上面的 lambda，或者为每个实例定义一个命名函数。

然而，lambda 语法并不为所有人所喜欢，至少这么说。此外，Python 使用位置参数和关键字的灵活参数传递提供了一个泛化偏应用概念并做 lambda 无法做的事情的机会。

## 示例实现

这是在 Python 中创建具有部分应用参数的可调用对象的一种方法。下面的实现基于 Scott David Daniels 提供的改进：

```python
class partial(object):

    def __init__(*args, **kw):
        self = args[0]
        self.fn, self.args, self.kw = (args[1], args[2:], kw)

    def __call__(self, *args, **kw):
        if kw and self.kw:
            d = self.kw.copy()
            d.update(kw)
        else:
            d = kw or self.kw
        return self.fn(*(self.args + args), **d)
```

（Python Cookbook 中已经有类似这样的食谱 [3]。）

注意，当对象像函数一样被调用时，位置参数被追加到提供给构造函数的参数，关键字参数覆盖和增强提供给构造函数的参数。

在创建对象和调用它时，可以供应位置参数、关键字参数或两者。

## 使用示例

所以 partial(operator.add, 1) 有点像 (lambda x: 1 + x)。当然这不是一个你看到好处的例子。

还要注意，你也可以用同样的方式包装一个类，因为类本身就是对象的可调用工厂。所以在某些情况下，而不是定义一个子类，你可以通过对构造函数的参数进行偏应用来特化类。

例如，partial(Tkinter.Label, fg='blue') 创建默认具有蓝色前景的 Tkinter Labels。

这是一个简单的示例，使用偏应用动态构建 Tkinter 小部件的回调：

```python
from Tkinter import Tk, Canvas, Button
import sys
from functional import partial

win = Tk()
c = Canvas(win,width=200,height=50)
c.pack()

for colour in sys.argv[1:]:
    b = Button(win, text=colour,
               command=partial(c.config, bg=colour))
    b.pack(side='left')

win.mainloop()
```

## 被放弃的语法提议

我最初建议语法 fn@(*args, **kw)，意思与 partial(fn, *args, **kw) 相同。

@ 符号在一些汇编语言中用于暗示寄存器间接寻址，这里的用法也是一种间接寻址。f@(x) 不是 f(x)，而是一个当你调用它时变成 f(x) 的东西。

它没有被很好地接受，所以我撤回了这个提议的一部分。无论如何，@ 已被新的装饰器语法采用。

## 来自 comp.lang.python 和 python-dev 的反馈

表达的意见包括以下几点（我总结）：

- Lambda 已经足够好了。

- @ 语法很丑陋（一致意见）。

- 它真的是柯里化而不是闭包。ActiveState 的 Python Cookbook 上有一个几乎相同的柯里化类实现。

- 柯里化类确实是标准库的有用补充。

- 它不是函数柯里化，而是偏应用。因此现在提议名称为 partial()。

- 它可能不够有用以至于放入内置函数中。

- 名为 functional 的模块的想法被很好地接受，还有其他东西属于那里（例如函数组合）。

- 为了完整性，另一个在函数调用提供的参数之后追加偏参数的对象（可能称为 rightcurry）已被建议。

我同意 lambda 通常足够好，只是不总是。我想要有用的检查和子类化的可能性。

我不同意 @ 特别丑陋，但可能我只是奇怪。我们有字典、列表和元组字面量通过特殊标点符号 neatly 区分——直接表达部分应用的函数字面量的方式不是那么大的延伸。然而，没有一个人说他们喜欢它，所以就我而言它是一只死鹦鹉。

我同意将类称为 partial 而不是 curry 或 closure，所以我相应地修改了本 PEP 中的提议。但不是完全：一些对'curry'的不正确引用已被保留，因为那是当时讨论的地方。

从右边部分应用参数，或在任意位置插入参数创造了它自己的问题，但 pending 发现一个好的实现和不令人困惑的语义，我认为它不应该被排除。

Carl Banks 发布了一个作为真正的函数式闭包的实现：

```python
def curry(fn, *cargs, **ckwargs):
    def call_fn(*fargs, **fkwargs):
        d = ckwargs.copy()
        d.update(fkwargs)
        return fn(*(cargs + fargs), **d)
    return call_fn
```

他向我保证它更高效。

我也用 Pyrex 编写了这个类，以估计通过用 C 编写它性能可能如何提高：

```python
cdef class curry:

    cdef object fn, args, kw

    def __init__(self, fn, *args, **kw):
        self.fn=fn
        self.args=args
        self.kw = kw

    def __call__(self, *args, **kw):
        if self.kw: # from Python Cookbook version
            d = self.kw.copy()
            d.update(kw)
        else:
            d=kw
        return self.fn(*(self.args + args), **d)
```

Pyrex 中的性能增益比嵌套函数实现少 100%，因为为了完全通用它必须通过 Python API 调用操作。出于同样的原因，C 实现也不太可能快很多，所以内置 C 代码的理由不是很强。

## 总结

我更喜欢一些部分应用函数和其他可调用对象的方法应该存在于标准库中。

标准库模块 functional 应该包含 partial 的实现，以及社区想要的任何其他高阶函数。其他可能属于那里的函数超出了本 PEP 的范围。

实现的补丁、文档和单元测试（分别为 SF patches 931005、931007 和 931010）已被提交但尚未检出。

Hye-Shik Chang 的 C 实现也已提交，虽然预计它不会被包含，直到 Python 实现证明本身有用到值得优化的程度。
