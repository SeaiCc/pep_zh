# PEP 443 – 装饰器 functools.singledispatch
**元数据：**

- **作者**：Łukasz Langa <lukasz&#32;&#97;t&#32;python.org>
- **创建时间**：2013 年 5 月 22 日
- **Python 版本**：3.4
- **发布历史**：2013 年 5 月 22 日，2013 年 5 月 25 日，2013 年 5 月 31 日

## 摘要

本 PEP 提议在 functools 标准库模块中添加一个新机制，提供已知为单分发通用函数的简单形式的通用编程。

通用函数由多个函数组成，这些函数为不同类型实现相同的操作。在调用期间应使用哪个实现由分发算法确定。当实现基于单个参数的类型选择时，这被称为单分发。

## 理由和目标

Python 一直提供各种内置和标准库通用函数，如 len()、iter()、pprint.pprint()、copy.copy()，以及 operator 模块中的大多数函数。但是，它当前：

- 没有简单或直接的方法让开发者创建新的通用函数，

- 没有标准的方法向现有通用函数添加方法（即，一些使用注册函数添加，其他需要定义__special__方法，可能通过猴子补丁）。

此外，Python 代码当前常见的一种反模式是检查接收参数的类型，以决定如何处理对象。

例如，代码可能希望接受某个类型的对象或该类型对象的序列。当前，执行此操作的"明显方法"是通过类型检查，但这很脆弱且对扩展封闭。

抽象基类使发现现有行为更容易，但不帮助添加新行为。使用已编写库的开发者可能无法更改此类代码如何处理他们的对象，特别是如果他们使用的对象是由第三方创建的。

因此，本 PEP 提议一个统一的 API，通过使用装饰器解决动态重载。

## 用户 API

要定义通用函数，使用@singledispatch 装饰器装饰它。注意分发发生在第一个参数的类型上。相应地创建函数：

```python
>>> from functools import singledispatch
>>> @singledispatch
... def fun(arg, verbose=False):
...     if verbose:
...         print("Let me just say,", end=" ")
...     print(arg)
```

要向函数添加重载实现，使用通用函数的 register() 属性。这是一个装饰器，接受类型参数并装饰实现该类型操作的函数：

```python
>>> @fun.register(int)
... def _(arg, verbose=False):
...     if verbose:
...         print("Strength in numbers, eh?", end=" ")
...     print(arg)
...
>>> @fun.register(list)
... def _(arg, verbose=False):
...     if verbose:
...         print("Enumerate this:")
...     for i, elem in enumerate(arg):
...         print(i, elem)
```

要启用注册 lambda 和现有函数，register() 属性可以以函数形式使用：

```python
>>> def nothing(arg, verbose=False):
...     print("Nothing.")
...
>>> fun.register(type(None), nothing)
```

register() 属性返回未装饰的函数。这启用装饰器堆叠、序列化，以及独立创建每个变体的单元测试：

```python
>>> @fun.register(float)
... @fun.register(Decimal)
... def fun_num(arg, verbose=False):
...     if verbose:
...         print("Half of your number:", end=" ")
...     print(arg / 2)
...
>>> fun_num is fun
False
```

调用时，通用函数根据第一个参数的类型分发：

```python
>>> fun("Hello, world.")
Hello, world.
>>> fun("test.", verbose=True)
Let me just say, test.
>>> fun(42, verbose=True)
Strength in numbers, eh? 42
>>> fun(['spam', 'spam', 'eggs', 'spam'], verbose=True)
Enumerate this:
0 spam
1 spam
2 eggs
3 spam
>>> fun(None)
Nothing.
>>> fun(1.23)
0.615
```

当没有特定类型的注册实现时，使用其方法解析顺序查找更通用的实现。使用@singledispatch 装饰的原始函数注册为基本对象类型，这意味着如果没有找到更好的实现，它将使用它。

要检查通用函数为给定类型选择哪个实现，使用 dispatch() 属性：

```python
>>> fun.dispatch(float)
<function fun_num at 0x104319058>
>>> fun.dispatch(dict) # 注意：默认实现
<function fun at 0x103fe0000>
```

要访问所有注册实现，使用只读 registry 属性：

```python
>>> fun.registry.keys()
dict_keys([<class 'NoneType'>, <class 'int'>, <class 'object'>,
 <class 'decimal.Decimal'>, <class 'list'>,
 <class 'float'>])
>>> fun.registry[float]
<function fun_num at 0x1035a2840>
>>> fun.registry[object]
<function fun at 0x103fe0000>
```

提议的 API 故意有限和有观点，以确保它易于解释和使用，以及与 functools 模块中的现有成员保持一致。

## 实现说明

本 PEP 描述的功能已在 pkgutil 标准库模块中作为 simplegeneric 实现。因为此实现已成熟，目标是几乎原封不动地移动它。参考实现可在 hg.python.org [1] 获得。

分发类型指定为装饰器参数。考虑了使用函数注释的替代形式，但其包含已被拒绝。截至 2013 年 5 月，此使用模式超出标准库范围 [2]，注释使用的最佳实践仍有待讨论。

基于当前的 pkgutil.simplegeneric 实现，并遵循在抽象基类上注册虚拟子类的约定，分发注册表将不是线程安全的。

### 抽象基类

pkgutil.simplegeneric 实现依赖于几种方法解析顺序 (MRO) 形式。@singledispatch 移除了对旧式类和 Zope 的 ExtensionClasses 的特殊处理。更重要的是，它引入了对抽象基类 (ABC) 的支持。

当为 ABC 注册通用函数实现时，分发算法切换到 C3 线性化的扩展形式，其中包括相关 ABC 在提供的参数的 MRO 中。算法在 ABC 引入其功能的地方插入 ABC，即 issubclass(cls, abc) 对类本身返回 True 但对其所有直接基类返回 False。给定类的隐式 ABC（注册或从特殊方法如__len__() 的存在推断）直接插入在所述类的 MRO 中明确列出的最后一个 ABC 之后。

在最基本形式中，此线性化返回给定类型的 MRO：

```python
>>> _compose_mro(dict, [])
[<class 'dict'>, <class 'object'>]
```

当第二个参数包含指定类型是其子类的 ABC 时，它们以可预测的顺序插入：

```python
>>> _compose_mro(dict, [Sized, MutableMapping, str,
... Sequence, Iterable])
[<class 'dict'>, <class 'collections.abc.MutableMapping'>,
 <class 'collections.abc.Mapping'>, <class 'collections.abc.Sized'>,
 <class 'collections.abc.Iterable'>, <class 'collections.abc.Container'>,
 <class 'object'>]
```

虽然此操作模式明显更慢，但所有分发决定都被缓存。在通用函数上注册新实现或用户代码在 ABC 上调用 register() 隐式子类化时，缓存被无效。在后一种情况下，可能创建具有歧义分发的情况，例如：

```python
>>> from collections.abc import Iterable, Container
>>> class P:
...     pass
>>> Iterable.register(P)
<class '__main__.P'>
>>> Container.register(P)
<class '__main__.P'>
```

面对歧义，@singledispatch 拒绝猜测的诱惑：

```python
>>> @singledispatch
... def g(arg):
...     return "base"
...
>>> g.register(Iterable, lambda arg: "iterable")
<function <lambda> at 0x108b49110>
>>> g.register(Container, lambda arg: "container")
<function <lambda> at 0x108b491c8>
>>> g(P())
Traceback (most recent call last):
...
RuntimeError: Ambiguous dispatch: <class 'collections.abc.Container'>
or <class 'collections.abc.Iterable'>
```

注意，如果一个或多个 ABC 在类定义期间明确提供为基类，则不会引发此异常。在这种情况下，分发按 MRO 顺序发生：

```python
>>> class Ten(Iterable, Container):
...     def __iter__(self):
...         for i in range(10):
...             yield i
...     def __contains__(self, value):
...         return value in range(10)
...
>>> g(Ten())
'iterable'
```

当从__len__() 或__contains__() 等特殊方法的存在推断子类化 ABC 时，出现类似的冲突：

```python
>>> class Q:
...     def __contains__(self, value):
...         return False
...
>>> issubclass(Q, Container)
True
>>> Iterable.register(Q)
>>> g(Q())
Traceback (most recent call last):
...
RuntimeError: Ambiguous dispatch: <class 'collections.abc.Container'>
or <class 'collections.abc.Iterable'>
```

PEP 的早期版本包含一个自定义方法，更简单但创建了一些具有令人惊讶结果的边缘情况 [3]。

## 使用模式

本 PEP 提议仅扩展明确标记为通用的函数的行为。就像基类方法可能被子类重写一样，函数也可以重载以为给定类型提供自定义功能。

通用重载不等于任意重载，在我们需要期望人们以不可预测的方式重新定义现有函数行为的意义下。相反，实际程序中的通用函数使用往往遵循非常可预测的模式，并且在常见情况下注册的实现高度可发现。

如果模块定义新的通用操作，它通常也在同一位置定义现有类型所需的任何实现。同样，如果模块定义新类型，则它通常在那里为任何它知道或关心的通用函数定义实现。因此，绝大多数注册实现可以在被重载的函数旁边找到，或者为新定义的添加支持的类型找到。

只有在相当不频繁的情况下，人们才将在包含函数或添加实现的类型都不存在的模块中注册实现。在没有无能或故意模糊的情况下，不在相关类型或函数旁边注册的少数实现通常不需要在定义它们的范围之外被理解或知道。（除了"支持模块"情况，最佳实践建议相应地命名它们。）

如前所述，单分发通用函数在整个标准库中已经 prolific。一个干净的标准方法为重构这些自定义实现以使用通用实现提供前进之路，同时为用户可扩展性开放它们。

## 替代方法

在 PEP 3124 中，Phillip J. Eby 提出了一个完全成熟的解决方案，基于任意规则集的重载（默认实现根据参数类型分发），以及接口、适配和方法组合。PEAK-Rules [4] 是 PJE 的 PEP 中描述概念的参考实现。

如此广泛的方法本质上是复杂的，这使得达成共识困难。相比之下，本 PEP 专注于一个简单的功能片段，易于推理。重要的是要注意，这不排除现在或将来使用其他方法。

在 2005 年 Artima 上的一篇文章 [5] 中，Guido van Rossum 展示了一个通用函数实现，该实现根据函数的所有参数的类型分发。相同的方法在 Andrey Popp 的 generic 包中可用 [6]，以及 David Mertz 的 gnosis.magic.multimethods [7]。

虽然这起初看起来是可取的，但我同意 Fredrik Lundh 的评论，"如果你设计 API 时用几页逻辑来理清函数应该执行什么代码，你应该可能将 API 设计交给其他人"。换句话说，本 PEP 提议的单参数方法不仅更容易实现，而且清楚地传达了在更复杂状态上分发是反模式。它还具有与面向对象编程中熟悉的方法分发机制直接对应的优点。唯一的区别是自定义实现是更接近数据（面向对象方法）还是算法（单分发重载）关联。

PyPy 的 RPython 提供 extendabletype [8]，一个元类，使类可以从外部扩展。与 pairtype() 和 pair() 工厂组合，这提供了一种形式的单分发通用函数。
