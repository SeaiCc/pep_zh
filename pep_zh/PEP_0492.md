# PEP 492 – 使用 async 和 await 语法的协程

**元数据：**

- **作者**: Yury Selivanov <yury at edgedb.com>
- **创建日期**: 2015 年 4 月 9 日
- **Python 版本**: 3.5
- **发布历史**: 2015 年 4 月 17 日，2015 年 4 月 21 日，2015 年 4 月 27 日，2015 年 4 月 29 日，2015 年 5 月 5 日

## 摘要

互联网和通用连接性的增长引发了对响应式和可扩展代码的相应需求。本提案旨在通过将编写显式异步并发 Python 代码变得更加简单和符合 Python 风格来满足这一需求。

提议将协程作为 Python 中的一个独立概念，并引入新的支持语法。最终目标是帮助在 Python 中建立一个通用的、易于理解的异步编程思维模型，并使其尽可能接近同步编程。

本 PEP 假设异步任务由类似于标准库模块 asyncio.events.AbstractEventLoop 的事件循环进行调度和协调。虽然该 PEP 不绑定于任何特定的事件循环实现，但它仅适用于将 yield 用作向调度程序发出信号以表示协程将等待某个事件（如 IO）完成的协程类型。

我们相信，此处提出的更改将有助于 Python 在异步编程这一快速增长的领域中保持相关性和竞争力，因为许多其他语言已经采用或计划采用类似功能：[2], [5], [6], [7], [8], [10]。

## API 设计和实现修订

- 对 Python 3.5 初始测试版发布的反馈导致了对支持本 PEP 的对象模型进行了重新设计，以更清晰地区分原生协程和生成器——原生协程不再是新类型的生成器，而是现在成为自己完全独立的类型（在 [17] 中实现）。此更改主要是基于在尝试将原生协程支持集成到 Tornado Web 服务器时遇到的问题（在 [18] 中报告）。

- 在 CPython 3.5.2 中，`__aiter__` 协议已更新。在 3.5.2 之前，`__aiter__` 应该返回一个解析为异步迭代器的可等待对象。从 3.5.2 开始，`__aiter__` 应该直接返回异步迭代器。

如果在 3.5.2 中使用旧协议，Python 将引发 PendingDeprecationWarning。

在 CPython 3.6 中，旧的 `__aiter__` 协议仍将受支持，但会引发 DeprecationWarning。

在 CPython 3.7 中，旧的 `__aiter__` 协议将不再受支持：如果 `__aiter__` 返回异步迭代器以外的任何内容，将引发 RuntimeError。

更多详细信息请参阅 [19] 和 [20]。

## 理由和目标

当前 Python 通过生成器（PEP 342）支持实现协程，并进一步通过 PEP 380 中引入的 yield from 语法得到增强。这种方法有一些缺点：

- 协程与普通生成器很容易混淆，因为它们共享相同的语法；这对于新开发人员来说尤其如此。

- 一个函数是否是协程是由其函数体中是否存在 yield 或 yield from 语句决定的，这可能导致在重构过程中此类语句出现在或消失于函数体中时产生不明显的错误。

- 对异步调用的支持仅限于语法上允许 yield 的表达式，限制了 with 和 for 语句等语法功能的有用性。

本提案将协程作为 Python 原生语言特性，并将它们与生成器明确区分开来。这消除了生成器/协程的歧义，并使它能够可靠地定义协程而无需依赖特定库。这也使 linter 和 IDE 能够改进静态代码分析和重构。

原生协程和相关的新的语法特性使得能够以异步术语定义上下文管理器协议和迭代协议。如本提案后面所示，新的 async with 语句让 Python 程序在执行和退出运行时上下文时执行异步调用，新的 async for 语句使得能够在迭代器中执行异步调用。

## 规范

本提案引入新的语法和语义以增强 Python 中的协程支持。

本规范假定了解 Python 中协程的实现（PEP 342 和 PEP 380）。此处提出的语法更改的动机来自 asyncio 框架（PEP 3156）和"Cofunctions"提案（PEP 3152，现已否决，转而采用本规范）。

从本文档的这一点开始，我们使用术语原生协程来指代使用新语法声明的函数。generator-based coroutine（基于生成器的协程）在必要时用于指代基于生成器语法的协程。coroutine（协程）在两个定义都适用的上下文中使用。

### 新的协程声明语法

以下新语法用于声明原生协程：

```python
async def read_data(db):
  pass
```

协程的关键属性：

- async def 函数始终是协程，即使它们不包含 await 表达式。

- 在 async 函数中包含 yield 或 yield from 表达式是 SyntaxError。

- 内部引入了两个新的代码对象标志：
CO_COROUTINE 用于标记原生协程（使用新语法定义）。

- CO_ITERABLE_COROUTINE 用于使基于生成器的协程与原生协程兼容（由 types.coroutine() 函数设置）。

- 普通生成器被调用时返回生成器对象；类似地，协程返回协程对象。

- StopIteration 异常不会从协程中传播出来，而是被替换为 RuntimeError。对于普通生成器，这种行为需要 future import（参见 PEP 479）。

- 当原生协程被垃圾回收时，如果它从未被等待过，会引发 RuntimeWarning（另见调试功能）。

- 另见协程对象部分。

### types.coroutine()

一个名为 coroutine(fn) 的新函数被添加到 types 模块中。它允许现有的 asyncio 中基于生成器的协程与本 PEP 引入的原生协程之间的互操作性：

```python
@types.coroutine
def process_data(db):
  data = yield from read_data(db)
  ...
```

该函数将 CO_ITERABLE_COROUTINE 标志应用于生成器函数的代码对象，使其返回协程对象。

如果 fn 不是生成器函数，它将被包装。如果它返回生成器，它将被包装在一个可等待代理对象中（见下文可等待对象的定义）。

注意，CO_COROUTINE 标志不会被 types.coroutine() 应用，以便将使用新语法定义的原生协程与基于生成器的协程区分开来。

### Await 表达式

以下新的 await 表达式用于获取协程执行的结果：

```python
async def read_data(db):
  data = await db.fetch('SELECT ...')
  ...
```

await，类似于 yield from，挂起 read_data 协程的执行，直到 db.fetch 可等待对象完成并返回结果 data。

它使用 yield from 实现，并额外验证其参数。await 只接受可等待对象，它可以是以下之一：

- 从原生协程函数返回的原生协程对象。

- 从使用 types.coroutine() 装饰的函数返回的基于生成器的协程对象。

- 具有返回迭代器的 `__await__` 方法的任何对象。Any yield from 调用链最终以 yield 结束。这是 Future 如何实现的基本机制。由于协程内部是一种特殊类型的生成器，每个 await 都由 await 调用链中的某个 yield 挂起（请参照 PEP 3156 获取详细解释）。

为了使协程能够实现这种行为，添加了一个名为 `__await__` 的新魔术方法。例如，在 asyncio 中，要使 Future 对象在 await 语句中可用，唯一的变化是在 asyncio.Future 类中添加 `__await__ = __iter__` 行。

具有 `__await__` 方法的对象在本 PEP 的其余部分称为 Future-like objects（类 Future 对象）。

如果 `__await__` 返回迭代器以外的任何内容，则为 TypeError。

- 使用 CPython C API 定义的具有 tp_as_async.am_await 函数并返回迭代器的对象（类似于 `__await__` 方法）。

在 async def 函数之外使用 await 是 SyntaxError（就像在 def 函数之外使用 yield 是 SyntaxError 一样）。

将非可等待对象传递给 await 表达式是 TypeError。

#### 更新的操作符优先级表

await 关键字定义如下：

```
power ::= await ["**" u_expr]
await ::= ["await"] primary
```

其中"primary"代表语言中绑定最紧密的操作。其语法为：

```
primary ::= atom | attributeref | subscription | slicing | call
```

详见 Python 文档 [12] 和本提案的语法更新部分。

await 与 yield 和 yield from 操作符的关键区别在于，await 表达式大多数时候不需要用括号括起来。

另外，yield from 允许任何表达式作为其参数，包括诸如 yield from a() + b() 之类的表达式，这将被解析为 yield from (a() + b())，这几乎总是一个 bug。一般来说，任何算术运算的结果都不是可等待对象。为了避免这种错误，决定使 await 的优先级低于 [], (), 和 .，但高于 ** 操作符。

| 操作符 | 描述 |
|--------|------|
| yield x, yield from x | 生成表达式 |
| lambda | Lambda 表达式 |
| if – else | 条件表达式 |
| or | 布尔或 |
| and | 布尔与 |
| not x | 布尔非 |
| in, not in, is, is not, <, <=, >, >=, !=, == | 比较运算，包括成员资格测试和身份测试 |
| \| | 按位或 |
| ^ | 按位异或 |
| & | 按位与 |
| <<, >> | 移位 |
| +, - | 加法和减法 |
| *, @, /, //, % | 乘法、矩阵乘法、除法、取余 |
| +x, -x, ~x | 正数、负数、按位非 |
| ** | 幂运算 |
| await x | Await 表达式 |
| x[index], x[index:index], x(arguments...), x.attribute | 下标、切片、调用、属性引用 |
| (expressions...), [expressions...], {key: value...}, {expressions...} | 绑定或元组显示、列表显示、字典显示、集合显示 |

#### "await"表达式示例

有效语法示例：

| 表达式 | 被解析为 |
|--------|----------|
| if await fut: pass | if (await fut): pass |
| if await fut + 1: pass | if (await fut) + 1: pass |
| pair = await fut, 'spam' | pair = (await fut), 'spam' |
| with await fut, open(): pass | with (await fut), open(): pass |
| await foo()['spam'].baz()() | await ( foo()['spam'].baz()() ) |
| return await coro() | return ( await coro() ) |
| res = await coro() ** 2 | res = (await coro()) ** 2 |
| func(a1=await coro(), a2=0) | func(a1=(await coro()), a2=0) |
| await foo() + await bar() | (await foo()) + (await bar()) |
| -await foo() | -(await foo()) |

无效语法示例：

| 表达式 | 应写为 |
|--------|--------|
| await await coro() | await (await coro()) |
| await -coro() | await (-coro()) |

### 异步上下文管理器和"async with"

异步上下文管理器是一个能够在其 enter 和 exit 方法中挂起执行的上下文管理器。

为了使这成为可能，提议了一个用于异步上下文管理器的新协议。添加了两个新的魔术方法：`__aenter__` 和 `__aexit__`。两者都必须返回一个可等待对象。

异步上下文管理器的示例：

```python
class AsyncContextManager:
  async def __aenter__(self):
    await log('entering context')

  async def __aexit__(self, exc_type, exc, tb):
    await log('exiting context')
```

#### 新语法

提议了一个用于异步上下文管理器的新语句：

```python
async with EXPR as VAR:
  BLOCK
```

它在语义上等价于：

```python
mgr = (EXPR)
aexit = type(mgr).__aexit__
aenter = type(mgr).__aenter__

VAR = await aenter(mgr)
try:
  BLOCK
except:
  if not await aexit(mgr, *sys.exc_info()):
    raise
else:
  await aexit(mgr, None, None, None)
```

与普通 with 语句一样，可以在单个 async with 语句中指定多个上下文管理器。

将没有 `__aenter__` 和 `__aexit__` 方法的普通上下文管理器传递给 async with 是错误的。在 async def 函数之外使用 async with 是 SyntaxError。

#### 示例

使用异步上下文管理器，可以轻松地为协程实现适当的数据库事务管理器：

```python
async def commit(session, data):
  ...

  async with session.transaction():
    ...
    await session.update(data)
    ...
```

需要锁定的代码也看起来更简洁：

```python
async with lock:
  ...
```

而不是：

```python
with (yield from lock):
  ...
```

### 异步迭代器和"async for"

异步可迭代对象能够在其 iter 实现中调用异步代码，异步迭代器能够在其 next 方法中调用异步代码。为了支持异步迭代：

- 对象必须实现一个 `__aiter__` 方法（或者，如果使用 CPython C API 定义，则为 tp_as_async.am_aiter 槽），返回一个异步迭代器对象。

- 异步迭代器对象必须实现一个 `__anext__` 方法（或者，如果使用 CPython C API 定义，则为 tp_as_async.am_anext 槽），返回一个可等待对象。

- 要停止迭代，`__anext__` 必须引发 StopAsyncIteration 异常。

异步可迭代对象的示例：

```python
class AsyncIterable:
  def __aiter__(self):
    return self

  async def __anext__(self):
    data = await self.fetch_data()
    if data:
      return data
    else:
      raise StopAsyncIteration

  async def fetch_data(self):
    ...
```

#### 新语法

提议了一个用于遍历异步迭代器的新语句：

```python
async for TARGET in ITER:
  BLOCK
else:
  BLOCK2
```

它在语义上等价于：

```python
iter = (ITER)
iter = type(iter).__aiter__(iter)
running = True
while running:
  try:
    TARGET = await type(iter).__anext__(iter)
  except StopAsyncIteration:
    running = False
  else:
    BLOCK
else:
  BLOCK2
```

将没有 `__aiter__` 方法的普通可迭代对象传递给 async for 是 TypeError。在 async def 函数之外使用 async for 是 SyntaxError。

与普通 for 语句一样，async for 有一个可选的 else 子句。

#### 示例 1

使用异步迭代协议，可以在迭代期间异步缓冲数据：

```python
async for data in cursor:
  ...
```

其中 cursor 是一个异步迭代器，每 N 次迭代后从数据库预取 N 行数据。

以下代码说明了新的异步迭代协议：

```python
class Cursor:
  def __init__(self):
    self.buffer = collections.deque()

  async def _prefetch(self):
    ...

  def __aiter__(self):
    return self

  async def __anext__(self):
    if not self.buffer:
      self.buffer = await self._prefetch()
    if not self.buffer:
      raise StopAsyncIteration
    return self.buffer.popleft()
```

然后 Cursor 类可以如下使用：

```python
async for row in Cursor():
  print(row)
```

这等价于以下代码：

i = Cursor().__aiter__()
while True:
  try:
    row = await i.__anext__()
  except StopAsyncIteration:
    break
  else:
    print(row)

#### 示例 2

以下是一个工具类，它将普通可迭代对象转换为异步可迭代对象。虽然这不是很有用的事情，但代码说明了普通迭代器和异步迭代器之间的关系。

```python
class AsyncIteratorWrapper:
  def __init__(self, obj):
    self._it = iter(obj)

  def __aiter__(self):
    return self

  async def __anext__(self):
    try:
      value = next(self._it)
    except StopIteration:
      raise StopAsyncIteration
    return value

async for letter in AsyncIteratorWrapper("abc"):
  print(letter)
```

#### 为什么是 StopAsyncIteration？

协程内部仍然基于生成器。因此，在 PEP 479 之前，以下两者之间没有根本区别：

```python
def g1():
  yield from fut
  return 'spam'

和：

def g2():
  yield from fut
  raise StopIteration('spam')
```

由于 PEP 479 已被接受并默认对协程启用，以下示例的 StopIteration 将被包装到 RuntimeError 中：

```python
async def a1():
  await fut
  raise StopIteration('spam')
```

告诉外部代码迭代已结束的唯一方法是引发 StopIteration 以外的东西。因此，添加了一个新的内置异常类 StopAsyncIteration。

此外，使用 PEP 479 的语义，协程中引发的所有 StopIteration 异常都被包装在 RuntimeError 中。

### 协程对象

#### 与生成器的区别

本节仅适用于具有 CO_COROUTINE 标志的原生协程，即使用新的 async def 语法定义的协程。

asyncio 中现有基于生成器的协程的行为保持不变。

已经做了很大的努力来确保协程和生成器被视为不同的概念：

- 原生协程对象不实现 `__iter__` 和 `__next__` 方法。因此，它们不能被迭代，也不能传递给 iter()、list()、tuple() 和其他内置函数。它们也不能用于 for..in 循环中。尝试在原生协程对象上使用 `__iter__` 或 `__next__` 将导致 TypeError。

- 纯生成器不能从原生协程中 yield from：这样做将导致 TypeError。

- 基于生成器的协程（用于 asyncio 代码必须使用 @asyncio.coroutine 装饰 [1]）可以从原生协程对象 yield from。

- `inspect.isgenerator()` 和 `inspect.isgeneratorfunction()` 对原生协程对象和原生协程函数返回 False。

#### 协程对象方法

协程内部基于生成器，因此它们共享实现。类似于生成器对象，协程具有 throw()、send() 和 close() 方法。StopIteration 和 GeneratorExit 对协程起相同的作用（尽管 PEP 479 默认对协程启用）。详见 PEP 342、PEP 380 和 Python 文档 [11]。

协程的 throw()、send() 方法用于向类 Future 对象推送值和引发错误。

### 调试功能

一个常见的初学者错误是忘记在协程上使用 yield from：

```python
@asyncio.coroutine
def useful():
  asyncio.sleep(1) # 没有'yield from'这将不起作用
```

为了调试这种错误，asyncio 中有一个特殊的调试模式，其中 @coroutine 装饰器将所有函数包装在一个具有记录警告的析构函数的特殊对象中。每当包装的生成器被垃圾回收时，会生成一个详细的日志消息，其中包含装饰器函数确切定义的位置、它被回收的堆栈跟踪等信息。包装对象还提供了一个方便的 `__repr__` 函数，其中包含有关生成器的详细信息。

唯一的问题是如何启用这些调试功能。由于调试设施应该在生产模式下不产生任何操作，@coroutine 装饰器根据操作系统环境变量 PYTHONASYNCIODEBUG 决定是否包装或不包装。这样就可以运行带有 asyncio 自身函数插装的 asyncio 程序。EventLoop.set_debug，一个不同的调试设施，对 @coroutine 装饰器的行为没有影响。

本提案中，协程是一个原生的、与生成器不同的概念。除了对从未被等待的协程引发 RuntimeWarning 外，还提议向 sys 模块添加两个新函数：set_coroutine_wrapper 和 get_coroutine_wrapper。这是为了在 asyncio 和其他框架中启用高级调试功能（例如显示协程确切创建的位置，以及它被垃圾回收的更详细的堆栈跟踪）。

### 新的标准库函数

- types.coroutine(gen)。详见 types.coroutine() 部分。

- inspect.iscoroutine(obj) 如果 obj 是原生协程对象则返回 True。

- inspect.iscoroutinefunction(obj) 如果 obj 是原生协程函数则返回 True。

- inspect.isawaitable(obj) 如果 obj 是可等待的则返回 True。

- inspect.getcoroutinestate(coro) 返回原生协程对象的当前状态（镜像 inspect.getgeneratorstate(gen)）。

- inspect.getcoroutinelocals(coro) 返回原生协程对象的局部变量到其值的映射（镜像 inspect.getgeneratorlocals(gen)）。

- sys.set_coroutine_wrapper(wrapper) 允许拦截原生协程对象的创建。wrapper 必须是接受一个参数（协程对象）的可调用对象，或者为 None。None 重置包装器。如果调用两次，新包装器将替换之前的包装器。该函数是线程特定的。详见调试功能部分。

- sys.get_coroutine_wrapper() 返回当前的包装器对象。如果没有设置包装器则返回 None。该函数是线程特定的。详见调试功能部分。

### 新的抽象基类

为了允许与现有框架（如 Tornado，见 [13]）和编译器（如 Cython，见 [16]）更好地集成，添加了两个新的抽象基类（ABC）：

- collections.abc.Awaitable ABC 用于实现 `__await__` 方法的类 Future 类。

- collections.abc.Coroutine ABC 用于实现 `send(value)`、`throw(type, exc, tb)`、`close()` 和 `__await__()` 方法的协程对象。注意具有 CO_ITERABLE_COROUTINE 标志的基于生成器的协程不实现 `__await__` 方法，因此不是 collections.abc.Coroutine 和 collections.abc.Awaitable ABC 的实例：

```python
@types.coroutine
def gencoro():
  yield

assert not isinstance(gencoro(), collections.abc.Coroutine)

# 但是：
assert inspect.isawaitable(gencoro())
```

为了允许轻松测试对象是否支持异步迭代，添加了两个更多的 ABC：

collections.abc.AsyncIterable – 测试 `__aiter__` 方法。

- collections.abc.AsyncIterator – 测试 `__aiter__` 和 `__anext__` 方法。

## 术语表

**原生协程函数** – 使用 async def 声明的协程函数。它使用 await 和 return value；详见 [新的协程声明语法](#新的协程声明语法)。

**原生协程** – 从原生协程函数返回。详见 [Await 表达式](#await-表达式)。

**基于生成器的协程函数** – 基于生成器语法的协程。最常见的例子是使用 @asyncio.coroutine 装饰的函数。

**基于生成器的协程** – 从基于生成器的协程函数返回。

**协程** – *原生协程*或*基于生成器的协程*。

**协程对象** – *原生协程*对象或*基于生成器的协程*对象。

**类 Future 对象** – 具有 `__await__` 方法的对象，或具有 tp_as_async->am_await 函数并返回*迭代器*的 C 对象。可以在协程中被 await 表达式消耗。等待类 Future 对象的协程会挂起，直到类 Future 对象的 `__await__` 完成，并返回结果。详见 [Await 表达式](#await-表达式)。

**可等待对象** – *类 Future*对象或*协程*对象。详见 [Await 表达式](#await-表达式)。

**异步上下文管理器** – 异步上下文管理器具有 `__aenter__` 和 `__aexit__` 方法，可以与 async with 一起使用。详见 [异步上下文管理器和"async with"](#异步上下文管理器和-async-with-)。

**异步可迭代对象** – 具有 `__aiter__` 方法的对象，该方法必须返回*异步迭代器*对象。可以与 async for 一起使用。详见 [异步迭代器和"async for"](#异步迭代器和-async-for-)。

**异步迭代器** – 异步迭代器具有 `__anext__` 方法。详见 [异步迭代器和"async for"](#异步迭代器和-async-for-)。

## 过渡计划

为了避免与 async 和 await 关键字的后向兼容性问题，决定修改 tokenizer.c，使其：

- 识别 async def NAME 令牌组合；

- 在标记化 async def 块时，它将'async' NAME 令牌替换为 ASYNC，将'await' NAME 令牌替换为 AWAIT；

- 在标记化 def 块时，它将'async'和'await' NAME 令牌原样输出。

这种方法允许将新的语法特性（都仅在 async 函数中可用）与任何现有代码无缝组合。

一个在一段代码中同时具有"async def"和"async"属性的示例：

```python
class Spam:
  async = 42

async def ham():
  print(getattr(Spam, 'async'))
```

# 协程可以执行并打印'42'

### 后向兼容性

本提案保持 100% 的后向兼容性。

#### asyncio

asyncio 模块已适配并测试可以与协程和新语句一起工作。后向兼容性保持 100%，即所有现有代码都将按原样工作。

所需的更改主要是：

- 修改 @asyncio.coroutine 装饰器以使用新的 types.coroutine() 函数。

- 在 asyncio.Future 类中添加 `__await__ = __iter__` 行。

- 添加 ensure_future() 作为 async() 函数的别名。弃用 async() 函数。

#### asyncio 迁移策略

由于纯生成器不能从原生协程对象 yield from（详见与生成器的区别部分），建议确保所有基于生成器的协程在使用新语法之前都使用 @asyncio.coroutine 装饰。

#### CPython 代码库中的 async/await

CPython 中没有使用 await 名称。

async 主要由 asyncio 使用。我们通过将 async() 函数重命名为 ensure_future() 来解决这个问题（详见 asyncio 部分）。

async 关键字的另一个用途是在 Lib/xml/dom/xmlbuilder.py 中，用于为 DocumentLS 类定义 async = False 属性。它没有文档或测试，在 CPython 的其他地方都没有使用。它被一个 getter 替换，该 getter 引发 DeprecationWarning，建议使用 async_ 属性。'async'属性没有文档，在 CPython 代码库中也没有使用。

### 语法更新

语法更改相当小：

```
decorated: decorators (classdef | funcdef | async_funcdef)
async_funcdef: ASYNC funcdef

compound_stmt: (if_stmt | while_stmt | for_stmt | try_stmt | with_stmt | funcdef | classdef | decorated | async_stmt)

async_stmt: ASYNC (funcdef | with_stmt | for_stmt)

power: atom_expr ['**' factor]
atom_expr: [AWAIT] atom trailer*
```

### 弃用计划

async 和 await 名称将在 CPython 3.5 和 3.6 中软弃用。在 3.7 中，我们将把它们转换为正式的关键字。在 3.7 之前使 async 和 await 成为正式关键字可能会使人们将代码移植到 Python 3 变得更加困难。

## 设计考虑

### PEP 3152

Gregory Ewing 的 PEP 3152 提出了一种不同的协程机制（称为"cofunctions"）。一些关键点：

- 一个新关键字 codef 用于声明 cofunction。Cofunction 始终是一个生成器，即使其中没有 cocall 表达式。映射到本提案中的 async def。

- 一个新关键字 cocall 用于调用 cofunction。只能在 cofunction 内使用。映射到本提案中的 await（有一些差异，见下文）。

- 不使用 cocall 关键字无法调用 cofunction。

- cocall 在语法上要求后面有括号：atom: cocall | <existing alternatives for atom>
cocall: 'cocall' atom cotrailer* '(' [arglist] ')'
cotrailer: '[' subscriptlist ']' | '.' NAME

`cocall f(*args, **kwds)` 在语义上等价于 `yield from f.__cocall__(*args, **kwds)`。

与本提案的差异：

本 PEP 中没有 `__cocall__` 的等价物，它在 cocall 表达式中被调用，其结果传递给 yield from。await 关键字期望一个可等待对象，验证类型，并在其上执行 yield from。虽然 `__await__` 方法类似于 `__cocall__`，但仅用于定义类 Future 对象。

- await 在语法上几乎与 yield from 相同的方式定义（稍后强制 await 只能在 async def 内）。可以简单地写 await future，而 cocall 总是需要括号。

- 为了使 asyncio 与 PEP 3152 一起工作，需要修改 @asyncio.coroutine 装饰器将所有函数包装在具有 `__cocall__` 方法的对象中，或者在生成器上实现 `__cocall__`。要从现有的基于生成器的协程调用 cofunction，需要使用 costart(cofunc, *args, **kwargs) 内置函数。

- 由于不使用 cocall 关键字无法调用 cofunction，它自动防止了忘记在基于生成器的协程上使用 yield from 的常见错误。本提案通过不同的方法解决了这个问题，详见调试功能。

- 需要 cocall 关键字来调用协程的一个缺点是，如果决定实现 coroutine-generators（具有 yield 或 async yield 表达式的协程），我们不需要 cocall 关键字来调用它们。所以我们将最终拥有__cocall__而没有__call__用于普通协程，以及拥有__call__而没有__cocall__用于 coroutine-generators。

- 在语法上需要括号也引入了很多问题。以下代码：

```python
await fut
await function_returning_future()
await asyncio.gather(coro1(arg1, arg2), coro2(arg1, arg2))
```

看起来会像：

```python
cocall fut() # 或 cocall costart(fut)
cocall (function_returning_future())()
cocall asyncio.gather(costart(coro1, arg1, arg2), costart(coro2, arg1, arg2))
```

PEP 3152 中没有 async for 和 async with 的等价物。

### Coroutine-generators

使用 async for 关键字，最好有一个 coroutine-generator 的概念——一个具有 yield 和 yield from 表达式的协程。为了避免与普通生成器的任何歧义，我们可能要求在 yield 之前有一个 async 关键字，async yield from 将引发 StopAsyncIteration 异常。

虽然可以实现 coroutine-generators，但我们认为它们超出了本提案的范围。这是一个高级概念，应该仔细考虑和权衡，需要对当前生成器对象的实现进行非琐碎的更改。这是另一个 PEP 的事情。

### 为什么选择"async"和"await"关键字

async/await 不是编程语言中的新概念：

C# 很久以前就有了 [5]；

- 在 ECMAScript 7 中添加 async/await 的提案 [2]；另见 Traceur 项目 [9]；

- Facebook 的 Hack/HHVM [6]；

- Google 的 Dart 语言 [7]；

- Scala [8]；

- 在 C++ 中添加 async/await 的提案 [10]；

- 以及许多其他不太流行的语言。

这是一个巨大的优势，因为一些用户已经有 async/await 的经验，而且因为它使在一个项目中使用多种语言更容易（例如 Python 与 ECMAScript 7）。

### 为什么 "\_\_aiter\_\_" 不返回可等待对象

PEP 492 在 CPython 3.5.0 中被接受，其中 `__aiter__` 被定义为一个方法，该方法应该返回一个解析为异步迭代器的可等待对象。

在 3.5.2（因为 PEP 492 是在临时基础上接受的）中，`__aiter__` 协议已更新为直接返回异步迭代器。

这一更改背后的动机是为了使在 Python 中实现异步生成器成为可能。更多详细信息请参阅 [19] 和 [20]。

### "async"关键字的重要性

虽然可以只实现 await 表达式并将所有至少有一个 await 的函数视为协程，但这种方法使 API 设计、代码重构和长期支持变得更加困难。

让我们假设 Python 只有 await 关键字：

```python
def useful():
  ...
  await log(...)
  ...

def important():
  await useful()
```

如果 useful() 函数被重构，有人从中删除了所有 await 表达式，它将变成一个普通的 Python 函数，所有依赖于它的代码，包括 important()，都将无法工作。为了缓解这个问题，必须引入类似于 @asyncio.coroutine 的装饰器。

### 为什么是"async def"

对于一些人来说，`async name(): pass` 语法可能看起来比 `async def name(): pass` 更有吸引力。当然它更容易输入。但另一方面，它打破了 async def、async with 和 async for 之间的对称性，其中 async 是一个修饰符，表示该语句是异步的。它也符合现有的语法。

### 为什么不是"await for"和"await with"

async 是一个形容词，因此它作为语句限定符关键字的更好选择。await for/with 将意味着某个东西在等待 for 或 with 语句完成。

### 为什么是"async def"而不是"def async"

async 关键字是一个语句限定符。一个很好的类比是其他语言中的"static"、"public"、"unsafe"关键字。"async for"是一个异步"for"语句，"async with"是一个异步"with"语句，"async def"是一个异步函数。

将"async"放在主语句关键字之后可能会引入一些混淆，例如"for async item in iterator"可以被读作"for each asynchronous item in iterator"。

将 async 关键字放在 def、with 和 for 之前也使语言语法更简单。而且"async def"在视觉上更好地将协程与普通函数区分开来。

### 为什么不是 __future__ import

过渡计划部分解释了如何修改 tokenizer 以仅在 async def 块中将 async 和 await 视为关键字。因此 async def 填充了模块级编译器声明（如 `from __future__ import async_await`）的角色。

### 为什么魔术方法以"a"开头

新的异步魔术方法 `__aiter__`、`__anext__`、`__aenter__` 和 `__aexit__` 都以相同的前缀"a"开头。一个替代方案是使用"async"前缀，这样 `__anext__` 就变成了 `__async_next__`。然而，为了将新的魔术方法与现有的魔术方法（如 `__radd__` 和 `__iadd__`）对齐，决定使用较短的版本。

### 为什么不重用现有的魔术名称

关于新的异步迭代器和上下文管理器的一个替代想法是重用现有的魔术方法，通过在其声明中添加 async 关键字：

```python
class CM:
  async def __enter__(self): # 而不是__aenter__
    ...
```

这种方法有以下缺点：

- 将无法创建一个同时在 with 和 async with 语句中工作的对象；

- 它将破坏后向兼容性，因为没有什么阻止在 Python <= 3.4 中从 `__enter__` 和/或 `__exit__` 返回类 Future 对象；

- 本提案的主要要点之一是使原生协程尽可能简单和防错，因此协议的清晰分离。

### 为什么不重用现有的"for"和"with"语句

现有基于生成器的协程和本提案背后的愿景是让用户能够轻松看到代码可能在何处挂起。使现有的"for"和"with"语句识别异步迭代器和上下文管理器将不可避免地创建隐式挂起点，使代码更难理解。

### 推导式

可以为异步推导式提供语法，但该构造超出了本提案的范围。

### 异步 lambda 函数

可以为异步 lambda 函数提供语法，但该构造超出了本提案的范围。

### async/await

以下微基准测试用于确定"async"函数和生成器之间的性能差异：

```python
import sys
import time

def binary(n):
  if n <= 0:
    return 1
  l = yield from binary(n - 1)
  r = yield from binary(n - 1)
  return l + 1 + r

async def abinary(n):
  if n <= 0:
    return 1
  l = await abinary(n - 1)
  r = await abinary(n - 1)
  return l + 1 + r

def timeit(func, depth, repeat):
  t0 = time.time()
  for _ in range(repeat):
    o = func(depth)
    try:
      while True:
        o.send(None)
    except StopIteration:
      pass
  t1 = time.time()
  print('{}({}) * {}: total {:.3f}s'.format(
    func.__name__, depth, repeat, t1-t0))
```


### 高级更改和新协议列表

- 定义协程的新语法：async def 和新的 await 关键字。

- 用于类 Future 对象的新 `__await__` 方法，以及 PyTypeObject 中的新 tp_as_async.am_await 槽。

- 用于异步上下文管理器的新语法：async with。以及相关的协议，具有 __aenter__ 和 __aexit__ 方法。

- 用于异步迭代的新语法：async for。以及相关的协议，具有 `__aiter__`、`__aexit__` 和新的内置异常 StopAsyncIteration。PyTypeObject 中的新 tp_as_async.am_aiter 和 tp_as_async.am_anext 槽。

- 新的 AST 节点：AsyncFunctionDef、AsyncFor、AsyncWith、Await。

- 新函数：sys.set_coroutine_wrapper(callback)、sys.get_coroutine_wrapper()、types.coroutine(gen)、inspect.iscoroutinefunction(func)、inspect.iscoroutine(obj)、inspect.isawaitable(obj)、inspect.getcoroutinestate(coro) 和 inspect.getcoroutinelocals(coro)。

- 代码对象的新 CO_COROUTINE 和 CO_ITERABLE_COROUTINE 位标志。

- 新的 ABC：collections.abc.Awaitable、collections.abc.Coroutine、collections.abc.AsyncIterable 和 collections.abc.AsyncIterator。

- C API 更改：新的 PyCoro_Type（在 Python 中暴露为 types.CoroutineType）和 PyCoroObject。PyCoro_CheckExact(*o*) 用于测试 o 是否为原生协程。

虽然更改和新内容的列表不长，但重要的是要理解，大多数用户将不会直接使用这些功能。它旨在在框架和库中使用，以为用户提供方便使用和明确的 API，具有 async def、await、async for 和 async with 语法。

### 工作示例

本 PEP 中提出的所有概念都已实现 [3] 并且可以测试。

```python
import asyncio

async def echo_server():
  print('Serving on localhost:8000')
  await asyncio.start_server(handle_connection,
                            'localhost', 8000)

async def handle_connection(reader, writer):
  print('New connection...')

  while True:
    data = await reader.read(8192)

    if not data:
      break

    print('Sending {:.10}... back'.format(repr(data)))
    writer.write(data)

loop = asyncio.get_event_loop()
loop.run_until_complete(echo_server())
try:
  loop.run_forever()
finally:
  loop.close()
```
